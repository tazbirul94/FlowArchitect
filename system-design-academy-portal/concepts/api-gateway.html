API Gateway

A single entry point for all client requests — handling authentication, routing, rate limiting, SSL termination, and request transformation before traffic reaches your backend services.

Intermediate Networking Microservices Infrastructure

Definition & Analogy

Simple Analogy

Think of an API Gateway as the front desk of a large corporate building. Every visitor — employees, couriers, clients — enters through the same entrance. The receptionist checks their ID (authentication), directs them to the right floor (routing), limits how many people from a single company can enter at once (rate limiting), and logs everyone who comes through (observability). No one knows or cares how many floors are in the building, or how many people work on each one.

In a microservices architecture, clients would otherwise need to know the address of every service — user-service:3001, ticket-service:8080, payment-service:5000 — and handle authentication themselves. An API Gateway collapses all of that into one endpoint. It receives every request, applies cross-cutting concerns (auth, rate limiting, logging), and routes the request to the correct backend service. The client only ever talks to the gateway.

This is why the API Gateway pattern emerged alongside microservices. When you go from one monolith to dozens of services, the question of "who handles auth?" and "where do I put rate limiting?" has a clean answer: the gateway, once, at the edge.

The Story

Real-World Story
A travel booking startup decomposed their monolith into 12 microservices over 18 months: search-service, booking-service, payment-service, notification-service, and eight others. Each team owned their service and exposed REST endpoints directly. The mobile app and web app called each service directly using individual base URLs hardcoded in client code.
Every service implemented its own JWT validation — six slightly different ways. When the security team found a vulnerability in the token parsing logic, they had to patch 12 services separately, coordinating 12 deployments. The mobile app had 12 different base URLs; when a service moved, the app broke in production until users force-updated. Rate limiting didn't exist, and a buggy mobile client hammered search-service with 10,000 requests/min, starving real users. An audit found that 3 services had never validated tokens at all — the auth had silently been skipped in a refactor six months ago.
The team introduced an API Gateway (Kong) in front of all services. All clients now call api.travelapp.io. The gateway handles JWT validation once — a single plugin, a single patch point. Path-based routing sends /search/* to search-service and /bookings/* to booking-service. Rate limiting is configured per route: search allows 100 req/min per client, payment allows 10. The gateway logs every request centrally for the audit team.
The next security patch was a single config change deployed in 3 minutes, not 12 coordinated rollouts. When notification-service was rewritten in a different language and moved to a new host, clients noticed nothing — only the gateway routing config changed. The buggy mobile client that fired 10,000 req/min was automatically throttled; real users saw no impact. Client-side code dropped from 1,800 lines of service-discovery boilerplate to 120 lines calling a single base URL.

Learn at Your Level

Beginner

Imagine you run an airport. Passengers (clients) arrive wanting to reach their destination (backend service). Without an API Gateway, every passenger would have to navigate directly to each terminal, go through security at every gate, and figure out the boarding process on their own. Chaos.

An API Gateway is the main terminal entrance. Every passenger goes through one place. Security is done once. Staff direct everyone to the right gate (routing). If too many passengers arrive at once, they queue or some are turned away (rate limiting). The airline can move gates around behind the scenes and passengers never know — they still go to the same entrance.

In software: clients only know one URL. The gateway handles all the complicated routing, security checks, and logging behind the scenes. Backend services can change freely without breaking clients.

What an API Gateway Does

  • Routing — sends /api/users to user-service, /api/orders to order-service
  • Authentication — checks if the user's token is valid before the request reaches any backend
  • Rate Limiting — stops one client from overwhelming the system with too many requests
  • SSL Termination — handles HTTPS so backend services can use plain HTTP internally
  • Logging — records every request in one place for debugging and auditing

Key Terms

  • Route — a rule that maps a URL path to a backend service
  • Plugin/Middleware — a piece of logic the gateway runs on every request (e.g., JWT validation, rate limiting)
  • Upstream — the backend service that the gateway forwards requests to
  • Downstream — the client that sends requests to the gateway
  • BFF (Backend for Frontend) — a pattern where different clients (mobile, web) get their own gateway tailored to their needs

An API Gateway operates at Layer 7 (HTTP/HTTPS) of the network stack. This means it can inspect and modify request/response payloads, headers, and paths — not just IP addresses and ports. This is what makes it fundamentally different from a load balancer.

Request Flow Through the Gateway

  1. Client sends GET /api/users/42/profile to api.example.com with Authorization: Bearer <jwt>
  2. Gateway receives the request, starts processing pipeline
  3. Rate limit plugin checks: client IP has 45/100 requests used this minute — allow
  4. Auth plugin validates the JWT: signature valid, not expired, user has read:profile scope
  5. Router matches /api/users/* → user-service:3001
  6. Request transformation: gateway strips the Authorization header and injects X-User-ID: 42 and X-User-Role: member
  7. Gateway forwards GET /users/42/profile to user-service:3001 with injected headers
  8. user-service responds with 200 OK and JSON body
  9. Gateway logs: method, path, upstream, latency, status code
  10. Gateway returns response to client

Path-Based Routing Configuration (NGINX)

upstream user_service {
    server user-service:3001;
    keepalive 32;
}

upstream ticket_service {
    server ticket-service:8080;
    keepalive 32;
}

upstream payment_service {
    server payment-service:5000;
    keepalive 16;
}

server {
    listen 443 ssl http2;
    server_name api.example.com;

    ssl_certificate     /etc/ssl/api.example.com.crt;
    ssl_certificate_key /etc/ssl/api.example.com.key;
    ssl_protocols       TLSv1.2 TLSv1.3;

    # JWT validation via auth_request
    location = /auth {
        internal;
        proxy_pass http://auth-service:4000/validate;
        proxy_pass_request_body off;
        proxy_set_header Content-Length "";
        proxy_set_header X-Original-URI $request_uri;
    }

    location /api/users/ {
        auth_request /auth;
        auth_request_set $user_id $upstream_http_x_user_id;

        proxy_pass http://user_service/;
        proxy_set_header X-User-ID $user_id;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header Host $host;
    }

    location /api/tickets/ {
        auth_request /auth;
        auth_request_set $user_id $upstream_http_x_user_id;

        proxy_pass http://ticket_service/;
        proxy_set_header X-User-ID $user_id;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }

    location /api/payments/ {
        auth_request /auth;
        auth_request_set $user_id $upstream_http_x_user_id;

        # Payments get a stricter rate limit
        limit_req zone=payment_zone burst=5 nodelay;

        proxy_pass http://payment_service/;
        proxy_set_header X-User-ID $user_id;
    }
}

Core Gateway Capabilities

  • SSL/TLS Termination — HTTPS terminates at the gateway; backend services use plain HTTP on the internal network, reducing CPU overhead and simplifying cert management
  • JWT Validation at Edge — validate tokens once at the gateway, inject parsed claims (user ID, roles) as headers to backend services — backends never touch JWT parsing code
  • Rate Limiting — per-client (IP or API key), per-route, per-service — prevents one misbehaving client from taking down the entire system
  • Request/Response Transformation — add/remove/rename headers; transform request body for legacy backends; aggregate responses from multiple services
  • Observability — centralized access logs, distributed trace injection, latency metrics per route — no need to instrument each service separately
Design Principle

The gateway is a thin pipe. It should handle cross-cutting infrastructure concerns — auth, routing, rate limiting, logging. It should never contain business logic. If you find yourself writing if user.country == "US": route_to(us_service) else: route_to(eu_service) based on user data you had to query a database for, that logic belongs in a dedicated service, not the gateway.

At scale, the API Gateway is simultaneously your most critical infrastructure component and your most common architectural mistake. The gateway's design directly controls system latency, availability, and the cognitive load on every team that ships APIs.

BFF Pattern — Backend for Frontend

A single gateway serving all clients creates tension: mobile apps want small payloads and aggregated data; web apps want richer responses; third-party API consumers want stability and versioning. The BFF (Backend for Frontend) pattern solves this with separate gateways per client type.

The mobile BFF aggregates a GET /home call into one response by calling user-service, ticket-service, and notification-service in parallel — saving the mobile client three round trips. The web BFF returns richer data since bandwidth is less constrained. The public API gateway enforces strict versioning (/v1/, /v2/) and has harsher rate limits since third-party consumers are untrusted.

Circuit Breaking at Gateway Layer

A well-configured gateway should detect when a backend service is failing and stop routing traffic to it — rather than letting requests pile up and time out. This is circuit breaking at the gateway layer, and it prevents one failing service from cascading to slow down the entire gateway.

# Traefik circuit breaker via middleware
apiVersion: traefik.io/v1alpha1
kind: Middleware
metadata:
  name: payment-circuit-breaker
  namespace: production
spec:
  circuitBreaker:
    # Trip breaker when 50% of requests fail over 30s
    expression: "ResponseCodeRatio(500, 600, 0, 600) > 0.5"
    # Check every 10 seconds
    checkPeriod: 10s
    # Stay open for 30 seconds before half-open retry
    fallbackDuration: 30s
    recoveryDuration: 10s

---
# IngressRoute using the circuit breaker
apiVersion: traefik.io/v1alpha1
kind: IngressRoute
metadata:
  name: payment-routes
spec:
  entryPoints: [websecure]
  routes:
    - match: PathPrefix(`/api/payments`)
      kind: Rule
      middlewares:
        - name: payment-circuit-breaker
        - name: rate-limit-payments
        - name: jwt-auth
      services:
        - name: payment-service
          port: 5000

When NOT to Use an API Gateway

  • Monolithic applications — A gateway adds a network hop and operational overhead for no benefit when you have one service.
  • Internal service-to-service communication — Services calling each other internally don't need to go through the gateway. Use service mesh (sidecar proxies) for east-west traffic. The gateway handles north-south (client to cluster) traffic only.
  • Early-stage startups with 1–3 services — The operational overhead (TLS certs, HA config, monitoring) is not worth it until you have enough services to justify the abstraction. A well-configured NGINX reverse proxy is sufficient for 3 services.
  • Latency-critical paths where 5–30ms matters — AWS API Gateway's documented overhead (~29ms) is measurable. For latency-sensitive workloads (HFT, gaming), design to minimize gateway hops or use a lower-overhead proxy.
Anti-Pattern Alert

The Smart Gateway Anti-Pattern: Teams slowly add business logic to the gateway — "it's faster to put this check at the edge." The gateway ends up with database calls, complex conditional routing, and shared business rules. It becomes a distributed monolith with all the operational pain of a gateway plus all the coupling of a monolith. Keep the gateway dumb. Route, auth, rate-limit, log. Everything else belongs in services.

At enterprise scale, the API Gateway strategy evolves from "one gateway" to a layered traffic management architecture. The gateway decisions made early become load-bearing architectural constraints that are very expensive to change later.

Netflix: From Zuul to Envoy

Netflix Case Study Netflix — Gateway Evolution (2012–2023)

Phase 1 — Zuul 1 (2012): Netflix open-sourced Zuul as their edge gateway. Zuul 1 used a blocking I/O model (one thread per connection). At 100K req/sec, this required thousands of threads — high memory, high context-switch overhead. Max throughput was bounded by thread pool exhaustion.

Phase 2 — Zuul 2 (2016): Rewrote Zuul with async, non-blocking I/O (Netty-based). A small number of event loop threads handled thousands of concurrent connections. Same hardware handled 3× more traffic. The 2B+ daily requests Netflix processes go through this layer.

Phase 3 — Envoy at the Edge (2020+): Netflix began migrating portions of their edge to Envoy. Reasons: xDS API for dynamic configuration without restarts, native gRPC support, richer observability (distributed tracing with headers injected at L7), and alignment with the CNCF ecosystem. Zuul and Envoy coexist for different traffic classes.

Scale numbers: Netflix serves 238M+ subscribers. Peak traffic: approximately 15% of global internet bandwidth in the evening. The edge gateway infrastructure handles over 2 billion API requests per day, with p99 latency targets of under 50ms for gateway processing alone.

Key lesson: The blocking I/O vs non-blocking I/O decision in a gateway is not academic. At 10K req/sec, it doesn't matter. At 500K req/sec, it's the difference between 500 servers and 150 servers.

API Gateway vs Service Mesh

The most common architectural confusion in post-2020 microservices: "Do I need a service mesh (Istio) or an API gateway?" The answer is usually both, for different purposes.

Dimension API Gateway Service Mesh (Istio/Linkerd)
Traffic direction North-South (client ↔ cluster boundary) East-West (service ↔ service inside cluster)
Auth JWT, API key, OAuth2 — external client auth mTLS — mutual TLS between services (workload identity)
Configuration Route-based config (paths, methods, headers) Sidecar-injected per pod — no app code changes
Observability Request logs, gateway metrics Full service topology, distributed traces, per-service SLO dashboards
Operational complexity Moderate — one component to manage High — sidecar injection, control plane (Istiod), CRDs, debug complexity
Latency overhead 1–30ms per request (varies by product) ~1ms per hop (sidecar adds one hop per service call)
Architecture Decision Rule

Use an API Gateway for north-south (ingress) traffic. Use a service mesh for east-west (intra-cluster) traffic if you need mTLS, zero-trust networking, or fine-grained traffic policies between services. You don't need a service mesh until you have east-west security requirements, multi-cluster traffic management, or fine-grained canary deployments between internal services. Most teams at under 50 services should start with just a gateway.

Compliance and Multi-Region Gateway Design

GDPR and data residency regulations affect gateway routing design directly. A user in Germany making requests to api.example.com must have their data processed in the EU. This requires geo-aware routing at the gateway layer.

Production Operations Checklist

  • Gateway deployed with at least 2 replicas behind a load balancer (no SPOF)
  • Health check endpoint on gateway itself monitored by external synthetic
  • All routes version-controlled in declarative config (Kong deck / Terraform / Helm values)
  • JWT secret rotation tested — gateway can hot-reload key without downtime
  • Rate limit thresholds documented per route with reasoning
  • Gateway latency tracked separately (p50, p95, p99) — not bundled with backend latency
  • Circuit breaker thresholds configured for all payment/critical routes
  • Access logs shipped to SIEM — PCI compliance requires log retention
  • mTLS between gateway and internal services (zero-trust posture)
  • Runbook for gateway emergency bypass (direct service access during gateway outage)
  • Load test gateway independently at 2× expected peak before production launch
  • TLS 1.2 minimum enforced; TLS 1.3 preferred; cipher suites audited quarterly

Enterprise Use Case

Enterprise Use Case Netflix — 2B+ Requests Per Day Through a Single Edge Layer

Context: Netflix serves 238 million subscribers across 190 countries. Their backend consists of hundreds of microservices. Every stream start, playback event, search query, and account action flows through their edge gateway layer before reaching any backend service.

Architecture:

  1. Clients (smart TVs, mobile, browser) call a single API endpoint — the edge gateway cluster
  2. The gateway validates session tokens using a distributed token validation service (sub-millisecond latency via in-memory cache)
  3. Country-of-origin is extracted from the request — EU requests route to EU-region services (GDPR compliance)
  4. Path routing sends /api/title to the content metadata cluster, /api/playback to the playback service, /api/account to the account service
  5. Rate limiting: 1,000 req/min per authenticated account — protects against credential stuffing and account scraping
  6. Every request injects a X-Netflix-Device-Id and X-Netflix-Request-Id header for distributed tracing across 100+ microservices downstream

Numbers: Peak throughput exceeds 600,000 requests per second globally. The Zuul 2 / Envoy gateway layer adds ~3–8ms of latency overhead. Gateway infrastructure runs on auto-scaled EC2 clusters with target: never less than 30% headroom capacity above current peak. An entire region can be evacuated (traffic shifted to adjacent region) in under 2 minutes using DNS-based routing.

Lessons for your team:

  1. Non-blocking I/O is non-negotiable at scale — Blocking thread-per-request models hit a hard ceiling. Async gateways (Netty, Envoy, NGINX event loop) handle the same load at a fraction of the memory. Migrate before you hit the ceiling, not after.
  2. Token validation must be in-memory — Calling a database or remote service for every request to validate a JWT adds 20–100ms per call. Cache validated tokens in-memory with a short TTL (30–60 seconds). Revocation is handled via short expiry + revocation list check, not per-request DB calls.
  3. Trace injection at the gateway is worth its weight in gold — Injecting a single request ID at the gateway allows you to correlate a client-reported bug to the exact chain of service calls that caused it. Without this, debugging distributed systems is forensic archaeology.

Common Mistakes

Putting Business Logic in the Gateway

The most insidious API gateway mistake: gradually moving business decisions into the gateway because "it's faster to reject at the edge." First it's checking a feature flag. Then it's looking up the user's subscription tier to decide which backend to route to. Then it's calling a database to verify account status. Now your gateway has application logic, database dependencies, and service coupling. It has become the very monolith you decomposed.

The gateway must remain a dumb pipe for infrastructure concerns. Routing rules should be based on request properties (path, headers, method) — never on data fetched from a database. If you need dynamic routing based on business data, build a dedicated routing service and have the gateway call it — keeping the logic testable and independently deployable.

Single Gateway Instance — Single Point of Failure

A team adds an API gateway to eliminate SPOFs on backend services. The gateway itself is deployed as a single instance. Now the entire system goes down when the gateway goes down — and it will. Gateway processes crash, nodes get evicted by the scheduler, TLS certificates expire and the process fails to restart.

Every production gateway must run with at least two replicas behind a load balancer, with health checks on each. A gateway cluster going from 3 to 2 instances should page an engineer; going to 1 should be treated as an incident. Use a load balancer (AWS ALB, NGINX, or cloud NLB) in front of the gateway replicas — the irony of load balancing your load balancer is real but necessary.

Not Monitoring Gateway Latency Separately

Teams bundle gateway latency into backend service latency and then spend weeks investigating why payment-service p99 is "high" — when the actual problem is gateway JWT validation taking 150ms due to an expired key cache and falling back to a remote validation endpoint. Because all latency was attributed to the backend, the gateway was never in scope for the investigation.

Gateway latency must be tracked independently: total gateway processing time (before forwarding to upstream), auth plugin latency, and upstream response time. A dashboard showing "gateway overhead = X ms" separate from "upstream latency = Y ms" is the difference between 10 minutes and 3 days to root-cause a latency regression. Export these as separate metrics: gateway_auth_duration_ms, gateway_routing_duration_ms, gateway_total_duration_ms.

Interview Prep

An API Gateway is a single entry point that handles authentication, routing, rate limiting, SSL termination, and observability for all client-to-service traffic — centralizing cross-cutting concerns so each backend service stays focused on its business logic.

2-Minute Structured Answer

"An API Gateway solves the problem of cross-cutting concerns in microservices. Without one, every service needs to implement auth, rate limiting, and logging individually — which means 12 services implementing JWT validation 12 different ways, and patching a security vulnerability requires 12 coordinated deployments. The gateway collapses all of that into one place. Clients call one URL; the gateway validates tokens, enforces rate limits, routes by path to the correct backend, injects parsed user context as headers, and logs everything centrally. I think of it as the public API contract between the outside world and your internal service topology. For example, Netflix handles over 2 billion requests per day through their edge gateway layer, which adds roughly 3–8ms of overhead but gives them centralized observability and consistent security policy across hundreds of services. The key trade-off is that the gateway becomes your most critical component — it must be HA, low-latency, and operationally simple, because if it goes down, everything goes down."

Common Interview Mistakes

  • Confusing an API Gateway with a load balancer — a load balancer distributes traffic across instances of the same service; a gateway routes to different services by path and handles application-layer concerns like auth.
  • Not mentioning high availability — interviewers expect you to note that the gateway must itself be fault-tolerant, not just the services behind it.
  • Treating the gateway as the right place for business logic — it should be a thin infrastructure pipe, not an application layer. This distinction matters to senior engineers.
  • Ignoring the latency impact — AWS API Gateway adds ~29ms; NGINX adds ~0.5ms; these differences are architecturally significant at scale. Know the numbers.
