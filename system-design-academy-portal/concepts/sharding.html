Sharding

Horizontally partitions a database across multiple machines so each node holds a subset of the data — enabling scale beyond what a single server can store or serve.

Advanced Data Databases Scalability

Definition & Analogy

Simple Analogy

Imagine a library that has grown too large for one building. Instead of buying a bigger building, you split the collection: books A–F go to Branch 1, G–M to Branch 2, N–Z to Branch 3. Each branch holds a manageable chunk, and a central directory tells you which branch has the book you need.

Sharding does the same for a database: rows are split across multiple database servers (shards) based on a shard key. A router knows which shard holds any given row and sends queries there directly.

Vertical scaling (bigger machine) has a ceiling — at some point you cannot buy a server large enough or fast enough. Sharding breaks that ceiling by distributing data horizontally across many commodity machines. Instagram stores billions of photos, Twitter serves billions of tweets, and Uber tracks millions of rides per day — none of these fits on a single database server. Sharding is how they scale their data tier.

The key distinction: sharding (horizontal partitioning) splits rows across servers. Vertical partitioning splits columns across servers — keeping "hot" frequently-accessed columns together. Both are tools; sharding is what most people mean when they talk about database scaling beyond replication.

The Story

Real-World Story
A social platform launched with a single PostgreSQL instance. At 500K users it handled everything fine — 8-core server, 64 GB RAM, 2 TB SSD. The team added read replicas when read load grew, and that bought another year. User count hit 12 million. The posts table crossed 4 billion rows. Writes were queueing; primary CPU was at 85% continuously.
Adding more read replicas no longer helped — the bottleneck was write throughput to a single primary. Upgrading to a 96-core machine with 768 GB RAM cost $40K/month and still had the same fundamental constraint: all writes serialized through one node. Table vacuums started falling behind, bloat grew, and query plans degraded. The single-server model had hit its wall.
The team sharded by user_id using consistent hashing across 8 shards. Each shard got its own PostgreSQL primary with two replicas. A lightweight router layer (built on PgBouncer + custom middleware) mapped user_id % 8 to the correct shard. The migration ran online: dual-write to old and new during cutover, then flip the read path, then decommission the monolith.
Each shard now holds ~500M rows instead of 4B. Write throughput scaled 8x. P99 write latency dropped from 340ms to 28ms. CPU across all shards sits at 20–30% during peak. The team can add more shards independently when any single shard grows too large — without touching the others. Operational cost actually went down: 8 standard cloud instances are cheaper than one ultra-high-end machine.

Learn at Your Level

Beginner

Think about a phone book. One city's phone book is one thick volume. A country's phone book would be impossibly large. So they split it: one volume per city, or alphabetically across multiple volumes. If you want to find "Smith", you know to go to the S–Z volume without looking at A–R.

Sharding is the same idea for a database. Instead of one giant table with all user data, you split the rows across multiple database servers. User IDs 1–1,000,000 go to Server 1. User IDs 1,000,001–2,000,000 go to Server 2. And so on. Each server is called a shard. The value used to decide which shard a row belongs to (in this case user_id) is called the shard key.

When your application needs user #1,500,000, it knows to go to Server 2. It doesn't have to search all servers — it goes directly to the right one.

Key Terms

  • Shard — One of the database server partitions. Each shard is a full database that holds a subset of rows.
  • Shard key — The column (or combination of columns) used to decide which shard a row belongs to. Choosing this well is the most important decision in sharding.
  • Shard router — Middleware that looks at the shard key in a query and routes it to the correct shard. The application talks to the router, not individual shards directly.
  • Horizontal partitioning — The technical term for sharding. "Horizontal" means splitting by rows (as opposed to "vertical" partitioning which splits by columns).

Consistent Hash Ring

A consistent hash ring is how many systems decide which shard holds a given key. Imagine arranging all possible hash values (0 to 232) in a circle. Each shard node sits at a position on the ring. A key's hash value is mapped to the ring, and it goes to the first node clockwise from that position.

The ring diagram above shows four shards, each responsible for a hash range. When a new shard is added, only the keys in the adjacent range need to move — not all the data. This minimises data migration compared to a simple modulo approach.

There are three primary sharding strategies, each with clear trade-offs. Choosing the right one depends on your access patterns.

Strategy 1 — Range Sharding

Split data into contiguous ranges of the shard key. For example: users 1–1,000,000 on Shard 1, users 1,000,001–2,000,000 on Shard 2.

-- Range sharding on user_id
-- Shard 1: user_id BETWEEN 1 AND 1000000
-- Shard 2: user_id BETWEEN 1000001 AND 2000000
-- Shard 3: user_id BETWEEN 2000001 AND 3000000

-- Router logic (pseudocode)
function get_shard(user_id):
    if user_id <= 1_000_000:  return shard_1
    if user_id <= 2_000_000:  return shard_2
    if user_id <= 3_000_000:  return shard_3
    raise Error("shard not found")

Good for: Range queries — "get all orders from Q1" maps to a single shard if ordered by date. Easy to reason about.
Bad for: Sequential auto-increment IDs — all new rows hit the last shard (hotspot). Requires shard rebalancing as data grows unevenly.

Strategy 2 — Hash Sharding

Apply a hash function to the shard key and take the modulo. This distributes rows evenly but destroys range ordering.

import hashlib

def get_shard(user_id: int, num_shards: int = 8) -> int:
    """Assign user to shard using MD5 hash mod N."""
    key_bytes = str(user_id).encode('utf-8')
    hash_value = int(hashlib.md5(key_bytes).hexdigest(), 16)
    return hash_value % num_shards

# user_id=1042  → shard 3
# user_id=1043  → shard 7  (adjacent IDs spread across shards)
# user_id=5000  → shard 1
# Even distribution, but range queries require hitting all shards

Good for: Even write distribution, no hotspots, simple routing logic.
Bad for: Range queries ("get all users registered in March") scatter to all shards — expensive scatter-gather required.

Strategy 3 — Directory Sharding

A lookup table explicitly maps each key (or key range) to a shard. Maximum flexibility, but the directory becomes a critical component.

-- Directory table (stored in Redis or dedicated metadata DB)
CREATE TABLE shard_directory (
    tenant_id   VARCHAR(64) PRIMARY KEY,
    shard_id    INT NOT NULL,
    shard_host  VARCHAR(255) NOT NULL,
    created_at  TIMESTAMP DEFAULT NOW()
);

INSERT INTO shard_directory VALUES
  ('acme-corp',   3, 'shard3.db.internal', NOW()),
  ('globex',      1, 'shard1.db.internal', NOW()),
  ('initech',     3, 'shard3.db.internal', NOW());  -- same shard as acme

-- Router: SELECT shard_host FROM shard_directory WHERE tenant_id = ?

Good for: Multi-tenant SaaS where tenants vary wildly in size. A large tenant can get its own dedicated shard. Flexible shard reassignment.
Bad for: Directory is a single point of failure (needs HA). Every query requires a directory lookup (latency overhead, mitigated by caching).

Shard Key Selection Criteria

  • High cardinality — The key must have many distinct values (e.g., user_id, not boolean active/inactive). Low cardinality means you can't have more shards than distinct key values.
  • Even distribution — Values must spread evenly across shards. Auto-increment IDs with hash sharding distribute evenly. Timestamps with range sharding skew toward recent data.
  • Avoid hotspots — If 80% of your traffic is for a single user (e.g., a celebrity), putting that user on one shard creates a hotspot. Consider composite keys or secondary sharding.
  • Align with query patterns — Queries that filter by the shard key hit one shard. Queries that filter by other columns scatter to all shards. Choose the key that matches your most critical query.
  • Immutability — The shard key should not change after record creation. Changing a shard key means migrating the row to a different shard — operationally painful.

The Hotspot Problem

Even with perfect hash distribution, certain keys attract disproportionate traffic. Classic examples:

  • Celebrity key — A user with 100M followers: every follow/unfollow, every tweet, every profile view hits the same shard. Kylie Jenner's first Instagram post received 18M likes; all those writes went to one shard.
  • Viral content — A post that goes viral generates millions of reads per minute from a single row on a single shard.
  • Temporal hotspots — With timestamp-based range sharding, all new writes hit the "current" shard. Yesterday's shards sit idle.

Mitigation strategies:

  • Shard splitting — Split the hot shard into multiple sub-shards. Requires rerouting logic update.
  • Key salting — Append a random suffix to the key (user_id_0 through user_id_9) to spread writes across multiple shards. Reads must fan out to all suffixed variants and merge results.
  • Read replicas on hot shards — Add read replicas specifically to heavily-read shards. Writes still concentrate but reads distribute.
  • Application-level caching — Cache viral content at the application layer. The database shard only gets cache misses.
import random

# Key salting to spread hot user writes across 10 sub-shards
SALT_RANGE = 10

def write_user_counter(user_id: int, increment: int):
    """Spread writes for a hot user across multiple shards via salting."""
    salt = random.randint(0, SALT_RANGE - 1)
    salted_key = f"{user_id}_{salt}"
    shard = get_shard(salted_key)
    shard.incr(f"counter:{salted_key}", increment)

def read_user_counter(user_id: int) -> int:
    """Read requires fan-out to all salt variants and sum."""
    total = 0
    for salt in range(SALT_RANGE):
        salted_key = f"{user_id}_{salt}"
        shard = get_shard(salted_key)
        total += shard.get(f"counter:{salted_key}") or 0
    return total
    # Trade-off: writes are 10x faster (parallelised), reads are 10x more requests

Resharding — Adding Shards

Adding a shard to a running system is one of the hardest operations in sharding. With simple hash mod N, changing N from 4 to 5 remaps almost every key — a full data migration while serving live traffic.

Consistent hashing solves this: only the data between the new node and its predecessor in the ring needs to move. With 4 nodes and adding a 5th, only ~20% of data migrates instead of ~80%.

import hashlib
import bisect
from typing import Optional

class ConsistentHashRing:
    """
    Minimal consistent hash ring implementation.
    Virtual nodes (replicas) ensure even distribution even with few physical nodes.
    """
    def __init__(self, virtual_nodes: int = 150):
        self.virtual_nodes = virtual_nodes
        self.ring: dict[int, str] = {}     # hash_position -> node_id
        self.sorted_keys: list[int] = []    # sorted list of hash positions

    def add_node(self, node_id: str) -> None:
        for i in range(self.virtual_nodes):
            vnode_key = f"{node_id}:vnode:{i}"
            h = self._hash(vnode_key)
            self.ring[h] = node_id
            bisect.insort(self.sorted_keys, h)

    def remove_node(self, node_id: str) -> None:
        for i in range(self.virtual_nodes):
            vnode_key = f"{node_id}:vnode:{i}"
            h = self._hash(vnode_key)
            self.ring.pop(h, None)
            idx = bisect.bisect_left(self.sorted_keys, h)
            if idx < len(self.sorted_keys) and self.sorted_keys[idx] == h:
                self.sorted_keys.pop(idx)

    def get_node(self, key: str) -> Optional[str]:
        if not self.ring:
            return None
        h = self._hash(key)
        idx = bisect.bisect(self.sorted_keys, h) % len(self.sorted_keys)
        return self.ring[self.sorted_keys[idx]]

    @staticmethod
    def _hash(key: str) -> int:
        return int(hashlib.md5(key.encode()).hexdigest(), 16)

# Usage
ring = ConsistentHashRing(virtual_nodes=150)
ring.add_node("shard-a")
ring.add_node("shard-b")
ring.add_node("shard-c")

print(ring.get_node("user:1042"))   # e.g. "shard-b"
print(ring.get_node("user:9999"))   # e.g. "shard-a"

# Adding shard-d only migrates ~25% of keys (1/4 of the ring)
ring.add_node("shard-d")
Architect Note — Virtual Nodes

Without virtual nodes, 3 physical nodes on a ring create 3 hash positions that may cluster unevenly. One node might own 60% of the ring, another 10%. With virtual nodes (each physical node maps to N positions on the ring), distribution averages out. Cassandra uses 256 virtual nodes per physical node by default. DynamoDB uses a variant internally. 100–200 virtual nodes is a practical sweet spot.

Cross-Shard Queries

The fundamental limitation of sharding: JOIN across shards is impossible at the database level. Any cross-shard operation must happen at the application layer. Two approaches:

1. Scatter-Gather: Send the query to all shards in parallel, collect results, merge and sort at the application layer. Works but is expensive — N shards means N parallel queries, N network round-trips, and an in-memory merge.

import asyncio
from typing import Any

async def scatter_gather_search(
    query: str,
    shards: list,
    merge_key: str = "score",
    limit: int = 20
) -> list[dict]:
    """
    Execute query on all shards concurrently, merge and re-sort results.
    Classic scatter-gather for cross-shard queries.
    """
    async def query_shard(shard) -> list[dict]:
        return await shard.execute(query, limit=limit)

    # Fire all shard queries in parallel
    shard_results = await asyncio.gather(
        *[query_shard(shard) for shard in shards],
        return_exceptions=True
    )

    # Flatten, filter errors, sort by merge key, truncate
    merged = []
    for result in shard_results:
        if isinstance(result, Exception):
            # Log, increment error counter, continue — partial results preferred
            continue
        merged.extend(result)

    merged.sort(key=lambda x: x[merge_key], reverse=True)
    return merged[:limit]

# Problem: if limit=20 and you have 8 shards, you must fetch 20×8=160 rows
# total just to return 20. Offset pagination compounds this badly.

2. Global Secondary Indexes (GSI): Maintain a separate index shard (or a distributed index like Elasticsearch) that maps non-shard-key attributes to shard locations. The lookup shard returns which primary shards to query, then you fetch from those specific shards only — not all shards.

-- Global Secondary Index pattern
-- Primary data sharded by user_id
-- GSI sharded by email (for email-based lookup)

-- GSI shard stores: email -> (user_id, shard_id)
CREATE TABLE gsi_email_to_user (
    email       VARCHAR(320) PRIMARY KEY,
    user_id     BIGINT NOT NULL,
    shard_id    SMALLINT NOT NULL
);

-- Lookup flow:
-- 1. Hash(email) → GSI shard
-- 2. SELECT user_id, shard_id FROM gsi_email_to_user WHERE email = ?
-- 3. Use returned shard_id to query primary shard directly
-- 4. Fetch full user record: SELECT * FROM users WHERE user_id = ?

-- Two hops, but only 2 targeted queries instead of N scatter-gather queries
-- GSI must be kept consistent with primary (dual-write or CDC)

Transactions Across Shards

ACID transactions spanning multiple shards require distributed transaction protocols (2-phase commit, Saga pattern). These are orders of magnitude more complex than single-shard transactions. The practical guidance:

  • Design your data model so transactional operations stay within a single shard. If an order and its items share the same order_id shard key, they co-locate on the same shard.
  • When cross-shard transactions are unavoidable, use the Saga pattern (compensating transactions) rather than 2PC. 2PC is blocking and failure-prone at scale.
  • NewSQL databases (CockroachDB, Google Spanner, YugabyteDB) handle distributed transactions internally — consider them before hand-rolling cross-shard transaction logic.

Online Resharding with Zero Downtime

# Online resharding playbook (doubling from 4 to 8 shards)

phase_1_prepare:
  - Spin up 4 new empty shard instances
  - Configure router for dual-write: all writes go to both old and new shards
  - Enable change data capture (CDC) on all old shards

phase_2_backfill:
  - Stream CDC events to new shards continuously
  - Run parallel bulk copy of historical data (use low-watermark cursioning)
  - Monitor replication lag — must reach < 1 second before proceeding

phase_3_cutover:
  - Pause all writes for ~200ms (connection draining at load balancer)
  - Confirm CDC lag = 0 (new shards are fully caught up)
  - Atomic router config swap: point all reads to new shards
  - Resume writes — now hitting new shards only
  - Total downtime window: 200–500ms

phase_4_cleanup:
  - Run old shards in read-only mode for 48h (rollback safety net)
  - Verify no application reads still hitting old shards via access logs
  - Decommission old shards

# Tools: AWS DMS, Debezium (CDC), Vitess (MySQL), Citus (PostgreSQL)
Architect Note — When Not to Shard

Sharding adds enormous operational complexity: N deployment pipelines, N backup jobs, cross-shard query rewrites, no database-level JOINs, and resharding risk. Before sharding, exhaust: (1) read replicas — most traffic is reads, (2) caching layers — Redis eliminates many DB reads, (3) vertical scaling to the next tier, (4) data archiving — move cold data to cheaper storage. Many systems sharding at 100M rows shouldn't be sharding at all. Shard only when you've hit limits on all other approaches and the data tier is genuinely the bottleneck.

Enterprise Use Case

Enterprise Use Case MongoDB Atlas — Sharding at Forbes, EA Games, and Lyft

Context: MongoDB's native sharding architecture is one of the most widely deployed sharding implementations in production. MongoDB uses a mongos router process, config servers (metadata), and shard members (replica sets). Sharding is transparent to the application — it queries mongos as if it were a single server.

Forbes (Media — 200M documents): Forbes shards its content database by article_id hash across 6 shards. The article collection holds 200M+ documents. A single MongoDB instance would require a 24-core machine with 256 GB RAM at this scale; 6 shards each running on 8-core instances cost 40% less and write throughput scales linearly with shard count. Viral article reads hit all shards (scatter-gather on tag-based queries) — mitigated by a Redis cache layer in front of MongoDB that absorbs 92% of read traffic.

EA Games (Game State): EA uses MongoDB sharding for player state in live service games. Shard key is player_id — all player state (inventory, progress, achievements) co-locates on one shard, enabling single-shard transactions for all game state mutations. 60M active players across 16 shards. Chunk auto-balancer runs during off-peak hours. When a new game launch spikes player counts 5x, they horizontally scale by adding shards via Atlas's managed resharding.

Lyft (Real-Time Data): Lyft's ride-matching database shards on region_id + timestamp (compound shard key). Region locality means queries for "available drivers in San Francisco" hit 1–2 shards, not all. Timestamp component prevents temporal hotspots. This compound key was chosen after an initial deployment with only driver_id caused scatter-gather on all regional queries — a classic shard key mistake that required a costly remigration.

mongos Routing Architecture: The mongos process caches shard metadata from config servers. Shard key-targeted queries resolve in one network hop. Scatter-gather queries fan out in parallel; mongos merges results. Chunk migration (MongoDB's automatic rebalancing) moves 64 MB chunks between shards when one shard holds >10% more data than others — all online, no downtime.

Common Mistakes

Using an Auto-Increment Integer as the Shard Key

The most common sharding mistake. Auto-increment IDs are sequential: new rows always have the highest ID, so they always land on the last shard. All writes go to one shard — exactly what sharding was supposed to prevent.

The last shard becomes a write hotspot: CPU at 90% while the first 7 shards sit idle. The system doesn't scale horizontally — it just adds idle infrastructure.

Fix: Use a hash of the ID instead of the raw ID. Or use UUIDs (which distribute evenly). Or choose a different shard key entirely — often the natural partition boundary is a tenant ID, region, or entity relationship that aligns with your access patterns.

Not Planning for Resharding from Day One

Teams shard at 4 nodes and never think about what happens at 40 nodes. When they need to add capacity a year later, they discover that their router is hardcoded with user_id % 4 — a constant. Changing 4 to 8 remaps 75% of all keys. Their migration tooling doesn't exist, their backfill scripts haven't been written, and their cutover window is unclear.

Retrofitting resharding capability onto a live production system under capacity pressure is extremely painful. Engineers spend weeks writing one-off migration scripts, the cutover requires an extended maintenance window, and data consistency bugs surface during the migration.

Fix: Choose consistent hashing from the start — it makes adding nodes cheap. Build migration infrastructure before you need it. Document the resharding runbook. Test it in staging. Treat resharding as a first-class operation, not a future problem.

Ignoring Cross-Shard Query Performance

Engineers shard the database and discover that their most critical query — "get all orders for a customer across all time" — now requires hitting every shard. Query latency goes from 5ms to 200ms (scatter-gather across 8 shards, serial result merge).

This happens when the shard key is chosen based on "what distributes data evenly" rather than "what matches our most critical query patterns." Even distribution is important, but a system where 80% of queries are scatter-gathers has no real performance benefit from sharding — it just added operational complexity.

Fix: Before choosing a shard key, audit your top 10 queries by frequency and by business criticality. The shard key should allow at least your top 3 most critical queries to be single-shard targeted. Cross-shard queries for analytics and reporting are acceptable; cross-shard queries on every user-facing page load are not.

Interview Prep

Sharding horizontally partitions a database by splitting rows across multiple servers based on a shard key, enabling write throughput and storage to scale beyond a single machine's limits.

2-Minute Structured Answer

"Sharding is horizontal partitioning — you split a table's rows across multiple database instances based on a shard key. The key decision is choosing that shard key well: it needs high cardinality, even distribution, and alignment with your most common access patterns. There are three main strategies — range sharding (good for range queries but risks hotspots), hash sharding (even distribution but no range queries), and consistent hashing (adds nodes with minimal data migration). The main trade-offs are that cross-shard JOINs become scatter-gather operations, ACID transactions are limited to a single shard, and operational complexity multiplies with shard count. I'd reach for sharding only after exhausting read replicas, caching, and vertical scaling — because retrofitting sharding onto an existing system is expensive. A concrete example: Instagram shards user media data by user_id hash. All a user's media co-locates on one shard, so profile loads are single-shard targeted. Cross-user queries like 'trending posts' use a separate aggregation pipeline that tolerates scatter-gather latency."

Common Interview Mistakes

  • Confusing sharding with replication — replication copies all data to multiple nodes for redundancy/read scale; sharding splits data across nodes for write scale and storage scale. They are complementary, not alternatives.
  • Proposing sharding as the first solution to a slow database — interviewers expect awareness that read replicas, caching, and query optimization come first.
  • Forgetting that shard key is immutable — saying "we can change the shard key later if needed" shows a gap in understanding resharding cost.
  • Not discussing the hotspot problem — any discussion of shard key selection that doesn't mention hotspot risk is incomplete.

Trade-Offs Summary

Dimension Benefit Cost
Write Throughput Scales linearly — N shards = N× write capacity Only if writes distribute evenly; hotspots negate gains
Storage Each shard holds a fraction of total data — commodity hardware works Total storage cost increases (replication factor × N shards)
Query Performance Shard-key-targeted queries are fast — smaller index, less data scanned Cross-shard queries require scatter-gather — latency multiplies with shard count
Transactions Single-shard transactions are fully ACID — same performance as unsharded Cross-shard transactions require distributed protocols (2PC, Saga) — complex and slow
Operational Complexity Each shard is independently upgradeable and recoverable N pipelines, N backup jobs, N monitoring targets; resharding is a major operation
Schema Changes N/A — same schema change process per shard DDL must run on all N shards; rolling schema changes require careful sequencing
Development Complexity Application can often be shard-agnostic if using a router layer Cross-shard queries, pagination, aggregation, and joins must be rewritten
