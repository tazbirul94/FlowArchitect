OAuth 2.0

An authorization framework that lets applications access resources on behalf of a user — without ever seeing the user's password.

Intermediate Security Authorization Tokens

Definition & Analogy

Simple Analogy

Imagine a hotel key card. You check in at the front desk (the Authorization Server) — they verify your identity and hand you a key card (access token). That key card only opens your room and the gym, not the staff office or other guests' rooms. You never hand your house keys to the hotel. The key card expires at checkout.

OAuth 2.0 works the same way: you authorize an app to access specific resources with a scoped, time-limited token — without ever sharing your actual credentials.

Critical distinction: OAuth 2.0 is an authorization framework — it answers "what can this app access?" not "who is this user?" Authentication (proving identity) is handled by OpenID Connect (OIDC), which layers on top of OAuth 2.0. Treating OAuth as authentication is one of the most common and dangerous mistakes in the field.

OAuth 2.0 solves the credential-sharing problem. Before OAuth, if you wanted a third-party app to read your emails, you'd hand over your Gmail password. That app now has your full credentials forever. OAuth replaces this with a scoped, revocable token that expires — you stay in control.

The Story

Real-World Story
A fintech startup, BudgetPal, needed to read users' bank transaction history to generate spending reports. Their first approach: ask users to enter their online banking username and password. Users entered credentials, BudgetPal stored them, and used them to screen-scrape the bank's website nightly.
Three months in, a BudgetPal server was compromised. Attackers had plaintext banking credentials for 40,000 users. The bank's fraud team detected mass unauthorized logins at 3 AM. BudgetPal received a cease-and-desist, faced regulatory investigation under PSD2 and GDPR, and lost 80% of users. The fundamental error: BudgetPal owned credentials they should never have seen.
BudgetPal rebuilt on OAuth 2.0 using the bank's Open Banking API. Users now click "Connect Bank" — they are redirected to their bank's login page (which BudgetPal never sees). After login, the bank shows a consent screen: "BudgetPal wants to read your transactions from the last 90 days. Allow?" The user approves. The bank issues a scoped access token (read-only, 90-day transactions only, expires in 1 hour) and a refresh token. BudgetPal stores the refresh token — never the password.
After the rebuild: zero credential-storage risk. Users can revoke BudgetPal's access from their bank's dashboard at any time without changing their password. The bank's compliance team approved the integration. BudgetPal passed PCI-DSS and PSD2 audits. When an access token expires, BudgetPal silently exchanges the refresh token for a new one — users never notice. The architecture is now the industry standard.

Learn at Your Level

Beginner

Have you ever clicked "Login with Google" or "Sign in with GitHub"? That's OAuth 2.0 in action. The website you're logging into never sees your Google password. Instead, Google asks you "Do you want to let this site see your name and email?" You click yes, Google hands the site a temporary pass. That's it.

Three parties are always involved:

  • You (Resource Owner) — the person who owns the data and decides who can see it.
  • The App (Client) — the third-party app that wants to access your data.
  • The Provider (Authorization Server + Resource Server) — Google, GitHub, etc. — they hold your data and decide whether to grant access.

OAuth hands out scoped tokens — like a ticket that only gets you into certain rooms, and automatically expires.

The Basic Flow (Authorization Code)

The most common OAuth flow — used by web apps and mobile apps — works in five steps:

  1. You click "Login with GitHub" on an app.
  2. The app redirects you to GitHub's login page (with a request saying which permissions it needs).
  3. You log in to GitHub and approve the permissions.
  4. GitHub redirects you back to the app with a short-lived authorization code.
  5. The app exchanges that code (server-side, secretly) for an access token.

Authorization Code Flow Diagram

sequenceDiagram participant U as You (Browser) participant A as App (Client) participant G as GitHub (Auth Server) U->>A: Click "Login with GitHub" A->>U: Redirect to GitHub with client_id + scopes U->>G: Login + approve permissions G->>U: Redirect back to App with auth code U->>A: Deliver auth code (in URL) A->>G: Exchange code for access token (server-to-server) G->>A: Access token + refresh token A->>G: Call API with access token G->>A: Your profile data

Key Terms

  • Access Token — A short-lived credential (like a day pass). The app includes this in every API request. Typically expires in 15 minutes to 1 hour.
  • Refresh Token — A long-lived credential used to get new access tokens when they expire. Never sent to APIs — only to the auth server.
  • Scope — Defines what the token can do. read:user means read-only user profile. repo means full repository access. Always request the minimum scope needed.
  • Authorization Code — A one-time, short-lived code that the app exchanges for tokens. It travels through the browser (less secure) so it must be swapped quickly for tokens (which travel server-to-server).

OAuth 2.0 defines four grant types (flows) for different use cases. Picking the wrong flow is a common source of security bugs.

The Four OAuth 2.0 Flows

1. Authorization Code + PKCE (Most Secure — Use by Default)

When to use: Web apps with a server, single-page apps (SPAs), and mobile apps. The "PKCE" extension (Proof Key for Code Exchange) is mandatory for public clients (SPAs, mobile apps) that can't keep a secret.

How PKCE prevents code interception: Without PKCE, a malicious app could intercept the authorization code in the URL redirect (especially on mobile) and exchange it for tokens. PKCE prevents this by making the app generate a random code_verifier, hash it into a code_challenge, and send the challenge upfront. The auth server only issues tokens if the app provides the original code_verifier — which only the legitimate app knows.

# Step 1: App generates PKCE values
code_verifier = random_string(43-128 chars)  # store this
code_challenge = BASE64URL(SHA256(code_verifier))

# Step 2: Redirect user to authorization endpoint
GET /authorize?
  response_type=code
  &client_id=your_client_id
  &redirect_uri=https://yourapp.com/callback
  &scope=read:user user:email
  &state=random_csrf_token           # prevents CSRF
  &code_challenge=BASE64URL_HASH
  &code_challenge_method=S256

# Step 3: After user approves, exchange code for tokens
POST /token
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code
&code=AUTH_CODE_FROM_REDIRECT
&redirect_uri=https://yourapp.com/callback
&client_id=your_client_id
&code_verifier=ORIGINAL_VERIFIER   # PKCE verification

# Step 4: Token response
{
  "access_token": "eyJhbGciOiJSUzI1NiJ9...",
  "token_type": "Bearer",
  "expires_in": 3600,
  "refresh_token": "8xLOxBtZp8",
  "scope": "read:user user:email"
}

2. Client Credentials (Machine-to-Machine)

When to use: Service-to-service communication with no user involved. A background job, a cron task, or a microservice calling another microservice. The "client" is the service itself — it authenticates with its own credentials (client ID + secret) to get a token.

POST /token
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials
&client_id=service_account_id
&client_secret=super_secret_value
&scope=read:metrics write:events

# Response — no refresh token (machine can re-auth anytime)
{
  "access_token": "eyJhbGciOiJSUzI1NiJ9...",
  "token_type": "Bearer",
  "expires_in": 3600,
  "scope": "read:metrics write:events"
}

3. Device Authorization Flow (TVs & CLIs)

When to use: Devices with no browser or limited input (smart TVs, game consoles, IoT devices, CLI tools). The device shows a code and URL. The user visits the URL on their phone and enters the code. The device polls until the user approves.

# Device requests code
POST /device/code
client_id=tv_app_client_id
&scope=read:library

# Response shown on TV screen
{
  "device_code": "GmRhmhcxhwAzkoEqiMEg_DnyEysNkuNhszIySk9eS",
  "user_code": "WDJB-MJHT",
  "verification_uri": "https://provider.com/activate",
  "expires_in": 1800,
  "interval": 5
}

# TV displays: "Go to provider.com/activate and enter WDJB-MJHT"
# TV polls every 5 seconds:
POST /token
grant_type=urn:ietf:params:oauth:grant-type:device_code
&device_code=GmRhmhcxhwAzkoEqiMEg_DnyEysNkuNhszIySk9eS
&client_id=tv_app_client_id
# Returns 'authorization_pending' until user approves on phone

4. Implicit Flow (DEPRECATED — Do Not Use)

The implicit flow was designed for SPAs before PKCE existed. It returned the access token directly in the URL fragment (#access_token=...). This is dangerous because:

  • Tokens appear in browser history.
  • Tokens can be leaked via the Referer header when navigating to external links.
  • No way to rotate tokens.

Use Authorization Code + PKCE for SPAs instead. The implicit flow was removed from the OAuth 2.1 draft specification entirely.

Scopes — Least-Privilege Access

Scopes are space-separated strings that declare what the token can do. Common examples:

  • read:user — Read user profile (GitHub)
  • user:email — Read user's email address (GitHub)
  • repo — Full access to repositories (GitHub) — very broad, avoid unless necessary
  • openid — Required for OIDC (returns ID token)
  • profile — Access to user's name, picture, locale (OIDC)
  • email — Access to user's email (OIDC)
  • offline_access — Request a refresh token

Always request the minimum scopes required. If your app only needs to read a user's email, don't request repo or write scopes. Users see the requested scopes on the consent screen — broad scopes reduce trust and conversion.

Access Tokens vs Refresh Tokens

Property Access Token Refresh Token
Purpose Authorize API calls Get new access tokens
Lifetime Short (15 min – 1 hour) Long (days, weeks, or indefinite)
Sent to Resource APIs on every request Auth server only (token endpoint)
Storage Memory (JS variable) — safest HttpOnly cookie — safest for web
Revocable Difficult (stateless JWT) or DB lookup (opaque) Yes — revoke at auth server immediately
Rotation Expires automatically Rotate on each use (detect reuse attacks)
Token Storage Rules

Access tokens: Store in memory (a JavaScript variable). If the user refreshes the page, re-acquire silently via a hidden iframe or cookie-based session. Never store access tokens in localStorage or sessionStorage — they are readable by any JavaScript running on the page (XSS attack vector).

Refresh tokens: Store in an HttpOnly, Secure, SameSite=Strict cookie. HttpOnly means JavaScript cannot read it — only the server can. If you must store it client-side in a mobile app, use the OS secure keychain (iOS Keychain, Android Keystore).

OAuth 2.0 vs OpenID Connect (OIDC)

This distinction matters architecturally. They solve different problems and are often confused.

Dimension OAuth 2.0 OpenID Connect (OIDC)
What it answers "Can this app access my Google Drive?" "Who is this user?" + "Can this app access...?"
Protocol type Authorization framework Identity layer built on OAuth 2.0
What you get Access token (opaque or JWT) Access token + ID token (always JWT)
ID Token No Yes — contains user identity claims (sub, email, name)
UserInfo endpoint Not defined by spec Standard /userinfo endpoint
Required scope Any (e.g. read:user) Must include openid
Use for login? Not designed for it Yes — that's what it's for
Common providers GitHub (OAuth only) Google, Auth0, Okta, Microsoft Entra

In practice: if you're implementing "Login with Google" and need the user's identity, you're using OIDC (which uses OAuth 2.0 underneath). If you're a CI/CD pipeline fetching repository data from GitHub API with no user, you're using OAuth 2.0 Client Credentials.

Token Validation — JWT Structure

Access tokens are often JSON Web Tokens (JWTs). A JWT has three base64url-encoded parts separated by dots:

// Header (algorithm + token type)
{
  "alg": "RS256",
  "typ": "JWT",
  "kid": "key-id-for-rotation"
}

// Payload (claims)
{
  "iss": "https://accounts.google.com",     // issuer
  "sub": "1234567890",                       // subject (user ID)
  "aud": "your-client-id",                   // audience (must match your app)
  "exp": 1720000000,                         // expiry timestamp (Unix)
  "iat": 1719996400,                         // issued at
  "scope": "read:user user:email",           // granted scopes
  "email": "user@example.com"
}

// Signature (RS256 = RSA private key from auth server)
// You verify using the auth server's PUBLIC key from JWKS endpoint

Validating a JWT access token correctly requires checking all of these:

  1. Signature is valid (fetch public key from /.well-known/jwks.json)
  2. exp has not passed
  3. iss matches the expected issuer
  4. aud contains your client ID
  5. Scopes in the token cover what the request requires
Never Skip Audience Validation

If you don't validate the aud (audience) claim, an attacker can obtain a valid JWT token issued for a different application at the same provider, and replay it against your API. Your server would accept it because the signature is valid and it hasn't expired. Always verify aud matches your specific client ID.

PKCE Deep Dive

PKCE (RFC 7636) protects the authorization code exchange. Here is the full mechanism:

// 1. Generate cryptographically random verifier (43-128 chars)
const codeVerifier = crypto.randomBytes(32).toString('base64url');
// Example: "dBjftJeZ4CVP-mB92K27uhbUJU1p1r_wW1gFWFOEjXk"

// 2. Hash it to create the challenge
const codeChallenge = crypto
  .createHash('sha256')
  .update(codeVerifier)
  .digest('base64url');
// Example: "E9Melhoa2OwvFrEMTJguCHaoeK1t8URWbuGJSstw-cM"

// 3. Store verifier in session storage (not localStorage)
sessionStorage.setItem('pkce_verifier', codeVerifier);

// 4. Redirect to authorization endpoint with challenge
const params = new URLSearchParams({
  response_type: 'code',
  client_id: 'YOUR_CLIENT_ID',
  redirect_uri: 'https://app.com/callback',
  scope: 'openid email profile',
  state: crypto.randomUUID(),  // CSRF protection
  code_challenge: codeChallenge,
  code_challenge_method: 'S256'
});
window.location.href = `https://auth.example.com/authorize?${params}`;

// 5. On callback, exchange code using the stored verifier
const verifier = sessionStorage.getItem('pkce_verifier');
const tokenResponse = await fetch('https://auth.example.com/token', {
  method: 'POST',
  body: new URLSearchParams({
    grant_type: 'authorization_code',
    code: callbackCode,
    redirect_uri: 'https://app.com/callback',
    client_id: 'YOUR_CLIENT_ID',
    code_verifier: verifier  // Auth server hashes this and compares to challenge
  })
});
// If verifier doesn't match the challenge sent upfront, token is denied

Refresh Token Rotation

With token rotation, each time you use a refresh token to get a new access token, the auth server invalidates the old refresh token and issues a new one. This enables detection of refresh token theft:

  1. Attacker steals refresh token RT-1.
  2. Legitimate app uses RT-1 → gets AT-2 + RT-2. RT-1 is now invalidated.
  3. Attacker attempts to use RT-1 → auth server detects reuse of an invalidated token.
  4. Auth server revokes the entire token family — all active refresh tokens for that session.
  5. User is forced to re-authenticate. This limits the blast radius of theft.

Token Introspection (RFC 7662)

Resource servers (your APIs) need to validate access tokens. Two approaches:

  • Local JWT validation — Verify the signature with the auth server's public key (fetch from JWKS endpoint). Fast (no network call), but you can't revoke tokens before they expire. Use for stateless APIs that tolerate token-window exposure.
  • Token introspection — Call the auth server's /introspect endpoint on every request. Returns real-time status including whether the token has been revoked. Slow (adds ~10-50ms), but enables immediate revocation. Use for high-security operations (financial transactions, admin APIs).
# Token introspection request (RFC 7662)
POST /introspect
Authorization: Basic base64(resource_server_id:resource_server_secret)
Content-Type: application/x-www-form-urlencoded

token=eyJhbGciOiJSUzI1NiJ9...
&token_type_hint=access_token

# Active token response
{
  "active": true,
  "scope": "read:accounts write:transactions",
  "client_id": "budgetpal-prod",
  "username": "user@example.com",
  "sub": "user-uuid-1234",
  "aud": ["banking-api", "analytics-api"],
  "exp": 1720000000,
  "iat": 1719996400,
  "token_type": "Bearer"
}

# Revoked or expired token response
{ "active": false }

Dynamic Client Registration (RFC 7591)

In large enterprise environments, manually pre-registering every OAuth client is impractical. Dynamic Client Registration lets clients register themselves programmatically at runtime:

POST /register
Authorization: Bearer initial_access_token  # bootstrapped via out-of-band process
Content-Type: application/json

{
  "client_name": "Order Service v2.3",
  "client_uri": "https://order-service.internal",
  "redirect_uris": ["https://order-service.internal/oauth/callback"],
  "grant_types": ["client_credentials"],
  "token_endpoint_auth_method": "private_key_jwt",
  "jwks_uri": "https://order-service.internal/.well-known/jwks.json",
  "scope": "read:inventory write:orders"
}

# Server issues client credentials
{
  "client_id": "auto-generated-uuid",
  "client_secret": "generated-secret",  # or use private_key_jwt (no secret stored)
  "client_id_issued_at": 1720000000,
  "client_secret_expires_at": 0  # 0 = never expires
}

Multi-Tenant OAuth Architecture

Multi-tenant SaaS (GitHub, Google Workspace, Salesforce) requires isolating tokens per tenant:

# Multi-tenant token strategy options

# Option A: Single authorization server, tenant claim in token
token_claims:
  sub: "user-uuid"
  tenant_id: "acme-corp"          # custom claim
  org: "acme-corp"                # GitHub-style org claim
  permissions: ["read:data"]

# Resource server validates:
# 1. Token signature valid
# 2. Token not expired
# 3. tenant_id matches the requested resource's tenant
# 4. permissions include required scope

# Option B: Per-tenant authorization server (Okta / Auth0 organizations)
auth_servers:
  acme-corp: "https://acme-corp.auth0.com"
  beta-inc:  "https://beta-inc.auth0.com"

# Each tenant has isolated:
# - Users and groups
# - Client registrations
# - Signing keys (no cross-tenant token reuse)
# - Audit logs

# Option C: Tenant-scoped client credentials (service mesh)
# Each tenant's services get a dedicated client_id
# Token audience must include tenant identifier
# Tokens from acme-corp namespace cannot call beta-inc APIs

Compliance Considerations

  • GDPR / Data Residency: OAuth tokens must not carry PII (like email) in access tokens sent to third-party APIs — the third party can log and store the token payload. Use opaque tokens (random strings) for external APIs; only use JWT tokens with user claims for internal services.
  • PSD2 / Open Banking: Mandates OAuth 2.0 with PKCE for TPP (Third Party Provider) access to banking APIs. Refresh token lifetime capped at 90 days per regulation. Strong Customer Authentication (SCA) required for high-value operations — re-consent required.
  • HIPAA: Access tokens must expire within 8 hours for administrative access to PHI. Refresh tokens require MFA re-authentication. All token issuance events must be logged with user identifier, client identifier, requested scopes, and timestamp.
  • SOC 2 / Audit: Token issuance, refresh, revocation, and introspection events must flow to a SIEM. Privileged scope grants require human approval workflow before client can request them. Client credential rotations must be recorded.
  • OAuth 2.1 (draft): Consolidates the spec by: requiring PKCE for all authorization code flows, removing implicit flow, removing resource owner password credentials (ROPC) flow, requiring refresh token rotation, and mandating exact redirect URI matching.

Token Performance at Scale

# Production token validation architecture (100K req/sec)

# Problem: Every API request needs token validation
# Option 1: Local JWT validation (fast, no revocation)
validation_latency: ~0.1ms (pure CPU, no network)
revocation_lag: up to token expiry (worst case 1 hour)
suitable_for: read APIs, non-critical paths

# Option 2: Introspection with caching

  store: Redis
  key: SHA256(access_token)
  value: introspection_response
  ttl: 30s  # balance freshness vs performance
  hit_rate: ~95% (most requests reuse same token)
validation_latency: ~0.2ms (cache hit) / ~15ms (cache miss)
revocation_lag: 0-30s (cache TTL)
suitable_for: financial APIs, admin operations

# Option 3: Token binding to request fingerprint
# Access token includes hash of client certificate / device fingerprint
# Prevents token theft even if token is intercepted

Enterprise Use Case

Enterprise Use Case GitHub OAuth Apps — 100M+ Developers, Thousands of Integrations

Context: GitHub provides OAuth 2.0 for third-party apps to integrate with GitHub on behalf of users. Over 100 million developers have connected at least one third-party tool — CI/CD systems, IDEs, project management platforms, security scanners — via GitHub OAuth. This is the world's largest developer OAuth ecosystem.

Fine-grained scopes: GitHub OAuth defines extremely granular scopes so users can grant minimal access:

  • repo — Full repository access (read + write). Rarely needed. Most apps only need public_repo.
  • public_repo — Access to public repositories only. Sufficient for 90% of CI integrations.
  • read:user — Read user profile. Used by apps that just need your name/avatar.
  • user:email — Read primary email. Used for account creation flows.
  • workflow — Manage GitHub Actions workflows. Required by CI/CD tools that create or modify pipelines.
  • read:org — Read organization membership. Used by tools that need to know which team you're on.

GitHub Apps vs OAuth Apps: GitHub now recommends "GitHub Apps" over "OAuth Apps" for new integrations. GitHub Apps use Installation Access Tokens (server-to-server, no user involved), have per-repository permissions rather than per-user, and can operate as their own identity rather than impersonating a user. For user-facing "Login with GitHub" flows, OAuth 2.0 with PKCE is still the right tool.

At scale — what happens when you revoke access: When a user revokes a GitHub OAuth app from their Settings page, GitHub immediately invalidates the app's refresh token. The app's existing access tokens continue to work until they naturally expire (up to 8 hours). This is the revocation lag problem inherent to stateless JWTs. GitHub addresses this with a Webhook event (installation.deleted) — apps that care about immediate revocation must subscribe and remove the user's data on receipt. This is the production pattern used by enterprise GitHub integrations with compliance requirements.

Concrete numbers (public GitHub data): GitHub OAuth processes ~2 billion API requests per day across authorized third-party apps. Token issuance at peak: ~50,000 new access tokens per minute during business hours. Scope breakdown: 68% of apps request only read-only scopes; 31% request repo (often overkill — a common user complaint on consent screens).

Common Mistakes

Using Implicit Flow (Tokens in the URL)

The implicit flow delivers access tokens directly in the URL fragment: https://app.com/callback#access_token=eyJ.... This means the token appears in browser history, server access logs (if the fragment gets forwarded), and is visible to any JavaScript running on the page. The implicit flow was designed as a shortcut for SPAs before PKCE existed — that constraint no longer applies.

Fix: Use Authorization Code + PKCE for all SPAs and mobile apps. PKCE is supported by every modern OAuth provider. The implicit flow is removed in OAuth 2.1. If your library only supports implicit flow, it is outdated — replace it.

Storing Tokens in localStorage (XSS Vulnerability)

Storing access tokens or refresh tokens in localStorage or sessionStorage is an XSS vulnerability. Any JavaScript on your page — including injected scripts from a compromised CDN dependency, a browser extension, or an XSS attack via user content — can read localStorage and steal your tokens. Once stolen, an attacker can use the access token immediately and, if they have the refresh token, maintain access indefinitely.

Fix: Store access tokens in memory (a JavaScript variable that lives only during the page session). Store refresh tokens in HttpOnly, Secure, SameSite=Strict cookies — JavaScript cannot access HttpOnly cookies. Implement a silent refresh mechanism (hidden iframe or service worker) to re-acquire tokens on page reload without re-authentication.

No PKCE for Public Clients (Code Interception Attack)

On mobile apps, the OS routes the OAuth redirect URI (myapp://callback?code=XXXX) to registered apps. A malicious app can register the same custom URI scheme and intercept the authorization code before your legitimate app receives it. Without PKCE, the attacker can exchange that code for tokens — you'll never know.

Similarly, in SPAs without PKCE, a race condition in some browsers could allow a compromised browser extension to intercept the code from the URL before your JavaScript reads it.

Fix: Always use PKCE for public clients (SPAs, native mobile apps, CLI tools). The code_verifier is generated locally and never transmitted during the redirect — an intercepted code is worthless without it. PKCE has zero UX cost and should be a non-negotiable requirement in your OAuth implementation checklist.

Interview Prep

OAuth 2.0 is an authorization framework that lets applications access protected resources on behalf of a user by issuing scoped, time-limited tokens — without the application ever seeing the user's credentials.

2-Minute Structured Answer

"OAuth 2.0 solves the credential-sharing problem. Without it, you'd have to give a third-party app your actual password to let it act on your behalf — and you'd have no way to revoke access without changing your password. OAuth replaces this with scoped, expiring tokens. The core flow — Authorization Code with PKCE — redirects the user to the provider, who authenticates them and issues a short-lived authorization code. The app exchanges that code server-side for an access token and a refresh token. The access token is included in every API call; when it expires, the refresh token silently gets a new one. It's important to distinguish OAuth 2.0 from OpenID Connect: OAuth handles authorization ('what can this app do'), while OIDC layers identity on top ('who is this user'). The most common mistakes I see are using the deprecated implicit flow, storing tokens in localStorage where they're XSS-accessible, and forgetting PKCE on public clients where code interception is a real attack."

Common Interview Mistakes

  • Saying "OAuth is for authentication" — OAuth is for authorization. OIDC adds authentication. This distinction is tested in most security interviews.
  • Not mentioning PKCE when asked about SPA or mobile OAuth — PKCE is now table stakes; omitting it signals outdated knowledge.
  • Saying "store the token in localStorage — it's fine for most apps" — it is not fine; localStorage is readable by any JavaScript and is the most exploited OAuth vulnerability.
  • Conflating access tokens and refresh tokens — forgetting that refresh tokens are only sent to the auth server, never to resource APIs.
  • Forgetting the state parameter — the state value is your CSRF protection for the OAuth redirect. Interviewers who go deep will ask about it.

Trade-Offs Summary

Dimension Benefit Cost / Challenge
