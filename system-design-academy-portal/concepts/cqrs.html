CQRS

Command Query Responsibility Segregation — separate your read and write models so each can be independently optimised, scaled, and evolved.

Advanced Architecture Scalability Event Sourcing

Definition & Analogy

Simple Analogy

Think of a bank. There is one window where you deposit or withdraw money — every transaction goes through strict identity checks, auditing, and balance validation. There is a separate ATM display that just shows your balance — it reads from a cached summary and doesn't need to lock the ledger. CQRS is the same idea applied to your codebase: one path for writes (commands), a completely separate path for reads (queries).

CQRS stands for Command Query Responsibility Segregation. It is a pattern, first articulated by Greg Young in 2010 and popularised through Martin Fowler's enterprise architecture catalog, that splits an application's data model into two distinct sides:

  • Command side — handles operations that change state: PlaceOrder, UpdateInventory, CancelOrder. Commands return void or an ID. They go through validation and business rules. The underlying storage is typically normalised (3NF) to ensure consistency.
  • Query side — handles read-only operations: GetOrderDashboard, GetInventoryReport. Queries have no side effects. The underlying storage is often denormalised, pre-joined, and optimised for the exact shape each consumer needs.

Without CQRS, a single model tries to serve both purposes. Business logic for writes pollutes query paths with unnecessary complexity. Optimising for fast reads (denormalisation) conflicts with write consistency requirements. The result is a model that does neither job well. CQRS lets you optimise each side independently.

The Story

Real-World Story
A fast-growing e-commerce platform processes thousands of orders a minute. The engineering team had built a clean domain model: an Order entity with full business logic for placing, updating, and cancelling orders. The same Order model was also used to power the customer dashboard, the seller analytics screen, the warehouse picking queue, and the executive revenue report — all via the same repository methods.
As the platform scaled, query performance collapsed. The dashboard needed data from Orders, Products, Users, and Shipping — all joined across normalised tables under heavy locking. Indexes that helped writes hurt reads and vice versa. Business logic methods on the domain model were accidentally invoked from read paths, firing validation rules mid-query. The team added caching, but cache invalidation bugs caused stale dashboards. Reads were 99% of traffic but were fighting the write model for resources. P99 dashboard latency hit 8 seconds.
The team introduced CQRS. Commands (PlaceOrder, CancelOrder) continued to go through the domain model with full validation, writing to a normalised write database. Whenever a command succeeded, an event was published (OrderPlaced, OrderCancelled). Separate event consumers built purpose-built read models: a denormalised order_dashboard_view table pre-joined with customer names and product photos, a warehouse_queue_view sorted by dispatch priority, and an analytics_summary_view updated asynchronously. Each read model was a simple SELECT * from a purpose-built table.
Dashboard P99 latency dropped from 8 seconds to 60ms. The read replicas scaled independently — the team added three more read nodes without touching write infrastructure. Business logic on the write side became cleaner because it no longer carried query-related concerns. The only trade-off: the dashboard could lag by up to 500ms after an order was placed. The team added a "refreshing..." indicator for the 0.5% of users who opened the dashboard within a second of placing an order. Customers didn't notice.

Learn at Your Level

Beginner

Imagine a library. When you return a book, a librarian takes it, checks for damage, logs the return, updates records, and puts it back on the right shelf. That is the write path — it requires rules and careful handling.

When you want to find a book, you walk to the catalogue display — a simple lookup board that just shows title, location, and availability. That is the read path — it is fast, simple, and needs no rules. You don't check for damage when browsing the catalogue.

CQRS does the same thing for your software. Instead of one model that handles both reading and writing, you have two separate models. The write model enforces all the business rules. The read model is a simple, fast, pre-built view of the data optimised for what the user actually sees on screen.

The key insight: reads and writes have very different requirements. Reads are frequent and need to be fast. Writes are less frequent but need to be correct. Trying to serve both from one model is like using the librarian's processing desk as your browsing catalogue.

Key Terms

  • Command — An instruction to change something. "Place this order." "Cancel that booking." Commands have business logic. They can be rejected.
  • Query — A request to see something. "Show me my orders." "What is in stock?" Queries are read-only and always succeed (or return empty).
  • Write model — The internal representation of your data optimised for consistent updates. Usually normalised, with relationships enforced.
  • Read model — A pre-built, denormalised view of the data optimised for a specific screen or consumer. Can be a flat table, a JSON document, or a cache.

Command vs Query Path

flowchart TD Client([Client]) subgraph CommandPath["Command Path (Write)"] direction TB C1[Command\ne.g. PlaceOrder] --> C2[CommandHandler\nValidation + Business Rules] --> C3[(Write DB\nNormalised)] end subgraph QueryPath["Query Path (Read)"] direction TB Q1[Query\ne.g. GetDashboard] --> Q2[QueryHandler\nSimple lookup] --> Q3[(Read DB\nDenormalised)] end Client --> C1 Client --> Q1 C3 -- "Event / Sync" --> Q3

The two paths share no code. The write side runs business logic. The read side is a simple database read from a pre-built view. An event (or a background sync process) keeps the read model up to date after writes happen.

At the intermediate level, CQRS is about concrete implementation patterns: how commands, command handlers, query handlers, and read models are structured in code.

The Command Side

A command is a plain object representing intent. It is named in the imperative voice and carries only the data needed to perform the action. A command handler receives it, loads the relevant aggregate, applies business logic, and persists the result.

# commands.py — plain data objects, no logic
from dataclasses import dataclass
from uuid import UUID

@dataclass(frozen=True)
class PlaceOrderCommand:
    customer_id: UUID
    items: list[dict]
    payment_method: str

@dataclass(frozen=True)
class CancelOrderCommand:
    order_id: UUID
    reason: str


# command_handlers.py — one handler per command
class PlaceOrderCommandHandler:
    def __init__(self, order_repo, payment_service, event_bus):
        self._order_repo = order_repo
        self._payment_service = payment_service
        self._event_bus = event_bus

    def handle(self, cmd: PlaceOrderCommand) -> UUID:
        # 1. Load / create aggregate
        order = Order.create(
            customer_id=cmd.customer_id,
            items=cmd.items
        )
        # 2. Apply business rules (raises DomainError if invalid)
        order.validate_stock()
        order.apply_payment(cmd.payment_method, self._payment_service)
        # 3. Persist write model
        self._order_repo.save(order)
        # 4. Publish domain event for read-side consumers
        self._event_bus.publish(OrderPlacedEvent(order_id=order.id, ...))
        return order.id


class CancelOrderCommandHandler:
    def __init__(self, order_repo, event_bus):
        self._order_repo = order_repo
        self._event_bus = event_bus

    def handle(self, cmd: CancelOrderCommand) -> None:
        order = self._order_repo.get(cmd.order_id)
        order.cancel(reason=cmd.reason)  # raises if not cancellable
        self._order_repo.save(order)
        self._event_bus.publish(OrderCancelledEvent(order_id=order.id, reason=cmd.reason))

The Query Side

A query handler reads from a read model — a denormalised table or document store purpose-built for that specific view. There is no domain model involved, no business logic, no side effects.

# queries.py — plain data objects
@dataclass(frozen=True)
class GetOrderDashboardQuery:
    customer_id: UUID
    page: int = 1
    page_size: int = 20

# query_handlers.py — direct DB reads, no domain model
class GetOrderDashboardQueryHandler:
    def __init__(self, read_db):
        self._read_db = read_db

    def handle(self, query: GetOrderDashboardQuery) -> list[OrderDashboardRow]:
        # Read model is pre-joined: no complex SQL here
        rows = self._read_db.execute("""
            SELECT order_id, placed_at, status, total_amount,
                   item_count, customer_name, shipping_address
            FROM   order_dashboard_view
            WHERE  customer_id = %s
            ORDER  BY placed_at DESC
            LIMIT  %s OFFSET %s
        """, [query.customer_id, query.page_size, (query.page - 1) * query.page_size])
        return [OrderDashboardRow(**row) for row in rows]

Read Model Synchronisation

The read model must stay in sync with the write model. There are two approaches:

  • Synchronous (inline) — the command handler updates the read model in the same transaction. Simple but couples write and read infrastructure. Good for small systems.
  • Asynchronous (event-driven) — the command handler publishes an event; a separate consumer updates the read model. Slightly eventual but decoupled. Used in most production CQRS systems.
# read_model_updater.py — event consumer that keeps read model fresh
class OrderDashboardViewUpdater:
    def __init__(self, read_db):
        self._read_db = read_db

    def on_order_placed(self, event: OrderPlacedEvent) -> None:
        self._read_db.execute("""
            INSERT INTO order_dashboard_view
                (order_id, customer_id, customer_name, placed_at,
                 status, total_amount, item_count, shipping_address)
            VALUES (%s, %s, %s, NOW(), 'PLACED', %s, %s, %s)
        """, [event.order_id, event.customer_id, event.customer_name,
              event.total_amount, event.item_count, event.shipping_address])

    def on_order_cancelled(self, event: OrderCancelledEvent) -> None:
        self._read_db.execute("""
            UPDATE order_dashboard_view
            SET    status = 'CANCELLED', cancelled_at = NOW()
            WHERE  order_id = %s
        """, [event.order_id])

Multiple Read Models for Different Consumers

The power of CQRS is that you can build as many read models as you need, each optimised for its consumer:

  • Mobile app — needs a summary: order ID, status, total, item count. One row, minimal payload.
  • Warehouse picking — needs items, quantities, bin locations, sorted by pick route. Completely different shape.
  • Analytics — needs aggregates: revenue per day, cancellation rate, top products. Pre-aggregated, updated in batch.
  • Admin console — needs full detail: all fields, audit history, customer contact info.

Without CQRS, one query has to serve all four. With CQRS, each consumer gets its own read model shaped exactly for its needs.

At the advanced level, CQRS is about trade-offs, combining it with Event Sourcing, and understanding when it creates more problems than it solves.

CQRS + Event Sourcing

CQRS pairs naturally with Event Sourcing. Instead of storing the current state of an entity, you store the sequence of events that led to it. The write model becomes an append-only event log. Read models are projections — views built by replaying events.

flowchart LR CMD[Command\nPlaceOrder] --> H[Command Handler] --> ES[(Event Store\nAppend-only)] ES --> P1[Projection:\nDashboard View] --> RDB1[(Read DB:\ndashboard_view)] ES --> P2[Projection:\nWarehouse Queue] --> RDB2[(Read DB:\nwarehouse_view)] ES --> P3[Projection:\nAnalytics Summary] --> RDB3[(Read DB:\nanalytics_view)] RDB1 --> QH[Query Handlers] RDB2 --> QH RDB3 --> QH QH --> CLIENT([Client])

The key benefit: if a projection has a bug, you can fix the code and replay all events from the beginning to rebuild it. Your data is never lost — only the derived view was wrong.

# event_store.py
class EventStore:
    def append(self, aggregate_id: UUID, events: list[DomainEvent], expected_version: int) -> None:
        """Optimistic concurrency: fail if another writer updated since we loaded."""
        current_version = self._get_version(aggregate_id)
        if current_version != expected_version:
            raise ConcurrencyConflictError(aggregate_id)
        for event in events:
            self._db.execute(
                "INSERT INTO events (aggregate_id, event_type, payload, version, occurred_at) VALUES (%s, %s, %s, %s, NOW())",
                [aggregate_id, type(event).__name__, serialize(event), current_version + 1]
            )
            current_version += 1

    def load(self, aggregate_id: UUID) -> list[DomainEvent]:
        rows = self._db.execute(
            "SELECT event_type, payload FROM events WHERE aggregate_id = %s ORDER BY version",
            [aggregate_id]
        )
        return [deserialize(row['event_type'], row['payload']) for row in rows]


# order_aggregate.py — rebuilt from events every time (or cached)
class Order:
    @classmethod
    def load(cls, event_store: EventStore, order_id: UUID) -> 'Order':
        events = event_store.load(order_id)
        order = cls.__new__(cls)
        order._apply_all(events)
        return order

    def _apply_all(self, events: list[DomainEvent]) -> None:
        for event in events:
            self._apply(event)

    def _apply(self, event: DomainEvent) -> None:
        handlers = {
            OrderPlacedEvent: self._on_placed,
            OrderCancelledEvent: self._on_cancelled,
            OrderShippedEvent: self._on_shipped,
        }
        handlers[type(event)](event)

    def _on_placed(self, event: OrderPlacedEvent) -> None:
        self.id = event.order_id
        self.status = 'PLACED'
        self.items = event.items

Trade-offs Table

Dimension Benefit Cost
Read performance Reads hit denormalised views — near-instant even at scale Read model must be maintained and kept in sync
Write performance Write model is lean — no join overhead, no read-optimisation pollution Must publish events / trigger projections after every write
Scalability Read and write sides scale independently — add read replicas without touching write DB Infrastructure doubles: separate write DB, read DB(s), event bus
Multiple consumers Each consumer gets its own read model shaped for its exact needs N consumers = N read models to build, maintain, and monitor
Consistency Write side is strongly consistent Read side is eventually consistent — may lag by milliseconds to seconds
Debugging Clear separation makes write-side bugs easier to isolate Tracing a read anomaly requires looking at event flow + projections
Team complexity Write and read teams can work independently Requires strong understanding of eventual consistency across the team

When to Use CQRS

  • Complex domain with rich business rules on the write side that would pollute read queries.
  • Significant read/write scale mismatch — e.g., 99% reads, 1% writes — where scaling independently saves cost.
  • Multiple consumers needing different representations of the same data (mobile, analytics, admin).
  • You are already using or moving toward Event Sourcing.
  • Domain Driven Design with aggregates — CQRS and DDD were designed together.

When NOT to Use CQRS

  • Simple CRUD applications — if your domain is mostly create/read/update/delete with no real business rules, CQRS adds massive complexity for zero benefit.
  • Small teams — maintaining two models, an event bus, and projections requires engineering bandwidth. On a 2-person team, it is premature.
  • No clear read/write imbalance — if reads and writes are roughly equal, the effort of separate models yields nothing.
  • Strong consistency required on reads — if users must see their own writes immediately (no eventual consistency acceptable), CQRS requires extra work (read-your-own-writes patterns) that may negate the benefits.
Architect Note

CQRS is not binary. You can apply it at the service level (one service with separate command and query handlers using the same database, just separate code paths) before committing to separate databases. Start with logical separation. Add physical separation (separate DBs, event bus, projections) only when you have measured that you need it. "CQRS Lite" — same DB, separate models — solves most of the complexity problems at a fraction of the infrastructure cost.

At the expert level, CQRS is about projection design, handling projection failures, eventual consistency guarantees, and enterprise-scale patterns.

Read-Side Projections at Scale

A projection is a function from a stream of events to a read model. At enterprise scale, projections become their own engineering problem:

  • Catch-up subscriptions — a new projection must process all historical events before it can serve queries. For a system with 10 billion events, this takes hours. Plan for it.
  • Checkpointing — projections must record their position in the event stream. If the consumer crashes and restarts, it must resume from the last checkpoint, not the beginning.
  • Idempotency — the same event may be delivered more than once (at-least-once delivery). Projections must be idempotent: applying the same event twice produces the same result as applying it once.
  • Partition ordering — Kafka partitions give ordering within a partition. If events for the same aggregate can land in different partitions, projections may receive them out of order. Design aggregates to always publish to the same partition (use aggregate ID as partition key).
# projection_runner.py — production-grade projection with checkpointing
class OrderDashboardProjection:
    def __init__(self, event_stream, read_db, checkpoint_store):
        self._stream = event_stream
        self._db = read_db
        self._checkpoints = checkpoint_store

    def run(self) -> None:
        last_position = self._checkpoints.get('order_dashboard_projection') or 0
        for event, position in self._stream.subscribe_from(last_position):
            try:
                self._handle(event)
                # Checkpoint every 100 events for performance
                if position % 100 == 0:
                    self._checkpoints.set('order_dashboard_projection', position)
            except Exception as e:
                log.error("Projection failure at position %d: %s", position, e)
                # Dead-letter queue for manual inspection
                self._stream.dead_letter(event, error=str(e))
                # Continue processing — don't halt the stream

    def _handle(self, event: DomainEvent) -> None:
        # Idempotent upsert — safe to apply twice
        if isinstance(event, OrderPlacedEvent):
            self._db.execute("""
                INSERT INTO order_dashboard_view
                    (order_id, customer_id, status, total_amount, placed_at)
                VALUES (%s, %s, 'PLACED', %s, %s)
                ON CONFLICT (order_id) DO UPDATE
                    SET status = EXCLUDED.status,
                        total_amount = EXCLUDED.total_amount
            """, [event.order_id, event.customer_id, event.total_amount, event.occurred_at])

Projection Failure Recovery

The canonical recovery approach with Event Sourcing:

  1. Fix the projection code bug.
  2. Drop the corrupted read model table.
  3. Reset the checkpoint to position 0.
  4. Restart the projection — it replays all events from the beginning.
  5. Once caught up, route queries to it.

This is the single most powerful argument for Event Sourcing + CQRS over traditional CRUD. Your events are the source of truth. Read models are derived and always rebuildable.

Azure Event Store Pattern (EventGrid + Cosmos DB)

Microsoft's production CQRS pattern for Azure-native systems uses:

  • Azure Cosmos DB as the event store (append-only, change feed for subscriptions).
  • Cosmos DB Change Feed as the event stream — projections subscribe to the change feed of the events container.
  • Azure Functions as projection runners — serverless, scale-to-zero, triggered by change feed events.
  • Cosmos DB (separate container) or Azure SQL as read model storage, with each projection writing to its own container.
# azure_pattern/projection_function.py
# Azure Function triggered by Cosmos DB Change Feed
import azure.functions as func
import json

app = func.FunctionApp()

@app.cosmos_db_trigger(
    arg_name="events",
    container_name="events",
    database_name="orders_db",
    connection="COSMOS_CONNECTION_STRING"
)
def order_dashboard_projection(events: func.DocumentList) -> None:
    for event_doc in events:
        event_type = event_doc["event_type"]
        payload = event_doc["payload"]

        if event_type == "OrderPlaced":
            upsert_dashboard_view(
                order_id=payload["order_id"],
                customer_id=payload["customer_id"],
                status="PLACED",
                total_amount=payload["total_amount"],
                placed_at=event_doc["occurred_at"]
            )
        elif event_type == "OrderCancelled":
            update_dashboard_view_status(
                order_id=payload["order_id"],
                status="CANCELLED"
            )

Eventual Consistency Patterns

Managing user expectations around eventual consistency:

  • Read-your-own-writes — After a command, redirect the user to a query with a version token. The query handler waits (up to N ms) for the read model to catch up to that version before responding. If it times out, return the last-known state with a "refreshing" indicator.
  • Optimistic UI — The client immediately shows the result of a command in the UI without waiting for the read model. If the command later fails, the client rolls back. Used widely in mobile apps.
  • Polling with version — The command returns the event position. The client polls the read model with the expected version. The read model returns 202 Accepted until it has caught up, then returns the data.
Expert Architect Note

The hardest part of CQRS at enterprise scale is not the technical implementation — it is the organisational knowledge. Every engineer on the team must understand eventual consistency deeply enough to make correct decisions when building features. A single developer who treats the read model as strongly consistent will introduce subtle bugs that are extremely hard to reproduce. Invest in training and coding standards before adopting CQRS. Greg Young himself said: "CQRS is not an architecture. It's a principle. The architecture that emerges from it is up to you."

Enterprise Use Case

Enterprise Use Case Microsoft Azure DevOps — Work Item Tracking

Context: Azure DevOps manages hundreds of millions of work items (bugs, tasks, user stories) across thousands of enterprise tenants. Work items have complex domain rules: state machine transitions, field validation based on process templates, parent-child link constraints, iteration path hierarchies, and custom field definitions per organisation. Simultaneously, work items must power multiple radically different views: a Kanban board (columns = states), a backlog list (hierarchy of items), a sprint board (assigned items in current iteration), burndown charts (velocity over time), and query-based reports with arbitrary filters.

How CQRS is applied:

  1. Every state change to a work item — create, update, link, move, resolve — is processed through the command side, which enforces the full process template rules, field validation, and state machine. Commands go through ~40 validation rules before being committed. The write model is normalised with full referential integrity.
  2. When a work item changes, domain events are published: WorkItemCreated, WorkItemUpdated, WorkItemStateChanged, WorkItemLinked.
  3. Separate projections consume these events and maintain purpose-built read models: a board_view document per board (pre-bucketed by column), a backlog_view document tree (pre-computed parent-child hierarchy), an analytics_facts table for burndown and velocity queries.
  4. Each tenant's board view is a single Cosmos DB document — a GET with no joins, returning in under 10ms regardless of item count.
  5. Read replicas handle 98% of traffic. The write database handles only mutation operations.

Concrete numbers: Azure DevOps serves over 30 million work item reads per day. Average board load time: 120ms. Average work item update latency (command path): 280ms. Read model lag after a write: typically 50–200ms, maximum 2 seconds under load. The team estimates that without CQRS, serving this read volume from the write database would require 15x more database compute at 4x the cost.

Lessons from production:

  1. Projection versioning — when the board view schema changes, the team creates a new projection version (board_view_v2), rebuilds it from events in the background, then switches the query handler to it atomically. No downtime.
  2. Lag monitoring is mandatory — they alert when read model lag exceeds 5 seconds. Lag spikes are the first sign of a backed-up event consumer.
  3. Not all views need eventual consistency — the "just saved" confirmation screen queries the write database directly. Only the heavy-traffic views use CQRS read models.

Common Mistakes

Applying CQRS to Simple CRUD

The most common mistake: introducing CQRS to a service that does nothing more than create, read, update, and delete records with minimal business logic. You now have two models, an event bus, projection runners, and eventual consistency — all to serve what a single well-indexed table would handle trivially. A user settings service does not need CQRS. An order fulfilment engine with 40 domain rules might.

Rule of thumb: if you can't name the commands (imperative verbs with business meaning like PlaceOrder, ApproveRefund), you probably don't need CQRS. If your commands are named UpdateUser, SetField, or SaveRecord, your domain is CRUD, not CQRS territory.

Ignoring Projection Failure — Silent Stale Read Models

Teams set up projections, observe them working in staging, and deploy without monitoring. A projection consumer crashes silently in production. The read model stops updating. Users see stale data. No alert fires because the read DB is up and queries are fast — they are just returning old data. This continues for hours or days before someone notices that cancelled orders still appear as active.

Fix: every projection must expose a lag metric: current_event_position - projection_position. Alert when lag exceeds your SLA threshold (usually 10–30 seconds for user-facing views). Dead-letter queues must be monitored and alarmed on non-zero depth. Treat projection failure as a SEV-2 incident.

Tight Coupling Between Write and Read Models

The entire point of CQRS is that the read model is independently evolvable. A common failure mode: the read model is just a different SQL view on the same normalised write tables, requiring joins. Now you have the naming overhead of CQRS without the performance or scalability benefits. Another failure: the command handler calls the read model projection synchronously in the same transaction. Now they are not independent — you can't scale, deploy, or migrate them separately.

Fix: the read model must be physically separate — ideally a different database, at minimum a different schema. The write model publishes events; the read model consumes them. The command handler does not know the read model exists. The read model does not know the write model exists. Enforce this at code review: no shared imports between write-side and read-side packages.

Interview Prep

CQRS separates read and write models so each can be independently optimised — writes go through business-rule-enforcing command handlers against a normalised write DB; reads hit denormalised, pre-built views via query handlers, enabling independent scaling and multiple consumer representations.

2-Minute Structured Answer

"CQRS stands for Command Query Responsibility Segregation. The core idea is that reading data and writing data have fundamentally different requirements, so they should use different models. On the write side, you have command handlers that enforce business rules and update a normalised database. On the read side, you have query handlers that read from denormalised, purpose-built views — no business logic, no joins, just fast lookups. The two sides communicate via events: when a command succeeds, it publishes a domain event that updates the read models asynchronously. The main benefit is that you can scale reads and writes independently — if you have 99% reads and 1% writes, you add read replicas without touching your write infrastructure. You can also build multiple read models for different consumers — a mobile summary view, a warehouse queue view, an analytics dashboard — each shaped exactly for its consumer. The trade-off is eventual consistency: the read model lags slightly behind writes. For most systems this is 50–500ms, which users don't notice. The pattern is most valuable in complex domains with rich business logic. For simple CRUD, it's over-engineering."

Common Interview Mistakes

  • Confusing CQRS with Event Sourcing — they are complementary patterns but neither requires the other. You can have CQRS without event sourcing (synchronous read model sync) and event sourcing without CQRS (though CQRS makes event sourcing significantly more practical).
  • Not mentioning eventual consistency — any senior interviewer will ask "what's the trade-off?" and the first answer must be eventual consistency on the read side. Failing to name it signals shallow understanding.
  • Saying "CQRS is always a good idea" — interviewers expect you to know when not to use it. Naming "simple CRUD" as a counter-case shows maturity.
  • Treating commands and queries as just HTTP POST vs GET — CQRS is a domain model pattern, not an API naming convention. Commands are objects that go through handlers with business logic, not just non-idempotent HTTP requests.
