Saga Pattern

Manages long-running distributed transactions across microservices using a sequence of local transactions with compensating rollbacks — no locking, no blocking, eventual consistency.

Advanced Architecture Microservices Distributed Transactions

Definition & Analogy

Simple Analogy

Booking a holiday through a travel agent: the agent reserves your flight, then hotel, then car rental — three separate bookings with three separate companies. If the car rental falls through, the agent calls back the hotel and cancels it, then cancels the flight. Each step is independent; the "undo" chain is the saga compensation.

A saga is that coordinated sequence of bookings-and-cancellations, applied to microservices.

In a monolithic system, a single database transaction with ACID guarantees means all-or-nothing: every step commits together or all roll back automatically. In microservices, each service owns its own database. There is no shared transaction manager. Trying to span a single ACID transaction across service boundaries requires Two-Phase Commit (2PC), which introduces distributed locking, coordinator single points of failure, and poor scalability under high load.

The Saga pattern solves this by decomposing a business transaction into a sequence of local transactions. Each local transaction updates one service's database and publishes an event or message to trigger the next step. If any step fails, the saga executes compensating transactions — explicit undo operations — for all previously completed steps, restoring the system to a consistent state.

Sagas trade strong consistency for availability and partition tolerance. The system is eventually consistent: for a brief window during execution, some services may reflect a partial state. For most business workflows — orders, payments, bookings — this is acceptable and preferable to blocking locks that kill throughput.

The Story

Real-World Story
A food-delivery startup built their order flow as a monolith: place order → deduct inventory → charge payment → notify kitchen — all in one database transaction. It worked perfectly until they split into microservices. Suddenly each step touched a different service with a different database. They tried Two-Phase Commit to keep ACID guarantees. Under Black Friday load, the distributed lock coordinator became the bottleneck. At 500 concurrent orders/sec, lock timeouts cascaded. 30% of orders failed. Revenue evaporated.
The core problem: 2PC requires every participating service to hold a lock through the entire prepare + commit round-trip. With network latency between services and 8 participating microservices, lock hold times averaged 120ms. At high concurrency this created contention so severe the coordinator itself crashed, leaving thousands of orders in an undefined "prepared but not committed" limbo — requiring manual reconciliation over three days.
The team replaced the 2PC flow with an orchestration-based saga. A dedicated OrderSaga service was introduced as the state machine. It called each downstream service in sequence: Reserve Inventory → Charge Payment → Notify Kitchen. Each call was idempotent with a unique saga ID. If Charge Payment failed, the orchestrator called Release Inventory with the same saga ID (compensation). No distributed locks. Each service held a lock only for its own local transaction — milliseconds, not seconds.
Order throughput went from 500 to 4,000 concurrent orders/sec. Lock-related failures dropped to zero. Failed payments triggered clean compensation — inventory was always released. The compensation path was tested explicitly in CI with chaos injection. Reconciliation jobs that used to run weekly became unnecessary. The saga state machine also gave ops a single dashboard showing where every in-flight order was stuck, dramatically reducing incident response time.

Learn at Your Level

Beginner

Imagine you're building a Lego set that requires steps in order: attach the base, build the walls, add the roof. Each step is separate. If you finish the walls and then realize you used the wrong colour bricks, you have to dismantle the walls (compensation) before you can fix the base.

A saga is the same idea for software. When an online order involves multiple systems — stock warehouse, payment provider, delivery scheduler — you can't do them all at once with a magic "undo" button. Instead, you do them one at a time. If one fails, you send explicit cancellation messages to undo the previous steps.

The key insight: compensating transactions are just normal business operations that reverse a previous step. "Charge card" is forward. "Refund card" is the compensation. Both are ordinary API calls — no special magic.

Orchestration Saga — Happy Path & Compensation

In an orchestration saga, one central coordinator (the saga orchestrator) is in charge. It calls each service one at a time and reacts to success or failure. Here is what both paths look like:

Key Terms

  • Local transaction — A single-service database operation. Fast, no cross-service locking.
  • Compensating transaction — The explicit "undo" operation for a completed step.
  • Saga orchestrator — The central service that drives the saga sequence and handles failures.
  • Eventual consistency — The system will be consistent once all steps (or compensations) complete, but not necessarily at every moment in between.

There are two implementation styles for sagas: choreography and orchestration. They represent different trade-offs between decoupling and observability.

Choreography-Based Saga

In choreography, there is no central coordinator. Each service listens for events and publishes the next event when it completes its step. The overall workflow "emerges" from event reactions across services.

// Choreography: each service reacts to events independently

// InventoryService — listens for order.created
orderCreatedQueue.subscribe(async (event) => {
  const { orderId, items } = event;
  try {
    await db.reserveInventory(orderId, items);
    await eventBus.publish('inventory.reserved', { orderId });
  } catch (err) {
    await eventBus.publish('inventory.failed', { orderId, reason: err.message });
  }
});

// PaymentService — listens for inventory.reserved
inventoryReservedQueue.subscribe(async (event) => {
  const { orderId } = event;
  try {
    const order = await orderService.getOrder(orderId);
    await paymentGateway.charge(order.customerId, order.amount);
    await eventBus.publish('payment.charged', { orderId });
  } catch (err) {
    // Publish failure — InventoryService must listen and compensate
    await eventBus.publish('payment.failed', { orderId, reason: err.message });
  }
});

// InventoryService — also listens for payment.failed (compensation)
paymentFailedQueue.subscribe(async (event) => {
  const { orderId } = event;
  await db.releaseInventory(orderId);    // Compensating transaction
  await eventBus.publish('inventory.released', { orderId });
});

Choreography pros: No single point of failure for coordination. Services are fully decoupled — InventoryService doesn't know PaymentService exists, only its events. Easy to add new steps by adding new listeners.

Choreography cons: Tracing the overall saga state requires correlating events across many services and logs. Compensation chains are implicit — you must manually ensure every possible failure path has a corresponding compensation listener. Cyclic event dependencies can create hard-to-debug feedback loops.

Orchestration-Based Saga

In orchestration, a dedicated saga orchestrator holds the state machine and explicitly calls each service in order. The orchestrator knows the full workflow and manages all compensation.

// Orchestration: central OrderSaga manages all steps

class OrderSaga {
  async execute(orderId) {
    const saga = await SagaState.create({ orderId, state: 'PENDING' });

    // Step 1: Reserve inventory
    try {
      await inventoryService.reserveInventory({ sagaId: saga.id, orderId });
      await saga.transition('INVENTORY_RESERVED');
    } catch (err) {
      await saga.transition('FAILED');
      return; // Nothing to compensate yet
    }

    // Step 2: Charge payment
    try {
      await paymentService.chargePayment({ sagaId: saga.id, orderId });
      await saga.transition('PAYMENT_CHARGED');
    } catch (err) {
      // Compensate step 1
      await inventoryService.releaseInventory({ sagaId: saga.id, orderId });
      await saga.transition('FAILED');
      return;
    }

    // Step 3: Notify kitchen
    try {
      await notificationService.notifyKitchen({ sagaId: saga.id, orderId });
      await saga.transition('COMPLETED');
    } catch (err) {
      // Compensate steps 1 and 2
      await paymentService.refundPayment({ sagaId: saga.id, orderId });
      await inventoryService.releaseInventory({ sagaId: saga.id, orderId });
      await saga.transition('FAILED');
    }
  }
}

Idempotency is Non-Negotiable

Every step and every compensating transaction must be idempotent. The orchestrator may retry a call if it times out — the downstream service must handle receiving the same sagaId twice without double-charging or double-releasing.

// Idempotent charge — uses sagaId as idempotency key
async function chargePayment({ sagaId, orderId }) {
  // Check if we already processed this saga step
  const existing = await db.findTransaction({ sagaId, type: 'CHARGE' });
  if (existing) {
    return existing; // Already done — return same result
  }

  const charge = await paymentGateway.charge(order.amount, {
    idempotencyKey: `saga-${sagaId}-charge`,
  });

  await db.saveTransaction({ sagaId, type: 'CHARGE', chargeId: charge.id });
  return charge;
}
Design Principle

Never design a saga step without designing its compensating transaction at the same time. If you can't define a reliable compensation for a step, the step may not be suitable for a saga — consider making it the last step (no compensation needed) or using a different consistency strategy for that service boundary.

Saga State Machine States

A well-implemented orchestration saga tracks every transition explicitly. This enables monitoring, retries, and debugging.

Failure Mode Analysis

Three primary failure modes require specific handling strategies:

  • Compensating transaction fails: The compensation itself (e.g., releaseInventory) fails due to network error or service downtime. Strategy: retry with exponential backoff and dead-letter queue. Compensations must be idempotent — retrying must be safe. Alert operations if compensation is stuck after N retries.
  • Orchestrator crashes mid-saga: The saga orchestrator restarts and must resume from the last known persisted state. Strategy: persist every state transition to durable storage (database or Kafka) before making the next service call. On restart, load all in-flight sagas and replay from last known good state.
  • Partial completion with one non-compensatable step: Some operations cannot be undone (e.g., sending a confirmation email). These are called pivot transactions. Design your saga so the pivot is the last step, placed after all compensatable steps. If it fails, you don't need to undo it — the data state is already consistent.

Uber Trip Saga — Production Scale

Uber processes millions of trips per day. Each trip is a saga spanning 6+ microservices. A simplified view of the trip lifecycle saga:

flowchart TD A[trip.requested] --> B[driver.assigned] B --> C[trip.started] C --> D[trip.completed] D --> E[payment.charged] E --> F[receipt.sent] F --> G([SAGA COMPLETED]) E -->|payment fails| H[trip.cancelled] H --> I[driver.notified] I --> J[fare.voided] J --> K([SAGA FAILED — compensated]) style G fill:#166534,color:#fff style K fill:#991b1b,color:#fff

At Uber scale (millions of trips/day), key engineering decisions were:

  • Every step stored in Cadence/Temporal workflow engine: Temporal persists the full event history of every saga, enabling replay after any failure. If a worker crashes mid-trip, Temporal replays from the last checkpoint automatically.
  • Idempotency via trip ID: All downstream service calls carry the tripId as the idempotency key. Payment charged twice for the same tripId returns the first result, not a double charge.
  • Compensation retries with alerting: If fare.voided compensation fails, it retries up to 10 times with exponential backoff. If all retries fail, a PagerDuty alert fires and a reconciliation job processes it within 1 hour.
  • Saga timeout: A trip saga that stays in any non-terminal state for more than 2 hours is automatically escalated for manual review. This catches edge cases like network partitions during driver assignment.

Saga with Temporal (Code Pattern)

// Temporal workflow — durable saga with automatic replay
import { proxyActivities, sleep } from '@temporalio/workflow';
import type * as activities from './activities';

const { reserveInventory, chargePayment, notifyKitchen,
        releaseInventory, refundPayment } = proxyActivities<typeof activities>({
  startToCloseTimeout: '30 seconds',
  retry: { maximumAttempts: 3 },
});

export async function orderSagaWorkflow(orderId: string): Promise<void> {
  let inventoryReserved = false;
  let paymentCharged = false;

  try {
    await reserveInventory(orderId);
    inventoryReserved = true;

    await chargePayment(orderId);
    paymentCharged = true;

    await notifyKitchen(orderId);
    // Saga complete
  } catch (err) {
    // Compensation: undo in reverse order
    if (paymentCharged) {
      await refundPayment(orderId);   // idempotent — safe to retry
    }
    if (inventoryReserved) {
      await releaseInventory(orderId); // idempotent — safe to retry
    }
    throw err; // Mark workflow as failed
  }
}

Outbox Pattern — Guaranteed Event Publishing

A critical reliability concern: after a local transaction commits, the event publish to the message broker might fail. This leaves the saga stalled with no trigger for the next step. The Outbox Pattern solves this by writing the event to a local database table (the outbox) within the same transaction, then having a separate polling process relay it to the broker atomically.

-- Transactional outbox: event stored atomically with business data
BEGIN;

INSERT INTO inventory_reservations (order_id, items, status)
  VALUES ($1, $2, 'RESERVED');

INSERT INTO outbox_events (aggregate_id, event_type, payload, created_at)
  VALUES ($1, 'inventory.reserved', $3, NOW());

COMMIT;

Observability: What to Monitor

  • Saga completion rate: Percentage of sagas reaching COMPLETED vs FAILED state. Drop below 99% should trigger alert.
  • Saga p99 duration: End-to-end time from saga start to COMPLETED. Sudden increase signals downstream service degradation.
  • Compensation rate: Number of sagas entering compensation per minute. Spike indicates a failing service (e.g., payment gateway outage).
  • Stuck sagas: Sagas in non-terminal state older than N minutes. These require investigation — often indicate a bug in the state machine or a dead-letter queue not being processed.
  • Idempotency key collisions: If a compensation is retried, the idempotency check should fire. Track these — a high rate indicates saga replay loops.
Expert Insight

The hardest part of the saga pattern is not the forward path — it is ensuring your compensation logic is as robust as your primary logic. Test compensation paths with the same rigour as happy paths. Use chaos engineering (kill a service after step N) in staging to verify compensations actually execute correctly. Teams that skip compensation testing discover their gaps in production, usually at the worst possible moment.

Enterprise Use Case

Enterprise Use Case Amazon — Order Fulfillment Saga at 1M+ orders/day

Context: Amazon's order fulfillment pipeline involves at minimum: fraud check, inventory reservation across multiple warehouses, payment authorization, warehouse pick-pack-ship assignment, carrier integration, and buyer notification. Each step is a separate domain with its own service and database. A single order may touch 12+ microservices. The system must handle 1M+ orders per day with sub-5-second checkout confirmation.

Architecture decisions:

  1. Orchestration-based saga with a dedicated OrderOrchestrator service per business domain (checkout saga, fulfillment saga, shipping saga — chained but independent sagas).
  2. Each saga step is persisted before the call is made — if the orchestrator restarts, it replays only unconfirmed steps.
  3. Payment authorization is treated as a pivot transaction: placed last among the compensatable steps. All inventory reservations happen before payment. If payment fails, inventory compensations are triggered.
  4. The outbox pattern ensures events are never lost even if the service crashes between the DB commit and the Kafka publish.
  5. Every compensating transaction retries indefinitely with capped exponential backoff — compensations are expected to eventually succeed. If a compensation cannot succeed after 1 hour, a human-review queue is populated automatically.

Business Impact: Before sagas: 2PC-based checkout had 0.8% failure rate at peak load and required a distributed lock coordinator that became the bottleneck during Prime Day (they ran 10x normal load). After saga adoption: failure rate at peak dropped to 0.02%. Prime Day throughput handled 8x normal without degradation. Payment double-charges (caused by 2PC lock timeouts creating ambiguous state) eliminated entirely — saving an estimated $40M/year in refund processing and customer trust recovery costs.

Lessons learned:

  1. Design for compensation from day one: Retrofitting compensation logic onto existing services is far more expensive than building it in from the start. Every API that a saga calls must expose an idempotent undo endpoint from the beginning.
  2. Saga visibility is a product requirement: Customer service agents need to see exactly which step an order is stuck on. Build a saga dashboard into your ops tooling — not as an afterthought, but as a core deliverable of the saga implementation.
  3. Choreography becomes unmaintainable at 10+ steps: Amazon shifted from choreography (simpler to start) to orchestration as sagas grew in complexity. The tipping point was when tracing a failed order required correlating events across 7 services and nobody could tell at a glance what had happened.

Common Mistakes

No Compensating Transaction Defined

The most dangerous mistake: implementing the forward path of a saga step without defining its compensating transaction. If ReserveInventory has no corresponding ReleaseInventory and a later step fails, the inventory stays reserved forever. The system accumulates "ghost reservations" that silently drain available stock. Customers see items as out-of-stock when they are not. This is a data consistency nightmare that may go undetected for days.

Fix: Treat each saga step as a pair — forward and backward — designed simultaneously. Make the compensating endpoint a hard acceptance criterion before any saga step can be merged. If a step genuinely cannot be reversed (e.g., sending an SMS), document it explicitly and position it as the final step in the saga, after all compensatable operations.

Compensating Transactions Are Not Idempotent

The orchestrator may call ReleaseInventory and get a timeout — it does not know if the call succeeded. It retries. If ReleaseInventory is not idempotent, the inventory is released twice. The same problem applies to refunds: a non-idempotent RefundPayment called twice issues two refunds. This is a real-money bug that has cost companies millions in duplicate refunds and stock mismatches.

Fix: Every compensating transaction must accept an idempotency key (typically the sagaId + step name) and return the same result if called multiple times. Check the key at the start of the handler — if already processed, return the previous result immediately without re-executing the side effect.

Too Many Steps in One Saga (10+ Steps)

Engineering teams sometimes model an entire business process as a single saga. An order saga with 12 steps means up to 12 compensation steps if the last step fails. The compensation logic grows quadratically in complexity — step N must know which of the previous N-1 steps completed. With 12 steps the state machine has 2^12 possible partial completion states to handle correctly. Testing this exhaustively is practically impossible, and production failures will reveal unhandled states.

Fix: Decompose large business processes into a chain of smaller, independent sagas of 3–5 steps each. For example: checkout saga (fraud + inventory + payment), fulfilment saga (warehouse + carrier), notification saga (email + push). Each saga is independently reliable and testable. Compose them by having the successful completion of one saga trigger the start of the next.

Interview Prep

A saga is a sequence of local transactions where each step publishes an event or message to trigger the next step, and compensating transactions are used to undo completed steps if any step fails — achieving distributed consistency without distributed locking.

2-Minute Structured Answer

"The saga pattern solves the distributed transaction problem in microservices. In a monolith you have one database so you can wrap everything in an ACID transaction. In microservices each service has its own database, so a single ACID transaction is impossible without Two-Phase Commit — which blocks all participants with distributed locks and becomes a bottleneck and single point of failure at scale. A saga replaces this with a sequence of local transactions. Each step commits locally and triggers the next step via event or direct call. If any step fails, previously completed steps are reversed by compensating transactions — explicit undo operations. There are two implementations: choreography, where each service reacts to events autonomously, and orchestration, where a central saga manager drives the sequence and manages compensations. The key trade-off is that sagas are eventually consistent — there is a brief window where the system is in a partial state. For most business transactions like orders and payments this is acceptable and far preferable to locking the entire system."

Common Interview Mistakes

  • Saying "sagas give you ACID consistency" — sagas give you eventual consistency, not ACID. Interviewers will immediately probe this.
  • Confusing choreography with orchestration, or treating them as equivalent — they have very different observability and coupling trade-offs.
  • Ignoring compensation entirely and describing only the happy path — the entire value of the saga pattern is in reliable compensation. An answer without compensation is incomplete.
  • Not mentioning idempotency — it is the foundation that makes retries safe. Missing this signals unfamiliarity with distributed systems fundamentals.
  • Saying "use sagas for everything" — sagas add significant complexity. If you are not crossing service boundaries, a simple database transaction is always preferable.
