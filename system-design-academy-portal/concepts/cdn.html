CDN

A Content Delivery Network distributes copies of your static assets across hundreds of edge servers worldwide so every user gets content from the closest location possible.

Beginner Intermediate Networking Performance Caching

Definition & Analogy

Simple Analogy

Imagine your favourite book is only stocked at a single warehouse in New York. Every time someone in London, Tokyo, or Sydney wants a copy, they have to wait for it to be shipped from New York. Now imagine the publisher pre-ships copies to local bookshops in every major city. Customers in Tokyo grab it from the shop around the corner — same book, delivered in minutes instead of days.

A CDN does the same for your website's files. Your origin server is the warehouse. The CDN's edge nodes are the local bookshops.

Without a CDN, every user worldwide sends requests to your single origin server. A visitor in Singapore waiting for a JavaScript bundle hosted in Virginia adds ~200ms of round-trip latency before the browser even starts rendering. Multiply that by every asset on every page and the performance impact compounds quickly.

A CDN caches your static assets — images, JS, CSS, fonts, videos — at hundreds of Points of Presence (PoPs) distributed globally. When a user requests a file, Anycast DNS routes them to the nearest PoP. If that PoP already has the file cached (a cache HIT), it is served immediately without touching your origin. Only on a cache MISS does the request flow back to origin, and even then the CDN caches the response for all future requests from that region.

The Story

Real-World Story
A fast-growing SaaS company launched a video-heavy marketing site to coincide with a major product announcement. Their origin server — a single EC2 instance in us-east-1 — hosted all HTML, CSS, JavaScript, and product demo videos totalling ~80 MB per page load.
The announcement went viral. Within 30 minutes, 40,000 simultaneous visitors arrived — 60% from Europe and Asia-Pacific. Median page load time jumped from 1.2 seconds to 18 seconds. Product demo videos buffered endlessly. The EC2 instance's outbound bandwidth hit its cap. The AWS bill for that single day was $4,200 in data transfer costs alone. Bounce rate hit 78%.
The team spent two hours configuring Cloudflare as a CDN in front of their origin. They set Cache-Control: public, max-age=31536000, immutable on hashed asset filenames (e.g. app.7f3ab2.js), and Cache-Control: public, max-age=300 on HTML. No code changes to the app — just DNS pointed to Cloudflare and cache headers added to the web server config.
Within an hour of the CDN being live, cache HIT ratio reached 94%. Median page load time dropped to 0.9 seconds globally — faster than the original single-region performance. The origin EC2 instance dropped from 100% CPU to 8%. The equivalent traffic day the following week cost $180 in data transfer. The company never removed the CDN.

Learn at Your Level

Beginner

When you visit a website, your browser downloads the page's files — images, stylesheets, JavaScript — before anything appears on screen. Each file is a separate request to a server. If that server is on the other side of the world, each request takes hundreds of milliseconds just to travel there and back.

A CDN fixes this by storing copies of those files on servers that are physically close to your users. When someone in Berlin requests your logo image, they get it from a CDN server in Frankfurt instead of one in California. The file travels 10ms instead of 150ms.

The magic word is caching: the CDN saves a copy of each file the first time anyone requests it from that region. Every subsequent request gets the cached copy instantly — the origin server is never contacted again until the cache expires.

Key Terms

  • PoP (Point of Presence) — A CDN data centre location. Major CDNs have 200–300+ PoPs worldwide. Cloudflare has 300+, Fastly has 80+.
  • Edge node / Edge server — A server inside a PoP that stores cached copies of your files and serves them to nearby users.
  • Origin server — Your actual web server. The CDN fetches files from here when it doesn't have a cached copy yet.
  • Cache HIT — The CDN has the file cached and serves it directly. Fast and cheap.
  • Cache MISS — The CDN doesn't have the file yet; it fetches from origin, caches it, then serves it. Slightly slower on first request.
  • TTL (Time To Live) — How long the CDN keeps a cached file before it expires and must be refreshed from origin.

What a CDN Caches

  • Images (.jpg, .png, .webp, .svg)
  • JavaScript bundles (.js)
  • CSS stylesheets (.css)
  • Fonts (.woff2, .ttf)
  • Videos and media files
  • Static HTML (less common — careful with personalised content)
What a CDN does NOT cache by default

Anything requiring personalisation or authentication — your dashboard page, your shopping cart, API responses with user data. These must always come from your origin. Serving cached authenticated content to the wrong user would be a serious security breach.

Understanding CDNs at an intermediate level means understanding how cache control actually works — how you tell the CDN what to cache, for how long, and how to invalidate it when you deploy.

Cache-Control Headers — The Full Picture

The Cache-Control response header is the primary signal both browsers and CDNs use to decide caching behaviour. Key directives:

  • public — Response may be cached by CDNs (not just browsers). Required for CDN caching.
  • private — Only the user's browser may cache this. CDNs must not cache it. Use for authenticated responses.
  • max-age=N — Cache for N seconds (used by browsers).
  • s-maxage=N — Cache for N seconds specifically at CDN/shared caches. Overrides max-age for CDNs. Lets you have short browser cache but long CDN cache.
  • no-cache — Don't serve from cache without revalidating with origin first. CDN must check if the file changed.
  • no-store — Never cache. Not even by the browser. Use for sensitive data (authentication tokens, bank statements).
  • immutable — The file will never change. Browser and CDN can serve it forever. Only safe if you use content-hash filenames.
  • stale-while-revalidate=N — Serve the stale cached version while fetching fresh in the background. Eliminates wait during revalidation.

The Content-Hash Strategy

The most effective CDN caching strategy for static assets uses content hashing: the build tool appends a hash of the file's content to its filename.

# Before build
styles.css
app.js

# After build (webpack/vite generates these)
styles.a3f7bc.css
app.8d2e14.js

Because the filename changes every time the content changes, you can safely set a very long TTL:

# nginx config for hashed static assets
location ~* \.(js|css|woff2|png|jpg|webp|svg)$ {
    # Only applies to hashed assets (e.g. app.8d2e14.js)
    add_header Cache-Control "public, max-age=31536000, immutable";
    # 1 year TTL — safe because filename changes on every deploy
}

# HTML files: short TTL so users get the latest shell quickly
location ~* \.html$ {
    add_header Cache-Control "public, max-age=300, s-maxage=3600";
    # Browser caches 5 min, CDN caches 1 hour
}

Cache Invalidation Methods

When you deploy and need users to get new assets immediately:

  • Content-hash filenames (recommended) — New deploy creates app.9f1c22.js. Old file URL stays valid. New file URL is automatically a cache MISS. Zero cost, instant.
  • CDN purge API — Explicitly tell the CDN to remove a cached path. Instant but costs money at scale. Cloudflare charges for bulk purge operations; Fastly purge in real-time (<150ms globally) but at API cost.
  • Surrogate Keys / Cache Tags — Tag cached objects (e.g. tag all product page variants with product-123). Purge the tag to invalidate all variants at once. Much more granular than path-based purge.
  • TTL expiry — Do nothing and wait for the TTL. Only acceptable for assets with short TTLs. Never acceptable for critical assets after a bug fix deploy.

Vary Header — The Cache Segmentation Trap

The Vary header tells CDNs to store separate cache entries per value of a named request header. Common misuse:

# BAD: Vary: Cookie causes CDN to cache a separate copy per unique cookie value
# This effectively defeats CDN caching for authenticated users
Vary: Cookie

# BAD: Vary: User-Agent stores separate copies per browser/device string
# Thousands of User-Agent strings = terrible cache efficiency
Vary: User-Agent

# OK: Vary: Accept-Encoding is standard — separate copies for gzip vs brotli
Vary: Accept-Encoding

Code Example: Setting Cache Headers in Express (Node.js)

const express = require('express');
const path = require('path');
const app = express();

// Hashed static assets — 1 year, immutable
app.use('/static', express.static(path.join(__dirname, 'public/static'), {
  maxAge: '1y',
  immutable: true,
  setHeaders(res, filePath) {
    // Ensure CDNs can also cache these
    res.setHeader('Cache-Control', 'public, max-age=31536000, immutable');
  }
}));

// HTML entry point — short TTL so deploys propagate quickly
app.get('/', (req, res) => {
  res.setHeader('Cache-Control', 'public, max-age=0, s-maxage=3600, stale-while-revalidate=86400');
  res.sendFile(path.join(__dirname, 'public/index.html'));
});

// API responses — never cache at CDN
app.get('/api/user/:id', authenticateMiddleware, (req, res) => {
  res.setHeader('Cache-Control', 'private, no-store');
  res.json({ userId: req.params.id, data: '...' });
});
Cache Header Debugging Tip

Use curl -I https://yoursite.com/app.js to inspect response headers. Look for cf-cache-status: HIT (Cloudflare), x-cache: HIT (CloudFront), or fastly-cache-status: HIT (Fastly) to confirm CDN caching is working. If you see MISS on every request, check your Cache-Control headers and Vary header for issues.

At the advanced level, CDN becomes an architectural component — not just a cache layer but an active part of request routing, security, and dynamic content delivery.

Origin Shield

On a cache MISS, every PoP independently fetches from origin. With 300 PoPs, a popular new asset could result in 300 simultaneous origin requests — the thundering herd problem. Origin Shield solves this:

  • One designated PoP acts as a buffer ("shield") between all other PoPs and origin.
  • On a cache MISS, edge PoPs forward the request to the Shield PoP instead of directly to origin.
  • The Shield PoP has a high-capacity cache. If it already has the file, it serves the edge PoP. If not, it fetches from origin — and only one PoP hits origin at a time.
  • Dramatically reduces origin load and egress costs, especially on deploys when the entire cache is cold.

Dynamic Content at the Edge

CDNs are no longer just caches. Edge computing runtimes let you run JavaScript logic at every PoP:

  • Cloudflare Workers — V8 isolates running at Cloudflare's 300+ PoPs. <1ms cold start. Can rewrite requests, A/B test, generate HTML, call APIs. No concept of a "region" — code runs wherever the user is.
  • AWS Lambda@Edge — Node.js functions running at CloudFront PoPs. Can modify request/response headers, redirect, authentication checks. Slightly higher latency than Workers due to Lambda cold starts.
  • Fastly Compute@Edge — WebAssembly-based, supports Rust/Go/JS. Fastest edge runtime for compute-heavy tasks.

CDN Cost Model

CDN pricing has two main components:

  • Egress / Data Transfer — charged per GB served to end users. Cloudflare charges $0 (unlimited on paid plans). AWS CloudFront: ~$0.008–0.08/GB depending on region. Fastly: ~$0.12/GB base.
  • Request charges — charged per HTTP request. CloudFront: $0.0075–$0.016 per 10,000 requests. Cloudflare: unlimited on Business/Enterprise.

Cache HIT ratio is the most important cost driver. At 90% HIT ratio, only 10% of requests hit origin (reducing origin compute and egress). At 50% HIT ratio, you pay full price for half your traffic and get no performance benefit from the CDN for those requests.

HTTPS and Certificate Management

CDNs terminate TLS at the edge, meaning:

  • The CDN manages certificates for your domain — auto-renewal via Let's Encrypt or vendor CA.
  • Connection between CDN and origin can be HTTP (insecure) or HTTPS with a self-signed or origin CA cert.
  • Full end-to-end encryption ("Full Strict" in Cloudflare) requires a valid cert on origin — the most secure option.
  • CDNs handle TLS handshake overhead so your origin doesn't have to, reducing origin CPU.

At expert level, CDN strategy touches multi-region resilience, security architecture, DDoS mitigation, and cost optimisation at scale — CDN is a core reliability component, not an add-on.

Multi-CDN Strategy

Relying on a single CDN provider is itself a SPOF. Major outages are rare but consequential — Fastly's June 2021 outage took down GitHub, Reddit, Twitch, and The Guardian simultaneously for ~1 hour. Multi-CDN approaches:

  • DNS-based failover — Use primary CDN for normal traffic. Health check monitors edge response times. Automated DNS failover to secondary CDN if latency exceeds threshold or if primary returns errors. 60–120 second DNS propagation lag limits switchover speed.
  • Weighted DNS split — Permanently split traffic (90%/10%) between two CDNs. Both are always warm. Failover redirects the minority share. No cold-cache problem on failover.
  • Client-side multi-CDN — Application fetches assets from the faster CDN based on real-time performance measurement. Used by major video platforms (Netflix, Amazon Video).
  • Anycast routing with BGP — Operate your own Anycast IP space. Announce the same IP from multiple CDN PoPs. Traffic routes to nearest by BGP — no DNS involved. Extremely fast failover (<1s), but requires running your own CDN infrastructure.

CDN Security — DDoS Mitigation and WAF

CDNs are the first line of defence against volumetric attacks:

  • Volumetric DDoS absorption — Cloudflare's network capacity is 230+ Tbps. A 500 Gbps DDoS attack that would saturate a normal ISP connection is a rounding error to Cloudflare. The CDN absorbs and drops malicious traffic before it reaches origin. Origin IP is never exposed — attackers cannot bypass CDN protection by hitting origin directly (assuming origin egress firewall only allows CDN IPs).
  • Layer 7 DDoS (application layer) — Valid-looking HTTP requests that exhaust application resources. CDN WAF (Web Application Firewall) applies rate limiting, bot scoring, challenge pages (JS/CAPTCHA challenges), and ML-based anomaly detection. Cloudflare blocked 212 billion cyberattacks per day in 2023.
  • Bot management — Distinguish legitimate crawlers (Googlebot) from scrapers/credential stuffers. Challenge or block based on browser fingerprint, request patterns, IP reputation.
  • Origin IP protection — Never expose origin IP in DNS. Never mention it in email headers or error messages. Rotate origin IP if it becomes known (e.g. via old DNS records in certificate transparency logs).

Production Operations Checklist

  • Cache HIT ratio monitored — alert if drops below 80% for 10 minutes
  • Origin error rate monitored separately from CDN error rate
  • Content-hash filenames enforced in build pipeline (webpack/vite config verified)
  • Cache purge API integrated into deploy pipeline (purge HTML on deploy)
  • Origin IP not exposed in DNS — firewall only allows CDN IP ranges
  • WAF enabled with at minimum OWASP Core Rule Set
  • DDoS protection tier matched to business risk level
  • TLS "Full Strict" mode enabled — origin has valid certificate
  • Certificate expiry monitoring on origin cert (alert at 30 days)
  • Multi-CDN or CDN failover plan documented and tested
  • Egress costs reviewed monthly — alert if >20% MoM increase
  • No Vary: Cookie or Vary: User-Agent on public static assets

Enterprise Cost Model at Scale

For a site serving 100TB/month of assets at 90% CDN HIT ratio:

  • CDN egress: 100 TB × $0.02/GB (CloudFront average) = $2,000/month
  • Origin egress (10% MISSes): 10 TB × $0.09/GB (AWS inter-region) = $900/month
  • Origin compute savings: 90% fewer requests means 10× smaller origin fleet vs no CDN
  • Cloudflare Enterprise flat-rate eliminates per-GB egress charges — economical above ~50TB/month
Architecture Principle

Treat CDN configuration as infrastructure code. Cache headers, WAF rules, routing logic, and purge policies belong in version-controlled Terraform/Pulumi, not clicked through a web console. A misconfigured cache header (e.g. accidentally caching authenticated responses) is a data breach vector — it must go through code review and tested in staging before production. CDN misconfigurations have caused serious incidents at major companies including exposing private S3 bucket contents and caching payment pages.

Enterprise Use Case

Enterprise Use Case Cloudflare — Black Friday 2023 at Internet Scale

Context: Cloudflare operates the world's largest CDN by network capacity. On Black Friday / Cyber Monday 2023, the company published verified metrics from its network that quantify what CDN performance looks like at genuine internet scale.

Numbers:

  • 71 million requests/second — peak traffic handled on Black Friday, up from 55M/sec the prior year
  • 90% cache HIT ratio — only 1 in 10 requests reached origin servers behind the CDN
  • 230 Tbps — network capacity available for traffic absorption
  • 212 billion — cyberattacks blocked per day across the network
  • 300+ PoPs — cities worldwide where edge nodes served content
  • <50ms — median TTFB (Time To First Byte) for cached assets globally

Architecture behind the numbers: Cloudflare uses Anycast routing — all PoPs announce the same IP ranges via BGP. User DNS resolves to the same IP globally; the internet's routing layer steers each user to the nearest PoP automatically. No DNS-based geo-routing latency. Each PoP runs the same Cloudflare software stack including WAF, bot management, Workers runtime, and cache. The 90% HIT ratio means that for every 10 requests that arrived at 71M/sec, only 7.1M/sec hit any origin server worldwide — origin operators served 10% of peak Black Friday traffic because the CDN absorbed the rest.

Lessons for application teams:

  • Cache HIT ratio is the primary metric — if your CDN HIT ratio is below 70%, you're paying for CDN infrastructure but not getting its core benefit. Audit Vary headers and Cache-Control headers first.
  • Black Friday spikes are predictable but still hard — pre-warming CDN caches before a sale (by hitting key URLs from scripts in each region) avoids the cold-cache MISS thundering herd that can overwhelm origin in the first 2–3 minutes after traffic arrives.
  • DDoS protection is not optional at scale — 212B attacks/day means roughly 2.5M attacks per second. Origin servers exposed without CDN protection cannot absorb this; a CDN is both a performance and a security investment.

Common Mistakes

Caching Authenticated or Personalised Responses

The most dangerous CDN mistake: serving a cached response containing User A's data to User B. This happens when a response that includes personal data (name, email, cart contents) is returned with Cache-Control: public or no Cache-Control header, and the CDN caches it. The next user who requests the same URL receives User A's data.

It has caused real data breaches. Always set Cache-Control: private, no-store on any response that includes session-specific, user-specific, or payment-related data. Never rely on headers like Authorization or Cookie being present to prevent CDN caching — some CDNs ignore these signals unless private is explicitly set.

Setting no-cache on Static Assets — Getting Zero CDN Benefit

A common mistake in legacy applications: setting Cache-Control: no-cache on JavaScript and CSS files "to be safe." The no-cache directive does not mean "never cache" — it means "always revalidate before serving from cache." But the more common result is that developers or ops teams set it because they were afraid of stale assets after deploy, and the CDN either treats it as uncacheable or checks origin on every request.

The solution is not no-cache on assets — it is content-hash filenames with immutable. A file named app.abc123.js never changes, so max-age=31536000, immutable is completely safe. When you deploy, the HTML references app.def456.js — a new URL that automatically cache-MISSes at the CDN.

Not Purging the CDN on Deploy — Users Get Old JS/CSS for Hours

A team deploys a critical bug fix. The HTML index.html is updated to reference new JS. But the old JS is still cached at CDN PoPs with a 24-hour TTL. Users who receive the new HTML but load the old JS from CDN cache see a broken page — mismatched HTML shell and JS bundle. This is especially bad for SPAs where the HTML references a specific hashed JS bundle that no longer exists on origin.

Two fixes work together: (1) Use content-hash filenames so new JS gets a new URL — old URL stays valid and cached, new URL is a cold cache MISS. (2) For HTML files that don't have hashed names, integrate CDN purge into the deploy pipeline: after deploying, call the CDN purge API to invalidate /index.html and any other HTML paths. This should be automated — manual CDN purges after deploys are always forgotten at 2 AM during an incident.

Interview Prep

A CDN is a geographically distributed network of edge servers that cache and serve static content from locations close to users, reducing latency and origin load by eliminating round-trips to a single central server.

2-Minute Structured Answer

"A CDN solves two problems: latency and origin load. Without one, every user worldwide waits for round-trip time to your single origin — a user in Tokyo hitting a server in Virginia adds 200ms before the browser receives the first byte. A CDN places cached copies at edge nodes worldwide, so Tokyo users get assets from a Tokyo PoP in 20ms instead. On the origin load side, at 90% cache HIT ratio, your origin only sees 10% of traffic — the CDN absorbs the rest. The key design decision is cache strategy: for static assets like JS and CSS, you use content-hash filenames plus a 1-year immutable TTL for perfect caching with zero stale risk. For HTML, you use a short TTL so deploys propagate quickly. The main trade-off is stale content risk — you must either accept brief staleness, use short TTLs and pay with more origin requests, or build cache invalidation into your deploy pipeline."

Common Interview Mistakes

  • Saying "CDNs just cache everything" — interviewers expect you to know that authenticated/personalised content must never be cached at a shared CDN layer.
  • Not mentioning cache invalidation strategy — explaining CDN without explaining how you handle deploys and stale content shows incomplete understanding of the real operational challenge.
  • Conflating CDN with load balancer — CDNs cache and serve static content at the edge; load balancers distribute dynamic requests across origin server instances. They solve different problems (though CDNs can include some routing features).
  • Forgetting security — CDNs absorb DDoS, provide WAF, and hide origin IPs. In senior interviews, omitting the security dimension is a missed opportunity.
