Rate Limiting

Controls how many requests a client can make in a given time window — protecting systems from abuse, controlling costs, and ensuring fair resource access for all users.

Intermediate Reliability API Design Security

Definition & Analogy

Simple Analogy

Think of a nightclub with a velvet rope. The bouncer lets in 10 people per minute regardless of how many are queued outside. If you've already been let in 10 times tonight, you wait at the rope even if you're not causing trouble. The rule applies equally to everyone — the bouncer doesn't evaluate each person's intentions, just the count.

Rate limiting is a technique that restricts how frequently a client (a user, IP address, or API key) can call an endpoint within a defined time window. When the limit is exceeded, the server returns an HTTP 429 Too Many Requests response instead of processing the request.

Without rate limiting, a single bad actor — or even a well-intentioned client with a bug in a retry loop — can consume all your server capacity, exhaust your database connection pool, run up your LLM API bill, or simply crowd out other legitimate users. Rate limiting is the mechanism that makes your API fair, affordable, and resilient.

The Story

Real-World Story
A fintech startup launched a public REST API for currency conversion. The API called a paid third-party exchange rate provider at $0.001 per request. The founders expected modest usage of a few thousand calls per day during beta.
Three days after launch, a developer integrating the API had a bug: their polling loop had no sleep interval. The loop fired 200 requests per second for 18 hours before they noticed. The startup received a $12,960 bill from the exchange rate provider in a single day. The API was also completely unresponsive to other beta testers for most of that day because the server's outbound connection pool was saturated.
The team implemented rate limiting using Redis with a sliding window counter. Each API key was limited to 100 requests per minute. Any key exceeding the limit received a 429 response with a Retry-After: 43 header telling the client exactly how long to wait. The misbehaving client's integration immediately started seeing 429s and the developer was alerted.
Third-party API costs dropped to $8/day — within budget. Other users regained consistent response times. The rate limit headers also helped every integration developer understand their usage patterns. Two enterprise customers later upgraded to paid plans specifically because they needed higher rate limits — rate limiting became a revenue mechanism, not just a protective one.

Learn at Your Level

Beginner

Imagine you're at an all-you-can-eat buffet. The rule is: you can take food as many times as you want, but no more than 5 trips in any 10-minute window. This is rate limiting. The restaurant isn't saying you can never eat — it's saying you can't eat so fast that you empty the buffet before other diners get a turn.

In software, when your app calls an API (like a weather service, a payment processor, or an AI assistant), the server on the other end is doing real work for every request. If you send 10,000 requests in a second, you're consuming the same resources that all other users share. Rate limiting says: "You get 100 requests per minute. After that, slow down."

The most common response you'll see when you hit a rate limit is HTTP 429 with a message like "Too Many Requests" and a header like Retry-After: 30 telling you to wait 30 seconds before trying again.

Why APIs Rate Limit

  • Prevent abuse — Stops scrapers, credential-stuffing bots, and DDoS-style floods from overwhelming the service.
  • Control costs — APIs that call paid downstream services (LLMs, payment processors, SMS) need to cap spend per customer.
  • Ensure fairness — One heavy user shouldn't starve 1,000 light users of capacity.
  • Protect stability — Sudden traffic spikes are predictably absorbed rather than crashing servers.

Token Bucket — The Most Common Algorithm

The token bucket algorithm is the most widely used approach. Picture a bucket that holds tokens:

  • The bucket has a capacity (e.g., 10 tokens max).
  • Tokens are added at a constant refill rate (e.g., 2 tokens per second).
  • Each incoming request consumes 1 token.
  • If there are no tokens, the request is rejected with HTTP 429.
  • Burst traffic is allowed up to bucket capacity, then throttled.

At the intermediate level you need to understand the four major algorithms, when to use each, and how to implement rate limiting with Redis — the standard backend for distributed rate limiting.

The Four Algorithms Compared

1. Token Bucket — A bucket holds up to N tokens. Tokens refill at a constant rate. Each request consumes one token. Burst traffic up to bucket size is allowed. Most common in practice (GitHub API, AWS API Gateway, Stripe).

2. Leaky Bucket — Requests enter a queue (the "bucket") and are processed at a constant drain rate. Excess requests overflow and are dropped. Smooths bursty traffic into a uniform output rate. Good for protecting downstream systems from load spikes. Implemented as a FIFO queue with a fixed drain interval.

3. Fixed Window Counter — Count requests in fixed time windows (e.g., 0–60s, 60–120s). If count exceeds limit, reject. Simple and memory-efficient. Has a well-known boundary burst problem: a client can make N requests just before the window resets and N more just after — getting 2N requests in a very short interval.

4. Sliding Window Log — Stores a timestamp for each request in a sorted set. On each new request, remove timestamps older than the window, count remaining entries. Exact — no boundary burst problem. Memory-intensive because every request timestamp is stored. Best for low-volume, high-accuracy requirements.

Fixed Window Boundary Burst Problem

If limit = 100 req/min with a fixed window at :00 seconds:

  • Client sends 100 requests at 11:59:58 — all pass (window 1).
  • Window resets at 12:00:00.
  • Client sends 100 requests at 12:00:02 — all pass (window 2).
  • Result: 200 requests in 4 seconds — 50x the intended rate.

Redis Implementation — Fixed Window Counter

import redis
import time

r = redis.Redis(host='localhost', port=6379, decode_responses=True)

def is_rate_limited(user_id: str, limit: int = 100, window_seconds: int = 60) -> bool:
    """
    Fixed-window counter using Redis INCR + EXPIRE.
    Returns True if the request should be rate-limited (rejected).
    """
    # Key format: rate::
    current_window = int(time.time()) // window_seconds
    key = f"rate:{user_id}:{current_window}"

    # Increment the counter atomically
    count = r.incr(key)

    # On first increment, set expiry so keys auto-clean
    if count == 1:
        r.expire(key, window_seconds * 2)  # 2x window for safety

    return count > limit


def get_rate_limit_headers(user_id: str, limit: int = 100, window_seconds: int = 60) -> dict:
    """Returns headers to include in every API response."""
    current_window = int(time.time()) // window_seconds
    key = f"rate:{user_id}:{current_window}"
    count = int(r.get(key) or 0)
    reset_time = (current_window + 1) * window_seconds

    return {
        "X-RateLimit-Limit": str(limit),
        "X-RateLimit-Remaining": str(max(0, limit - count)),
        "X-RateLimit-Reset": str(reset_time),
    }


# Flask middleware example
from flask import Flask, request, jsonify, g

app = Flask(__name__)

@app.before_request
def check_rate_limit():
    user_id = request.headers.get("X-API-Key", request.remote_addr)
    if is_rate_limited(user_id):
        headers = get_rate_limit_headers(user_id)
        headers["Retry-After"] = "60"
        return jsonify({"error": "Too Many Requests"}), 429, headers

Redis Implementation — Sliding Window with Sorted Sets

import redis
import time

r = redis.Redis(host='localhost', port=6379, decode_responses=True)

def is_rate_limited_sliding(user_id: str, limit: int = 100, window_seconds: int = 60) -> bool:
    """
    Sliding window log using a Redis sorted set.
    Score = timestamp, Member = unique request ID.
    Exact accuracy — no boundary burst problem.
    """
    now = time.time()
    window_start = now - window_seconds
    key = f"ratelimit:sliding:{user_id}"

    pipe = r.pipeline()
    # Remove timestamps outside the window
    pipe.zremrangebyscore(key, '-inf', window_start)
    # Count remaining requests in window
    pipe.zcard(key)
    # Add current request with timestamp as score
    pipe.zadd(key, {str(now): now})
    # Expire the key after the window (cleanup)
    pipe.expire(key, window_seconds + 1)
    results = pipe.execute()

    request_count = results[1]  # Count before adding current request
    return request_count >= limit

Rate Limit Keys — What to Limit On

  • Per API key (authenticated) — Most common for developer APIs. Granular, traceable, supports tiered plans. rate:{api_key}:{window}
  • Per IP address (unauthenticated) — Only option for public endpoints before login. Vulnerable to IP spoofing and shared NAT (office building with one IP). rate:{ip}:{window}
  • Per user ID (post-auth) — Best for consumer apps after login. Survives IP changes. rate:{user_id}:{window}
  • Per endpoint — Different limits per route. /search might allow 10 req/min while /profile allows 1000 req/min. rate:{user_id}:{endpoint}:{window}
  • Composite — Combine multiple dimensions: per-user + per-endpoint + global. GitHub uses this approach.

At advanced level, the key concerns are: accuracy vs. performance trade-offs across algorithms, atomic operations to prevent race conditions in distributed environments, and the subtle failure modes that appear at scale.

Sliding Window Counter — The Practical Hybrid

Approximates a sliding window using only two fixed window counters — the current window and the previous window — weighted by how far the current request falls into the current window:

import redis
import time

r = redis.Redis(host='localhost', port=6379, decode_responses=True)

def is_rate_limited_sliding_counter(
    user_id: str, limit: int = 100, window_seconds: int = 60
) -> tuple[bool, dict]:
    """
    Sliding window counter — approximates sliding window using two
    fixed counters. Memory-efficient, ~0.003% error rate in practice.
    """
    now = time.time()
    current_window = int(now) // window_seconds
    prev_window = current_window - 1

    # Position within the current window (0.0 to 1.0)
    window_fraction = (now % window_seconds) / window_seconds

    current_key = f"rate:{user_id}:{current_window}"
    prev_key = f"rate:{user_id}:{prev_window}"

    pipe = r.pipeline()
    pipe.get(current_key)
    pipe.get(prev_key)
    pipe.incr(current_key)
    pipe.expire(current_key, window_seconds * 2)
    results = pipe.execute()

    current_count = int(results[0] or 0)
    prev_count = int(results[1] or 0)

    # Weighted estimate: prev window contributes less as current window progresses
    estimated_count = (prev_count * (1 - window_fraction)) + current_count

    reset_at = (current_window + 1) * window_seconds
    headers = {
        "X-RateLimit-Limit": str(limit),
        "X-RateLimit-Remaining": str(max(0, limit - int(estimated_count))),
        "X-RateLimit-Reset": str(int(reset_at)),
    }

    return estimated_count > limit, headers

Atomic Operations — Preventing Race Conditions

In a distributed system, the check-then-increment pattern has a race condition: two requests can both read "count = 99" and both pass, exceeding the limit. Use Redis Lua scripts for atomic check-and-increment:

-- Rate limit check + increment as atomic Lua script
-- KEYS[1] = the rate limit key
-- ARGV[1] = limit
-- ARGV[2] = window TTL in seconds
-- Returns: {current_count, is_limited}

local key = KEYS[1]
local limit = tonumber(ARGV[1])
local ttl = tonumber(ARGV[2])

local count = redis.call('INCR', key)

if count == 1 then
  redis.call('EXPIRE', key, ttl)
end

local is_limited = count > limit and 1 or 0
return {count, is_limited}

import redis

r = redis.Redis(host='localhost', port=6379, decode_responses=True)

RATE_LIMIT_SCRIPT = """
local key = KEYS[1]
local limit = tonumber(ARGV[1])
local ttl = tonumber(ARGV[2])
local count = redis.call('INCR', key)
if count == 1 then
  redis.call('EXPIRE', key, ttl)
end
local is_limited = count > limit and 1 or 0
return {count, is_limited}
"""

rate_limit_sha = r.script_load(RATE_LIMIT_SCRIPT)

def check_rate_limit_atomic(user_id: str, limit: int = 100, window: int = 60) -> bool:
    import time
    current_window = int(time.time()) // window
    key = f"rate:{user_id}:{current_window}"
    count, is_limited = r.evalsha(rate_limit_sha, 1, key, limit, window * 2)
    return bool(is_limited)

Distributed Rate Limiting Architecture

Per-instance rate limiting is a common mistake at scale. If you have 10 API server instances each enforcing a 100 req/min limit, you effectively allow 1,000 req/min per user. The fix: centralize rate limit state in Redis (or a Redis Cluster for HA):

flowchart LR C1[Client A] --> LB[Load Balancer] C2[Client B] --> LB LB --> S1[API Server 1] LB --> S2[API Server 2] LB --> S3[API Server 3] S1 & S2 & S3 --> RC[(Redis Cluster\nShared Rate Limit\nCounters)] RC -->|"429: limit exceeded"| S1 RC -->|"429: limit exceeded"| S2 RC -->|"429: limit exceeded"| S3
Performance Note

Redis INCR is O(1) and executes in <0.1ms. Adding rate limiting via Redis to every API request adds ~1ms of latency in a well-configured deployment (Redis co-located in same AZ). This is acceptable overhead. If Redis is unavailable, decide your fallback policy: fail-open (allow all requests, risk abuse) or fail-closed (reject all requests, risk availability). Most APIs choose fail-open with alerting.

At expert scale, rate limiting becomes part of a multi-layered defense strategy and a product differentiator. The concerns shift to multi-region consistency, adaptive limits, and distinguishing legitimate high-volume users from abusive traffic.

Rate Limiting vs. DDoS Protection — Critical Distinction

Rate limiting and DDoS protection are complementary but distinct:

  • Rate limiting — Per-client enforcement at the application layer. Effective against: API abuse, credential stuffing, scraping, runaway retry loops. Operates at L7. Response: HTTP 429.
  • DDoS protection — Volume-based traffic filtering at the network/CDN edge. Effective against: volumetric floods (100Gbps UDP floods), SYN floods, amplification attacks. Operates at L3/L4. Response: packet drop or BGP blackholing. Tools: AWS Shield Advanced, Cloudflare, Akamai.
  • The gap — Application-layer DDoS (L7 floods like HTTP GET floods from botnets) fall in both categories. Cloudflare's rate limiting operates at the edge to handle this. Application-side rate limiting alone cannot protect against a 100K IP botnet each sending 1 req/min (all within per-IP limits).

Multi-Region Rate Limiting

Globally distributed systems face a fundamental tension: strong consistency for rate limits requires cross-region coordination (adds 50–150ms latency to every request) versus eventual consistency (allows small overcounts during partition windows).

Most teams choose eventual consistency with bounded error:

  • Each region tracks local counts in a regional Redis cluster.
  • Counts are periodically synchronized (every 1–5 seconds) to a global aggregator.
  • Local limits are set to global_limit / num_regions * 1.2 (20% headroom for imbalanced traffic).
  • A user could briefly exceed the global limit during the sync window — acceptable for most rate limiting use cases.
  • For strict enforcement (financial APIs), use global coordination via a strongly-consistent store like Spanner or a custom two-phase check protocol.

Adaptive Rate Limiting

Static limits are a blunt instrument. Production systems at scale use adaptive limits:

  • Server-side load-based throttling — When P95 latency exceeds SLO threshold, dynamically lower rate limits across all clients to shed load. Netflix's Concurrency Limiter adjusts based on round-trip time.
  • Client reputation scoring — High-trust clients (verified enterprise accounts, known good bots like Googlebot) get elevated limits. New, unverified clients get conservative limits that grow with demonstrated good behavior.
  • Request cost weighting — Not all requests are equal. GraphQL APIs assign "points" per query based on complexity. GitHub's GraphQL API uses a point system — a simple query costs 1 point, a query touching 100 nodes costs 100 points. Total budget: 5,000 points/hour.

Rate Limit Tiers as a Business Model

Rate limits directly drive revenue for API-first businesses:

  • Free tier — 100 req/min. Good for exploration, personal projects. No SLA.
  • Starter ($99/mo) — 1,000 req/min. Startup use cases. 99.9% SLA.
  • Professional ($499/mo) — 10,000 req/min. Growing businesses. Priority queue.
  • Enterprise (custom) — Unlimited or negotiated. Dedicated infrastructure, committed spend.

Twilio, Stripe, OpenAI, and Cloudflare all use this model. The rate limit is not just a technical constraint — it's a product tier feature.

Production Rate Limiting Checklist

  • All public API endpoints have rate limits configured
  • Rate limits enforced centrally (Redis cluster), not per-instance
  • 429 responses always include Retry-After and X-RateLimit-* headers
  • Redis unavailability has a defined fallback policy (fail-open or fail-closed)
  • Different limits per endpoint sensitivity (/search stricter than /status)
  • Internal services are rate-limited separately from external clients
  • Rate limit events are logged and dashboarded (top offenders visible)
  • Alerting when any single key consumes > 20% of global capacity
  • Lua scripts used for atomic check-and-increment (no race conditions)
  • Documentation published for developer-facing limits with examples

Enterprise Use Case

Enterprise Use Case OpenAI API — Multi-Dimensional Rate Limiting at Massive Scale

Context: OpenAI's API serves millions of developers and enterprise customers. LLM inference is computationally expensive — a single GPT-4 request can cost $0.03–$0.12 in GPU time. Without rate limiting, a single misconfigured client could run up tens of thousands of dollars in costs in minutes and crowd out other users from GPU capacity.

How it works:

  1. Rate limits are enforced on two dimensions simultaneously: RPM (requests per minute) and TPM (tokens per minute).
  2. For GPT-4 on the free tier: 3 RPM and 40,000 TPM. For Tier 5 (enterprise): 10,000 RPM and 800,000 TPM.
  3. The TPM limit is more meaningful than RPM for LLMs — a single request that generates a 4,000-token response consumes more GPU than 20 small requests.
  4. Headers returned on every response: x-ratelimit-limit-requests, x-ratelimit-limit-tokens, x-ratelimit-remaining-requests, x-ratelimit-remaining-tokens, x-ratelimit-reset-requests, x-ratelimit-reset-tokens.
  5. The official client libraries implement exponential backoff with jitter automatically on 429 responses — preventing client thundering herds on rate limit reset.
  6. Tier upgrades are automatic based on cumulative spend — as customers demonstrate responsible usage and pay more, their limits increase without manual approval.

Architectural decision: Token-bucket algorithm per API key. The bucket refills based on token count consumed, not request count alone. This prevents a client from burning through the token budget in one huge batch request and then being frozen, while still smoothing out traffic fairly.

Lesson for architects: Rate limiting dimensions should match the actual resource being protected. For a database API, limit by query complexity. For a video processing API, limit by minutes of video per hour. For an LLM API, limit by tokens. Generic "requests per minute" is often the wrong metric.

Common Mistakes

Per-Instance Rate Limiting (The Most Dangerous Mistake)

Each application server instance enforces its own local rate limit in memory. If you have 10 instances with a limit of 100 req/min each, you've actually allowed 1,000 req/min per user — 10x your intended limit. This mistake is invisible during development (single instance) and only reveals itself in production under load.

Fix: Always store rate limit counters in a centralized shared store (Redis, Memcached) accessible to all instances. The rate limit state must live outside any individual server.

No Retry-After Header (Causing Thundering Herds)

When clients receive a 429 without a Retry-After header, well-behaved clients retry immediately — often thousands of times per second — creating a feedback loop that makes the problem worse. This turns a rate limit event into a partial DDoS from your own clients. Every client SDK that retries on 429 needs to know when to retry.

Fix: Always include Retry-After: N (seconds to wait) and X-RateLimit-Reset (Unix timestamp of window reset) on every 429 response. Add jitter recommendations in your API documentation to prevent synchronized retry storms.

Not Rate Limiting Internal Services

Teams implement rate limiting only on the public-facing API gateway but leave internal service-to-service calls unrestricted. A runaway background job or a cascading retry storm between internal services can exhaust shared resources (database connections, message queue consumers) just as effectively as external abuse. The payment-service calling the notification-service 50,000 times/second during an incident is an internal rate limiting failure.

Fix: Apply rate limiting at every service boundary, not just the public edge. Use service mesh policies (Istio, Linkerd) or implement per-caller limits in each service's ingress middleware. Internal callers should be identified by service account or mTLS identity, not IP.

Interview Prep

Rate limiting restricts how many requests a client can make in a time window, protecting systems from abuse, controlling costs, and ensuring fair access — enforced via algorithms like token bucket or sliding window, with HTTP 429 responses and Retry-After headers when limits are exceeded.

2-Minute Structured Answer

"Rate limiting controls how frequently a client can call an API. There are three main reasons to rate limit: prevent abuse like scraping or credential stuffing, control costs when downstream calls are expensive, and ensure fairness so one heavy user doesn't starve others. The most common algorithm is the token bucket — clients get a bucket of tokens that refill at a constant rate. Each request consumes one token; when empty, you get HTTP 429 with a Retry-After header. The critical implementation detail is that rate limits must be centralized — storing counts per-instance means each of your 20 servers allows the full limit, so users get 20x the intended quota. Redis is the standard backend because INCR is atomic and O(1). In distributed systems, you also want Lua scripts to make the check-and-increment operation atomic, preventing the race condition where two requests both see 'count=99' and both pass."
