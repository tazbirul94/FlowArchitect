Retry & Idempotency

Make distributed systems resilient to transient failures — and ensure that retrying an operation never causes unintended side effects like double charges.

Intermediate Reliability Distributed Systems Resilience

Definition & Analogy

Simple Analogy

Retry is like pressing the elevator button again after it doesn't light up — you assume a brief hiccup and try once more. Idempotency is the guarantee that pressing that button ten times only calls the elevator once — you can't summon two elevators by pressing repeatedly.

Together they answer the question every distributed system must answer: "When my request fails and I don't know why, is it safe to try again?"

In a distributed system, things fail transiently all the time — a network packet drops, a server restarts, a database runs a short garbage collection pause. Most of these failures resolve in milliseconds to seconds. Retrying the request is the right instinct. The problem is that some operations are not safe to retry without extra safeguards: charging a credit card twice, creating a duplicate order, or sending a notification email twice all cause real user harm. Idempotency is the design pattern that makes retrying safe by ensuring the system produces the same result no matter how many times the same logical operation is submitted.

The Story

Real-World Story
A fintech startup built a mobile payments app. During checkout, the app sent a POST /charge request to its payment-service. The mobile client had a 5-second timeout. On slow connections, the charge sometimes succeeded server-side but the 200 OK response never reached the app before the timeout fired.
The app treated any timeout as a failure and automatically retried the charge. On flaky 3G connections, this happened dozens of times a day. Customers started reporting double and triple charges. Support tickets exploded. One user was charged seven times for a single $12 coffee order. The startup issued $47,000 in refunds in a single month and nearly lost their payment processor relationship.
The team added two changes. First, the mobile app generates a UUID before each payment attempt and sends it as an Idempotency-Key header. Second, the payment-service stores each idempotency key with its result for 24 hours — if the same key arrives again, the service returns the cached original response instead of processing a new charge. The key is generated once per user checkout session, so any number of network retries map to exactly one payment.
Double charges dropped to zero. The team also added exponential backoff with jitter to the retry logic — instead of all devices hammering the server simultaneously after a regional blip, each client waited a randomised interval before retrying. Server load spikes from retry storms fell by 80%. Customer support tickets related to payments dropped by 94%.

Learn at Your Level

Beginner

Imagine you're sending a text message and your phone shows "Failed to send." You tap retry. That's a retry — try the same thing again because something went wrong temporarily.

Now imagine you accidentally send the same message three times. The recipient gets three identical texts. That's the problem with retries when the operation isn't idempotent. An idempotent operation is one where doing it multiple times is the same as doing it once. Pressing "like" on a post is idempotent — no matter how many times you press it, you can only like it once. Charging a credit card is NOT idempotent by default — each call charges you again.

Systems solve this by attaching a unique request ID to each attempt. The server sees: "I already processed request abc-123, here's the result" — and returns the original result instead of doing the work again.

Key Terms

  • Transient failure — A temporary error that goes away on its own (network hiccup, server restarting). Worth retrying.
  • Permanent failure — An error that won't fix itself (invalid card number, item doesn't exist). Do NOT retry — it will keep failing.
  • Idempotent — An operation where calling it multiple times has the same effect as calling it once.
  • Idempotency key — A unique ID the client generates and sends with its request so the server can detect duplicates.

Which HTTP methods are safe to retry?

  • GET — Always safe to retry. Just fetches data, doesn't change anything.
  • DELETE — Usually safe to retry. Deleting something that's already deleted still results in "it's gone."
  • PUT — Safe to retry. Sets a value — setting it to the same value twice is fine.
  • POST — NOT safe to retry by default. Creates something new each time. Needs an idempotency key to be safe.

Exponential Backoff

Retrying immediately after a failure is usually the worst thing you can do — the server is probably still struggling. Exponential backoff means each retry waits twice as long as the previous one:

wait_time = base * 2^attempt

Attempt 0 (first retry): 1 * 2^0 = 1s
Attempt 1:               1 * 2^1 = 2s
Attempt 2:               1 * 2^2 = 4s
Attempt 3:               1 * 2^3 = 8s
Attempt 4:               1 * 2^4 = 16s  ← cap here (max_wait)

Jitter — Why You Must Add Randomness

Without jitter, every client that received the same 503 at the same moment will all retry at exactly the same time. If 1,000 clients retry after exactly 2 seconds, you've recreated the same traffic spike that caused the failure. Full jitter randomises the wait within the backoff window:

Full jitter:
  wait_time = random(0, min(cap, base * 2^attempt))

Example for attempt 2 (cap = 30s, base = 1s):
  max_wait = min(30, 1 * 4) = 4s
  wait_time = random(0, 4)  → could be 0.7s, 2.3s, 3.9s ...

Result: 1,000 clients spread their retries randomly across 4s
        instead of all hitting at t+4s simultaneously.

Implementing Retry with Exponential Backoff + Jitter

import time
import random
import requests

def retry_with_backoff(url, payload, max_attempts=5, base_delay=1.0, max_delay=30.0):
    """
    Retry a POST request with full-jitter exponential backoff.
    Only retries on transient errors (5xx, connection errors).
    Does NOT retry on 4xx — those are permanent client errors.
    """
    for attempt in range(max_attempts):
        try:
            response = requests.post(url, json=payload, timeout=5)

            # Success
            if response.status_code == 200:
                return response

            # Permanent failure — do not retry
            if 400 <= response.status_code < 500:
                raise ValueError(f"Client error {response.status_code} — not retrying")

            # Transient failure — fall through to backoff
            print(f"Attempt {attempt + 1} failed with {response.status_code}")

        except requests.exceptions.ConnectionError as e:
            print(f"Attempt {attempt + 1} connection error: {e}")

        # Last attempt — give up
        if attempt == max_attempts - 1:
            raise RuntimeError(f"All {max_attempts} attempts failed")

        # Exponential backoff with full jitter
        cap = min(max_delay, base_delay * (2 ** attempt))
        sleep_time = random.uniform(0, cap)
        print(f"Waiting {sleep_time:.2f}s before attempt {attempt + 2}...")
        time.sleep(sleep_time)

    raise RuntimeError("Retry loop exited unexpectedly")

Idempotency Keys

The client generates a UUID once before the first attempt, and sends it with every retry via a header. The server uses this key to detect duplicate requests:

import uuid

def charge_customer(user_id: str, amount_cents: int) -> dict:
    """
    Charge a customer. Safe to retry — idempotency key ensures
    at most one charge regardless of how many times this is called.
    """
    # Generate ONCE before the retry loop — same key for all attempts
    idempotency_key = str(uuid.uuid4())

    payload = {"user_id": user_id, "amount_cents": amount_cents}
    headers = {"Idempotency-Key": idempotency_key}

    for attempt in range(5):
        try:
            response = requests.post(
                "https://api.payment.example/charge",
                json=payload,
                headers=headers,   # same key on every retry
                timeout=5
            )
            if response.status_code in (200, 201):
                return response.json()
            if 400 <= response.status_code < 500:
                raise ValueError(f"Permanent error: {response.status_code}")
        except requests.exceptions.Timeout:
            pass  # retry
        # ... backoff logic

Server-Side Idempotency Key Handling

from flask import Flask, request, jsonify
import redis
import json

app = Flask(__name__)
cache = redis.Redis()
IDEMPOTENCY_KEY_TTL = 86400  # 24 hours

@app.route('/charge', methods=['POST'])
def charge():
    idem_key = request.headers.get('Idempotency-Key')

    if idem_key:
        # Check cache first
        cached = cache.get(f"idem:{idem_key}")
        if cached:
            # Duplicate request — return original response, don't charge again
            return jsonify(json.loads(cached)), 200

    # Process the charge (new request or no idempotency key)
    result = process_payment(request.json)

    if idem_key:
        # Store result for 24h — future retries get this cached response
        cache.setex(f"idem:{idem_key}", IDEMPOTENCY_KEY_TTL, json.dumps(result))

    return jsonify(result), 201

HTTP Method Idempotency Reference

Method Idempotent? Safe (read-only)? Notes
GET Yes Yes Always safe to retry
PUT Yes No Sets resource to value — same result if repeated
DELETE Yes No Second call may return 404, but end state is same
HEAD Yes Yes Like GET but no body
POST No (by default) No Creates new resource each call — needs idempotency key
PATCH No (by default) No Depends on operation — append is not idempotent, set is

Retry Storms — The Silent Killer

A retry storm occurs when a downstream service degrades and all its callers simultaneously start retrying at their configured max rate. The math is brutal:

Normal operation:
  100 upstream services × 10 req/s each = 1,000 req/s to database

Database slows (high CPU, GC pause):
  All 100 services detect timeouts and retry 3× each
  100 services × 10 req/s × 3 retries = 3,000 req/s to already-struggling DB

Result: 3× amplification turns a recoverable slowdown into a full outage.
        The database never recovers because retries keep it at 300% load.

Retry Budgets

A retry budget caps the total fraction of requests that are retries, not just per-client but system-wide. Popularised by Google's SRE practices:

Retry budget rule: retries must not exceed 10% of total requests

If a service receives 10,000 req/s:
  Max retry traffic = 1,000 req/s
  If retries exceed this, start rejecting retry attempts at source

Implementation approaches:
  1. Token bucket: each service gets N retry tokens/second, refills constantly
  2. Server-side: server tracks retry header (Retry-Attempt: 2) and
     actively rejects excess retries with 429 Too Many Requests
  3. Client-side: exponential backoff + max_attempts cap
     (e.g., 3 attempts total, never more)

Trade-Off Summary

Approach Benefit Risk When to Use
Immediate retry (no backoff) Fastest recovery for very brief glitches Thundering herd — amplifies load on degraded service Almost never. Only if 1 retry max and glitch < 50ms
Fixed interval retry Simple to implement and reason about Synchronised retries → spikes. No adaptation to recovery time Low-concurrency internal tools only
Exponential backoff Load decreases over time, gives service room to recover All clients on same schedule still synchronise if started at same time All production retry scenarios
Backoff + full jitter Eliminates thundering herd, even distribution of retries over time Max wait is unpredictable (could be 0s or cap) Production default — use this
Backoff + decorrelated jitter Even better temporal spread than full jitter in some workloads More complex algorithm, harder to explain to teammates High-concurrency write workloads with many clients
Idempotency key (no retry logic) Client can retry freely without server-side duplication risk Requires server-side storage of keys + TTL management Any non-idempotent operation (payments, order creation)

What NOT to Retry

Never retry 4xx errors

HTTP 400 (Bad Request), 401 (Unauthorized), 403 (Forbidden), 404 (Not Found), 422 (Unprocessable Entity) — these will not resolve with retrying. They represent permanent failures about the request itself. Retrying a 400 wastes resources and masks a real bug in the client. Retrying a 403 could trigger rate limiting or account suspension.

Only retry: 429 (Too Many Requests — after honouring Retry-After header), 500, 502, 503, 504, and connection/timeout errors.

Non-Idempotent Operations That Look Idempotent

Architect Note

Relative updates are not idempotent. UPDATE account SET balance = balance - 10 is NOT idempotent — running it twice subtracts 20. UPDATE account SET balance = 90 (absolute) IS idempotent.

Appending to a log is not idempotent. Inserting a row with no unique constraint creates duplicates. Add a unique constraint on operation_id and use INSERT ... ON CONFLICT DO NOTHING.

Sending notifications is not idempotent. Sending an email twice delivers two emails. Deduplication by message ID at the notification layer is required before retrying.

Stripe's Idempotency Key Implementation

Stripe is the gold standard reference implementation. Every mutating API call should include an Idempotency-Key header. Stripe stores keys and results for 24 hours.

import stripe
import uuid
import time

stripe.api_key = "sk_live_..."

def create_payment_intent_idempotent(amount: int, currency: str, customer_id: str) -> dict:
    """
    Create a PaymentIntent safely with idempotency.
    The key is tied to the logical operation, not the HTTP request.
    Generate it from stable business inputs so a server restart
    with the same parameters reuses the same key.
    """
    # Stable key: tied to customer + amount + currency
    # Same business operation always generates the same key
    # This prevents duplicates even across process restarts
    idempotency_key = f"pi-{customer_id}-{amount}-{currency}-{int(time.time() // 3600)}"
    # ↑ Bucketed to 1-hour window: same customer can only create
    #   one $X payment per hour without a new key

    try:
        intent = stripe.PaymentIntent.create(
            amount=amount,
            currency=currency,
            customer=customer_id,
            idempotency_key=idempotency_key,
        )
        return {"status": "created", "intent_id": intent.id}

    except stripe.error.IdempotencyError as e:
        # Stripe detected the same key with different parameters
        # This is a client bug — different request with same key
        raise ValueError(f"Idempotency key reused with different params: {e}")

    except stripe.error.RateLimitError:
        # 429 — honour Retry-After, then retry with SAME key
        raise  # caller handles backoff

Database-Level Idempotency

For operations that write to a database, use the database itself as the idempotency store — avoids the need for a separate cache:

-- Schema: store each idempotency key with its result
CREATE TABLE idempotency_keys (
    key         VARCHAR(255) PRIMARY KEY,
    status      VARCHAR(50)  NOT NULL,   -- 'processing', 'complete', 'failed'
    response    JSONB,                    -- cached response body
    created_at  TIMESTAMPTZ DEFAULT NOW(),
    expires_at  TIMESTAMPTZ DEFAULT NOW() + INTERVAL '24 hours'
);

-- Cleanup job (run every hour)
DELETE FROM idempotency_keys WHERE expires_at < NOW();

-- Index for TTL cleanup
CREATE INDEX idx_idem_expires ON idempotency_keys(expires_at);
from contextlib import contextmanager
import psycopg2
import json

def process_with_idempotency(conn, idempotency_key: str, operation_fn):
    """
    Execute operation_fn exactly once, regardless of how many times called
    with the same idempotency_key.

    Uses advisory locks to prevent race condition where two concurrent
    requests with the same key both see "not cached" simultaneously.
    """
    with conn.cursor() as cur:
        # 1. Acquire advisory lock on hash of the key
        # Prevents two concurrent identical requests from both running
        key_hash = hash(idempotency_key) % (2**31)
        cur.execute("SELECT pg_try_advisory_xact_lock(%s)", (key_hash,))
        lock_acquired = cur.fetchone()[0]

        if not lock_acquired:
            # Another process is handling the same key right now
            # Wait briefly and retry lookup
            raise RetryAfterError("Key being processed by another worker")

        # 2. Check for existing result
        cur.execute(
            "SELECT status, response FROM idempotency_keys WHERE key = %s AND expires_at > NOW()",
            (idempotency_key,)
        )
        row = cur.fetchone()

        if row:
            status, response = row
            if status == 'complete':
                return json.loads(response)  # Return cached result
            elif status == 'processing':
                raise RetryAfterError("Still processing — retry in 1s")
            # status == 'failed' — re-run the operation

        # 3. Mark as processing (idempotency fence)
        cur.execute(
            """INSERT INTO idempotency_keys (key, status)
               VALUES (%s, 'processing')
               ON CONFLICT (key) DO UPDATE SET status = 'processing'""",
            (idempotency_key,)
        )
        conn.commit()

    # 4. Execute the actual operation (outside of lock scope)
    try:
        result = operation_fn()
        # 5. Store result
        with conn.cursor() as cur:
            cur.execute(
                "UPDATE idempotency_keys SET status='complete', response=%s WHERE key=%s",
                (json.dumps(result), idempotency_key)
            )
            conn.commit()
        return result

    except Exception as e:
        with conn.cursor() as cur:
            cur.execute(
                "UPDATE idempotency_keys SET status='failed' WHERE key=%s",
                (idempotency_key,)
            )
            conn.commit()
        raise

Circuit Breaker Integration

Retry and circuit breaker work together — retry handles individual transient failures, circuit breaker handles sustained failure. Without the circuit breaker, retries pile up indefinitely during an outage:

from enum import Enum
import threading
import time

class CircuitState(Enum):
    CLOSED = "closed"         # Normal — all requests pass through
    OPEN = "open"             # Failing — all requests rejected immediately (fast fail)
    HALF_OPEN = "half_open"   # Recovery probe — one request allowed through

class CircuitBreaker:
    def __init__(self, failure_threshold=5, recovery_timeout=30, half_open_max=1):
        self.failure_threshold = failure_threshold
        self.recovery_timeout = recovery_timeout
        self.half_open_max = half_open_max
        self.state = CircuitState.CLOSED
        self.failure_count = 0
        self.last_failure_time = None
        self.half_open_attempts = 0
        self._lock = threading.Lock()

    def call(self, fn, *args, **kwargs):
        with self._lock:
            if self.state == CircuitState.OPEN:
                elapsed = time.time() - self.last_failure_time
                if elapsed >= self.recovery_timeout:
                    self.state = CircuitState.HALF_OPEN
                    self.half_open_attempts = 0
                else:
                    raise CircuitOpenError(f"Circuit OPEN — retry in {self.recovery_timeout - elapsed:.0f}s")

            if self.state == CircuitState.HALF_OPEN:
                if self.half_open_attempts >= self.half_open_max:
                    raise CircuitOpenError("Half-open probe already in flight")
                self.half_open_attempts += 1

        try:
            result = fn(*args, **kwargs)
            with self._lock:
                # Success — reset circuit
                if self.state == CircuitState.HALF_OPEN:
                    self.state = CircuitState.CLOSED
                self.failure_count = 0
            return result

        except Exception as e:
            with self._lock:
                self.failure_count += 1
                self.last_failure_time = time.time()
                if self.failure_count >= self.failure_threshold:
                    self.state = CircuitState.OPEN
            raise

# Usage: circuit breaker wraps retry — don't retry when circuit is OPEN
payment_cb = CircuitBreaker(failure_threshold=5, recovery_timeout=30)

def charge_with_circuit_breaker(payload):
    try:
        return payment_cb.call(lambda: requests.post("/charge", json=payload))
    except CircuitOpenError:
        # Fast fail — no retry, no waiting
        return {"error": "payment_service_unavailable", "retry_after": 30}
Production Design Pattern

Layered defence: Idempotency key (prevents duplicates) + Exponential backoff with jitter (prevents thundering herd) + Retry budget (prevents retry amplification) + Circuit breaker (prevents cascade failures). Each layer addresses a different failure mode. Use all four together in any service that handles money, inventory, or notifications.

Enterprise Use Case

Enterprise Use Case Stripe — Idempotent Payment API at Scale

Context: Stripe processes hundreds of billions of dollars in payments annually. Every payment intent creation, capture, and refund is a POST request. Mobile clients on unreliable connections time out constantly. Server-side, deployments happen dozens of times per day, creating brief windows where in-flight requests may not receive responses. Without idempotency, even a 0.001% duplicate rate at $500B volume means $500M in erroneous charges.

How Stripe implements it:

  1. Every mutating API endpoint accepts an Idempotency-Key header (up to 255 chars, client-generated UUID).
  2. Stripe stores the key, the HTTP status, and the serialised response body in a distributed store with a 24-hour TTL.
  3. If the same key arrives before the original request completes, Stripe returns 409 Conflict — the client should wait and retry.
  4. If the key arrives with different parameters than the original, Stripe returns 422 — the client has a bug (can't reuse a key for a different operation).
  5. Idempotency keys are scoped per API key — a key "abc-123" from Merchant A doesn't conflict with "abc-123" from Merchant B.
  6. Their official SDKs automatically generate and attach idempotency keys, and implement exponential backoff with jitter for all retries.

Concrete numbers: Stripe's engineering blog reports that idempotency keys prevent an estimated 0.5–2% of all API calls from creating duplicate charges (depending on client SDK quality). At their scale, that represents billions of dollars in prevented duplicate charges each year. Their retry logic with jitter reduces peak retry traffic by approximately 75% compared to naive immediate retry.

Lessons for your system:

  1. Make idempotency opt-out, not opt-in — Stripe's SDKs attach keys automatically. If developers have to remember to add the key, they'll forget under deadline pressure.
  2. Store the full response, not just "processed" — On replay, return the exact original status code and body. Some clients check the response body; returning a generic 200 breaks them.
  3. 24 hours is the right TTL — Long enough for mobile devices to come back online after being offline; short enough to keep storage costs manageable.

Common Mistakes

Retrying Non-Idempotent POST Without an Idempotency Key

The most costly mistake. The timeout fires, the client retries, the server processes both requests. A customer gets charged twice, an order is created twice, an email is sent twice. The severity depends on the operation — double-charging a customer is a chargeback and a regulatory incident. The fix is mandatory: always generate and attach an idempotency key before the retry loop begins, not after the first failure.

Red flag in code review: Any retry loop around a POST/create call that doesn't carry an idempotency key through all attempts.

No Jitter on Retries (Thundering Herd)

A developer adds exponential backoff — good. But no jitter — bad. If 500 mobile clients all experience a connection error at the same second (e.g. during a brief BGP route flap), they all back off for exactly 2 seconds, then all hit the server simultaneously again. The spike recreates the original failure. Jitter — even just sleep(backoff * random()) — smears those 500 retries across the full backoff window and prevents the spike.

Real numbers: AWS's "Exponential Backoff and Jitter" blog post shows full jitter achieving 7.5× better throughput than fixed backoff in high-contention scenarios.

Infinite or Uncapped Retries

A developer sets max_attempts = None "to make the system more resilient." When a downstream service goes down for 2 hours, every caller retries indefinitely. Background job queues fill with thousands of stuck tasks, each holding threads and database connections. The retry storm takes down services that had nothing to do with the original failure. Always cap retries (3–5 is typical) and combine with circuit breakers to stop retrying against services that are clearly down for extended periods.

Symptom: CPU and memory on callers climbs over time during a downstream outage, even though those callers "aren't doing anything."

Interview Prep

Retries handle transient failures by trying the operation again; idempotency ensures those retries never cause duplicate side effects like double charges — together they make distributed systems both resilient and correct.

2-Minute Structured Answer

"In distributed systems, transient failures are inevitable — network packets drop, servers restart, load spikes cause timeouts. The right response is to retry, but retrying safely requires idempotency. Without it, retrying a payment POST can double-charge a customer. The pattern is: the client generates a UUID idempotency key once before the retry loop and sends it on every attempt; the server stores the key and caches the response, returning the cached result on duplicate requests. For the retry timing itself, you always want exponential backoff — each attempt waits twice as long — plus full jitter, which adds randomness to prevent hundreds of clients from retrying simultaneously and creating a thundering herd. The key trade-off is between retry amplification — more retries means more load on an already-struggling service — and availability — fewer retries means more user-visible errors. Retry budgets and circuit breakers cap the amplification."

Common Interview Mistakes

  • Saying "just retry on any error" without distinguishing transient vs permanent — retrying a 400 wastes resources and masks bugs.
  • Describing exponential backoff without mentioning jitter — this is the most common gap interviewers probe for specifically.
  • Treating idempotency as only a payment concern — any operation with side effects (email, notification, inventory decrement) needs it.
  • Not knowing that GET/PUT/DELETE are idempotent by definition but POST is not — HTTP fundamentals are fair game.
  • Forgetting that idempotency keys need a TTL — unlimited storage of keys isn't operationally viable.
