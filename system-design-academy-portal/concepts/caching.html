Caching

Stores copies of frequently requested data in fast-access storage so repeated requests can be served without hitting the original, slower data source.

Category Data
Difficulty All Levels
Read Time ~12 min
Related CDN, Databases, Load Balancer

Definition & Analogy

Real-World Analogy

A chef who keeps frequently used ingredients on the counter (the cache) instead of walking to the pantry every time (the database). The counter has limited space, so only the most-used items stay there. When someone asks for salt, the chef grabs it from the counter instantly. For a rare spice, they walk to the pantry.

Databases are slow relative to memory. A database query might take 50–200 ms. Serving the same result from an in-memory cache takes 1–5 ms. For read-heavy systems where the same data is requested thousands of times per second, caching is the difference between a system that works and one that doesn't. Without caching, popular content causes database overload even when the data hasn't changed.

At its core, caching is a trade-off: you accept a degree of data staleness in exchange for dramatically lower latency and reduced backend load. Choosing the right TTL, eviction policy, and invalidation strategy turns that trade-off into a controlled, measurable engineering decision rather than a liability.

The Story

Real-World Scenario
Scenario: A social media platform shows a "Trending Topics" sidebar on every page. The trending topics query runs against a database that aggregates 50 million posts from the last hour. The query itself takes 800 ms and hits a complex aggregation pipeline.
The Problem: With 5 million daily active users, the trending topics query was executing 2 million times per hour — even though the results changed only once every 5 minutes. The database was spending 60% of its capacity running the same expensive query repeatedly. Response times degraded, other queries slowed, and at peak hours the database became a bottleneck for the entire platform.
The Solution: The team cached the trending topics result in Redis with a 5-minute TTL (time-to-live). The first request each 5 minutes runs the expensive database query and stores the result in Redis. All subsequent requests within that 5-minute window read from Redis in under 2 ms. A single cache-aside pattern, one Redis cluster, zero application rewrites.
What Changed: Database load dropped 94%. Trending topics response time fell from 800 ms to 2 ms. The database recovered capacity for user-generated writes and other complex queries. Peak-hour performance stabilized. Cost: one Redis cluster.

Learn at Your Level

Beginner

Your computer is fast but your hard drive is slow. That's why your computer uses RAM — it copies your most-used files into fast memory so you don't have to wait for the hard drive every time. That's caching.

In web systems, the "hard drive" is a database — it stores everything permanently but takes time to answer questions. The "RAM" is a cache — it stores recent answers in fast memory.

When your app gets a question it has answered recently (like "what are the trending topics?"), it checks the cache first. If the answer is there (a cache hit), it returns it instantly. If not (a cache miss), it goes to the database, gets the answer, stores a copy in the cache, and returns it.

The catch: the cache has limited space, and data in it can become stale (outdated) if the original data changes. That's why each cached item has an expiry time (TTL) — after which it's removed and the next request goes back to the database for a fresh answer.

Key Terms

  • Cache hit — The requested data is found in the cache. Fast. Cheap.
  • Cache miss — The data is not in the cache. Must go to the database. Slow. Expensive.
  • TTL (Time to Live) — How long a cached item stays before it expires and is deleted.
  • Eviction — Removing items from the cache when it's full to make room for new ones.

Mental Model

Think of caching as a "cheat sheet." The first time you solve a hard problem, you write the answer down. The next time someone asks the same question, you read the cheat sheet instead of re-solving it. The cheat sheet eventually goes out of date — that's TTL.

Caches live at multiple layers of a system. Each layer serves a different purpose and has different characteristics.

Cache Layers

  • Client-side cache (browser) — Stores static assets (JS, CSS, images) locally. HTTP cache headers control TTL. Zero network cost on hit.
  • CDN cache — Serves static and semi-static content from edge nodes near the user. No origin hit needed. Geographic distribution of cached content.
  • Application cache (Redis / Memcached) — Stores computed results, session data, or frequently read records in shared memory. The most commonly added cache layer in backend systems.
  • Database query cache — Database engines can cache query results internally. Less common in practice — often disabled in favor of application-level caching where developers have more control.

Cache-Aside Pattern (Most Common)

Write-Through Pattern

  1. Application writes new data
  2. Write to cache AND database simultaneously
  3. Cache always has the latest version
  4. Higher write latency, but reads never go stale

Failure Scenario: Cache Stampede

Redis cluster loses its primary node. Failover to replica takes 30–60 seconds. During that window, all cache reads miss. Every request hits the database. If the database isn't sized for 100% traffic load (it usually isn't, because the cache absorbs 90%+), it gets overwhelmed. This is a cache stampede — the scenario that breaks systems after a cache outage.

Caching introduces consistency trade-offs that become harder to manage at scale. The choice of invalidation strategy is the most consequential architectural decision in a caching system.

Cache Invalidation Strategies

Strategy Pros Cons Best For
TTL-based expiry Simple, no coordination needed Stale data for up to TTL duration Read-heavy, freshness not critical
Write-through Always consistent with source Higher write latency, double writes User settings, profile data
Write-behind (write-back) Fastest writes Data loss risk on cache failure, complex Write-heavy, async persistence OK
Event-driven invalidation Precise, low staleness window High complexity, requires event bus Financial data, inventory, real-time
Cache-aside (lazy loading) Flexible, only caches what's read Thundering herd on miss, stale risk General-purpose read optimization

Eviction Policies

  • LRU (Least Recently Used) — Evicts the key that hasn't been accessed for the longest time. Best for most workloads with temporal locality.
  • LFU (Least Frequently Used) — Evicts the key accessed the fewest times overall. Better for long-lived hot keys. Requires frequency counters.
  • TTL-based (volatile-ttl) — Evicts keys with the nearest expiry first. Good when you want to control freshness budget.
  • Random — Evicts a random key. Surprisingly effective under uniform access patterns, zero overhead.
  • No eviction (noeviction) — Returns errors when memory is full. Only use for caches where you guarantee bounded key growth.

Scale Considerations

Architect's Scale Model

At 1K req/sec: Single Redis instance is sufficient. Hit rate optimization matters more than infrastructure. Focus on choosing the right keys and TTLs.

At 100K req/sec: Redis Cluster with 3+ shards. The hot key problem emerges — a single popular key overwhelms one shard. Use key replication (prefix sharding) or local in-process cache as a second layer in front of Redis.

At 1M req/sec: Multi-layer caching is mandatory. Local in-process LRU cache (1,000-item limit per instance) → Redis Cluster → Database. The local cache absorbs 80–90% of reads within each app server, Redis handles cross-instance sharing, and the database handles only misses through both layers.

When NOT to Use Caching

  • Highly dynamic data with strict consistency: Financial balances, inventory counts. Cache invalidation complexity outweighs benefits. Use read replicas instead.
  • Data unique per request: User-specific pages where no two requests share a cache key. Hit rate approaches zero — all overhead, no benefit.
  • Write-heavy with rare reads: Caching adds write complexity for data that's rarely read back. Measure before adding cache.
Performance Reference

Redis get/set latency: ~0.1–0.3 ms on local network. Compare to database query: 5–200 ms. At 100K reads/sec, caching a 50 ms query saves 5,000 seconds of database CPU per second. That translates to significant infrastructure cost reduction at any meaningful scale.

At enterprise scale, caching is a distributed consistency problem wrapped in a performance optimization. The infrastructure is the easy part; the hard parts are key design, ownership, compliance, and stampede protection.

Multi-Region Cache Architecture

Each region has its own Redis cluster. Cross-region cache synchronization is generally not worth it — network latency defeats the purpose of a cache. Accept that each region's cache warms independently. On failover or traffic switchover, expect cold-cache performance for 5–15 minutes. Design your application to handle this gracefully: circuit breakers on cache misses, graceful degradation to slower DB reads, and auto-warming scripts triggered on deploy.

Compliance: GDPR & HIPAA

  • Redis is typically not an approved long-term data store. Cache must not store raw PII.
  • Cache user IDs as keys, not names, emails, or health identifiers.
  • PHI in cache requires the same encryption at rest and in transit as the primary database.
  • Cache eviction must be auditable for compliance reporting — document TTL policy per key pattern.

Security Posture

Default Redis is Insecure

Redis default config has no authentication and binds to all interfaces. Production Redis must have: AUTH password enabled, TLS in transit, network-level access controls (VPC security groups, no public exposure). Cache poisoning is a real attack vector — never allow user input to directly form cache keys. Always namespace and sanitize keys.

Migration Path: Adding Cache to an Existing System

  1. Identify top 5 most expensive, most frequent read queries using the slow query log.
  2. Implement cache-aside for those 5 queries. Measure hit rate after 24 hours.
  3. If hit rate > 60%, proceed. If < 60%, reconsider TTL or query selection — you may be caching the wrong things.
  4. Add monitoring: hit/miss ratio, memory usage, and eviction rate per key prefix.
  5. Expand to more queries. Aim for 80%+ overall hit rate.
  6. Add stampede protection (mutex lock or probabilistic early expiry) for keys that cause DB spikes on miss.

Enterprise Use Case

Enterprise Scenario High-Scale SaaS Analytics Platform (Salesforce-scale CRM)

Context: A SaaS analytics platform serves dashboards to 500,000 business users. Dashboard data aggregates transactions across hundreds of tables. Each dashboard has 20+ widgets. During business hours, 50,000 users refresh dashboards simultaneously. The underlying data warehouse processes 10 billion rows. Warehouse queries take 3–30 seconds. Users expect dashboard load times under 2 seconds.

Multi-layer caching strategy deployed:

1. CDN layer: Static assets (JS, CSS, images) cached at edge nodes with 30-day TTL via cache-control headers. Zero origin hits for static content.

2. Application cache (Redis Cluster): Computed dashboard results cached per user, per dashboard, per time range with a 5-minute TTL. Each cache key encodes the user ID, dashboard ID, and query time bucket.

3. Query result cache (materialized views): Common aggregations materialized at the database level and refreshed every 15 minutes. Reduces query complexity for cache misses.

4. Object cache: Individual records (company profiles, user settings) cached with 60-second TTL and write-through invalidation on update. Always consistent for profile data.

5. Cache warming: New deployments pre-warm caches for the top 10,000 dashboards (by access frequency) before cutover. Users on hot dashboards never see a cold-cache latency spike after deploy.

Results: Average dashboard load time fell from 8 seconds to 1.4 seconds. User satisfaction score rose from 3.1 to 4.3 out of 5. Support tickets about slowness dropped from 400/week to 40/week. Database compute cost reduced 35%.

Common Mistakes

Mistake 1 — No TTL Set (Unbounded Cache Growth)

Caching without expiry fills memory until Redis starts evicting randomly or crashes under memory pressure. Every cache key must have a TTL. If you can't decide on a TTL, start at 60 seconds and tune up from there based on measured staleness tolerance. The default Redis policy noeviction returns errors when full — a cache that errors is worse than no cache.

Mistake 2 — Caching Data That Changes on Every Write

Caching something that gets invalidated more often than it gets read means you pay the write complexity without getting the read benefit. If your cache is being invalidated on every write and the data is only read once between writes, your hit rate approaches zero. Measure hit rate. If it's under 40%, the caching strategy is wrong — you're adding a write-path cost with no read benefit.

Mistake 3 — Cache Stampede on Expiry

At peak load, a popular key expires. Thousands of requests simultaneously miss the cache and hammer the database with the same expensive query. The database spikes, latency climbs, and cascading failures begin. Three protection patterns: probabilistic early expiry (start refreshing before TTL expires based on a random probability), mutex lock (first miss acquires lock, other requests wait for it to populate, then serve from cache), or background refresh (async job refreshes popular keys before they expire, key never goes cold under load).

Interview Prep

Caching stores copies of expensive-to-compute data in fast storage so repeated requests can be served without going to the original slow source.

Structured Answer (2 minutes)

"Caching stores results of expensive operations in fast memory so future requests for the same data skip the slow path. It solves the read amplification problem — when the same data is requested thousands of times per second but only changes occasionally. For example, Twitter caches tweet content and user timelines in Redis — the tweet exists once in the database but might be read 50 million times."

"The main trade-off is freshness vs performance: a long TTL means more cache hits but potentially stale data; a short TTL means fresher data but more database load. The hardest problem is cache invalidation — knowing when to remove stale data — which is why 'there are only two hard problems in computer science: cache invalidation and naming things.'"

"In production I always monitor hit rate (target 80%+), eviction rate (signals the cache is too small), and replica lag. For high-traffic keys I implement stampede protection via mutex or probabilistic early expiry."

Practice Questions

What is the difference between cache-aside and write-through caching?

Cache-aside (lazy loading): The application is responsible for reading from the cache and populating it on miss. On a miss: read from DB, write to cache, return data. The cache only contains data that has actually been requested. Risk: cache miss causes a "thundering herd" if many requests miss simultaneously for a popular key.

Write-through: Every write goes to both cache and database simultaneously. The cache is always consistent with the source. Risk: higher write latency (two writes instead of one), and the cache may contain data that's never read (write amplification for write-heavy datasets).

Cache-aside is better for read-heavy systems with occasional writes. Write-through is better when consistency is critical and write volume is manageable.

How would you prevent a cache stampede?

A cache stampede (thundering herd) occurs when a popular key expires and many requests simultaneously miss the cache, hammering the database. Three mitigations:

1. Mutex / lock: The first request to miss acquires a distributed lock (e.g., Redis SETNX), queries the DB, and repopulates the cache. All other requests wait for the lock and then read from the newly populated cache. Simple but adds latency for waiting requests.

2. Probabilistic early expiry (XFetch): Before the TTL expires, each request has a small random probability of triggering a background refresh. The probability increases as the key approaches expiry. The cache key is refreshed proactively without ever going cold.

3. Background refresh: A background process (cron or event-driven) refreshes keys asynchronously before they expire. The key is never served stale to the user, and the database is hit by exactly one process per refresh cycle, not by thousands of users simultaneously.

What data should you NOT cache?

Financial balances and inventory counts: Staleness is unacceptable. If your cache says a user has $500 and the real balance is $0, you have a fraud risk. Use read replicas for read scaling instead of application caches.

Highly unique per-request data: If every request has a different cache key (e.g., personalized recommendations with 10+ parameters), hit rate approaches zero. You pay the overhead of a cache check and store with no benefit.

Raw PII or PHI: Unless your cache is encrypted at rest and in transit with the same controls as your database, personal data (names, emails, health data) should not be stored in Redis. Use IDs as cache keys, look up names from the DB only when needed.

Write-heavy data with rare reads: If data is updated every second but only read once per hour, caching it means you're doing constant cache invalidation with almost no cache hits. The write complexity is pure overhead.

How does Redis Cluster handle hot keys?

In Redis Cluster, each key is assigned to a hash slot, and each shard owns a range of hash slots. A "hot key" — a single key with extremely high read/write volume — routes all its traffic to one shard, creating a hotspot regardless of cluster size.

Solutions:

1. Key replication with local scatter: Store the same value under N keys (e.g., trending:global:0 through trending:global:9) and randomly select one on read. Each shard handles 1/N of the load.

2. Local in-process cache: Add an in-process LRU cache (e.g., a 1,000-entry map) per application server in front of Redis. Hot keys are served entirely from process memory. Only misses hit Redis. This eliminates the hot-key problem for read-only data entirely.

3. Redis read replicas: Route reads to replicas of the hot shard. Adds read capacity without sharding the key.

Trade-Offs Summary

Dimension Benefit Cost
Performance Read latency 10–100× lower (ms vs sub-ms) Write complexity — invalidation strategy required on every write
Complexity Simple to add cache-aside to an existing system Hard to get right: invalidation, stampede protection, consistency windows
Cost Reduces DB compute cost significantly — cache pays for itself at modest scale Redis/Memcached cluster cost + ops overhead + monitoring
Consistency Configurable freshness window per key pattern Potential for stale data reads within TTL window
Scalability Dramatically extends DB read capacity without DB hardware changes Cache itself needs scaling: cluster, hot key mitigation, multi-layer
Reliability Reduces DB load so DB handles failures more gracefully Cache failure causes DB stampede — system must be designed for cache-down scenarios
