Databases

The persistent backbone of every application — choosing the right database is one of the most consequential architectural decisions you will ever make.

Beginner Data Storage ACID

Definition & Analogy

Simple Analogy

A database is like a filing cabinet in a highly organized office. A relational database (SQL) is a cabinet with perfectly labelled folders, rigid dividers, and cross-referenced indexes — finding anything is blazing fast, and the cabinet refuses to accept misfiled paperwork. A NoSQL database is more like a big open shelf where boxes can hold any shape of contents — flexible to add new kinds of documents, but you need to know roughly where you put things.

Every application eventually needs to remember something between requests. Databases are the systems that store, retrieve, and manage that data reliably. The choice is not simply "SQL or NoSQL" — it is matching the database engine's data model, consistency guarantees, and scaling characteristics to the specific read/write patterns and reliability requirements of your service.

Understanding databases at the design level — ACID properties, indexing, connection pooling, read replicas — separates engineers who can build greenfield apps from engineers who can build systems that survive at scale.

The Story

Real-World Story
Instagram launched in October 2010 on a single PostgreSQL instance running on Amazon EC2. The app was a photo-sharing product; the team was three people. They chose PostgreSQL because it was rock-solid, well-understood, and could handle the initial workload without operational complexity.
Within 18 months Instagram had 27 million users. Their single PostgreSQL database was struggling — write throughput was hitting the ceiling, read queries for popular users' feeds were taking seconds, and every new follow/like/comment relationship required multiple table joins across hundreds of millions of rows. A single VACUUM run could lock the entire table for minutes. The engineering team had zero headroom for growth.
The team introduced read replicas to offload all read traffic from the primary write node. They added Redis as a caching layer in front of PostgreSQL for hot data (user profile lookups, follower counts). They sharded the user table horizontally across multiple PostgreSQL instances — each shard owned a range of user IDs. Critical counters (likes, follower counts) moved to Redis entirely, providing sub-millisecond reads with periodic flushing back to PostgreSQL for durability.
Instagram scaled to over 1 billion monthly active users on an architecture still anchored by PostgreSQL — but with read replicas, connection pooling via PgBouncer, strategic Redis caching, and horizontal sharding. The lesson: start with a proven relational database, add specialized stores (Redis, Cassandra) only where the data model genuinely demands it, and scale with read replicas before reaching for a full rewrite.

Learn at Your Level

Beginner

Think of a database as a very smart spreadsheet that lives on a server. A relational database (like PostgreSQL or MySQL) is a collection of linked spreadsheets called "tables." Each row is one record. Each column is one field. The "relational" part means rows in different tables can link to each other — an order row links to a customer row, which links to an address row.

A NoSQL database (like MongoDB or Redis) stores data differently. MongoDB stores documents — like JSON objects — so one document can hold all the information about a customer including their orders and address, nested inside one record. Redis stores simple key-value pairs in memory and is extraordinarily fast because it never touches a disk during normal reads.

The fundamental question when choosing a database: does your data have a fixed, well-known structure that benefits from relationships (use SQL), or is it flexible, document-shaped, or needs sub-millisecond speed (consider NoSQL)?

The Four Database Types You'll Actually Use

  • PostgreSQL — Relational, ACID-compliant, complex queries, joins, transactions. The right default for most applications.
  • MongoDB — Document store. JSON-like documents. Flexible schema. Good for content, catalogues, user profiles with variable attributes.
  • Redis — In-memory key-value store. Under 1ms reads. Used for caching, sessions, rate limiting, real-time counters.
  • Cassandra — Wide-column store. Designed for write-heavy workloads at massive scale. Multi-region by default. Good for time-series, logs, IoT data.

Key Terms

  • Schema — The structure definition of a table: column names, data types, constraints.
  • Index — A data structure (like a book index) that speeds up lookups on a specific column. Without one, every query scans every row.
  • Primary key — A unique identifier for each row (e.g., user_id). Always indexed automatically.
  • Foreign key — A column that references the primary key of another table, creating the "relationship."
  • Transaction — A group of operations that either all succeed or all fail together (e.g., deducting money from one account and adding it to another).

At the intermediate level, you need to understand how databases guarantee correctness (ACID), how indexes actually work (B-trees), and the patterns used to scale a single database before reaching for distributed systems.

ACID Properties — What They Actually Mean

ACID is the set of four guarantees that make relational databases trustworthy for financial and business-critical operations.

  • Atomicity — A transaction is all-or-nothing. If you transfer $100 from account A to account B and the server crashes after the debit but before the credit, the transaction rolls back entirely. No partial writes.
  • Consistency — A transaction can only bring the database from one valid state to another. A row cannot violate a NOT NULL constraint, a foreign key cannot point to a non-existent record, and a balance cannot go below zero if a CHECK constraint prevents it.
  • Isolation — Concurrent transactions cannot see each other's in-progress changes. If two users both try to book the last ticket simultaneously, isolation ensures only one succeeds. PostgreSQL supports multiple isolation levels: READ COMMITTED (default), REPEATABLE READ, and SERIALIZABLE (strictest).
  • Durability — Once a transaction commits, it stays committed even if the server crashes immediately after. PostgreSQL achieves this via Write-Ahead Logging (WAL): every change is written to a log on disk before being applied to the data file.

How Indexes Work (B-Tree)

A B-tree index is a balanced tree where each node holds sorted keys and pointers. Looking up WHERE user_id = 12345 traverses the tree in O(log n) time instead of scanning all rows O(n). For a table with 10 million rows, that's the difference between checking ~24 nodes vs. 10,000,000 rows.

-- Without index: full table scan (slow on millions of rows)
SELECT * FROM tickets WHERE user_id = 42;

-- Create a B-tree index on the queried column
CREATE INDEX idx_tickets_user_id ON tickets (user_id);

-- Now the same query uses the index (fast)
SELECT * FROM tickets WHERE user_id = 42;

-- Check which index is being used with EXPLAIN ANALYZE
EXPLAIN ANALYZE SELECT * FROM tickets WHERE user_id = 42;
-- Output will show: Index Scan using idx_tickets_user_id
-- vs. Seq Scan (full table scan) without the index

Connection Pooling with PgBouncer

Each PostgreSQL connection consumes ~5–10 MB of memory and takes 20–40ms to establish. A web app with 500 concurrent requests cannot open 500 direct database connections — the database will run out of memory and reject connections. PgBouncer sits between your app and PostgreSQL, maintaining a small pool of real database connections (e.g., 50) and multiplexing thousands of application requests through them.

[databases]
# Map application database name to real PostgreSQL host
ticketflow = host=postgres-primary.internal port=5432 dbname=ticketflow

[pgbouncer]
listen_addr = 0.0.0.0
listen_port = 5432
auth_type = md5
auth_file = /etc/pgbouncer/userlist.txt

# Pool mode: transaction (release connection after each transaction)
# vs session (hold connection for full client session)
pool_mode = transaction

# Max connections to the actual PostgreSQL server
server_pool_size = 50

# Max client connections to PgBouncer (what the app connects to)
max_client_conn = 1000

# Timeout for idle server connections
server_idle_timeout = 600

Read Replicas

PostgreSQL streaming replication sends WAL records from the primary to one or more read replicas in real time. Replicas are read-only copies of the primary. Application code must direct writes to the primary and reads to replicas. This is the first scaling step for read-heavy applications — Instagram's architecture ran 1 primary + multiple read replicas before needing sharding.

import psycopg2

# Primary: all writes go here
primary_conn = psycopg2.connect(
    host="postgres-primary.internal",
    port=5432,
    dbname="ticketflow",
    user="app_user",
    password="secret"
)

# Read replica: all reads go here
replica_conn = psycopg2.connect(
    host="postgres-replica.internal",
    port=5432,
    dbname="ticketflow",
    user="app_user",
    password="secret"
)

def create_ticket(event_id, user_id, seat):
    """Writes always go to primary."""
    with primary_conn.cursor() as cur:
        cur.execute(
            "INSERT INTO tickets (event_id, user_id, seat) VALUES (%s, %s, %s) RETURNING id",
            (event_id, user_id, seat)
        )
        primary_conn.commit()
        return cur.fetchone()[0]

def get_user_tickets(user_id):
    """Reads go to replica — primary is not touched."""
    with replica_conn.cursor() as cur:
        cur.execute(
            "SELECT id, event_id, seat FROM tickets WHERE user_id = %s",
            (user_id,)
        )
        return cur.fetchall()
Replication Lag Warning

Read replicas introduce replication lag — typically 10–100ms on a healthy system, but can spike to seconds under high write load or network issues. If a user just created a ticket and immediately queries for their tickets via a read replica, they may not see it yet. Design around this: either read from primary for "read-your-own-writes" scenarios, or accept eventual consistency for non-critical reads.

At the advanced level, you're making database selection and scaling decisions for systems handling millions of operations per day. The right database choice depends on the specific access patterns, not just the data model.

Query Planning — Using EXPLAIN ANALYZE

The query planner decides how to execute your SQL. EXPLAIN ANALYZE shows you what it chose and how long it actually took. Reading this output is a core skill for database performance work.

-- Run EXPLAIN ANALYZE to see the actual execution plan
EXPLAIN ANALYZE
SELECT t.id, t.seat, e.name, u.email
FROM tickets t
JOIN events e ON t.event_id = e.id
JOIN users u ON t.user_id = u.id
WHERE t.event_id = 1234
  AND t.status = 'confirmed';

-- Typical output to look for:
--
-- Nested Loop  (cost=0.43..1850.23 rows=42 width=89) (actual time=0.045..12.3 ms rows=38 loops=1)
--   -> Index Scan using idx_tickets_event_id on tickets t
--        (actual time=0.021..3.1 ms rows=38 loops=1)
--        Index Cond: (event_id = 1234)
--        Filter: (status = 'confirmed')
--        Rows Removed by Filter: 2
--   -> Index Scan using users_pkey on users u
--        (actual time=0.21..0.22 ms rows=1 loops=38)
--
-- Key numbers to watch:
-- - "Seq Scan" instead of "Index Scan" → missing index
-- - "Rows Removed by Filter" very high → index not selective
-- - actual time >> estimated time → stale statistics (run ANALYZE)
-- - loops=N on inner join side → N×M problem (classic N+1)

The N+1 Query Problem

The most common performance mistake in ORM-heavy codebases. Fetching a list of N records, then executing one additional query per record = N+1 round trips to the database.

# BAD: N+1 queries
# 1 query to get all events, then 1 query per event to get ticket count
events = db.query("SELECT * FROM events WHERE date = '2025-01-15'")  # 1 query
for event in events:
    # This runs once per event! 100 events = 100 additional queries
    count = db.query(
        "SELECT COUNT(*) FROM tickets WHERE event_id = %s",
        (event['id'],)
    )
    event['ticket_count'] = count

# GOOD: 2 queries total (or 1 with JOIN)
events = db.query("SELECT * FROM events WHERE date = '2025-01-15'")
event_ids = [e['id'] for e in events]

# Fetch all counts in a single query
counts = db.query("""
    SELECT event_id, COUNT(*) as ticket_count
    FROM tickets
    WHERE event_id = ANY(%s)
    GROUP BY event_id
""", (event_ids,))

counts_map = {row['event_id']: row['ticket_count'] for row in counts}
for event in events:
    event['ticket_count'] = counts_map.get(event['id'], 0)

# BEST: Single JOIN query
events = db.query("""
    SELECT e.*, COUNT(t.id) as ticket_count
    FROM events e
    LEFT JOIN tickets t ON t.event_id = e.id
    WHERE e.date = '2025-01-15'
    GROUP BY e.id
""")

When NOT to Use Each Database

  • Don't use PostgreSQL when your write volume exceeds what a single node can sustain (roughly 50K+ writes/sec for complex transactions) and you're not willing to implement manual sharding.
  • Don't use MongoDB as your primary store if your data has natural relationships requiring joins — you'll end up manually joining in application code, which is worse than SQL joins.
  • Don't use Redis as your system of record. Redis persistence (AOF/RDB) can lose up to 1 second of data. It is a cache and secondary store, not a durable primary database.
  • Don't use Cassandra for read-heavy OLTP workloads or when you need flexible ad-hoc queries — its data model is optimized for known access patterns at write scale.

At enterprise scale, the database layer is where architecture complexity concentrates. Multi-region consistency, compliance, cost modelling, and operational overhead all intersect here.

Multi-Region Database Strategies

Multi-region databases present a fundamental tension defined by the CAP theorem: you can have Consistency or Availability across a network partition, not both simultaneously.

  • Active-Passive (one primary region, replicas elsewhere) — Strong consistency for writes (all writes go to primary). Read latency for other regions. Failover takes 30–120 seconds. Used by most systems that cannot tolerate stale reads for business-critical data.
  • Active-Active (multi-master) — Writes accepted in any region. Conflict resolution required (last-write-wins, CRDTs, application-level merging). Used by systems that prioritize availability — social feeds, IoT data ingestion. CockroachDB and Google Spanner implement this with consensus protocols (Raft/Paxos).
  • Regional isolation with event sync — Each region owns its data, changes propagate via event streams (Kafka). Eventual consistency. Used for GDPR compliance where EU data must not leave EU, with eventual sync to a global analytics store.

Compliance Considerations

  • GDPR (EU) — Personal data of EU users must be stored in the EU or in countries with adequacy decisions. Database encryption at rest (AES-256) and in transit (TLS 1.2+). Right-to-erasure requires soft-delete + permanent purge pipeline. Data retention policies must be enforced at the database layer, not just application logic.
  • PCI DSS — Cardholder data (PANs) must never be stored in plaintext. Database-level encryption + tokenization. All access to card tables must be logged. Database audit logs retained for 1 year minimum.
  • HIPAA — PHI must be encrypted at rest and in transit. Access controls at row/column level (PostgreSQL row security policies). Audit trail of all accesses to PHI rows. Database backups must be encrypted.
-- PostgreSQL Row-Level Security for HIPAA compliance
-- Patients can only see their own records; doctors see their patients

-- Enable RLS on the table
ALTER TABLE patient_records ENABLE ROW LEVEL SECURITY;

-- Policy: patients see only their own records
CREATE POLICY patient_self_access ON patient_records
  FOR SELECT
  USING (patient_id = current_setting('app.current_user_id')::INTEGER);

-- Policy: doctors see records for their assigned patients
CREATE POLICY doctor_patient_access ON patient_records
  FOR SELECT
  USING (
    EXISTS (
      SELECT 1 FROM doctor_patient_assignments dpa
      WHERE dpa.doctor_id = current_setting('app.current_user_id')::INTEGER
        AND dpa.patient_id = patient_records.patient_id
    )
  );

-- Audit log trigger: record every access to sensitive data
CREATE OR REPLACE FUNCTION log_phi_access()
RETURNS TRIGGER AS $$
BEGIN
  INSERT INTO phi_audit_log (table_name, record_id, accessed_by, accessed_at, operation)
  VALUES (TG_TABLE_NAME, NEW.id, current_setting('app.current_user_id'), NOW(), TG_OP);
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER phi_access_audit
  AFTER SELECT OR INSERT OR UPDATE OR DELETE ON patient_records
  FOR EACH ROW EXECUTE FUNCTION log_phi_access();

Production Operations Checklist

  • Connection pooler (PgBouncer or RDS Proxy) in front of all PostgreSQL instances
  • Indexes created for all columns that appear in WHERE, JOIN ON, or ORDER BY clauses on large tables
  • EXPLAIN ANALYZE run on all queries added during development
  • Automated backups enabled with point-in-time recovery; restore tested monthly
  • Read replicas for all read-heavy services; primary reserved for writes only
  • Replication lag alert configured (>5 seconds triggers alert)
  • Slow query log enabled (log queries >100ms); reviewed weekly
  • autovacuum monitored; table bloat alert configured
  • Database credentials rotated via secrets manager (not hardcoded in app config)
  • Column-level encryption for PII fields (name, email, phone) using pgcrypto or application-layer encryption
  • Database access audit log shipped to SIEM for compliance

Enterprise Use Case

Enterprise Use Case Instagram — 1 Billion Users on PostgreSQL + Redis + Cassandra

Context: Instagram serves over 1 billion monthly active users. Every day, 100 million photos are uploaded, 4.2 billion likes are registered, and 500 million Stories are viewed. The data layer must sustain millions of writes per second, serve personalized feeds under 100ms, and maintain 99.99% availability globally.

Architecture:

  1. PostgreSQL (primary + read replicas, sharded) — The system of record for users, follows, media metadata. User table sharded by user_id across dozens of PostgreSQL clusters. Each shard has one primary (writes) and two replicas (reads). PgBouncer pools connections in front of every node.
  2. Redis Cluster — Follower/following counts, like counts, session tokens, feed assembly cache. Counts stored as Redis integers with atomic INCR — no locking, no write amplification. A like increments a Redis counter and enqueues a background job to flush to PostgreSQL. Result: sub-millisecond like registration under any load.
  3. Cassandra — Direct messaging storage. Wide-column model: partition key is the conversation_id, clustering key is message_timestamp. A query for "last 50 messages in conversation X" reads from a single Cassandra partition — one disk seek, deterministic latency. 13 billion rows in messaging at last public disclosure.
  4. Feed Fanout — When a user with 1 million followers posts a photo, Instagram writes a notification to Cassandra (timeline table) for each follower asynchronously via a distributed task queue. The feed is pre-computed and stored. Reading your feed is a fast Cassandra range query, not a real-time computation.

Business Impact: PostgreSQL's ACID guarantees ensure that follow/unfollow operations are consistent — you cannot simultaneously follow and not-follow someone. Redis ensures that likes, views, and follower counts respond under 1ms even during viral events. Cassandra ensures messaging remains available and consistent even when individual data centers have issues. The combination handles 10× traffic spikes (Super Bowl, product launches) without architectural changes.

Lessons for architects:

  1. Match the database to the access pattern — messaging (append-only, range reads) is a natural Cassandra workload. User relationships (complex joins, ACID) are a natural PostgreSQL workload. Using one database for both would have required either schema gymnastics or consistency trade-offs.
  2. Counters are not trivial at scale — a like counter that requires a SQL UPDATE + SELECT on 1 billion rows creates lock contention. Moving counters to Redis atomic increments with async persistence eliminated this bottleneck entirely.
  3. Caching is not optional at scale — it is the architecture — 95%+ of Instagram's reads are served from Redis. PostgreSQL handles less than 5% of reads at steady state. Your database schema and caching strategy are designed together, not independently.

Common Mistakes

No Indexes on Queried Columns

The single most common cause of database performance problems in production. Adding a column to a WHERE clause without creating an index means every query does a full sequential scan — reading every row in the table. A table with 10 million rows and no index on user_id will scan all 10 million rows every time you look up a user's records.

Symptom: Queries are fast in development (thousands of rows) and slow in production (millions of rows). EXPLAIN ANALYZE shows "Seq Scan" instead of "Index Scan."

Fix: Run EXPLAIN ANALYZE on every query during development. Create indexes for every column that appears in WHERE, JOIN ON, ORDER BY, or GROUP BY clauses on large tables. Use pg_stat_user_indexes to find unused indexes (they consume write overhead for no benefit).

N+1 Queries — Fetching Rows in a Loop

Loading a list of parent records and then issuing one database query per record to fetch related data. 100 orders → 100 + 1 = 101 database round-trips. At 10ms per round-trip, that's 1 full second of database time for what could be a single 20ms query. This pattern is invisible in unit tests and devastating in production.

Symptom: Page load times proportional to the number of records displayed. Database shows thousands of nearly identical queries per second. New Relic or Datadog shows high query count with low individual query time.

Fix: Use JOIN or batch SELECT ... WHERE id IN (...) to fetch all related records in one round-trip. In ORMs, use eager loading (include in ActiveRecord, select_related in Django, joinedload in SQLAlchemy). Enable query logging in development and fail CI if a test issues more than N queries.

Single Database as Single Point of Failure

Running your entire application against a single database instance with no replication, no backups, or no failover is a single point of failure that will eventually result in data loss or extended downtime. A disk failure, OS crash, or runaway query can take down the entire application with no recovery path.

Symptom: Any database-level incident (hardware failure, runaway migration, resource exhaustion) causes complete application downtime. Recovery from backup takes hours, not seconds.

Fix: Always run at least one read replica, even in early production — it doubles as a warm standby for failover. Enable automated backups with point-in-time recovery. Test restore from backup monthly (untested backups are not backups). Use managed database services (AWS RDS Multi-AZ, Google Cloud SQL with failover replica) that automate failover.

Interview Prep

A database persistently stores and retrieves structured data. SQL databases (PostgreSQL, MySQL) provide ACID guarantees and relational joins, best for transactional workloads. NoSQL databases (Redis, Cassandra, MongoDB) trade some of those guarantees for specific performance or scaling characteristics suited to specific access patterns.

2-Minute Structured Answer

"When choosing a database I start with the access patterns. For transactional systems with complex relationships — user accounts, orders, financial data — I default to PostgreSQL because ACID guarantees prevent data corruption and complex joins are handled natively. As read traffic grows I add read replicas and put PgBouncer in front for connection pooling. For sub-millisecond caching and session storage I add Redis — it's not a primary database, it's a fast layer in front of the primary. If I have write-heavy time-series data or need multi-region writes with no single master, Cassandra becomes the right tool. The key is that a production system often uses multiple databases — each for the workload it's best at, not one database for everything."

Common Interview Mistakes

  • Saying "I'd use MongoDB because SQL is too rigid" — without explaining what specific access patterns justify schema flexibility over ACID guarantees.
  • Treating Redis as a primary database — Redis persistence can lose data. Always clarify it's a cache/secondary store backed by a durable primary.
  • Not mentioning indexes when asked about slow database queries — the answer to "how do you fix a slow query" almost always starts with "check if the relevant columns are indexed."
  • Forgetting that read replicas have replication lag — this matters for read-your-own-writes consistency and real-time applications.
  • Conflating sharding with replication — sharding splits data horizontally for write scale; replication copies data for read scale and fault tolerance. They solve different problems.
