SQL vs NoSQL

Understand when to use relational databases with ACID guarantees and when to reach for document, key-value, wide-column, or graph stores.

Beginner Intermediate Data Databases Architecture

Definition & Analogy

Simple Analogy

SQL databases are like a carefully organised filing cabinet: every document goes into a labelled folder, every folder into a named drawer, every drawer into the same cabinet. The rigid structure means you can always find anything instantly — but you must decide on the folder labels before you file anything.

NoSQL databases are like a backpack: you can throw in a book, a laptop, a water bottle, and a crumpled receipt all at once. You get flexibility and quick access to whatever you dropped in last — but "find me everything under $10 that I bought in March" is considerably harder.

The choice between SQL and NoSQL is not about which technology is "better" — it is about matching the data model and access patterns to the problem. SQL excels at structured, relational data with complex queries. NoSQL excels at scale, flexibility, and specialised access patterns. Most large systems use both.

Understanding this trade-off is foundational to system design. Every interview question about database selection, every architectural decision about where to store data, and every production incident caused by the wrong database choice comes back to these fundamentals.

The Story

Real-World Story
A fast-growing SaaS company built their entire product on a single PostgreSQL instance. The schema was clean, queries were fast, and the team was happy. They stored users, subscriptions, feature flags, activity logs, cached search results, and real-time notifications — all in one Postgres database, because "it's relational data and we know SQL."
At 500,000 users the problems started. The activity_logs table had 800 million rows — every query touched it. Feature flag lookups fired 200 times per page load, each a round-trip to Postgres under full connection-pool pressure. The notification fan-out query took 4.2 seconds. The database CPU pegged at 95% during peak hours. Adding read replicas helped briefly, then the primary write volume overwhelmed the instance. Their $12K/month RDS bill was growing faster than revenue.
The team analysed their access patterns and separated concerns by data type. PostgreSQL kept what it was good at: users, subscriptions, billing (ACID, complex JOINs, reporting). Redis replaced the feature flag lookups (O(1) hash lookup, 0.3ms vs 12ms). Cassandra absorbed activity logs (append-only writes, time-range reads — exactly what wide-column is built for). Elasticsearch handled search. Each database solved the problem it was designed for.
Page load time dropped from 3.8 seconds to 400ms. Database costs fell by 60% because each store was right-sized for its workload. The Postgres primary CPU dropped to 15% at peak. The team could scale each component independently: Cassandra for write-heavy logging, Redis for ultra-low-latency lookups, Postgres for transactional correctness. This pattern — called polyglot persistence — is how every large-scale system is built.

Learn at Your Level

Beginner

Imagine you are organising your music collection. The SQL way is to build a spreadsheet: one sheet for Artists (ID, Name, Country), one sheet for Albums (ID, ArtistID, Title, Year), one sheet for Songs (ID, AlbumID, Title, DurationSeconds). Every piece of data lives in exactly one place. If an artist changes their name, you update one row and every song automatically reflects the change. This is called normalisation.

The NoSQL way is a folder of JSON files: one file per artist, and inside each file you embed all their albums and songs directly. Fetching "everything about Taylor Swift" is instant — one read, one file. But if Taylor Swift changes her name, you have to update it everywhere it appears.

Neither approach is wrong. SQL wins when you need consistency and complex queries across many things. NoSQL wins when you usually fetch one thing at a time and need to do it very fast at very large scale.

The Relational Model at a Glance

  • Tables — Data lives in rows and columns, like a spreadsheet.
  • Schema — You define the columns up front. Every row must have the same columns.
  • Primary Key — A unique identifier for each row (e.g., user_id = 42).
  • Foreign Key — A column that points to a row in another table, creating a relationship.
  • JOIN — Combining rows from two tables based on a shared column.
  • ACID — A guarantee that transactions are Atomic, Consistent, Isolated, and Durable.

SQL version: fetching Alice's orders requires a JOIN across three tables. MongoDB version: one document read returns everything. But to find all orders over $50 across all users, SQL runs one clean query; MongoDB must scan every user document.

At the intermediate level the question becomes: what are the actual technical strengths of each approach, and when do those strengths matter?

SQL Strengths

  • ACID transactions — Either all steps of a transaction complete, or none do. Critical for financial operations (debit one account, credit another — both must succeed or neither does).
  • JOINs — Query data across multiple tables without duplicating it. Normalised data is stored once and referenced everywhere.
  • Complex queries — Aggregations, filters, groupings, window functions — SQL is a full query language, not a key lookup API.
  • Schema enforcement — The database rejects bad data. You cannot accidentally insert a string into an integer column or leave a required field null.
  • Mature tooling — Decades of ORMs, query planners, migration tools, monitoring, and operational knowledge.
-- SQL: "Total revenue per country for orders placed in Q1 2024"
-- This query spans 3 tables — trivial in SQL
SELECT
    u.country,
    COUNT(DISTINCT o.order_id)  AS total_orders,
    SUM(oi.price * oi.quantity) AS total_revenue
FROM users u
JOIN orders o       ON o.user_id  = u.user_id
JOIN order_items oi ON oi.order_id = o.order_id
WHERE o.created_at BETWEEN '2024-01-01' AND '2024-03-31'
GROUP BY u.country
ORDER BY total_revenue DESC;

The Four NoSQL Families

1. Document Store (MongoDB, Firestore, CouchDB)

Stores JSON-like documents. Related data is embedded in one document. Best for: content management, user profiles, product catalogues — any entity that is always fetched as a whole and has variable structure.

// MongoDB: find all products in the "electronics" category priced under $100
db.products.find(
  { category: "electronics", price: { $lt: 100 } },
  { name: 1, price: 1, rating: 1 }
).sort({ rating: -1 }).limit(20);

2. Key-Value Store (Redis, DynamoDB, Memcached)

The simplest model: a key maps to a value. Lookup is O(1) — always sub-millisecond at any scale. Best for: session storage, caching, rate-limiting counters, feature flags, leaderboards.

# Redis: cache a user's profile with 15-minute TTL
import redis
r = redis.Redis()

# Write (SET with expiry)
r.setex(f"user:42:profile", 900, json.dumps(user_profile))

# Read (GET)
cached = r.get(f"user:42:profile")
if cached:
    return json.loads(cached)   # O(1), ~0.3ms

# Increment a counter atomically (rate limiting)
count = r.incr(f"rate_limit:user:42")
r.expire(f"rate_limit:user:42", 60)   # reset every minute

3. Wide-Column Store (Cassandra, HBase, DynamoDB in table mode)

Rows have a primary key but columns can vary per row and are grouped into "column families". Data is distributed across nodes by partition key. Best for: time-series data, activity logs, IoT sensor readings, write-heavy workloads across multiple regions.

-- Cassandra: activity log table optimised for "recent events for user X"
CREATE TABLE activity_log (
    user_id   UUID,
    event_ts  TIMESTAMP,
    event_type TEXT,
    metadata  MAP<TEXT, TEXT>,
    PRIMARY KEY (user_id, event_ts)   -- partition by user, sort by time
) WITH CLUSTERING ORDER BY (event_ts DESC);

-- Insert: O(1) regardless of table size (append to partition)
INSERT INTO activity_log (user_id, event_ts, event_type, metadata)
VALUES (uuid(), toTimestamp(now()), 'page_view', {'page': '/checkout'});

-- Query: "Last 50 events for user X" — reads a single partition, fast
SELECT * FROM activity_log
WHERE user_id = :user_id
LIMIT 50;

4. Graph Database (Neo4j, Amazon Neptune)

Relationships are first-class citizens — stored as edges, not foreign keys. Traversing "friends of friends" in a graph DB is O(depth), not O(table size). Best for: social networks, fraud detection, recommendation engines, knowledge graphs.

// Neo4j Cypher: "Find products that friends of Alice bought, that Alice hasn't bought"
MATCH (alice:User {name: "Alice"})-[:FRIEND]->(friend:User)
      -[:PURCHASED]->(product:Product)
WHERE NOT (alice)-[:PURCHASED]->(product)
RETURN product.name, COUNT(friend) AS friend_purchases
ORDER BY friend_purchases DESC
LIMIT 10;

Quick Decision Heuristic

  • You need ACID transactions (payments, bookings, inventory) → SQL
  • You have a fixed, well-understood schema → SQL
  • You need complex reporting/analytics queries → SQL
  • Your data is a flexible JSON document, read as a whole → Document (MongoDB)
  • You need sub-millisecond lookup by a single key → Key-Value (Redis)
  • You have write-heavy time-series or log data → Wide-Column (Cassandra)
  • Your most important queries traverse relationships → Graph (Neo4j)

At the advanced level the question is about trade-offs under constraints — and how the CAP theorem explains why NoSQL systems make the choices they do.

CAP Theorem Connection

The CAP theorem states that a distributed system can guarantee at most two of three properties simultaneously: Consistency (every read returns the most recent write), Availability (every request gets a response), and Partition tolerance (the system continues operating when network partitions occur). Since network partitions are inevitable in distributed systems, the real choice is CP vs AP.

Database CAP Choice Consistency Model Implication
PostgreSQL CA (single-node) Strong (serialisable) Cannot horizontally partition — scale with vertical sizing or read replicas
MongoDB CP (default) Tunable (strong to eventual) Majority reads/writes; minority nodes can serve stale data
Cassandra AP Eventual (tunable quorum) Always available; may return stale data; must handle conflicts
Redis CP (Cluster mode) Strong for single keys Cluster partitions by keyspace; cross-slot transactions limited
DynamoDB AP (default) Eventually consistent reads (cheap) or strongly consistent (2x cost) Choose per-request; pay for stronger guarantees

When "NoSQL is Faster" is False

A common misconception is that NoSQL is inherently faster than SQL. Speed depends entirely on the access pattern:

  • Redis is faster for single-key lookups. Always. SQL cannot compete with an in-memory hash map.
  • PostgreSQL is faster for complex multi-table aggregations. A Mongo aggregation pipeline over embedded documents is often slower than a well-indexed SQL query.
  • Cassandra is faster for high-volume time-series writes distributed across nodes. Postgres write throughput is bounded by a single primary node.
  • SQL is faster for ad-hoc reporting. Running an analytics query against MongoDB requires reading entire documents to extract one field. SQL column-oriented storage or even PostgreSQL is far more efficient.
Architect Note

The root cause of most bad database choices is designing for data storage instead of access patterns. Before choosing a database, answer: what are the three most frequent read queries? What are the most frequent write operations? What are the latency and throughput requirements for each? The database that serves those access patterns best at the required scale is the right choice — regardless of which family it belongs to.

At the expert level the questions become: how do you manage a polyglot persistence architecture in production, how do you migrate between stores, and how do compliance requirements constrain database choice?

Polyglot Persistence: Operational Reality

Polyglot persistence — using multiple database types in one system — solves performance problems but creates operational complexity. The real costs are:

  • Cross-database transactions — You cannot atomically write to Postgres and Cassandra in one transaction. Solutions: Saga pattern (compensating transactions), outbox pattern (write to one DB, fan out via event stream), or accept eventual consistency.
  • Data duplication — The same entity may live in Postgres (authoritative), Redis (cached), and Elasticsearch (searchable). Keeping them in sync requires event-driven architecture or change data capture (CDC).
  • Operational overhead — Each database type has a different backup strategy, scaling model, monitoring approach, and failure mode. Your team must be competent across all of them.
  • Debugging complexity — A "wrong data" bug could be a Postgres constraint failure, a cache invalidation race, a Cassandra eventual-consistency window, or an Elasticsearch reindex lag. Tracing it requires knowing all systems.
# Outbox Pattern: atomically commit to Postgres + publish event
# (solves cross-database consistency without distributed transactions)

Migration Patterns: SQL → NoSQL (and back)

Migrating a production database without downtime requires a careful multi-phase approach:

# Phase 1: Dual-write (write to both, read from old)
# Zero risk — old system is authoritative, new system catches up
deploy_dual_write_code
run_backfill_job --source postgres --target cassandra --table activity_logs

# Phase 2: Dark read (compare results between old and new)
# Validate correctness before switching reads
deploy_shadow_read_code     # reads both, logs discrepancies, returns old result
monitor_discrepancy_rate    # should converge to 0% over 24–48h

# Phase 3: Read from new (canary → ramp)
flip_feature_flag --read_from=cassandra --percentage=5
monitor_error_rate --alert_threshold=0.1%
flip_feature_flag --read_from=cassandra --percentage=100

# Phase 4: Remove old writes
deploy_single_write_code    # writes only to Cassandra
schedule_postgres_table_cleanup --after=90_days

GDPR and Compliance Across Database Types

GDPR's "right to be forgotten" requires deleting a user's personal data across every store it appears in. Each database handles this differently:

Database Deletion Mechanism Compliance Challenge
PostgreSQL DELETE WHERE user_id = :id Straightforward. Cascades via FK constraints if configured.
MongoDB deleteMany({ user_id: id }) Must find all collections where user data was embedded — easy to miss one.
Redis DEL key, SCAN + DEL pattern If user data is encoded in cached keys, must enumerate all patterns.
Cassandra DELETE (tombstone) Tombstones are not physical deletes — data persists until compaction. Compaction can take days. Encryption + key deletion is safer.
Elasticsearch delete by query Deletes are eventually consistent — re-indexed segments may still contain data briefly.
Expert Pattern

The cleanest GDPR compliance pattern for polyglot systems: use a user data service as the single source of truth for PII. All other databases reference a user_id only — no PII stored. When a deletion request arrives, the user data service deletes the record and publishes a user.deleted event. Downstream services tombstone or anonymise their records. Cassandra rows with only a user_id reference contain no PII and do not need deletion. Encryption per-user with key deletion (crypto-erasure) is an alternative that works well for Cassandra's tombstone latency problem.

Database Selection at Scale — Decision Checklist

  • Access patterns documented: primary read queries, primary write operations, QPS estimates
  • Consistency requirements assessed: does wrong data cause money loss or just UX degradation?
  • Schema stability assessed: will the data model change frequently?
  • Scale requirements: data volume (GB/TB/PB), throughput (ops/sec), latency SLO (p99)
  • GDPR/compliance requirements reviewed for all stored PII fields
  • Team operational competency confirmed for the chosen database type
  • Managed vs self-hosted decision made (AWS RDS vs self-managed Postgres, Atlas vs self-hosted Mongo)
  • Cross-database consistency strategy defined (saga/outbox/eventual) if polyglot

Enterprise Use Case

Enterprise Use Case Airbnb — Polyglot Persistence at 150M+ Users

Context: Airbnb's platform handles 150M+ active users, millions of listings, real-time availability calendars, billions of activity events, and billions of dollars in transactions annually. No single database family could serve all these workloads efficiently.

How they use each database:

  1. PostgreSQL — Core booking and payment data. ACID transactions are mandatory: a booking must simultaneously reserve the listing, charge the guest, and update the host's payout ledger. Postgres's serialisable isolation prevents double-bookings. Data size: hundreds of GB, but not TB — correctness over scale.
  2. MySQL — User profiles, host settings, listing metadata. Strong consistency required (a host's payout settings must be read correctly), but access patterns are simpler than bookings. MySQL's battle-tested replication is well-understood by the team.
  3. Cassandra — Messaging inbox and activity feed. Hosts and guests send millions of messages per day. Airbnb's access pattern is: "fetch the last 50 messages for conversation X" — a perfect Cassandra time-series query. Write throughput vastly exceeds what a single Postgres primary can handle. Eventual consistency is acceptable: a message appearing 100ms late is fine.
  4. Redis — Session tokens, search result caching, rate limiting, and the "availability calendar" cache. A listing availability check fires on every search result — with millions of concurrent searches, even a 5ms Postgres query per listing would collapse the system. Redis returns the cached calendar in 0.3ms.
  5. Elasticsearch — Full-text and geo-spatial search ("find entire homes in Paris under $150/night, pet-friendly, with free cancellation"). Postgres full-text search cannot handle this at Airbnb's query volume. Elasticsearch indexes are rebuilt nightly from Postgres via a Kafka-based CDC pipeline.

Business Impact: Before the polyglot architecture, a single Cassandra incident or Redis outage could cascade into booking failures because all systems shared one overloaded database tier. After separation: a Redis outage means slower search (falls back to Postgres, higher latency) but bookings continue. A Cassandra hiccup means delayed messages — no revenue impact. Independence between data tiers is as important as the per-tier performance gain.

Key lesson: Airbnb did not choose Cassandra because it is "NoSQL and modern." They chose it because their messaging workload is literally a time-series append log — the exact problem wide-column stores are designed for. The data model drove the decision, not the technology label.

Common Mistakes

"NoSQL is Faster Than SQL"

This is the most pervasive misconception in database selection. Speed is entirely workload-dependent. Redis is faster than Postgres for a single-key hash lookup — by two orders of magnitude. Postgres is faster than MongoDB for a multi-collection aggregation query. Cassandra is faster than Postgres for sustained million-row-per-second time-series writes across a distributed cluster. Postgres is faster than Cassandra for ad-hoc analytical queries.

The correct framing: which database is fastest for the specific access pattern at the specific scale I need? Choosing MongoDB because "NoSQL scales better" for an app that primarily runs complex reporting queries will result in worse performance and a painful migration.

Choosing MongoDB Because "JSON is Natural"

Teams familiar with JavaScript or Python often choose MongoDB because storing JSON documents "feels natural" compared to SQL tables. This reasoning ignores access patterns entirely. If your application frequently queries data across multiple entity types (e.g., "all orders for users in a specific region who signed up after a date"), the document model forces either expensive $lookup aggregations (essentially a slow JOIN) or massive data duplication. Many production MongoDB clusters were migrated back to Postgres precisely because the team discovered, after going live, that their query patterns were inherently relational.

The right question is not "what format is my data in?" but "how do I most often need to query it?"

Ignoring ACID Requirements for Financial Data

Building payment or inventory systems on an eventually-consistent NoSQL store is a serious architectural error. Cassandra's eventual consistency means two writes to the same balance can arrive at different replicas in different orders — your account balance becomes undefined. Redis (without Redis Cluster transactions and careful scripting) can lose writes during failover. MongoDB's multi-document transactions (introduced in v4.0) work but add significant performance overhead and are often disabled in high-throughput configurations.

Any operation that must be atomic — deducting stock and creating a reservation, debiting one account and crediting another, creating an order and capturing payment — belongs in a database with full ACID transaction support. Use Postgres or MySQL. Do not fight the tool.

Interview Prep

SQL gives you ACID transactions, complex JOINs, and strict schemas — ideal for relational data where correctness is paramount. NoSQL gives you scale, flexibility, and access-pattern-optimised data models — ideal when you've outgrown a single machine or need specialised performance characteristics.

2-Minute Structured Answer

"SQL and NoSQL serve different access patterns. SQL — Postgres, MySQL — is the right choice when you need ACID transactions, like a payment system where you can't afford inconsistency, or when you have complex query requirements like analytics across many tables. The schema enforcement also prevents data quality problems. NoSQL covers four families: document stores like MongoDB for flexible-schema entities fetched as a whole; key-value stores like Redis for O(1) lookups at microsecond latency; wide-column stores like Cassandra for write-heavy time-series data distributed across nodes; and graph databases like Neo4j when relationships are the primary query axis. In practice, large systems use both — this is called polyglot persistence. The decision should be driven by access patterns, consistency requirements, and scale — not by familiarity or hype."

Common Interview Mistakes

  • Saying "NoSQL scales better" without qualifying what type of workload or what kind of scale — interviewers will immediately probe this.
  • Treating this as SQL vs. "one NoSQL" — there are four distinct NoSQL families with completely different models; conflating them signals shallow knowledge.
  • Forgetting to mention consistency trade-offs when recommending a NoSQL store for financial data.
  • Not connecting the choice to access patterns — the answer "I'd use MongoDB for user profiles" is weak without explaining the access pattern that makes documents better than tables for that case.
