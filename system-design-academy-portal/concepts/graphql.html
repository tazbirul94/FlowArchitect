GraphQL

A query language for APIs that lets clients request exactly the data they need — nothing more, nothing less — through a single, self-documenting endpoint.

Intermediate APIs Query Language Schema-First

Definition & Analogy

Simple Analogy

REST is like ordering from a set menu — you get what the kitchen decided to put on the plate. GraphQL is like ordering à la carte — you specify exactly which items you want, in what combination, and they arrive on one tray instead of five separate trips from the waiter.

GraphQL is a query language for your API and a server-side runtime for executing those queries. Developed at Facebook in 2012 and open-sourced in 2015, it solves two chronic problems with REST: over-fetching (getting back fields you don't need, wasting bandwidth) and under-fetching (needing to call three endpoints to get the data for one screen).

With GraphQL, the client describes its data requirements in a query document. The server validates the query against a strongly-typed schema and executes it, returning exactly the requested shape. One network round-trip. No wasted bytes. No waterfall of sequential HTTP calls.

The Story

Real-World Story
Facebook's mobile engineering team in 2012 was rebuilding the iOS app. The existing REST API was designed around the web experience — it returned massive JSON objects built for desktop screens. On a 3G connection, every news-feed page load required 6–8 separate REST calls, and each response contained dozens of fields the mobile app never rendered.
A single news-feed refresh caused: GET /feed returning 2 KB of user profile data the app already had, then GET /users/{id}/posts returning posts with 40 fields when 5 were needed, then GET /posts/{id}/comments called once per post — 20 posts meant 20 additional requests. On 3G, this cascade took 8–12 seconds. Battery drain from network keep-alive chewed through 15% of charge per hour of scrolling.
The team designed GraphQL: a single endpoint where the client sends a query describing exactly what it wants. The news-feed screen sends one query asking for feed items with only the fields it displays. The server resolves all sub-queries internally (using its own fast internal network) and returns one response. The mobile client receives exactly the shape it needs — nothing extra, nothing missing.
The Facebook iOS app feed load time dropped from 8 seconds to under 1.5 seconds on 3G. Network payload per feed refresh dropped 70%. The API became self-documenting — mobile engineers could introspect the schema to see all available fields without reading docs or emailing the backend team. Facebook open-sourced GraphQL in 2015. GitHub adopted it for their v4 API. Today 100M+ developers interact with GitHub's GraphQL API daily.

Learn at Your Level

Beginner

Imagine you're building a mobile app that shows a user's profile card: their name, avatar, and number of followers. With REST, you'd call GET /users/42 and get back a 50-field JSON object — birth date, email, settings, preferences — even though you only need three fields. That's over-fetching.

Now imagine you also need their latest three posts. With REST you'd make a second call to GET /users/42/posts?limit=3. That's two round-trips for one screen. That's under-fetching.

With GraphQL, you write one query that says "give me the user's name, avatar URL, follower count, and their last 3 posts' titles." The server returns exactly that shape — in one round-trip, with no extra fields.

The Three Operation Types

  • Query — Read data. Like a GET request. Fetches information from the server.
  • Mutation — Write data. Like POST/PUT/DELETE. Creates or changes something on the server.
  • Subscription — Real-time updates. The server pushes changes to you over a WebSocket connection (e.g., live chat messages, stock prices).

Your First GraphQL Query

# Ask for exactly the fields you need
query GetUserProfile {
  user(id: "42") {
    name
    avatarUrl
    followerCount
    posts(limit: 3) {
      title
      publishedAt
    }
  }
}

The server responds with a JSON object whose shape exactly mirrors the query structure — no surprise fields, no missing data:

{
  "data": {
    "user": {
      "name": "Alice",
      "avatarUrl": "https://cdn.example.com/alice.jpg",
      "followerCount": 1284,
      "posts": [
        { "title": "GraphQL is great", "publishedAt": "2025-05-01" },
        { "title": "REST vs GraphQL", "publishedAt": "2025-04-20" },
        { "title": "My first API", "publishedAt": "2025-03-14" }
      ]
    }
  }
}

Schema Definition Language (SDL)

Every GraphQL API starts with a schema — a strongly-typed contract that defines every type, field, and operation available. The schema is the single source of truth for both client and server.

# Schema Definition Language (SDL)

type User {
  id: ID!           # ID! means non-null — this field is always present
  name: String!
  email: String!
  avatarUrl: String
  followerCount: Int!
  posts: [Post!]!   # List of Post, list and items are both non-null
  createdAt: String!
}

type Post {
  id: ID!
  title: String!
  body: String!
  author: User!
  comments: [Comment!]!
  publishedAt: String
  tags: [String!]!
}

type Comment {
  id: ID!
  text: String!
  author: User!
  createdAt: String!
}

# Root operation types
type Query {
  user(id: ID!): User           # Nullable — returns null if not found
  users(limit: Int, offset: Int): [User!]!
  post(id: ID!): Post
  posts(authorId: ID, limit: Int): [Post!]!
}

type Mutation {
  createPost(input: CreatePostInput!): Post!
  updatePost(id: ID!, input: UpdatePostInput!): Post!
  deletePost(id: ID!): Boolean!
}

type Subscription {
  commentAdded(postId: ID!): Comment!
}

input CreatePostInput {
  title: String!
  body: String!
  tags: [String!]
}

input UpdatePostInput {
  title: String
  body: String
  tags: [String!]
}

Queries, Mutations, and Subscriptions

# QUERY — read data
query GetPostWithComments($postId: ID!) {
  post(id: $postId) {
    title
    body
    author {
      name
      avatarUrl
    }
    comments {
      text
      author {
        name
      }
      createdAt
    }
  }
}

# MUTATION — write data
mutation CreateNewPost($input: CreatePostInput!) {
  createPost(input: $input) {
    id
    title
    publishedAt
  }
}

# SUBSCRIPTION — real-time push
subscription WatchComments($postId: ID!) {
  commentAdded(postId: $postId) {
    id
    text
    author {
      name
    }
    createdAt
  }
}

Fragments — Reusable Field Selections

Fragments let you define a named set of fields once and reuse them across multiple queries. This avoids duplication and keeps queries maintainable.

# Define a fragment
fragment UserBasic on User {
  id
  name
  avatarUrl
}

# Reuse it in two different queries
query GetPost($id: ID!) {
  post(id: $id) {
    title
    author {
      ...UserBasic    # Spread the fragment
    }
    comments {
      text
      author {
        ...UserBasic  # Same fragment, different context
      }
    }
  }
}

Resolver Chain — How Fields Get Their Values

On the server, each field in the schema maps to a resolver function. When a query is executed, GraphQL walks the query tree and calls each resolver to get the value for each field. Resolvers can independently fetch data from databases, caches, or other services.

// Apollo Server resolver map (Node.js)
const resolvers = {
  Query: {
    // Top-level resolvers receive: (parent, args, context, info)
    user: async (_, { id }, { dataSources }) => {
      return dataSources.usersDB.findById(id);
    },
    post: async (_, { id }, { dataSources }) => {
      return dataSources.postsDB.findById(id);
    },
  },

  // Type-level resolvers — called when a User field is requested
  User: {
    // 'parent' is the User object already fetched
    posts: async (parent, { limit = 10 }, { dataSources }) => {
      return dataSources.postsDB.findByAuthorId(parent.id, { limit });
    },
    followerCount: async (parent, _, { dataSources }) => {
      return dataSources.followsDB.countFollowers(parent.id);
    },
  },

  Post: {
    author: async (parent, _, { dataSources }) => {
      return dataSources.usersDB.findById(parent.authorId);
    },
    comments: async (parent, _, { dataSources }) => {
      return dataSources.commentsDB.findByPostId(parent.id);
    },
  },
};

Introspection — The Self-Documenting API

GraphQL APIs are introspective. You can query the schema itself to discover all available types, fields, and operations. This powers tools like GraphiQL and Postman's GraphQL support.

# Ask the API what types are available
query IntrospectSchema {
  __schema {
    types {
      name
      kind
      fields {
        name
        type {
          name
          kind
        }
      }
    }
  }
}

# Ask about a specific type
query InspectUser {
  __type(name: "User") {
    name
    fields {
      name
      type {
        name
        kind
        ofType {
          name
          kind
        }
      }
    }
  }
}

The N+1 Problem

The most critical performance pitfall in GraphQL. When you query a list of items and each item has a nested resolver, you get one database query per item instead of one query total.

# This innocent-looking query causes N+1 in a naive implementation
query GetAllPosts {
  posts(limit: 100) {   # 1 DB query for posts
    title
    author {            # 100 separate DB queries, one per post!
      name
    }
  }
}

The naive resolver calls usersDB.findById(post.authorId) for each of the 100 posts — 101 database round-trips for what should be 2. At 100 posts this is noticeable; at 1000 posts it brings down your database.

DataLoader — The Solution

DataLoader is a utility (originally from Facebook) that batches and caches data-loading. Instead of executing one DB call per item, DataLoader collects all IDs requested in the same event-loop tick and issues a single batched query.

import DataLoader from 'dataloader';

// One batching function — called with an array of IDs collected in one tick
const userLoader = new DataLoader(async (userIds) => {
  // Single DB query for all requested user IDs at once
  const users = await db.query(
    'SELECT * FROM users WHERE id = ANY($1)',
    [userIds]
  );
  // Return results in the same order as the input IDs
  const userMap = new Map(users.map(u => [u.id, u]));
  return userIds.map(id => userMap.get(id) || null);
});

// Resolver — uses loader instead of direct DB call
const resolvers = {
  Post: {
    author: (post, _, { loaders }) => {
      // loader.load() schedules this ID — DataLoader batches all
      // calls made in the same event-loop tick into ONE DB query
      return loaders.user.load(post.authorId);
    },
  },
};

// Result: 100 posts → 1 query for posts + 1 batched query for all authors
// Instead of: 1 query for posts + 100 queries for authors

Query Depth Limiting

Without protection, a malicious (or careless) client can send an arbitrarily deep query that triggers an exponential number of resolver calls — a denial-of-service through the API itself.

# This query nests deeply and could resolve millions of records
query MaliciousDeepQuery {
  users {
    posts {
      comments {
        author {
          posts {
            comments {
              author {
                posts {           # Each level multiplies resolver calls
                  title
                }
              }
            }
          }
        }
      }
    }
  }
}
import depthLimit from 'graphql-depth-limit';
import costAnalysis from 'graphql-cost-analysis';

const server = new ApolloServer({
  typeDefs,
  resolvers,
  validationRules: [
    depthLimit(6),               // Reject queries deeper than 6 levels
    costAnalysis({
      maximumCost: 1000,         // Each field has a cost; reject if total > 1000
      defaultCost: 1,
      scalarCost: 1,
      objectCost: 2,
      listFactor: 10,            // Lists multiply cost by 10
    }),
  ],
});
Architecture Note

Depth limiting alone is not sufficient. A query 3 levels deep that requests a list of 10,000 users, each with 50 posts, is 500,000 resolver calls. Combine depth limiting with query cost analysis and field-level rate limiting for production security. Also disable introspection in production — it gives attackers a roadmap of your entire data model.

Persisted Queries

Standard GraphQL POSTs the full query string on every request. Persisted queries register a query string on the server (or CDN) ahead of time and send only a hash at runtime. This provides three benefits: smaller request payloads, CDN cacheability (GET with hash), and security (only pre-approved queries can execute).

// Apollo Client — automatic persisted queries (APQ)
import { ApolloClient, InMemoryCache, createHttpLink } from '@apollo/client';
import { createPersistedQueryLink } from '@apollo/client/link/persisted-queries';
import { sha256 } from 'crypto-hash';

const persistedQueryLink = createPersistedQueryLink({ sha256 });
const httpLink = createHttpLink({ uri: '/graphql' });

const client = new ApolloClient({
  link: persistedQueryLink.concat(httpLink),
  cache: new InMemoryCache(),
});

// First request: sends hash + full query (server stores hash → query mapping)
// Subsequent requests: sends ONLY the hash — full query never transmitted again
// GET /graphql?extensions={"persistedQuery":{"version":1,"sha256Hash":"abc123"}}
// Now CDNs can cache GET requests normally!

Schema Federation with Apollo Router

At microservices scale, maintaining one monolithic GraphQL schema is impractical. Apollo Federation lets each service own its portion of the schema. The Apollo Router stitches them into a unified supergraph, routing sub-queries to the correct service automatically.

# user-service schema (subgraph)
type User @key(fields: "id") {
  id: ID!
  name: String!
  email: String!
}

type Query {
  user(id: ID!): User
  me: User
}

# post-service schema (subgraph) — extends User from user-service
type Post @key(fields: "id") {
  id: ID!
  title: String!
  authorId: ID!
  # Federated reference: resolves User across service boundary
  author: User
}

extend type User @key(fields: "id") {
  id: ID! @external       # Tells router this field comes from user-service
  posts: [Post!]!         # This service contributes posts to the User type
}

type Query {
  posts: [Post!]!
  post(id: ID!): Post
}
# Apollo Router supergraph config
federation_version: =2.6.0

subgraphs:
  users:
    routing_url: http://user-service:4001/graphql
    schema:
      file: ./schemas/users.graphql
  posts:
    routing_url: http://post-service:4002/graphql
    schema:
      file: ./schemas/posts.graphql
  comments:
    routing_url: http://comment-service:4003/graphql
    schema:
      file: ./schemas/comments.graphql

Compliance and Security Posture

Production Security Checklist Enterprise GraphQL Hardening

Introspection: Disable in production (introspection: false in ApolloServer). Introspection gives attackers a complete map of your data model. Use Schema Registry (Apollo Studio) for authorized developers to explore the schema.

Query depth limiting: Set maximum depth to 6–10 levels depending on your data model. Use graphql-depth-limit package. Test with adversarial queries before going live.

Query complexity analysis: Assign cost points to fields (lists cost more). Reject queries over a total cost budget. This prevents the "many shallow resolvers" variant of DoS that depth limits miss.

Field-level rate limiting: Expensive computed fields (aggregations, ML inference) need per-field rate limits. Standard rate limiting on the endpoint level is insufficient — a client can call one cheap endpoint frequently vs one expensive field once.

Authorization in resolvers: Never rely on the schema alone to protect data. Authorize at the resolver level. Use a pattern like @auth directives or middleware. Row-level security: filter data by the authenticated user's permissions inside each resolver, not at the query level.

GDPR/Data residency: Federation routing rules can enforce that queries requesting EU user data are routed only to EU-region subgraph instances. Apollo Router's traffic shaping and coprocessor hooks enable compliance enforcement at the gateway layer.

Enterprise Use Case

Enterprise Use Case GitHub — REST v3 to GraphQL v4 Migration

Context: GitHub has 100M+ developers using its API. The REST v3 API served over 1 billion API requests per day. Mobile clients and third-party integrations commonly needed to combine data from /repos, /issues, /pulls, /commits, and /users endpoints — 5 round-trips to render a single pull request review screen.

Why GraphQL: GitHub's REST API returned enormous response objects (a repository object had 100+ fields). Clients would request an entire repository object just to get the defaultBranch name — over-fetching on every call. Additionally, paginating through nested resources (commits in a PR, reviews in a commit) required understanding GitHub's custom Link header pagination.

What changed with v4: A single GraphQL endpoint at api.github.com/graphql. Clients request exactly the fields they need — a repository listing for a dashboard fetches 4 fields instead of 100. Cursor-based Relay-style pagination is consistent across all list types. The schema is introspectable — IDE plugins auto-complete GitHub API fields as you type.

Concrete numbers:

  • Average payload size per API call dropped by ~60% for typical integrations
  • GitHub's own mobile app reduced its network calls per screen by 3–5x
  • The schema has grown to 250+ types and 1,000+ fields — all discoverable via introspection
  • 100M+ developers interact with the GraphQL API daily through CI/CD integrations, IDE extensions, and third-party tooling

Lessons learned:

  • Schema design outlasts implementation: Once developers adopt your schema, breaking changes are painful. GitHub has kept backward compatibility for years through field deprecation rather than removal.
  • Rate limiting had to change: REST rate limits were per-endpoint (5,000 requests/hour). GraphQL needed cost-based rate limits — one complex query costs more than one simple query. GitHub implemented a point-based system where each field access costs 1 point and nodes cost 1 point, with a cap of 5,000 points/hour.
  • Persisted queries were essential: Without them, every API call sent the full query string, inflating payload size and preventing CDN caching for safe queries.

Common Mistakes

No Query Depth Limiting (DoS via Deep Nesting)

GraphQL allows clients to request arbitrarily nested data. Without depth or complexity limits, a query like { users { posts { comments { author { posts { comments { ... } } } } } } } can trigger an exponential chain of resolver calls that saturates your database in seconds. This is not theoretical — GraphQL endpoints without limits have been taken offline by a single malformed query from a developer debugging locally.

Fix: Install graphql-depth-limit and set depthLimit(6) as a validation rule. Also add graphql-cost-analysis for budget-based complexity limits. Reject queries over the limit before any resolvers execute.

Missing DataLoader (N+1 Queries Kill Production)

The most common performance bug in GraphQL servers. A query for 100 posts where each post's author field fires a separate DB query results in 101 database calls instead of 2. At low traffic this is invisible; at production load (1,000 concurrent users each triggering 100 resolver calls) it fires 100,000 DB queries per second and crashes your database. The insidious part: it works perfectly in development (where you test with 5 posts) and fails catastrophically in production.

Fix: Always use DataLoader for any resolver that fetches by a foreign key. Install dataloader, create one loader per data source per request context, and use loader.load(id) instead of direct DB calls. DataLoader batches all loads in the same event-loop tick into one query.

Introspection Enabled in Production

Introspection is invaluable during development — it powers GraphiQL, Apollo Studio, and IDE auto-complete. In production, it's a security liability. Introspection gives any unauthenticated caller a complete map of every type, field, resolver argument, and relationship in your data model. Attackers use this to identify sensitive fields (internalAdminFlag, creditCardLast4) and craft targeted queries. It's like leaving your API documentation inside the API itself with no authentication.

Fix: Set introspection: false in your production ApolloServer config. Use Apollo Schema Registry or a private GraphiQL instance behind authentication for developer exploration. Consider allowing introspection only for authenticated requests with an INTROSPECTION role.

Interview Prep

GraphQL is a query language and runtime where the client declares exactly what data it needs, the server validates the request against a typed schema, and returns precisely that shape — eliminating over-fetching and under-fetching from a single endpoint.

2-Minute Structured Answer

"GraphQL solves two problems REST has at scale: over-fetching, where endpoints return more data than the client needs, and under-fetching, where clients need multiple round-trips to assemble one screen's worth of data. With GraphQL, you define a schema — a typed contract — and clients query exactly the fields they want in one request. The server resolves each field independently using resolver functions, which can fetch from databases, caches, or other services. The major trade-off is HTTP caching: since all requests are POSTed to one endpoint, standard CDN caching breaks — you need persisted queries to restore cacheability. The other critical pitfall is the N+1 problem: naive resolvers fire one DB query per item in a list. Facebook created DataLoader to fix this by batching all ID lookups within the same event-loop tick into a single query. GitHub migrated their entire public API to GraphQL v4 and cut payload size 60% while eliminating 5-call waterfalls for mobile screens."

Common Interview Mistakes

  • Saying "GraphQL is always better than REST" — interviewers want nuance. GraphQL adds complexity and breaks HTTP caching; REST is simpler for CRUD-heavy APIs with simple clients.
  • Forgetting to mention the N+1 problem — this is the most common GraphQL production failure and interviewers expect you to know it. If you know GraphQL, you know DataLoader.
  • Confusing subscriptions with polling — subscriptions use WebSockets (persistent connection, server pushes). Polling is the client repeatedly querying. They solve the same use case differently.
  • Claiming GraphQL eliminates versioning entirely — field deprecation and schema evolution reduce versioning needs, but breaking schema changes still require coordination and migration periods.
