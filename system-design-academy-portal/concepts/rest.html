REST

The architectural style that powers the web — how applications communicate over HTTP using stateless, resource-based requests.

All Levels APIs HTTP Web

Definition & Analogy

Simple Analogy

A waiter at a restaurant takes your order (request), walks to the kitchen (server), retrieves your food (resource), and brings it back — without needing to remember who you are between visits. Each interaction is self-contained. REST APIs work exactly the same way: every request carries all the information the server needs, and the server returns a resource without storing any session state.

REST (Representational State Transfer) is an architectural style for designing networked applications. It was defined by Roy Fielding in his 2000 doctoral dissertation and has since become the dominant pattern for building web APIs. REST is not a protocol or standard — it is a set of constraints that, when followed, produce scalable, maintainable, and interoperable systems.

The six REST constraints are: client-server separation, statelessness, cacheability, uniform interface, layered system, and code-on-demand (optional). In practice, the most impactful constraints are statelessness and uniform interface — these are what most engineers mean when they say "RESTful API."

The Story

Real-World Story
A fintech startup was building a mobile banking app. The backend team had built a working internal API, but it used custom HTTP calls like GET /getAccountBalance?accountId=123, POST /doTransfer, and GET /deleteTransaction?id=456. Every mobile client, web app, and third-party integration had to learn bespoke conventions.
When they onboarded a new mobile contractor, it took two weeks just to understand the API. The GET /deleteTransaction endpoint caused a production incident when a caching layer cached the "delete" call and replayed it. Third-party integrators complained the API was unpredictable. Developers couldn't use standard HTTP tooling because nothing followed convention.
The team redesigned the API following REST conventions. Accounts became /v1/accounts/{id}. Transactions were /v1/accounts/{id}/transactions. They used GET for reads, POST for creates, PUT for full updates, PATCH for partial updates, and DELETE for removals. Status codes became meaningful: 201 for created, 404 for not found, 409 for conflicts.
New developers were productive in one day instead of two weeks. Standard HTTP clients, caching proxies, and API gateways worked out of the box. The accidental replay incident became impossible because GET requests are now safe (no side effects) and DELETE requests are idempotent. Third-party integrations doubled in six months. The API documentation wrote itself because the conventions were predictable.

Learn at Your Level

Beginner

Think about how you use a website. You type a URL, press enter, and the page loads. Behind the scenes, your browser sent an HTTP request to a server, and the server responded with HTML. REST APIs work the same way — but instead of HTML pages, they return data (usually JSON), and instead of a browser, it might be your mobile app or another service calling the API.

REST has four core operations that map to everyday actions:

  • GET — Read something. Like looking up a contact in your phone.
  • POST — Create something new. Like adding a new contact.
  • PUT / PATCH — Update something. Like editing a contact's phone number.
  • DELETE — Remove something. Like deleting a contact.

These operations work on resources — things your API manages, like users, orders, or products. Each resource has a URL (called an endpoint), like /users/42 or /orders/789.

Key Status Codes to Know

  • 200 OK — Request succeeded. For GET, PUT, PATCH.
  • 201 Created — New resource created. For POST.
  • 204 No Content — Success, nothing to return. For DELETE.
  • 400 Bad Request — Client sent invalid data.
  • 401 Unauthorized — Not authenticated (no valid token).
  • 403 Forbidden — Authenticated but not allowed.
  • 404 Not Found — Resource doesn't exist.
  • 500 Internal Server Error — Server crashed.

At the intermediate level, REST is about designing URLs and using HTTP correctly. Good API design follows predictable conventions that let developers reason about your API before reading documentation.

Resource Design Rules

  • Use nouns, not verbs — /users not /getUsers; /orders not /createOrder. The HTTP method IS the verb.
  • Use plural nouns — /users not /user; /products not /product.
  • Nest related resources — /users/42/orders for orders belonging to user 42. But keep nesting shallow (max 2 levels).
  • Lowercase, hyphens for multi-word — /ticket-types not /ticketTypes or /TicketTypes.

HTTP Methods and Idempotency

Method Use For Idempotent? Safe?
GET Read a resource or list Yes Yes (no side effects)
POST Create a new resource No No
PUT Full replace of a resource Yes No
PATCH Partial update of a resource Not always No
DELETE Remove a resource Yes No

Idempotent means calling the operation multiple times produces the same result as calling it once. DELETE /users/42 called twice should return 204 or 404 — not delete two different users. Safe means the operation has no side effects; GET requests must never modify state.

Pagination Patterns

Never return all records for a large collection. Two common strategies:

Offset-based pagination — simple but has gaps with concurrent writes:

GET /users?page=2&per_page=20
GET /users?offset=40&limit=20

Response:
{
  "data": [...],
  "meta": {
    "total": 1543,
    "page": 2,
    "per_page": 20,
    "total_pages": 78
  }
}

Cursor-based pagination — stable with concurrent writes, preferred for real-time feeds:

GET /users?cursor=eyJpZCI6NDB9&limit=20

Response:
{
  "data": [...],
  "meta": {
    "next_cursor": "eyJpZCI6NjB9",
    "has_more": true
  }
}

API Versioning Strategies

Your API will change. Versioning prevents breaking existing clients:

  • URL path versioning (most common) — /v1/users, /v2/users. Simple, visible, cacheable. GitHub uses this.
  • Header versioning — Accept: application/vnd.myapi.v2+json. Cleaner URLs but harder to test in a browser.
  • Query param versioning — /users?version=2. Easy but pollutes query strings.
# URL versioning — recommended for public APIs
GET https://api.github.com/v3/repos/octocat/Hello-World

# Accept header versioning
GET https://api.example.com/users/42
Accept: application/vnd.example.v2+json

Complete Status Code Reference

  • 200 OK — Successful GET, PUT, PATCH.
  • 201 Created — POST succeeded, include Location header pointing to new resource.
  • 204 No Content — DELETE or PATCH with no response body needed.
  • 400 Bad Request — Malformed JSON, missing required fields, invalid values.
  • 401 Unauthorized — Token missing or expired. Client must authenticate.
  • 403 Forbidden — Authenticated but lacks permission. Don't reveal resource existence to unauthorized users.
  • 404 Not Found — Resource doesn't exist at this URL.
  • 409 Conflict — State conflict — e.g. creating a user with a duplicate email.
  • 422 Unprocessable Entity — Valid JSON but semantically invalid (e.g. end date before start date).
  • 429 Too Many Requests — Rate limit exceeded. Include Retry-After header.
  • 500 Internal Server Error — Unexpected server failure.

At the advanced level, REST design is about consistency at scale, trade-offs against alternatives, and operational concerns like rate limiting, caching headers, and idempotency keys.

REST vs GraphQL vs gRPC Trade-Offs

Dimension REST GraphQL gRPC
Over-fetching Common — fixed response shapes return unused fields Eliminated — client requests exact fields Minimal — typed contract defines exact payload
Under-fetching Requires multiple round trips for related resources Single query fetches nested data Single RPC call returns structured data
Cacheability HTTP cache headers work natively (CDN, browser) POST-based queries bypass HTTP caching Not HTTP-cacheable by default
Tooling & ecosystem Curl, Postman, browsers, OpenAPI — universal Good but less universal Requires gRPC clients, proto toolchain
Performance Good (HTTP/1.1 or HTTP/2) Good Excellent — binary Protocol Buffers, HTTP/2
Learning curve Low — HTTP is universal knowledge Medium — schema, resolvers, N+1 problem High — proto files, code gen, streaming
Best for Public APIs, mobile, browser clients Data-heavy UIs, multiple client types Internal microservice-to-microservice calls

HTTP Caching Headers

REST's cacheability is a key advantage. Use it correctly:

# Immutable resources (e.g. CDN-hosted versioned assets)
Cache-Control: public, max-age=31536000, immutable

# Short-lived public resources (product listings)
Cache-Control: public, max-age=60, s-maxage=300

# Private user data (never cache in shared proxies)
Cache-Control: private, max-age=0, no-store

# Conditional GET — efficient revalidation
ETag: "abc123"
Last-Modified: Wed, 01 Jan 2025 00:00:00 GMT

# Client revalidation
If-None-Match: "abc123"
# Server responds 304 Not Modified if unchanged (0 bytes payload)

Idempotency Keys for Non-Idempotent Operations

POST requests are not idempotent — retrying a failed payment POST could charge a customer twice. Idempotency keys solve this:

POST /v1/payments
Idempotency-Key: 7a8f3b2e-1c4d-4e6f-a9b0-c2d3e4f5a6b7
Content-Type: application/json

{
  "amount": 5000,
  "currency": "usd",
  "source": "card_abc123"
}

# Server stores the result keyed by Idempotency-Key.
# If client retries with the same key within 24h, server
# returns the original stored response — no double charge.

HATEOAS — Hypermedia as the Engine of Application State

A fully RESTful API includes links in responses that tell clients what actions are available next. Few APIs implement this fully, but it's the purist definition:

{
  "id": 42,
  "status": "pending",
  "amount": 5000,
  "_links": {
    "self":    { "href": "/orders/42" },
    "cancel":  { "href": "/orders/42/cancellation", "method": "POST" },
    "payment": { "href": "/orders/42/payment",      "method": "POST" }
  }
}

When NOT to Use REST

  • Real-time bidirectional communication — REST is request-response. Use WebSockets or Server-Sent Events for live feeds (chat, notifications, price tickers).
  • High-performance internal microservice calls — gRPC's binary encoding and HTTP/2 multiplexing is significantly faster. REST overhead is measurable at 100K+ req/sec.
  • Complex, nested data fetching in SPAs — If every page requires 5 REST calls that could be one GraphQL query, the over/under-fetching problem is real.
  • Streaming large payloads — REST works but gRPC's native streaming is more ergonomic for server-streaming scenarios.
Architect Insight

REST is not a binary — it's a spectrum. Most real-world APIs are "REST-ish" rather than fully RESTful (HATEOAS is rarely implemented). That's fine. The practical value is in the predictable URL structure, correct HTTP method usage, meaningful status codes, and statelessness. Chasing full REST compliance often delivers diminishing returns.

At enterprise scale, REST API design intersects with governance, breaking change management, security posture, and platform economics.

Expert Considerations Enterprise REST API Platform

API governance at scale: When you have 200 microservices each exposing REST APIs, inconsistency compounds. Teams use different error formats, different pagination conventions, different versioning schemes. The solution is an API style guide enforced via linting (Spectral + OpenAPI) in CI. Every PR that touches an API spec runs the linter. Violations block merge. GitHub's API style guide is public and used as a reference by many orgs.

Breaking vs non-breaking changes: Adding fields to a response is non-breaking (clients ignore unknown fields). Removing fields, renaming fields, changing types, or changing URL structure are breaking changes requiring a version bump. Deprecation windows of 6–12 months are standard for public APIs. Internal APIs can be shorter — but communicate via Slack/email and give teams a sprint to migrate before removal.

Security at the API layer: REST APIs are the most common attack surface. Every endpoint needs authentication (OAuth 2.0 bearer tokens), authorization (RBAC/ABAC checks), rate limiting (429 with Retry-After), input validation (reject oversized payloads — default to 1MB max), and structured logging of every request with user ID, path, method, latency, and status. OWASP API Security Top 10 is the checklist.

Cost model: REST over HTTP/1.1 means one connection per request at high concurrency. HTTP/2 multiplexing reduces connection overhead. At 1M req/sec, the difference between an optimized REST API (HTTP/2, connection pooling, response compression) and a naive one is 40–60% infrastructure cost. gzip or brotli on JSON responses routinely reduces payload size by 80%, cutting bandwidth bills significantly at scale.

Multi-region REST APIs: Statelessness is a superpower here — any region can serve any request. Route 53 latency-based routing directs clients to the nearest region. Read operations can be served from any replica. Write operations must route to the primary region or use eventual-consistency patterns. The hardest problem is user-created resources: if a user registers in EU-WEST and then calls the US-EAST endpoint before replication completes, they get a 404. Solve with global databases (CockroachDB, Aurora Global), strong routing consistency (always route user X to region Y), or async replication with read-your-writes consistency tokens.

Production Operations Checklist

  • OpenAPI spec checked into version control and validated in CI (Spectral lint)
  • All endpoints authenticated — no accidental public routes
  • Rate limiting applied at API gateway level (not per-service)
  • Request size limits enforced (default 1MB, media uploads separate)
  • Structured logs include: user_id, path, method, status, latency_ms, request_id
  • Correlation/request IDs propagated across microservices (X-Request-ID header)
  • Deprecation warnings returned in headers for sunset endpoints
  • CORS policy explicitly configured (not wildcard * in production)
  • HTTP/2 enabled on load balancer (reduces connection overhead)
  • Response compression enabled (gzip/brotli — typically 70–80% JSON reduction)

Enterprise Use Case

Enterprise Use Case GitHub REST API — api.github.com

Context: GitHub's REST API has been public since 2009. It serves billions of requests per month across millions of developers, CI/CD pipelines, bots, and integrations. The base URL is https://api.github.com. As of 2024, they're on API version 3 (path-based), and have been carefully maintaining backward compatibility for 15+ years while adding features.

Resource design in practice: GitHub's URLs are textbook REST. Repositories are /repos/{owner}/{repo}. Pull requests are /repos/{owner}/{repo}/pulls/{pull_number}. Commits are /repos/{owner}/{repo}/commits. Every resource is a noun. Every nested resource follows the parent chain. Collections are plural. IDs are in the path.

# Get a repository
GET https://api.github.com/repos/octocat/Hello-World
Authorization: Bearer ghp_xxxxxxxxxxxx

# List open pull requests with cursor pagination
GET https://api.github.com/repos/octocat/Hello-World/pulls?state=open&per_page=30&page=2

# Create a new issue
POST https://api.github.com/repos/octocat/Hello-World/issues
Content-Type: application/json

{
  "title": "Found a bug",
  "body": "I'm having a problem with this.",
  "labels": ["bug"]
}
# Returns 201 Created with Location header

# Update an issue (partial update)
PATCH https://api.github.com/repos/octocat/Hello-World/issues/1
{
  "state": "closed"
}
# Returns 200 OK

Rate limiting: GitHub rate limits unauthenticated requests to 60/hour and authenticated requests to 5,000/hour. Enterprise Cloud gets 15,000/hour. Rate limit state is returned in every response via X-RateLimit-* headers. When exceeded, 403 (not 429 — a non-standard choice) with a Retry-After header.

Versioning strategy: GitHub uses a hybrid approach — the URL path has /v3/ as an implied convention (the base URL itself is the version), while individual endpoint behavior can be customized via Accept header media types for experimental features: Accept: application/vnd.github.v3.full+json.

Metrics (2024): The GitHub API processes over 2 billion requests per day. Average P50 latency is under 100ms. The API surface has over 400 endpoints. Breaking changes require 12-month deprecation windows. They use OpenAPI specs for documentation generation and SDK generation (official SDKs in Python, Go, .NET, Ruby, JS).

Lessons learned: GitHub's original API had inconsistencies they've had to live with for decades because they can't break existing clients. Their blog cites maintaining backward compatibility as the single hardest ongoing cost of running a public API. The lesson: design your URL structure and response shapes carefully up front — changing them later is enormously expensive at scale.

Common Mistakes

Using GET for Mutations

GET /deleteUser?id=42 or GET /sendEmail?userId=5 are dangerous anti-patterns. GET requests are assumed to be safe — browsers prefetch them, CDNs cache them, link crawlers follow them, and caching layers replay them. In 2010, a malicious image tag (<img src="https://example.com/deleteAccount">) became a widespread CSRF attack vector precisely because GET requests were used for destructive actions. Always use the appropriate HTTP method: DELETE to delete, POST to trigger actions with side effects.

Returning 200 OK for Errors

A common mistake is returning 200 OK with an error in the body: {"success": false, "error": "User not found"}. This breaks every tool in the HTTP ecosystem — monitoring systems that alert on 5xx rates, load balancers that remove unhealthy backends, retry logic that replays on 5xx, and CDN caching policies that cache 200s but not 4xx/5xx. Use correct status codes: 404 for not found, 400 for validation errors, 500 for server errors. Your monitoring, alerting, and client error handling will all work correctly without custom logic.

No API Versioning Strategy

Launching a public API without versioning is a time bomb. When you need to change a response structure, rename a field, or remove an endpoint, you'll break every client simultaneously. Retrofitting versioning into an existing API is painful — you can't easily add /v2/ in the path when existing clients have hardcoded api.example.com/users. Start with versioning from day one (/v1/ in the path). When you're ready to introduce breaking changes, add /v2/ and give clients a 6-12 month migration window. Deprecate with Sunset and Deprecation response headers so clients can detect and act on the signal automatically.

Interview Prep

REST is an architectural style for building APIs over HTTP using stateless, resource-based interactions where the HTTP method expresses the action (GET/POST/PUT/DELETE) and URL paths identify the resources.

2-Minute Structured Answer

"REST is an architectural style that Roy Fielding defined in 2000, built on top of HTTP. Its core idea is that your API exposes resources — like users, orders, or products — each with a stable URL, and clients interact with those resources using standard HTTP methods: GET to read, POST to create, PUT or PATCH to update, DELETE to remove. The key constraint is statelessness: every request must contain all the information the server needs — no session stored on the server. This makes REST APIs easy to scale horizontally because any server can handle any request. The main trade-off is over-fetching and under-fetching — you get a fixed response shape, which can mean multiple round trips or returning more data than needed. GraphQL solves that for data-heavy clients, and gRPC solves it for internal service-to-service calls where performance is critical. But REST's universal tooling and HTTP cacheability make it the right default for public-facing APIs."

Common Interview Mistakes

  • Confusing 401 and 403 — 401 means "not authenticated" (no valid credentials), 403 means "authenticated but not authorized." They have different client-side responses.
  • Saying PATCH is always idempotent — it's not. PATCH /counter {"operation": "increment"} is not idempotent. A PATCH that sets a field to a fixed value is. The operation defines idempotency, not the method.
  • Designing URLs with verbs — /createUser instead of POST /users. This is the most common REST design mistake in real codebases.
  • Not mentioning caching — REST's HTTP cacheability is a genuine architectural advantage over GraphQL/gRPC. Interviewers notice when candidates miss this.
