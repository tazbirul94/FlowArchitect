Authorization

Controlling what authenticated users are allowed to do — who can read, write, or delete which resources.

Intermediate Security RBAC Access Control

Definition & Analogy

Simple Analogy

Authentication is your hotel key card — it proves who you are and gets you past the lobby. Authorization is the list of floors your card is actually programmed to open. A guest card opens floors 3–8. A housekeeper card opens floors 3–8 but only between 8 AM and 4 PM. A manager card opens the executive lounge too. Same building, different permissions per card.

Authentication answers "who are you?" Authorization answers "what are you allowed to do?" They are distinct systems that work together. A logged-in user might be able to view their own profile but not delete another user's account. A finance employee can read salary reports but not modify them. Authorization is the set of rules that enforces these boundaries — and getting it wrong is one of the most common causes of data breaches.

The difference matters architecturally: authentication happens once at login, while authorization is evaluated on every request. Authorization must live in backend services, not just the UI — hiding a "Delete" button in the frontend is not authorization. The API itself must enforce access control on every call.

The Story

Real-World Story
A healthcare startup built a patient records portal. Doctors could view patient records for their department, nurses could add notes, and admin staff could edit billing information. The engineering team wired up authentication properly — every user needed to log in with MFA. For authorization, they added simple role checks: if user.role === 'doctor' before rendering patient data in the UI.
A junior developer noticed that if you changed the patient ID in the URL from /patients/1042 to /patients/1043, you could see a different patient's full record — even one outside your department, even across hospitals in the same network. The backend API never checked whether the requesting doctor was actually assigned to that patient. The UI had no "View" button for those records, but the API endpoint had no authorization check at all. Over 40,000 patient records were accessible to any authenticated user.
The team introduced resource-level authorization. Every API call now validated: (1) is the user authenticated? (2) does the user's role include read access to patient records? (3) is this specific patient in the user's assigned department or explicitly shared with this doctor? They deployed OPA (Open Policy Agent) as a policy sidecar, separating policy logic from application code. Policies were written in Rego, stored in version control, and evaluated on every request.
The vulnerability was closed with zero patient data exposure confirmed. A full audit log of authorization decisions — who accessed which record, when, and whether it was permitted or denied — was added for HIPAA compliance. Six months later, the audit log caught an insider threat: a billing admin who was querying patient diagnoses outside their role. The logs provided exact evidence for the investigation.

Learn at Your Level

Beginner

Think of a company building. Everyone who works there has a key card (authentication — proves they're an employee). But the key card is programmed differently for each person. The intern can get into the lobby and their own floor. The HR manager can get into the HR floor and employee files. The CEO can get into every floor. The security guard can get into every floor but only during working hours.

Authorization is the system that defines and enforces those rules. In software, instead of floors, we have actions: "can read document", "can delete user", "can approve payment". Instead of a key card programmer, we have code that checks these rules before allowing any action to proceed.

The most common form is Role-Based Access Control (RBAC): you assign users to roles (admin, editor, viewer), and roles have permissions. Instead of giving 500 employees permissions one by one, you give each employee a role, and the role has the permissions.

Key Terms

  • Permission — A specific action allowed on a resource (e.g., "can read invoices", "can delete users").
  • Role — A named collection of permissions (e.g., "editor" = can read + write, but not delete).
  • Principal — The entity being authorized (a user, a service, an API key).
  • Resource — The thing being protected (a document, an API endpoint, a database row).
  • Deny by default — If no rule explicitly allows an action, it is denied. The safe baseline.

RBAC Permission Check Flow

flowchart TD A[Incoming Request\nUser: alice, Action: delete, Resource: invoice-42] --> B{Is user authenticated?} B -- No --> Z1[401 Unauthorized] B -- Yes --> C[Look up alice's roles\nalice → editor] C --> D[Look up editor's permissions\neditor: read, write] D --> E{Does role include\n'delete' permission?} E -- No --> Z2[403 Forbidden\nInsufficient permissions] E -- Yes --> F[Allow request\n200 OK]

Notice there are two different error codes: 401 means "I don't know who you are" (not authenticated). 403 means "I know who you are, but you're not allowed to do this" (not authorized). This distinction is important for debugging and for API clients to handle correctly.

Real systems need more than flat role lists. Users need permissions scoped to specific resources, and roles need hierarchy.

RBAC Implementation

A typical RBAC schema in a database:

-- Users table (already exists)
-- users(id, email, name, ...)

-- Roles
CREATE TABLE roles (
  id   SERIAL PRIMARY KEY,
  name VARCHAR(64) UNIQUE NOT NULL  -- 'admin', 'editor', 'viewer'
);

-- Permissions
CREATE TABLE permissions (
  id       SERIAL PRIMARY KEY,
  action   VARCHAR(64) NOT NULL,    -- 'read', 'write', 'delete'
  resource VARCHAR(64) NOT NULL,    -- 'invoice', 'user', 'report'
  UNIQUE (action, resource)
);

-- Map roles → permissions
CREATE TABLE role_permissions (
  role_id       INT REFERENCES roles(id),
  permission_id INT REFERENCES permissions(id),
  PRIMARY KEY (role_id, permission_id)
);

-- Map users → roles (many-to-many: user can have multiple roles)
CREATE TABLE user_roles (
  user_id INT REFERENCES users(id),
  role_id INT REFERENCES roles(id),
  PRIMARY KEY (user_id, role_id)
);

An authorization check in application code:

def can_perform(user_id: int, action: str, resource: str) -> bool:
    """Check if user has permission to perform action on resource."""
    result = db.execute("""
        SELECT 1
        FROM user_roles ur
        JOIN role_permissions rp ON rp.role_id = ur.role_id
        JOIN permissions p ON p.id = rp.permission_id
        WHERE ur.user_id = %s
          AND p.action    = %s
          AND p.resource  = %s
        LIMIT 1
    """, [user_id, action, resource])
    return result.rowcount > 0

# Usage in an API handler
@app.delete("/invoices/{invoice_id}")
def delete_invoice(invoice_id: int, current_user: User):
    if not can_perform(current_user.id, "delete", "invoice"):
        raise HTTPException(status_code=403, detail="Insufficient permissions")
    db.execute("DELETE FROM invoices WHERE id = %s", [invoice_id])
    return {"deleted": invoice_id}

Permission Hierarchy — Resource-Level vs Field-Level

Permissions exist at multiple granularities:

  • Endpoint-level — Can the user call DELETE /invoices at all?
  • Resource-level (row-level) — Can the user delete this specific invoice (e.g., only ones they created)?
  • Field-level — Can the user see the salary field in an employee record, even if they can read the overall employee record?
def get_employee(employee_id: int, current_user: User) -> dict:
    employee = db.get_employee(employee_id)

    # Resource-level: can this user read this employee's record at all?
    if not can_perform(current_user.id, "read", "employee"):
        raise HTTPException(status_code=403)

    # Row-level: can they read employees in the same department only?
    if current_user.department != employee.department:
        if not current_user.has_role("hr_admin"):
            raise HTTPException(status_code=403, detail="Cross-department access denied")

    # Field-level: strip sensitive fields unless user is HR or finance
    result = employee.to_dict()
    if not current_user.has_any_role(["hr_admin", "finance"]):
        del result["salary"]
        del result["ssn"]

    return result

Principle of Least Privilege

Every user, service, and API key should have the minimum permissions needed to do its job — nothing more. Common violations:

  • A reporting service with write access to the database it only reads.
  • All developers with production database admin access "just in case".
  • A new employee defaulting to the same role as a 10-year senior employee.
  • Service accounts with wildcard permissions (*) because it was convenient to set up.
Implementation Pattern

Cache authorization decisions, but with care. A permission check might hit the DB on every request. Cache the result (e.g., in Redis) with a short TTL — 1 to 5 minutes. When a user's role changes, invalidate their cache entry. Never cache for longer than your acceptable "stale permission" window — if you revoke a user's access, how quickly must it take effect?

RBAC breaks down when permissions depend on context — the time of day, the data itself, or the relationship between the user and the resource. Two more advanced models address this.

ABAC — Attribute-Based Access Control

Policy decisions are based on attributes of the user, the resource, and the environment — not just the role.

# OPA Policy — Rego language
package authz.invoices

import future.keywords.if

# Allow a user to read an invoice if:
# - their department matches the invoice's owner department
# - AND it's during business hours
default allow := false

allow if {
    input.action == "read"
    input.user.department == input.resource.owner_department
    time.clock(time.now_ns())[0] >= 8     # hour >= 8
    time.clock(time.now_ns())[0] < 18    # hour < 18
}

# Finance team can always read any invoice
allow if {
    input.action == "read"
    "finance" in input.user.roles
}

# Nobody can delete invoices older than 7 years (legal hold)
deny if {
    input.action == "delete"
    age_years := (time.now_ns() - input.resource.created_at_ns) / (365 * 24 * 3600 * 1e9)
    age_years >= 7
}

ReBAC — Relationship-Based Access Control

Permissions derive from the relationship graph between users and resources. This is what powers Google Drive: you own a document, you share it with a group, members of that group can view it.

graph LR alice([alice]) -- OWNER --> doc1([doc: proposal.pdf]) alice([alice]) -- MEMBER --> teamA([team: product]) teamA -- EDITOR --> doc2([doc: roadmap.pdf]) bob([bob]) -- MEMBER --> teamA carol([carol]) -- VIEWER --> doc1 style alice fill:#3b82f6,color:#fff style bob fill:#3b82f6,color:#fff style carol fill:#3b82f6,color:#fff style doc1 fill:#8b5cf6,color:#fff style doc2 fill:#8b5cf6,color:#fff style teamA fill:#f59e0b,color:#fff

To check if bob can edit roadmap.pdf: traverse the graph — bob is MEMBER of teamA, teamA is EDITOR of roadmap.pdf → bob can edit. This traversal is the key operation, and it can be expensive at scale.

RBAC vs ABAC vs ReBAC Trade-offs

Model Strengths Weaknesses Best For
RBAC Simple, fast, easy to audit. Works for 1000s of users with a handful of roles. Can't do row-level or context-sensitive checks. Role explosion when fine-grained rules needed. Enterprise apps with clear, stable role boundaries.
ABAC Extremely flexible — any attribute can be used. Handles time, location, data sensitivity. Policy complexity explodes. Hard to audit ("why was this denied?"). Performance cost of evaluating policies. Compliance-heavy domains (healthcare, finance) with conditional access rules.
ReBAC Natural model for user-generated content with sharing. Handles inherited permissions elegantly. Graph traversal is expensive at scale. Complex to debug. Hard to answer "who has access to this?" (reverse lookup). Collaboration platforms (docs, files, tickets) where users share resources dynamically.

OPA — Open Policy Agent

OPA decouples authorization policy from application code. Instead of if user.role == 'admin' scattered across services, all policy lives in a central OPA service queried via HTTP or sidecar.

# OPA input document — sent from your service on each request
{
  "input": {
    "user": {
      "id": "user-42",
      "roles": ["editor"],
      "department": "engineering"
    },
    "action": "delete",
    "resource": {
      "type": "invoice",
      "id": "inv-1099",
      "owner_department": "finance",
      "created_at_ns": 1680000000000000000
    }
  }
}
import httpx

def is_authorized(user: dict, action: str, resource: dict) -> bool:
    """Query OPA for an authorization decision."""
    response = httpx.post(
        "http://opa:8181/v1/data/authz/invoices/allow",
        json={"input": {"user": user, "action": action, "resource": resource}},
        timeout=0.1  # authorization must be fast — 100ms budget
    )
    return response.json().get("result", False)
Production Note

OPA can run as a sidecar (co-deployed with each service, ultra-low latency) or as a central service (easier to update policies centrally, single network hop). For latency-sensitive paths, sidecar mode with policy bundle pre-loaded is preferred. Bundle updates push new policies to all sidecars without a service restart.

Google Zanzibar — The Model Behind Global Authorization

In 2019 Google published the Zanzibar paper describing the authorization system behind Google Drive, YouTube, Maps, and Calendar. It manages over 10 trillion ACL entries and handles 10 million+ QPS at 95th percentile latency under 10ms globally.

Zanzibar is a ReBAC system. Every permission is an ACL tuple:

# Tuple format: <resource>#<relation>@<user-or-userset>
doc:proposal#owner@alice
doc:proposal#viewer@carol
group:product#member@bob
doc:roadmap#editor@group:product#member   # All members of group:product are editors of doc:roadmap

A permission check — "can bob edit doc:roadmap?" — becomes a graph traversal over these tuples. Zanzibar calls this a check request. The traversal can go multiple hops: bob → member of group:product → group:product is editor of doc:roadmap → allow.

Open-source implementations inspired by Zanzibar: OpenFGA (by Auth0/Okta), SpiceDB (by Authzed), Ory Keto. All expose a similar tuple API and check endpoint.

import openfga_sdk

# Write a relationship tuple
client.write(body={
    "writes": {
        "tuple_keys": [
            {"object": "document:proposal", "relation": "owner", "user": "user:alice"},
            {"object": "document:proposal", "relation": "viewer", "user": "user:carol"},
        ]
    }
})

# Check permission — Zanzibar-style
result = client.check(body={
    "tuple_key": {
        "object": "document:proposal",
        "relation": "viewer",
        "user": "user:carol"
    }
})
# result.allowed == True

Audit Logging — SOC 2 & GDPR Requirements

Every authorization decision — not just denials, but every allow too — should be logged. This is a hard requirement for SOC 2 Type II (CC6.2, CC6.3) and GDPR Article 32 (accountability). The log must capture:

  • Who — user ID, service account ID, session ID
  • What — action attempted, resource ID, resource type
  • When — ISO 8601 timestamp with milliseconds
  • Decision — allow / deny, and which policy rule triggered
  • Context — IP address, user agent, request ID for correlation
{
  "ts": "2024-11-15T14:32:11.443Z",
  "event": "authz.decision",
  "user_id": "usr_8fg2k9",
  "session_id": "sess_3jd8k1",
  "action": "delete",
  "resource_type": "invoice",
  "resource_id": "inv-1099",
  "decision": "deny",
  "policy_rule": "authz.invoices/deny — resource.age_years >= 7",
  "ip": "10.0.4.22",
  "request_id": "req_a9bc23",
  "latency_ms": 2
}

Compliance Checklist

  • SOC 2 CC6.2: Logical access controls implemented and enforced at all layers
  • SOC 2 CC6.3: Role assignments reviewed quarterly; terminated employees de-provisioned within 24 hours
  • GDPR Art. 25: Data minimization — authorization restricts access to minimum required fields (field-level)
  • GDPR Art. 32: Audit logs retained for minimum 90 days (SOC 2) or 1 year (many compliance programs)
  • HIPAA 164.312(a)(1): Role-based access control for PHI; unique user IDs in all audit logs
  • PCI DSS 7: Access to cardholder data limited to business need-to-know; access matrix documented
  • Policies in version control — every policy change is a PR with a reviewer
  • Automated tests for critical deny paths — regression tests for known privilege escalation scenarios
Zanzibar Consistency Model

Zanzibar introduced the concept of a zookie (consistency token) to handle a key problem: if Alice shares a document with Bob and Bob immediately tries to open it, the check might run on a replica that hasn't seen the write yet — and deny him. Zanzibar solves this by returning a zookie with every write. Bob passes the zookie in his check request, forcing evaluation on a replica that is at least as up-to-date as that write. This is "snapshot reads with external consistency" — a key design pattern for globally distributed authorization systems.

Enterprise Use Case

Enterprise Use Case Google — Zanzibar Global Authorization (10 trillion ACLs, 10M+ QPS)

Context: Google runs a single unified authorization system — Zanzibar — for Google Drive, YouTube, Google Maps, Google Calendar, Google Photos, and more. At peak it evaluates over 10 million authorization checks per second, with a 95th percentile latency under 10ms globally, managing over 10 trillion ACL relationship tuples. It was published as a research paper in 2019 and became the reference model for modern authorization infrastructure.

Why a shared system: Before Zanzibar, each Google product managed its own ACL system. Drive had different sharing semantics than Calendar. Integrations (e.g., "share a Google Doc link in a Calendar invite") required cross-product authorization checks that each team implemented differently — inconsistently and insecurely. Zanzibar unified all of them behind a single tuple model and check API.

The architecture: Zanzibar stores tuples in Spanner (Google's globally distributed relational database). Checks traverse the relationship graph: "can user U perform relation R on object O?" is resolved by expanding relation R recursively through userset rewrite rules until reaching a leaf user. Negative caching (caching "deny" results) and Leopard indexing (precomputed group membership) reduce the cost of deep traversals.

Concrete numbers from the paper:

  1. 10 trillion ACL entries under management
  2. 10 million+ check QPS at production peak
  3. 95th percentile check latency: <10ms globally
  4. 99th percentile check latency: <25ms
  5. Consistency guaranteed via zookie tokens (new-enemy problem solved)

Business impact and open-source ecosystem: The Zanzibar paper inspired a generation of open-source implementations. Auth0 (now Okta) released OpenFGA. Authzed released SpiceDB. Ory released Ory Keto. Companies like Airbnb, Carta, and Duolingo now use Zanzibar-model authorization instead of building custom RBAC. The model enables fine-grained sharing (exactly what a Google Drive "share with specific person" requires) while remaining consistent at global scale.

Lesson: For consumer-scale systems with user-generated content and complex sharing needs, graph-based authorization (ReBAC) is the right model — but it requires infrastructure investment. Simple RBAC is appropriate until you need row-level or dynamic sharing semantics.

Common Mistakes

Authorization Only in the Frontend

Hiding a "Delete" button in the UI for users who aren't admins is not authorization. It's UI logic. Any user with browser dev tools or a tool like curl can call the API directly and bypass every frontend check. Authorization must be enforced in the backend on every API call — the frontend is convenience, not security. This is one of the most common causes of OWASP A01 (Broken Access Control) vulnerabilities.

Fix: Every API endpoint that accesses or mutates data must have an authorization check. Treat frontend permission checks as a UX nicety, never as a security control.

No Audit Log of Authorization Decisions

Most teams log authentication events (login, logout, failed login). Few log authorization decisions. But authorization logs are what SOC 2 Type II requires (CC6.2, CC6.3), what GDPR Article 32 accountability demands, and what your security team needs to detect insider threats and investigate breaches. "Who accessed which data, when, and was it permitted?" must be answerable from logs.

Fix: Log every authorization decision — not just denials. Include user ID, resource ID, action, decision, and a request correlation ID. Ship logs to a SIEM where they cannot be tampered with by the application itself.

Hardcoded Role Strings Scattered Across the Codebase

Checking if user.role === "admin" in 47 different places means one typo or one undocumented role name bypasses security. When a new role is added, every check has to be found and updated. There is no central place to audit what "admin" is actually allowed to do. This is policy-as-code's worst anti-pattern: policy-as-string-literals.

Fix: Centralize permission checks in a single authorization module or policy engine. Define permissions as constants (Permission.DELETE_INVOICE), not magic strings. Use a policy engine like OPA, Casbin, or a dedicated authorization service so that all policy lives in one auditable place.

Interview Prep

Authorization is the enforcement of who is allowed to do what to which resources — implemented on the backend on every request, distinct from authentication, using models like RBAC, ABAC, or ReBAC depending on how dynamic and fine-grained the access rules need to be.

2-Minute Structured Answer

"Authorization is separate from authentication. Authentication proves who you are; authorization decides what you're allowed to do. It must live in the backend — never only in the UI. The most common model is RBAC: users get roles, roles have permissions, and every API call checks whether the current user's role includes the required permission. For more dynamic needs, ABAC adds context — you can write policies like 'allow read if user.department matches resource.owner_department and it's business hours.' For user-generated content with sharing semantics, ReBAC models permissions as a relationship graph — this is how Google Drive works. At scale, a policy engine like OPA separates policy from code: policies are in version control, deployed independently, and every authorization decision is evaluated by OPA rather than scattered if-statements. Critical production requirement: log every authorization decision for compliance and incident response."

Common Interview Mistakes

  • Conflating authentication with authorization — they are distinct systems. Authentication is a prerequisite, not a substitute.
  • Describing only frontend checks — interviewers specifically probe whether you understand that authorization lives in the backend API layer.
  • Treating RBAC as the only model — for systems with dynamic sharing or attribute-based rules, RBAC alone is insufficient.
  • Forgetting audit logging — in any compliance or security context, the interviewer expects you to mention authorization logs as a requirement, not an afterthought.

Trade-Offs Summary

Dimension RBAC ABAC ReBAC (Zanzibar)
Simplicity High — roles and permissions are easy to reason about Low — policy combinatorics become complex quickly Medium — the graph model is intuitive but graph traversal logic is hard
Flexibility Low — can't easily express "only on business hours" or "only their own records" Very high — any attribute can be used in any combination High — relationship hierarchies handle complex sharing semantics naturally
Performance Fast — role lookup is a single DB join or cache hit Slower — policy evaluation requires loading attribute data for user and resource Variable — graph traversal cost scales with depth and fan-out of relationships
Auditability Easy — "what can role X do?" is a simple query Hard — "who has access to resource Y?" requires evaluating all policies for all users Medium — reverse lookups are possible but expensive at scale
Scalability Scales to millions of users with a few hundred roles Policy evaluation adds latency at high QPS — policy engine must be co-located Requires specialized infrastructure (Spanner-level) at Google scale; OpenFGA/SpiceDB at startup scale
