JWT — JSON Web Token

A compact, self-contained token format for securely transmitting identity and claims between parties — no database lookup required to validate a request.

Intermediate Security Authentication Stateless

Definition & Analogy

Simple Analogy

Think of a JWT like a digitally signed concert wristband. The wristband says who you are, what areas you're allowed into, and when it expires — all printed directly on it. Any security guard can read the wristband and verify it's genuine by checking the venue's seal, without calling the ticket office every time. If the seal is intact and the wristband hasn't expired, you're in.

In software: the "wristband" is the token, the "seal" is the cryptographic signature, and the "security guard" is any service that knows the public key.

A JWT (JSON Web Token) is a URL-safe, base64url-encoded string composed of three parts separated by dots: header.payload.signature. The header identifies the token type and signing algorithm. The payload carries claims — facts about the user like their ID, roles, and when the token expires. The signature proves the token was issued by a trusted authority and hasn't been tampered with since.

The critical property that makes JWTs useful at scale is statelessness. A traditional session-based system requires the server to look up a session record in a database on every request. With a JWT, the server only needs its signing key — it can verify the token's authenticity and read the user's claims entirely in memory. This removes a database round-trip from the hot path of every authenticated API call, which at millions of requests per second is a meaningful performance gain.

The Story

Real-World Story
A fintech startup built a microservices platform — five independent services: user-service, account-service, transaction-service, notification-service, and an API gateway. They used server-side sessions for authentication: on login, a session ID was created and stored in a shared Redis instance. Every service queried Redis on each request to validate the session.
At 50,000 concurrent users, Redis became the bottleneck. Every API call made a synchronous Redis lookup. Latency spiked during peak hours. When Redis had a 4-minute outage, all five services rejected every request — not just authentication — because they couldn't validate any session. The Redis cluster was now a distributed single point of failure. The ops team got paged at 2 AM.
The team migrated to JWTs. On login, the auth service issued a signed JWT containing the user's ID, roles, and an expiry time. The JWT was signed with a private RSA key held only by the auth service. Every other service was given the public key — a one-time configuration step. Services now validated tokens locally: decode the JWT, verify the RSA signature against the public key, check the expiry claim. No Redis, no network call, no shared state.
P99 authentication latency dropped from 22ms (Redis roundtrip) to under 1ms (local verification). The Redis cluster was decommissioned from the auth hot path — kept only for refresh token management. When a downstream service went down, it no longer took authentication with it. The platform handled 3× the user load without additional infrastructure. The 2 AM Redis pager alerts stopped.

Learn at Your Level

Beginner

When you log in to a website, the server needs a way to remember who you are on every future request. HTTP is stateless — it doesn't remember anything between requests. Servers used to solve this by storing a session cookie tied to a database record. But that means every page load hits the database just to ask "is this person logged in?"

A JWT solves this differently: instead of the server storing your session, it gives you a token that already contains your identity. The token is cryptographically signed — the server can verify it's genuine without asking anyone. You attach the token to every request in the Authorization header. The server reads it, verifies the signature, and knows exactly who you are and what you're allowed to do.

Think of it like a government-issued ID card. You carry it. Any official can verify it by checking the hologram (signature). They don't need to call the government office every time — the card itself is the proof.

Key Terms

  • Token — A string that proves your identity to a server.
  • Claim — A key-value pair inside the JWT payload, e.g. "role": "admin".
  • Signature — A cryptographic proof that the token was issued by a trusted authority and hasn't changed.
  • Expiry (exp) — A Unix timestamp after which the token is no longer valid.
  • Bearer token — The convention of sending the JWT in the HTTP header: Authorization: Bearer <token>.

A JWT has three base64url-encoded sections joined by dots. Understanding each section is fundamental to using JWTs correctly.

Header

The header declares the token type and the signing algorithm:

{
  "alg": "RS256",
  "typ": "JWT"
}

Algorithm options:

  • HS256 (HMAC-SHA256) — Symmetric: the same secret is used to both sign and verify. Simple to set up, but every service that verifies tokens must have the secret. A leaked secret in any one service compromises all tokens across all services.
  • RS256 (RSA-SHA256) — Asymmetric: a private key signs the token, a public key verifies it. The auth service holds the private key exclusively. All other services get the public key. A compromised downstream service cannot forge tokens. Recommended for multi-service architectures.
  • ES256 (ECDSA P-256) — Asymmetric like RS256 but with elliptic curve cryptography. Produces smaller signatures and is faster to verify than RSA. Use this for high-throughput systems or mobile where bandwidth and CPU matter.

Payload — Standard Claims (RFC 7519)

{
  "iss": "https://auth.ticketflow.io",
  "sub": "user_456",
  "aud": "ticket-service",
  "exp": 1717001800,
  "iat": 1717000000,
  "jti": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "roles": ["user", "organiser"],
  "email": "alice@example.com"
}
  • iss (Issuer) — Who created the token. Services should reject tokens from unknown issuers.
  • sub (Subject) — The user this token represents. Typically the user's ID in your system.
  • aud (Audience) — Which service this token is intended for. A token for ticket-service should be rejected by payment-service.
  • exp (Expiration) — Unix timestamp after which the token must be rejected. Keep access tokens short-lived (5–15 minutes).
  • iat (Issued At) — When the token was created. Useful for calculating token age.
  • jti (JWT ID) — A unique identifier for this specific token. Used to implement a revocation blocklist.

Signature

The signature is computed over the base64url-encoded header and payload:

signature = RSA_SHA256(
  private_key,
  base64url(header) + "." + base64url(payload)
)

When a service receives a JWT, it re-computes this check using the public key. If the signatures match, the token is genuine. If anyone tampered with even one character of the payload, the signature check fails and the token is rejected.

Validating a JWT in Node.js

const jwt = require('jsonwebtoken');
const fs = require('fs');

// Load public key once at startup — NOT on every request
const publicKey = fs.readFileSync('./keys/public.pem');

function validateToken(token, expectedAudience) {
  try {
    const payload = jwt.verify(token, publicKey, {
      algorithms: ['RS256'],   // Reject HS256 — prevents "alg:none" attacks
      audience: expectedAudience,
      issuer: 'https://auth.ticketflow.io',
    });
    return { valid: true, payload };
  } catch (err) {
    // err.name: 'TokenExpiredError' | 'JsonWebTokenError' | 'NotBeforeError'
    return { valid: false, reason: err.message };
  }
}

// Usage in Express middleware
app.use((req, res, next) => {
  const auth = req.headers.authorization;
  if (!auth?.startsWith('Bearer ')) return res.status(401).json({ error: 'Missing token' });

  const token = auth.slice(7);
  const result = validateToken(token, 'ticket-service');

  if (!result.valid) return res.status(401).json({ error: result.reason });

  req.user = result.payload;
  next();
});

At production scale, three problems dominate JWT architecture discussions: token revocation, refresh token rotation, and algorithm confusion attacks.

The Token Revocation Problem

JWTs are stateless by design — but statelessness means you cannot revoke them before expiry. If a user logs out, their JWT remains valid until exp. If an admin disables an account, that user's existing JWT still works. This is the core JWT trade-off.

Solutions in order of complexity:

  • Short expiry (5–15 min) — The simplest mitigation. A stolen token is only valid briefly. Requires refresh tokens for good UX.
  • JTI blocklist in Redis — On logout or account disable, add the token's jti to a Redis set with a TTL matching the token's remaining lifetime. Every service checks the blocklist. This re-introduces a shared data store — small lookup, but now Redis is in the critical path again.
  • Version claim — Store a token_version per user in the database. Include it as a claim in the JWT. Services validate the claim matches the current version. Invalidating a user's tokens means incrementing their version. Trades one Redis lookup for one DB lookup — similar overhead, better semantics.

Refresh Token Rotation

Refresh tokens solve the UX problem of short-lived access tokens without sacrificing security. The pattern:

  1. On login, issue two tokens: a short-lived access token (15 min, JWT) and a long-lived refresh token (30 days, opaque string stored in DB).
  2. Client stores the refresh token in an HTTP-only cookie (not localStorage — XSS protection).
  3. When the access token expires, the client sends the refresh token to /auth/refresh.
  4. Auth service verifies the refresh token against the database, issues a new access token and a new refresh token, and invalidates the old refresh token.
  5. If an attacker steals and uses the refresh token first, the old token is marked used. When the legitimate client tries to refresh with the now-invalidated token, the auth service detects reuse and revokes the entire session family.

Trade-offs: HS256 vs RS256 vs ES256

Algorithm Confusion Attack

Critical Vulnerability: "alg:none" and Algorithm Switching

An attacker can modify a JWT's header to set "alg": "none" and strip the signature. Naive implementations that trust the algorithm declared in the header will accept this as valid. Always specify allowed algorithms explicitly in verification code — never accept whatever algorithm the token claims.

A related attack: if your system accepts both RS256 and HS256, an attacker can take an RS256 token, re-sign it using HS256 with the public key as the HMAC secret. Libraries that don't enforce algorithm type will verify this as valid. Fix: always hard-code the expected algorithm in your verify call.

Audience Validation Is Non-Optional

Without aud validation, a token issued for payment-service can be sent to admin-service and accepted. In a microservices architecture, this is a privilege escalation vector — if one low-privilege service is compromised, an attacker can replay its tokens against higher-privilege services. Always validate aud on every service.

Refresh Token Implementation

// Refresh token rotation — Node.js / TypeScript example

interface RefreshTokenRecord {
  tokenHash: string;    // bcrypt hash of the refresh token
  userId: string;
  familyId: string;     // all tokens from one login share a family
  used: boolean;
  expiresAt: Date;
}

async function refreshTokens(incomingRefreshToken: string): Promise<TokenPair> {
  const tokenHash = await bcrypt.hash(incomingRefreshToken, 10);
  const record = await db.refreshTokens.findOne({ tokenHash });

  if (!record) throw new UnauthorizedError('Invalid refresh token');

  // Detect reuse — token was already consumed
  if (record.used) {
    // Revoke the entire family — session compromise assumed
    await db.refreshTokens.deleteMany({ familyId: record.familyId });
    throw new UnauthorizedError('Refresh token reuse detected — session revoked');
  }

  if (record.expiresAt < new Date()) throw new UnauthorizedError('Refresh token expired');

  // Mark old token as used
  await db.refreshTokens.updateOne({ tokenHash }, { used: true });

  // Issue new token pair
  const newRefreshToken = crypto.randomUUID() + crypto.randomUUID(); // 72 chars entropy
  const newAccessToken = jwt.sign(
    { sub: record.userId, aud: 'api', iss: 'https://auth.ticketflow.io' },
    privateKey,
    { algorithm: 'RS256', expiresIn: '15m' }
  );

  await db.refreshTokens.insert({
    tokenHash: await bcrypt.hash(newRefreshToken, 10),
    userId: record.userId,
    familyId: record.familyId,  // Inherit family from parent token
    used: false,
    expiresAt: new Date(Date.now() + 30 * 24 * 60 * 60 * 1000),
  });

  return { accessToken: newAccessToken, refreshToken: newRefreshToken };
}
Architecture Note

Refresh tokens are stateful by necessity — you need to track which ones have been used. This is acceptable because the refresh endpoint is called infrequently (every 15 minutes per active session) compared to the access token check (every API call). The stateful overhead is bounded: O(active sessions) rows in the database, not O(requests/second).

At enterprise scale, JWT infrastructure spans key management, compliance, multi-cloud scenarios, and token binding for high-assurance systems.

JWK Endpoints and Key Rotation

Hard-coding a public key in each service creates a deployment problem: when the key rotates, every service must be redeployed or reconfigured. The solution is the JWKS (JSON Web Key Set) endpoint — a standardized URL where services fetch current public keys dynamically.

// GET /.well-known/jwks.json
{
  "keys": [
    {
      "kty": "RSA",
      "use": "sig",
      "kid": "key-2024-v3",      // Key ID — referenced in JWT header
      "alg": "RS256",
      "n": "0vx7agoebGcQSuuPiLJXZptN9nndrQmbXEps2aiAFbW...",
      "e": "AQAB"
    },
    {
      "kty": "RSA",
      "use": "sig",
      "kid": "key-2023-v2",      // Old key — kept for in-flight tokens during rotation
      "alg": "RS256",
      "n": "sGu5kVpUFNa7GvVlBVe5a82WvCoEFRx94wDXEL...",
      "e": "AQAB"
    }
  ]
}

The JWT header includes a kid (Key ID) field. The receiving service fetches the JWKS endpoint (cached with a short TTL), looks up the key matching the kid, and uses it for verification. Key rotation becomes a two-phase operation: add the new key to JWKS → wait for in-flight tokens signed with the old key to expire → remove the old key. Zero-downtime rotation with no service redeployment.

Fetching JWKS with Caching

// Service startup — cache JWKS, refresh every 12h or on cache miss
import jwksClient from 'jwks-rsa';

const client = jwksClient({
  jwksUri: 'https://auth.ticketflow.io/.well-known/jwks.json',
  cache: true,
  cacheMaxEntries: 10,
  cacheMaxAge: 12 * 60 * 60 * 1000, // 12 hours
  rateLimit: true,
  jwksRequestsPerMinute: 5,  // prevent JWKS DDoS from bad tokens
});

async function getPublicKey(kid: string): Promise<string> {
  const key = await client.getSigningKey(kid);
  return key.getPublicKey();
}

async function verifyToken(token: string): Promise<JWTPayload> {
  const decoded = jwt.decode(token, { complete: true }) as any;
  if (!decoded?.header?.kid) throw new Error('Missing kid in token header');

  const publicKey = await getPublicKey(decoded.header.kid);

  return jwt.verify(token, publicKey, {
    algorithms: ['RS256'],
    audience: process.env.SERVICE_NAME,
    issuer: 'https://auth.ticketflow.io',
  }) as JWTPayload;
}

Token Binding

A bearer token stolen from network traffic can be replayed from any client. Token binding cryptographically ties a JWT to a specific TLS connection, making captured tokens useless on other connections. The DPoP (Demonstrating Proof of Possession) RFC 9449 is the practical implementation: the client generates an asymmetric key pair per session, includes the public key as a claim in the JWT, and signs each request with the private key. The server verifies both the JWT signature and the DPoP proof. Even if a token is stolen, it cannot be used without the private key.

Compliance Considerations

  • GDPR / CCPA: JWT payloads are base64url decoded — not encrypted. Do not include PII (name, email, address) in standard JWTs unless you use JWE (JSON Web Encryption). Include only an opaque user ID and look up PII when needed. If a user requests data erasure, their token claims cannot be erased until the token expires.
  • PCI-DSS: Tokens used in payment flows must have expiry enforced. Audit logs must record token issuance and validation events. Signing keys must be rotated per key management policy (typically annually, immediately on suspected compromise).
  • HIPAA: PHI (Protected Health Information) must not appear in JWT payloads transmitted outside controlled networks. Use opaque references. Token lifetimes should reflect minimum necessary access.
  • SOC 2: Key rotation procedures, access to private key material, and token issuance audit trails are scope items. Key storage in HSM (Hardware Security Module) or cloud KMS (AWS KMS, Azure Key Vault, GCP Cloud KMS) is expected for Type II certification.

Key Storage — Never in Code

// Wrong — private key in source code or env var
const privateKey = process.env.JWT_PRIVATE_KEY; // Bad: env vars leak in logs/crash dumps

// Right — fetch from AWS KMS or Secrets Manager at startup
import { SecretsManagerClient, GetSecretValueCommand } from '@aws-sdk/client-secrets-manager';

const secretsClient = new SecretsManagerClient({ region: 'eu-west-1' });

async function loadSigningKey(): Promise<string> {
  const response = await secretsClient.send(new GetSecretValueCommand({
    SecretId: 'ticketflow/jwt/private-key',
  }));
  return response.SecretString!;
}

// For AWS KMS-backed signing (key never leaves KMS hardware):
import { KMSClient, SignCommand } from '@aws-sdk/client-kms';

const kmsClient = new KMSClient({ region: 'eu-west-1' });

async function signWithKMS(message: string): Promise<Buffer> {
  const response = await kmsClient.send(new SignCommand({
    KeyId: 'arn:aws:kms:eu-west-1:123456789:key/mrk-abc123',
    Message: Buffer.from(message),
    MessageType: 'RAW',
    SigningAlgorithm: 'RSASSA_PKCS1_V1_5_SHA_256',
  }));
  return Buffer.from(response.Signature!);
}
Expert Consideration

For very high-throughput systems, asymmetric JWT signing (RS256/ES256) at the auth service can become a bottleneck — RSA signing is CPU-intensive. Mitigations: use ES256 (faster than RS256), cache signed tokens for identical requests within a short window, or use dedicated crypto hardware. At 100K logins/second, even ES256 signing may require horizontal scaling of the auth service itself.

Enterprise Use Case

Enterprise Use Case Auth0 — Identity Platform for 10,000+ Companies (Mozilla, Atlassian, Siemens)

Context: Auth0 is an identity-as-a-service platform that issues JWTs for thousands of client applications. It handles over 1 billion authentication events per month. Customers include Mozilla, Atlassian, Siemens, and JetBlue. Auth0 manages the entire JWT lifecycle — issuance, rotation, revocation — as a managed service so developers don't build it themselves.

How Auth0 uses JWTs:

  1. Auth0 issues RS256-signed access tokens with a 15-minute expiry. Each token includes iss, sub, aud, exp, iat, and custom claims defined by the customer (e.g. user roles, subscription tier).
  2. The JWKS endpoint at https://<tenant>.auth0.com/.well-known/jwks.json is publicly available. Customer services fetch and cache the public keys — no coordination required across deployments.
  3. Refresh token rotation is enabled by default with absolute lifetime of 30 days and inactivity lifetime of 3 days. Reuse detection is automatic — a reused refresh token triggers revocation of the entire token family and notifies the customer via webhook.
  4. Customers never see the private signing key. Key rotation is handled transparently by Auth0 — services pick up new keys via JWKS without any configuration change.
  5. For enterprise customers, Auth0 offers custom domains (e.g. auth.atlassian.com) so JWTs appear to originate from the customer's domain — important for the iss claim consistency across environments.

Key metrics: Median token validation latency under 0.5ms (local verification with cached keys). Auth service availability 99.99% SLA. Supports 50,000+ concurrent logins per second at peak without degradation. Refresh token store runs on distributed Postgres with read replicas — refresh is rare enough that DB latency doesn't dominate.

Lessons:

  1. JWKS caching is critical: At 1B auth events/month, the JWKS endpoint itself must be cached aggressively by clients. Auth0 uses CDN-backed JWKS with long cache headers — fetching the JWKS on every token validation would overwhelm any infrastructure.
  2. Refresh token reuse detection catches real attacks: Auth0's telemetry shows reuse detection fires on actual credential theft cases, not just implementation bugs. The automatic family revocation is what turns detection into remediation.
  3. Custom claims drive adoption: The most-used Auth0 feature after basic auth is custom claims in the access token — embedding subscription tier, user permissions, and tenant ID so downstream services don't need extra DB lookups for common authorization decisions.

Common Mistakes

No Expiry — The Immortal Token

JWTs without an exp claim never expire. If such a token is leaked — from a log file, a compromised client, or a data breach — it grants permanent access to your system with no remediation option. This is the single most dangerous JWT misconfiguration. Always set a short expiry (5–15 minutes for access tokens).

A common anti-pattern is setting expiry to 1 year "for convenience during development" and then deploying to production. Audit every JWT-issuing code path and enforce a maximum expiry at the token creation layer, not just documentation.

Skipping Audience Validation

If your services don't validate the aud claim, a token issued for user-service can be replayed against admin-service and accepted. In a microservices architecture, not every service has the same privilege level. A token valid for reading user profiles should not grant access to payment operations.

This becomes a real attack vector when one service is compromised: the attacker extracts tokens from that service's memory or logs and replays them against other services. Validating aud limits the blast radius. Every service must explicitly check that the token's aud matches its own service identifier.

Storing Sensitive Data in the Payload

Base64url encoding is not encryption. Anyone who receives a JWT can decode the payload by running it through any base64 decoder — no key required. Yet developers routinely put passwords, credit card numbers, social security numbers, and other PII into JWT payloads.

The payload is visible to: the user's browser (if stored in localStorage), browser extensions, backend logs that record full request headers, CDN access logs, proxy servers, and anyone who intercepts unencrypted HTTP traffic. Store only non-sensitive identifiers in the payload (user ID, roles). Retrieve sensitive data from your database using those identifiers after validation. If you must embed sensitive data, use JWE (JSON Web Encryption) — not regular JWTs.

Interview Prep

A JWT is a signed, self-contained token that carries identity claims — the server verifies the signature locally without a database lookup, enabling stateless authentication across distributed services.

2-Minute Structured Answer

"A JWT has three parts: header, payload, and signature — all base64url encoded and joined by dots. The payload carries standard claims like sub (user ID), exp (expiry), aud (intended audience), and custom claims like roles. The signature is computed with a private key by the auth service, and any service with the public key can verify it locally — no database round-trip required. This is the key trade-off: stateless validation is fast but you can't revoke tokens before they expire. In practice you mitigate this with short expiry — 15 minutes — combined with refresh token rotation. For algorithm choice, I'd use RS256 or ES256 in a multi-service system because the private key stays only in the auth service; downstream services can't forge tokens even if they're compromised. I'd avoid HS256 because every verifying service shares the secret, which multiplies the blast radius of a key leak."

Common Interview Mistakes

  • Saying "JWTs are encrypted" — they are signed, not encrypted. The payload is readable by anyone with the token unless you use JWE specifically.
  • Saying "just set a long expiry to avoid the refresh token complexity" — this misunderstands the security model. Long-lived tokens are the problem, not the solution.
  • Not mentioning the revocation problem — interviewers expect you to know this is the core limitation of stateless tokens and to have a concrete mitigation strategy.
  • Recommending HS256 for microservices — shows lack of understanding of shared-secret risk in multi-service environments.
  • Forgetting audience validation — a commonly-skipped security check that interviewers specifically probe.
