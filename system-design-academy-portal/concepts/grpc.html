gRPC

A high-performance, open-source RPC framework that uses Protocol Buffers as its interface definition language and HTTP/2 as its transport, enabling efficient bi-directional streaming between services.

Intermediate Advanced APIs Microservices Protocol Buffers HTTP/2

Definition & Analogy

Simple Analogy

REST is like sending handwritten letters — easy to read for anyone, but large and slow to deliver. gRPC is like using a high-speed fax machine with a shared codebook — both sides agree on the exact format upfront, messages are compressed to the minimum, and the phone line stays open to send many documents at once. Much faster and more efficient, but you need the codebook (the .proto file) on both ends.

gRPC was born inside Google as "Stubby" — the internal RPC system that connected thousands of microservices for over a decade. In 2016 Google open-sourced it as gRPC, built on top of two open standards: Protocol Buffers (protobuf) for schema-defined binary serialization, and HTTP/2 for multiplexed transport. Today it powers the data planes of Uber, Netflix, CoreOS (etcd), and hundreds of cloud-native systems.

Where REST sends JSON text over HTTP/1.1 — human-readable but verbose and one-request-per-connection — gRPC sends compact binary over a persistent HTTP/2 connection, with auto-generated client and server code in any supported language, and built-in support for four communication patterns including real-time bidirectional streaming.

The Story

Real-World Story
A FinTech startup built its microservices architecture with REST+JSON between all services. Their risk-service received a loan application decision from credit-service, and the payload included the full customer profile: name, address, income breakdown, credit history — roughly 14 KB of JSON per request. The team felt REST was fine — after all, their endpoints worked and Postman could hit them directly.
As they scaled to 50,000 loan applications per day — about 3,500 requests per hour at peak — the internal JSON serialization and deserialization was burning 22% of CPU time on every service instance. Worse, their fraud-detection service needed to push real-time risk signals back while the credit check was still running (a streaming pattern), which REST simply couldn't model. They polled every 200ms as a workaround, adding even more overhead. Latency between services averaged 38ms.
The team migrated the high-throughput internal APIs to gRPC. They defined their schemas in .proto files: field names became field numbers (1 byte instead of a string), unused fields were omitted entirely, and data was serialized as binary rather than UTF-8 JSON. For the fraud-detection flow, they used gRPC's server-streaming RPC — the credit service streamed risk signals back as they were computed, eliminating the polling loop entirely.
Payload size dropped from 14 KB to 1.1 KB per request — an 8x reduction. Internal service latency fell from 38ms to 4ms. CPU consumption for serialization dropped by 78%. The fraud-detection service eliminated 17,500 polling requests per hour. REST APIs were kept for external partners and the public-facing mobile app — gRPC was used only on the internal data plane where performance mattered most.

Learn at Your Level

Beginner

You already know APIs: one service asks another for data, and the other responds. REST is the most common way to do this — the asking service sends an HTTP request with a JSON body, and the responding service sends JSON back.

gRPC is a different way of doing the same thing, but much faster. Instead of sending a JSON message like {"user_id": 123, "name": "Alice"} (36 bytes of text), it sends a tiny binary message that means the same thing but might be only 6 bytes. To achieve this, both services agree on an exact schema upfront — a .proto file that says "field 1 is user_id, field 2 is name". Neither service sends field names over the wire; just field numbers and values.

gRPC also runs over HTTP/2, which means one open connection can carry many requests simultaneously. HTTP/1.1 (what most REST APIs use) can only process one request at a time per connection — if you want to send 10 requests at once, you need 10 separate connections.

Key Terms

  • .proto file — A schema file that defines your data structures and service methods. Both the client and server use this file to generate code automatically.
  • Protocol Buffers (protobuf) — Google's binary serialization format. Smaller and faster than JSON, but not human-readable.
  • Stub — Auto-generated client code. Instead of manually building HTTP requests, you call a method on a generated object as if the remote function were local.
  • RPC (Remote Procedure Call) — Calling a function on a different machine as if it were a local function call.
  • HTTP/2 — The transport layer gRPC runs on. Allows multiple requests over one connection simultaneously.

When to Use gRPC vs REST

  • Use gRPC when: internal service-to-service communication, high throughput, streaming data, multiple languages in your stack.
  • Use REST when: public APIs, browser clients, simplicity matters more than speed, third-party integrations.

gRPC has three core technical pillars: Protocol Buffers for schema definition and serialization, HTTP/2 for transport, and code generation for client/server stubs. Understanding each pillar explains both why gRPC is fast and why it requires more setup than REST.

Protocol Buffers: The Schema

Every gRPC API starts with a .proto file. This file defines the shape of all messages and the methods of every service. Field numbers (not names) are what's serialized — this is how protobuf achieves compact binary encoding and backward compatibility.

// user.proto
syntax = "proto3";

package user;

// Message definitions — field numbers are what travel over the wire
message GetUserRequest {
  int32 user_id = 1;          // field 1
}

message User {
  int32  user_id   = 1;       // field 1: 1 byte tag + value
  string name      = 2;       // field 2
  string email     = 3;       // field 3
  bool   is_active = 4;       // field 4
  // Adding a new field later is safe — old clients ignore unknown fields
  string phone     = 5;       // field 5 (added in v2, backward-compatible)
}

// Service definition — describes RPC methods
service UserService {
  // Unary: 1 request → 1 response (like REST)
  rpc GetUser (GetUserRequest) returns (User);

  // Server-streaming: 1 request → stream of responses
  rpc ListUsers (ListUsersRequest) returns (stream User);

  // Client-streaming: stream of requests → 1 response
  rpc BulkCreateUsers (stream CreateUserRequest) returns (BulkCreateResponse);

  // Bidirectional streaming: both sides stream simultaneously
  rpc SyncUsers (stream SyncRequest) returns (stream SyncResponse);
}

Code Generation: No More Manual HTTP Clients

The protoc compiler takes your .proto file and generates typed client and server code. You never write HTTP requests, parse JSON, or build URL strings.

# Install protoc and Go gRPC plugin
go install google.golang.org/protobuf/cmd/protoc-gen-go@latest
go install google.golang.org/grpc/cmd/protoc-gen-go-grpc@latest

# Generate Go code from the .proto file
protoc --go_out=. --go-grpc_out=. user.proto

# This generates:
#   user.pb.go        — message types (User, GetUserRequest, etc.)
#   user_grpc.pb.go   — UserServiceClient interface + UserServiceServer interface

Using the Generated Client (Go)

package main

import (
    "context"
    "log"
    "time"

    "google.golang.org/grpc"
    pb "github.com/myorg/user-service/proto"
)

func main() {
    // Dial once — reuse this connection for all calls
    conn, err := grpc.Dial("user-service:50051", grpc.WithInsecure())
    if err != nil {
        log.Fatalf("failed to connect: %v", err)
    }
    defer conn.Close()

    client := pb.NewUserServiceClient(conn)

    // Always set a deadline — if you don't, goroutines can leak
    ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
    defer cancel()

    // This looks like a local function call
    // Under the hood: serializes to binary, sends over HTTP/2, deserializes response
    user, err := client.GetUser(ctx, &pb.GetUserRequest{UserId: 123})
    if err != nil {
        log.Fatalf("GetUser failed: %v", err)
    }

    log.Printf("Got user: %s (%s)", user.Name, user.Email)
}

Implementing the Server (Go)

package main

import (
    "context"
    "net"

    "google.golang.org/grpc"
    pb "github.com/myorg/user-service/proto"
)

// Implement the generated UserServiceServer interface
type userServiceServer struct {
    pb.UnimplementedUserServiceServer
    db UserRepository
}

func (s *userServiceServer) GetUser(ctx context.Context, req *pb.GetUserRequest) (*pb.User, error) {
    user, err := s.db.FindByID(ctx, req.UserId)
    if err != nil {
        // Use gRPC status codes — NOT HTTP codes
        return nil, status.Errorf(codes.NotFound, "user %d not found: %v", req.UserId, err)
    }
    return &pb.User{
        UserId:   user.ID,
        Name:     user.Name,
        Email:    user.Email,
        IsActive: user.Active,
    }, nil
}

func main() {
    lis, err := net.Listen("tcp", ":50051")
    if err != nil {
        log.Fatalf("failed to listen: %v", err)
    }

    s := grpc.NewServer()
    pb.RegisterUserServiceServer(s, &userServiceServer{db: NewUserRepo()})

    log.Println("gRPC server listening on :50051")
    s.Serve(lis)
}

gRPC Status Codes (Not HTTP Status Codes)

gRPC has its own status code system. Do NOT use HTTP 404 or 500 — use the gRPC codes. This is one of the most common mistakes when switching from REST.

  • codes.OK — Success
  • codes.InvalidArgument — Client sent bad input (like HTTP 400)
  • codes.NotFound — Resource not found (like HTTP 404)
  • codes.AlreadyExists — Duplicate creation attempt (like HTTP 409)
  • codes.PermissionDenied — Not authorized (like HTTP 403)
  • codes.Unauthenticated — No valid auth credentials (like HTTP 401)
  • codes.ResourceExhausted — Rate limit or quota exceeded (like HTTP 429)
  • codes.Unavailable — Server temporarily unavailable, safe to retry (like HTTP 503)
  • codes.DeadlineExceeded — Operation took too long
  • codes.Internal — Unexpected server error (like HTTP 500)

At the advanced level, gRPC's performance comes from understanding what HTTP/2 actually does differently, how to design backward-compatible proto schemas, and how to tune gRPC for production throughput and resilience.

HTTP/2 Multiplexing — Why It Matters

HTTP/1.1 has a fundamental limitation: only one request can be in-flight per TCP connection. Browsers work around this by opening 6 parallel connections per domain — wasteful and bounded. HTTP/2 introduces streams: multiple independent request/response pairs multiplexed over a single TCP connection.

HTTP/2 also compresses headers using HPACK — for gRPC, this matters because metadata (auth tokens, tracing headers) is sent with every call, and HPACK prevents that from becoming expensive.

Proto Schema Evolution Rules

Protobuf's backward compatibility is only guaranteed if you follow these rules. Break them and you'll get silent data corruption — old clients can't decode new messages or vice versa.

// SAFE CHANGES — backward and forward compatible:
// 1. Add new fields (with new field numbers)
// 2. Mark old fields as "reserved" instead of deleting
// 3. Rename a field (the field number is what matters, not the name)

// Example of a safe v2 evolution:
message User {
  int32  user_id   = 1;
  string name      = 2;
  string email     = 3;
  bool   is_active = 4;

  // SAFE: new field with new number
  string phone = 5;

  // SAFE: field 6 was deleted but its number is reserved
  // DO NOT reuse field number 6 for a new field — data corruption
  reserved 6;
  reserved "old_field_name";

  // SAFE: new fields in a nested message
  Address address = 7;
}

// UNSAFE CHANGES — will break compatibility:
// - Changing a field's type (e.g., int32 → int64 may work, string → int32 will not)
// - Reusing a field number for a different field
// - Removing a field without reserving its number

Deadlines and Cancellation

gRPC propagates context cancellation automatically across service boundaries. If a client sets a 5-second deadline and the downstream service hasn't responded, the context is cancelled — and if the downstream service is properly checking ctx.Err(), it stops work too. This prevents cascading resource waste.

// ALWAYS set a deadline. Without one, a hanging downstream
// service will hold your goroutine open indefinitely.
ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
defer cancel()

resp, err := client.GetUser(ctx, &pb.GetUserRequest{UserId: id})
if err != nil {
    st, ok := status.FromError(err)
    if ok && st.Code() == codes.DeadlineExceeded {
        // The call timed out — log and return a sensible error
        log.Printf("GetUser timed out for user %d", id)
        return nil, status.Error(codes.DeadlineExceeded, "user lookup timed out")
    }
    // Other error handling...
    return nil, err
}

Server-Streaming Pattern: Real-World Example

// Server: stream log entries as they arrive
func (s *logServiceServer) TailLogs(req *pb.TailLogsRequest, stream pb.LogService_TailLogsServer) error {
    watcher := s.logStore.Watch(req.ServiceName)
    defer watcher.Close()

    for {
        select {
        case entry := <-watcher.Entries():
            // Send one log entry — client receives each as it arrives
            if err := stream.Send(&pb.LogEntry{
                Timestamp: entry.Ts.Unix(),
                Level:     entry.Level,
                Message:   entry.Msg,
            }); err != nil {
                // Client disconnected or deadline exceeded
                return err
            }
        case <-stream.Context().Done():
            // Client cancelled or deadline exceeded — stop gracefully
            return stream.Context().Err()
        }
    }
}

// Client: iterate over the stream
stream, err := client.TailLogs(ctx, &pb.TailLogsRequest{ServiceName: "payment-service"})
if err != nil {
    log.Fatal(err)
}
for {
    entry, err := stream.Recv()
    if err == io.EOF {
        break // Server closed the stream
    }
    if err != nil {
        log.Fatalf("stream error: %v", err)
    }
    fmt.Printf("[%s] %s\n", entry.Level, entry.Message)
}
Performance Note

A single gRPC connection with HTTP/2 multiplexing can handle thousands of concurrent RPCs. Unlike REST, you should NOT create a new connection per request — create one connection pool at startup and reuse it. Most gRPC client libraries manage this automatically, but be aware that connection pool exhaustion behaves differently than HTTP/1.1 connection limits.

At the expert level, gRPC is not just a protocol — it's a platform for service communication with interceptors, health protocols, service mesh integration, and its own browser bridging story.

gRPC-Web: Bridging Browsers

Browsers cannot use gRPC natively because the Fetch API doesn't expose the HTTP/2 framing required for gRPC's binary protocol. gRPC-web is a specification and a JavaScript client that wraps gRPC messages in HTTP/1.1 (or HTTP/2 without trailers), relying on a proxy (Envoy or nginx-grpc-web) to translate to real gRPC on the backend.

# Envoy config: translate gRPC-web (from browsers) to gRPC (to backend)
filter_chains:
  - filters:
      - name: envoy.filters.network.http_connection_manager
        typed_config:
          http_filters:
            # This filter handles the gRPC-web → gRPC translation
            - name: envoy.filters.http.grpc_web
              typed_config:
                "@type": type.googleapis.com/envoy.extensions.filters.http.grpc_web.v3.GrpcWeb
            - name: envoy.filters.http.router
              typed_config:
                "@type": type.googleapis.com/envoy.extensions.filters.http.router.v3.Router

Practical limitation: gRPC-web only supports unary and server-streaming RPCs. Client-streaming and bidirectional streaming require a WebSocket upgrade and are not part of the gRPC-web standard.

gRPC Interceptors (Middleware)

Interceptors are gRPC's equivalent of HTTP middleware — they run around every RPC call. Common uses: authentication, logging, tracing, metrics, retry logic, and circuit breaking.

// Unary server interceptor: logging + auth
func authInterceptor(
    ctx context.Context,
    req interface{},
    info *grpc.UnaryServerInfo,
    handler grpc.UnaryHandler,
) (interface{}, error) {
    // Extract token from metadata
    md, ok := metadata.FromIncomingContext(ctx)
    if !ok {
        return nil, status.Error(codes.Unauthenticated, "missing metadata")
    }
    tokens := md["authorization"]
    if len(tokens) == 0 || !isValidToken(tokens[0]) {
        return nil, status.Error(codes.Unauthenticated, "invalid token")
    }

    // Inject user into context for downstream handlers
    ctx = context.WithValue(ctx, userKey, extractUser(tokens[0]))

    start := time.Now()
    resp, err := handler(ctx, req)

    // Log every RPC with timing
    log.Printf("method=%s duration=%s err=%v", info.FullMethod, time.Since(start), err)
    return resp, err
}

// Register interceptors when creating the server
s := grpc.NewServer(
    grpc.UnaryInterceptor(authInterceptor),
    grpc.StreamInterceptor(streamAuthInterceptor),
)

// Or chain multiple interceptors using go-grpc-middleware
import grpcmiddleware "github.com/grpc-ecosystem/go-grpc-middleware"
s := grpc.NewServer(
    grpc.UnaryInterceptor(grpcmiddleware.ChainUnaryServer(
        grpcrecovery.UnaryServerInterceptor(),  // panic recovery
        grpcopentelemetry.UnaryServerInterceptor(),  // distributed tracing
        authInterceptor,
    )),
)

Service Mesh Integration

gRPC integrates naturally with service meshes (Istio, Linkerd, Envoy). The mesh sidecar understands gRPC natively, enabling:

  • Method-level traffic management — route 10% of UserService.GetUser calls to a canary version, not just IP-level routing.
  • Automatic mTLS — the mesh handles certificate rotation and mutual authentication between services; your application code doesn't change.
  • Load balancing at the RPC level — traditional L4 load balancers are stream-unaware; they'll pin all HTTP/2 streams to one backend. Envoy understands HTTP/2 streams and balances at the RPC level.
  • Observability — method-level RED metrics (Rate, Errors, Duration) automatically from the mesh, without any instrumentation in service code.
# Istio VirtualService: route gRPC method to canary
apiVersion: networking.istio.io/v1alpha3
kind: VirtualService
metadata:
  name: user-service-vs
spec:
  hosts:
    - user-service
  http:
    # Route 10% of GetUser calls to the canary
    - match:
        - headers:
            content-type:
              prefix: application/grpc
          uri:
            prefix: /user.UserService/GetUser
      route:
        - destination:
            host: user-service
            subset: stable
          weight: 90
        - destination:
            host: user-service
            subset: canary
          weight: 10
    # All other gRPC methods go to stable
    - route:
        - destination:
            host: user-service
            subset: stable

gRPC Health Checking Protocol

gRPC defines a standard health checking protocol (grpc.health.v1). Kubernetes livenessProbe and Envoy health checks understand this protocol natively — no custom HTTP endpoint needed.

import "google.golang.org/grpc/health"
import healthpb "google.golang.org/grpc/health/grpc_health_v1"

// Register the health service when creating your gRPC server
healthServer := health.NewServer()
healthpb.RegisterHealthServer(s, healthServer)

// Mark a service as healthy
healthServer.SetServingStatus("user.UserService", healthpb.HealthCheckResponse_SERVING)

// Mark a service as not ready (during drain / rolling update)
healthServer.SetServingStatus("user.UserService", healthpb.HealthCheckResponse_NOT_SERVING)
# Kubernetes: use gRPC health probe for readiness
readinessProbe:
  grpc:
    port: 50051
    service: "user.UserService"  # empty string checks overall server health
  initialDelaySeconds: 5
  periodSeconds: 10
livenessProbe:
  grpc:
    port: 50051
  initialDelaySeconds: 10
  periodSeconds: 15

Error Handling with Rich Error Details

Standard gRPC errors only carry a code and a message. For production APIs, use the google.rpc.Status proto to attach structured error details — field violations, retry info, resource references.

import (
    "google.golang.org/grpc/status"
    "google.golang.org/grpc/codes"
    errdetails "google.golang.org/genproto/googleapis/rpc/errdetails"
)

// Return a rich validation error with per-field details
func validateAndCreateUser(req *pb.CreateUserRequest) error {
    st := status.New(codes.InvalidArgument, "request validation failed")

    // Add structured details the client can parse programmatically
    br := &errdetails.BadRequest{}
    if req.Email == "" {
        br.FieldViolations = append(br.FieldViolations, &errdetails.BadRequest_FieldViolation{
            Field:       "email",
            Description: "email is required",
        })
    }
    if len(req.Name) < 2 {
        br.FieldViolations = append(br.FieldViolations, &errdetails.BadRequest_FieldViolation{
            Field:       "name",
            Description: "name must be at least 2 characters",
        })
    }

    if len(br.FieldViolations) > 0 {
        st, _ = st.WithDetails(br)
        return st.Err()
    }
    return nil
}

Enterprise Use Case

Enterprise Use Case Uber — gRPC on the data plane across 4,000+ microservices

Context: Uber's engineering platform evolved from a monolith to hundreds, then thousands of microservices between 2014 and 2018. Their initial inter-service communication used REST+JSON, then Thrift (Facebook's binary RPC), before migrating their data plane to gRPC. As of 2019, gRPC handles the majority of internal Uber service-to-service communication.

The Problem with REST at Uber's scale: At 15 million trips per day, internal services like dispatch-service, pricing-service, eta-service, and supply-service exchange data with sub-10ms latency requirements. REST+JSON serialization overhead was measurable: for a 1 KB payload, JSON parse/serialization consumed ~0.8ms on average. Across 4,000 services making hundreds of calls each per request path, this compounded to significant latency and CPU burn.

How Uber uses gRPC:

  1. All new Uber microservices are required to expose a gRPC interface defined in a central proto registry — the "schema store". This ensures cross-team API contracts are versioned and discoverable.
  2. The dispatch service uses bidirectional streaming between driver and rider matching — streaming location updates and match candidates simultaneously over a single connection per active session.
  3. The ETA pipeline uses server streaming to push route recalculations to the client as new traffic data arrives, rather than polling every few seconds.
  4. Uber's internal load balancer ("NGINX → Envoy migration") was specifically driven by needing HTTP/2-aware L7 routing that could balance at the gRPC stream level, not the TCP connection level.
  5. All internal gRPC services use a standardized interceptor chain for auth (SVID from SPIFFE), distributed tracing (Jaeger), and rate limiting — injected at the platform level, invisible to service authors.

Measured Impact: After migrating the high-throughput dispatch data plane from Thrift to gRPC, Uber reported ~10x throughput improvement for the same hardware. Serialization CPU dropped by 65% for equivalent request volumes. The shared proto registry eliminated a class of schema drift bugs that had caused 6 production incidents in 2017.

What they kept REST for: External partner APIs (for third-party map integrations, fleet management vendors), the driver and rider mobile apps (using REST at the HTTP gateway layer, with gRPC only for internal traffic), and any service that needed to be callable by non-engineering teams without tooling setup.

Common Mistakes

Not Setting Deadlines — Goroutine Leaks in Production

In gRPC, if you don't set a deadline (via context.WithTimeout or context.WithDeadline), a call will wait indefinitely. If the downstream service hangs — network partition, GC pause, database lock — your goroutine sits blocked with no timeout. In a high-throughput service, this causes goroutine accumulation. At 1,000 req/sec with a hanging downstream, you can exhaust all goroutines (default limit: 10,000) in under 10 seconds, crashing the service.

Fix: Every gRPC call must use a context with a timeout. Set timeouts at the call site, not as global defaults. Propagate deadlines from the inbound request context to outbound calls so the entire call tree respects the client's deadline. Most gRPC frameworks support deadline propagation automatically if you pass the parent context.

Using HTTP Status Codes Instead of gRPC Status Codes

Teams migrating from REST habitually return HTTP-style errors: returning a JSON {"error": "not found", "code": 404} body in a gRPC response field, or mapping gRPC errors to HTTP codes in the wrong layer. gRPC has its own canonical 16-code status system (codes.NotFound, codes.InvalidArgument, etc.). Using gRPC status codes matters because: client libraries and proxies use them for automatic retry decisions (UNAVAILABLE is retryable, INVALID_ARGUMENT is not), Envoy and Istio use them for circuit breaker and outlier detection, and observability dashboards show method-level error rates by status code.

Fix: Always return status.Errorf(codes.NotFound, "...") from gRPC handlers. Never put error information in the response message body — that's a REST pattern that breaks gRPC tooling.

Breaking Proto Backward Compatibility (Silent Data Corruption)

The most dangerous proto mistake is reusing a deleted field's number. Suppose field 3 was string phone = 3. You delete it and later add int32 score = 3. Old services that still send a phone number will now serialize it into field 3. New services reading field 3 expect an int32, but get a string binary encoding — result: silent data corruption or a panic during deserialization. No compile error, no obvious runtime error, just wrong data silently computed upon.

Fix: Never reuse field numbers. When removing a field, always add reserved 3; and reserved "phone"; to the message definition. This causes a compile error if anyone accidentally tries to reuse the number or name. Additionally, run proto lint tooling (buf lint with FILE_SAME_TYPE rule) in CI to catch breaking changes before they reach production.

Interview Prep

gRPC is a high-performance RPC framework using Protocol Buffers for binary serialization and HTTP/2 for multiplexed transport, making it 3–10x more efficient than REST+JSON for internal service communication — at the cost of browser compatibility and debuggability.

2-Minute Structured Answer

"gRPC is a framework for service-to-service communication that solves the performance and type-safety limitations of REST+JSON. It uses Protocol Buffers — a binary serialization format where both sides share a schema file, so you never send field names over the wire, just field numbers and values. This makes payloads 3–10x smaller and serialization 5–7x faster than JSON. It runs over HTTP/2, which means multiple in-flight requests share one connection through multiplexing. Beyond unary request-response, gRPC natively supports server streaming, client streaming, and bidirectional streaming — which eliminates the polling patterns you'd need with REST. The main trade-off is that it's not browser-friendly and binary data is harder to debug. Most mature architectures use gRPC for internal microservice communication and REST for external or public APIs — Uber, for example, uses gRPC on their data plane across 4,000+ services, reporting a 10x throughput improvement over their previous RPC system."

  • How does gRPC handle authentication and authorization?
  • When would you choose REST over gRPC for an internal API?

Common Interview Mistakes

  • Saying "gRPC is always better than REST" — REST wins for external APIs, browser clients, human-readable debugging, and anywhere schema tooling would be a burden.
  • Confusing HTTP status codes with gRPC status codes — interviewers test this specifically because it reveals whether you've actually used gRPC or just read about it.
  • Forgetting that HTTP/2 load balancing is different — standard L4 load balancers pin all gRPC calls to one backend per connection; you need an L7-aware proxy like Envoy to balance at the RPC level.
  • Not mentioning deadlines — any experienced interviewer will ask "what happens if you don't set one?" — the answer is goroutine/thread leaks and eventual service crash.
