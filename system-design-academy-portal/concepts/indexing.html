Indexing

A data structure that lets the database engine find rows without scanning every single record — the difference between millisecond and multi-second queries.

Intermediate Data Databases Performance

Definition & Analogy

Simple Analogy

Imagine a 1,000-page textbook with no index. To find every mention of "B-tree" you must read every page from cover to cover. Now imagine the same book with a sorted index in the back: "B-tree — pages 42, 78, 391." You jump straight there. A database index is that back-of-book index — a separate, sorted data structure that maps values to the rows that contain them.

Without an index, the database performs a sequential scan — it reads every row in a table to find the matching ones. On a table with 100 million rows that can take seconds. An index pre-organises the values in a column (or columns) so the engine can skip to the right rows in O(log n) time instead of O(n). This is the single most impactful performance optimisation available in relational databases and most NoSQL stores.

The trade-off: indexes cost storage and slow down writes. Every INSERT, UPDATE, or DELETE must update not just the table but every index on it. Good index design is therefore a balance between read speed and write overhead.

The Story

Real-World Story — Slack Messages Table
Slack stores every message ever sent in a messages table. By mid-2023 that table held over 10 billion rows across all workspaces globally. Engineers needed to let users search their channel history — fetching the most recent 50 messages in a given channel is the most common query pattern in the application.
Without a targeted index, the query SELECT * FROM messages WHERE channel_id = 'C123' ORDER BY created_at DESC LIMIT 50 triggered a sequential scan across billions of rows. Even with the table partitioned by workspace, scans within a large workspace took 8 seconds. The search feature was unusable. P99 latency hit 12 seconds on active channels.
The team added a composite index: CREATE INDEX idx_messages_channel_time ON messages(channel_id, created_at DESC). The column order was deliberate — channel_id first because users always filter by channel (high cardinality, excellent selectivity), then created_at DESC so the index pages are pre-sorted in the direction the query needs. The database could now use an index-only scan to satisfy the entire query without touching the table heap.
Query time dropped from 8 seconds to 12 milliseconds — a 650x improvement. P99 latency normalised under 50ms. The index added ~80 GB of storage overhead but eliminated three database replicas that had been provisioned purely to absorb the sequential scan load, netting a significant infrastructure cost reduction.

Learn at Your Level

Beginner

Think of a phone book. It lists every subscriber in alphabetical order. To find "Smith, John" you don't read every entry — you flip to the S section, then to Sm, then scan a few entries. That's an index. A database index works the same way: it's a sorted copy of one or more columns, stored separately, that the engine can binary-search in fractions of a millisecond.

When you write SELECT * FROM users WHERE email = 'alice@example.com', the database has two options. Without an index: read every user row until it finds Alice's email. With an index on the email column: jump directly to Alice's row in O(log n) time. For a table with 1 million users, that's the difference between scanning 1,000,000 rows and scanning roughly 20 (since log₂ 1,000,000 ≈ 20).

Creating a basic index is a single SQL command and takes effect immediately:

-- Create an index on the email column of the users table
CREATE INDEX idx_users_email ON users(email);

-- Now this query uses the index instead of a full table scan
SELECT * FROM users WHERE email = 'alice@example.com';

-- Drop an index
DROP INDEX idx_users_email;

Key Terms

  • Sequential scan (Seq Scan) — Reading every row in a table from start to finish. Always works, but slow on large tables.
  • Index scan — Using the index to find the row locations, then fetching those specific rows from the table.
  • Index-only scan — The index contains all the data the query needs; no visit to the table required. Fastest option.
  • Cardinality — The number of distinct values in a column. High cardinality (e.g., email addresses) = very useful index. Low cardinality (e.g., a boolean is_active) = index rarely helps.

B-Tree Structure (Most Common Index)

Each leaf node stores the indexed value and a pointer to the actual row on disk. To find a value, the engine starts at the root, compares the search key, and navigates down the appropriate branch — typically 3–5 levels deep even for tables with hundreds of millions of rows.

There are several index types, each optimised for different access patterns. Choosing the right type — and understanding EXPLAIN ANALYZE — separates average from strong engineers.

Index Types

  • B-tree (default) — Balanced tree. Supports equality (=), range (<, >, BETWEEN), and ORDER BY. Used in PostgreSQL, MySQL, Oracle, SQL Server by default.
  • Hash — Stores a hash of each value. O(1) exact-match lookup. Cannot be used for range queries or sorting. Best in Redis and in-memory use cases. PostgreSQL supports hash indexes but they don't persist across crash recovery prior to v10.
  • GIN (Generalized Inverted Index) — Designed for composite values: full-text search (tsvector), JSONB columns, arrays. Slower to build and update but excellent for @> containment queries and @@ text search operators.
  • GiST (Generalized Search Tree) — Supports geometric data, IP ranges, fuzzy matching. Used for PostGIS spatial queries.
  • BRIN (Block Range Index) — Stores min/max values per disk block range. Tiny storage cost, only useful on naturally ordered columns (e.g., created_at in append-only tables).

Reading EXPLAIN ANALYZE

Before adding indexes, always run EXPLAIN ANALYZE to understand what the planner is doing and what it costs:

EXPLAIN ANALYZE
SELECT id, title, created_at
FROM tickets
WHERE user_id = 1042
ORDER BY created_at DESC
LIMIT 20;

-- Example output BEFORE index:
-- Seq Scan on tickets  (cost=0.00..28345.00 rows=47 width=52)
--                      (actual time=0.042..312.847 rows=47 loops=1)
--   Filter: (user_id = 1042)
--   Rows Removed by Filter: 1249953
-- Planning time: 0.612 ms
-- Execution time: 312.991 ms

-- After: CREATE INDEX idx_tickets_user_time ON tickets(user_id, created_at DESC);

-- Index Scan using idx_tickets_user_time on tickets
--                      (cost=0.43..84.21 rows=47 width=52)
--                      (actual time=0.028..0.347 rows=47 loops=1)
--   Index Cond: (user_id = 1042)
-- Planning time: 0.389 ms
-- Execution time: 0.412 ms

Key output nodes to recognise:

  • Seq Scan — Full table scan. Normal for small tables or when the planner estimates index won't help (e.g., fetching 60% of rows).
  • Index Scan — Uses the index to find rows, then fetches from the heap. cost is planner estimate; actual time is wall clock.
  • Index Only Scan — The index satisfies the query entirely — no heap access. Only possible if all selected columns are in the index (covering index).
  • Bitmap Index Scan + Bitmap Heap Scan — Mid-ground: builds a bitmap of matching row locations, then fetches heap pages in order. Efficient for moderate selectivity.

Composite Index: Column Order Matters

A composite index on (a, b, c) supports queries that filter on the leftmost prefix of columns. This is called the left-prefix rule:

-- Index: CREATE INDEX ON orders(status, user_id, created_at);

-- ✅ Uses index — filters on leftmost column
SELECT * FROM orders WHERE status = 'pending';

-- ✅ Uses index — filters on first two columns
SELECT * FROM orders WHERE status = 'pending' AND user_id = 42;

-- ✅ Uses index fully
SELECT * FROM orders WHERE status = 'pending' AND user_id = 42
  AND created_at > NOW() - INTERVAL '7 days';

-- ❌ Cannot use index — skips leftmost column
SELECT * FROM orders WHERE user_id = 42;

-- ❌ Cannot use index — skips both leading columns
SELECT * FROM orders WHERE created_at > NOW() - INTERVAL '7 days';

Rule of thumb for column order: Put the column with the highest selectivity (most distinct values) first — unless you always filter on a fixed column, in which case that fixed equality column goes first. The goal is to eliminate as many rows as possible at each tree level.

Advanced index techniques go beyond the default B-tree and unlock significant performance gains with minimal storage overhead.

Covering Index (Index-Only Scan)

A covering index includes every column a query needs — the engine never touches the table heap. In PostgreSQL, add non-key columns with INCLUDE:

-- Query: fetch recent tickets for a user (id, title, status, created_at)
SELECT id, title, status, created_at
FROM tickets
WHERE user_id = 1042
ORDER BY created_at DESC
LIMIT 20;

-- Standard index — still requires heap fetch for title and status
CREATE INDEX idx_tickets_user ON tickets(user_id, created_at DESC);

-- Covering index — includes all projected columns in the index leaf
-- No heap access needed → Index Only Scan
CREATE INDEX idx_tickets_user_covering
  ON tickets(user_id, created_at DESC)
  INCLUDE (id, title, status);

-- MySQL/MariaDB: equivalent is a composite index with all needed columns
CREATE INDEX idx_tickets_user_covering
  ON tickets(user_id, created_at, id, title, status);

Partial Index

A partial index only indexes rows that match a WHERE clause. This massively reduces index size when you only ever query a subset of rows:

-- 99% of orders are 'completed'. Queries only ever touch 'pending'.
-- Full index on status = waste of 99% of index space.

-- Partial index: only index pending orders
CREATE INDEX idx_orders_pending
  ON orders(created_at DESC)
  WHERE status = 'pending';

-- This query uses the partial index — fast, tiny index
SELECT * FROM orders WHERE status = 'pending' ORDER BY created_at DESC;

-- This query does NOT use the partial index (status = 'completed')
SELECT * FROM orders WHERE status = 'completed' ORDER BY created_at DESC;

-- Practical impact: table has 10M rows, 100K pending orders
-- Full index: 10M entries (~800 MB)
-- Partial index: 100K entries (~8 MB) — 99% smaller, fits in RAM

Index Selectivity and When Indexes Are Ignored

The query planner estimates whether using the index is cheaper than a sequential scan based on selectivity — what fraction of rows match the predicate. If the index would return more than ~10–15% of the table, a sequential scan is often faster (avoids random I/O):

-- Low selectivity: ~50% of users are active — planner ignores index
SELECT * FROM users WHERE is_active = true;   -- Seq Scan likely

-- High selectivity: very few users have this role — index used
SELECT * FROM users WHERE role = 'admin';     -- Index Scan likely

-- Force the planner to show its reasoning (PostgreSQL)
EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM users WHERE is_active = true;

-- Disable sequential scan to force index path for testing
SET enable_seqscan = off;
EXPLAIN ANALYZE SELECT * FROM users WHERE is_active = true;
SET enable_seqscan = on;

Index Trade-Offs

Dimension Benefit Cost
Read performance O(log n) lookup vs O(n) scan; 100–10000x faster on large tables Small per-query planning overhead; planner must evaluate index paths
Write performance No change to INSERT/UPDATE logic from application perspective Every write must update all indexes; 10 indexes = ~10x write amplification
Storage Enables smaller, cheaper instances B-tree indexes typically 10–30% of table size; covering indexes larger
Maintenance Automatic; no application changes needed Bloat over time; VACUUM/REINDEX needed in PostgreSQL for write-heavy tables
Lock behaviour CREATE INDEX CONCURRENTLY avoids full table lock in PostgreSQL Standard CREATE INDEX locks writes for the duration of the build
Architect Note

Always use CREATE INDEX CONCURRENTLY in production. The standard CREATE INDEX holds a ShareLock that blocks all writes until the index build completes — on a 100M-row table that could be 10+ minutes of write downtime. CONCURRENTLY takes ~2x longer but never blocks writes. The only downside is that it cannot be run inside a transaction block.

At expert level, index strategy intersects with write amplification, vacuum mechanics, full-text search architecture, and the economics of storage vs compute.

Index Maintenance Overhead (PostgreSQL Internals)

PostgreSQL uses MVCC (Multi-Version Concurrency Control). When a row is updated, the old version is kept in the heap until VACUUM removes it — but the index entries for the old version become "dead tuples." Accumulated dead tuples cause index bloat — the index grows larger than necessary, slowing scans and wasting I/O bandwidth.

-- Check index bloat (PostgreSQL)
SELECT
  schemaname,
  tablename,
  indexname,
  pg_size_pretty(pg_relation_size(indexrelid)) AS index_size,
  idx_scan,
  idx_tup_read,
  idx_tup_fetch
FROM pg_stat_user_indexes
ORDER BY pg_relation_size(indexrelid) DESC;

-- Find unused indexes (zero scans since last statistics reset)
SELECT schemaname, tablename, indexname, idx_scan
FROM pg_stat_user_indexes
WHERE idx_scan = 0
  AND schemaname NOT IN ('pg_catalog', 'information_schema')
ORDER BY pg_relation_size(indexrelid) DESC;

-- Rebuild a bloated index without a full lock
REINDEX INDEX CONCURRENTLY idx_orders_status;

-- Check table bloat and dead tuple ratio
SELECT relname, n_dead_tup, n_live_tup,
       round(n_dead_tup * 100.0 / nullif(n_live_tup + n_dead_tup, 0), 1) AS dead_ratio
FROM pg_stat_user_tables
WHERE n_dead_tup > 10000
ORDER BY n_dead_tup DESC;

Write-Heavy Workloads: When to Avoid or Defer Indexes

On write-heavy tables (event logs, IoT telemetry, audit trails), every additional index multiplies I/O cost. Strategies:

  • BRIN indexes — For append-only, time-ordered data. A BRIN on created_at is ~200 pages vs ~50,000 pages for a B-tree on a 100M-row table. Trade-off: requires data to be physically sorted on disk; random updates make it useless.
  • Deferred/partial index rebuild — Bulk-load data with indexes dropped, then rebuild: DROP INDEX; COPY ...; CREATE INDEX CONCURRENTLY;. Load speed improves 5–20x.
  • Fillfactor tuning — Set fillfactor=70 on B-trees for heavily updated tables: CREATE INDEX ... WITH (fillfactor=70). Leaves 30% of each page free so HOT (Heap-Only Tuple) updates can be accommodated without moving index entries.
  • Separate write and read replicas — Route all writes to primary (fewer indexes), add read replicas with richer indexes tuned for query patterns.

Full-Text Search: GIN Index vs Elasticsearch

PostgreSQL GIN indexes on tsvector columns handle moderate full-text search workloads well, without adding infrastructure complexity:

-- Add a tsvector column and GIN index for full-text search
ALTER TABLE tickets ADD COLUMN search_vector tsvector;

UPDATE tickets
SET search_vector = to_tsvector('english', coalesce(title,'') || ' ' || coalesce(description,''));

CREATE INDEX idx_tickets_fts ON tickets USING GIN(search_vector);

-- Automatically update search_vector on insert/update using a trigger
CREATE FUNCTION tickets_tsvector_update() RETURNS trigger AS $$
BEGIN
  NEW.search_vector := to_tsvector('english',
    coalesce(NEW.title,'') || ' ' || coalesce(NEW.description,''));
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER tsvector_update BEFORE INSERT OR UPDATE ON tickets
FOR EACH ROW EXECUTE FUNCTION tickets_tsvector_update();

-- Full-text search query
SELECT id, title, ts_rank(search_vector, query) AS rank
FROM tickets, to_tsquery('english', 'payment & failed') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 20;

When to choose Elasticsearch over PostgreSQL FTS:

  • Need relevance tuning with boosting, fuzzy matching, or custom analyzers
  • Full-text search is the primary access pattern (not a secondary feature)
  • Table has 100M+ rows with high update frequency (GIN index rebuild becomes expensive)
  • Need faceted search, aggregations, or auto-complete at scale
  • Multi-language stemming and language detection required
Architect Note — Index Strategy by Stage

Startup (0–1M rows): Add indexes when you first observe slow queries. Don't pre-optimise. EXPLAIN ANALYZE is your guide.

Growth (1M–100M rows): Audit unused indexes quarterly, eliminate them. Profile composite index column order against actual query patterns. Monitor dead tuple ratios — tune autovacuum if needed.

Scale (100M+ rows): Index strategy becomes a first-class architectural concern. Partial indexes, covering indexes, and BRIN for time-series become essential. Consider table partitioning (local indexes per partition are far smaller). Evaluate whether FTS should move to a dedicated search service.

Enterprise Use Case

Enterprise Use Case Slack — messages table, 10B+ rows, 650x query improvement

Context: Slack's messages table is one of the largest OLTP tables in production anywhere. Hundreds of millions of messages are written every day. The core query — "give me the last 50 messages in channel X" — runs billions of times per day across every active workspace. It must return in under 100ms to keep the real-time messaging UX responsive.

Index design: The composite index (channel_id, created_at DESC) was chosen deliberately. channel_id is placed first because the query always filters on it (high selectivity — each channel is a small fraction of the total table). created_at DESC is placed second so results come out pre-sorted in descending order, eliminating a sort step. The index is a covering index for the most common projection (id, channel_id, user_id, text, created_at), enabling index-only scans on the hot path.

Concrete numbers: Before the index: 8 second P50 query time. After: 12 milliseconds P50, under 50ms P99. Index size: ~80 GB on the primary. Table size: ~4 TB. Storage overhead: 2%. This is within budget — the 80 GB is in RAM on the primary for the hot channels, so it's effectively zero I/O cost in steady state.

Ongoing operations:

  • Autovacuum is tuned aggressively for the messages table: autovacuum_vacuum_scale_factor = 0.01 (trigger vacuum when 1% of rows are dead, vs the default 20%).
  • The table is partitioned by month. Each partition has its own index, keeping individual index sizes manageable.
  • New indexes are built with CREATE INDEX CONCURRENTLY — blocking writes on a 10B-row table would mean hours of outage.
  • Unused indexes are dropped quarterly — 14 legacy indexes were found and removed in one audit, reducing write amplification by ~25%.

Lessons learned: The most common mistake was adding indexes reactively without removing old ones. At peak, the messages table had 22 indexes — almost all writes triggered 22 B-tree updates. After the audit, 8 well-chosen indexes replaced 22 redundant ones. Write throughput increased 18% with zero read regression.

Common Mistakes

Over-Indexing: 10+ Indexes on One Table

Engineers add indexes to fix slow queries and never remove them. A table that accumulates 15 indexes pays a 15x write amplification penalty. Every INSERT must update 15 B-trees. Every UPDATE on any indexed column updates those trees. On write-heavy tables this compounds quickly — teams have observed 5–10x write throughput degradation from index sprawl alone.

Fix: Audit indexes with SELECT * FROM pg_stat_user_indexes WHERE idx_scan = 0. Remove any index with zero scans since the last statistics reset (typically 2+ weeks in production). Combine overlapping indexes into composite indexes.

Wrong Column Order in Composite Index

The index (status, user_id) and (user_id, status) are very different. If your most common query is WHERE user_id = ?, the first index cannot be used (left-prefix rule violation). Engineers frequently put the "logical" column first rather than the column that drives selectivity. Low-selectivity columns (like a status enum with 4 values) placed first create wide fan-out at the root — the index is nearly useless.

Fix: Design composite indexes around actual query patterns. Run EXPLAIN ANALYZE for every slow query and verify the index is actually used. Put the equality column first, then the range/sort column. Put high-selectivity columns before low-selectivity ones when both are equality filters.

Missing Index on Foreign Key Columns

In PostgreSQL, foreign key constraints do not automatically create indexes on the referencing column (MySQL InnoDB does create them). This means a JOIN on an unindexed foreign key triggers a sequential scan on the child table. On a large table, joining orders.user_id to users.id without an index on orders.user_id will scan the entire orders table for every user lookup.

Fix: Always index foreign key columns: CREATE INDEX ON orders(user_id). Run the following query after schema changes to catch missing FK indexes:

-- Find foreign key columns with no index (PostgreSQL)
SELECT
  tc.table_name,
  kcu.column_name,
  ccu.table_name AS foreign_table
FROM information_schema.table_constraints tc
JOIN information_schema.key_column_usage kcu
  ON tc.constraint_name = kcu.constraint_name
JOIN information_schema.referential_constraints rc
  ON tc.constraint_name = rc.constraint_name
JOIN information_schema.constraint_column_usage ccu
  ON rc.unique_constraint_name = ccu.constraint_name
WHERE tc.constraint_type = 'FOREIGN KEY'
  AND NOT EXISTS (
    SELECT 1 FROM pg_indexes
    WHERE tablename = tc.table_name
      AND indexdef LIKE '%' || kcu.column_name || '%'
  );

Interview Prep

A database index is a sorted auxiliary data structure that maps column values to row locations, enabling O(log n) lookups instead of O(n) sequential scans — at the cost of additional storage and write amplification.

2-Minute Structured Answer

"An index is a separate data structure — most commonly a B-tree — that stores a sorted copy of one or more columns alongside pointers to the corresponding table rows. When a query filters or sorts on an indexed column, the engine traverses the tree in O(log n) time instead of scanning the entire table. The main trade-off is write amplification: every insert, update, or delete must update all indexes. A practical example is a messages table — without an index on channel_id and created_at, fetching recent messages requires a full table scan that takes seconds. With a composite index on those two columns in that order, the same query takes 12 milliseconds. Choosing the right index requires understanding the left-prefix rule for composite indexes, the selectivity of your predicates, and whether a partial or covering index can further reduce overhead."

Common Interview Mistakes

  • Saying "just add an index" without mentioning write overhead — interviewers will probe trade-offs.
  • Claiming hash indexes support range queries — they don't. Hash indexes are equality-only.
  • Forgetting the left-prefix rule when designing composite indexes — a very common interview gotcha.
  • Ignoring index selectivity — indexes on low-cardinality columns (booleans, small enums) are often ignored by the query planner.
  • Not mentioning CREATE INDEX CONCURRENTLY for production — standard CREATE INDEX blocks writes.
