Pub/Sub

A messaging pattern where publishers broadcast events to a topic and multiple independent subscribers receive each message — without knowing anything about each other.

Intermediate Messaging Event-Driven Decoupling

Definition & Analogy

Simple Analogy

A newspaper publisher prints one edition and thousands of subscribers each get their own copy. The publisher doesn't know who reads it — and adding a new subscriber doesn't require the publisher to change anything. Contrast this with a direct phone call (point-to-point messaging), where you have to call each person individually.

Publish/Subscribe (Pub/Sub) is a messaging pattern where publishers send messages to a named topic — not to specific receivers. Subscribers declare interest in a topic and the messaging system delivers a copy of every message to every active subscriber.

This decoupling is the key architectural win. A publisher doesn't need to know how many consumers exist, what they do, or whether they are even online right now. Adding a new downstream service means subscribing to the existing topic — zero code changes to the publisher. This makes Pub/Sub the natural backbone for event-driven architectures where one event (like order.created) needs to trigger multiple independent workflows simultaneously.

Contrast this with a traditional message queue: in a queue, each message is delivered to exactly one consumer. In Pub/Sub, each message is delivered to every subscriber of the topic — this is called fan-out. If you need both behaviors (fan-out at the topic level, competing-consumers at the service level), the standard pattern is to put a dedicated queue behind each subscriber, so each service gets its own copy and can scale its workers independently.

The Story

Real-World Story
An e-commerce startup had an order-service that created orders. Every time a new order was placed, the engineering team wired up direct HTTP calls: order-service called billing-service, then called email-service, then called analytics-service. Three sequential HTTP calls before the order was confirmed. Total p99 latency: 1.4 seconds.
A product manager requested that warehouse picking start immediately after order creation. The engineer added a fourth HTTP call to warehouse-service. Then marketing wanted a loyalty-points service notified. Then compliance required an audit log. Each new requirement meant changing order-service, redeploying it, and hoping none of the downstream services were slow that day. One afternoon, email-service went down during a deployment. order-service started failing completely — all orders were broken because one non-critical notification service was unavailable.
The team introduced Pub/Sub. order-service now publishes a single order.created event to a topic and returns HTTP 201 immediately. Each downstream service subscribes to that topic independently: billing, email, analytics, warehouse, loyalty, audit — all receive their own copy of the event. order-service no longer knows or cares who is consuming it.
Order confirmation latency dropped from 1.4 seconds to 80ms. When email-service went down, orders kept processing — the email queue backed up and flushed when the service came back online. Adding a new downstream service required zero changes to order-service. The team added three new subscribers in the next quarter without a single order-service deployment.

Learn at Your Level

Beginner

Imagine a radio station broadcasting on FM 101.5. The station (publisher) transmits a signal to the airwaves (the topic). Anyone who tunes to that frequency (subscriber) receives the broadcast. The radio station doesn't call each listener individually — it just broadcasts, and any number of radios can pick it up simultaneously.

In software, Pub/Sub works the same way. A service (the publisher) sends a message to a named "channel" called a topic. Any number of other services (subscribers) that are tuned to that topic each receive a copy of the message. The publisher sends once; everyone subscribed gets it.

The big benefit? The publisher doesn't need to know anything about the subscribers. You can add a new subscriber any time without touching the publisher. This makes the system much easier to grow and change over time.

Key Terms

  • Publisher — The service that creates and sends a message to a topic.
  • Topic — A named channel. Messages published here are delivered to all subscribers of that topic.
  • Subscriber — A service that has registered interest in a topic and receives a copy of every message sent to it.
  • Fan-out — One message going to many subscribers. Pub/Sub is the canonical fan-out mechanism.
  • Event — The thing that happened, expressed as a message. e.g. order.created, user.signed-up, video.uploaded.

Pub/Sub vs Message Queues

These two patterns are often confused because both involve a broker delivering messages asynchronously. The key distinction is how many consumers receive each message.

Dimension Message Queue Pub/Sub Topic
Delivery One message → one consumer (competing consumers) One message → all subscribers (fan-out)
Use case Work distribution (only one worker should process each job) Event broadcasting (many services need to react to one event)
Example Image resize queue — only one worker resizes each image order.created — billing, email, analytics all react
Common pattern SQS, RabbitMQ queue SNS, Google Cloud Pub/Sub, Kafka topic

In practice, Pub/Sub fan-out and queues are often combined: an SNS topic fans out to multiple SQS queues, and each service pulls from its own queue using competing consumers. This gives you both fan-out (every service gets the event) and work distribution (each service can scale its workers independently).

Push vs Pull Delivery

Pub/Sub systems deliver messages in one of two modes:

  • Push — The broker sends (HTTP POST) messages to a subscriber's endpoint as they arrive. Low latency, simple subscriber code. The subscriber must be available and able to handle the incoming rate. Good for: real-time webhooks, low-volume events, subscribers that need immediate processing.
  • Pull — The subscriber calls the broker to fetch a batch of messages. The subscriber controls the pace. Good for: batch processing, bursty workloads, subscribers that need to control concurrency.

Code Example: Publishing an Event

# Google Cloud Pub/Sub — Publishing
from google.cloud import pubsub_v1
import json

publisher = pubsub_v1.PublisherClient()
topic_path = publisher.topic_path("my-project", "order.created")

def publish_order_created(order_id: str, user_id: str, total: float):
    message = {
        "event": "order.created",
        "order_id": order_id,
        "user_id": user_id,
        "total": total,
        "timestamp": "2025-06-03T12:00:00Z"
    }
    # Encode to bytes and publish
    data = json.dumps(message).encode("utf-8")

    # Attributes for server-side filtering
    future = publisher.publish(
        topic_path,
        data=data,
        event_type="order.created",   # filterable attribute
        region="us-east-1"
    )
    message_id = future.result()      # blocks until confirmed
    print(f"Published message {message_id}")

publish_order_created("ord-789", "usr-42", 149.99)

Code Example: Subscribing (Pull)

# Google Cloud Pub/Sub — Pull Subscriber (billing-service)
from google.cloud import pubsub_v1
import json

subscriber = pubsub_v1.SubscriberClient()
# Each service has its OWN subscription (its own independent cursor)
subscription_path = subscriber.subscription_path(
    "my-project", "order.created-billing-sub"
)

def handle_message(message):
    payload = json.loads(message.data.decode("utf-8"))
    order_id = payload["order_id"]
    total = payload["total"]

    try:
        charge_card(order_id, total)     # idempotent — safe to retry
        message.ack()                    # tell broker: done, don't redeliver
    except Exception as e:
        print(f"Billing failed: {e}")
        message.nack()                   # broker will redeliver after ack_deadline

streaming_pull_future = subscriber.subscribe(
    subscription_path, callback=handle_message
)
print(f"Listening for messages on {subscription_path}...")
streaming_pull_future.result()           # blocks indefinitely

Ordering Guarantees

Most Pub/Sub systems are unordered by default. A message published first may be delivered after a message published second. This is a deliberate design decision — strict ordering requires coordination that hurts throughput and availability.

  • Google Cloud Pub/Sub — Unordered by default. Ordering keys (message attribute) enable in-order delivery within a key, at the cost of lower throughput for that key.
  • AWS SNS — Standard topics: unordered. FIFO topics: strict order within message group, up to 300 msg/sec per group.
  • Kafka — Ordered within a partition. Messages with the same partition key always land on the same partition, preserving order per key.

If your subscribers need strict ordering (e.g. account balance updates must process oldest-first), either use Kafka with a consistent partition key or use FIFO-mode SNS/SQS and accept the throughput limit.

At-Least-Once Delivery and Idempotency

All major Pub/Sub systems guarantee at-least-once delivery: a message will be delivered to your subscriber at least once, but possibly more than once. Duplicates occur when:

  • Your subscriber processes the message but crashes before calling ack() — the broker retries after the ack deadline expires.
  • A network partition causes the broker to lose the ack and redeliver.
  • The broker itself duplicates the message due to internal retries.

The consequence: subscribers must be idempotent. Processing the same message twice must produce the same result as processing it once. Standard patterns:

  • Deduplication ID — Store the message ID in a Redis set or DB table. Before processing, check if the ID exists. Skip if seen before.
  • Upsert instead of insert — Use database upsert semantics: if the record already exists, update it to the same value rather than inserting a duplicate.
  • Idempotent keys — Design business logic to be safe on replay. Charging a card "for order-789" twice should fail gracefully on the second attempt, not double-charge.

Message Filtering

When a topic carries multiple event types, subscribers can use attribute-based filtering to receive only relevant messages — without the broker needing separate topics.

# Google Cloud Pub/Sub — Subscription with filter
# Only delivers messages where event_type = "order.created"
# analytics-service also subscribes but filters for "order.completed"

from google.cloud import pubsub_v1

subscriber = pubsub_v1.SubscriberClient()

# Create subscription with a server-side filter
# (done once during provisioning, not in application code)
subscription = subscriber.create_subscription(
    request={
        "name": "projects/my-project/subscriptions/billing-order-created",
        "topic": "projects/my-project/topics/orders",
        "filter": 'attributes.event_type = "order.created"',
        "ack_deadline_seconds": 30,
    }
)
print(f"Created filtered subscription: {subscription.name}")

Dead Letter Topics

When a subscriber consistently fails to process a message (e.g. malformed payload it cannot parse), the broker retries indefinitely by default — filling up the subscription backlog and blocking all other messages. A dead letter topic (DLT) is the safety valve: after N failed delivery attempts, the broker moves the message to a separate DLT where engineers can inspect, repair, and replay it.

# Configure dead letter topic on a subscription (Terraform / SDK example)
# After 5 failed delivery attempts, message goes to dead-letter topic

subscription = subscriber.create_subscription(
    request={
        "name": "projects/my-project/subscriptions/billing-order-created",
        "topic": "projects/my-project/topics/orders",
        "dead_letter_policy": {
            "dead_letter_topic": "projects/my-project/topics/orders-dead-letter",
            "max_delivery_attempts": 5,
        },
        "retry_policy": {
            "minimum_backoff": {"seconds": 10},
            "maximum_backoff": {"seconds": 600},   # exponential backoff up to 10 min
        },
    }
)

Pub/Sub Trade-offs

Aspect Strength Weakness Mitigation
Decoupling Publisher needs zero knowledge of subscribers Hard to trace end-to-end flow across services Distributed tracing (trace ID in message attributes)
Fan-out One publish, N consumers — free scale-out N copies of every message stored in broker Set retention windows; use message filtering
Reliability At-least-once delivery with broker durability Duplicate delivery requires idempotent consumers Deduplication table in Redis/DB
Ordering High throughput when order not required No ordering guarantee by default Ordering keys (GCP), FIFO mode (SNS), Kafka partitions
Backpressure Pull model lets subscriber control pace Push model overwhelms slow subscribers Use pull, or configure flow control on push endpoint
Architect Note

The SNS+SQS fan-out pattern on AWS is the most commonly used Pub/Sub architecture in enterprise systems. SNS provides the topic (fan-out), and each subscriber has its own SQS queue (independent backlog, independent scaling, independent DLQ). This means a slow subscriber can't block a fast one — each queue grows at its own rate. Always provision a DLQ on the SQS queues, not just on SNS.

Google Cloud Pub/Sub Internals

Google Cloud Pub/Sub is one of the world's largest managed messaging systems, processing hundreds of billions of messages per day. Understanding its architecture explains its guarantees and limitations.

  • Global distribution — Topics are global by default. Messages can be published from any region and delivered to subscribers in any region. The broker replicates messages across multiple zones for durability before acknowledging the publish.
  • Message storage — Messages are stored in Bigtable-backed storage with configurable retention (default 7 days, up to 31 days). Subscriptions maintain an independent read cursor — each subscription sees all messages independently.
  • Exactly-once delivery (limited) — GCP Pub/Sub now supports exactly-once delivery within a region on a subscription. The broker tracks which messages have been successfully acknowledged and guarantees no redelivery, at the cost of higher latency (~50ms overhead per message). Only available on pull subscriptions.
  • Flow control — The client library implements local flow control: maximum outstanding messages and maximum outstanding bytes. This prevents the pull subscriber from pulling faster than it can process, avoiding OOM crashes on burst traffic.

AWS SNS + SQS Fan-Out Pattern

# Terraform: SNS topic → multiple SQS queues (fan-out pattern)

resource "aws_sns_topic" "order_created" {
  name = "order-created"
}

# Each subscriber gets its own SQS queue
resource "aws_sqs_queue" "billing_queue" {
  name                       = "billing-order-created"
  visibility_timeout_seconds = 30
  message_retention_seconds  = 86400   # 1 day

  redrive_policy = jsonencode({
    deadLetterTargetArn = aws_sqs_queue.billing_dlq.arn
    maxReceiveCount     = 5
  })
}

resource "aws_sqs_queue" "billing_dlq" {
  name = "billing-order-created-dlq"
}

# Subscribe billing queue to SNS topic
resource "aws_sns_topic_subscription" "billing_subscription" {
  topic_arn = aws_sns_topic.order_created.arn
  protocol  = "sqs"
  endpoint  = aws_sqs_queue.billing_queue.arn

  filter_policy = jsonencode({
    event_type = ["order.created"]
  })
}

# SQS policy — allow SNS to send to queue
resource "aws_sqs_queue_policy" "billing_policy" {
  queue_url = aws_sqs_queue.billing_queue.id
  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Effect    = "Allow"
      Principal = { Service = "sns.amazonaws.com" }
      Action    = "sqs:SendMessage"
      Resource  = aws_sqs_queue.billing_queue.arn
      Condition = {
        ArnEquals = { "aws:SourceArn" = aws_sns_topic.order_created.arn }
      }
    }]
  })
}

# Repeat aws_sqs_queue + subscription for email-service, analytics-service, etc.

Multi-Region Pub/Sub

For global applications with strict data-residency requirements, Pub/Sub topology must be carefully designed:

  • Replicated topics — GCP Pub/Sub topics are global; publishers in any region write to the same topic. AWS SNS is regional — you must create separate topics per region and replicate events using EventBridge or custom Lambda forwarders.
  • Message schema registry — At scale, schema drift breaks subscribers silently. Use a schema registry (Confluent Schema Registry, AWS Glue Schema Registry) to enforce backward-compatible evolution. Avro or Protobuf over plain JSON enables schema validation at publish time.
  • Observability — Monitor subscription backlog (oldest_unacked_message_age), not just message count. A backlog of 10K messages is healthy if your subscriber clears it in seconds; it's critical if the oldest message is 4 hours old.

Production Operations Checklist

  • Dead letter topic configured on every subscription (max 5 delivery attempts)
  • Alerting on DLT message count > 0 (any DLT message warrants investigation)
  • Alerting on subscription backlog oldest_unacked_message_age > 5 minutes
  • Subscribers implement idempotent message handling with deduplication
  • Message schema versioned and validated against schema registry
  • Distributed trace ID injected into message attributes at publish time
  • Retention period set explicitly (not default) per topic criticality
  • Message ordering requirements documented and enforcement mechanism confirmed
  • Load tested: subscriber can clear backlog at 2x peak publish rate
  • Runbook for DLT replay documented and tested

Enterprise Use Case

Enterprise Use Case YouTube — video.uploaded at massive scale

Context: YouTube users upload 500 hours of video every minute. Each uploaded video must trigger: transcoding into multiple resolutions (360p, 720p, 1080p, 4K), thumbnail generation, content moderation (CSAM scanning, copyright matching), notification to subscribers of the channel, and analytics ingestion. That is five independent, heavyweight workflows — each with its own scaling requirements and SLAs.

How Pub/Sub is used: When the upload service confirms a video file is stored in GCS, it publishes a single video.uploaded event to a Google Cloud Pub/Sub topic. The event payload contains the video ID, uploader ID, GCS path, and metadata. Five services subscribe to the topic, each with their own subscription and independent processing backlog.

  1. transcoding-service — pulls the event, launches a Dataflow job per video. p99 completion time: 8 minutes for 1080p. Has its own DLT for videos that fail transcoding.
  2. thumbnail-service — generates 3 candidate thumbnails within 30 seconds. Uses pull subscription with flow control (max 500 concurrent jobs).
  3. moderation-service — highest priority subscription. Uses push delivery with a dedicated endpoint for sub-second latency. Video is restricted from public view until moderation completes.
  4. notification-service — fans out channel subscriber notifications via FCM/email. Separate rate-limiting layer to avoid notification spam during bulk uploads.
  5. analytics-service — batch pull subscription. Processes messages in 1-minute windows. Tolerates hours of backlog without impacting other services.

Scale numbers: 500M+ messages/day on this topic. Peak throughput: ~6,000 messages/second during YouTube premiere events. The transcoding-service subscription maintains a backlog of up to 4 million messages during peak hours — this is expected and healthy. Backlog metrics trigger auto-scaling of Dataflow workers, not alerts.

Key engineering decisions: Each subscription has independent ack deadline configured to match the subscriber's expected processing time (moderation: 60s, transcoding: 900s). Ordering keys are unused — transcoding jobs are independent and order doesn't matter. Schema is Protobuf, validated via GCP Schema Registry, with strict backward-compatibility enforced at publish time.

Common Mistakes

Assuming Messages Arrive in Order

This is the most common and most dangerous Pub/Sub misconception. Standard Pub/Sub topics deliver messages out of order. If you publish events 1, 2, 3 — subscribers may receive them as 2, 1, 3 or 3, 1, 2. Engineers who assume ordering write bugs that are subtle and hard to reproduce.

The failure mode: a user.subscription-cancelled event arrives before user.subscription-started. The billing service processes the cancellation first (finds no active subscription, does nothing) and then processes the start (creates a subscription that never gets cancelled). The user is charged indefinitely.

Fix: Design message handlers to be order-independent where possible. If strict ordering is required, use Kafka with a consistent partition key, or GCP Pub/Sub ordering keys / AWS SNS FIFO. Include a timestamp in every event and apply last-writer-wins logic in the subscriber.

Not Handling Duplicate Delivery

At-least-once delivery means your subscriber will receive the same message more than once — this is not a bug, it is a guarantee. Teams that don't design for this ship bugs like double-charging customers, sending duplicate emails, or inserting duplicate analytics records.

The failure mode: billing-service charges the card successfully but the network hiccups before message.ack() is called. The broker's ack deadline expires and the message is redelivered. billing-service charges the card a second time.

Fix: Store the message_id in a Redis set (TTL matching your message retention period). At the start of every handler: check if this message ID has been seen before, and skip if so. Alternatively, design business operations as idempotent upserts — "set balance to X" is safe on replay; "add X to balance" is not.

No Dead Letter Topic for Unprocessable Messages

Without a dead letter topic, a message that your subscriber cannot process (malformed JSON, unexpected field, downstream timeout) will be retried indefinitely until the retention period expires. During that time it consumes retry budget, generates alerts, and depending on your subscriber logic, may block processing of later messages.

The failure mode: a schema change in the publisher sends a field as a string instead of an integer. Every subscriber that tries to cast it to int throws an exception, nacks the message, and the broker retries it every 10 seconds — generating thousands of errors per hour. The message is eventually dropped when retention expires, silently. No one investigates.

Fix: Configure a dead letter topic on every subscription with max_delivery_attempts = 5. Set an alert on any message arriving in the DLT. Maintain a runbook describing how to inspect, fix, and replay DLT messages. Treat the DLT as a first-class production signal — every message there is a bug.

Interview Prep

Pub/Sub decouples publishers from subscribers through a topic — publishers broadcast events without knowing who receives them, and multiple subscribers independently receive every message, enabling fan-out and loose coupling in distributed systems.

2-Minute Structured Answer

"Pub/Sub is a messaging pattern where publishers send events to a named topic, and any number of subscribers independently receive a copy of each message. The key property is decoupling: the publisher doesn't know who's consuming, which means you can add new subscribers without changing the publisher. The classic use case is an e-commerce system — when an order is created, a single event fans out to billing, email, analytics, and warehouse simultaneously, rather than the order service calling each downstream service sequentially. The main trade-offs to know: first, Pub/Sub is generally unordered, so if you need strict message ordering you need Kafka partitions or FIFO mode. Second, delivery is at-least-once, so subscribers must be idempotent — you will receive duplicates. Third, you need a dead letter topic to handle unprocessable messages, otherwise broken messages retry forever. In practice at AWS, the standard pattern is SNS fan-out to multiple SQS queues — SNS gives you the topic abstraction, SQS gives each service its own independent backlog and dead-letter queue."

Common Interview Mistakes

  • Saying "Pub/Sub guarantees delivery in order" — it does not, unless you explicitly opt into ordering mechanisms with significant trade-offs.
  • Conflating Pub/Sub with message queues — interviewers specifically test whether you know the fan-out vs competing-consumers distinction.
  • Forgetting idempotency — any senior interviewer will ask how you handle duplicates; saying "it won't happen" is a red flag.
  • Not mentioning dead letter topics when discussing reliability — omitting DLTs suggests you haven't operated Pub/Sub in production.
  • Saying "just use Kafka for everything" — Kafka is powerful but operationally heavy. Managed Pub/Sub (GCP, SNS+SQS) is often the right default and interviewers notice when you can articulate the trade-off.
