The TicketFlow Enterprise Scenario

🏢 Meet TicketFlow — Your Learning Context

Every page in this portal uses the same fictional company: TicketFlow, a high-traffic ticketing platform processing millions of events per month. The company runs a microservices architecture across AWS EKS, and every concept you learn is applied directly to a realistic problem their engineering team faces.

The team and their services:

  • user-service — Node.js: authentication, profiles, preferences
  • ticket-service — Spring Boot (Java): ticket inventory, reservations, pricing
  • payment-service — Python (FastAPI): payment processing, refunds, fraud detection
  • order-service — Go: order lifecycle, confirmation, history
  • notification-service — Node.js: email, SMS, and push via third-party APIs
  • api-gateway — NGINX: routing, rate limiting, SSL termination

Their infrastructure stack:

  • Container registry: Nexus at nexus.ticketflow.io:5000
  • Kubernetes: AWS EKS (multi-AZ, production + staging clusters)
  • CI/CD: GitHub Actions → Nexus → Helm → Argo CD
  • Monitoring: Prometheus + Grafana + Loki + Jaeger

By the end of this portal, you will have seen every part of this stack in depth — and you will understand how every technology choice connects to a business requirement.

Architect's Perspective on This Path

🏛️ Architect Note

This roadmap is deliberately bottom-up. Many engineers try to learn Kubernetes without understanding Docker, or try to learn Argo CD without understanding Kubernetes deployments. The result is fragile knowledge: they can follow a tutorial but cannot reason through a production incident or a design decision.

The sequence matters:

  • Docker first — containers are the fundamental unit. Until you understand how images, layers, namespaces, and cgroups work, Kubernetes is magic you cannot debug.
  • Networking and volumes before K8s — K8s networking is a superset of Docker networking. Understanding bridge networks, DNS, and port mapping makes the Kubernetes network model obvious rather than mysterious.
  • CI/CD before GitOps — Argo CD is a better CI/CD. You will only appreciate why if you have first experienced the limitations of imperative pipelines.
  • Observability before security — You cannot secure what you cannot see. Understanding metrics, logs, and traces first makes security hardening decisions concrete and measurable.
  • Architecture last — Architecture patterns are only valuable when you understand the tools well enough to reason about their trade-offs. Pattern names without deep tool knowledge are just vocabulary.

The engineers who command the highest salaries and the most influence are not the ones who have memorised the most kubectl commands. They are the ones who can look at a failing system and reason from first principles about what is wrong — and why a particular architectural decision made two years ago is causing today's incident.

That is the goal of this portal. Not just the "how," but the "why."

Prerequisites Checklist

Before starting, make sure your environment is ready. You only need a few things to follow every lab in this portal:

  • Docker Desktop installed (docs.docker.com/desktop) — required for pages 01–08
  • A terminal with bash or zsh (macOS/Linux native; Windows users use WSL2 or Git Bash)
  • A code editor — VS Code recommended (free, cross-platform, excellent Docker and Kubernetes extensions)
  • kubectl installed (kubernetes.io/docs/tasks/tools) — required for pages 09+
  • A local Kubernetes cluster — enable Kubernetes in Docker Desktop, or install kind (kind.sigs.k8s.io) or minikube
  • Helm CLI installed (helm.sh) — required for pages 16+
  • Git installed and a GitHub account — required for CI/CD and GitOps pages
  • Basic Linux command-line familiarity (ls, cd, cat, grep, curl) — assumed throughout
