Docker Commands

The complete Docker CLI reference — from pulling your first image to debugging running containers in production.

Story

Learning Docker commands is like learning to drive. At first the dashboard is overwhelming — all those buttons and levers. But you only need a handful to get started: ignition, steering, accelerator, brake. The advanced controls (ABS, traction control, cruise) reveal themselves when you actually need them. This page covers your everyday commands first, then the advanced ones you'll eventually reach for.

Core Commands by OS

Docker's CLI is cross-platform, but path syntax and shell conventions differ between Linux/macOS and Windows. The examples below use the same Docker commands — only the shell syntax around them changes.

Running a container with port mapping and a volume

# Run nginx, map host port 8080 to container port 80,
# and bind-mount the current directory's "html" folder as the web root
docker run -d \
  --name my-nginx \
  -p 8080:80 \
  -v $(pwd)/html:/usr/share/nginx/html:ro \
  nginx:1.25

# Run a Node app with an environment variable and a named volume
docker run -d \
  --name my-node-app \
  -p 3000:3000 \
  -e NODE_ENV=production \
  -v app-data:/app/data \
  my-node-app:latest

docker run Flags Reference

The docker run command is the most flag-rich command in Docker. Below is a complete annotated example followed by a reference table for each flag.

docker run \
  -d \                            # detached mode (background)
  -p 8080:80 \                    # host:container port mapping
  -p 443:443 \                    # multiple ports — add one -p per port
  -v myvolume:/data \             # named volume mount
  -v $(pwd)/config:/etc/config \  # bind mount (host directory → container path)
  -e DB_HOST=postgres \           # environment variable
  -e DB_PORT=5432 \               # another environment variable
  --name my-app \                 # assign a name (otherwise random)
  --network mynetwork \           # attach to a custom bridge network
  --restart unless-stopped \      # restart policy
  --memory 512m \                 # hard memory limit — OOMKill if exceeded
  --cpus 0.5 \                    # CPU limit (0.5 = half a CPU core)
  --user 1001 \                   # run as non-root user (UID)
  --read-only \                   # read-only root filesystem
  --rm \                          # auto-remove container when it stops
  nginx:1.25
Flag Type Description Production recommended?
-d Boolean Detached mode — runs container in the background. Without it the container attaches to your terminal and Ctrl+C kills it. Yes — always
-p host:container String Publish a container port to the host. Format: hostPort:containerPort. Add multiple -p flags for multiple ports. Yes — required for external access
-v name:/path String Named volume mount. Docker manages the storage. Survives container removal. Use for persistent data like databases. Yes — for stateful services
-v /host:/container String Bind mount. Maps a host directory directly into the container. Good for dev (live code reload) but ties container to host filesystem. Dev only — avoid in production
-e KEY=VALUE String Set an environment variable. Use for runtime config: DB host, ports, feature flags. Never put secrets in -e — they're visible in docker inspect. Yes — for non-secret config
--name String Assign a human-readable name. Without this, Docker generates a random name. Named containers are addressable by name on custom networks. Yes — makes debugging easier
--network String Attach to a Docker network. Containers on the same network can reach each other by name. Default is the bridge network. Yes — for multi-container apps
--restart String Restart policy. Options: no (default), always, unless-stopped, on-failure. unless-stopped is safest for long-running services. Yes — use unless-stopped
--memory String Hard memory limit (e.g. 512m, 2g). If the container exceeds it, Linux OOM Killer kills the process (exit code 137). Maps to Kubernetes resource limit. Yes — always set in production
--cpus Float CPU quota in CPU cores. 0.5 = 50% of one core, 2 = two full cores. Maps directly to Kubernetes CPU limit. Yes — always set in production
--user String Run as a specific user (UID or username). Default is root inside container which is a security risk. Use non-root for production workloads. Yes — security hardening
--read-only Boolean Mount the container's root filesystem as read-only. Forces explicit volume declarations for writable paths. Prevents malware from writing to the container. Yes — security hardening
--rm Boolean Auto-remove the container when it exits. Great for one-off tasks and dev workflows. Do not combine with --restart. Dev/CI only — not for services

docker logs Deep Dive

docker logs shows everything the container process wrote to stdout and stderr. It's almost always the first command you run when something goes wrong.

# Follow logs in real time (-f = follow, like tail -f)
docker logs -f my-app

# Show last 100 lines
docker logs --tail 100 my-app

# Show logs with timestamps
docker logs -t my-app

# Show logs since a specific time
docker logs --since 2024-01-15T10:00:00 my-app

# Show logs since 30 minutes ago
docker logs --since 30m my-app

# Show logs between two timestamps
docker logs --since 2024-01-15T10:00:00 --until 2024-01-15T11:00:00 my-app

# Follow logs with timestamps (most useful for debugging)
docker logs -f -t my-app

# Pipe to grep to filter for errors only
docker logs my-app 2>&1 | grep ERROR

# Save logs to a file for further analysis
docker logs my-app > app.log 2>&1
Beginner Note

The 2>&1 at the end of a command redirects stderr (file descriptor 2) to stdout (file descriptor 1). Docker logs shows both stdout and stderr mixed together in the terminal, but when you pipe or redirect, stderr goes to a different stream by default. Adding 2>&1 ensures you capture everything — both normal output and error messages — in your pipe or file. If you only see some errors but not others, make sure you're capturing both streams.

docker exec Deep Dive

docker exec is your entry point into a running container. It creates a new process inside the container's existing namespaces without stopping or modifying the container.

# Get an interactive bash shell (Debian/Ubuntu-based images)
docker exec -it my-app bash

# Get sh (for Alpine-based images that don't have bash installed)
docker exec -it my-app sh

# Run a single non-interactive command
docker exec my-app cat /etc/hosts

# Run as root even if the container uses a non-root user
docker exec -u root -it my-app bash

# Check all environment variables set inside the container
docker exec my-app env

# List all running processes inside the container
docker exec my-app ps aux

# Check network connectivity from inside the container
docker exec my-app curl -s http://other-service:8080/health

# Check disk usage from inside the container
docker exec my-app df -h

# Tail an application log file that lives inside the container
docker exec my-app tail -f /app/logs/app.log
Architect Note

In production Kubernetes, docker exec has a direct equivalent: kubectl exec -it <pod-name> -- bash. Same concept — reach inside a running container without modifying it or redeploying it. This is a non-negotiable production debugging skill. Understanding docker exec first makes kubectl exec immediately intuitive when you reach Kubernetes. The -it flags mean the same thing in both tools: -i keeps stdin open, -t allocates a pseudo-TTY for an interactive session.

docker inspect

docker inspect returns the complete configuration and state of a container or image as a JSON object. It's the source of truth for everything Docker knows about a running container.

# Full JSON inspection of a container — outputs everything
docker inspect my-app

# Get just the container's IP address (using Go template syntax)
docker inspect -f '{{range .NetworkSettings.Networks}}{{.IPAddress}}{{end}}' my-app

# Get all environment variables set in the container
docker inspect -f '{{range .Config.Env}}{{.}}{{"\n"}}{{end}}' my-app

# Get mounted volumes formatted as pretty JSON
docker inspect -f '{{json .Mounts}}' my-app | python3 -m json.tool

# Get the container's restart policy
docker inspect -f '{{.HostConfig.RestartPolicy.Name}}' my-app

# Inspect an image (not a container) — works the same way
docker inspect nginx:1.25

# Get image creation date and compressed size in bytes
docker inspect -f 'Created: {{.Created}}, Size: {{.Size}}' nginx:1.25

docker stats

docker stats streams live resource usage — CPU, memory, network, and disk I/O — for running containers. It's Docker's built-in top command.

# Live stats for all running containers (streams until Ctrl+C)
docker stats

# Live stats for a specific container
docker stats my-app

# One-time snapshot — print stats once and exit (no streaming)
docker stats --no-stream

# Custom format — useful for scripting and CI dashboards
docker stats --format "table {{.Container}}\t{{.CPUPerc}}\t{{.MemUsage}}"

What each column means

Column What it shows What to look for
CONTAINER ID Shortened container ID Cross-reference with docker ps
NAME Container name Human-readable identifier
CPU % Percentage of host CPU used Sustained high % means app needs more CPU or has a runaway loop
MEM USAGE / LIMIT Current memory used vs. the set limit Approaching the limit means OOMKill is imminent
MEM % Memory used as a percentage of the limit Above 90% is a warning sign; 100% triggers OOMKill
NET I/O Total network bytes sent/received High values on a service with no expected traffic suggests a loop or DDoS
BLOCK I/O Total disk bytes read/written High write I/O on a database signals an I/O bottleneck
PIDS Number of processes inside the container Rapidly growing PID count suggests a fork bomb or process leak
Production Tip

High MEM % approaching the limit means OOMKilled is coming — either raise the memory limit or investigate the memory leak before the container crashes. Sustained high CPU % means your application is CPU-bound: add more replicas (horizontal scaling) or increase the --cpus limit. If BLOCK I/O is consistently high on a database container, you're hitting an I/O bottleneck — consider using a faster storage driver, moving to an SSD-backed volume, or adding read replicas to distribute the load.

Image Management

Images are the blueprints for containers. Managing them well keeps your registry clean, your builds reproducible, and your disk usage under control.

# List all local images with details
docker images

# List with custom formatted output — useful for scripting
docker images --format "table {{.Repository}}:{{.Tag}}\t{{.Size}}\t{{.CreatedAt}}"

# Pull a specific version (always use explicit tags — never rely on :latest in production)
docker pull node:18-alpine

# Pull by digest — immutable reference, guaranteed to always be the same image
docker pull nginx@sha256:a4c4106dd4f8b66be4f8b28b7a4aeda3e4f12e2d1b0f6b6d4d7a9b83d9d1d2c4

# Tag an image for a private registry
docker tag myapp:v1.2.0 nexus.company.com:5000/myapp:v1.2.0
docker tag myapp:v1.2.0 nexus.company.com:5000/myapp:latest  # only tag latest on release

# Authenticate with a private registry, then push
docker login nexus.company.com:5000
docker push nexus.company.com:5000/myapp:v1.2.0

# Remove a specific image (containers using it must be stopped first)
docker rmi myapp:v1.2.0

# Remove dangling images — untagged intermediate build layers
docker image prune

# Remove ALL unused images (not just dangling) — reclaims significant disk space
docker image prune -a

# See the full layer history of an image (what each Dockerfile instruction added)
docker history nginx:1.25 --no-trunc

Cleanup Commands

Docker accumulates disk usage fast — stopped containers, old image layers, build cache, and unused volumes. On CI servers this can fill a disk within days without a cleanup strategy.

Production Tip

Always run docker system df before any prune command to see exactly what's using space. On CI servers, add a post-build cleanup step to every pipeline. This prevents the "disk full at 3 AM" incident. A good CI cleanup strategy: remove the specific built image after pushing it, then run docker image prune -f to clear dangling layers. Reserve the nuclear options for manual maintenance windows.

# See disk usage before pruning — shows containers, images, volumes, build cache
docker system df

# Remove all stopped containers
docker container prune

# Remove dangling images (untagged, not referenced by any container)
docker image prune

# Remove ALL unused images (not just dangling — reclaims much more space)
docker image prune -a

# Remove all unused volumes
# CAREFUL — this deletes data! Only run if you understand what's in the volumes.
docker volume prune

# Remove all unused networks
docker network prune

# Remove stopped containers, unused networks, dangling images, and build cache
# Does NOT touch volumes by default
docker system prune

# Also removes all unused images (not just dangling)
docker system prune -a

# Nuclear option — also removes named volumes (PERMANENT DATA LOSS)
docker system prune -a --volumes
Common Mistake

Running docker system prune -a --volumes on a production server deletes ALL container data including database volumes. Postgres data, Redis snapshots, uploaded files — gone permanently with no undo. Only run volume pruning on developer machines or CI build servers where you are certain no persistent data matters. Production servers should never need this command. If you find yourself considering it on a production host, stop and think about what data you're about to destroy.

Behind the Scenes

Understanding what happens under the hood helps you debug unexpected behaviour and reason about Docker's limitations.

How does docker exec work?

When you run docker exec -it my-app bash, Docker calls the nsenter system call on Linux to enter the existing namespaces of the running container process. The key namespaces it enters are the network namespace (so the new process shares the container's network interfaces and routing) and the mount namespace (so it sees the container's filesystem, including all its mounted volumes).

The new process gets its own separate PID — it does not share the container's process tree — but it shares everything else. This is why you can check network connectivity with curl inside an exec session, and why filesystem changes you make inside exec persist: you're genuinely inside the container's environment, not a simulation of it.

This also means exec sessions count against the container's resource limits. If the container has a 512 MB memory limit and your exec session starts a memory-heavy tool, it can push the container over the limit and trigger an OOMKill.

Where are Docker files stored on disk?

On Linux, Docker stores everything under /var/lib/docker. The storage is layered using the overlay2 storage driver (the default since Docker 1.13). Image layers live at /var/lib/docker/overlay2 — each image layer is a directory containing just the filesystem diff for that layer. Containers add a thin writable layer on top.

Named volumes are stored at /var/lib/docker/volumes. Each named volume gets its own subdirectory. On Windows with WSL2, the equivalent path is inside the WSL2 virtual machine, accessible via \\wsl$\docker-desktop-data\data\docker.

This directory can fill up a disk on CI servers in days if builds are frequent and cleanup is absent. Use docker system df to see a breakdown of what's consuming space. The build cache alone can be tens of gigabytes on an active CI agent. Add docker builder prune -af to your CI cleanup scripts to reclaim build cache after each run.

What is the difference between docker stop and docker kill?

docker stop sends SIGTERM to the container's main process (PID 1) first. SIGTERM is a polite request to shut down — the application can catch this signal and perform a graceful shutdown: finish handling in-flight HTTP requests, close database connections, flush write buffers to disk, deregister from service discovery. After a configurable timeout (10 seconds by default, changeable with --time), if the process has not exited, Docker escalates to SIGKILL.

docker kill sends SIGKILL immediately, bypassing the grace period entirely. SIGKILL cannot be caught or ignored by the application — the kernel terminates the process immediately. In-flight requests are dropped, write buffers may not be flushed, and connection pools are torn down without cleanup.

Always prefer docker stop for normal operations. Use docker kill only when a container is genuinely frozen and not responding to SIGTERM, which usually indicates a bug in the application's signal handling. Well-written applications listen for SIGTERM and shut down cleanly within the 10-second window.

Common Mistakes

Mistake 1 — Forgetting the -d flag

Running docker run nginx without -d attaches the container to your terminal. You'll see logs streaming, and the moment you press Ctrl+C, the container stops. New users often think the container is running in the background — it isn't. Always add -d for services you want running persistently: docker run -d -p 8080:80 nginx.

Mistake 2 — Port conflict

If you see Error starting userland proxy: listen tcp4 0.0.0.0:8080: bind: address already in use, another process is already using port 8080 on your host. To find it on Linux/macOS: lsof -i :8080 or ss -tlnp | grep 8080. On Windows: netstat -ano | findstr 8080, then check Task Manager for the PID. Either stop the conflicting process, or map to a different host port: -p 9090:80.

Mistake 3 — Container exits immediately

You run docker run -d myapp, then docker ps shows nothing. The container started and stopped in milliseconds. This almost always means the main process in the container exited — either it crashed, or the Dockerfile CMD/ENTRYPOINT doesn't run a long-lived process. The fix: run docker logs <container-id> immediately (use docker ps -a to find the stopped container ID) to see the exit message. Common causes: missing environment variable the app required on startup, wrong file path in CMD, or an application error on boot.

Mistake 4 — Exec into the wrong container

When you have multiple containers with similar names (e.g. my-app-1, my-app-2, my-app-staging), it's easy to exec into the wrong one and spend 20 minutes confused why config changes aren't taking effect. Always run docker ps first to confirm container names and IDs before exec. Tab completion helps: most shells with Docker completion will autocomplete container names from docker ps output.

Mistake 5 — Cleaning containers but not images

docker rm removes containers but leaves the images on disk. On CI servers that build dozens of images per day, this fills the disk quickly with old image layers. The symptom is a slow build server that eventually refuses to start new builds with "no space left on device". The fix: add image cleanup to every CI pipeline. At minimum, run docker image prune -f after each build. Better: use docker system prune -af in a scheduled cleanup job that runs nightly on CI agents.

Hands-On Lab

Lab — Container Debugging Workflow

Scenario: A container is misbehaving. Practice the full debugging sequence — the same sequence you'd use in production when someone pages you at 2 AM.

# Step 1: Start a "broken" nginx with a custom environment variable
docker run -d --name debug-nginx \
  -p 8090:80 \
  -e NGINX_ENV=production \
  nginx:1.25

# Step 2: Confirm it's running
docker ps

# Step 3: Check the logs for any startup errors
docker logs debug-nginx

# Step 4: Enter the container and inspect its internals
docker exec -it debug-nginx bash
# Inside the container, run these one by one:
cat /etc/nginx/nginx.conf
ls -la /etc/nginx/conf.d/
env | grep NGINX
ps aux
exit

# Step 5: Inspect full container metadata (first 80 lines of JSON)
docker inspect debug-nginx | python3 -m json.tool | head -80

# Step 6: Check live resource usage — one-time snapshot
docker stats debug-nginx --no-stream

# Step 7: Copy the default nginx config out to your host for review
docker cp debug-nginx:/etc/nginx/conf.d/default.conf ./nginx-default.conf
cat ./nginx-default.conf

# Step 8: Clean up after the lab
docker stop debug-nginx
docker rm debug-nginx

# Verify the container is gone
docker ps -a | grep debug-nginx

After completing this lab you should be comfortable moving through the full debugging cycle: check if running → check logs → enter container → inspect metadata → review resource usage → extract files → clean up. This cycle is the same regardless of which application is inside the container.

Interview Questions

These are the Docker CLI questions that come up repeatedly in DevOps and platform engineering interviews. Click each question to reveal the full answer.

Q1: How do you debug a container that keeps crashing (the Docker equivalent of CrashLoopBackOff)?

Start with docker logs <container-name> to see the last output before the exit. This usually tells you exactly what went wrong — a missing environment variable, a failed database connection, or an unhandled exception.

If the container exits so fast that it's gone before you can check logs, use docker ps -a to find its ID, then docker logs <container-id>. The container must still exist (not been docker rm'd) for logs to be accessible.

If the logs don't tell you enough and you need to poke around inside, override the entrypoint to prevent the app from starting: docker run -it --entrypoint sh <image-name>. This drops you into a shell where you can manually run the application and see exactly what fails.

Finally, check the exit code with docker inspect <container-id> --format '{{.State.ExitCode}}'. Exit code 1 means the application itself returned a non-zero exit status (application error). Exit code 137 means the process was killed with SIGKILL — almost always OOMKilled (out of memory). Exit code 139 means segmentation fault. Exit code 126 means the entrypoint command wasn't executable. Fix the root cause — don't just restart and hope.

Q2: What is the difference between COPY/ADD in a Dockerfile and docker cp?

COPY and ADD are Dockerfile build-time instructions. They run when you execute docker build and copy files from your build context (the directory on your host containing the Dockerfile) into the image being constructed. The result is baked permanently into the image layer. Every container started from that image will have those files.

docker cp is a runtime command. It copies files between your host filesystem and a running (or stopped) container without modifying the image. Changes made with docker cp live only in that specific container's writable layer — they disappear when the container is removed and are not reflected in any other containers started from the same image.

Use docker cp exclusively for debugging and emergency access — grabbing a log file, injecting a temporary config fix. Never build a deployment workflow around docker cp. The correct deployment workflow is: change the source code → rebuild the image → redeploy the container.

Q3: How do you run a Docker container as a different user?

There are three places where the user can be set, in order of precedence:

Dockerfile (recommended for production): Add USER 1001 or USER appuser near the end of your Dockerfile, after installing dependencies. Every container started from the image will use that user by default. This is the most robust approach because the intent is captured in the image definition.

docker run flag (runtime override): Use docker run --user 1001:1001 image to override the user at container start. The format is UID:GID. This works but the override can be forgotten, making it less reliable than the Dockerfile approach.

docker exec flag: Use docker exec -u root -it my-app bash to override the user for a specific exec session. This is useful for debugging — you can enter as root to inspect files that the application user can't read, without changing how the container itself runs.

Always prefer the Dockerfile USER instruction for production workloads. Running as root inside a container is a significant security risk — if an attacker escapes the container namespace, they may gain root on the host.

Q4: What exactly does docker system prune remove, and what does it leave alone?

docker system prune (without flags) removes four categories of resources: stopped containers (any container not in the running state), dangling images (untagged images not referenced by any container — typically intermediate build layers), unused networks (networks not connected to any running container), and build cache (cached build layers from previous docker build runs).

It does not remove: running containers, named images that are tagged (even if no container is using them), or named volumes. This is intentional — removing volumes would delete persistent data.

docker system prune -a additionally removes unused images that are tagged but not referenced by any running container — this reclaims significantly more space and is safe on CI servers where images are rebuilt regularly.

Adding --volumes to either form will also remove named volumes. This is permanent and irreversible — every database, every uploaded file, every stateful piece of data stored in volumes is gone. Only use --volumes on machines where you are absolutely certain no persistent data exists, such as a fresh CI runner being reset between jobs.

Q5: How do you limit container memory and CPU, and how do these map to Kubernetes?

Memory: --memory 512m sets a hard memory limit. If the container's memory usage exceeds this, the Linux kernel's Out-of-Memory (OOM) Killer terminates a process inside the container — the container exits with code 137. --memory-reservation 256m sets a soft limit: Docker tries to keep memory usage below this under memory pressure, but it's not enforced strictly.

CPU: --cpus 0.5 limits the container to using at most 0.5 CPU cores (50% of one core). Internally, Docker translates this into a Linux CFS (Completely Fair Scheduler) quota. --cpu-shares 512 is relative weighting (default 1024) — it only matters when the system is under CPU contention and does not set a hard cap.

Kubernetes mapping: These flags map directly to the Kubernetes container resource spec. --memory 512m → resources.limits.memory: 512Mi. --memory-reservation 256m → resources.requests.memory: 256Mi. --cpus 0.5 → resources.limits.cpu: 500m (millicores). Understanding the Docker flags first makes the Kubernetes resource model immediately intuitive because they implement the same underlying Linux kernel mechanisms.
