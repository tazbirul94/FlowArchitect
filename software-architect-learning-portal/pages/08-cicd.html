CI/CD Pipelines

Build, test, scan, push, and deploy automatically — from a git push to a running container in Kubernetes without a single manual step.

The Story Behind CI/CD

🏭 The Assembly Line That Changed Manufacturing

In 1913, Henry Ford introduced the moving assembly line at Highland Park. Before it, building a car required a team of skilled workers to assemble the whole vehicle from scratch — each one slightly different, each one taking days. Ford's insight was radical: break the process into repeatable, automated steps. Each station does one thing consistently. The car moves through the line and emerges at the other end fully assembled, every time, in 93 minutes instead of 12 hours.

Software deployment was stuck in the pre-assembly-line era for most of the industry's history. A developer would finish a feature, hand it to QA, QA would test it manually for days or weeks, an ops engineer would coordinate a weekend deployment window, manually copy files to servers, cross fingers, and hope nothing broke. Every deployment was a unique, high-risk event.

CI/CD is the assembly line for software. Every commit enters the line: it gets compiled, tested, scanned for vulnerabilities, packaged into a container image, pushed to a registry, and deployed to Kubernetes — automatically, in minutes, every single time. The process is identical for every commit. Deployment becomes a non-event. Teams deploy dozens of times per day instead of once per month.

What is CI/CD?

🥪 Beginner Analogy

Imagine you run a bakery. Every time a baker finishes a new recipe, you want it tested, packaged, and on the shelf — without the baker having to personally carry it to the store, talk to the packaging team, file paperwork, and schedule shelf space. CI/CD is the conveyor belt system that does all of that automatically the moment the baker says "done". The baker just bakes. The system handles everything else.

In software: you write code and push it to git. The CI/CD system automatically compiles it, runs all your tests, builds a Docker image, checks it for security vulnerabilities, pushes it to your container registry, updates your deployment configuration, and triggers a rollout in Kubernetes. You just write code. The pipeline handles the rest.

CI/CD stands for two related but distinct practices that together form a continuous delivery loop:

  • Continuous Integration (CI) — Every commit to the shared repository triggers an automated build and test suite. The goal is to detect integration problems early, when they are cheap to fix. "Integration" refers to the act of merging individual developers' work together — CI does this constantly, rather than in a big-bang merge at the end of a sprint.
  • Continuous Delivery (CD) — Every commit that passes CI is automatically packaged and deployed to a staging environment, and is always in a releasable state. Deployment to production may still require a manual approval gate, but it is a single button click.
  • Continuous Deployment — A stricter form of CD where every commit that passes all automated checks is automatically deployed to production with no human intervention. Requires extremely high confidence in your test suite and monitoring.

The tools we focus on in this page are GitHub Actions (YAML-based workflows triggered by git events) and GitLab CI (equivalent functionality, different syntax). Both integrate with Nexus, Trivy, Helm, and Argo CD to form the complete TicketFlow pipeline.

Technical Deep Dive

The Five Stages of a Production-Grade Pipeline

A mature CI/CD pipeline for a containerized microservice consists of five sequential stages, each acting as a quality gate that must pass before the next stage executes:

  1. Build & Test — Compile the application, run unit tests, run integration tests. Produces a tested binary or JAR. If any test fails, the pipeline halts immediately and the developer is notified. No artifact is produced for a failed build.
  2. Container Build — Build a Docker image from the Dockerfile, using the tested binary from stage 1. Apply semantic version tags based on the git tag or SHA. The image is not yet pushed to a shared registry.
  3. Security Scan — Run Trivy (or Snyk) against the freshly built image. Check for known CVEs in OS packages and application dependencies. If critical or high vulnerabilities are found, fail the pipeline. The image is not pushed until it passes the security gate.
  4. Push to Registry — Push the scanned, tagged image to Nexus at nexus.ticketflow.io:5000. The image now has an immutable address: nexus.ticketflow.io:5000/ticket-service:1.4.2-abc1234.
  5. Deploy — Update the Helm values.yaml with the new image tag and commit it to the GitOps repository. Argo CD detects the change and syncs the Kubernetes cluster to the new desired state. The deployment is triggered by a git commit — not by a direct kubectl apply.

Semantic Versioning in CI

Image tags must be deterministic, unique, and informative. The TicketFlow team uses a two-component tag strategy:

  • Release tags — When a git tag like v1.4.2 is pushed, the pipeline produces nexus.ticketflow.io:5000/ticket-service:1.4.2. This is the production-ready image.
  • Branch/SHA tags — Every commit to main produces nexus.ticketflow.io:5000/ticket-service:main-abc1234 where abc1234 is the short git SHA. This is used for staging deployments.
  • Never just :latest — :latest is pushed as a convenience alias only, never used in any Kubernetes manifest or Helm values file. See Common Mistakes below.

GitHub Actions Key Concepts

  • Workflow — A YAML file in .github/workflows/. Triggered by events (push, pull_request, schedule, workflow_dispatch). Can contain multiple jobs.
  • Job — A set of steps that run on the same runner (VM or container). Jobs run in parallel by default; use needs: to express dependencies and enforce sequential execution.
  • Step — An individual task within a job. Either a shell command (run:) or a pre-built action (uses:). Steps within a job always run sequentially and share the same filesystem.
  • Runner — The machine that executes jobs. GitHub provides hosted runners (ubuntu-latest, windows-latest, macos-latest). Organizations can run self-hosted runners inside their own infrastructure for security or performance.
  • Secrets — Encrypted key-value pairs stored at the repository or organization level. Referenced as ${{ secrets.MY_SECRET }} in workflows. Never appear in logs.
  • Outputs — Values produced by one step and consumed by later steps or jobs. Used to pass image tags, version numbers, or build artifacts between stages.

Enterprise Scenario: TicketFlow

🏢 TicketFlow's Before and After

Before CI/CD: The TicketFlow team had 6 microservices. Deploying a new version of ticket-service required: a developer manually building a JAR, writing an email to the ops team with the JAR attached, an ops engineer logging into the server, stopping the old process, copying the JAR, updating a config file by hand, restarting the service, and posting a message in Slack confirming the deploy. This took 2–4 hours per service. Deployments happened every 2 weeks, accumulating 50+ changes per release. Rollback meant reversing all of it manually.

After CI/CD: A developer merges a PR to main. GitHub Actions triggers automatically. In 8 minutes: the Spring Boot app is compiled, all 247 unit tests pass, a Docker image is built, Trivy finds 0 critical CVEs, the image is pushed to Nexus as nexus.ticketflow.io:5000/ticket-service:main-abc1234, the Helm values file is updated with the new tag, Argo CD detects the git change and performs a rolling update on EKS, and Slack posts a deploy notification. The developer is already working on the next feature. Total human effort: writing the code and clicking "Merge".

TicketFlow's pipeline covers all 6 services:

  • user-service (Node.js) — npm ci && npm test → Docker build → Trivy → Nexus push
  • ticket-service (Spring Boot) — mvn package → Docker build → Trivy → Nexus push
  • payment-service (Python) — pytest --cov → Docker build → Trivy → Nexus push
  • order-service (Go) — go test ./... → Docker build → Trivy → Nexus push
  • notification-service (Node.js) — npm ci && npm test → Docker build → Trivy → Nexus push
  • api-gateway (NGINX) — config validation → Docker build → Trivy → Nexus push

Full Pipeline Code

GitHub Actions — ticket-service Full Pipeline

This is the complete .github/workflows/ci-cd.yml for TicketFlow's ticket-service (Spring Boot). It implements all five pipeline stages with proper secrets handling, semantic versioning, and Argo CD deployment triggering.

# .github/workflows/ci-cd.yml
# Full CI/CD pipeline for ticket-service (Spring Boot)
# Triggers: push to main (staging deploy) and git tags v* (production deploy)

name: ticket-service CI/CD

on:
  push:
    branches: [ main ]
    tags:
      - 'v*'           # e.g. v1.4.2 — triggers production release
  pull_request:
    branches: [ main ] # CI only — no deploy on PRs

env:
  REGISTRY: nexus.ticketflow.io:5000
  IMAGE_NAME: ticket-service
  HELM_REPO: ticketflow/gitops   # GitOps repo containing Helm values

jobs:
  # ─────────────────────────────────────────────────────────────────
  # JOB 1: Build and Test
  # Compile the Spring Boot app and run all tests
  # ─────────────────────────────────────────────────────────────────
  build-and-test:
    name: Build & Test
    runs-on: ubuntu-latest

    steps:
      - name: Checkout source code
        uses: actions/checkout@v4
        with:
          fetch-depth: 0  # Full history needed for semantic versioning

      - name: Set up JDK 17
        uses: actions/setup-java@v4
        with:
          java-version: '17'
          distribution: 'temurin'
          cache: maven

      - name: Build with Maven and run tests
        run: mvn package --no-transfer-progress -B

      - name: Upload test results
        if: always()  # Upload even if tests fail — helps debug CI failures
        uses: actions/upload-artifact@v4
        with:
          name: test-results
          path: target/surefire-reports/
          retention-days: 7

      - name: Upload built JAR
        uses: actions/upload-artifact@v4
        with:
          name: app-jar
          path: target/ticket-service-*.jar
          retention-days: 1

  # ─────────────────────────────────────────────────────────────────
  # JOB 2: Determine version tag
  # Produces a deterministic image tag from git context
  # ─────────────────────────────────────────────────────────────────
  version:
    name: Determine Version
    runs-on: ubuntu-latest
    outputs:
      image_tag: ${{ steps.tag.outputs.image_tag }}
      semver: ${{ steps.tag.outputs.semver }}
      is_release: ${{ steps.tag.outputs.is_release }}

    steps:
      - name: Checkout
        uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Compute image tag
        id: tag
        run: |
          SHORT_SHA=$(git rev-parse --short=7 HEAD)

          if [[ "${GITHUB_REF}" == refs/tags/v* ]]; then
            # Production release: strip the leading 'v' from the git tag
            SEMVER="${GITHUB_REF_NAME#v}"   # e.g. v1.4.2 → 1.4.2
            IMAGE_TAG="${SEMVER}"
            IS_RELEASE="true"
          else
            # Branch build: branch-name + short SHA
            BRANCH="${GITHUB_REF_NAME//\//-}"   # replace / with - in branch names
            IMAGE_TAG="${BRANCH}-${SHORT_SHA}"  # e.g. main-abc1234
            SEMVER="${IMAGE_TAG}"
            IS_RELEASE="false"
          fi

          echo "image_tag=${IMAGE_TAG}" >> "$GITHUB_OUTPUT"
          echo "semver=${SEMVER}" >> "$GITHUB_OUTPUT"
          echo "is_release=${IS_RELEASE}" >> "$GITHUB_OUTPUT"
          echo "Computed image tag: ${IMAGE_TAG}"

  # ─────────────────────────────────────────────────────────────────
  # JOB 3: Build Docker image, scan, and push to Nexus
  # ─────────────────────────────────────────────────────────────────
  docker-build-scan-push:
    name: Docker Build → Trivy Scan → Push
    runs-on: ubuntu-latest
    needs: [ build-and-test, version ]

    steps:
      - name: Checkout source code
        uses: actions/checkout@v4

      - name: Download built JAR
        uses: actions/download-artifact@v4
        with:
          name: app-jar
          path: target/

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3
        # Buildx enables cache-from/cache-to features for faster builds

      - name: Log in to Nexus registry
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ secrets.NEXUS_USERNAME }}
          password: ${{ secrets.NEXUS_PASSWORD }}

      - name: Build Docker image (not yet pushed)
        uses: docker/build-push-action@v5
        with:
          context: .
          file: ./Dockerfile
          push: false          # Build only — we scan before pushing
          load: true           # Load into local Docker daemon for Trivy
          tags: |
            ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ needs.version.outputs.image_tag }}
            ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:latest
          cache-from: type=gha   # Use GitHub Actions cache for layer caching
          cache-to: type=gha,mode=max

      - name: Run Trivy vulnerability scanner
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ needs.version.outputs.image_tag }}
          format: 'table'
          exit-code: '1'         # Fail the pipeline if vulnerabilities found
          ignore-unfixed: true   # Only fail on CVEs that have a fix available
          vuln-type: 'os,library'
          severity: 'CRITICAL,HIGH'

      - name: Run Trivy — produce SARIF report for GitHub Security tab
        if: always()   # Generate report even if scan fails
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ needs.version.outputs.image_tag }}
          format: 'sarif'
          output: 'trivy-results.sarif'
          ignore-unfixed: true
          vuln-type: 'os,library'
          severity: 'CRITICAL,HIGH'

      - name: Upload Trivy SARIF to GitHub Security
        if: always()
        uses: github/codeql-action/upload-sarif@v3
        with:
          sarif_file: 'trivy-results.sarif'

      # Only push after the security scan passes
      - name: Push image to Nexus
        uses: docker/build-push-action@v5
        with:
          context: .
          file: ./Dockerfile
          push: true
          tags: |
            ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ needs.version.outputs.image_tag }}
            ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:latest
          cache-from: type=gha

      - name: Image summary
        run: |
          echo "### Docker Image Pushed" >> $GITHUB_STEP_SUMMARY
          echo "- **Registry:** ${{ env.REGISTRY }}" >> $GITHUB_STEP_SUMMARY
          echo "- **Image:** ${{ env.IMAGE_NAME }}:${{ needs.version.outputs.image_tag }}" >> $GITHUB_STEP_SUMMARY
          echo "- **Is release:** ${{ needs.version.outputs.is_release }}" >> $GITHUB_STEP_SUMMARY

  # ─────────────────────────────────────────────────────────────────
  # JOB 4: Update Helm values.yaml and trigger Argo CD sync
  # Updates the GitOps repo — Argo CD picks up the change automatically
  # ─────────────────────────────────────────────────────────────────
  deploy:
    name: Update Helm values & Trigger ArgoCD
    runs-on: ubuntu-latest
    needs: [ docker-build-scan-push, version ]
    # Only deploy on pushes to main or version tags — not on PRs
    if: github.event_name != 'pull_request'

    steps:
      - name: Checkout GitOps repository
        uses: actions/checkout@v4
        with:
          repository: ${{ env.HELM_REPO }}
          token: ${{ secrets.GITOPS_PAT }}   # Personal Access Token with repo write permission
          path: gitops

      - name: Update image tag in Helm values
        run: |
          cd gitops

          # Determine environment directory based on release type
          if [[ "${{ needs.version.outputs.is_release }}" == "true" ]]; then
            VALUES_FILE="environments/production/ticket-service/values.yaml"
          else
            VALUES_FILE="environments/staging/ticket-service/values.yaml"
          fi

          # Update the image tag using yq (YAML processor)
          # Safer than sed — handles YAML structure correctly
          yq e '.image.tag = "${{ needs.version.outputs.image_tag }}"' -i "${VALUES_FILE}"
          yq e '.image.repository = "${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}"' -i "${VALUES_FILE}"

          echo "Updated ${VALUES_FILE}:"
          cat "${VALUES_FILE}"

      - name: Commit and push values change
        run: |
          cd gitops
          git config user.name "github-actions[bot]"
          git config user.email "github-actions[bot]@users.noreply.github.com"

          git add .
          git diff --staged --quiet || git commit -m \
            "chore(deploy): update ticket-service to ${{ needs.version.outputs.image_tag }}

            Triggered by: ${{ github.sha }}
            Workflow: ${{ github.workflow }}
            Actor: ${{ github.actor }}"

          git push

      # Optional: trigger Argo CD sync immediately rather than waiting for poll interval
      - name: Trigger Argo CD sync (optional — Argo CD polls automatically)
        env:
          ARGOCD_SERVER: argocd.ticketflow.io
          ARGOCD_AUTH_TOKEN: ${{ secrets.ARGOCD_AUTH_TOKEN }}
        run: |
          # Install argocd CLI
          curl -sSL -o argocd https://github.com/argoproj/argo-cd/releases/latest/download/argocd-linux-amd64
          chmod +x argocd

          # Trigger sync — Argo CD will detect the Helm values change and roll out
          ./argocd app sync ticket-service \
            --server "${ARGOCD_SERVER}" \
            --auth-token "${ARGOCD_AUTH_TOKEN}" \
            --insecure \
            --timeout 300

          # Wait for rollout to complete
          ./argocd app wait ticket-service \
            --server "${ARGOCD_SERVER}" \
            --auth-token "${ARGOCD_AUTH_TOKEN}" \
            --insecure \
            --health \
            --timeout 300

Dockerfile for ticket-service (Spring Boot)

The Dockerfile used in the pipeline above. Uses multi-stage build to keep the final image small and free of build tools.

# Stage 1: Build (uses Maven + JDK — NOT included in final image)
FROM eclipse-temurin:17-jdk-alpine AS builder
WORKDIR /build
COPY pom.xml .
COPY src ./src
# Download dependencies separately so they can be cached as a Docker layer
RUN mvn dependency:go-offline -B -q
RUN mvn package -DskipTests -B -q

# Stage 2: Runtime (minimal JRE — no Maven, no JDK, no source code)
FROM eclipse-temurin:17-jre-alpine AS runtime
WORKDIR /app

# Create non-root user for security
RUN addgroup -g 1001 appgroup && adduser -u 1001 -G appgroup -D appuser

# Copy only the built JAR from the builder stage
COPY --from=builder /build/target/ticket-service-*.jar app.jar

# Set ownership before switching user
RUN chown appuser:appgroup app.jar

USER appuser

# Health check — Kubernetes liveness/readiness probes use this
HEALTHCHECK --interval=30s --timeout=10s --start-period=60s --retries=3 \
  CMD wget -qO- http://localhost:8080/actuator/health || exit 1

EXPOSE 8080

ENTRYPOINT ["java", \
  "-XX:+UseContainerSupport", \
  "-XX:MaxRAMPercentage=75.0", \
  "-Djava.security.egd=file:/dev/./urandom", \
  "-jar", "app.jar"]

GitLab CI Equivalent

For teams using GitLab, this .gitlab-ci.yml implements the same five-stage pipeline with the same security and versioning behaviour.

# .gitlab-ci.yml
# Full CI/CD pipeline for ticket-service — GitLab CI equivalent of the GitHub Actions workflow

variables:
  REGISTRY: nexus.ticketflow.io:5000
  IMAGE_NAME: ticket-service
  MAVEN_OPTS: "-Dmaven.repo.local=$CI_PROJECT_DIR/.m2/repository"

# Cache Maven dependencies between pipeline runs
cache:
  key: "$CI_COMMIT_REF_SLUG-maven"
  paths:
    - .m2/repository/

stages:
  - build
  - docker
  - scan
  - push
  - deploy

# ─────────────────────────────────────────────────────────────────
# Stage 1: Build and Test
# ─────────────────────────────────────────────────────────────────
build-test:
  stage: build
  image: eclipse-temurin:17-jdk-alpine
  script:
    - mvn package --no-transfer-progress -B
  artifacts:
    paths:
      - target/ticket-service-*.jar
    expire_in: 1 hour
    reports:
      junit: target/surefire-reports/TEST-*.xml

# ─────────────────────────────────────────────────────────────────
# Stage 2: Build Docker Image (not pushed yet)
# ─────────────────────────────────────────────────────────────────
docker-build:
  stage: docker
  image: docker:24
  services:
    - docker:24-dind
  needs: [ build-test ]
  variables:
    DOCKER_TLS_CERTDIR: "/certs"
  script:
    # Compute image tag: semver for git tags, branch-sha for branches
    - |
      if [ -n "$CI_COMMIT_TAG" ]; then
        IMAGE_TAG="${CI_COMMIT_TAG#v}"
      else
        SHORT_SHA=$(echo "$CI_COMMIT_SHA" | cut -c1-7)
        IMAGE_TAG="${CI_COMMIT_REF_SLUG}-${SHORT_SHA}"
      fi
      echo "IMAGE_TAG=${IMAGE_TAG}" > image_tag.env
      echo "Building image: ${REGISTRY}/${IMAGE_NAME}:${IMAGE_TAG}"

    - source image_tag.env
    - docker build -t "${REGISTRY}/${IMAGE_NAME}:${IMAGE_TAG}" .
    - docker save "${REGISTRY}/${IMAGE_NAME}:${IMAGE_TAG}" | gzip > image.tar.gz
  artifacts:
    paths:
      - image.tar.gz
      - image_tag.env
    expire_in: 1 hour

# ─────────────────────────────────────────────────────────────────
# Stage 3: Security Scan with Trivy
# ─────────────────────────────────────────────────────────────────
trivy-scan:
  stage: scan
  image: aquasec/trivy:latest
  needs: [ docker-build ]
  script:
    - source image_tag.env
    - docker load < image.tar.gz
    - |
      trivy image \
        --exit-code 1 \
        --ignore-unfixed \
        --vuln-type os,library \
        --severity CRITICAL,HIGH \
        --format table \
        "${REGISTRY}/${IMAGE_NAME}:${IMAGE_TAG}"
  artifacts:
    paths:
      - image_tag.env
    expire_in: 1 hour

# ─────────────────────────────────────────────────────────────────
# Stage 4: Push to Nexus (only after successful scan)
# ─────────────────────────────────────────────────────────────────
push-to-nexus:
  stage: push
  image: docker:24
  services:
    - docker:24-dind
  needs: [ trivy-scan ]
  script:
    - source image_tag.env
    - docker load < image.tar.gz
    - echo "${NEXUS_PASSWORD}" | docker login "${REGISTRY}" -u "${NEXUS_USERNAME}" --password-stdin
    - docker push "${REGISTRY}/${IMAGE_NAME}:${IMAGE_TAG}"
    - docker tag "${REGISTRY}/${IMAGE_NAME}:${IMAGE_TAG}" "${REGISTRY}/${IMAGE_NAME}:latest"
    - docker push "${REGISTRY}/${IMAGE_NAME}:latest"
  rules:
    - if: $CI_COMMIT_BRANCH == "main"
    - if: $CI_COMMIT_TAG =~ /^v/

# ─────────────────────────────────────────────────────────────────
# Stage 5: Update Helm values and trigger ArgoCD
# ─────────────────────────────────────────────────────────────────
deploy:
  stage: deploy
  image: alpine/git:latest
  needs: [ push-to-nexus ]
  rules:
    - if: $CI_COMMIT_BRANCH == "main"
    - if: $CI_COMMIT_TAG =~ /^v/
  script:
    - source image_tag.env
    - apk add --no-cache yq curl

    # Clone GitOps repository
    - git clone "https://oauth2:${GITOPS_TOKEN}@gitlab.com/ticketflow/gitops.git" gitops
    - cd gitops

    # Determine environment
    - |
      if [ -n "$CI_COMMIT_TAG" ]; then
        VALUES_FILE="environments/production/ticket-service/values.yaml"
      else
        VALUES_FILE="environments/staging/ticket-service/values.yaml"
      fi

    # Update image tag
    - yq e ".image.tag = \"${IMAGE_TAG}\"" -i "${VALUES_FILE}"
    - yq e ".image.repository = \"${REGISTRY}/${IMAGE_NAME}\"" -i "${VALUES_FILE}"

    # Commit and push
    - git config user.email "gitlab-ci@ticketflow.io"
    - git config user.name "GitLab CI"
    - git add .
    - git diff --staged --quiet || git commit -m "chore(deploy): update ticket-service to ${IMAGE_TAG}"
    - git push

Semantic Versioning Helper Script

This reusable shell script encodes the TicketFlow versioning strategy. It can be called from any CI system.

#!/usr/bin/env bash
# scripts/compute-version.sh
# Outputs a deterministic image tag based on git context.
# Usage: source scripts/compute-version.sh
# Exports: IMAGE_TAG, SEMVER, IS_RELEASE

set -euo pipefail

SHORT_SHA=$(git rev-parse --short=7 HEAD)
BRANCH=$(git rev-parse --abbrev-ref HEAD | tr '/' '-')

# Check if we are on a semver tag (e.g. v1.4.2)
GIT_TAG=$(git tag --points-at HEAD | grep -E '^v[0-9]+\.[0-9]+\.[0-9]+$' | head -1 || true)

if [[ -n "${GIT_TAG}" ]]; then
  # Strip leading 'v': v1.4.2 → 1.4.2
  SEMVER="${GIT_TAG#v}"
  IMAGE_TAG="${SEMVER}"
  IS_RELEASE="true"
elif [[ "${BRANCH}" == "main" ]]; then
  # Main branch build: append SHA for traceability
  IMAGE_TAG="main-${SHORT_SHA}"
  SEMVER="${IMAGE_TAG}"
  IS_RELEASE="false"
else
  # Feature branch: sanitize name and append SHA
  SAFE_BRANCH=$(echo "${BRANCH}" | sed 's/[^a-zA-Z0-9-]/-/g' | tr '[:upper:]' '[:lower:]')
  IMAGE_TAG="${SAFE_BRANCH}-${SHORT_SHA}"
  SEMVER="${IMAGE_TAG}"
  IS_RELEASE="false"
fi

export IMAGE_TAG
export SEMVER
export IS_RELEASE

echo "IMAGE_TAG:  ${IMAGE_TAG}"
echo "SEMVER:     ${SEMVER}"
echo "IS_RELEASE: ${IS_RELEASE}"

Helm values.yaml (staging) after pipeline update

This is what the ticket-service Helm values file looks like after the deploy job updates it. Argo CD reads this file from the GitOps repository and applies it to the cluster.

Behind the Scenes

What happens inside a GitHub Actions runner when the workflow triggers?
  1. GitHub receives the push event via the git server. It evaluates all workflows in .github/workflows/ to find those matching the event and branch. If multiple workflows match, they run in parallel.
  2. GitHub allocates a fresh virtual machine from its hosted runner pool (a 2-core, 7 GB RAM Ubuntu VM for ubuntu-latest). This VM is provisioned from a pre-baked image so it starts in seconds. The VM has Docker, git, Node.js, Java, Python, and hundreds of other tools pre-installed.
  3. The runner agent (a small Go binary) starts on the VM and registers with GitHub's API. It receives the job's YAML instructions.
  4. For each step: if it is a uses: action, the runner downloads the action's code from its repository (or from the actions cache). If it is a run: step, it executes shell commands in a subprocess.
  5. Environment variables and secrets are injected into the process environment. Secrets are masked in log output — any log line containing a secret value is replaced with ***. This masking is best-effort and not a security boundary; it prevents accidental leakage in logs, not intentional exfiltration.
  6. Each step writes its output to a log stream that GitHub captures in real time (visible in the Actions UI). Steps can also write to $GITHUB_OUTPUT to pass values to later steps, or to $GITHUB_STEP_SUMMARY to add Markdown summaries visible in the workflow UI.
  7. When all jobs complete (or any required job fails), the runner sends a final status to GitHub, which updates the commit status check, notifies subscribers, and marks pull requests as passing or failing. The VM is wiped and returned to the pool.

The VM is ephemeral — nothing persists between workflow runs. This is why you use actions/cache to save Maven/npm dependencies, and actions/upload-artifact / actions/download-artifact to pass build outputs between jobs within the same workflow run.

How does Argo CD detect the Helm values change and deploy?

Argo CD operates on the GitOps principle: the desired state of the cluster lives in git. Argo CD continuously reconciles the cluster's actual state against the desired state in the repository.

  1. The CI pipeline's deploy job commits an updated values.yaml to the GitOps repository. The only change is a single line: image.tag: "main-abc1234" → image.tag: "main-def5678".
  2. Argo CD polls the GitOps repository every 3 minutes (default) using a shallow git fetch. It compares the repository's current HEAD SHA against the last-synced SHA stored in its own database.
  3. When it detects a new commit, Argo CD renders the Helm chart with the updated values, producing a set of Kubernetes manifests (Deployment, Service, HPA, etc.).
  4. It compares these rendered manifests against the live resources in the cluster using a three-way diff: desired (git), live (cluster), last-applied. Resources that differ are flagged as "OutOfSync".
  5. If auto-sync is enabled (configured in the Argo CD Application CRD), it immediately applies the diff — patching the Deployment with the new image tag. Kubernetes then initiates a rolling update: new pods are scheduled with the new image, old pods are terminated as new ones become ready.
  6. Argo CD monitors the rollout and reports back to its UI. If the new pods fail health checks within the timeout, Argo CD can automatically roll back by syncing to the previous git commit.
How does Trivy scan work and what does it find?

Trivy is an open-source vulnerability scanner by Aqua Security. It works by:

  1. Extracting layers — Trivy decompresses the Docker image and extracts each filesystem layer.
  2. Inventory collection — It reads OS package manifests (/var/lib/dpkg/status for Debian/Ubuntu, /lib/apk/db/installed for Alpine), language-specific lock files (pom.xml, package-lock.json, requirements.txt, go.sum), and binary metadata.
  3. CVE matching — It compares the collected package versions against its embedded vulnerability database (updated daily from NVD, GHSA, and OS-specific advisories like Alpine Linux secdb, Debian Security Tracker, Red Hat Security Data).
  4. Reporting — For each match, Trivy reports: CVE ID, severity (CRITICAL/HIGH/MEDIUM/LOW), affected package, installed version, fixed version, and a link to the advisory.

In the TicketFlow pipeline, --ignore-unfixed is set so Trivy only fails on CVEs that have a patch available. This prevents the pipeline from being blocked by vulnerabilities in packages where no upgrade path yet exists — which would make the gate impossible to pass. --exit-code 1 causes the pipeline to fail (non-zero exit) if any CRITICAL or HIGH CVE with a fix is found, blocking the image from being pushed to Nexus.

Common Mistakes

❌ Mistake 1: Pushing :latest Only

Teams configure their CI pipeline to build and push only ticket-service:latest. This tag is mutable — every new pipeline run overwrites it. After 10 deploys, you have no idea which version is actually running in production, you cannot roll back to a specific previous version, and your Kubernetes deployment history is meaningless (every revision says "latest"). Debugging a production incident becomes "which latest are we on?".

Fix: Always push a deterministic, immutable tag alongside :latest. In TicketFlow: nexus.ticketflow.io:5000/ticket-service:1.4.2 for releases and nexus.ticketflow.io:5000/ticket-service:main-abc1234 for branch builds. Kubernetes manifests and Helm values reference only the immutable tag. :latest is pushed as a convenience for humans to docker pull quickly, never for automated deployments.

❌ Mistake 2: Storing Registry Credentials in the Workflow File

Engineers hardcode NEXUS_PASSWORD: mysecretpassword directly in the .github/workflows/ci-cd.yml file or in the Dockerfile. The workflow file is committed to git — which means the password is in every developer's local clone, in every git log, and visible to anyone with repository read access. Rotating the password requires editing the YAML and committing again.

Fix: Use GitHub Secrets (${{ secrets.NEXUS_PASSWORD }}) for sensitive values. Secrets are encrypted at rest, never appear in logs, and can be rotated without touching the workflow file. For organizations, use environment-scoped secrets so staging and production use different credentials with different permissions.

❌ Mistake 3: Pushing Before the Security Scan

The pipeline builds the Docker image, immediately pushes it to Nexus, and then runs Trivy as a later step. If Trivy finds CRITICAL vulnerabilities and the pipeline fails, the image is already in the registry and could be pulled and deployed by someone who doesn't notice the failed pipeline. The security gate has no teeth.

Fix: Always scan before push. Build the image, load it into the local Docker daemon with push: false, load: true, run Trivy against the local image. Only if Trivy exits with code 0 (no blocking CVEs) do you run a second build-push with push: true. Layer caching means the second build is nearly instantaneous — it reuses all cached layers. This is exactly how the TicketFlow workflow above is structured.

❌ Mistake 4: Running kubectl apply Directly From CI

The pipeline has a deploy step that runs kubectl apply -f k8s/deployment.yaml with hardcoded cluster credentials stored as CI secrets. This works until you have multiple engineers running pipelines simultaneously (two concurrent deployments race against each other), need to audit what changed and when (git history only shows code changes, not deployment history), or need to roll back (you have to re-run a previous pipeline manually).

Fix: Use GitOps. The pipeline's only deployment action is to commit the new image tag to the GitOps repository. Argo CD handles the actual kubectl operations. This gives you: a full git audit trail of every deployment, automatic drift detection (if someone manually edits the cluster, Argo CD resets it), automatic rollback by reverting a git commit, and no cluster credentials in CI (Argo CD has read access to git, not the other way around).

❌ Mistake 5: No Parallelism — Sequential Jobs for Independent Tasks

Teams write a single job with 20 steps running entirely sequentially: build the backend, then build the frontend, then lint the backend, then lint the frontend, then run backend tests, then run frontend tests. Each step waits for the previous one. A pipeline that could complete in 4 minutes takes 15 because parallelisable work is serialised.

Fix: Split independent work into separate jobs and use needs: only to express true dependencies. In GitHub Actions, jobs without needs: run in parallel. In the TicketFlow pipeline: lint, unit tests, and dependency audit can run in parallel in the first stage. Docker build depends only on the build artifact, not on lint. Security scan depends only on the Docker build. This shapes the pipeline as a DAG, not a linear chain, and minimises end-to-end time.

Architect Perspective

🏛️ Architect Note

CI/CD is not a tooling decision — it is an organizational capability. The pipeline is the fastest feedback loop your engineering organization has. Every architectural decision you make either shortens or lengthens that loop.

  • The pipeline is the deployment contract. When the pipeline passes, the artifact is correct, tested, scanned, and ready to deploy. Every engineer trusts this contract implicitly. When engineers bypass the pipeline (manual deploys, hotfixes applied directly to the cluster), they break this contract and reintroduce the variability CI/CD was built to eliminate.
  • Pipeline speed is a product metric. A 40-minute pipeline discourages small commits and encourages large batches of changes — exactly the opposite of what you want. Invest engineering time in making the pipeline fast: layer caching, test parallelization, selective test runs on changed modules. Keep the critical path under 10 minutes.
  • Security gates in the pipeline are shift-left security. Running Trivy in CI means a vulnerability is caught before the image reaches any environment. Running it in production monitoring means you catch it after deployment — after it was exploitable. The earlier in the pipeline a defect is caught, the cheaper it is to fix. This applies to security vulnerabilities, performance regressions, and functional bugs equally.
  • The GitOps boundary matters architecturally. Separating the CI pipeline (produces artifact) from the CD process (deploys artifact) via a git repository creates an important decoupling. CI doesn't need cluster credentials. Deployments have a full audit trail in git. Rollback is a git revert. Multiple environments (staging, prod-eu, prod-us) can all track the same Helm chart but deploy different image tags, controlled entirely by their respective values files in git.

Trade-off to understand: Full GitOps with Argo CD introduces a propagation delay. A commit to the GitOps repo does not deploy instantly — Argo CD polls every 3 minutes by default. For incident response where seconds matter, you can trigger an immediate sync via the argocd app sync CLI command or the Argo CD API, but the usual flow involves up to a few minutes of latency. For organizations used to direct kubectl apply in CI, this feels slow — but the audit trail, drift detection, and rollback capability are worth it for production systems.

Production Readiness Checklist

Before considering your CI/CD pipeline production-ready, verify all of the following:

  • Every image is pushed with a deterministic, immutable tag (semver or branch-SHA) — never :latest only
  • Kubernetes manifests and Helm values files reference the immutable tag, never :latest
  • Registry credentials stored as CI secrets — never in workflow YAML, Dockerfiles, or code
  • Trivy (or equivalent) security scan runs before image push — pipeline fails on CRITICAL/HIGH fixable CVEs
  • Multi-stage Dockerfile in use — final image contains no build tools, test dependencies, or source code
  • Final image runs as a non-root user (USER instruction in Dockerfile)
  • GitOps pattern in use — deploys happen via git commit to values file, not direct kubectl from CI
  • Argo CD (or Flux) reconciles cluster state — drift is auto-detected and auto-corrected
  • Pipeline has explicit job dependencies (needs:) ensuring scan runs before push, push before deploy
  • Test results and scan reports uploaded as artifacts for post-run analysis
  • Branch protection rules require pipeline to pass before merge to main
  • Pipeline completes in under 10 minutes for the critical path (build → test → scan → push)
  • Rollback procedure documented and tested: revert values.yaml git commit → Argo CD auto-resyncs
  • Slack / PagerDuty notification on deploy failure configured in pipeline

Interview Questions

These are the most frequently asked CI/CD questions in software engineering and DevOps interviews. Click each question to reveal a complete answer.

Q1: What is the difference between Continuous Integration, Continuous Delivery, and Continuous Deployment?

Continuous Integration (CI) is the practice of automatically building and testing every commit to the shared repository. The goal is to detect integration failures early. "Integration" refers to merging multiple developers' work — CI does this continuously rather than in a big-bang merge at the end of a sprint. A CI pipeline typically runs: compile → unit tests → integration tests → code analysis.

Continuous Delivery (CD) extends CI by ensuring that every commit which passes CI is automatically packaged and deployed to a staging environment, making it deployable to production at any time. Deployment to production still requires a manual approval gate — but it is a single button click, not a multi-hour process. The key property: the software is always in a releasable state.

Continuous Deployment is a stricter form of CD where every commit that passes all automated checks is automatically deployed to production with zero human intervention. This requires extremely high confidence in test coverage, monitoring, and automated rollback capabilities. Companies like Netflix and Amazon deploy to production hundreds of times per day using this model.

Practical guidance: Most organizations start with CI, progress to Continuous Delivery, and only adopt full Continuous Deployment once they have strong monitoring and automated rollback in place. Continuous Deployment without good observability is dangerous — you need to know within seconds if a deployment has broken production.

Q2: How do you implement semantic versioning in a CI pipeline? What image tag strategy do you recommend?

Semantic versioning in CI means producing deterministic, meaningful image tags that encode the version, source, and traceability of the artifact.

The three-tag strategy used in TicketFlow:

  1. Release tag — triggered by a git tag like v1.4.2. Strip the v prefix: push nexus.ticketflow.io:5000/ticket-service:1.4.2. This is the production-ready, human-readable semver tag. Created by the release manager tagging a commit.
  2. Branch-SHA tag — triggered by every commit to main. Format: main-abc1234 where abc1234 is the 7-character git short SHA. Immutable and traceable — you can run git show abc1234 to see exactly what code is in the image. Used for staging deployments.
  3. :latest — pushed as a convenience alias alongside the deterministic tag. Never referenced in any Kubernetes manifest, Helm values file, or automated deployment.

In a GitHub Actions workflow:

if [[ "${GITHUB_REF}" == refs/tags/v* ]]; then
  IMAGE_TAG="${GITHUB_REF_NAME#v}"    # v1.4.2 → 1.4.2
else
  SHORT_SHA=$(git rev-parse --short=7 HEAD)
  IMAGE_TAG="${GITHUB_REF_NAME}-${SHORT_SHA}"  # main-abc1234
fi

The key property: every image tag must be both unique and traceable. Given a tag, you can answer: "exactly what code is in this image?" A tag like latest or build-42 cannot answer that question without external context.

Q3: Why should you never push a Docker image to the registry before running the security scan? How do you structure the pipeline to prevent this?

If the image is pushed to the registry before scanning, the security gate is effectively decorative. A developer or automated process could pull and deploy the image before the scan completes or after it fails. The image exists in the registry as a real, pullable artifact — regardless of subsequent scan results. This is especially risky in organisations with Argo CD auto-sync enabled: a new image tag in the registry could be deployed automatically within minutes.

The correct sequence:

  1. Build the image with push: false, load: true — this builds the image locally in the CI runner's Docker daemon without pushing it to any registry.
  2. Run Trivy against the local image with exit-code: 1 and severity: CRITICAL,HIGH. If Trivy finds blocking vulnerabilities, it exits with code 1, failing the CI step. The pipeline halts. The image has never touched the registry.
  3. Only if Trivy exits with code 0 (no blocking CVEs), run a second build-push step with push: true. Layer caching means this second build is nearly instantaneous — it does not re-download or re-compile anything.

In GitHub Actions with docker/build-push-action:

# Step 1: Build only (no push)
- uses: docker/build-push-action@v5
  with:
    push: false
    load: true
    tags: myimage:${{ env.TAG }}

# Step 2: Scan
- uses: aquasecurity/trivy-action@master
  with:
    image-ref: myimage:${{ env.TAG }}
    exit-code: '1'
    severity: CRITICAL,HIGH

# Step 3: Push (only reached if scan passed)
- uses: docker/build-push-action@v5
  with:
    push: true
    tags: myimage:${{ env.TAG }}
Q4: What is the GitOps pattern and why is it better than running kubectl apply from CI?

GitOps is a deployment pattern where the desired state of infrastructure and applications is stored entirely in git. A reconciliation agent (Argo CD or Flux) continuously compares the git state to the live cluster state and applies any differences. Deployments happen by committing to git — not by running imperative commands like kubectl apply.

Problems with kubectl apply from CI:

  • Credentials in CI: The CI system needs cluster credentials with kubectl apply permissions. If CI is compromised, the attacker has direct cluster write access.
  • No audit trail: Who deployed what and when is recorded in CI logs, not in a queryable git history. Correlating a production incident to a specific deployment requires searching CI logs.
  • No drift detection: If someone runs kubectl edit directly on the cluster, CI has no knowledge of the drift. The cluster's actual state diverges silently from what CI last applied.
  • Rollback is manual: To roll back, you must re-run a previous pipeline or manually construct a kubectl rollout undo command.

GitOps advantages:

  • CI only needs git write access — not cluster access. The security boundary is dramatically smaller.
  • Every deployment is a git commit with author, timestamp, diff, and commit message. Full audit trail.
  • Argo CD continuously detects and corrects drift — unauthorized manual changes are reverted automatically.
  • Rollback is a git revert of the values commit. Argo CD auto-syncs within minutes.
  • Multiple environments (staging, prod-eu, prod-ap) are managed by separate values files in the same git repository — no separate CI logic needed per environment.
Q5: How do you handle secrets in a CI/CD pipeline? What are the anti-patterns?

Secrets in CI/CD pipelines (registry passwords, cloud credentials, API tokens, database connection strings) must never appear in plain text anywhere in the repository, workflow files, or build logs.

Anti-patterns to avoid:

  • Hardcoding secrets in workflow YAML: password: mysecretpassword
  • Storing secrets in a .env file committed to git (even if the file is later added to .gitignore — git history still contains it)
  • Printing secrets in build logs: echo $DB_PASSWORD — even with GitHub's log masking, this is fragile
  • Baking secrets into Docker image layers: ENV DB_PASSWORD=secret in Dockerfile — anyone who can pull the image can run docker history and read it
  • Using the same credentials for CI and production with full permissions

Best practices:

  • GitHub Secrets — encrypted key-value pairs stored at repo or org level. Referenced as ${{ secrets.NEXUS_PASSWORD }}. Never appear in logs. Scoped to specific environments (staging vs production) for separation.
  • OIDC federation — instead of storing long-lived credentials, configure GitHub Actions to exchange a short-lived GitHub OIDC token for cloud credentials (AWS IAM, GCP Workload Identity). Zero static secrets for cloud access.
  • Least-privilege accounts — CI credentials should have only the permissions needed: push to one Nexus repository, commit to one git repository. Not admin access.
  • Secret rotation — rotate CI credentials regularly and on personnel changes. GitHub Secrets allow rotation without changing workflow files.
  • Secret scanning — enable GitHub's secret scanning to detect accidental secret commits before they land in the main branch.
Q6: How do you optimise CI pipeline speed? What are the key techniques for a build that's taking 30+ minutes?

A slow CI pipeline is a tax on every developer, every commit, every hour of the workday. Reducing a 30-minute pipeline to under 10 minutes is one of the highest-leverage engineering investments a team can make. Here are the techniques in rough order of impact:

  1. Parallelise independent jobs. Identify which jobs do not depend on each other and remove the artificial serialisation. Linting, type checking, unit tests, dependency audits, and Docker builds for independent services can often run simultaneously. In GitHub Actions, any job without needs: runs in parallel.
  2. Cache dependencies aggressively. Maven, npm, pip, and go all download packages on every fresh runner. Cache them: actions/setup-java with cache: maven, or explicit actions/cache keyed on the lock file hash. Restoring from cache takes seconds; downloading from the internet takes minutes.
  3. Use Docker layer caching in CI. Add cache-from: type=gha and cache-to: type=gha,mode=max to docker/build-push-action. This caches Docker layers in GitHub's cache. Unchanged base layers (JRE, OS packages) are restored from cache — only the application layer is rebuilt when code changes.
  4. Skip unnecessary work on unchanged code. In a monorepo, use path filters (on: push: paths: in GitHub Actions) so the ticket-service pipeline only triggers when ticket-service code changes. The notification-service pipeline doesn't need to run when you modify ticket-service.
  5. Use faster test strategies. Run unit tests in parallel (Surefire -T 1C for one thread per CPU core in Maven). Separate fast unit tests (run on every commit) from slow integration tests (run only on main branch merges or nightly).
  6. Profile before optimising. Identify which step is the bottleneck. Is it the test suite? The Docker build? The Trivy scan? Optimising the wrong stage wastes time. GitHub Actions provides step timing data in the workflow run UI.
Q7: Walk me through what happens when a developer pushes a hotfix tag v2.1.1 to production in the TicketFlow system.

This question tests understanding of the full CI/CD loop. Here is the complete sequence for TicketFlow:

  1. Developer action: Engineer applies the hotfix commit to the main branch, creates an annotated git tag v2.1.1 and pushes it: git tag -a v2.1.1 -m "fix: patch payment timeout race condition" && git push origin v2.1.1.
  2. Workflow triggers: GitHub receives the tag push event. The ci-cd.yml workflow matches the tags: - 'v*' trigger. A fresh Ubuntu runner VM is allocated.
  3. Job 1 — Build & Test: Maven compiles the Spring Boot app and runs all 247 unit tests plus integration tests. If any test fails, the pipeline halts — the hotfix is rejected and does not proceed. Assume all tests pass (30 seconds).
  4. Job 2 — Version: The version computation detects GITHUB_REF = refs/tags/v2.1.1. It strips v to produce IMAGE_TAG=2.1.1 and sets IS_RELEASE=true.
  5. Job 3 — Docker Build → Scan → Push: Docker builds the multi-stage image. Trivy scans it — finds 0 CRITICAL/HIGH fixable CVEs. The image is pushed as nexus.ticketflow.io:5000/ticket-service:2.1.1 (and :latest as alias).
  6. Job 4 — Deploy: Because IS_RELEASE=true, the job checks out the GitOps repository and modifies environments/production/ticket-service/values.yaml: image.tag: "2.1.1". It commits with message chore(deploy): update ticket-service to 2.1.1 and pushes.
  7. Argo CD detection: Within 3 minutes (or immediately if triggered via sync), Argo CD detects the new commit in the GitOps repo. It renders the updated Helm chart, sees the Deployment's image field has changed from 2.1.0 to 2.1.1, and marks the app as OutOfSync.
  8. Kubernetes rolling update: Argo CD applies the updated Deployment. Kubernetes starts new ticket-service pods with image 2.1.1 from Nexus. As each new pod passes readiness probes, an old pod is terminated. Zero downtime for end users.
  9. Confirmation: Argo CD transitions the app to Healthy & Synced. A Slack notification is sent: "ticket-service 2.1.1 deployed to production by [engineer] at 14:32 UTC". Total elapsed time from git push to production: approximately 8 minutes.

If something goes wrong: The engineer runs git revert on the values.yaml commit in the GitOps repo and pushes. Argo CD detects the revert within 3 minutes and rolls back to 2.1.0 — identical process, no special rollback commands needed.
