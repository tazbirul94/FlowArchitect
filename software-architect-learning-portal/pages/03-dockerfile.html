Dockerfile & Image Engineering

Master every Dockerfile instruction, understand layer caching, build multi-stage images, and engineer minimal, secure container images.

Story

Building a Docker image is like packing a go-bag — you take exactly what you need, nothing more.

  • Pack: app code, dependencies, runtime.
  • Leave behind: full OS, dev tools, unrelated packages.
  • The Dockerfile is your packing list — written once, reproducible forever.

What is a Dockerfile?

Beginner

A Dockerfile is a text file with step-by-step instructions telling Docker how to build your image. Each instruction creates a new layer on top of the previous one.

Docker reads the Dockerfile top to bottom; each instruction creates a cached layer. Layers stack together using OverlayFS — a Linux filesystem that overlays read-only image layers under a single writable container layer.

If an instruction's text and all inputs match a previous build, Docker reuses the cached layer. The moment anything changes, the cache is invalidated from that instruction forward — which is why instruction order matters enormously.

Every Dockerfile Instruction

FROM

FROM is always the first instruction; it sets the base layer everything builds upon.

# Syntax
FROM <image>[:<tag>] [AS <name>]

# Examples
FROM ubuntu:22.04
FROM node:18-alpine
FROM node:18-alpine AS builder   # for multi-stage builds
FROM scratch                     # empty base — for Go static binaries
  • Always pin to a specific version tag — never use :latest in production.
  • Choose the smallest base: a 7 MB Alpine image has far less attack surface than a 1.1 GB full Node image.
  • Use AS <name> to name stages so later stages can reference them with COPY --from=<name>.
  • FROM scratch is a zero-byte base — only viable for fully static binaries (Go, Rust with musl).

WORKDIR

WORKDIR sets the working directory for all subsequent RUN, COPY, CMD, and ENTRYPOINT instructions — creating the directory if it doesn't exist.

WORKDIR /app

# Creates directory if it doesn't exist
# All subsequent RUN, COPY, CMD, ENTRYPOINT use this as the working directory
# Prefer over: RUN mkdir /app && cd /app

You can call WORKDIR multiple times; relative paths build on the previous value.

COPY vs ADD

COPY and ADD both transfer files from the build context into the image, but the differences matter for security.

# COPY: simple, predictable, always preferred for file copying
COPY package.json package-lock.json ./
COPY src/ ./src/
COPY . .

# ADD: do NOT use for simple file copying
# ADD has special powers: auto-extracts tarballs, supports URLs
# These "powers" are surprising and can create security issues
ADD app.tar.gz /app/     # extracts automatically into /app/
ADD https://example.com/file.tar.gz /tmp/   # downloads from URL (avoid!)
Common Mistake
  • Use COPY for all file copying — always.
  • ADD's URL download skips checksum verification; use RUN wget with explicit hash verification instead.
  • ADD's auto-extract bypasses security scanning in your pipeline.

RUN

RUN executes a shell command during the build, creating a new image layer. Cleanup must happen in the same RUN instruction as installation — a cleanup in a later layer leaves the cache in the image.

# BAD: three separate layers — cache in layer 1 still lives in the image
RUN apt-get update
RUN apt-get install -y curl
RUN rm -rf /var/lib/apt/lists/*

# GOOD: one layer with cleanup — apt cache never exists in the final image
RUN apt-get update && \
    apt-get install -y --no-install-recommends curl wget && \
    rm -rf /var/lib/apt/lists/*

# Shell form vs exec form
RUN echo "Hello"           # shell form: /bin/sh -c "echo Hello"
RUN ["echo", "Hello"]      # exec form: no shell invoked, direct exec
Production Tip
  • Always clean package manager caches in the same RUN as the install.
  • A separate RUN apt-get clean does nothing — the cache already exists in the previous layer.
  • Use --no-install-recommends to avoid pulling in suggested packages.

CMD vs ENTRYPOINT

CMD and ENTRYPOINT both specify what runs when a container starts; they serve different purposes and are most powerful together.

# CMD: default command, easily overridden at runtime
CMD ["nginx", "-g", "daemon off;"]
CMD ["node", "server.js"]

# ENTRYPOINT: the fixed executable that always runs
ENTRYPOINT ["java", "-jar"]
CMD ["app.jar"]   # CMD becomes the default argument to ENTRYPOINT

# docker run myapp               → java -jar app.jar
# docker run myapp other.jar     → java -jar other.jar  (CMD overridden)
# docker run --entrypoint sh myapp → sh  (ENTRYPOINT overridden)

# ---------------------------------------------------------------
# Shell form vs exec form — CRITICAL difference for signal handling
# ---------------------------------------------------------------

CMD node server.js          # shell form: wraps in /bin/sh -c
                            # PID 1 = sh, Node gets PID 2
                            # SIGTERM goes to sh, NOT to Node
                            # Graceful shutdown BROKEN

CMD ["node", "server.js"]   # exec form: Node is PID 1 directly
                            # SIGTERM goes directly to Node
                            # Graceful shutdown WORKS
Common Mistake
  • Shell form makes /bin/sh PID 1 — SIGTERM never reaches your app.
  • Your app can't drain connections, flush buffers, or deregister from service discovery.
  • Always use exec form (JSON array) for CMD and ENTRYPOINT.

EXPOSE

EXPOSE is documentation — it records which port the app listens on but does not publish it.

EXPOSE 8080
EXPOSE 8080/tcp
EXPOSE 5353/udp

Port publishing happens at runtime via docker run -p 8080:8080, Compose ports:, or a Kubernetes Service.

ENV vs ARG

ARG and ENV both declare variables but have very different scopes.

# ARG: build-time variable only — NOT available in the running container
ARG BUILD_VERSION=unknown
ARG NODE_ENV=production

# ENV: available both at build time AND at runtime inside the container
ENV APP_PORT=8080
ENV LOG_LEVEL=info
ENV NODE_ENV=production

# Using ARG to set ENV (common pattern for embedding build metadata)
ARG APP_VERSION
ENV APP_VERSION=$APP_VERSION

# Build with:
docker build --build-arg APP_VERSION=1.2.0 .
Common Mistake
  • ARG values are visible in docker history; ENV values in docker inspect.
  • Never bake passwords, API keys, or secrets into either — anyone who pulls the image can read them.
  • Inject secrets at runtime via -e, Docker Swarm secrets, or Kubernetes Secrets.

USER

USER sets the user for all subsequent RUN, CMD, and ENTRYPOINT instructions, and determines the user the container process runs as at startup.

# Create a non-root user on Debian/Ubuntu
RUN groupadd -r appuser && useradd -r -g appuser appuser

# Create a non-root user on Alpine (uses addgroup/adduser)
RUN addgroup -S appuser && adduser -S appuser -G appuser

# Switch to non-root user
USER appuser

# All subsequent instructions and the container startup run as appuser
Production Tip
  • Running as root gives attackers root inside the container — a stepping stone to the host.
  • UID 1001 is a common convention for application users.
  • Set all file permissions and directory ownership before the USER instruction.

HEALTHCHECK

HEALTHCHECK tells Docker how to test whether the container is functioning. Orchestrators use health status to route traffic and restart broken containers.

# HTTP health check using curl
HEALTHCHECK --interval=30s --timeout=10s --start-period=5s --retries=3 \
  CMD curl -f http://localhost:8080/health || exit 1

# HTTP health check using wget (Alpine-friendly — no curl by default)
HEALTHCHECK --interval=30s --timeout=3s \
  CMD wget --no-verbose --tries=1 --spider http://localhost:8080/health || exit 1

# Disable health check (useful when the orchestrator handles health probes)
HEALTHCHECK NONE
  • --interval: how often to run the check (default 30s)
  • --timeout: how long to wait before marking it failed (default 30s)
  • --start-period: grace period after start — failures here don't count toward retries (default 0s)
  • --retries: consecutive failures before marking unhealthy (default 3)

VOLUME

VOLUME declares a mount point — primarily documentation telling operators which paths need external storage.

# Declare mount points
VOLUME ["/data", "/logs"]

# Creates an anonymous volume at build time if nothing is mounted
# Prefer named volumes via: docker run -v mydata:/data
# Or in Docker Compose: volumes: - mydata:/data

In practice, define named volumes in docker-compose.yml or Kubernetes PersistentVolumeClaims rather than relying on this instruction.

.dockerignore

Without a .dockerignore, docker build sends your entire project — node_modules, .git, test files, IDE config — to the Docker daemon as the build context, slowing every build.

The .dockerignore file excludes files from the build context using the same glob syntax as .gitignore.

# Version control
.git
.gitignore

# CI/CD
.github
.gitlab-ci.yml
Jenkinsfile

# Development files
.env

*.local

# Dependencies (let Docker install them fresh from package manifests)
node_modules/

# Build artifacts

*.war
*.class

# IDE files

# Documentation (don't bloat the image)

# Test files

# Logs

# Temp files

Common Mistake
  • Without .dockerignore, COPY . . can send 500 MB+ of node_modules to the daemon.
  • Local native binaries (compiled for macOS/Windows) end up in a Linux image — and break.
  • Your .dockerignore is as important as your .gitignore — place it alongside your Dockerfile.

Layer Caching Deep Dive

Cache optimization is the single most impactful build pipeline improvement: it can cut a 3-minute build to 12 seconds.

One rule: if a layer changes, every layer after it must also be rebuilt. The diagram below shows what happens when source code changes but package.json doesn't:

flowchart TD F["FROM node:18-alpine\n✅ Cache HIT (never changes)"] --> W W["WORKDIR /app\n✅ Cache HIT"] --> CP1 CP1["COPY package*.json ./\n✅ Cache HIT (package.json unchanged)"] --> RUN RUN["RUN npm install\n✅ Cache HIT (dependencies unchanged)"] --> CP2 CP2["COPY . .\n❌ Cache MISS (src code changed)"] --> BUILD BUILD["RUN npm run build\n❌ Must rebuild"] --> CMD CMD["CMD node server.js\n❌ Must rebuild"] style F fill:#22c55e,color:#fff style W fill:#22c55e,color:#fff style CP1 fill:#22c55e,color:#fff style RUN fill:#22c55e,color:#fff style CP2 fill:#ef4444,color:#fff style BUILD fill:#ef4444,color:#fff style CMD fill:#ef4444,color:#fff

Copy package.json before source code — npm install is then cached until dependencies actually change, not on every commit.

# BAD: cache-busting approach
FROM node:18-alpine
WORKDIR /app
COPY . .                  # copies everything including src files
RUN npm install           # runs on EVERY code change — wasteful!
CMD ["node", "server.js"]


# GOOD: cache-friendly approach
FROM node:18-alpine
WORKDIR /app
COPY package.json package-lock.json ./  # only package files first
RUN npm install                          # cached until package.json changes
COPY . .                                 # source code after dependencies
CMD ["node", "server.js"]

Same principle applies everywhere: pom.xml before Java source, go.mod/go.sum before Go source, requirements.txt before Python source.

Multi-Stage Builds

A multi-stage build uses multiple FROM instructions. Each starts a new stage; later stages copy artifacts from earlier ones via COPY --from=<stage-name>. Only the final stage becomes the image you push.

This solves a core tension: build tools (compilers, test runners, dev dependencies) must not ship in the production image. Multi-stage lets you use whatever you need at build time while the runtime image contains only what's needed to execute.

Example 1: Node.js React App

# Stage 1: Build — full Node with dev dependencies
FROM node:18-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production=false     # install all deps including devDeps
COPY . .
RUN npm run build                      # produces /app/dist

# Stage 2: Serve — minimal nginx with only compiled static files
FROM nginx:1.25-alpine AS runtime
COPY --from=builder /app/dist /usr/share/nginx/html
COPY nginx.conf /etc/nginx/conf.d/default.conf
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]

Result: builder ~400 MB; final runtime image ~25 MB — nginx plus compiled static files only.

Example 2: Java Spring Boot App

# Stage 1: Build — Maven + full JDK
FROM maven:3.9-eclipse-temurin-17 AS builder
WORKDIR /build
COPY pom.xml .
RUN mvn dependency:go-offline -B          # cache Maven deps as separate layer
COPY src ./src
RUN mvn package -DskipTests -B

# Stage 2: Runtime — only JRE, no JDK, no Maven
FROM eclipse-temurin:17-jre-alpine AS runtime
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
WORKDIR /app
COPY --from=builder /build/target/*.jar app.jar
USER appuser
EXPOSE 8080
HEALTHCHECK --interval=30s --timeout=10s --start-period=20s --retries=3 \
  CMD wget --no-verbose --tries=1 --spider http://localhost:8080/actuator/health || exit 1
CMD ["java", "-jar", "app.jar"]

Result: builder ~800 MB; final runtime image ~180 MB — JRE plus compiled JAR only.

Example 3: Go Binary (Smallest Possible)

# Stage 1: Build — full Go toolchain
FROM golang:1.21-alpine AS builder
WORKDIR /build
COPY go.mod go.sum ./
RUN go mod download                                       # cache module downloads
COPY . .
RUN CGO_ENABLED=0 GOOS=linux \
    go build -a -installsuffix cgo -o app .              # fully static binary

# Stage 2: Minimal runtime — completely empty base image
FROM scratch AS runtime
COPY --from=builder /etc/ssl/certs/ca-certificates.crt /etc/ssl/certs/
COPY --from=builder /build/app /app
EXPOSE 8080
CMD ["/app"]

Result: ~6 MB final image. No OS, no shell, no utilities — minimum achievable attack surface.

Base Image Comparison

Choosing the right base image is the biggest lever for reducing image size and attack surface.

Base Image Size Shell Package Manager Use Case
ubuntu:22.04 77 MB bash apt General purpose, debugging-friendly
debian:bookworm-slim 74 MB bash apt Java/Python apps needing glibc
alpine:3.19 7 MB sh apk Go, static binaries, minimal footprint
node:18 1.1 GB bash apt Node apps — use node:18-alpine instead
node:18-alpine 175 MB sh apk Node apps — much smaller baseline
gcr.io/distroless/nodejs18 112 MB none none Production Node — no shell, minimal attack surface
gcr.io/distroless/java17 220 MB none none Production Java — no shell, glibc-compatible
scratch 0 MB none none Go/Rust fully static binaries only
Architect Note
  • Alpine uses musl libc — some native extensions (npm packages, Python C extensions) fail silently; test thoroughly.
  • Distroless is Debian-based (glibc-compatible) but stripped of shell, package manager, and debug tools — harder to pivot from if compromised.
  • Distroless :debug variants include busybox for incident response — never run them in production.

Complete Production Dockerfile Example

A production-grade Dockerfile for TicketFlow's ticket-service — multi-stage, layered JAR, non-root user, OCI labels, health check, proper signal handling.

# ==========================================
# ticket-service Dockerfile
# Spring Boot microservice
# ==========================================

# Stage 1: Dependencies — cached separately so code changes don't
#           re-download Maven dependencies
FROM maven:3.9-eclipse-temurin-17 AS deps
WORKDIR /build
COPY pom.xml .
RUN mvn dependency:go-offline -B --no-transfer-progress

# Stage 2: Build — builds on the deps stage (inherits its layer cache)
FROM deps AS builder
COPY src ./src
ARG APP_VERSION=unknown
ARG GIT_COMMIT=unknown
RUN mvn package -DskipTests -B --no-transfer-progress \
    -Dapp.version=${APP_VERSION} \
    -Dgit.commit=${GIT_COMMIT}

# Stage 3: Extract layers — Spring Boot layered jars split the JAR into
#           four layers ordered from least to most frequently changed,
#           so only application code changes invalidate the innermost layer
FROM eclipse-temurin:17-jre AS extractor
WORKDIR /extract
COPY --from=builder /build/target/*.jar app.jar
RUN java -Djarmode=layertools -jar app.jar extract

# Stage 4: Runtime — minimal JRE only, no JDK or Maven
FROM eclipse-temurin:17-jre-alpine AS runtime

# OCI image labels for registries, SBOM tools, and audit trails
LABEL org.opencontainers.image.title="ticket-service"
LABEL org.opencontainers.image.description="TicketFlow ticket management service"
LABEL org.opencontainers.image.vendor="TicketFlow Engineering"

# Security: create and switch to non-root user
RUN addgroup -S ticketflow && adduser -S ticketuser -G ticketflow

WORKDIR /app

# Copy Spring Boot layers in optimal order (least → most frequently changed)
# dependencies: frameworks, libraries
# spring-boot-loader: Spring's fat-jar launcher
# snapshot-dependencies: in-development library snapshots (if any)
# application: your compiled classes and resources (changes every commit)
COPY --from=extractor /extract/dependencies/ ./
COPY --from=extractor /extract/spring-boot-loader/ ./
COPY --from=extractor /extract/snapshot-dependencies/ ./
COPY --from=extractor /extract/application/ ./

# Set ownership before USER instruction
RUN chown -R ticketuser:ticketflow /app

# Drop privileges
USER ticketuser

# Runtime configuration via environment variables (overridden in K8s via ConfigMap)
ENV SPRING_PROFILES_ACTIVE=production
ENV SERVER_PORT=8080
ENV JAVA_OPTS="-Xms256m -Xmx512m -XX:+UseContainerSupport"

EXPOSE 8080

# Health check using Spring Boot Actuator — start-period gives JVM time to warm up
HEALTHCHECK --interval=30s --timeout=10s --start-period=60s --retries=3 \
  CMD wget --no-verbose --tries=1 --spider \
      http://localhost:8080/actuator/health || exit 1

# Use exec form so Java is PID 1 and receives SIGTERM for graceful shutdown.
# The 'exec' shell builtin replaces sh with java (avoids double-process problem
# when JAVA_OPTS needs shell variable expansion).
ENTRYPOINT ["sh", "-c", "exec java $JAVA_OPTS org.springframework.boot.loader.JarLauncher"]

Image Scanning

Scan images for known CVEs in OS packages and app dependencies. Trivy by Aqua Security is the most widely adopted open-source scanner — fast, accurate, and integrates with every major CI platform.

# Install Trivy
brew install aquasecurity/trivy/trivy     # macOS
apt-get install -y trivy                  # Ubuntu/Debian

# Scan a local image — shows all vulnerabilities by severity
trivy image myapp:v1.2.0

# Filter to HIGH and CRITICAL only (useful for actionable CI reports)
trivy image --severity HIGH,CRITICAL myapp:v1.2.0

# Output as JSON for downstream tooling or dashboards
trivy image --format json --output trivy-results.json myapp:v1.2.0

# Fail CI pipeline if any CRITICAL vulnerability is found
trivy image --exit-code 1 --severity CRITICAL myapp:v1.2.0

# Scan the Dockerfile itself for configuration mistakes
trivy config ./Dockerfile

# GitHub Actions integration
# - name: Scan image with Trivy
#   uses: aquasecurity/trivy-action@master
#   with:
#     image-ref: myapp:${{ github.sha }}
#     exit-code: '1'
#     severity: 'CRITICAL,HIGH'
Architect Note — SBOM
  • An SBOM (Software Bill of Materials) inventories every OS package and transitive dependency in your image.
  • Generate with: trivy sbom myapp:v1.2.0 --format cyclonedx
  • Required by US Executive Order 14028 for software sold to federal agencies — costs nothing extra if Trivy already runs in CI.

Image Size Optimization

A naive Node.js Dockerfile produces ~1.2 GB. An optimized one produces ~85 MB — 93% smaller.

# BEFORE: naive Dockerfile — 1.2 GB
FROM node:18                  # full Debian + Node
WORKDIR /app
COPY . .                      # copies node_modules if not dockerignored
RUN npm install               # installs dev AND prod dependencies
CMD ["node", "server.js"]


# AFTER: optimized — 85 MB (93% smaller)
FROM node:18-alpine AS deps   # Alpine base: 7 MB vs 350 MB
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production  # production dependencies only, no dev deps

FROM node:18-alpine AS runtime
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY src/ ./src/
COPY package.json .
USER node                     # non-root: the 'node' user is built into node images
CMD ["node", "src/server.js"]

Checklist for every image you build:

  • Use Alpine or distroless base image
  • Multi-stage build — exclude compilers, build tools, dev dependencies
  • npm ci --only=production or equivalent (no dev dependencies in runtime)
  • Combine related RUN commands to reduce layer count
  • Remove package manager caches in the same RUN layer as installation
  • .dockerignore excludes node_modules, .git, .env, and test files
  • Copy only what's needed — avoid COPY . . without a solid .dockerignore
  • Measure image size after every change: docker images myapp

Behind the Scenes

How does layer caching work internally?
  1. Each layer is identified by a content hash of: (a) the instruction text + any referenced file checksums, and (b) the previous layer's hash.
  2. This chain means reordering instructions busts the cache even if individual layer content is unchanged.
  3. For RUN instructions that fetch remote packages, Docker caches on instruction text only — it can't detect upstream version changes.
  4. Force a full rebuild with docker build --no-cache, or selectively invalidate with --build-arg CACHE_BUST=$(date +%s).
What is OverlayFS and how does Docker use it?
  1. Images are stacks of read-only layers; running a container adds one writable layer on top.
  2. OverlayFS presents a unified filesystem view across all layers without copying data.
  3. Writing to a file in a read-only layer triggers copy-on-write: the file is copied to the writable layer first, then modified.
  4. Containers that modify many large files (e.g., compiling inside a container) incur high I/O from copy-on-write.
  5. When the container is removed, the writable layer is discarded; read-only image layers persist.
What is BuildKit and how does it improve builds?

BuildKit is the modern Docker build engine, default since Docker 23. Key improvements:

  • Parallel stage execution: independent stages build simultaneously.
  • Better cache semantics: content-addressed cache plus --mount=type=cache to share package manager caches (~/.m2, ~/.npm) across builds without baking them into the image.
  • Build secrets: RUN --mount=type=secret,id=npmrc,target=/root/.npmrc npm install mounts a secret for one instruction only — never in any layer, never in docker history.
  • SSH forwarding: --mount=type=ssh forwards the host SSH agent for private git dependencies without embedding keys.

Common Mistakes

Mistake 1 — Package manager cache left in the image

Cleanup in a separate RUN layer does nothing — the cache already exists in the previous layer. Always chain: RUN apt-get update && apt-get install -y --no-install-recommends nginx && apt-get clean && rm -rf /var/lib/apt/lists/*.

Mistake 2 — Copying node_modules into the image
  • Local node_modules may contain native binaries compiled for macOS/Windows — they break on Linux.
  • They may include dev packages, local patches, or symlinks that don't survive the copy.
  • Always install fresh from the lockfile with npm ci.
Mistake 3 — Using ENTRYPOINT without CMD together carelessly

Without CMD, docker run myimage sh passes sh as an argument to your entrypoint instead of opening a shell. Use ENTRYPOINT for the fixed executable and CMD for default arguments.

Mistake 4 — Not setting a non-root USER
  • Containers default to root (UID 0) — unnecessary and dangerous.
  • Set directory permissions before the USER instruction.
  • Many Kubernetes clusters enforce runAsNonRoot: true — root images will fail to start.
Mistake 5 — Large build context from missing .dockerignore

Without .dockerignore, a Node project can send gigabytes to the daemon before a single instruction runs. Verify with docker build --no-cache . and watch the "Sending build context" line.

Mistake 6 — Pinning only to major version
  • FROM node:18 resolves to a different image every patch release — builds are not reproducible.
  • Use fully pinned tags: FROM node:18.19.1-alpine3.19.
  • Update deliberately after reviewing changelogs, not silently on every build.

Production Checklist

Before pushing to a registry or deploying, verify each item:

  • Base image uses a fully pinned version tag (not :latest)
  • Multi-stage build separates build tools from the runtime image
  • Non-root USER instruction set before CMD / ENTRYPOINT
  • HEALTHCHECK instruction defined with appropriate --start-period
  • .dockerignore present and excludes node_modules, .git, .env
  • Package manager cache cleaned in the same RUN layer as installation
  • Secrets not stored in ENV or ARG — injected at runtime
  • COPY used instead of ADD for all file copying
  • CMD and ENTRYPOINT use exec form (JSON array), not shell form
  • Dependency manifests copied before source code (cache optimization)
  • Image size measured and tracked: docker images myapp
  • Image scanned with Trivy before push to registry
  • OCI labels added: title, description, vendor, version

Hands-On Lab

Lab — Build a Progressively Optimized Node.js Image

Start with a naive Dockerfile, measure image size, then apply each optimization and watch the size shrink. All commands run in a single terminal session.

# Step 0: Create the test application
mkdir dockerfile-lab && cd dockerfile-lab

cat > server.js << 'EOF'
const http = require('http');
const server = http.createServer((req, res) => {
  res.writeHead(200);
  res.end('Hello from TicketFlow!\n');
});
server.listen(3000, () => console.log('Listening on port 3000'));
EOF

cat > package.json << 'EOF'
{
  "name": "ticket-service",
  "version": "1.0.0",
  "main": "server.js",
  "dependencies": {
    "express": "^4.18.2"
  },
  "devDependencies": {
    "jest": "^29.0.0",
    "eslint": "^8.0.0"
  }
}
EOF

# Step 1: Naive Dockerfile
cat > Dockerfile.naive << 'EOF'
FROM node:18
WORKDIR /app
COPY . .
RUN npm install
CMD ["node", "server.js"]
EOF

docker build -f Dockerfile.naive -t ticket-service:naive .
docker images ticket-service:naive    # note the size — likely ~1.1 GB+


# Step 2: Alpine base + cache-friendly layer ordering
cat > Dockerfile.alpine << 'EOF'
FROM node:18-alpine
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
CMD ["node", "server.js"]
EOF

docker build -f Dockerfile.alpine -t ticket-service:alpine .
docker images ticket-service:alpine   # compare — should be ~250 MB


# Step 3: Production-optimized (multi-stage + prod deps only + non-root)
cat > Dockerfile.prod << 'EOF'
FROM node:18-alpine AS deps
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production

FROM node:18-alpine AS runtime
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY server.js .
USER node
EXPOSE 3000
HEALTHCHECK --interval=30s CMD wget -q --spider http://localhost:3000/ || exit 1
CMD ["node", "server.js"]
EOF

docker build -f Dockerfile.prod -t ticket-service:prod .
docker images ticket-service:prod     # should be ~85 MB — 93% smaller than naive!


# Compare all three side by side
docker images ticket-service


# Run and test the production-optimized image
docker run -d -p 3000:3000 --name ticket-test ticket-service:prod
curl http://localhost:3000
docker logs ticket-test

# Inspect the running user — verify it's not root
docker exec ticket-test whoami        # should print: node


# Clean up
docker stop ticket-test && docker rm ticket-test

Interview Questions

Q1: What is the difference between CMD and ENTRYPOINT?

ENTRYPOINT is the fixed executable — only overridable via --entrypoint. CMD provides default arguments that any docker run myimage arg replaces.

  • Together: ENTRYPOINT is the program, CMD is its defaults.
  • docker run myapp override replaces CMD but keeps ENTRYPOINT.
  • Use ENTRYPOINT when the container is an executable (takes subcommands); use only CMD for flexible services.
  • Always use exec form (JSON array) for both.
Q2: Why does instruction order in a Dockerfile matter?

When a layer changes, all subsequent layers are invalidated and rebuilt — even if their content is unchanged.

  • Put rarely-changing instructions first, frequently-changing instructions last.
  • Classic Node pattern: copy package.json → npm install (cached) → copy source → build.
  • Without this, every code commit rebuilds the dependency layer — minutes of wasted CI time per commit.
Q3: What is a multi-stage build and when would you use it?

Multiple FROM instructions in one Dockerfile; later stages copy artifacts from earlier ones via COPY --from=<stage>. Only the last stage becomes the final image.

  • Use it when build tools (compilers, Maven, Go toolchain, dev deps) must not ship to production.
  • A Java service: builder ~800 MB, final runtime image ~180 MB — Maven and JDK never reach the registry.
  • Also prevents build-time secrets or intermediate artifacts from leaking into the final image.
Q4: How do you pass secrets to a Docker build without them ending up in the image?

Use BuildKit's --secret flag with --mount=type=secret — the secret is a temporary file for that single RUN only, never written to any layer.

# Build command
docker build --secret id=npmrc,src=$HOME/.npmrc .
# In Dockerfile
RUN --mount=type=secret,id=npmrc,target=/root/.npmrc npm install
  • ENV — visible in docker inspect. Not safe.
  • ARG — visible in docker history. Not safe.
  • Copy-then-delete — original file still exists in the earlier layer. Not safe.
Q5: What is distroless and why would you use it?

Distroless images (gcr.io/distroless/) contain only the language runtime — no shell, no package manager, no system tools. Debian-based, so glibc-compatible (unlike Alpine).

  • No bash, curl, or apt for attackers to pivot with post-exploitation.
  • Trade-off: you can't exec a shell into the container for debugging.
  • Use :debug variants (includes busybox) only for incident response — never in production.
  • Use kubectl debug with a sidecar image instead of modifying the production image.
Q6: How does the Docker layer cache work and how can it be busted?

Each layer's cache key = hash of (instruction + file checksums) + previous layer's key. Change any layer → all subsequent layers invalidated.

  • docker build --no-cache . — disables cache entirely.
  • --build-arg CACHE_BUST=$(date +%s) — invalidates from that instruction forward.
  • Touching any file referenced by COPY busts the cache from that point.
  • Limitation: Docker can't detect upstream package changes — apt-get install nginx may return a cached (stale) layer even after a new nginx version ships.
Q7: What is the security risk of running containers as root?

Root in a container means any code execution vulnerability gives an attacker root-level access inside that container.

  • Mounted Docker socket (/var/run/docker.sock) grants full host control from root.
  • Privileged containers bypass namespace isolation entirely.
  • Kernel exploits only reachable from root become accessible.
  • Non-root reduces blast radius — even with a container escape, attacker gets an unprivileged host user.
