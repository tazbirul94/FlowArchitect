Dockerfile & Image Engineering

Master every Dockerfile instruction, understand layer caching, build multi-stage images, and engineer minimal, secure container images.

Story

Building a Docker image is like packing a perfect go-bag for travel. You don't throw your entire house in a suitcase. You pick exactly what you need: your clothes (app code), your toiletries (dependencies), your charger (runtime). You don't bring your couch (full OS), your refrigerator (dev tools), or your neighbor's belongings (unrelated packages). The Dockerfile is your packing list. Written once, reproducible forever.

What is a Dockerfile?

Beginner

A Dockerfile is a text file with step-by-step instructions telling Docker how to build your image. Like a recipe. Each instruction creates a new layer on top of the previous one.

Docker reads the Dockerfile top to bottom. Each instruction creates a layer. Layers are cached. If a layer hasn't changed since the last build, Docker reuses the cached version — making subsequent builds much faster. The final image is all layers stacked together using OverlayFS: a Linux filesystem that overlays read-only image layers under a single writable container layer.

Every time you run docker build, Docker compares each instruction against its cache. If the instruction text and all inputs (file contents, parent layer hash) are identical to a previous build, it skips recomputing that layer and pulls from cache. The moment anything changes, the cache is invalidated from that instruction forward — which is why instruction order matters enormously.

Every Dockerfile Instruction

FROM

FROM is always the first instruction in a Dockerfile (only an ARG may precede it). It sets the base layer that every subsequent instruction builds upon. Think of it as choosing what kind of soil you're planting in — everything else grows on top of this foundation.

# Syntax
FROM <image>[:<tag>] [AS <name>]

# Examples
FROM ubuntu:22.04
FROM node:18-alpine
FROM node:18-alpine AS builder   # for multi-stage builds
FROM scratch                     # empty base — for Go static binaries
  • Always pin to a specific version tag — never use :latest in production. :latest is a moving target that breaks reproducibility.
  • Choose the smallest base that meets your needs. A 7 MB Alpine image has less attack surface than a 1.1 GB full Node image.
  • Use the AS <name> syntax to name stages in multi-stage builds so later stages can reference them with COPY --from=<name>.
  • FROM scratch is a special empty base with zero OS packages — only viable for fully static binaries (Go, Rust with musl).

WORKDIR

WORKDIR sets the working directory for all subsequent RUN, COPY, CMD, and ENTRYPOINT instructions. If the directory doesn't exist, Docker creates it. Using WORKDIR is cleaner and more explicit than chaining RUN mkdir && cd.

WORKDIR /app

# Creates directory if it doesn't exist
# All subsequent RUN, COPY, CMD, ENTRYPOINT use this as the working directory
# Prefer over: RUN mkdir /app && cd /app

You can use WORKDIR multiple times in a single Dockerfile. Each call changes the current directory for subsequent instructions. Relative paths build on the previous WORKDIR.

COPY vs ADD

COPY and ADD both transfer files from the build context (your local machine) into the image. They look similar, but the differences matter for security and predictability.

# COPY: simple, predictable, always preferred for file copying
COPY package.json package-lock.json ./
COPY src/ ./src/
COPY . .

# ADD: do NOT use for simple file copying
# ADD has special powers: auto-extracts tarballs, supports URLs
# These "powers" are surprising and can create security issues
ADD app.tar.gz /app/     # extracts automatically into /app/
ADD https://example.com/file.tar.gz /tmp/   # downloads from URL (avoid!)
Common Mistake

Using ADD instead of COPY is a common mistake from old tutorials. COPY is always preferred for simple file copying. ADD's URL download feature doesn't verify checksums — use RUN wget or RUN curl with explicit hash verification instead. ADD's auto-extract skips any security scanning you might have in your pipeline. Use COPY.

RUN

RUN executes a shell command during the build, creating a new image layer with the result. Every RUN instruction creates one layer. Fewer layers means a smaller image — but the real constraint is that cleanup must happen in the same RUN instruction as the installation.

# BAD: three separate layers — cache in layer 1 still lives in the image
RUN apt-get update
RUN apt-get install -y curl
RUN rm -rf /var/lib/apt/lists/*

# GOOD: one layer with cleanup — apt cache never exists in the final image
RUN apt-get update && \
    apt-get install -y --no-install-recommends curl wget && \
    rm -rf /var/lib/apt/lists/*

# Shell form vs exec form
RUN echo "Hello"           # shell form: /bin/sh -c "echo Hello"
RUN ["echo", "Hello"]      # exec form: no shell invoked, direct exec
Production Tip

Always clean up package manager caches in the same RUN instruction as the installation. If you clean in a separate RUN layer, the cache still exists in the previous layer — it's part of the image, just not accessible from the filesystem. Running apt-get clean in a separate RUN after apt-get install does nothing to reduce image size.

CMD vs ENTRYPOINT

CMD and ENTRYPOINT both specify what runs when a container starts. They serve different purposes and are most powerful when used together.

# CMD: default command, easily overridden at runtime
CMD ["nginx", "-g", "daemon off;"]
CMD ["node", "server.js"]

# ENTRYPOINT: the fixed executable that always runs
ENTRYPOINT ["java", "-jar"]
CMD ["app.jar"]   # CMD becomes the default argument to ENTRYPOINT

# docker run myapp               → java -jar app.jar
# docker run myapp other.jar     → java -jar other.jar  (CMD overridden)
# docker run --entrypoint sh myapp → sh  (ENTRYPOINT overridden)

# ---------------------------------------------------------------
# Shell form vs exec form — CRITICAL difference for signal handling
# ---------------------------------------------------------------

CMD node server.js          # shell form: wraps in /bin/sh -c
                            # PID 1 = sh, Node gets PID 2
                            # SIGTERM goes to sh, NOT to Node
                            # Graceful shutdown BROKEN

CMD ["node", "server.js"]   # exec form: Node is PID 1 directly
                            # SIGTERM goes directly to Node
                            # Graceful shutdown WORKS
Common Mistake

Using shell form for CMD or ENTRYPOINT breaks graceful shutdown. When Kubernetes sends SIGTERM to stop a pod, it goes to PID 1. If PID 1 is /bin/sh (shell form), it often doesn't forward the signal to your application. Your app never gets a chance to drain connections, flush buffers, or deregister from service discovery. Always use exec form — the JSON array syntax — for CMD and ENTRYPOINT.

EXPOSE

EXPOSE is documentation. It tells users and tooling what port your application listens on inside the container. It does not actually publish the port or make it accessible from outside the container.

EXPOSE 8080
EXPOSE 8080/tcp
EXPOSE 5353/udp

Port publishing happens at runtime: docker run -p 8080:8080 maps the host port to the container port. In Docker Compose, ports: does the mapping. In Kubernetes, the Service resource handles it. EXPOSE just records the intent — but it's still worth including for documentation and for tools like docker-compose that can use it with automatic port assignment.

ENV vs ARG

ARG and ENV both declare variables, but they have very different scopes and use cases.

# ARG: build-time variable only — NOT available in the running container
ARG BUILD_VERSION=unknown
ARG NODE_ENV=production

# ENV: available both at build time AND at runtime inside the container
ENV APP_PORT=8080
ENV LOG_LEVEL=info
ENV NODE_ENV=production

# Using ARG to set ENV (common pattern for embedding build metadata)
ARG APP_VERSION
ENV APP_VERSION=$APP_VERSION

# Build with:
docker build --build-arg APP_VERSION=1.2.0 .
Common Mistake

ARG values are visible in docker history output. ENV values are visible in docker inspect. Neither should contain passwords, API keys, or secrets. Pass secrets at runtime via the -e flag, Docker Swarm secrets, or Kubernetes Secrets — never bake them into the image. Anyone who pulls your image can retrieve baked-in secrets.

USER

USER sets the user (and optionally the group) that all subsequent RUN, CMD, and ENTRYPOINT instructions run as. It also determines what user the container process runs as at startup.

# Create a non-root user on Debian/Ubuntu
RUN groupadd -r appuser && useradd -r -g appuser appuser

# Create a non-root user on Alpine (uses addgroup/adduser)
RUN addgroup -S appuser && adduser -S appuser -G appuser

# Switch to non-root user
USER appuser

# All subsequent instructions and the container startup run as appuser
Production Tip

Running as root in a container means that if your app has a code execution vulnerability, an attacker gets root inside the container — and potentially a path to the host via mounted sockets or kernel exploits. Always use USER. UID 1001 is a common convention for application users. Make sure all file permissions and directory ownership are set before the USER instruction, since after switching you lose root privileges.

HEALTHCHECK

HEALTHCHECK tells Docker how to test whether the container is functioning correctly. Docker periodically runs the check and updates the container's health status to healthy, unhealthy, or starting. Orchestrators like Kubernetes and Docker Swarm use health status to route traffic and restart broken containers.

# HTTP health check using curl
HEALTHCHECK --interval=30s --timeout=10s --start-period=5s --retries=3 \
  CMD curl -f http://localhost:8080/health || exit 1

# HTTP health check using wget (Alpine-friendly — no curl by default)
HEALTHCHECK --interval=30s --timeout=3s \
  CMD wget --no-verbose --tries=1 --spider http://localhost:8080/health || exit 1

# Disable health check (useful when the orchestrator handles health probes)
HEALTHCHECK NONE
  • --interval: how often to run the check (default 30s)
  • --timeout: how long to wait for a response before considering it failed (default 30s)
  • --start-period: grace period after container starts — failures during this window don't count toward the retry limit (default 0s)
  • --retries: how many consecutive failures before the container is marked unhealthy (default 3)

VOLUME

VOLUME declares a mount point. At runtime, Docker creates an anonymous volume at the declared path if no volume is mounted there. It's primarily documentation — telling users which directories hold persistent or shared data.

# Declare mount points
VOLUME ["/data", "/logs"]

# Creates an anonymous volume at build time if nothing is mounted
# Prefer named volumes via: docker run -v mydata:/data
# Or in Docker Compose: volumes: - mydata:/data

In practice, rely on named volumes defined in docker-compose.yml or Kubernetes PersistentVolumeClaims rather than the VOLUME instruction. The instruction's main value is as documentation for operators to know which paths need external storage.

.dockerignore

When you run docker build, Docker sends everything in your build directory to the Docker daemon as a "build context." Without a .dockerignore file, this includes your entire project — node_modules, .git history, test files, IDE configuration, everything. The build context travels over a network socket even on localhost, and large contexts slow every build.

The .dockerignore file tells Docker which files and directories to exclude from the build context. It uses the same glob syntax as .gitignore.

# Version control
.git
.gitignore

# CI/CD
.github
.gitlab-ci.yml
Jenkinsfile

# Development files
.env
.env.*
*.local

# Dependencies (let Docker install them fresh from package manifests)
node_modules/
vendor/
.venv/
__pycache__/
*.pyc

# Build artifacts
dist/
build/
target/
*.jar
*.war
*.class

# IDE files
.vscode/
.idea/
*.iml
.DS_Store
Thumbs.db

# Documentation (don't bloat the image)
docs/
README.md
*.md

# Test files
tests/
test/
spec/
coverage/
*.test.js
*.spec.ts

# Logs
*.log
logs/

# Temp files
tmp/
temp/
*.tmp
*.swp
Common Mistake

Without .dockerignore, a COPY . . instruction sends node_modules (potentially 500 MB or more) to the Docker daemon and then copies your local — possibly incompatible — native binaries into the image. Builds become dramatically slower and images become huge. Your .dockerignore is as important as your .gitignore. Place it at the same level as your Dockerfile.

Layer Caching Deep Dive

Understanding Docker's layer cache is the single most impactful optimization you can make to your build pipeline. When Docker builds an image, it checks each layer against the cache. A cache hit skips that layer entirely — including all the compute, network, and I/O involved. For a Java or Node project, this can mean the difference between a 3-minute build and a 12-second build.

Cache invalidation follows one simple rule: if a layer changes, every layer after it must also be rebuilt. This is why instruction order matters so much. The diagram below shows what happens when source code changes but package.json doesn't:

flowchart TD F["FROM node:18-alpine\n✅ Cache HIT (never changes)"] --> W W["WORKDIR /app\n✅ Cache HIT"] --> CP1 CP1["COPY package*.json ./\n✅ Cache HIT (package.json unchanged)"] --> RUN RUN["RUN npm install\n✅ Cache HIT (dependencies unchanged)"] --> CP2 CP2["COPY . .\n❌ Cache MISS (src code changed)"] --> BUILD BUILD["RUN npm run build\n❌ Must rebuild"] --> CMD CMD["CMD node server.js\n❌ Must rebuild"] style F fill:#22c55e,color:#fff style W fill:#22c55e,color:#fff style CP1 fill:#22c55e,color:#fff style RUN fill:#22c55e,color:#fff style CP2 fill:#ef4444,color:#fff style BUILD fill:#ef4444,color:#fff style CMD fill:#ef4444,color:#fff

The key insight: copy package.json before copying source code. The npm install layer is cached as long as package.json doesn't change. Source code changes — which happen on every commit — don't invalidate the dependency installation layer. This alone can save minutes per build in a CI pipeline.

# BAD: cache-busting approach
FROM node:18-alpine
WORKDIR /app
COPY . .                  # copies everything including src files
RUN npm install           # runs on EVERY code change — wasteful!
CMD ["node", "server.js"]


# GOOD: cache-friendly approach
FROM node:18-alpine
WORKDIR /app
COPY package.json package-lock.json ./  # only package files first
RUN npm install                          # cached until package.json changes
COPY . .                                 # source code after dependencies
CMD ["node", "server.js"]

The same principle applies to other ecosystems: copy pom.xml before source in Maven builds, copy go.mod and go.sum before Go source, copy requirements.txt before Python source. Copy the dependency manifest first, install dependencies, then copy the actual application code.

Multi-Stage Builds

A multi-stage build uses multiple FROM instructions in a single Dockerfile. Each FROM starts a new stage. A later stage can copy files from an earlier stage using COPY --from=<stage-name>. Only the final stage becomes the image you push and run.

This solves a fundamental tension: building software requires compilers, build tools, test runners, and development libraries that have no business being in a production container. A Maven installation weighs ~300 MB. A Node.js installation with all dev dependencies can weigh over 500 MB. Multi-stage builds let you use whatever you need at build time while producing a runtime image that contains only what's needed to execute.

Example 1: Node.js React App

# Stage 1: Build — full Node with dev dependencies
FROM node:18-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production=false     # install all deps including devDeps
COPY . .
RUN npm run build                      # produces /app/dist

# Stage 2: Serve — minimal nginx with only compiled static files
FROM nginx:1.25-alpine AS runtime
COPY --from=builder /app/dist /usr/share/nginx/html
COPY nginx.conf /etc/nginx/conf.d/default.conf
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]

Result: the builder stage is ~400 MB (Node + dev dependencies + source). The final runtime image is ~25 MB — nginx plus compiled static HTML, CSS, and JavaScript. No Node runtime, no npm, no TypeScript compiler, no source maps in production.

Example 2: Java Spring Boot App

# Stage 1: Build — Maven + full JDK
FROM maven:3.9-eclipse-temurin-17 AS builder
WORKDIR /build
COPY pom.xml .
RUN mvn dependency:go-offline -B          # cache Maven deps as separate layer
COPY src ./src
RUN mvn package -DskipTests -B

# Stage 2: Runtime — only JRE, no JDK, no Maven
FROM eclipse-temurin:17-jre-alpine AS runtime
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
WORKDIR /app
COPY --from=builder /build/target/*.jar app.jar
USER appuser
EXPOSE 8080
HEALTHCHECK --interval=30s --timeout=10s --start-period=20s --retries=3 \
  CMD wget --no-verbose --tries=1 --spider http://localhost:8080/actuator/health || exit 1
CMD ["java", "-jar", "app.jar"]

Result: the builder stage is ~800 MB (JDK + Maven + source). The final runtime image is ~180 MB — just the JRE and the compiled JAR. No compiler, no build system, no source code.

Example 3: Go Binary (Smallest Possible)

# Stage 1: Build — full Go toolchain
FROM golang:1.21-alpine AS builder
WORKDIR /build
COPY go.mod go.sum ./
RUN go mod download                                       # cache module downloads
COPY . .
RUN CGO_ENABLED=0 GOOS=linux \
    go build -a -installsuffix cgo -o app .              # fully static binary

# Stage 2: Minimal runtime — completely empty base image
FROM scratch AS runtime
COPY --from=builder /etc/ssl/certs/ca-certificates.crt /etc/ssl/certs/
COPY --from=builder /build/app /app
EXPOSE 8080
CMD ["/app"]

Result: ~6 MB final image. A completely static Go binary in an entirely empty container. No OS, no shell, no utilities. The scratch base is zero bytes — the only contents are what you explicitly copy in. This is the minimum achievable attack surface.

Base Image Comparison

Choosing the right base image is the single biggest lever for reducing image size and attack surface. Here's a comparison of common base images:

Base Image Size Shell Package Manager Use Case
ubuntu:22.04 77 MB bash apt General purpose, debugging-friendly
debian:bookworm-slim 74 MB bash apt Java/Python apps needing glibc
alpine:3.19 7 MB sh apk Go, static binaries, minimal footprint
node:18 1.1 GB bash apt Node apps — use node:18-alpine instead
node:18-alpine 175 MB sh apk Node apps — much smaller baseline
gcr.io/distroless/nodejs18 112 MB none none Production Node — no shell, minimal attack surface
gcr.io/distroless/java17 220 MB none none Production Java — no shell, glibc-compatible
scratch 0 MB none none Go/Rust fully static binaries only
Architect Note

Alpine uses musl libc instead of glibc. Most apps work fine, but some native extensions — certain npm packages, Python C extensions, Ruby gems with native code — fail silently or crash with Alpine. Test thoroughly before adopting Alpine for apps with native dependencies.

Distroless is often the better production choice: it's based on Debian, so glibc-compatible, but stripped of shell, package manager, and debugging tools. If an attacker gets code execution in your distroless container, there's no bash, no curl, no apt to pivot with. Keep a debug-tagged variant (:debug) of distroless images — these include busybox for incident response — but never run debug images in production.

Complete Production Dockerfile Example

The following is a production-grade Dockerfile for the TicketFlow ticketing platform's ticket-service. It applies every technique covered in this page: multi-stage builds, Spring Boot layered jars for optimal caching, non-root user, OCI labels, health check, and proper signal handling.

# ==========================================
# ticket-service Dockerfile
# Spring Boot microservice
# ==========================================

# Stage 1: Dependencies — cached separately so code changes don't
#           re-download Maven dependencies
FROM maven:3.9-eclipse-temurin-17 AS deps
WORKDIR /build
COPY pom.xml .
RUN mvn dependency:go-offline -B --no-transfer-progress

# Stage 2: Build — builds on the deps stage (inherits its layer cache)
FROM deps AS builder
COPY src ./src
ARG APP_VERSION=unknown
ARG GIT_COMMIT=unknown
RUN mvn package -DskipTests -B --no-transfer-progress \
    -Dapp.version=${APP_VERSION} \
    -Dgit.commit=${GIT_COMMIT}

# Stage 3: Extract layers — Spring Boot layered jars split the JAR into
#           four layers ordered from least to most frequently changed,
#           so only application code changes invalidate the innermost layer
FROM eclipse-temurin:17-jre AS extractor
WORKDIR /extract
COPY --from=builder /build/target/*.jar app.jar
RUN java -Djarmode=layertools -jar app.jar extract

# Stage 4: Runtime — minimal JRE only, no JDK or Maven
FROM eclipse-temurin:17-jre-alpine AS runtime

# OCI image labels for registries, SBOM tools, and audit trails
LABEL org.opencontainers.image.title="ticket-service"
LABEL org.opencontainers.image.description="TicketFlow ticket management service"
LABEL org.opencontainers.image.vendor="TicketFlow Engineering"

# Security: create and switch to non-root user
RUN addgroup -S ticketflow && adduser -S ticketuser -G ticketflow

WORKDIR /app

# Copy Spring Boot layers in optimal order (least → most frequently changed)
# dependencies: frameworks, libraries
# spring-boot-loader: Spring's fat-jar launcher
# snapshot-dependencies: in-development library snapshots (if any)
# application: your compiled classes and resources (changes every commit)
COPY --from=extractor /extract/dependencies/ ./
COPY --from=extractor /extract/spring-boot-loader/ ./
COPY --from=extractor /extract/snapshot-dependencies/ ./
COPY --from=extractor /extract/application/ ./

# Set ownership before USER instruction
RUN chown -R ticketuser:ticketflow /app

# Drop privileges
USER ticketuser

# Runtime configuration via environment variables (overridden in K8s via ConfigMap)
ENV SPRING_PROFILES_ACTIVE=production
ENV SERVER_PORT=8080
ENV JAVA_OPTS="-Xms256m -Xmx512m -XX:+UseContainerSupport"

EXPOSE 8080

# Health check using Spring Boot Actuator — start-period gives JVM time to warm up
HEALTHCHECK --interval=30s --timeout=10s --start-period=60s --retries=3 \
  CMD wget --no-verbose --tries=1 --spider \
      http://localhost:8080/actuator/health || exit 1

# Use exec form so Java is PID 1 and receives SIGTERM for graceful shutdown.
# The 'exec' shell builtin replaces sh with java (avoids double-process problem
# when JAVA_OPTS needs shell variable expansion).
ENTRYPOINT ["sh", "-c", "exec java $JAVA_OPTS org.springframework.boot.loader.JarLauncher"]

Image Scanning

Building a small, well-structured image is only half the security story. You also need to scan your images for known vulnerabilities (CVEs) in OS packages and application dependencies. Trivy by Aqua Security is the most widely adopted open-source scanner — it's fast, accurate, and integrates with every major CI platform.

# Install Trivy
brew install aquasecurity/trivy/trivy     # macOS
apt-get install -y trivy                  # Ubuntu/Debian

# Scan a local image — shows all vulnerabilities by severity
trivy image myapp:v1.2.0

# Filter to HIGH and CRITICAL only (useful for actionable CI reports)
trivy image --severity HIGH,CRITICAL myapp:v1.2.0

# Output as JSON for downstream tooling or dashboards
trivy image --format json --output trivy-results.json myapp:v1.2.0

# Fail CI pipeline if any CRITICAL vulnerability is found
trivy image --exit-code 1 --severity CRITICAL myapp:v1.2.0

# Scan the Dockerfile itself for configuration mistakes
trivy config ./Dockerfile

# GitHub Actions integration
# - name: Scan image with Trivy
#   uses: aquasecurity/trivy-action@master
#   with:
#     image-ref: myapp:${{ github.sha }}
#     exit-code: '1'
#     severity: 'CRITICAL,HIGH'
Architect Note — SBOM

An SBOM (Software Bill of Materials) is a complete inventory of every component in your image — OS packages, language runtime libraries, transitive application dependencies. Generate one with trivy sbom myapp:v1.2.0 --format cyclonedx. SBOMs are required by some enterprise security policies and government regulations, including US Executive Order 14028 (May 2021), which mandates SBOMs for software sold to federal agencies. Generating them costs nothing extra if you already run Trivy in CI.

Image Size Optimization

A naive Dockerfile for a Node.js service can produce a 1.2 GB image. An optimized Dockerfile for the same application produces an 85 MB image — 93% smaller. Here's the before and after:

# BEFORE: naive Dockerfile — 1.2 GB
FROM node:18                  # full Debian + Node
WORKDIR /app
COPY . .                      # copies node_modules if not dockerignored
RUN npm install               # installs dev AND prod dependencies
CMD ["node", "server.js"]


# AFTER: optimized — 85 MB (93% smaller)
FROM node:18-alpine AS deps   # Alpine base: 7 MB vs 350 MB
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production  # production dependencies only, no dev deps

FROM node:18-alpine AS runtime
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY src/ ./src/
COPY package.json .
USER node                     # non-root: the 'node' user is built into node images
CMD ["node", "src/server.js"]

Checklist for every image you build:

  • Use Alpine or distroless base image
  • Multi-stage build — exclude compilers, build tools, dev dependencies
  • npm ci --only=production or equivalent (no dev dependencies in runtime)
  • Combine related RUN commands to reduce layer count
  • Remove package manager caches in the same RUN layer as installation
  • .dockerignore excludes node_modules, .git, .env, and test files
  • Copy only what's needed — avoid COPY . . without a solid .dockerignore
  • Measure image size after every change: docker images myapp

Behind the Scenes

How does layer caching work internally?

Each image layer is identified by a content hash computed from two inputs: the hash of the instruction itself (including any files referenced by COPY or ADD, checksummed by Docker) plus the hash of the previous layer. This creates a chain: change any layer, and all layers after it have a different hash and must be rebuilt.

This is why instruction order matters structurally — it's not just a convention. Layer A's cache key incorporates layer A-1's key. If you reorder instructions, you change the dependency chain and the cache no longer applies even if the layer content hasn't changed.

For RUN instructions that fetch external resources (like apt-get install), Docker only caches based on the instruction text — it doesn't know whether the remote package repository has new versions. Use docker build --no-cache to force a full rebuild, or use --build-arg CACHE_BUST=$(date +%s) to selectively invalidate from a specific point.

What is OverlayFS and how does Docker use it?

Docker images are stacks of read-only layers. At the bottom is the base image layer (e.g., Alpine's minimal filesystem). Each Dockerfile instruction adds another read-only layer on top. When you run a container, Docker adds a single writable layer on top of all the read-only layers.

OverlayFS is the Linux kernel filesystem driver that makes this work efficiently. It presents a unified view of multiple directories (the layers) without copying any data. Files are accessed from whichever layer they exist in, starting from the top (most recent).

When a container writes to a file that lives in a read-only layer, OverlayFS performs a copy-on-write: the file is copied to the writable layer first, then modified there. The original in the read-only layer is untouched. This is why containers that modify many large files (like compiling source code) have high I/O overhead — every touched file triggers a copy. It's also why writing large files in a container doesn't reduce image size; the write goes to the writable layer while the original file still occupies space in the image.

When the container is removed, the writable layer is discarded. Everything in the read-only image layers persists for the next container that starts from the same image.

What is BuildKit and how does it improve builds?

BuildKit is the modern Docker build engine, enabled by default since Docker 23 (and available since Docker 18.09 via DOCKER_BUILDKIT=1). It replaces the legacy builder with a fundamentally different architecture.

Key improvements over the legacy builder:

  • Parallel stage execution: BuildKit analyzes the dependency graph between stages. Stages that don't depend on each other are built in parallel. A multi-stage build with independent builder and tester stages runs both at the same time.
  • Better cache semantics: BuildKit uses a content-addressed cache that persists across builds more reliably than the legacy cache. It also supports --mount=type=cache for sharing package manager caches (like ~/.m2, ~/.npm) between build runs without baking them into the image.
  • Build secrets: RUN --mount=type=secret,id=npmrc,target=/root/.npmrc npm install mounts a secret file for a single RUN instruction only. The secret never exists in any image layer and doesn't appear in docker history.
  • SSH forwarding: --mount=type=ssh forwards the host's SSH agent into the build, enabling npm install of private git dependencies without embedding SSH keys.

To explicitly enable BuildKit: set the environment variable DOCKER_BUILDKIT=1, or use docker buildx build which always uses BuildKit.

Common Mistakes

Mistake 1 — Package manager cache left in the image

RUN apt-get install -y nginx leaves the entire apt package cache inside the layer. This can add tens of megabytes to every image silently. Always complete the cleanup in the same instruction: RUN apt-get update && apt-get install -y --no-install-recommends nginx && apt-get clean && rm -rf /var/lib/apt/lists/*. The --no-install-recommends flag also prevents apt from pulling in suggested packages you don't need.

Mistake 2 — Copying node_modules into the image

If node_modules is not in your .dockerignore and you COPY . ., you're copying your local development modules. These may contain native binaries compiled for your host OS (macOS or Windows) that won't run on the Linux container. They may also contain dev-only packages, local patches, or symlinks that don't survive the copy. Always let Docker install dependencies fresh from your lockfile with npm ci.

Mistake 3 — Using ENTRYPOINT without CMD together carelessly

If you use ENTRYPOINT in exec form without CMD, you can't easily debug a running container by doing docker run myimage sh. That command passes sh as an argument to your entrypoint instead of running a shell. Use ENTRYPOINT for the fixed executable and CMD for default arguments. Override CMD for alternate arguments; override ENTRYPOINT only when you need a completely different executable (e.g., for debug containers).

Mistake 4 — Not setting a non-root USER

By default, containers run as root (UID 0). This is unnecessary and dangerous. Always add a USER instruction. If your app needs to write files, create directories and set permissions before switching users. Many Kubernetes clusters run with the runAsNonRoot: true security context enforced via admission controllers — images that require root will fail to start.

Mistake 5 — Large build context from missing .dockerignore

Without a .dockerignore, running docker build . in a Node project sends hundreds of megabytes to the Docker daemon before a single instruction executes. Even on localhost, this traverses a Unix socket. In a CI environment it can mean uploading gigabytes. Place your .dockerignore at the repository root and verify it with docker build --no-cache . while watching the "Sending build context" output line shrink.

Mistake 6 — Pinning only to major version

FROM node:18 resolves to a different image every time Node 18.x.y is patched. Your build is not reproducible. A patch release can change behavior, break native addon compilation, or introduce new CVEs. Use fully pinned tags: FROM node:18.19.1-alpine3.19. You can update deliberately when you review changelogs, rather than receiving silent updates on every build.

Production Checklist

Before pushing an image to a registry or deploying to production, verify each of the following:

  • Base image uses a fully pinned version tag (not :latest)
  • Multi-stage build separates build tools from the runtime image
  • Non-root USER instruction set before CMD / ENTRYPOINT
  • HEALTHCHECK instruction defined with appropriate --start-period
  • .dockerignore present and excludes node_modules, .git, .env
  • Package manager cache cleaned in the same RUN layer as installation
  • Secrets not stored in ENV or ARG — injected at runtime
  • COPY used instead of ADD for all file copying
  • CMD and ENTRYPOINT use exec form (JSON array), not shell form
  • Dependency manifests copied before source code (cache optimization)
  • Image size measured and tracked: docker images myapp
  • Image scanned with Trivy before push to registry
  • OCI labels added: title, description, vendor, version

Hands-On Lab

Lab — Build a Progressively Optimized Node.js Image

Start with a naive Dockerfile, measure image size, then apply each optimization and watch the size shrink. All commands run in a single terminal session.

# Step 0: Create the test application
mkdir dockerfile-lab && cd dockerfile-lab

cat > server.js << 'EOF'
const http = require('http');
const server = http.createServer((req, res) => {
  res.writeHead(200);
  res.end('Hello from TicketFlow!\n');
});
server.listen(3000, () => console.log('Listening on port 3000'));
EOF

cat > package.json << 'EOF'
{
  "name": "ticket-service",
  "version": "1.0.0",
  "main": "server.js",
  "dependencies": {
    "express": "^4.18.2"
  },
  "devDependencies": {
    "jest": "^29.0.0",
    "eslint": "^8.0.0"
  }
}
EOF

# Step 1: Naive Dockerfile
cat > Dockerfile.naive << 'EOF'
FROM node:18
WORKDIR /app
COPY . .
RUN npm install
CMD ["node", "server.js"]
EOF

docker build -f Dockerfile.naive -t ticket-service:naive .
docker images ticket-service:naive    # note the size — likely ~1.1 GB+


# Step 2: Alpine base + cache-friendly layer ordering
cat > Dockerfile.alpine << 'EOF'
FROM node:18-alpine
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
CMD ["node", "server.js"]
EOF

docker build -f Dockerfile.alpine -t ticket-service:alpine .
docker images ticket-service:alpine   # compare — should be ~250 MB


# Step 3: Production-optimized (multi-stage + prod deps only + non-root)
cat > Dockerfile.prod << 'EOF'
FROM node:18-alpine AS deps
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production

FROM node:18-alpine AS runtime
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY server.js .
USER node
EXPOSE 3000
HEALTHCHECK --interval=30s CMD wget -q --spider http://localhost:3000/ || exit 1
CMD ["node", "server.js"]
EOF

docker build -f Dockerfile.prod -t ticket-service:prod .
docker images ticket-service:prod     # should be ~85 MB — 93% smaller than naive!


# Compare all three side by side
docker images ticket-service


# Run and test the production-optimized image
docker run -d -p 3000:3000 --name ticket-test ticket-service:prod
curl http://localhost:3000
docker logs ticket-test

# Inspect the running user — verify it's not root
docker exec ticket-test whoami        # should print: node


# Clean up
docker stop ticket-test && docker rm ticket-test

Interview Questions

Q1: What is the difference between CMD and ENTRYPOINT?

ENTRYPOINT defines the fixed executable that always runs when the container starts. It cannot be overridden by passing a command to docker run — only the --entrypoint flag overrides it. CMD provides default arguments that can be easily replaced by anything you pass to docker run myimage arg1 arg2.

When used together: ENTRYPOINT is the program, CMD is its default arguments. docker run myapp runs ENTRYPOINT CMD. docker run myapp override runs ENTRYPOINT override — the CMD is replaced but the entrypoint is not.

Rule of thumb: use ENTRYPOINT when your container is an executable (e.g., a compiled binary that takes subcommands). Use CMD when it's a service with configurable startup. For most microservices, using only CMD in exec form is sufficient and provides the most flexibility.

Q2: Why does instruction order in a Dockerfile matter?

Docker caches each layer. When a layer changes, all subsequent layers are invalidated and rebuilt from scratch — even if their own content is unchanged.

The practical implication: put rarely-changing instructions first and frequently-changing instructions last. The classic pattern for a Node.js service is: copy package.json first → run npm install (cached until package.json changes) → copy source code → run build. This means npm install only re-executes when you actually change dependencies, not on every code edit.

Without this discipline, every single code change rebuilds the dependency layer — which can mean minutes of wasted CI time per commit across a team.

Q3: What is a multi-stage build and when would you use it?

A multi-stage build uses multiple FROM instructions in a single Dockerfile. Each FROM starts a new build stage. Later stages can selectively copy artifacts from earlier stages using COPY --from=<stage-name>. Only the last stage becomes the final image.

Use it whenever your build process requires tools that shouldn't be in the production image: compilers, build systems (Maven, Gradle, Go toolchain), test runners, linters, or development dependencies. A Java service built with Maven produces an 800 MB builder image, but the final image containing only the JRE and compiled JAR is 180 MB. The Maven installation, JDK, and source code never appear in the registry or in production.

Beyond size, multi-stage builds reduce attack surface by excluding tools that could be exploited, and they prevent build-time secrets or intermediate artifacts from leaking into the final image.

Q4: How do you pass secrets to a Docker build without them ending up in the image?

With BuildKit (default in Docker 23+), use the --secret flag at build time and the --mount=type=secret syntax in the Dockerfile. The secret is mounted as a temporary file for that single RUN instruction only — it is never written to any image layer and never appears in docker history.

# Build command
docker build --secret id=npmrc,src=$HOME/.npmrc .
# In Dockerfile
RUN --mount=type=secret,id=npmrc,target=/root/.npmrc npm install

Alternatives that do NOT work safely: using ENV (visible in docker inspect), using ARG (visible in docker history), or copying a secrets file and then deleting it in a later layer (the file still exists in the earlier layer). For production registries, inject secrets at runtime via Kubernetes Secrets, AWS Secrets Manager, or HashiCorp Vault — never bake them into images.

Q5: What is distroless and why would you use it?

Distroless images (published by Google at gcr.io/distroless/) contain only the application runtime and its language-level dependencies — no shell, no package manager, no coreutils, no system tools. They're based on Debian so they use glibc (unlike Alpine), but they strip everything non-essential.

The security benefit is significant: if an attacker achieves code execution inside your container, there's no bash to drop into, no curl to download further payloads, no apt to install tools. The attack surface for post-exploitation is dramatically reduced.

The trade-off is debuggability. You can't exec a shell into a distroless container. For incident response, use the :debug tag variants which include a busybox shell — but never deploy debug images to production. For routine debugging during development, keep a sidecar debug image available and use kubectl debug to attach it to a running pod without modifying the production image.

Q6: How does the Docker layer cache work and how can it be busted?

Each layer's cache key is computed from two inputs: the content of the instruction itself (including checksums of any files referenced by COPY or ADD) plus the cache key of the preceding layer. This creates a chain — if any layer in the chain changes, all subsequent layers are invalidated.

Common ways to bust the cache deliberately:

  • docker build --no-cache . — disables cache entirely for this build
  • Adding a build argument that changes: ARG CACHE_BUST=$(date +%s) then passing --build-arg CACHE_BUST=$(date +%s) at build time invalidates from that instruction forward
  • Modifying any referenced file — Docker checksums files for COPY instructions, so touching a file busts the cache from that COPY forward

Important limitation: Docker cannot detect if an external resource has changed. RUN apt-get install nginx might install a newer nginx version today vs. six months ago, but Docker sees the same instruction text and returns the cached layer. This is intentional for reproducibility, but means you must occasionally force a rebuild to pick up security patches.

Q7: What is the security risk of running containers as root?

Running as root (UID 0) inside a container means that if your application has a vulnerability — remote code execution, path traversal, server-side template injection, deserialization — an attacker with container access has root-level privileges inside that container.

While Linux namespaces and cgroups provide isolation, root in a container is not the same as being sandboxed. Several escape vectors exist: mounted Docker socket (/var/run/docker.sock) grants full host control; privileged containers bypass namespace isolation entirely; kernel vulnerabilities exploitable only from root are more accessible; misconfigured volume mounts can expose host filesystems with root write access.

Running as a non-root user significantly reduces blast radius. Even with a container escape, the attacker gets an unprivileged user on the host rather than root. Additionally, many Kubernetes security policies — enforced via OPA Gatekeeper, Kyverno, or Pod Security Standards — reject pods that run as root. Using USER in your Dockerfile keeps your images compatible with security-hardened cluster configurations and communicates intent to platform teams.
