Kubernetes Ingress & API Gateway

One load balancer for all your services — path routing, TLS termination, rate limiting, and the difference between Ingress and an API Gateway.

The Story Behind Ingress

🏨 The Hotel Concierge

Imagine a large hotel with hundreds of rooms — each room offering a different service: a spa, a restaurant, a gym, a business center, a cinema. Without a concierge, every guest would need a separate entrance door and a separate phone number for each service. "Dial 555-0101 for spa, 555-0102 for restaurant, 555-0103 for gym..." The lobby would need a hundred separate reception desks. The hotel's street frontage would be a wall of doors.

Instead, every hotel has a single main entrance. You walk in and speak to the concierge: "I'd like the restaurant." They direct you. "I need the gym." They point the way. One front door, one phone number, one concierge — and they route you to the right service based on what you ask for.

Kubernetes Ingress is the concierge. Your cluster has dozens of services — user-service, ticket-service, payment-service, notification-service — each running on private ClusterIP addresses that the outside world cannot reach. Without Ingress, you'd need a separate cloud Load Balancer (at $20–$80/month each) per service. With Ingress, one load balancer handles all external traffic and routes it to the right service based on the URL path or hostname.

What Is Kubernetes Ingress?

Beginner Analogy

Think of your Kubernetes cluster as a shopping mall. Each shop (service) is inside the mall — they don't have their own street entrance. The mall has one main entrance, and a directory board inside tells you: "Electronics — Level 1, Food Court — Level 2, Cinema — Level 3." That directory board is the Ingress resource. The mall entrance doors are the Ingress Controller (NGINX). A customer types api.ticketflow.io/tickets in their browser — they reach the one entrance, the directory routes them to ticket-service.

Kubernetes has three types of Service that expose workloads:

  • ClusterIP — only reachable inside the cluster. The default. Used for service-to-service communication.
  • NodePort — exposes a port on every node's IP. Works but is awkward (random high port numbers) and bypasses the cloud load balancer.
  • LoadBalancer — provisions a cloud load balancer (AWS ALB/NLB, GCP LB). Works great, but one LB per Service means one cloud LB per microservice. For 10 services: 10 load balancers, 10 external IPs, and a significant monthly bill.

Ingress is a Kubernetes API object that defines HTTP/HTTPS routing rules. A single cloud LoadBalancer feeds into the Ingress Controller, which then routes to the correct internal ClusterIP service based on hostname and path. Ten services, one load balancer.

How Ingress Works Technically

Ingress has two distinct parts that must exist together:

1. The Ingress Resource

A declarative YAML object stored in the Kubernetes API server. It describes the routing rules: "requests for api.ticketflow.io with path /api/users go to user-service on port 3000." The Ingress resource itself does nothing by itself — it is just configuration data.

2. The Ingress Controller

A running Pod (or Deployment) inside the cluster that watches for Ingress resources via the Kubernetes API and enforces them. The most common is the NGINX Ingress Controller (maintained by the Kubernetes community). When you create or update an Ingress resource, the controller reads it and dynamically updates the NGINX configuration inside its pod — then reloads NGINX to apply the new routing rules.

Other controllers include: Traefik, Contour, HAProxy Ingress, and cloud-native controllers like AWS ALB Ingress Controller (which provisions an AWS Application Load Balancer directly rather than running NGINX in the cluster).

Routing types

  • Path-based routing — same host, different paths: /api/users → user-service, /api/tickets → ticket-service, /api/payments → payment-service.
  • Host-based routing — different subdomains to different services: users.ticketflow.io → user-service, tickets.ticketflow.io → ticket-service. Often used to separate internal APIs from public-facing frontends.
  • TLS termination — HTTPS is decrypted at the Ingress controller. Backend services communicate unencrypted inside the cluster (or with their own internal TLS). The Ingress resource references a Kubernetes Secret containing the TLS certificate and private key.

NGINX Ingress Controller — How it fits in AWS EKS

On EKS, installing the NGINX Ingress Controller creates a Kubernetes Service of type LoadBalancer. AWS provisions an NLB (Network Load Balancer) automatically. All external HTTPS traffic hits the NLB → NLB passes TCP to NGINX pods in the cluster → NGINX evaluates routing rules → forwards to the correct ClusterIP service → reaches your application pods.

TLS with cert-manager

cert-manager is a Kubernetes add-on that automates TLS certificate issuance and renewal. It integrates with Let's Encrypt (free, trusted certificates) via the ACME protocol. You annotate your Ingress resource with cert-manager.io/cluster-issuer: letsencrypt-prod and cert-manager handles the rest: domain validation, certificate request, storing the cert in a Secret, and renewing it 30 days before expiry. Zero manual certificate management.

Enterprise Scenario: TicketFlow

Real-World Example: TicketFlow on AWS EKS

TicketFlow's Kubernetes cluster on EKS runs five microservices. Initially the team deployed each service with type: LoadBalancer. After two weeks, the AWS bill showed five NLBs at $20/month each plus $0.008 per LCU-hour — an extra $200/month just for load balancers, before any traffic costs. The operations team also had five different DNS names to manage, five TLS certificates to renew, and five different places to configure rate limiting.

The architecture team consolidated to a single NGINX Ingress Controller. One NLB. One wildcard certificate (*.ticketflow.io) managed by cert-manager. Rate limiting configured centrally with annotations. All services reachable under api.ticketflow.io with path-based routing. Cost dropped from $200/month to $20/month for the single NLB.

The API Gateway pattern was implemented on top: the existing api-gateway (NGINX) service behind the Ingress handles authentication (JWT validation), request logging, and response transformation — responsibilities too application-specific for the raw Ingress controller.

Code Examples

Install NGINX Ingress Controller on EKS (Helm)

# Add the ingress-nginx Helm repository
helm repo add ingress-nginx https://kubernetes.github.io/ingress-nginx
helm repo update

# Install the NGINX Ingress Controller
# The service.type=LoadBalancer triggers AWS to provision an NLB
helm install ingress-nginx ingress-nginx/ingress-nginx \
  --namespace ingress-nginx \
  --create-namespace \
  --set controller.service.type=LoadBalancer \
  --set controller.service.annotations."service\.beta\.kubernetes\.io/aws-load-balancer-type"="nlb" \
  --set controller.replicaCount=2

# Verify the controller pods are running
kubectl get pods -n ingress-nginx

kubectl get svc -n ingress-nginx ingress-nginx-controller

kind: ClusterIssuer

Path-Based Ingress — All TicketFlow Services

# ticketflow-ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: ticketflow-ingress
  namespace: production
  annotations:
    # Specify which Ingress Controller handles this resource
    kubernetes.io/ingress.class: "nginx"

    # cert-manager: auto-issue and renew TLS certificate from Let's Encrypt
    cert-manager.io/cluster-issuer: "letsencrypt-prod"

    # Rewrite target: strip the path prefix before forwarding to the service
    # e.g., /api/users/profile → /profile at user-service
    nginx.ingress.kubernetes.io/rewrite-target: /$2

    # Rate limiting: 100 requests per minute per IP
    nginx.ingress.kubernetes.io/limit-rps: "100"
    nginx.ingress.kubernetes.io/limit-connections: "20"

    # Enable CORS for the frontend SPA
    nginx.ingress.kubernetes.io/enable-cors: "true"
    nginx.ingress.kubernetes.io/cors-allow-origin: "https://app.ticketflow.io"
    nginx.ingress.kubernetes.io/cors-allow-methods: "GET, POST, PUT, DELETE, OPTIONS"

    # Proxy timeouts (payment service may need longer)
    nginx.ingress.kubernetes.io/proxy-connect-timeout: "10"
    nginx.ingress.kubernetes.io/proxy-read-timeout: "60"
    nginx.ingress.kubernetes.io/proxy-send-timeout: "60"

    # Upload size limit (for ticket attachments)
    nginx.ingress.kubernetes.io/proxy-body-size: "10m"

spec:
  # TLS termination: cert-manager will create/renew this Secret automatically
  tls:
    - hosts:
        - api.ticketflow.io
      secretName: ticketflow-tls-cert  # cert-manager writes the cert here

  rules:
    - host: api.ticketflow.io
      http:
        paths:
          # user-service: Node.js — handles auth and user profiles
          - path: /api/users(/|$)(.*)
            pathType: Prefix
            backend:
              service:
                name: user-service
                port:
                  number: 3000

          # ticket-service: Spring Boot — handles ticket CRUD and search
          - path: /api/tickets(/|$)(.*)
            pathType: Prefix
            backend:
              service:
                name: ticket-service
                port:
                  number: 8080

          # payment-service: Python — handles payment processing
          - path: /api/payments(/|$)(.*)
            pathType: Prefix
            backend:
              service:
                name: payment-service
                port:
                  number: 5000

          # order-service: Go — handles order lifecycle
          - path: /api/orders(/|$)(.*)
            pathType: Prefix
            backend:
              service:
                name: order-service
                port:
                  number: 8090

          # notification-service: Node.js — handles email/push notifications
          - path: /api/notifications(/|$)(.*)
            pathType: Prefix
            backend:
              service:
                name: notification-service
                port:
                  number: 4000

Behind the Scenes

What happens when you kubectl apply -f ticketflow-ingress.yaml?
  1. kubectl sends an HTTP PUT to the Kubernetes API server (kube-apiserver). The API server validates the Ingress manifest against the networking.k8s.io/v1 schema and writes it to etcd.
  2. The NGINX Ingress Controller has a long-running watch (an HTTP long-poll) against the API server for Ingress objects. The API server notifies the controller about the new/updated Ingress.
  3. The controller's reconciliation loop runs. It reads all Ingress objects in all namespaces (filtered by its configured class). It builds an in-memory model of all routing rules across all Ingresses.
  4. The controller generates a complete nginx.conf from this model. Each location block in nginx.conf corresponds to a path rule in an Ingress resource. TLS configuration references the Kubernetes Secrets containing certificates.
  5. The controller checks if the new nginx.conf differs from the current one. If it does, it writes the new config to the NGINX process and sends it a signal to reload gracefully (NGINX reload is hot — in-flight requests complete before workers are replaced).
  6. Simultaneously, cert-manager watches for Ingress resources with the cert-manager.io/cluster-issuer annotation. It creates a Certificate object, which triggers the cert-manager controller to request a certificate from Let's Encrypt via HTTP-01 challenge. cert-manager temporarily serves the challenge response through the Ingress controller, Let's Encrypt validates domain ownership, and the certificate is issued and stored in the referenced Kubernetes Secret.
  7. NGINX reloads again to pick up the new TLS certificate from the Secret (cert-manager updates the Secret and the controller watches for Secret changes too).

The entire flow from kubectl apply to live HTTPS routing typically completes in under 90 seconds for a new certificate, or under 5 seconds for a routing change on an existing domain.

How does Let's Encrypt HTTP-01 challenge work inside a Kubernetes cluster?

The ACME HTTP-01 challenge requires that a file be served at http://yourdomain.com/.well-known/acme-challenge/TOKEN. Let's Encrypt makes an HTTP request to that URL from its servers; if the correct token is returned, it proves you control the domain.

  1. cert-manager creates a temporary Ingress rule: requests for api.ticketflow.io/.well-known/acme-challenge/TOKEN route to a temporary cert-manager pod running an HTTP server that responds with the correct token.
  2. cert-manager notifies Let's Encrypt that the challenge is ready.
  3. Let's Encrypt's servers make an HTTP request to http://api.ticketflow.io/.well-known/acme-challenge/TOKEN. This request comes in through the NLB → NGINX → temporary cert-manager pod.
  4. Let's Encrypt receives the correct token, validates domain ownership, and issues the certificate.
  5. cert-manager stores the certificate and private key in the Kubernetes Secret specified in the Ingress TLS configuration.
  6. cert-manager deletes the temporary Ingress rule and temporary pod.

For wildcard certificates (*.ticketflow.io), HTTP-01 cannot be used — you must use DNS-01 challenge, which requires cert-manager to create a DNS TXT record. cert-manager has plugins for Route 53 (AWS), Cloud DNS (GCP), and others.

Common Mistakes

Mistake 1: Using pathType: Exact when you need Prefix

pathType: Exact only matches the exact path string — /api/users matches, but /api/users/123 does not. Every request to a user profile returns 404. Most REST APIs need pathType: Prefix to match the base path and all sub-paths.

Fix: Use pathType: Prefix for all API services. Use pathType: Exact only for very specific single-endpoint rules (like webhooks at a fixed URL).

Mistake 2: Forgetting the rewrite-target annotation

You configure path: /api/users(/|$)(.*) to route to user-service. But user-service's routes start at /users/..., not /api/users/.... Without rewrite-target: /$2, NGINX forwards the full path /api/users/profile to user-service, which has no route matching /api/users/profile and returns 404.

Fix: Use the capture group pattern (/|$)(.*) in the path and annotate with nginx.ingress.kubernetes.io/rewrite-target: /$2. This strips the prefix before forwarding to the backend service.

Mistake 3: Not specifying the Ingress class

When multiple Ingress controllers exist in a cluster (e.g., both NGINX and AWS ALB controller), not specifying kubernetes.io/ingress.class annotation or the spec.ingressClassName field means both controllers try to reconcile the same Ingress resource. You get conflicting configurations and unpredictable behaviour — sometimes both controllers handle the rule, sometimes neither does.

Fix: Always set either the annotation (kubernetes.io/ingress.class: "nginx") or the spec field (spec.ingressClassName: nginx). The spec field is the current Kubernetes-standard approach (annotations are legacy but still widely used).

Mistake 4: Using the letsencrypt-staging issuer in production

cert-manager's Let's Encrypt staging issuer (https://acme-staging-v02.api.letsencrypt.org/directory) issues certificates that are not trusted by browsers — you get "Your connection is not private" errors. Teams set up staging for testing and forget to switch to production (https://acme-v02.api.letsencrypt.org/directory) before going live.

Fix: Always create two ClusterIssuers — letsencrypt-staging and letsencrypt-prod. Test with staging first (it has much higher rate limits). Switch to the prod issuer for the final deployment. Delete the staging Secret and certificate to trigger cert-manager to re-issue using the prod issuer.

Mistake 5: Expecting Ingress to handle non-HTTP protocols

Ingress is designed for HTTP and HTTPS only. Teams try to route gRPC, WebSocket, or database connections through a standard Ingress and wonder why it doesn't work. Standard NGINX Ingress supports WebSocket with nginx.ingress.kubernetes.io/proxy-read-timeout: "3600" and gRPC with nginx.ingress.kubernetes.io/backend-protocol: "GRPC", but raw TCP/UDP is not supported.

Fix: For gRPC, add the backend-protocol: "GRPC" annotation and configure H2 upstream. For raw TCP (Redis, PostgreSQL), use a separate LoadBalancer Service or configure TCP Services via the NGINX controller's TCP ConfigMap. For WebSocket, set the proxy timeout annotations to prevent connection drops.

Ingress vs API Gateway

This is a question architects frequently get wrong. Ingress and an API Gateway are related but distinct concepts — and in production, you often need both.

Capability NGINX Ingress Controller API Gateway (Kong, AWS API GW)
Path/host routing Yes Yes
TLS termination Yes Yes
Rate limiting Basic (per IP, global) Advanced (per user, per plan, token bucket)
Authentication Basic (auth-url annotation) Full (JWT, OAuth2, API keys, mTLS)
Request transformation Limited (header add/remove) Full (body rewrite, protocol translation)
Observability Access logs, basic metrics Per-route metrics, distributed tracing, analytics dashboard
Developer portal No Yes (API documentation, key management)
Circuit breaking No Yes
Canary / A/B routing Basic (NGINX annotations) Advanced (weighted routing, header-based)
Cost Free (open source) Paid (AWS API GW: per-request) or OSS (Kong)
When to use which

Use Ingress alone when: you have a small number of internal services, your authentication is handled by the services themselves (e.g., each service validates JWTs), you don't need to expose APIs to external partners, and advanced traffic management isn't required yet.

Add an API Gateway when: you need to expose APIs to third-party developers or external partners (requires developer portal, API key management), you want centralised authentication so no service ever receives unauthenticated requests, you need per-consumer rate limits and quotas, or you need request/response transformation between legacy and new API formats.

Common production pattern: Ingress Controller (NGINX) handles L7 routing and TLS termination at the cluster boundary. Behind it, a dedicated API Gateway service (Kong, Ambassador) handles authentication, authorization, rate limiting, and developer-facing API management. The API Gateway talks to backend services as a regular Kubernetes service-to-service call.

Production Readiness Checklist

  • NGINX Ingress Controller deployed with at least 2 replicas (single pod = single point of failure)
  • cert-manager installed and ClusterIssuer configured for letsencrypt-prod
  • All Ingress resources specify spec.ingressClassName or the class annotation
  • TLS configured for all external-facing hosts — no HTTP-only endpoints in production
  • Appropriate pathType selected per route (Prefix vs Exact vs ImplementationSpecific)
  • rewrite-target annotation set correctly to avoid double-path prefixes reaching backend services
  • Rate limiting annotations configured on sensitive endpoints (authentication, payments)
  • Proxy timeouts adjusted for services with long-running operations (file upload, report generation)
  • Ingress Controller resource requests and limits set — it handles ALL external traffic
  • Ingress Controller Pods spread across availability zones (PodAntiAffinity or topologySpreadConstraints)
  • Certificate expiry monitoring set up (Prometheus alert on certmanager_certificate_expiration_timestamp_seconds)
  • Let's Encrypt rate limits understood — 50 certificates per registered domain per week; wildcard certs use DNS-01
  • Default backend configured (custom 404/503 pages instead of NGINX default)
  • Access logs forwarded to Loki/ELK for security audit and debugging

Interview Questions

These are the most commonly asked questions about Kubernetes Ingress in DevOps, platform engineering, and software architecture interviews.

Q1: What is a Kubernetes Ingress, and why would you use it instead of a LoadBalancer Service for each microservice?

A Kubernetes Ingress is an API object that manages external HTTP/HTTPS access to services inside a cluster. It defines routing rules — mapping incoming hostnames and URL paths to backend Kubernetes Services.

The key reason to use Ingress instead of per-service LoadBalancer is cost and operational efficiency. A type: LoadBalancer Service on a cloud provider (AWS, GCP, Azure) provisions a dedicated cloud load balancer for that service. Each cloud load balancer has a fixed monthly cost — on AWS, an NLB costs approximately $20/month plus usage. For 10 microservices, that's $200/month just for load balancers, plus 10 separate DNS names, 10 separate TLS certificates to manage, and 10 places to configure rate limiting.

With Ingress, one cloud load balancer feeds into the Ingress Controller (e.g., NGINX running in the cluster). The controller routes to any number of services based on hostname and path. Ten services, one load balancer, one TLS cert, one place to configure cross-cutting concerns like rate limiting.

When to still use LoadBalancer directly: For non-HTTP protocols (gRPC, TCP), for services that need their own dedicated IP for compliance reasons, or for very high-traffic services that would bottleneck through a shared Ingress.

Q2: What is the difference between an Ingress resource and an Ingress Controller?

They are two distinct things that work together — a common source of confusion.

An Ingress resource is a declarative Kubernetes API object (a YAML file you apply with kubectl apply). It stores routing rules in the Kubernetes API server: "requests for host X with path Y should go to Service Z on port P." By itself, the resource does nothing — it is just configuration data sitting in etcd.

An Ingress Controller is a running application (a Deployment of pods) inside the cluster that watches for Ingress resources and acts on them. It reads the routing rules from the Ingress objects and configures an underlying proxy (NGINX, Traefik, HAProxy, etc.) to enforce those rules. Without an Ingress Controller installed, creating Ingress resources has zero effect — requests will not be routed anywhere.

Kubernetes provides the Ingress API as a standard interface, but it does not provide an Ingress Controller — you must install one yourself. Popular choices: NGINX Ingress Controller (community), Traefik, Contour, and cloud-native controllers like AWS ALB Ingress Controller.

Analogy: The Ingress resource is a letter of instructions ("route flights X to gate A, flights Y to gate B"). The Ingress Controller is the airport staff who read the letter and physically direct passengers to the correct gates.

Q3: How does TLS termination work with Kubernetes Ingress and cert-manager?

TLS termination at the Ingress means HTTPS is decrypted at the Ingress Controller. Backend services receive plain HTTP (or their own internal TLS if configured). The browser-to-Ingress connection is encrypted; the Ingress-to-service connection is unencrypted inside the cluster (generally acceptable since cluster network traffic is trusted).

The Ingress resource references a Kubernetes Secret containing the TLS certificate and private key via the spec.tls field. The NGINX controller reads this Secret and configures NGINX's SSL context with it.

cert-manager automates the lifecycle of this Secret:

  1. You create a ClusterIssuer resource that configures how certificates are obtained (e.g., from Let's Encrypt via ACME).
  2. You annotate your Ingress with cert-manager.io/cluster-issuer: "letsencrypt-prod".
  3. cert-manager watches for this annotation and creates a Certificate CRD resource, which triggers it to request a certificate from Let's Encrypt.
  4. Let's Encrypt validates domain ownership via HTTP-01 challenge (cert-manager temporarily routes the challenge through NGINX).
  5. Let's Encrypt issues the certificate; cert-manager stores it in the referenced Kubernetes Secret.
  6. cert-manager monitors the certificate's expiry and renews it automatically 30 days before expiry.

The result: zero manual certificate management. Add a new hostname to your Ingress with the annotation, and the certificate is issued and renewed forever without human intervention.

Q4: What is the difference between path-based routing and host-based routing in Ingress?

Path-based routing routes requests based on the URL path, all under the same hostname:

  • api.ticketflow.io/api/users → user-service
  • api.ticketflow.io/api/tickets → ticket-service
  • api.ticketflow.io/api/payments → payment-service

This uses a single domain, a single TLS certificate, and is the most common pattern for microservice APIs. Backend services must be aware their routes have a path prefix (or you use rewrite-target to strip it).

Host-based routing routes requests based on the HTTP Host header (the subdomain), regardless of path:

  • users.ticketflow.io → user-service (all paths)
  • tickets.ticketflow.io → ticket-service (all paths)
  • payments.ticketflow.io → payment-service (all paths)

This requires more DNS entries (one per service) and a wildcard certificate (*.ticketflow.io), but avoids path prefix complexity — each service sees requests at its own root path. Useful when services are developed independently and you don't want to coordinate URL prefixes across teams.

In practice, both can be combined: different hostnames for broad separation (internal vs external APIs, versioning like v2.api.ticketflow.io) and path routing within each hostname for the individual services.

Q5: What is the difference between an API Gateway and a Kubernetes Ingress Controller?

Both sit at the edge of the system and route external requests to internal services, but they operate at different layers of abstraction and offer different capabilities.

Kubernetes Ingress Controller (e.g., NGINX) is infrastructure-level L7 routing. It handles: URL path and hostname-based routing, TLS termination, basic rate limiting (by IP), basic authentication (via external auth service), and load balancing across service replicas. It is configured via Kubernetes Ingress resources and annotations. Its primary job is traffic routing — it is a proxy, not an application platform.

API Gateway (e.g., Kong, AWS API Gateway, Ambassador) is a higher-level application platform. It handles everything an Ingress does, plus: JWT and OAuth2 authentication (the gateway validates tokens — backend services never receive unauthenticated requests), per-user and per-plan rate limiting (not just per-IP), request/response transformation, API versioning, developer portal with documentation, API key management, circuit breaking, detailed per-route observability with analytics.

Common production pattern: Ingress Controller at the cluster boundary handles TLS and initial routing. An API Gateway pod runs inside the cluster — the Ingress routes /api/* to the API Gateway service, which then handles authentication and routes to individual microservices. This separates infrastructure concerns (Ingress) from API management concerns (Gateway).

When Ingress alone is sufficient: internal microservices not exposed to third parties, where each service handles its own authentication, and where advanced API management features are not needed.

Q6: How would you implement rate limiting on Kubernetes Ingress, and what are its limitations?

NGINX Ingress Controller supports rate limiting via annotations on the Ingress resource:

  • nginx.ingress.kubernetes.io/limit-rps: "100" — maximum 100 requests per second per IP address
  • nginx.ingress.kubernetes.io/limit-connections: "20" — maximum 20 concurrent connections per IP
  • nginx.ingress.kubernetes.io/limit-rpm: "1000" — maximum 1000 requests per minute per IP

These annotations add NGINX limit_req and limit_conn directives to the generated nginx.conf. NGINX enforces the limits using shared memory zones — all worker processes share the rate limiting state.

Limitations:

  • Per-IP only — you cannot rate limit by user ID, API key, or authenticated identity. One IP can represent thousands of users behind a NAT gateway.
  • Not distributed — each NGINX pod has its own rate limiting state. With 3 Ingress Controller replicas and a limit of 100 rps, a single client can effectively make 300 rps (100 per pod). The shared memory zone is per-pod, not cluster-wide.
  • No quotas — you cannot implement "100 requests per hour" burst quotas, only constant rate limits.
  • No per-user plans — you cannot give premium users 1000 rps and free users 100 rps based on their subscription.

For production rate limiting beyond these basics: use an API Gateway (Kong has a Rate Limiting plugin that stores counters in Redis, enabling distributed rate limiting across all gateway instances) or implement rate limiting in a custom admission webhook or in the services themselves using token bucket algorithms.

Q7: TicketFlow's payment-service is getting fraudulent traffic. How would you lock it down at the Ingress level?

This is a multi-layered defence problem. At the Ingress level, several controls are available:

1. IP allowlisting — if payment callbacks only come from a known payment processor (e.g., Stripe, PayPal), whitelist their published IP ranges and reject everything else:

nginx.ingress.kubernetes.io/whitelist-source-range: "54.187.174.169/32,54.187.205.235/32"

2. Strict rate limiting — reduce the per-IP rate limit aggressively for payment endpoints (10 rps instead of 100 rps for the general API):

nginx.ingress.kubernetes.io/limit-rps: "10"
nginx.ingress.kubernetes.io/limit-connections: "3"

3. Separate Ingress resource — create a dedicated Ingress just for /api/payments with stricter annotations, rather than modifying the shared Ingress resource and risk affecting other services.

4. External authentication — require requests to carry a valid JWT (issued during checkout) using the nginx.ingress.kubernetes.io/auth-url annotation to delegate authentication to an external service. Requests without valid tokens are rejected at the Ingress before reaching payment-service.

5. WAF (Web Application Firewall) — for production fraud prevention, deploy AWS WAF in front of the NLB. WAF provides managed rule sets for common attacks, geographic blocking, and custom rules based on request patterns.

Beyond Ingress (defence in depth): at the network level, use a Kubernetes NetworkPolicy to restrict which pods can call payment-service at all — only order-service should be allowed to initiate payment calls, never external traffic directly to payment-service's ClusterIP.
