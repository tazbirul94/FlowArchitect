Docker Volumes & Storage

Master container storage: named volumes, bind mounts, tmpfs, database persistence patterns, and backup/restore procedures.

Story

Containers are deliberately stateless — when you remove a container, everything inside it is gone. That's actually a feature, not a bug. It means containers are disposable, reproducible, and safe to replace.

But your database data? Your uploaded files? Your generated reports? Those cannot be disposable. You need a way to keep data alive even when the container that wrote it is long gone.

Think of it like a hotel room. You stay in the room (the container). When you leave, the room is cleaned and reset for the next guest. But your luggage stays at the bellhop desk (the volume) — safe, persistent, ready for your next stay or for a different room.

Why Containers Are Stateless by Default

Every container has a thin writable layer on top of its read-only image layers. All writes go to this writable layer. When the container is removed (docker rm), the writable layer is deleted. The image layers remain unchanged — this is why images can be shared across many running containers simultaneously.

This behavior is correct for application code and configs, but wrong for data that must outlive the container lifecycle: databases, user uploads, generated reports, and audit logs.

Beginner

A stopped container (docker stop) is not the same as a removed container (docker rm). Stopped containers keep their writable layer intact. docker start brings them back with all previously-written data still present. Data is only lost when you docker rm a container that has no volume attached.

Use volumes from the start — don't rely on the writable layer for anything you care about. The writable layer is scratch space, not storage.

Three Storage Types

Docker gives you three distinct mechanisms for persisting or sharing data. Choosing the right one depends on who manages the data, where it lives, and whether it must survive a container restart.

Type Where data lives Managed by Use case Survives docker rm?
Named Volume Docker's volume dir (/var/lib/docker/volumes/) Docker Databases, persistent app data ✓ Yes
Bind Mount Any path on the host filesystem Host OS Dev hot-reload, config files ✓ Yes (it's on the host)
tmpfs Host RAM (memory only) Kernel Temp secrets, sensitive data, performance ✗ No (RAM)

Named Volumes

Named volumes are the recommended way to persist data. Docker creates and manages the storage directory, handles permissions across platforms, and the volume exists completely independently of any individual container. You can stop, delete, and recreate containers freely — the data in the volume is untouched.

# Create a named volume explicitly
docker volume create postgres_data

# Use volume when running container
docker run -d \
  --name my-postgres \
  -v postgres_data:/var/lib/postgresql/data \
  -e POSTGRES_PASSWORD=secret \
  postgres:16-alpine

# The volume is created automatically if it doesn't exist
# -v volume_name:container_path

# List all volumes
docker volume ls

# Inspect volume (see actual path on host)
docker volume inspect postgres_data
# Output shows: "Mountpoint": "/var/lib/docker/volumes/postgres_data/_data"

# ---------------------------------------------------------------
# Proving data persistence across containers
# ---------------------------------------------------------------

# 1. Write some data to postgres
# 2. Stop and remove the container
docker stop my-postgres && docker rm my-postgres

# 3. Start a NEW container with the SAME volume
docker run -d \
  --name my-postgres-new \
  -v postgres_data:/var/lib/postgresql/data \
  -e POSTGRES_PASSWORD=secret \
  postgres:16-alpine

# 4. Data is still there — the volume survived the container deletion

# ---------------------------------------------------------------
# Volume cleanup
# ---------------------------------------------------------------

# Remove a specific volume (only when no container is using it)
docker volume rm postgres_data

# Remove all unused volumes (DANGER — data loss if you have important volumes)
docker volume prune

Bind Mounts

Bind mounts map any directory or file on your host filesystem directly into the container. Unlike named volumes, you choose the exact host path. This makes bind mounts ideal for development workflows where you want changes on the host to be immediately visible inside the container (hot-reload), or for injecting config files without rebuilding an image.

# Bind mount: map host directory into container
# Syntax: -v /absolute/host/path:/container/path[:options]

# Mount current directory's src/ as /app/src (dev hot-reload)
docker run -d \
  --name my-app \
  -v $(pwd)/src:/app/src \
  -p 3000:3000 \
  node:18-alpine \
  node --watch /app/src/server.js

# Mount a config file read-only
docker run -d \
  --name my-nginx \
  -v $(pwd)/nginx.conf:/etc/nginx/nginx.conf:ro \
  -p 80:80 \
  nginx:alpine

# Mount a log directory (app writes, host collects)
docker run -d \
  --name my-app \
  -v /var/log/myapp:/app/logs \
  my-app:latest
# Windows paths in PowerShell — use ${PWD} for current directory
docker run -d `
  --name my-app `
  -v ${PWD}/src:/app/src `
  -p 3000:3000 `
  node:18-alpine `
  node --watch /app/src/server.js

# Using explicit Windows-style path
docker run -d `
  --name my-app `
  -v C:\Users\me\project\src:/app/src `
  node:18-alpine

# Mount config file read-only
docker run -d `
  --name my-nginx `
  -v ${PWD}/nginx.conf:/etc/nginx/nginx.conf:ro `
  -p 80:80 `
  nginx:alpine

# NOTE: If using Windows-style paths (C:\...), ensure the drive
# is shared in Docker Desktop Settings > Resources > File Sharing
Common Mistake

Bind mounts require an absolute path on the host. Using ./src won't work — you must use $(pwd)/src on Linux/macOS or ${PWD}/src in PowerShell. On Windows with Docker Desktop, make sure the drive is shared under Docker Desktop Settings → Resources → File Sharing, otherwise Docker cannot mount directories from that drive.

Volume Mount Options

Both named volumes and bind mounts accept colon-separated options after the container path. These control access mode, sync behavior, and SELinux labeling.

# Read-only mount (container cannot write to this path)
-v myconfig:/config:ro

# Read-write (default — explicit form)
-v mydata:/data:rw

# ---------------------------------------------------------------
# macOS consistency options (Docker Desktop only)
# Control how quickly host and container filesystem views sync
# ---------------------------------------------------------------
-v $(pwd)/src:/app/src:consistent   # immediate sync both ways (slowest)
-v $(pwd)/src:/app/src:cached       # host is authoritative, container reads may lag (faster)
-v $(pwd)/src:/app/src:delegated    # container is authoritative, host reads may lag (fastest)

# ---------------------------------------------------------------
# SELinux labels — required on RHEL / Fedora / CentOS
# Without these, SELinux will block the mount
# ---------------------------------------------------------------
-v /host/data:/container/data:z     # relabel content for sharing with multiple containers
-v /host/data:/container/data:Z     # relabel content for exclusive use by THIS container only

tmpfs Mounts

tmpfs mounts write data into the host's RAM. The data never touches disk, which makes it ideal for sensitive data that must not persist after the container stops — session tokens, decrypted secrets, or intermediate cryptographic material. It is also faster than any disk-backed storage for high-throughput temporary files.

# Basic tmpfs mount — /tmp is RAM-backed, not written to disk
docker run -d \
  --name secure-app \
  --tmpfs /tmp \
  my-secure-app

# tmpfs with explicit options
docker run -d \
  --name secure-app \
  --tmpfs /run:rw,noexec,nosuid,size=100m \
  my-secure-app

# Options:
#   rw       — read-write (default)
#   noexec   — cannot execute binaries from this path
#   nosuid   — setuid/setgid bits are ignored
#   size=N   — maximum size in bytes (e.g. 100m, 512m)

# Use cases:
# - Temporary session files that must not persist to disk
# - Processing sensitive data (encryption keys, tokens)
# - High-speed scratch space (video frame processing, sort buffers)
# - /tmp in containers that handle many small temp files

Database Persistence Pattern

The single most important practical application of Docker volumes is running databases in containers. Getting this wrong means data loss. The following Compose file demonstrates every element of a production-like PostgreSQL setup: named volume with explicit declaration, init scripts, healthcheck, localhost-only port binding, and restart policy.

# docker-compose.yml — proper database persistence
version: '3.9'

services:
  postgres:
    image: postgres:16-alpine
    container_name: ticketflow-db
    environment:
      POSTGRES_DB: ticketflow
      POSTGRES_USER: ticketflow
      POSTGRES_PASSWORD: ${DB_PASSWORD:-devpassword}
      # Tune PostgreSQL encoding for the container environment
      POSTGRES_INITDB_ARGS: "--encoding=UTF8 --locale=en_US.UTF-8"
    volumes:
      # Main data volume — NEVER DELETE THIS
      - postgres_data:/var/lib/postgresql/data
      # SQL scripts in this directory run once on first start (volume empty)
      - ./db/init:/docker-entrypoint-initdb.d:ro
      # Custom postgresql.conf for memory/connection tuning
      - ./db/postgresql.conf:/etc/postgresql/postgresql.conf:ro
    command: postgres -c config_file=/etc/postgresql/postgresql.conf
    ports:
      - "127.0.0.1:5432:5432"   # bind to localhost only — not exposed to network
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ticketflow -d ticketflow"]
      interval: 10s
      timeout: 5s
      retries: 5
    restart: unless-stopped
    networks:
      - backend

volumes:
  postgres_data:
    driver: local
    # Uncomment to pin the data to a specific host path (e.g. fast SSD)
    # driver_opts:
    #   type: none
    #   o: bind
    #   device: /mnt/ssd/postgres-data

networks:
  backend:
    driver: bridge

Key decisions in this pattern: the postgres_data volume is declared at the top-level volumes: key, which makes it a named volume that persists across docker compose down runs. The port is bound to 127.0.0.1 only so the database is never exposed to the broader network. The healthcheck ensures dependent services wait until Postgres is actually ready to accept connections before starting.

Backup and Restore

A volume with no backup strategy is not a persistence strategy — it's deferred data loss. The commands below cover filesystem-level backups using Alpine as a temporary container, and database-native dumps for PostgreSQL which produce consistent, point-in-time snapshots.

# ───────────────────────────────────────────
# BACKUP A NAMED VOLUME
# ───────────────────────────────────────────

# Method 1: Run a temp Alpine container, archive the volume contents
# --rm     = remove the helper container after it exits
# :ro      = volume is read-only for safety during backup
docker run --rm \
  -v postgres_data:/source:ro \
  -v $(pwd)/backups:/backup \
  alpine \
  tar czf /backup/postgres_backup_$(date +%Y%m%d_%H%M%S).tar.gz -C /source .

# Method 2: pg_dump for PostgreSQL (preferred — consistent, point-in-time backup)
# This does NOT require stopping the container
docker exec my-postgres pg_dump \
  -U ticketflow \
  -d ticketflow \
  --format=custom \
  --compress=9 \
  > backups/ticketflow_$(date +%Y%m%d_%H%M%S).pgdump


# ───────────────────────────────────────────
# RESTORE A NAMED VOLUME
# ───────────────────────────────────────────

# Method 1: From tar archive
# STOP the container first — never restore into a running database
docker stop my-postgres

# Create a fresh volume to restore into
docker volume create postgres_data_restored

# Extract the archive into the new volume
docker run --rm \
  -v postgres_data_restored:/target \
  -v $(pwd)/backups:/backup:ro \
  alpine \
  tar xzf /backup/postgres_backup_20240115_103045.tar.gz -C /target

# Start a container with the restored volume
docker run -d \
  --name my-postgres-restored \
  -v postgres_data_restored:/var/lib/postgresql/data \
  -e POSTGRES_PASSWORD=secret \
  postgres:16-alpine

# Method 2: pg_restore for PostgreSQL
docker exec -i my-postgres pg_restore \
  -U ticketflow \
  -d ticketflow \
  --clean \
  < backups/ticketflow_20240115_103045.pgdump


# ───────────────────────────────────────────
# MIGRATE VOLUME TO A NEW HOST
# ───────────────────────────────────────────

# On source host: stream the archive directly over SSH
docker run --rm \
  -v postgres_data:/source:ro \
  alpine \
  tar czf - -C /source . | ssh user@newhost "cat > /tmp/postgres_data.tar.gz"

# On destination host: create the volume and extract
docker volume create postgres_data
docker run --rm \
  -v postgres_data:/target \
  -v /tmp:/backup:ro \
  alpine \
  tar xzf /backup/postgres_data.tar.gz -C /target

Volumes Shared Between Containers

Multiple containers can mount the same named volume. A common pattern is a write-capable application container sharing an uploads directory with a read-only nginx container that serves those files.

# Sharing a volume between containers
services:
  app:
    image: myapp
    volumes:
      - shared_uploads:/app/uploads    # app writes uploaded files here

  nginx:
    image: nginx:alpine
    volumes:
      - shared_uploads:/usr/share/nginx/html/uploads:ro  # nginx serves them read-only

volumes:
  shared_uploads:

When to share volumes: Single-writer / multiple-reader is safe — one service writes, others read with :ro. This covers upload directories, static asset generation, and report output.

When not to share volumes: Multiple-writer scenarios require coordination. Two containers writing to the same files simultaneously without locking will produce corrupted data. If two services genuinely need to share mutable state, use a database, a message queue, or a network filesystem with proper locking semantics — not a raw Docker volume.

Volume Management Commands

The full volume CLI lets you create, inspect, filter, label, and clean up volumes. Knowing these commands prevents the two most common accidents: unintended data loss and runaway disk usage.

# Full volume lifecycle
docker volume create myvolume
docker volume ls
docker volume inspect myvolume
docker volume rm myvolume
docker volume prune          # removes all volumes not currently attached to a container

# ---------------------------------------------------------------
# Filtering
# ---------------------------------------------------------------
docker volume ls -f dangling=true    # volumes not attached to ANY container (safe to prune)
docker volume ls -f name=postgres    # volumes whose name contains "postgres"

# ---------------------------------------------------------------
# Disk usage
# ---------------------------------------------------------------
docker system df -v          # show disk usage broken down by volumes, images, containers

# ---------------------------------------------------------------
# Labels for organization (use in multi-project environments)
# ---------------------------------------------------------------
docker volume create \
  --label app=ticketflow \
  --label env=development \
  --label created=$(date +%Y%m%d) \
  postgres_data

# Filter volumes by label
docker volume ls -f label=app=ticketflow
docker volume ls -f label=env=development

Behind the Scenes

Where do named volumes physically live?

On a Linux host running Docker Engine natively, named volumes live at /var/lib/docker/volumes/<volume-name>/_data. You can verify the exact path with docker volume inspect <name> — look for the Mountpoint field in the JSON output.

On macOS and Windows with Docker Desktop, Docker runs inside a hidden Linux virtual machine. The volume data lives inside that VM, not directly on your macOS or Windows filesystem. The Mountpoint path shown by docker volume inspect is a path inside the VM. To access it directly on macOS: docker run --rm -it -v /var/lib/docker:/docker alpine ls /docker/volumes/ — this mounts the VM's Docker directory into a temporary Alpine container so you can browse it.

How does Docker mount volumes into containers?

Docker uses Linux bind mounts (mount --bind) under the hood for both named volumes and bind mounts. The volume's directory on the host is bind-mounted into the container's filesystem namespace at the specified container path.

The full picture: the container's root filesystem is assembled using OverlayFS, which stacks the read-only image layers under a single writable container layer. Volume mounts sit outside the OverlayFS stack — they are separate bind mounts that overlay a specific path in the merged view. This means writes to a volume path bypass OverlayFS entirely and go directly to the volume's backing storage, which is why volume I/O is significantly faster than writing to the container's writable layer.

What is a volume driver?

By default, Docker uses the local volume driver, which stores volume data on the local host filesystem. Volume drivers are plugins that swap out the backing store for something else entirely.

Third-party drivers enable volumes backed by network and cloud storage: NFS, AWS EBS, Azure Disk, GlusterFS, Ceph, and others. The container application sees the same mount interface regardless of where the data actually lives.

In production Kubernetes, this concept maps directly to StorageClasses and PersistentVolumes. A StorageClass is the Kubernetes equivalent of a volume driver — it describes how to provision storage (e.g., AWS EBS with gp3 type, or an NFS provisioner). The same architecture: the pod sees a mounted path, and the storage layer handles where the bytes actually live.

Common Mistakes

Mistake 1 — Running docker compose down -v

The -v flag deletes all named volumes defined in the Compose file. A team member who thinks they're just cleaning up their local environment runs this command and deletes months of development database data. Add a prominent comment in your docker-compose.yml: # WARNING: never run 'docker compose down -v' without a backup. Use docker compose down (without -v) as the standard cleanup command. The -v variant should only be used deliberately, after an explicit backup has been confirmed.

Mistake 2 — Running a database container without a volume

Starting a Postgres or MySQL container without a -v flag means all data goes into the container's writable layer. docker rm postgres — your entire database is gone. Always declare a named volume for every stateful container from the very first docker run. There is no way to recover writable-layer data after docker rm.

Mistake 3 — Bind mounting over node_modules

A common development setup is COPY . . in the Dockerfile plus a bind mount of the project root at runtime: -v $(pwd):/app. This overwrites the container's /app/node_modules (installed from a clean lockfile on a Linux base) with the host's node_modules (potentially compiled for macOS, with native binaries that won't run on Linux). Fix: add an anonymous volume to shadow the node_modules directory: -v /app/node_modules. This tells Docker to use a fresh, anonymous volume at that path instead of the bind-mounted host directory.

Mistake 4 — Volume ownership mismatch

Container processes often run as non-root users (e.g., the postgres image runs as UID 999). If the named volume's directory was created or initialized by root, Postgres cannot write to it and fails with a permission denied error. Fix: run a one-time init container that chowns the volume: docker run --rm -v myvolume:/data alpine chown -R 999:999 /data. Alternatively, verify permissions with docker run --rm -v myvolume:/data alpine ls -la /data. Note that the official PostgreSQL image handles this automatically when the volume is completely empty — but not when it was pre-initialized by root.

Mistake 5 — Forgetting that stopped containers still count as volume users

docker volume prune removes volumes that are not attached to any container — running or stopped. If your container is stopped (not removed), its volume is still considered "in use" and is safe. But docker volume prune runs aggressively across all projects on the host. If another project's container was stopped, its volumes will be pruned. Back up important volumes before running prune, and prefer docker volume rm <specific-name> over the blanket prune command.

Architect Notes

Architect Note

Volume management is where Docker's "pets vs cattle" philosophy gets nuanced.

Stateless containers (cattle): throw away any container, start a new one, same result. This works for app servers, API services, and workers. Kubernetes Deployments are designed for this class of workload — any pod replica is interchangeable.

Stateful containers (pets): databases and other stateful services need persistent storage and often need stable identity. You can't simply throw them away. In Kubernetes, StatefulSets handle this — each instance gets a stable network identity and its own PersistentVolumeClaim that is never reassigned to another pod.

In production, you generally don't run critical databases in containers on a single host. The patterns you'll encounter on real teams are:

  • Managed databases (AWS RDS, Google Cloud SQL, Azure Database) — no Docker at all; the cloud provider handles HA, failover, backups, and patching.
  • Kubernetes StatefulSets with cloud storage — EBS, Azure Disk, or GCE Persistent Disk backing PersistentVolumeClaims. Kubernetes handles scheduling, but the storage is durable cloud-backed disk.
  • Dedicated database clusters — CockroachDB, Vitess, TiDB, or MongoDB Atlas Operator for clustered deployments where the database manages its own replication and sharding on top of container orchestration.

Docker volumes are the right tool for: local development databases, CI/CD test databases (name them with the build ID, reset between runs), and non-critical or easily-recoverable stateful services like a Redis cache that can be rebuilt from the source of truth.

The pattern for CI cleanup: name volumes with the build ID (postgres_data_${BUILD_ID}), use them for the test run, then delete them explicitly after. Never rely on prune in CI — it is too aggressive and can delete volumes from parallel builds running on the same agent.

Production Checklist

  • All stateful services (databases, caches, file stores) use named volumes
  • Volume names are descriptive and include the service name (e.g. ticketflow_postgres_data)
  • Backup procedure is documented and has been tested end-to-end at least once
  • docker compose down (without -v) is the default team cleanup command
  • docker compose down -v requires an explicit decision and a confirmed backup first
  • Volume permissions match the container's running user (no root-owned directories for non-root processes)
  • In CI: volumes are named with the build or job ID and deleted explicitly after the pipeline completes
  • Database backup runs before any destructive migration or schema change
  • Restore procedure has been tested — a backup you have never restored from is an untested backup
  • Sensitive runtime data (session tokens, decryption keys) uses tmpfs, not a named volume

Hands-On Lab

Lab — Prove Data Persistence with Volumes

This lab walks through the complete volume lifecycle: start a database, write data, destroy the container, start a new container on the same volume, and verify the data survived. Then back the volume up to disk.

# Step 1: Start postgres WITH a named volume
docker run -d \
  --name lab-postgres \
  -v lab_postgres_data:/var/lib/postgresql/data \
  -e POSTGRES_PASSWORD=labpassword \
  -e POSTGRES_DB=labdb \
  postgres:16-alpine

# Step 2: Wait for postgres to be ready
sleep 5
docker exec lab-postgres pg_isready -U postgres

# Step 3: Create a table and insert data
docker exec lab-postgres psql -U postgres -d labdb -c "
  CREATE TABLE tickets (
    id SERIAL PRIMARY KEY,
    title VARCHAR(100),
    created_at TIMESTAMPTZ DEFAULT NOW()
  );
  INSERT INTO tickets (title) VALUES
    ('Rock Concert - Floor Ticket'),
    ('Jazz Night - VIP Pass'),
    ('Comedy Show - Balcony Seat');
  SELECT * FROM tickets;
"

# Step 4: DESTROY the container (not the volume)
docker stop lab-postgres
docker rm lab-postgres

# Step 5: Verify the volume still exists — the container is gone but data lives on
docker volume ls | grep lab_postgres_data

# Step 6: Start a BRAND NEW postgres container with the SAME volume
docker run -d \
  --name lab-postgres-v2 \
  -v lab_postgres_data:/var/lib/postgresql/data \
  -e POSTGRES_PASSWORD=labpassword \
  postgres:16-alpine

# Step 7: Wait and verify data persisted across the container deletion
sleep 5
docker exec lab-postgres-v2 psql -U postgres -d labdb -c "SELECT * FROM tickets;"
# Expected output: your 3 ticket rows from Step 3

# Step 8: Back up the volume to disk
mkdir -p lab-backups
docker run --rm \
  -v lab_postgres_data:/source:ro \
  -v $(pwd)/lab-backups:/backup \
  alpine \
  tar czf /backup/postgres_backup.tar.gz -C /source .

ls -lh lab-backups/
# Expected: postgres_backup.tar.gz with a non-zero file size

# Step 9: Clean up everything
docker stop lab-postgres-v2
docker rm lab-postgres-v2
docker volume rm lab_postgres_data

Interview Questions

Q1: What is the difference between a Docker volume and a bind mount?

Named volumes are fully managed by Docker — Docker creates and manages the directory (/var/lib/docker/volumes/name/_data), handles permissions, and the volume persists independently of all containers. You reference it by name, not by path. Bind mounts map any host filesystem path directly into a container — you choose the exact host directory, and Docker makes the connection. The host path must exist before the container starts.

When to use named volumes: production data that must persist reliably across container restarts and host migrations (database files, application uploads). Named volumes work identically on any Docker host, are portable, and can be managed with Docker CLI commands.

When to use bind mounts: development workflows where you need to edit source code on the host and have changes immediately visible inside the container (hot-reload). Also for injecting config files from the host, or for collecting logs into a specific host directory. Bind mounts are host-specific — a path valid on your laptop may not exist on a CI server.

Q2: How do you back up and restore a Docker named volume?

The canonical technique uses a temporary Alpine container mounted to both the target volume and a host backup directory. To back up: docker run --rm -v myvolume:/source:ro -v $(pwd)/backups:/backup alpine tar czf /backup/backup.tar.gz -C /source .. To restore: stop the container using the volume, then docker run --rm -v myvolume:/target -v $(pwd)/backups:/backup:ro alpine tar xzf /backup/backup.tar.gz -C /target.

For databases, prefer native dump tools over filesystem-level backup. pg_dump for PostgreSQL and mysqldump for MySQL produce consistent snapshots that capture a transactionally-coherent point in time, even on a live database. Filesystem-level backups can capture the database files mid-write, producing a backup that cannot be restored cleanly. For PostgreSQL: docker exec my-postgres pg_dump -U user -d dbname --format=custom > backup.pgdump.

Q3: A container starts successfully but the database says "permission denied" on its data directory. What is the issue?

Volume ownership mismatch. The named volume's directory on the host was created or written by root (UID 0), but the database container runs as a non-root user — for example, the official postgres image runs as UID 999. When the volume is mounted, the directory is owned by root and the postgres process cannot write to it.

Diagnose it: docker run --rm -v myvolume:/data alpine ls -la /data. If the output shows root root ownership, that is the problem.

Fix it with a one-time init container: docker run --rm -v myvolume:/data alpine chown -R 999:999 /data. Alternatively, if the volume is empty, the official PostgreSQL image will initialize the data directory with correct ownership automatically — but it will not fix ownership on a pre-existing, non-empty volume.

Q4: What happens to data when you run docker stop vs docker rm?

docker stop sends SIGTERM to PID 1 in the container, waits for a graceful shutdown (default 10 seconds), then sends SIGKILL if the process has not exited. The container's writable layer is preserved — all data written since the container started is still there. docker start brings the container back to exactly the same state.

docker rm deletes the container record and its writable layer. All data written to the container's filesystem that is not backed by a volume is permanently gone. Named volumes are not deleted by docker rm. Only docker rm -v (with the explicit -v flag) also removes anonymous volumes that were created with the container. Named volumes are never deleted by docker rm regardless of flags — they require docker volume rm or docker compose down -v.

Q5: When would you use tmpfs in production?

tmpfs is RAM-backed — data never touches disk and is gone when the container stops. Production use cases:

  • Sensitive session data: GDPR and compliance requirements sometimes mandate that session tokens or PII must not be written to persistent storage. tmpfs satisfies this at the infrastructure level.
  • Temporary processing of secrets: decrypted private keys, HMAC signing keys, or OAuth tokens that a process needs briefly but must not survive a restart.
  • High-throughput temporary files: video processing pipelines that generate thousands of intermediate frame files, or sort operations that overflow RAM. tmpfs eliminates disk I/O bottleneck for scratch data.
  • /tmp in containers with many small temp files: some applications generate many temporary files in /tmp. Keeping those in RAM avoids filling up the container's writable layer or a backing volume with garbage.

Caveats: size is limited by available RAM. Data is unconditionally lost on container restart. Swapping reduces the in-memory guarantee. Never use tmpfs for anything that must survive a container restart or a node reboot.
