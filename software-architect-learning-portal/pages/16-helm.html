Helm: The Kubernetes Package Manager

Package, version, and deploy Kubernetes applications at scale — from chart structure to enterprise release management with helmfile.

The Story Behind Helm

📦 The Kubernetes YAML Explosion

A startup adopts Kubernetes and deploys their first microservice. The engineer writes a Deployment YAML, a Service YAML, a ConfigMap, an HorizontalPodAutoscaler, an Ingress, and a ServiceAccount. Six files for one service. Simple enough — for now.

Six months later, the platform has twelve microservices, three environments (dev, staging, production), and a new requirement: every deployment must have resource limits, pod disruption budgets, and network policies. That's twelve services × three environments × nine YAML files each = 324 YAML files to maintain. Changing the image tag for a rollout means finding and editing the right file. Deploying the same service to staging and production means duplicating YAML and manually substituting values. Engineers start copy-pasting and drift creeps in — production has a different replica count than staging because someone edited the wrong file eight months ago.

Helm solves this with a concept borrowed from Linux package managers like apt and yum: the chart. A chart is a parameterised, versioned, distributable package for Kubernetes resources. You define the shape of your deployment once, supply environment-specific values, and Helm renders and deploys the whole thing as a named release. Rollback, upgrade, and diff are built-in primitives.

What is Helm?

🥪 Beginner Analogy

Think of Helm like a cooking recipe book. The recipe (the chart) describes how to make a dish — what ingredients, what quantities, what steps. The recipe itself is generic. When you actually cook it, you fill in specifics: two cups of flour for the small version, four cups for the large. The final cooked dish served on a specific date is the release. You can make the same recipe on Monday (v1) and again on Friday with different spices (v2). If Friday's version is terrible, you can roll back to Monday's without starting from scratch.

In Helm: the chart is the recipe, values.yaml is where you fill in quantities, and a release is the running application installed from that chart at a specific version.

Helm is the official CNCF package manager for Kubernetes. It does four things that raw kubectl apply cannot:

  • Templating — write Kubernetes YAML once with placeholders; supply environment-specific values at deploy time.
  • Release management — every deployment is tracked as a named release with a version history. You can upgrade, rollback, or diff with a single command.
  • Packaging and distribution — charts are tarballs stored in repositories (Artifact Hub, Nexus, S3, GitHub Pages). Pull and deploy any published chart in one command.
  • Dependency management — a chart can declare dependencies on other charts (e.g., your app chart depends on the official PostgreSQL chart), and Helm resolves them automatically.

Chart Structure

Every Helm chart is a directory (or a tarball of that directory) with a mandatory layout. Understanding each file's role is the foundation of everything else:

user-service/                  # Chart root — named after the application
├── Chart.yaml                 # Chart metadata: name, version, appVersion
├── values.yaml                # Default configuration values
├── values-staging.yaml        # Optional: staging overrides (not Helm standard, loaded via -f)
├── values-production.yaml     # Optional: production overrides
├── charts/                    # Unpacked dependency charts (managed by helm dependency update)
├── templates/                 # Kubernetes manifest templates
│   ├── _helpers.tpl           # Named template definitions (not rendered as manifests)
│   ├── deployment.yaml        # Deployment resource template
│   ├── service.yaml           # Service resource template
│   ├── ingress.yaml           # Ingress resource template
│   ├── configmap.yaml         # ConfigMap resource template
│   ├── hpa.yaml               # HorizontalPodAutoscaler template
│   ├── serviceaccount.yaml    # ServiceAccount template
│   └── NOTES.txt              # Post-install instructions printed to the user
└── .helmignore                # Files to exclude when packaging the chart

Chart.yaml — The Identity File

apiVersion: v2                    # Helm 3 uses v2; Helm 2 used v1
name: user-service                # Chart name — must match the directory name
description: TicketFlow user authentication and profile management service
type: application                 # "application" (deployable) or "library" (helpers only)
version: 1.4.2                    # Chart version — increment when the chart itself changes
appVersion: "2.8.0"               # Version of the application being packaged (informational)
keywords:
  - nodejs
  - ticketflow
  - microservice
maintainers:
  - name: Platform Team
    email: platform@ticketflow.io
dependencies:
  - name: redis                   # This chart depends on the official Redis chart
    version: ">=17.0.0"
    repository: "https://charts.bitnami.com/bitnami"
    condition: redis.enabled       # Only install if redis.enabled=true in values

Key distinction: version is the chart version — bump it when you change templates or defaults. appVersion is the application version — it's informational only and typically matches your Docker image tag. They version independently: you can release chart v1.4.3 that still deploys appVersion 2.8.0 because you only fixed a template bug.

templates/_helpers.tpl — Named Templates

The _helpers.tpl file (any file starting with underscore is not rendered as a manifest) defines reusable named templates using the define action. These are called across your other templates with include:

{{/*
  Expand the name of the chart.
  Usage: {{ include "user-service.name" . }}
*/}}
{{- define "user-service.name" -}}
{{- default .Chart.Name .Values.nameOverride | trunc 63 | trimSuffix "-" }}
{{- end }}

{{/*
  Create a default fully qualified app name.
  Truncates to 63 chars because Kubernetes DNS component max length is 63.
  Usage: {{ include "user-service.fullname" . }}
*/}}
{{- define "user-service.fullname" -}}
{{- if .Values.fullnameOverride }}
{{- .Values.fullnameOverride | trunc 63 | trimSuffix "-" }}
{{- else }}
{{- $name := default .Chart.Name .Values.nameOverride }}
{{- if contains $name .Release.Name }}
{{- .Release.Name | trunc 63 | trimSuffix "-" }}
{{- else }}
{{- printf "%s-%s" .Release.Name $name | trunc 63 | trimSuffix "-" }}
{{- end }}
{{- end }}
{{- end }}

{{/*
  Common labels applied to every resource.
  Usage: {{ include "user-service.labels" . | nindent 4 }}
*/}}
{{- define "user-service.labels" -}}
helm.sh/chart: {{ include "user-service.chart" . }}
{{ include "user-service.selectorLabels" . }}
{{- if .Chart.AppVersion }}
app.kubernetes.io/version: {{ .Chart.AppVersion | quote }}
{{- end }}
app.kubernetes.io/managed-by: {{ .Release.Service }}
{{- end }}

{{/*
  Selector labels for Deployments and Services (must be stable — never change after first deploy).
*/}}
{{- define "user-service.selectorLabels" -}}
app.kubernetes.io/name: {{ include "user-service.name" . }}
app.kubernetes.io/instance: {{ .Release.Name }}
{{- end }}

Templating Syntax

Helm uses the Go text/template engine, augmented with the Sprig library (over 100 helper functions). Templates are delimited by {{ and }}. The leading dash {{- trims whitespace before the action; the trailing dash -}} trims whitespace after. This matters for YAML indentation.

Built-in Objects

Object Description Example
.Values Values from values.yaml and -f overrides {{ .Values.image.tag }}
.Release Information about the current release {{ .Release.Name }}, {{ .Release.Namespace }}
.Chart Contents of Chart.yaml {{ .Chart.Version }}, {{ .Chart.AppVersion }}
.Files Access non-template files in the chart {{ .Files.Get "config/app.conf" }}
.Capabilities Kubernetes API capabilities of the cluster {{ .Capabilities.KubeVersion.GitVersion }}
.Template Current template name and base path {{ .Template.Name }}

A Real Deployment Template

apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ include "user-service.fullname" . }}
  namespace: {{ .Release.Namespace }}
  labels:
    {{- include "user-service.labels" . | nindent 4 }}
spec:
  {{- if not .Values.autoscaling.enabled }}
  replicas: {{ .Values.replicaCount }}
  {{- end }}
  selector:
    matchLabels:
      {{- include "user-service.selectorLabels" . | nindent 6 }}
  template:
    metadata:
      labels:
        {{- include "user-service.selectorLabels" . | nindent 8 }}
      {{- with .Values.podAnnotations }}
      annotations:
        {{- toYaml . | nindent 8 }}
      {{- end }}
    spec:
      {{- with .Values.imagePullSecrets }}
      imagePullSecrets:
        {{- toYaml . | nindent 8 }}
      {{- end }}
      serviceAccountName: {{ include "user-service.serviceAccountName" . }}
      containers:
        - name: {{ .Chart.Name }}
          image: "{{ .Values.image.repository }}:{{ .Values.image.tag | default .Chart.AppVersion }}"
          imagePullPolicy: {{ .Values.image.pullPolicy }}
          ports:
            - name: http
              containerPort: {{ .Values.service.port }}
              protocol: TCP
          env:
            {{- range $key, $value := .Values.env }}
            - name: {{ $key }}
              value: {{ $value | quote }}
            {{- end }}
            - name: DB_HOST
              value: {{ .Values.postgresql.host | required "postgresql.host is required" | quote }}
            - name: DB_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: {{ include "user-service.fullname" . }}-db-secret
                  key: password
          resources:
            {{- toYaml .Values.resources | nindent 12 }}
          livenessProbe:
            httpGet:
              path: /health
              port: http
            initialDelaySeconds: 30
            periodSeconds: 10
          readinessProbe:
            httpGet:
              path: /ready
              port: http
            initialDelaySeconds: 5
            periodSeconds: 5
      {{- with .Values.nodeSelector }}
      nodeSelector:
        {{- toYaml . | nindent 8 }}
      {{- end }}
      {{- with .Values.affinity }}
      affinity:
        {{- toYaml . | nindent 8 }}
      {{- end }}

Helm Release Lifecycle

Key insight: Helm maintains a full release history in Kubernetes Secrets (one Secret per revision, in the same namespace as the release). This is how rollback works without any external state — Helm reads the previous revision's manifest from the Secret and re-applies it.

Enterprise Scenario: TicketFlow

🏢 TicketFlow Platform — Six Services, Three Environments

TicketFlow runs six microservices on AWS EKS across three environments (dev, staging, production). Before Helm, the DevOps team maintained 18 separate directories of YAML files — one per service per environment. A routine change to add a Prometheus scrape annotation required 18 manual edits. Release coordination was done via a shared spreadsheet tracking which YAML had been updated.

After migrating to Helm:

  • Each service has one chart in the charts/ directory of the TicketFlow monorepo
  • Each environment has one override values file (values-staging.yaml, values-production.yaml)
  • Adding a Prometheus annotation means editing _helpers.tpl once — all 18 environments pick it up on next deploy
  • GitHub Actions builds the image, pushes to Nexus, then calls helm upgrade --install --set image.tag=$GIT_SHA
  • Argo CD watches the Helm chart repository and auto-syncs to production on chart version bump

Release Management

Helm's release management commands form the core workflow for deploying and operating applications:

# Install a new release (fails if release already exists)
helm install user-service ./charts/user-service \
  --namespace ticketflow-prod \
  --create-namespace \
  -f charts/user-service/values-production.yaml \
  --set image.tag=a3f9c2d

# Upgrade an existing release (fails if release does not exist)
helm upgrade user-service ./charts/user-service \
  --namespace ticketflow-prod \
  -f charts/user-service/values-production.yaml \
  --set image.tag=b7e4a1c

# The canonical CI/CD command: install if missing, upgrade if exists
# --atomic: rolls back automatically if upgrade fails
# --timeout: time to wait for pods to become ready
helm upgrade --install user-service ./charts/user-service \
  --namespace ticketflow-prod \
  --create-namespace \
  -f charts/user-service/values-production.yaml \
  --set image.tag=$IMAGE_TAG \
  --atomic \
  --timeout 5m \
  --wait

# Preview what Helm WOULD apply without actually deploying (dry run)
helm upgrade --install user-service ./charts/user-service \
  --namespace ticketflow-prod \
  -f charts/user-service/values-production.yaml \
  --set image.tag=$IMAGE_TAG \
  --dry-run

# Render templates locally without a cluster (useful in CI for validation)
helm template user-service ./charts/user-service \
  -f charts/user-service/values-production.yaml \
  --set image.tag=$IMAGE_TAG \
  | kubectl apply --dry-run=client -f -
# List all releases in a namespace
helm list --namespace ticketflow-prod

# List releases across all namespaces
helm list --all-namespaces

# Show full release status and last deployment notes
helm status user-service --namespace ticketflow-prod

# Show the full release history (revision numbers are key for rollback)
helm history user-service --namespace ticketflow-prod
# REVISION  UPDATED                  STATUS     CHART              APP VERSION  DESCRIPTION
# 1         2024-01-15 09:00:00      superseded user-service-1.3.0  2.7.0       Install complete
# 2         2024-01-20 14:30:00      superseded user-service-1.4.0  2.7.5       Upgrade complete
# 3         2024-02-01 11:00:00      deployed   user-service-1.4.2  2.8.0       Upgrade complete

# Show the values currently deployed for a release
helm get values user-service --namespace ticketflow-prod

# Show all values (including defaults) — useful for debugging
helm get values user-service --namespace ticketflow-prod --all

# Show the rendered manifests for the currently deployed revision
helm get manifest user-service --namespace ticketflow-prod

# Diff current deployed vs pending upgrade (requires helm-diff plugin)
helm diff upgrade user-service ./charts/user-service \
  --namespace ticketflow-prod \
  -f charts/user-service/values-production.yaml \
  --set image.tag=$NEW_IMAGE_TAG
# Roll back to the previous revision (most common case)
helm rollback user-service --namespace ticketflow-prod

# Roll back to a specific revision (check revision numbers with helm history)
helm rollback user-service 2 --namespace ticketflow-prod

# Roll back with wait: block until rollback pods are healthy
helm rollback user-service --namespace ticketflow-prod --wait --timeout 3m

# Roll back all TicketFlow services to specific revisions after bad deploy
helm rollback ticket-service 5 --namespace ticketflow-prod --wait
helm rollback payment-service 3 --namespace ticketflow-prod --wait
helm rollback order-service 4 --namespace ticketflow-prod --wait

# Uninstall a release (removes all resources AND the release history)
helm uninstall user-service --namespace ticketflow-prod

# Uninstall but keep history (useful for auditing — can still see past revisions)
helm uninstall user-service --namespace ticketflow-prod --keep-history

Helm Repositories

Helm charts are distributed through repositories — HTTP servers that serve a repository index file (index.yaml) and chart tarballs. There are three types you will encounter in practice:

1. Artifact Hub (Public)

Artifact Hub (artifacthub.io) is the official aggregator for public Helm charts, including charts from Bitnami, Prometheus Community, Grafana, and hundreds of others. Use it to find community charts for infrastructure dependencies:

# Add commonly used repositories
helm repo add bitnami https://charts.bitnami.com/bitnami
helm repo add prometheus-community https://prometheus-community.github.io/helm-charts
helm repo add grafana https://grafana.github.io/helm-charts
helm repo add ingress-nginx https://kubernetes.github.io/ingress-nginx
helm repo add cert-manager https://charts.jetstack.io

# Update all repo indexes (like apt-get update)
helm repo update

# Search for charts across all added repositories
helm search repo postgresql
helm search repo redis --versions

# Pull a chart locally (without installing) to inspect or customize
helm pull bitnami/postgresql --version 12.5.8
helm pull bitnami/redis --untar --destination ./charts/vendor/

2. Nexus as a Private Helm Repository

TicketFlow uses Nexus at nexus.ticketflow.io as the private Helm repository for internal charts. Nexus supports Helm repositories natively — it hosts the index.yaml and serves chart tarballs.

# Add the TicketFlow Nexus Helm repository (credentials from env vars)
helm repo add ticketflow \
  https://nexus.ticketflow.io/repository/helm-releases/ \
  --username $NEXUS_USER \
  --password $NEXUS_PASSWORD

# Package a chart into a tarball for upload
helm package ./charts/user-service
# Produces: user-service-1.4.2.tgz

# Upload the packaged chart to Nexus via curl (Nexus REST API)
curl -u "$NEXUS_USER:$NEXUS_PASSWORD" \
  --upload-file user-service-1.4.2.tgz \
  https://nexus.ticketflow.io/repository/helm-releases/

# After uploading, update local index and verify
helm repo update
helm search repo ticketflow/user-service --versions

# Deploy from the Nexus repository
helm upgrade --install user-service ticketflow/user-service \
  --version 1.4.2 \
  --namespace ticketflow-prod \
  -f values-production.yaml \
  --set image.tag=$IMAGE_TAG

3. OCI Registries (Modern Standard)

Since Helm 3.8, charts can be stored as OCI artifacts in any OCI-compliant registry — including Nexus, AWS ECR, GHCR, and Docker Hub. This is the direction the ecosystem is moving: one registry for both Docker images and Helm charts.

# Login to Nexus OCI endpoint
helm registry login nexus.ticketflow.io:5000 \
  --username $NEXUS_USER \
  --password $NEXUS_PASSWORD

# Push chart as OCI artifact (no separate index.yaml needed)
helm push user-service-1.4.2.tgz oci://nexus.ticketflow.io:5000/ticketflow/charts

# Install directly from OCI registry
helm upgrade --install user-service \
  oci://nexus.ticketflow.io:5000/ticketflow/charts/user-service \
  --version 1.4.2 \
  --namespace ticketflow-prod \
  --set image.tag=$IMAGE_TAG

Helmfile: Multi-Chart Deployments

Helm manages one release at a time. Deploying all six TicketFlow services means running six separate helm upgrade --install commands — in the right order, with the right values, to the right namespaces. Helmfile is a declarative wrapper that orchestrates multiple Helm releases as a single unit.

# helmfile.yaml — TicketFlow full-stack deployment definition

environments:
  dev:
    values:
      - environments/dev.yaml
  staging:
    values:
      - environments/staging.yaml
  production:
    values:
      - environments/production.yaml

repositories:
  - name: ticketflow
    url: https://nexus.ticketflow.io/repository/helm-releases/
    username: "{{ requiredEnv \"NEXUS_USER\" }}"
    password: "{{ requiredEnv \"NEXUS_PASSWORD\" }}"
  - name: bitnami
    url: https://charts.bitnami.com/bitnami

releases:
  # Deploy ingress controller first (other services depend on it)
  - name: ingress-nginx
    namespace: ingress-nginx
    chart: ingress-nginx/ingress-nginx
    version: "4.8.3"
    needs:
      - kube-system/cert-manager

  # Deploy cert-manager before ingress
  - name: cert-manager
    namespace: kube-system
    chart: cert-manager/cert-manager
    version: "v1.13.3"
    set:
      - name: installCRDs
        value: "true"

  # TicketFlow microservices — deployed after infrastructure
  - name: user-service
    namespace: ticketflow
    chart: ticketflow/user-service
    version: "1.4.2"
    values:
      - charts/user-service/values.yaml
      - charts/user-service/values-{{ .Environment.Name }}.yaml
    set:
      - name: image.tag
        value: "{{ requiredEnv \"USER_SERVICE_IMAGE_TAG\" }}"
    needs:
      - ingress-nginx/ingress-nginx

  - name: ticket-service
    namespace: ticketflow
    chart: ticketflow/ticket-service
    version: "2.1.0"
    values:
      - charts/ticket-service/values.yaml
      - charts/ticket-service/values-{{ .Environment.Name }}.yaml
    set:
      - name: image.tag
        value: "{{ requiredEnv \"TICKET_SERVICE_IMAGE_TAG\" }}"
    needs:
      - user-service

  - name: payment-service
    namespace: ticketflow
    chart: ticketflow/payment-service
    version: "1.8.1"
    values:
      - charts/payment-service/values.yaml
      - charts/payment-service/values-{{ .Environment.Name }}.yaml
    set:
      - name: image.tag
        value: "{{ requiredEnv \"PAYMENT_SERVICE_IMAGE_TAG\" }}"
    needs:
      - user-service
      - ticket-service

  - name: order-service
    namespace: ticketflow
    chart: ticketflow/order-service
    version: "3.0.4"
    values:
      - charts/order-service/values.yaml
      - charts/order-service/values-{{ .Environment.Name }}.yaml
    set:
      - name: image.tag
        value: "{{ requiredEnv \"ORDER_SERVICE_IMAGE_TAG\" }}"
    needs:
      - payment-service

  - name: notification-service
    namespace: ticketflow
    chart: ticketflow/notification-service
    version: "1.2.7"
    values:
      - charts/notification-service/values.yaml
      - charts/notification-service/values-{{ .Environment.Name }}.yaml
    set:
      - name: image.tag
        value: "{{ requiredEnv \"NOTIFICATION_SERVICE_IMAGE_TAG\" }}"
    needs:
      - order-service

  - name: api-gateway
    namespace: ticketflow
    chart: ticketflow/api-gateway
    version: "2.4.0"
    values:
      - charts/api-gateway/values.yaml
      - charts/api-gateway/values-{{ .Environment.Name }}.yaml
    needs:
      - user-service
      - ticket-service
      - payment-service
      - order-service
      - notification-service
# Deploy all releases to staging
helmfile -e staging sync

# Preview diff for all releases before deploying to production
helmfile -e production diff

# Apply only changed releases (helmfile detects drift and skips unchanged releases)
helmfile -e production apply

# Target a specific release within helmfile
helmfile -e production -l name=user-service sync

# Destroy all releases (caution: irreversible in production)
helmfile -e staging destroy

Chart Versioning Strategy

Helm charts follow Semantic Versioning (SemVer). The convention for when to bump each component:

Change Type Version Bump Example
Breaking change: removed/renamed values, changed resource types Major (1.x.x → 2.0.0) Renamed replicaCount to replicas
New optional feature: new values with defaults, new optional resources Minor (1.3.x → 1.4.0) Added optional PodDisruptionBudget support
Backward-compatible fix: template bug, label correction Patch (1.4.1 → 1.4.2) Fixed wrong indentation in NOTES.txt
Application update only (new image, no chart changes) appVersion bump, no chart version change appVersion: 2.8.0 → 2.9.0

In practice, TicketFlow's CI/CD pipeline auto-increments chart patch version on every merge to main, and bumps minor/major versions via PR labels (using helm-chart-testing and chart-releaser-action).

CI/CD Integration

# .github/workflows/deploy-user-service.yml
name: Deploy user-service

on:
  push:
    branches: [main]
    paths:
      - 'services/user-service/**'
      - 'charts/user-service/**'

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Set image tag
        run: echo "IMAGE_TAG=${GITHUB_SHA::8}" >> $GITHUB_ENV

      - name: Build and push image to Nexus
        run: |
          docker login nexus.ticketflow.io:5000 \
            -u ${{ secrets.NEXUS_USER }} \
            -p ${{ secrets.NEXUS_PASSWORD }}
          docker build -t nexus.ticketflow.io:5000/ticketflow/user-service:$IMAGE_TAG \
            services/user-service/
          docker push nexus.ticketflow.io:5000/ticketflow/user-service:$IMAGE_TAG

      - name: Configure kubectl for EKS
        run: |
          aws eks update-kubeconfig \
            --region eu-west-1 \
            --name ticketflow-production

      - name: Helm lint (fail fast on template errors)
        run: |

        run: |

            --namespace ticketflow-staging \
            --create-namespace \

        run: ./scripts/smoke-test-user-service.sh staging

        if: success()
        run: |
          helm upgrade --install user-service ./charts/user-service \
            --namespace ticketflow-prod \
            --create-namespace \
            -f charts/user-service/values-production.yaml \
            --set image.tag=$IMAGE_TAG \
            --atomic \
            --timeout 8m \
            --wait

            -d "{\"text\":\"user-service deploy FAILED for $IMAGE_TAG — rolled back automatically\"}"

Behind the Scenes

What happens when you run helm upgrade --install user-service ./charts/user-service?
  1. Helm reads the chart directory: loads Chart.yaml, values.yaml, all -f override files, and --set flags. It merges values in order: values.yaml < -f files (left to right) < --set flags. Later values override earlier ones.
  2. Helm checks for existing release state in the namespace. It queries Kubernetes Secrets with label owner=helm and name=user-service. If none exist, this is an install; if one exists, this is an upgrade.
  3. Helm renders the templates using the Go text/template engine + Sprig, producing raw Kubernetes YAML manifests. Named templates from _helpers.tpl are resolved. The output is the same as helm template.
  4. If this is an upgrade: Helm performs a three-way strategic merge — current cluster state, previous release manifest, and new release manifest — to compute a minimal diff. Only resources that changed are sent to the Kubernetes API server.
  5. Helm calls the Kubernetes API server to create/update/delete resources. This happens in a deterministic order: Namespaces first, then CRDs, then RBAC resources, then Deployments/Services/etc.
  6. If --wait is set: Helm polls until all Deployments have their desired replica count ready and all Jobs have completed successfully. If --timeout is exceeded, Helm marks the release as FAILED.
  7. If --atomic is set and the release fails or times out: Helm automatically runs a rollback to the previous revision, then marks the release FAILED. Without --atomic, a failed upgrade leaves the cluster in a partially updated state.
  8. Helm serializes the final rendered manifests and the values used into a Kubernetes Secret (base64+gzip encoded, ~10KB typical). This Secret is the release revision record and is what powers helm history and helm rollback.

Note: Helm does not need any server-side component (unlike Helm 2's Tiller). All state is stored in Kubernetes Secrets. Helm itself is a purely client-side tool that talks directly to the Kubernetes API server.

How does helm rollback actually work?

When you run helm rollback user-service 2:

  1. Helm fetches the release Secret for revision 2 from Kubernetes (label version=2).
  2. It decodes and decompresses the stored manifests — these are the exact rendered templates from when revision 2 was deployed.
  3. Helm performs a three-way merge between the current cluster state, the current revision's manifest, and the target revision (2) manifest.
  4. It applies the diff — effectively re-applying revision 2's resources. Kubernetes Deployment controller detects the spec change and performs a rolling update to replace current pods with the revision 2 pods.
  5. Helm creates a new revision record (revision 4 in this case) describing the rollback, pointing back to the configuration of revision 2. The revision history is append-only — rollback does not delete history.

This is why --atomic is so valuable in CI/CD: a failed upgrade triggers an automatic rollback within the same command, without any external orchestration or error-handling logic in your pipeline script.

How does Helm store state? (No Tiller — Helm 3 difference)

Helm 2 required a server-side component called Tiller running in the cluster, which held all release state in its own data store and had cluster-admin privileges — a major security problem.

Helm 3 eliminated Tiller entirely. Release state is now stored as Kubernetes Secrets in the same namespace as the release, with labels like owner=helm, name=<release-name>, version=<revision>. Each Secret contains the compressed manifests of that revision.

This design has several benefits:

  • No special cluster-admin privileges for Helm — it uses your current kubectl context's RBAC permissions.
  • Release state lives with the workload in the same namespace — namespace deletion also deletes release history.
  • Multiple Helm clients can safely manage releases in different namespaces without conflict.
  • Release history is visible via kubectl get secrets -l owner=helm — no special tools needed to inspect state.

Common Mistakes

❌ Mistake 1: Hardcoding Values in Templates

Engineers new to Helm write replicas: 3 directly in deployment.yaml instead of replicas: {{ .Values.replicaCount }}. The template works, but it becomes a maintenance trap: changing replica counts requires editing template files rather than value overrides, breaking the entire point of parameterisation. Worse, hardcoded values make the chart useless for other teams or environments.

Fix: Treat values.yaml as your chart's public API. Every value that could differ between environments or between users of the chart must be expressed as a .Values reference in the template. Even values you think will "always" be the same — resource limits, port numbers, health check paths — should be values. The cost of parameterisation is near-zero; the cost of hardcoding is a future refactor under pressure.

❌ Mistake 2: Deploying Without --atomic in CI/CD

A CI/CD pipeline runs helm upgrade --install user-service ... without --atomic. The new image has a startup bug — pods crash-loop. Helm reports the upgrade as "in progress" and eventually times out, but the cluster is now in a broken state: some pods are on the old image, some on the broken new image. Traffic is split between working and broken instances. The on-call engineer has to manually identify and fix the state.

Fix: Always use --atomic --wait --timeout 5m in production CI/CD pipelines. --atomic guarantees that a failed upgrade is always followed by an automatic rollback, leaving the cluster in the last known-good state. Combined with --wait, Helm will not declare success until all pods are actually healthy.

❌ Mistake 3: Using --set for Secrets in CI/CD

Engineers pass database passwords via --set postgresql.password=supersecret in CI/CD commands. The password is now stored in: shell history, CI/CD logs (if commands are printed), Helm release Secrets in the cluster (the full rendered manifest including the password is stored base64-encoded in the Secret). Multiple exposure vectors for a single misconfiguration.

Fix: Never use --set for secrets. Store secrets in a dedicated system (AWS Secrets Manager, HashiCorp Vault, Kubernetes External Secrets Operator) and reference them in your chart via secretKeyRef. The chart should reference an already-existing Kubernetes Secret by name, not contain the secret value itself. This way, Helm never sees the secret value.

❌ Mistake 4: Not Bumping Chart Version After Template Changes

An engineer fixes a template bug in deployment.yaml but forgets to increment version in Chart.yaml. They package and push the chart to Nexus. The old chart tarball at version 1.4.2 is overwritten (if the repo allows it). Other teams running helm upgrade with the same version constraint get the silent fix — or if using --reuse-values, get confusing mixed behaviour because they think they have the same chart they tested with.

Fix: Treat chart version changes as immutable. Configure your Helm repository (Nexus or otherwise) to reject overwriting existing versions. Make chart version bumping part of your PR checklist. Better: use CI automation (like helm-chart-releaser-action) that auto-bumps chart version on any template change merged to main.

❌ Mistake 5: Ignoring selector label immutability

A chart uses matchLabels: { app: user-service } in the Deployment selector. An engineer later refactors to use the recommended app.kubernetes.io/name label. They update the chart. The upgrade fails with a cryptic error: field is immutable. Kubernetes does not allow Deployment selector labels to be changed after creation because they link the Deployment to its ReplicaSets.

Fix: Selector labels in matchLabels and pod metadata.labels (the intersection) must never change after first deployment. Use the standard labels recommended by the Helm docs from day one. If you must change them, you need to delete the old Deployment and re-create it — which means a brief downtime or a blue-green swap.

Architect Perspective

🏛️ Architect Note

Helm solves the YAML management problem, but it also introduces a new class of architectural concerns that architects must design around:

  • Chart as API contract — values.yaml is the public API between your platform team (who owns the chart) and your application teams (who supply values). Design it deliberately: document every value, provide sensible defaults, use required for values that must be supplied, and version the schema with JSON Schema validation (values.schema.json). A breaking change to a chart is a major version bump, same as any API.
  • The single-chart anti-pattern — some teams create one giant "platform chart" that deploys all services together. This is convenient until it is not: you cannot upgrade one service without potentially affecting others, rollback scope is all-or-nothing, and chart testing complexity explodes. Prefer one chart per deployable service with helmfile for orchestration.
  • Helm + Argo CD = GitOps — Helm alone is imperative: an engineer runs a command. The recommended production pattern is to commit all Helm values into Git and let Argo CD detect and apply changes automatically. The Git repo becomes the single source of truth for cluster state. No human should run helm upgrade against production directly.
  • Library charts for shared patterns — when multiple application charts repeat the same patterns (standard labels, standard probes, standard resource structures), extract those into a library chart (type: library in Chart.yaml). Library charts define named templates but are never deployed directly. Application charts declare the library as a dependency and include its templates. This eliminates cross-chart drift.
  • Drift detection — Helm does not continuously reconcile cluster state. If someone runs kubectl edit deployment directly in production, Helm has no idea. The cluster drifts from the Helm-managed state. Argo CD or Flux solve this by continuously comparing desired (Git/Helm) state to actual (cluster) state and alerting or auto-correcting on drift.

Production Readiness Checklist

  • All environment-varying values are in values.yaml — no hardcoded values in templates
  • values.schema.json exists to validate values against JSON Schema on install/upgrade
  • Secrets are referenced via secretKeyRef or External Secrets Operator — never in values.yaml
  • CI/CD deploys use --atomic --wait --timeout — automatic rollback on failure is guaranteed
  • Chart version is incremented for every template or default change
  • Private charts are stored in Nexus and access is restricted by RBAC
  • helm lint runs in CI on every PR that changes a chart
  • Selector labels use stable app.kubernetes.io/ standard labels — confirmed never changed after initial deploy
  • helm get values and helm history are used for incident investigation — not ad hoc kubectl describe
  • Argo CD or Flux manages production deploys from Git — no manual helm upgrade in production
  • Helmfile (or equivalent) coordinates multi-service deploys with explicit dependency ordering
  • At least 30 days of release history is retained (--history-max 30 in Helm config)
  • Resource requests and limits are set in values — no chart deploys pods with unbounded resources
  • Chart dependencies are pinned to specific versions in Chart.yaml — no floating ranges for production

Interview Questions

These questions cover both the conceptual understanding and hands-on practical knowledge interviewers test for Helm in senior engineering and DevOps roles.

Q1: What is Helm and why is it used instead of plain kubectl apply?

Helm is the Kubernetes package manager. It packages related Kubernetes resources — Deployments, Services, Ingresses, ConfigMaps, HPAs — into a single versioned unit called a chart. You parameterise the chart with values and deploy it as a named release.

The problems with plain kubectl apply that Helm solves:

  • No templating — with kubectl, you need separate YAML files per environment or resort to sed/envsubst hacks to substitute values. Helm provides a proper templating engine with conditionals, loops, and functions.
  • No release history — kubectl has no concept of "this is version 3 of the user-service deployment." Helm tracks every deployment as a revision and allows you to see history, diff revisions, and roll back to any previous revision with one command.
  • No dependency management — if your app needs PostgreSQL, you either manually deploy it or write a custom script. Helm charts declare dependencies that are resolved and deployed automatically.
  • No atomic rollback — with kubectl, a failed rolling update leaves you manually reverting changes across potentially many resources. Helm's --atomic flag automatically rolls back all resources if any part of the upgrade fails.
Q2: Explain the structure of a Helm chart and the role of each key file.

A Helm chart is a directory with a mandatory layout:

  • Chart.yaml — chart metadata: name, version (the chart's own SemVer), appVersion (the application version being packaged), dependencies. The version field is what Helm uses for release management.
  • values.yaml — default configuration values. Think of this as the chart's public API. All configurable parameters should be expressed here with sensible defaults. Users override these per environment with -f override files or --set.
  • templates/ — Kubernetes manifest templates using Go text/template syntax. Helm renders these files by injecting values from .Values, release metadata from .Release, and chart metadata from .Chart.
  • templates/_helpers.tpl — named template definitions (files starting with underscore are not rendered as Kubernetes manifests). Contains reusable template snippets for names, labels, and selectors that are included across other templates.
  • charts/ — unpacked dependency charts downloaded by helm dependency update.
  • templates/NOTES.txt — post-install instructions printed to the user's terminal after a successful install or upgrade.
Q3: How does Helm handle rollbacks, and how does it store release state?

Helm stores release state as Kubernetes Secrets (since Helm 3 — no more Tiller). Each revision of a release is stored as a separate Secret in the release's namespace, labelled with owner=helm, name=<release-name>, and version=<revision-number>. The Secret contains the full rendered manifests and values for that revision, compressed and base64-encoded.

When you run helm rollback user-service 2:

  1. Helm fetches the revision 2 Secret and decodes the stored manifests.
  2. It performs a three-way merge (current cluster state + current revision manifest + target revision manifest) to compute the diff.
  3. It applies the diff to Kubernetes — Deployments detect spec changes and perform rolling updates back to the old version.
  4. A new revision record (e.g., revision 4) is appended to the history describing the rollback. History is always append-only.

Key implication: rollback always creates a new revision. If revision 3 was the latest, after rolling back to revision 2, you have revision 4 which is functionally identical to revision 2. You can see the full audit trail of what happened and when.

Q4: What is the difference between version and appVersion in Chart.yaml?

version is the Helm chart's own semantic version. It describes the chart itself — the templates, the defaults, the dependencies. Helm uses this field for release tracking, repository indexing, and dependency resolution. You bump it when you change the chart.

appVersion describes the version of the application the chart deploys — typically the Docker image tag. It is informational only: Helm itself does not use appVersion for any logic. It appears in helm list output for human reference. By convention, it should match the image.tag default in values.yaml.

They version independently, which is the key point interviewers want to hear:

  • You can release chart 1.4.3 (fixed a template label bug) that still deploys appVersion 2.8.0 — because the fix was chart-only.
  • You can deploy a new appVersion 2.9.0 without changing the chart — by passing --set image.tag=2.9.0 or updating the CI/CD pipeline. Some teams never bump the chart version for application image changes; they always pass the tag at deploy time.
Q5: What is the risk of using --set for secrets in Helm, and what is the correct approach?

Using --set db.password=supersecret has multiple security exposure points:

  • Shell history on the machine that ran the command stores the plain-text password.
  • CI/CD systems often log the full command being run — the password appears in logs accessible to any developer with CI read access.
  • Helm stores the full rendered manifest (including the secret value) in a Kubernetes Secret as a base64-encoded string. Anyone with read access to Secrets in that namespace can decode and read it. Base64 is not encryption.

The correct approaches, in order of maturity:

  1. Reference existing Secrets — the chart should reference a pre-existing Kubernetes Secret by name using secretKeyRef. The Secret is created separately (by the platform team, an operator, or the app's own init), and Helm never sees the value. This is the minimum acceptable production pattern.
  2. External Secrets Operator (ESO) — a Kubernetes operator that syncs secrets from AWS Secrets Manager, HashiCorp Vault, or Azure Key Vault into Kubernetes Secrets automatically. The chart declares an ExternalSecret resource (a custom resource) specifying what to pull from the external store. No human ever sees the secret value in transit.
  3. Vault Agent Sidecar / CSI Provider — injects secrets directly into the pod's filesystem at runtime, without ever creating a Kubernetes Secret. The application reads the secret from a mounted file. Maximum security isolation.
Q6: What is helmfile, and when would you use it over plain Helm?

Helmfile is a declarative, YAML-based tool that manages multiple Helm releases as a unit. Where a Helmfile.yaml describes all the releases that should exist in a cluster (charts, versions, values, namespaces, dependencies between releases), Helm itself only manages one release at a time.

Use helmfile when you have:

  • Multiple services that need to be deployed together in dependency order (e.g., infrastructure charts before application charts).
  • Multi-environment configuration — helmfile's environments block selects different value files per environment cleanly, reducing the amount of scripting needed in CI/CD.
  • Drift detection at scale — helmfile diff shows you all changes across all releases before applying, similar to a terraform plan. This is very hard to achieve with a series of individual helm diff commands.
  • Declarative intent — the helmfile.yaml in Git is the single source of truth for "what should be running in this cluster." This is the foundation for GitOps with Helm (before adopting a full GitOps operator like Argo CD).

For very simple deployments (one or two services), helmfile adds complexity without much benefit. Adopt it when managing three or more releases that have inter-dependencies.

Q7: How do you design a Helm chart for reuse across multiple teams and environments?

Designing a reusable Helm chart is fundamentally an API design problem. The chart is the interface; values.yaml is the contract. Principles for reusable charts:

  • Maximise parameterisation — every value that could differ should be in values.yaml with a sensible default. Use required "error message" for values that have no safe default and must be explicitly provided (e.g., {{ .Values.postgresql.host | required "postgresql.host is required" }}).
  • Use conditional blocks for optional resources — wrap optional resources (Ingress, HPA, ServiceAccount) in {{- if .Values.ingress.enabled }} guards so teams that don't need them don't get them. This one chart can serve both minimal and full deployments.
  • Validate with JSON Schema — create a values.schema.json file that defines the expected types, required fields, and allowed values for your chart's values. Helm validates user-supplied values against this schema before rendering — type errors are caught at install time, not at runtime.
  • Document values thoroughly — every value in values.yaml should have a comment explaining its purpose, valid range, and impact. The helm-docs tool generates documentation from these comments automatically.
  • Provide a library chart for shared helpers — if you manage multiple application charts that share label structures, naming conventions, or resource patterns, extract them into a library chart. Application charts declare it as a dependency and include the shared templates. Changes to shared patterns propagate to all charts on the next dependency update.
  • Semantic versioning with strict immutability — configure your Nexus repo to reject re-pushing an existing chart version. Establish a clear breaking-change policy: major version for any breaking values change, minor for additive changes, patch for fixes. Document this in your platform team's runbook.
