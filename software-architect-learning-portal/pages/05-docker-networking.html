Docker Networking

Understand how containers communicate — bridge networks, host networking, DNS resolution, and service discovery in multi-container applications.

Story

Story

Imagine a large office building. Every floor is a separate department. People on the same floor can walk over and talk to each other freely. People on different floors need to take the elevator and have the right access card. The building's reception — the gateway — handles all visitors from outside.

Docker networking works the same way. Containers on the same network can reach each other by name, like office colleagues. Containers on different networks can't talk unless explicitly connected. The host machine is the building — it decides what traffic comes in from outside and where it goes.

Network Types Overview

Docker ships with several network drivers. Each serves a different purpose. Choosing the wrong one leads to security gaps, broken DNS, or unnecessary complexity. Here is the complete picture:

Default Bridge Network

When you run a container without specifying a network, Docker attaches it to the default bridge network (backed by the docker0 virtual interface on the host). This is fine for quick experiments, but it has a critical limitation: containers cannot reach each other by name.

# Default bridge: containers communicate by IP only
# Start two containers on default bridge
docker run -d --name container-a alpine sleep 3600
docker run -d --name container-b alpine sleep 3600

# Get container-a's IP
docker inspect container-a -f '{{range .NetworkSettings.Networks}}{{.IPAddress}}{{end}}'
# Returns: 172.17.0.2

# container-b can reach container-a by IP — but not by name!
docker exec container-b ping 172.17.0.2    # works
docker exec container-b ping container-a   # FAILS — no DNS on default bridge

# Cleanup
docker stop container-a container-b && docker rm container-a container-b
Common Mistake

The default bridge network does NOT support automatic DNS resolution between containers. You have to use IP addresses, which change every restart. Always create a custom bridge network for multi-container apps. This is the #1 Docker networking mistake beginners make.

Custom Bridge Networks (The Right Way)

A custom bridge network is identical in mechanism to the default bridge, but Docker registers every container's name and aliases with a built-in DNS server. Any container on the same custom network can reach any other container by name — no IP addresses needed, no configuration files to maintain.

# Create a custom bridge network
docker network create ticketflow-backend

# Or with options
docker network create \
  --driver bridge \
  --subnet 172.20.0.0/16 \
  --gateway 172.20.0.1 \
  --label app=ticketflow \
  ticketflow-backend

# Connect containers to custom network
docker run -d \
  --name user-service \
  --network ticketflow-backend \
  ticketflow/user-service:dev

docker run -d \
  --name postgres \
  --network ticketflow-backend \
  postgres:16-alpine

# Now DNS works! user-service can reach postgres by name
docker exec user-service ping postgres         # works!
docker exec user-service curl postgres:5432    # works!

# List networks
docker network ls

# Inspect network (see connected containers and IPs)
docker network inspect ticketflow-backend

# Connect an already-running container to a network
docker network connect ticketflow-frontend user-service

# Disconnect
docker network disconnect ticketflow-backend user-service

# Remove unused networks
docker network prune

Container DNS Resolution

Every container on a custom network has its /etc/resolv.conf automatically configured to point at Docker's embedded DNS resolver. This resolver — always reachable at 127.0.0.11 inside the container — maps container names to their current IPs on that network.

# Docker's embedded DNS server resolves container names
# On a custom network, container "postgres" resolves to its IP

# From inside user-service container:
docker exec user-service cat /etc/resolv.conf
# nameserver 127.0.0.11       ← Docker's embedded DNS
# options ndots:0

# This resolves:
# postgres          → 172.21.0.4  (container on same network)
# redis             → 172.21.0.5
# user-service      → 172.21.0.2  (self)

# Container aliases (same container, multiple names)
docker run -d \
  --name db-primary \
  --network-alias database \
  --network ticketflow-backend \
  postgres:16-alpine

# Now both "db-primary" and "database" resolve to postgres

DNS-based service discovery goes further when you run multiple containers with the same network alias. Docker's embedded DNS returns all their IP addresses in a round-robin rotation — giving you basic load balancing at zero configuration cost. This is how Docker Swarm spreads traffic across replicated services. In production Kubernetes, the equivalent is a ClusterIP Service with multiple backing Pods, but the DNS mechanism is conceptually identical.

Production Tip

Use --network-alias for service roles rather than individual instance names. If you name your database container db-primary and give it the alias database, your application code connects to database:5432. When you later promote a replica or replace the container, only the alias needs to be updated — not a single line of application config.

Port Mapping Deep Dive

Port mapping (-p) creates iptables NAT rules that forward traffic from a host port to a container port. It is the mechanism that makes containers accessible from outside the Docker network — from the host, from other machines on the network, and from the internet (if the host is publicly reachable).

# Syntax: -p [host_ip:]host_port:container_port[/protocol]

# Basic: map host port 8080 to container port 80
docker run -p 8080:80 nginx

# Bind to specific interface only (localhost only — not externally accessible)
docker run -p 127.0.0.1:8080:80 nginx

# Random host port (Docker assigns)
docker run -p 80 nginx
docker port my-container 80   # see what port was assigned

# UDP port
docker run -p 5353:5353/udp myapp

# Multiple ports
docker run -p 80:80 -p 443:443 nginx

# ALL exposed ports (random host ports)
docker run -P nginx
docker port nginx   # see all mappings

Beginner

You only need -p when you want something outside the Docker network to reach the container — your browser, another machine, a load balancer. Two containers on the same custom network talk to each other directly by name without any port publishing. Publishing ports unnecessarily exposes services to the host and potentially the internet.

Host Network

The host network driver removes all network isolation. The container shares the host machine's full network stack — the same interfaces, routing table, and port space. No virtual bridge, no NAT, no iptables rewrites.

# Host network: no isolation, container uses host's network stack
docker run --network host nginx

# nginx listens on port 80 of the HOST directly
# No port mapping needed (or possible)
# curl localhost:80 works immediately

# Check: inside container, network interfaces = host interfaces
docker run --rm --network host alpine ip addr
Architect Note

Host network gives maximum performance — no NAT overhead — and is sometimes needed for apps that require raw socket access or multicast. But it completely removes network isolation. A port conflict kills both the container process and the host service that was already using that port. It is rarely needed outside of specialized networking tools such as packet sniffers, service meshes doing eBPF-based monitoring, or legacy applications that use dynamic port negotiation protocols.

On macOS and Windows, Docker runs inside a lightweight VM. Host networking on those platforms connects to the VM's network stack, not the physical host's — behavior differs significantly from Linux. Never rely on host networking for cross-platform portability.

None Network

The none driver creates a container with no network interfaces except the loopback (lo). The container cannot send or receive any network traffic. It cannot reach the internet, other containers, or the host.

# No network at all
docker run --network none alpine ping 8.8.8.8
# FAIL: Network unreachable

# Only loopback interface available
docker run --rm --network none alpine ip addr
# 1: lo: <LOOPBACK,UP,LOWER_UP>

# Use case: batch processing containers that should never touch network
# Use case: containers that communicate only via shared volumes

The none network is the most secure option when a container genuinely does not need network access — file transformation jobs, data encryption pipelines, or model inference on locally-mounted files. It is also useful in security-sensitive testing environments where you want provable network isolation.

Network Troubleshooting

Most Docker networking problems fall into three categories: the containers are not on the same network, DNS is not resolving, or the target port has nothing listening on it. Work through these commands in order until you find the break.

# Check container's network config
docker inspect my-container --format '{{json .NetworkSettings.Networks}}' | python3 -m json.tool

# Test DNS resolution from inside container
docker exec my-app nslookup postgres
docker exec my-app cat /etc/resolv.conf

# Test connectivity between containers
docker exec my-app ping postgres
docker exec my-app nc -zv postgres 5432   # test TCP connection to port 5432
docker exec my-app wget -qO- http://user-service:3001/health

# Check open ports from inside container
docker exec my-app netstat -tlnp
docker exec my-app ss -tlnp

# Check iptables rules Docker created (on host)
sudo iptables -t nat -L -n -v | grep DOCKER

# Trace network path
docker exec my-app traceroute 8.8.8.8

# Check if containers can reach internet
docker exec my-app ping 8.8.8.8
docker exec my-app curl -s https://ifconfig.me  # what IP does container appear as?
Production Tip

Minimal base images like Alpine do not include ping, curl, or netstat by default. If you're debugging a running container and tools are missing, use docker run --rm --network container:<name> nicolaka/netshoot — this starts a purpose-built network debugging container attached to the same network namespace as your target container. You get a full suite of tools without modifying the production image.

Behind the Scenes

Understanding the kernel-level mechanisms makes networking problems much easier to reason about and debug.

How does Docker's virtual bridge actually work?

When you create a Docker network with the bridge driver, Linux creates a virtual bridge interface — docker0 for the default network, or a named interface like br-a1b2c3d4e5f6 for custom networks. This bridge behaves exactly like a virtual network switch: connected interfaces can send Ethernet frames to each other.

Each container gets a virtual ethernet pair (veth): two virtual interfaces joined at the kernel level. One end lives inside the container's network namespace and appears as eth0. The other end is attached to the bridge on the host and appears as vethXXXXXX in ip link show. Anything sent into one end of the pair comes out the other.

For outbound internet traffic, Docker installs iptables MASQUERADE rules that perform source NAT — rewriting the container's private source IP to the host's public IP before sending packets out. For published ports, DNAT rules rewrite the destination before forwarding inbound packets to the container's private IP.

How does Docker's embedded DNS work?

Each custom network has its own DNS resolver running at the loopback address 127.0.0.11 inside every container on that network. Docker configures each container's /etc/resolv.conf to point at this address at container creation time.

When a container is added to a network, Docker registers the container's name and any configured --network-alias values with that network's resolver. The resolver maps these names to the container's IP address on that network. When a container is removed or its IP changes, Docker immediately updates the DNS record.

When multiple containers share the same alias, the resolver returns all their IP addresses in a rotating round-robin order. The client receives a different IP each time it queries — basic load distribution without any load balancer. For sticky sessions or more sophisticated routing, you need a real load balancer or service mesh in front.

What happens to networking on container restart?

When a container restarts, it gets a brand new network configuration. The container's IP address on each network is reassigned from the subnet pool — it is not guaranteed to get the same IP it had before. Containers that were communicating with the restarted container by IP address will find that IP stale.

This is exactly why hardcoding container IP addresses is dangerous: any restart breaks all dependent services until they are updated or restarted themselves. DNS resolution — using the container name or network alias — always works correctly because Docker updates the DNS record the moment the new container IP is assigned, typically within milliseconds of the container coming online.

In Docker Compose, services always refer to each other by service name, which maps to the network alias Docker Compose creates automatically. This is why Compose applications tolerate container restarts gracefully out of the box.

Common Mistakes

Mistake 1 — Using the default bridge network

The default bridge network has no DNS resolver. Containers must communicate by IP address, which changes on every restart. New developers often spend hours wondering why ping postgres fails while ping 172.17.0.3 works — then the IP changes on the next deploy and everything breaks again. Always create a named custom bridge network and connect your containers to it.

Mistake 2 — Exposing internal services to the host

Mapping a database port as -p 5432:5432 makes Postgres reachable on all host interfaces — including the public IP if your server is internet-facing. Anyone who can reach your server's IP can attempt to connect to your database. Use -p 127.0.0.1:5432:5432 for local-only access, or better yet, don't publish the port at all if your application and database share a Docker network.

Mistake 3 — Forgetting network isolation

Putting all containers on a single network means every container can reach every other container. A compromised frontend service can attempt connections to your database, message queue, or internal admin APIs. Design your networks to match your trust boundaries: a frontend network for public-facing services, a backend network for internal services only, and a data network for databases that only application services can join.

Mistake 4 — ping not available in minimal images

When debugging network issues inside an Alpine-based container, ping will fail with "executable file not found". Alpine does not include iputils by default. Use wget -qO- http://service:port, nc -zv service port, or nslookup service instead — all available in Alpine's base package. For a temporary fix: apk add --no-cache iputils. Never commit this to your Dockerfile for production images.

Mistake 5 — Publishing ports for inter-container communication

Port mapping (-p) is only needed for HOST-to-container or external-to-container traffic. Two containers on the same Docker network reach each other directly by container name without any port publishing. Publishing internal service ports unnecessarily exposes them to the host and adds iptables overhead for every inter-container packet.

Hands-On Lab

Lab — Build an Isolated Network Architecture

Goal: Create a two-network architecture where an application container can reach a database container, but the database cannot reach the internet and cannot be reached from the host. This pattern is the baseline for secure container deployments.

# Create two isolated networks
docker network create --driver bridge tf-frontend
docker network create --driver bridge --internal tf-backend

# Start "database" on backend only
docker run -d \
  --name my-postgres \
  --network tf-backend \
  -e POSTGRES_PASSWORD=secret \
  postgres:16-alpine

# Start "app" on BOTH networks
docker run -d \
  --name my-app \
  --network tf-frontend \
  nginx:alpine

# Connect app to backend network too
docker network connect tf-backend my-app

# Test: app can reach database (same backend network)
docker exec my-app ping my-postgres   # WORKS (if ping available)
docker exec my-app wget -qO- http://my-postgres  # fails (not HTTP) but resolves

# Test: database CANNOT reach internet (internal network)
docker exec my-postgres ping 8.8.8.8  # FAILS — internal network

# Test: database CANNOT be reached from host (no port mapping)
curl localhost:5432  # FAILS

# Inspect what networks each container is on
docker inspect my-postgres --format '{{json .NetworkSettings.Networks}}' | python3 -m json.tool
docker inspect my-app --format '{{json .NetworkSettings.Networks}}' | python3 -m json.tool

# Cleanup
docker stop my-postgres my-app
docker rm my-postgres my-app
docker network rm tf-frontend tf-backend

After completing this lab you should be comfortable creating isolated networks, connecting containers to multiple networks, and verifying that the --internal flag truly prevents outbound traffic. This three-layer architecture — frontend, backend, data — is the foundation of virtually every production container deployment.

Interview Questions

These are the Docker networking questions that appear in DevOps, platform engineering, and senior backend interviews. Click each question to reveal the full answer.

Q1: What is the difference between the default bridge network and a custom bridge network?

The default bridge network (docker0) does not support automatic DNS resolution — containers can only reach each other by IP address, which changes on restart. Custom bridge networks have an embedded DNS server at 127.0.0.11 that resolves container names and aliases. Custom networks also provide better isolation (containers on different networks cannot communicate), support network-level configuration (subnet, gateway), and allow connecting and disconnecting containers at runtime without restarting them.

Always use custom networks in real applications. The default bridge exists for backward compatibility and single-container experiments, not for multi-service deployments.

Q2: How does container DNS resolution work in Docker?

Docker runs an embedded DNS resolver at 127.0.0.11 inside each container that is on a custom network. When a container starts on a custom network, Docker registers its name and any --network-alias values with that network's DNS resolver. Containers query this resolver for other container names, getting back the target container's IP on that shared network.

Round-robin DNS is used when multiple containers share the same alias — each DNS query returns the IPs in a different order, distributing client connections across replicas. The resolver configuration appears in /etc/resolv.conf inside the container as nameserver 127.0.0.11.

The default bridge network does not have this resolver. That is the single most important practical difference between the two network types.

Q3: What does --network host mean and when would you use it?

--network host removes network isolation entirely — the container shares the host's network namespace. There is no NAT, no virtual interfaces, no port mapping. The container binds directly to host ports and sees the same network interfaces as the host process.

Legitimate use cases include: network monitoring tools that need raw socket access or promiscuous mode, performance-critical applications where NAT overhead is measurable and matters (high-frequency trading, real-time data pipelines), and applications that use multicast or other protocols that behave incorrectly through NAT.

Never use host networking in multi-tenant environments — any container with host network can bind any host port, potentially conflicting with or intercepting traffic from other services. On macOS and Windows, host networking connects to the VM's network, not the physical host, so behavior differs from Linux.

Q4: How do you prevent a containerized database from being accessible from outside the host?

Three complementary approaches, most effective when combined:

Option 1 — Don't publish the port at all. If your application and database are on the same Docker custom network, the application reaches the database by container name without any -p flag. No iptables rule is created, so the port is genuinely unreachable from outside.

Option 2 — Bind to localhost only. Use -p 127.0.0.1:5432:5432 instead of -p 5432:5432. Docker creates an iptables rule bound to the loopback interface, so the port is accessible from the host machine but not from other machines on the network or the internet.

Option 3 — Use an internal network. Create the network with --internal. Containers on an internal network cannot make outbound connections and cannot be reached from outside the Docker network. The database cannot even reach the internet, let alone be reached from it.

In production, combine all three: no port publishing, place the database on an --internal network, and let the application bridge between the frontend and backend networks.

Q5: A container can't reach another container by name. How do you debug it?

Systematic diagnosis — work through each step until you find the break:

Step 1: docker network ls — confirm both containers are on the same custom network. If either is on the default bridge, DNS will not work.

Step 2: docker inspect <container-name> — check NetworkSettings.Networks and confirm the shared network appears for both containers with an assigned IP.

Step 3: docker exec container-a cat /etc/resolv.conf — verify nameserver 127.0.0.11 is present. Its absence means the container is on the default bridge, not a custom network.

Step 4: docker exec container-a nslookup container-b — test DNS directly. If this resolves but connection fails, DNS is working and the problem is at the TCP layer.

Step 5: If DNS resolves but connection still fails, verify the target container has a process actually listening on the expected port: docker exec container-b ss -tlnp.

Step 6: docker exec container-a nc -zv container-b 8080 — test TCP reachability to the specific port.

Q6: How does Docker networking differ from production Kubernetes networking?

Docker networking is host-scoped. Custom bridge networks span only the containers on a single host. Overlay networks extend this to multiple hosts in a Docker Swarm, but configuration and management are handled manually.

In Kubernetes, every Pod gets its own IP address from the cluster-wide CIDR, assigned by the Container Network Interface (CNI) plugin. Pods can communicate with any other Pod across any node in the cluster without NAT — a flat network model. There is no concept of separate networks for isolation; that is handled by NetworkPolicy objects.

Kubernetes Services provide stable DNS names and virtual IPs that load-balance across backing Pods, analogous to Docker's round-robin DNS for shared aliases but with significantly more control (session affinity, health-based routing via the endpoints controller, headless services for direct Pod IP discovery).

Understanding Docker's custom bridge networks makes Kubernetes networking easier to learn: Docker DNS by name maps to Kubernetes Service DNS, Docker's internal networks map to Kubernetes NetworkPolicies, and Docker's overlay driver conceptually explains why K8s needs a CNI plugin to build a flat cluster network across hosts.
