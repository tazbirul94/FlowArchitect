Docker Networking

Understand how containers communicate — bridge networks, host networking, DNS resolution, and service discovery in multi-container applications.

Story

Story
  • Containers on the same network reach each other by name — no IPs needed.
  • Containers on different networks can't talk unless explicitly connected.
  • The host decides what external traffic enters and where it goes.

Network Types Overview

Docker ships with several network drivers — pick the wrong one and you get broken DNS, security gaps, or unnecessary complexity.

Network Driver What it does Container sees Internet access Port publish needed Use case

Default Bridge Network

Without specifying a network, Docker attaches containers to the default bridge (docker0). Critical limitation: no DNS — containers can't reach each other by name, only by IP.

# Default bridge: containers communicate by IP only
# Start two containers on default bridge
docker run -d --name container-a alpine sleep 3600
docker run -d --name container-b alpine sleep 3600

# Get container-a's IP
docker inspect container-a -f '{{range .NetworkSettings.Networks}}{{.IPAddress}}{{end}}'
# Returns: 172.17.0.2

# container-b can reach container-a by IP — but not by name!
docker exec container-b ping 172.17.0.2    # works
docker exec container-b ping container-a   # FAILS — no DNS on default bridge

# Cleanup
docker stop container-a container-b && docker rm container-a container-b
Common Mistake

Default bridge has no DNS — you must use IP addresses, which change on every restart. Always use a custom bridge network for multi-container apps. This is the #1 Docker networking mistake.

Custom Bridge Networks (The Right Way)

Custom bridge networks have a built-in DNS server — containers reach each other by name, no IPs needed.

# Create a custom bridge network
docker network create ticketflow-backend

# Or with options
docker network create \
  --driver bridge \
  --subnet 172.20.0.0/16 \
  --gateway 172.20.0.1 \
  --label app=ticketflow \
  ticketflow-backend

# Connect containers to custom network
docker run -d \
  --name user-service \
  --network ticketflow-backend \
  ticketflow/user-service:dev

docker run -d \
  --name postgres \
  --network ticketflow-backend \
  postgres:16-alpine

# Now DNS works! user-service can reach postgres by name
docker exec user-service ping postgres         # works!
docker exec user-service curl postgres:5432    # works!

# List networks
docker network ls

# Inspect network (see connected containers and IPs)
docker network inspect ticketflow-backend

# Connect an already-running container to a network
docker network connect ticketflow-frontend user-service

# Disconnect
docker network disconnect ticketflow-backend user-service

# Remove unused networks
docker network prune

Use separate networks to enforce service boundaries — the API gateway bridges both, backend services stay isolated.

Container DNS Resolution

Every container on a custom network has /etc/resolv.conf pointing at Docker's embedded DNS resolver at 127.0.0.11, which maps container names to IPs.

# Docker's embedded DNS server resolves container names
# On a custom network, container "postgres" resolves to its IP

# From inside user-service container:
docker exec user-service cat /etc/resolv.conf
# nameserver 127.0.0.11       ← Docker's embedded DNS
# options ndots:0

# This resolves:
# postgres          → 172.21.0.4  (container on same network)
# redis             → 172.21.0.5
# user-service      → 172.21.0.2  (self)

# Container aliases (same container, multiple names)
docker run -d \
  --name db-primary \
  --network-alias database \
  --network ticketflow-backend \
  postgres:16-alpine

# Now both "db-primary" and "database" resolve to postgres

Multiple containers sharing the same alias get round-robin DNS — free basic load balancing with zero config.

Production Tip

Use --network-alias for roles, not instance names. App connects to database:5432; swap the underlying container without changing any app config.

Port Mapping Deep Dive

-p creates iptables NAT rules forwarding host ports to container ports — the only way to reach a container from outside Docker.

# Syntax: -p [host_ip:]host_port:container_port[/protocol]

# Basic: map host port 8080 to container port 80
docker run -p 8080:80 nginx

# Bind to specific interface only (localhost only — not externally accessible)
docker run -p 127.0.0.1:8080:80 nginx

# Random host port (Docker assigns)
docker run -p 80 nginx
docker port my-container 80   # see what port was assigned

# UDP port
docker run -p 5353:5353/udp myapp

# Multiple ports
docker run -p 80:80 -p 443:443 nginx

# ALL exposed ports (random host ports)
docker run -P nginx
docker port nginx   # see all mappings

Docker inserts iptables DNAT rules in the DOCKER chain that rewrite the destination to the container's private IP.

Beginner

Only use -p when something outside Docker needs to reach the container. Containers on the same network talk by name with no port publishing needed.

Host Network

host removes all network isolation — container shares the host's interfaces, routing table, and port space directly. No NAT, no virtual bridge.

# Host network: no isolation, container uses host's network stack
docker run --network host nginx

# nginx listens on port 80 of the HOST directly
# No port mapping needed (or possible)
# curl localhost:80 works immediately

# Check: inside container, network interfaces = host interfaces
docker run --rm --network host alpine ip addr
Architect Note
  • Use for: packet sniffers, eBPF monitoring, multicast, raw socket access.
  • Risk: port conflict kills both container and host service using that port.
  • macOS/Windows: connects to VM's network, not physical host — behavior differs from Linux.

None Network

none gives the container only loopback (lo) — no internet, no other containers, no host.

# No network at all
docker run --network none alpine ping 8.8.8.8
# FAIL: Network unreachable

# Only loopback interface available
docker run --rm --network none alpine ip addr
# 1: lo: <LOOPBACK,UP,LOWER_UP>

# Use case: batch processing containers that should never touch network
# Use case: containers that communicate only via shared volumes

Use none for batch jobs, encryption pipelines, or any workload that communicates only via volumes — provable network isolation.

Network Troubleshooting

Most problems fall into three categories: wrong network, DNS not resolving, or nothing listening on the target port.

# Check container's network config
docker inspect my-container --format '{{json .NetworkSettings.Networks}}' | python3 -m json.tool

# Test DNS resolution from inside container
docker exec my-app nslookup postgres
docker exec my-app cat /etc/resolv.conf

# Test connectivity between containers
docker exec my-app ping postgres
docker exec my-app nc -zv postgres 5432   # test TCP connection to port 5432
docker exec my-app wget -qO- http://user-service:3001/health

# Check open ports from inside container
docker exec my-app netstat -tlnp
docker exec my-app ss -tlnp

# Check iptables rules Docker created (on host)
sudo iptables -t nat -L -n -v | grep DOCKER

# Trace network path
docker exec my-app traceroute 8.8.8.8

# Check if containers can reach internet
docker exec my-app ping 8.8.8.8
docker exec my-app curl -s https://ifconfig.me  # what IP does container appear as?
Production Tip

Alpine lacks ping, curl, netstat by default. Debug without modifying the image: docker run --rm --network container:<name> nicolaka/netshoot — attaches a full tool suite to the same network namespace.

Behind the Scenes

How does Docker's virtual bridge actually work?
  1. Bridge driver creates a virtual network switch (docker0 or br-xxxx) on the host.
  2. Each container gets a veth pair: one end appears as eth0 inside the container, the other attaches to the bridge.
  3. Outbound traffic: iptables MASQUERADE rewrites the container's private IP to the host's public IP.
  4. Published ports: DNAT rules rewrite inbound destination to the container's private IP.
How does Docker's embedded DNS work?
  1. DNS resolver runs at 127.0.0.11 inside each container on a custom network.
  2. When container joins a network, Docker registers its name and aliases with that resolver.
  3. IP changes (on restart) update the DNS record immediately — within milliseconds.
  4. Multiple containers with the same alias → round-robin DNS responses for basic load distribution.
What happens to networking on container restart?
  • Container gets a new IP from the subnet pool — not guaranteed to be the same.
  • Anything connecting by hardcoded IP breaks until restarted.
  • DNS (name/alias) always works — record updates within milliseconds of the container coming online.
  • Docker Compose uses service names as aliases automatically — restarts are transparent.

Common Mistakes

Mistake 1 — Using the default bridge network

Default bridge has no DNS — containers communicate by IP only, which changes on every restart. Always create a named custom bridge network.

Mistake 2 — Exposing internal services to the host

-p 5432:5432 binds to all host interfaces, including public IPs. Use -p 127.0.0.1:5432:5432 for local-only, or skip publishing entirely if the app and DB share a network.

Mistake 3 — Forgetting network isolation

All containers on one network = every container can reach every other. Use separate networks per trust boundary: frontend, backend, data.

Mistake 4 — ping not available in minimal images

Alpine excludes iputils. Use wget -qO- http://service:port, nc -zv service port, or nslookup service instead — all in Alpine's base.

Mistake 5 — Publishing ports for inter-container communication

Containers on the same network reach each other by name with no -p needed. Publishing internal ports exposes them to the host and adds iptables overhead.

Hands-On Lab

Lab — Build an Isolated Network Architecture

Goal: Create a two-network architecture where an application container can reach a database container, but the database cannot reach the internet and cannot be reached from the host. This pattern is the baseline for secure container deployments.

# Create two isolated networks
docker network create --driver bridge tf-frontend
docker network create --driver bridge --internal tf-backend

# Start "database" on backend only
docker run -d \
  --name my-postgres \
  --network tf-backend \
  -e POSTGRES_PASSWORD=secret \
  postgres:16-alpine

# Start "app" on BOTH networks
docker run -d \
  --name my-app \
  --network tf-frontend \
  nginx:alpine

# Connect app to backend network too
docker network connect tf-backend my-app

# Test: app can reach database (same backend network)
docker exec my-app ping my-postgres   # WORKS (if ping available)
docker exec my-app wget -qO- http://my-postgres  # fails (not HTTP) but resolves

# Test: database CANNOT reach internet (internal network)
docker exec my-postgres ping 8.8.8.8  # FAILS — internal network

# Test: database CANNOT be reached from host (no port mapping)
curl localhost:5432  # FAILS

# Inspect what networks each container is on
docker inspect my-postgres --format '{{json .NetworkSettings.Networks}}' | python3 -m json.tool
docker inspect my-app --format '{{json .NetworkSettings.Networks}}' | python3 -m json.tool

# Cleanup
docker stop my-postgres my-app
docker rm my-postgres my-app
docker network rm tf-frontend tf-backend

After this lab: you can create isolated networks, connect containers to multiple networks, and verify --internal blocks outbound traffic. This frontend/backend/data pattern is the foundation of every production container deployment.

Interview Questions

Q1: What is the difference between the default bridge network and a custom bridge network?

Default bridge has no DNS — containers use IPs only, which change on restart. Custom bridge has embedded DNS at 127.0.0.11.

  • Custom: containers reach each other by name.
  • Custom: supports subnet/gateway config and runtime connect/disconnect.
  • Default: backward compatibility only — never use for multi-service apps.
Q2: How does container DNS resolution work in Docker?

Docker runs an embedded DNS resolver at 127.0.0.11 inside each container on a custom network.

  • Container name and aliases registered on join; updated immediately on restart.
  • Multiple containers with the same alias → round-robin DNS for basic load distribution.
  • Visible in container: cat /etc/resolv.conf → nameserver 127.0.0.11.
  • Default bridge has no resolver — this is the key practical difference.
Q3: What does --network host mean and when would you use it?

Container shares the host's full network namespace — no NAT, no virtual interfaces, no port mapping.

  • Use for: packet sniffers, eBPF monitors, multicast, high-frequency trading (no NAT overhead).
  • Risk: any port conflict kills both container and the host service.
  • macOS/Windows: connects to VM network, not physical host — behavior differs.
Q4: How do you prevent a containerized database from being accessible from outside the host?

Three approaches — use all three in production:

  • No port publish: app and DB on the same custom network — no -p flag, no iptables rule created.
  • Localhost bind: -p 127.0.0.1:5432:5432 — reachable from host only, not external IPs.
  • Internal network: docker network create --internal — no outbound or inbound internet at all.
Q5: A container can't reach another container by name. How do you debug it?
  1. docker network ls — confirm both containers share a custom network.
  2. docker inspect <name> → check NetworkSettings.Networks for shared network + assigned IP.
  3. docker exec container-a cat /etc/resolv.conf — must show nameserver 127.0.0.11.
  4. docker exec container-a nslookup container-b — DNS resolves? If yes, problem is at TCP layer.
  5. docker exec container-b ss -tlnp — verify something is actually listening on the port.
  6. docker exec container-a nc -zv container-b 8080 — test TCP reachability.
Q6: How does Docker networking differ from production Kubernetes networking?

Docker networking is host-scoped; Kubernetes uses a flat cluster-wide network model.

  • K8s: every Pod gets its own IP (CNI plugin) — Pods talk to any Pod on any node without NAT.
  • K8s: isolation via NetworkPolicy objects, not separate networks.
  • K8s Services ≈ Docker round-robin DNS aliases but with health-based routing, session affinity, headless mode.
  • Docker DNS by name → K8s Service DNS; Docker --internal → K8s NetworkPolicy.
