Kubernetes Workloads

Master every workload type Kubernetes offers — Deployment, ReplicaSet, StatefulSet, DaemonSet, Job, and CronJob — and know exactly when to use each one.

The Story Behind Kubernetes Workloads

The Army Analogy

Imagine you are a general commanding an army. You do not personally place every soldier on the battlefield and order them individually. That would not scale — and if a soldier fell in battle, you would need to notice and send a replacement yourself. That is unmanageable at scale.

Instead, you issue orders at a higher level: "I need 10 infantry units covering the northern flank at all times." Your officers continuously ensure that 10 units are present. One gets taken out? A replacement is dispatched. You want 20? The order is updated. You never think about individual soldiers — only about the desired state of the battlefield.

Kubernetes Workload resources work exactly like this. You describe the desired state — "3 replicas of ticket-service running" — and Kubernetes continuously reconciles reality to match that declaration. Pod crashes, node failures, rolling updates — all handled automatically by the control plane.

But not every workload behaves the same way. A stateless web service is very different from a database, which is different from a log collector, which is different from a batch job. Kubernetes provides distinct workload types — each designed for a specific operational pattern.

What Are Workload Resources?

Beginner Analogy

Think of a Pod as a single worker at a company. Workload resources are the HR policies that govern those workers:

  • Deployment — Like a staffing agency contract: "We always need exactly 3 developers on this project. If one quits, hire a replacement. If we need more, scale up." Perfect for stateless services.
  • StatefulSet — Like a specialized contractor agreement: "We need a DBA named db-0, and a replica named db-1. They have assigned desks (persistent storage), unique logins (stable DNS names), and are hired and fired in order." For databases and clustered apps.
  • DaemonSet — Like a security guard roster: "Every office floor must have exactly one guard on duty." As the building grows new floors, guards are automatically assigned. Perfect for cluster-wide infrastructure agents.
  • Job — Like a freelance project: "Hire a contractor to migrate this database. Once the work is done, no further employment needed."
  • CronJob — Like a recurring maintenance contract: "Every Monday at 2 AM, hire a cleaner. When the job is done, they leave."

Technical Deep Dive

The Kubernetes Controller Pattern

All workload resources operate via the controller pattern. Each workload type has a dedicated controller loop running in the control plane that continuously:

  1. Observes — reads the current state of Pods in the cluster
  2. Diffs — compares current state to the desired state declared in the spec
  3. Acts — creates, updates, or deletes Pods to reconcile the difference

This declarative, reconciliation-based approach means Kubernetes is self-healing by design. You never imperatively say "start a pod" — you declare intent and the system makes it so.

ReplicaSet — The Foundation (That You Don't Use Directly)

A ReplicaSet ensures a specified number of Pod replicas are running at any given time. It uses a label selector to identify the Pods it owns. If a Pod is deleted, the ReplicaSet controller immediately creates a replacement.

However, a raw ReplicaSet has a critical limitation: it has no concept of updates. If you change the Pod template, existing Pods are not replaced — the change only applies to new Pods. This makes it useless for deployments in practice. You almost never use ReplicaSet directly — instead, you use a Deployment, which manages ReplicaSets on your behalf and adds rolling update and rollback capabilities.

When you create a Deployment, Kubernetes automatically creates a ReplicaSet beneath it. When you perform a rolling update, the Deployment creates a new ReplicaSet with the updated template, scales it up, and scales down the old one — keeping the old ReplicaSet around (but at zero replicas) to enable instant rollback.

Deployment — For Stateless Services

A Deployment is the primary workload for stateless applications. It wraps a ReplicaSet and adds:

  • Rolling updates — controlled by maxSurge (how many extra Pods to create during update) and maxUnavailable (how many Pods can be down during update). Defaults to 25% each, ensuring zero downtime.
  • Rollback — Kubernetes keeps a configurable history of old ReplicaSets. kubectl rollout undo deployment/ticket-service instantly reverts to the previous version.
  • Scaling — manual (kubectl scale) or automatic via HPA (Horizontal Pod Autoscaler).
  • Pause/Resume — you can pause a rollout mid-way to observe before continuing.

Stateless means: any Pod in the Deployment is interchangeable. The app stores no session state locally — state lives in a database or cache. HTTP requests can be routed to any Pod. All of TicketFlow's application-tier services (user-service, ticket-service, payment-service, api-gateway) run as Deployments.

StatefulSet — For Stateful Applications

A StatefulSet manages Pods that need stable, persistent identity. Unlike Deployment Pods (which get random names like ticket-service-7d9f8b-xkp2v), StatefulSet Pods get stable, predictable names: postgres-0, postgres-1, postgres-2.

Three guarantees StatefulSet provides:

  • Stable network identity — each Pod gets a stable DNS name: <pod-name>.<service-name>.<namespace>.svc.cluster.local. postgres-0.postgres-headless.default.svc.cluster.local always resolves to the same Pod, regardless of restarts. Cluster members can find each other reliably.
  • Stable persistent storage — each Pod gets its own PersistentVolumeClaim via volumeClaimTemplates. postgres-0 always mounts data-postgres-0. Even if the Pod is rescheduled to a different node, it reconnects to the same volume.
  • Ordered deployment and scaling — Pods are created, updated, and deleted in order (0, 1, 2...). Pod N is not started until Pod N-1 is Running and Ready. This satisfies database leader-election requirements where the primary must be running before replicas join.

Use StatefulSet for: PostgreSQL, MySQL, MongoDB, Cassandra, Redis Cluster, Kafka, Zookeeper, Elasticsearch — any system that maintains cluster membership by identity.

DaemonSet — One Pod Per Node

A DaemonSet ensures that exactly one Pod runs on every node in the cluster (or a subset of nodes via node selectors). As nodes are added to the cluster, Pods are automatically created on them. As nodes are removed, Pods are garbage-collected.

DaemonSet is the workload for cluster-level infrastructure agents — services that need to run everywhere, not specific application replicas. Common use cases:

  • Log collection — Fluentd, Fluent Bit, Filebeat: reads log files from /var/log/containers/ on every node and ships them to a central logging backend (Loki, Elasticsearch, Splunk)
  • Metrics collection — Prometheus Node Exporter: collects host-level metrics (CPU, memory, disk, network) per node
  • Networking — CNI plugin agents (Calico, Cilium, Flannel): must run on every node to program the data plane
  • Security — Falco (runtime threat detection), Sysdig agents: must observe every node's kernel events
  • Storage — CSI node driver pods: must run on every node to mount volumes

Job — Run-to-Completion Workloads

A Job creates one or more Pods and retries until a specified number of completions succeed. Unlike Deployments, Jobs are not kept running — they terminate once their task is done.

Key parameters:

  • completions — total number of successful Pod completions needed (default: 1)
  • parallelism — how many Pods to run simultaneously (default: 1)
  • backoffLimit — how many times to retry before marking the Job failed (default: 6)
  • activeDeadlineSeconds — hard time limit on the whole Job

Use Jobs for: database migrations before a deployment, one-time data transformations, batch report generation, index rebuilds, sending bulk notifications.

CronJob — Scheduled Jobs

A CronJob creates a Job on a schedule defined in standard cron syntax. It manages Job creation, history retention, and concurrency policy.

Key parameters:

  • schedule — standard cron format: "0 2 * * *" = 2 AM daily
  • concurrencyPolicy — Allow (default), Forbid (skip if previous job still running), or Replace (kill previous job and start new)
  • successfulJobsHistoryLimit — how many completed Jobs to retain (default: 3)
  • failedJobsHistoryLimit — how many failed Jobs to retain (default: 1)
  • startingDeadlineSeconds — how late a job can start before being considered missed

Use CronJobs for: nightly database backups, hourly cache invalidation, daily report generation, weekly cleanup tasks.

Enterprise Scenario: TicketFlow on EKS

TicketFlow Kubernetes Architecture

TicketFlow runs on AWS EKS in the ticketflow-prod cluster. Here is how each Kubernetes workload type maps to a real service in the platform:

Deployments (stateless application tier):

  • user-service — Node.js, 3 replicas, rolling update strategy. Handles authentication, JWT issuance, and user profiles. Any replica can serve any request — no local state.
  • ticket-service — Spring Boot, 3 replicas, rolling update. Core ticket CRUD. Reads/writes PostgreSQL. Redis for caching ticket lookup results.
  • payment-service — Python (FastAPI), 2 replicas. Integrates with Stripe. All transaction state in PostgreSQL. Pods are ephemeral and interchangeable.
  • order-service — Go, 4 replicas. High-throughput order ingestion. Stateless, scales horizontally for concert peak loads.
  • notification-service — Node.js, 2 replicas. Consumes Kafka events, sends emails/SMS. Idempotent — safe to run multiple replicas.
  • api-gateway — NGINX, 2 replicas. Routes traffic to backend services. Config stored in ConfigMap, not in the Pod.

StatefulSets (stateful data tier):

  • postgres — 1 primary + 2 read replicas. postgres-0 is the primary via stable DNS. postgres-1 and postgres-2 stream from it. Each has a dedicated 100 GB EBS volume via PVC.
  • redis — 3-node Redis Cluster. Nodes identify each other by stable headless service DNS. Cluster state stored in per-Pod PVCs.
  • kafka — 3-broker Kafka cluster. Each broker has a stable identity (kafka-0, kafka-1, kafka-2) required for partition leadership and inter-broker replication.

DaemonSets (cluster infrastructure):

  • fluentd — Runs on every EKS worker node. Tails /var/log/containers/*.log and ships structured logs to Loki. Without a DaemonSet, nodes added by the Cluster Autoscaler would be dark (no log collection).
  • prometheus-node-exporter — Runs on every node. Exposes CPU, memory, disk, and network metrics for Grafana dashboards and alerting rules.

Jobs (one-time operations):

  • ticket-service-migration — A Job that runs Flyway to apply PostgreSQL schema migrations. Triggered by the CI/CD pipeline before each Deployment rollout. The Job must complete successfully before the new Deployment replicas start accepting traffic.
  • data-backfill-v2 — A one-time Job to populate a new ticket_search_index table from 3 years of historical ticket data. Ran with parallelism=4 to process 12M rows in under 2 hours.

CronJobs (scheduled operations):

  • postgres-backup — Runs at 02:00 UTC daily. Uses pg_dump to create a compressed backup and uploads to S3. concurrencyPolicy: Forbid prevents overlapping backup runs.
  • cache-warmer — Runs every hour at :05. Pre-loads the top-1000 event pages into Redis to prevent cache-miss storms after TTL expiry.
  • expired-session-cleanup — Runs at 03:30 UTC daily. Deletes JWT refresh tokens older than 30 days from the PostgreSQL sessions table.

Code Examples

Deployment: ticket-service with 3 Replicas

apiVersion: apps/v1
kind: Deployment
metadata:
  name: ticket-service
  namespace: ticketflow-prod
  labels:
    app: ticket-service
    tier: backend
    version: v2.5.1
spec:
  replicas: 3
  # Keep last 5 ReplicaSets for rollback history
  revisionHistoryLimit: 5
  selector:
    matchLabels:
      app: ticket-service
  strategy:
    type: RollingUpdate
    rollingUpdate:
      # Allow 1 extra Pod to be created during the update
      maxSurge: 1
      # Never allow more than 1 Pod to be unavailable
      maxUnavailable: 0
  template:
    metadata:
      labels:
        app: ticket-service
        version: v2.5.1
    spec:
      # Spread Pods across availability zones for resilience
      topologySpreadConstraints:
        - maxSkew: 1
          topologyKey: topology.kubernetes.io/zone
          whenUnsatisfiable: DoNotSchedule
          labelSelector:
            matchLabels:
              app: ticket-service
      containers:
        - name: ticket-service
          image: nexus.ticketflow.io:5000/ticket-service:v2.5.1
          ports:
            - containerPort: 8080
          env:
            - name: SPRING_PROFILES_ACTIVE
              value: production
            - name: DB_HOST
              value: postgres-0.postgres-headless.ticketflow-prod.svc.cluster.local
            - name: REDIS_URL
              value: redis-0.redis-headless.ticketflow-prod.svc.cluster.local:6379
          resources:
            requests:
              cpu: "250m"
              memory: "512Mi"
            limits:
              cpu: "1000m"
              memory: "1Gi"
          # Liveness: restart the container if it hangs
          livenessProbe:
            httpGet:
              path: /actuator/health/liveness
              port: 8080
            initialDelaySeconds: 30
            periodSeconds: 10
            failureThreshold: 3
          # Readiness: only send traffic when the app is fully started
          readinessProbe:
            httpGet:
              path: /actuator/health/readiness
              port: 8080
            initialDelaySeconds: 15
            periodSeconds: 5
            failureThreshold: 3

Rolling Update and Rollback Commands

# Trigger a rolling update by updating the image tag
kubectl set image deployment/ticket-service \
  ticket-service=nexus.ticketflow.io:5000/ticket-service:v2.5.2 \
  -n ticketflow-prod

# Watch the rollout progress in real time
kubectl rollout status deployment/ticket-service -n ticketflow-prod

# Inspect rollout history — shows all ReplicaSets kept as rollback targets
kubectl rollout history deployment/ticket-service -n ticketflow-prod

# View the details of a specific revision
kubectl rollout history deployment/ticket-service --revision=3 -n ticketflow-prod

# Rollback to the previous version instantly (uses old ReplicaSet — no rebuild)
kubectl rollout undo deployment/ticket-service -n ticketflow-prod

# Rollback to a specific revision
kubectl rollout undo deployment/ticket-service --to-revision=2 -n ticketflow-prod

# Pause a rollout mid-way to observe behaviour before continuing
kubectl rollout pause deployment/ticket-service -n ticketflow-prod
kubectl rollout resume deployment/ticket-service -n ticketflow-prod

# Scale manually (Deployment manages the ReplicaSet for you)
kubectl scale deployment/ticket-service --replicas=6 -n ticketflow-prod

kind: StatefulSet
metadata:
  name: postgres
  namespace: ticketflow-prod
spec:
  serviceName: "postgres-headless"   # Headless service provides stable DNS per Pod
  replicas: 3
  selector:
    matchLabels:

              valueFrom:
                secretKeyRef:
                  name: postgres-secret
                  key: username
            - name: POSTGRES_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: postgres-secret
                  key: password
          volumeMounts:

              mountPath: /var/lib/postgresql/data

            requests:
              cpu: "500m"
              memory: "1Gi"

              cpu: "2000m"

CronJob: Nightly PostgreSQL Backup

apiVersion: batch/v1
kind: CronJob
metadata:
  name: postgres-backup
  namespace: ticketflow-prod
spec:
  # Every day at 02:00 UTC
  schedule: "0 2 * * *"
  # Do not start a new Job if the previous one is still running
  concurrencyPolicy: Forbid
  # How late the job can start before being considered missed (5 minutes)
  startingDeadlineSeconds: 300
  # Keep last 3 successful and 2 failed job records for debugging
  successfulJobsHistoryLimit: 3
  failedJobsHistoryLimit: 2
  jobTemplate:
    spec:
      backoffLimit: 2
      activeDeadlineSeconds: 3600    # 1 hour max per backup run
      template:
        spec:
          restartPolicy: OnFailure
          containers:
            - name: pg-backup
              image: nexus.ticketflow.io:5000/pg-backup:15.4
              env:
                - name: PGHOST
                  value: postgres-0.postgres-headless.ticketflow-prod.svc.cluster.local
                - name: PGUSER
                  valueFrom:
                    secretKeyRef:
                      name: postgres-secret
                      key: username
                - name: PGPASSWORD
                  valueFrom:
                    secretKeyRef:
                      name: postgres-secret
                      key: password
                - name: S3_BUCKET
                  value: s3://ticketflow-backups/postgres
                - name: AWS_REGION
                  value: eu-west-1
              command:
                - /bin/sh
                - -c
                - |
                  TIMESTAMP=$(date +%Y%m%d_%H%M%S)
                  pg_dump -Fc ticketflow | aws s3 cp - \
                    ${S3_BUCKET}/ticketflow_${TIMESTAMP}.dump \
                    --sse AES256
                  echo "Backup completed: ticketflow_${TIMESTAMP}.dump"
              resources:
                requests:
                  cpu: "200m"
                  memory: "256Mi"
                limits:
                  cpu: "500m"
                  memory: "512Mi"

Behind the Scenes

What exactly happens during a Deployment rolling update?

When you update a Deployment's Pod template (e.g., change the image tag), the Deployment controller executes a precise sequence:

  1. New ReplicaSet created — The controller creates a new ReplicaSet with the updated Pod spec. It does NOT modify the old ReplicaSet.
  2. Scale up new ReplicaSet — New Pods are created up to the configured maxSurge above the desired replica count. With 3 replicas and maxSurge: 1, 4 Pods exist briefly.
  3. Readiness gate — New Pods must pass their readiness probe before the old Pods are terminated. This is what makes rolling updates safe — traffic is never routed to a Pod that is not ready.
  4. Scale down old ReplicaSet — Once new Pods are ready, old Pods are terminated respecting maxUnavailable. With maxUnavailable: 0, no old Pod is removed until a new one is ready — guaranteeing capacity is never reduced below 3.
  5. Repeat — Steps 2–4 repeat until the old ReplicaSet is at zero replicas.
  6. Old ReplicaSet retained — The old ReplicaSet is set to 0 replicas but kept in etcd. This is what enables instant rollback — kubectl rollout undo simply scales the old RS back up and the new RS back down. No Pod rebuild required.

The entire process is observable via kubectl rollout status. The Deployment controller writes events you can see with kubectl describe deployment. The ReplicaSet history is visible with kubectl get rs.

How does StatefulSet stable DNS actually work?

A StatefulSet requires a headless Service (a Service with clusterIP: None). A headless Service does not get a cluster IP — instead, Kubernetes DNS creates individual A records for each Pod.

For a StatefulSet named postgres in namespace ticketflow-prod with a headless service named postgres-headless:

  • postgres-0.postgres-headless.ticketflow-prod.svc.cluster.local → Pod postgres-0's IP
  • postgres-1.postgres-headless.ticketflow-prod.svc.cluster.local → Pod postgres-1's IP
  • postgres-2.postgres-headless.ticketflow-prod.svc.cluster.local → Pod postgres-2's IP

When postgres-1 is restarted and rescheduled to a different node, Kubernetes updates the DNS record to point to the new IP. Applications using the DNS name always reach the correct Pod. This is how PostgreSQL replicas know how to stream from postgres-0 — they use the stable DNS name, not a fragile IP address.

The headless service also has a regular A record (postgres-headless.ticketflow-prod.svc.cluster.local) that returns all Pod IPs — useful for clients that implement their own connection routing (like Cassandra or Elasticsearch drivers).

How does DaemonSet scheduling interact with node taints?

By default, a DaemonSet schedules Pods on all nodes that are schedulable. Nodes can have taints that repel Pods unless the Pod has a matching toleration.

EKS control plane nodes have the taint node-role.kubernetes.io/control-plane:NoSchedule, which prevents ordinary application Pods from running there. However, a Fluentd DaemonSet deployed to collect all logs should also run on control plane nodes to capture API server and controller logs.

By adding the matching toleration to the DaemonSet's Pod template, Fluentd will be scheduled even on tainted nodes. Without the toleration, control plane nodes would be invisible to your logging infrastructure — a common gap in observability setups.

DaemonSets also respect node selectors and node affinity rules, allowing you to target a DaemonSet to a specific node pool (e.g., only run the GPU driver agent on nodes with GPU labels).

What is the difference between Job restartPolicy OnFailure vs Never?

Job Pods cannot use restartPolicy: Always (that is reserved for Deployments). They have two options:

  • OnFailure — If the container exits with a non-zero code, Kubernetes restarts it on the same Pod. The Pod stays, the container restarts. Use this when your job is idempotent and can safely retry from wherever it failed — but you want to keep the Pod around for log inspection between retries.
  • Never — If the container fails, the Pod is marked as Failed and a new Pod is created for the retry. This is useful for jobs that are not idempotent, or when you want each attempt to start completely fresh (clean filesystem, no leftover temp files). Also useful when debugging — all failed Pod logs are retained separately.

For TicketFlow's Flyway migration job, OnFailure is used because Flyway's migrations are idempotent (already-applied migrations are skipped). For a one-time data backfill that writes to a temp table, Never is safer because each fresh Pod starts with no partial state.

Common Mistakes

Mistake 1: Using a StatefulSet for a Stateless Service

A developer sees that their payment-service needs to write temporary files during processing. They use a StatefulSet to give each Pod its own PVC, thinking this is "safer." The result: Pods now start in order and roll out one at a time. A 3-Pod update that would take 2 minutes with a Deployment now takes 6 minutes because each Pod waits for the previous to be fully ready. Rollbacks are painfully slow. HPA cannot scale quickly during traffic spikes.

Fix: Stateless services should write temporary files to an emptyDir volume (ephemeral, local to the node, does not require ordering). Use Deployment. Reserve StatefulSet strictly for databases, caches, and message brokers that need stable identity across restarts.

Mistake 2: Deleting StatefulSet Pods to "Speed Up" a Rolling Update

A DBA sees that the StatefulSet rolling update is waiting for postgres-2 to become ready before proceeding to postgres-1. Impatient, they delete the Pod manually to "force" the update. This bypasses the ordered rollout guarantee. postgres-1 may now start before postgres-2's replica stream is established, causing data inconsistency and potentially a split-brain scenario in the PostgreSQL replica set.

Fix: Trust the StatefulSet controller. Ordered rollouts are there for a reason — each database node must confirm it has caught up before the next is updated. If an update is genuinely stuck, investigate why the Pod is not becoming Ready (check kubectl describe pod events and probe failures) rather than forcibly deleting it.

Mistake 3: Not Setting a readinessProbe on a Deployment

A Spring Boot service takes 25 seconds to start up, run Flyway migrations on startup, and warm its connection pool. Without a readiness probe, Kubernetes considers the Pod ready the moment its container starts — which is immediately. The Service starts routing traffic to the Pod while Spring Boot is still in the middle of initialization. Users get 503 Service Unavailable errors during every deployment.

Fix: Always define both a readinessProbe (controls when the Pod receives traffic) and a livenessProbe (controls when the Pod is restarted). For Spring Boot, use the Actuator health endpoints: /actuator/health/readiness and /actuator/health/liveness. Set initialDelaySeconds generously — it is better to wait a little longer than to serve errors.

Mistake 4: Using restartPolicy: Always in a Job

An engineer writes a Job to migrate data and sets restartPolicy: Always (copying it from a Deployment template). Kubernetes rejects the manifest with a validation error — Jobs do not support Always. This blocks the CI/CD pipeline at 11 PM before a major release.

Fix: Jobs must use restartPolicy: OnFailure or restartPolicy: Never. Always is only valid for Deployments, StatefulSets, and DaemonSets (always-running workloads). Validate your YAML with kubectl apply --dry-run=client -f migration-job.yaml before adding it to the CI pipeline.

Mistake 5: CronJob Creating Duplicate Jobs Due to Missing concurrencyPolicy

TicketFlow's nightly PostgreSQL backup CronJob normally takes 20 minutes. One night, a large event sale causes the database to grow and the backup takes 80 minutes. The CronJob fires again at 02:00 the following day while the first backup is still running. Two pg_dump processes compete for I/O on the same database, both slow to a crawl, both eventually exceed their activeDeadlineSeconds, and both fail. The team wakes up to zero successful backups.

Fix: Set concurrencyPolicy: Forbid for any CronJob that must not overlap (backups, reports, migrations). Set an appropriate startingDeadlineSeconds so that a missed run is logged rather than silently skipped. Alert on CronJob failure using a Prometheus rule on the kube_cronjob_status_last_unsuccessful_time metric.

Architect Perspective

Architect Note

Choosing the right workload type is not a trivial decision — it has deep implications for operational complexity, upgrade risk, and blast radius during incidents. Here is the mental model I use at TicketFlow scale:

Default to Deployment. If your service is stateless — meaning any Pod instance is interchangeable and no data survives Pod deletion — use a Deployment. This covers the vast majority of application-tier services. Deployments give you the safest, most controllable rollout mechanism in Kubernetes with minimal operational burden.

StatefulSet is not "Deployment with persistence." StatefulSet is a contract with the cluster: "I need exactly these named Pods to exist, with these exact volumes, in this exact order." The operational cost is significantly higher. Each update must be tested carefully. Volume expansion requires manual steps. Backup strategy is more complex. Only accept this cost when the application genuinely requires it — typically, only data-tier services.

DaemonSets are infrastructure primitives, not application workloads. If you find yourself designing an application-tier service that "needs to run on every node," challenge that assumption. The more likely answer is that you need a Deployment with a Service, and the load balancer handles distribution. True DaemonSet use cases are almost exclusively cluster infrastructure: log collection, metrics export, network data plane, and security monitoring.

Jobs and CronJobs are deployment citizens, not afterthoughts. Database migrations as Jobs are a superior pattern to running migrations inside application startup code. Startup-based migrations cause cascading failures when multiple Pod replicas start simultaneously and all try to run the same migration. A Job runs once, completes, and the Deployment waits for it (or your CI/CD pipeline gates the Deployment rollout on Job completion). This is the pattern used at TicketFlow and it eliminates an entire class of deployment race conditions.

The ReplicaSet is the implementation detail, not the interface. Understanding that Deployments manage ReplicaSets is essential for debugging rollouts and understanding rollback mechanics — but you should never need to create or modify a ReplicaSet directly. If you are doing so, you are operating at the wrong level of abstraction.

Production Readiness Checklist

  • Every Deployment has both readinessProbe and livenessProbe configured with appropriate initialDelaySeconds
  • Every container has resources.requests and resources.limits defined for CPU and memory
  • Deployment revisionHistoryLimit is set (5 is a safe default — enough for rollback without bloating etcd)
  • Rolling update strategy uses maxUnavailable: 0 for zero-downtime deployments on critical services
  • StatefulSets have a headless Service defined with a matching serviceName
  • StatefulSet volumeClaimTemplates uses a StorageClass with appropriate IOPS for the workload (gp3 for PostgreSQL, io2 for Kafka)
  • DaemonSets have tolerations for any tainted nodes that must receive the daemon (control plane, spot, GPU nodes)
  • DaemonSets have resource limits to prevent a runaway logging agent from starving application Pods
  • Jobs set activeDeadlineSeconds and backoffLimit to prevent infinite retries and stuck jobs
  • Jobs use ttlSecondsAfterFinished to auto-clean completed Job objects from etcd
  • CronJobs define concurrencyPolicy: Forbid for any job that must not run overlapping (backups, migrations)
  • CronJob failedJobsHistoryLimit is at least 2 so debugging information is retained after failures
  • Pods spreading across availability zones via topologySpreadConstraints for services requiring high availability
  • All workloads pull images from nexus.ticketflow.io:5000 (internal registry) not Docker Hub, to avoid rate limits and ensure image availability

Hands-On Lab: Deploy ticket-service with 3 Replicas

Lab — Requires a running Kubernetes cluster (minikube, kind, or EKS)

In this lab you will deploy a simulated ticket-service as a Kubernetes Deployment with 3 replicas, perform a rolling update, and practice a rollback. We use the public nginx image to simulate the service without requiring internal registry access.

Step 1: Create the Deployment

# Create the namespace
kubectl create namespace ticketflow-lab

# Deploy ticket-service (using nginx to simulate the Spring Boot service)
kubectl create deployment ticket-service \
  --image=nginx:1.25-alpine \
  --replicas=3 \
  --namespace=ticketflow-lab

# Verify all 3 Pods are Running
kubectl get pods -n ticketflow-lab -l app=ticket-service

# Expected output: 3 Pods with STATUS=Running
# NAME                              READY   STATUS    RESTARTS
# ticket-service-7d9f8b-abc12       1/1     Running   0
# ticket-service-7d9f8b-def34       1/1     Running   0
# ticket-service-7d9f8b-ghi56       1/1     Running   0

Step 2: Add resource limits and probes via a full manifest

# Save this as ticket-service-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: ticket-service
  namespace: ticketflow-lab
spec:
  replicas: 3
  revisionHistoryLimit: 5
  selector:
    matchLabels:
      app: ticket-service
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0
  template:
    metadata:
      labels:
        app: ticket-service
    spec:
      containers:
        - name: ticket-service
          image: nginx:1.25-alpine
          ports:
            - containerPort: 80
          resources:
            requests:
              cpu: "50m"
              memory: "64Mi"
            limits:
              cpu: "200m"
              memory: "128Mi"
          readinessProbe:
            httpGet:
              path: /
              port: 80
            initialDelaySeconds: 5
            periodSeconds: 5
          livenessProbe:
            httpGet:
              path: /
              port: 80
            initialDelaySeconds: 10
            periodSeconds: 10
# Apply the full manifest
kubectl apply -f ticket-service-deployment.yaml

# Verify the update was applied
kubectl describe deployment ticket-service -n ticketflow-lab | grep -A5 "Strategy"

Step 3: Perform a rolling update

# Simulate a new version release — update to nginx:1.26-alpine
kubectl set image deployment/ticket-service \
  ticket-service=nginx:1.26-alpine \
  -n ticketflow-lab

# Watch the rolling update happen in real time (open a second terminal)
kubectl rollout status deployment/ticket-service -n ticketflow-lab

# While rolling, watch Pods change
kubectl get pods -n ticketflow-lab -l app=ticket-service -w

# After rollout: inspect the ReplicaSet history
# Notice two ReplicaSets — old (0 replicas) and new (3 replicas)
kubectl get replicasets -n ticketflow-lab -l app=ticket-service

Step 4: Simulate a bad deployment and rollback

# "Deploy" a bad image (non-existent tag — ImagePullBackOff)
kubectl set image deployment/ticket-service \
  ticket-service=nginx:99.99-bad-release \
  -n ticketflow-lab

# Watch the deployment stall — new Pods will fail with ImagePullBackOff
kubectl rollout status deployment/ticket-service -n ticketflow-lab
# (Ctrl+C to interrupt)

kubectl get pods -n ticketflow-lab -l app=ticket-service
# You will see: old Pods still Running, new Pods in ImagePullBackOff

# Roll back to the last known-good version instantly
kubectl rollout undo deployment/ticket-service -n ticketflow-lab

# Confirm rollback completed — all 3 Pods running nginx:1.26-alpine again
kubectl rollout status deployment/ticket-service -n ticketflow-lab
kubectl get pods -n ticketflow-lab -l app=ticket-service

Step 5: Explore the ReplicaSet history

# View rollout history — 3 revisions logged
kubectl rollout history deployment/ticket-service -n ticketflow-lab

# View details of revision 2 (the nginx:1.26-alpine version)
kubectl rollout history deployment/ticket-service --revision=2 -n ticketflow-lab

# List all ReplicaSets — notice revision 3 (bad) has 0/0 Pods
kubectl get replicasets -n ticketflow-lab -o wide

# Clean up the lab
kubectl delete namespace ticketflow-lab

What to observe during this lab:

  1. With maxUnavailable: 0, the old Pods are never terminated before a new Pod passes its readiness probe — you always have 3 serving Pods.
  2. The ReplicaSet from the bad deployment has 0 replicas but still exists in Kubernetes — this is the rollback target.
  3. Rollback is instant because the old ReplicaSet already has the container spec and does not need to rebuild anything — Kubernetes simply scales it back up.
  4. During the bad deployment, the 3 good Pods from the previous revision continued serving traffic. No downtime occurred.

Interview Questions

The most commonly asked Kubernetes workload questions in senior engineering and DevOps interviews. Click each question to reveal a complete answer.

Q1: What is the difference between a Deployment and a StatefulSet, and when would you use each?

A Deployment manages stateless applications where all Pod replicas are interchangeable. Pods get random names, share no stable identity, and can be scheduled to any node. Rolling updates replace Pods in any order. Deployments are designed for horizontal scaling — adding or removing Pods changes capacity but not behaviour.

A StatefulSet manages stateful applications that require stable, persistent identity. Pods get ordered, predictable names (app-0, app-1). Each Pod retains its identity across rescheduling via stable DNS names through a headless service. Each Pod gets its own PersistentVolumeClaim that follows it. Pods are created, updated, and deleted in strict order.

When to use Deployment: User-facing APIs, backend microservices, web servers — anything where requests can be served by any instance. All of TicketFlow's application-tier services (ticket-service, user-service, payment-service, api-gateway) are Deployments.

When to use StatefulSet: Databases (PostgreSQL, MySQL, MongoDB), distributed caches (Redis Cluster), message brokers (Kafka, Zookeeper), search engines (Elasticsearch) — any system that maintains cluster membership by stable identity and requires durable per-instance storage.

Key signal: If you need a Pod to reconnect to the same volume after a restart, or other Pods need to address specific instances by name, you need StatefulSet.

Q2: Why do you never use a ReplicaSet directly? What does a Deployment add?

A ReplicaSet guarantees that N replicas of a Pod are always running. However, it has no concept of updates. If you change the Pod template in a ReplicaSet, existing Pods are not replaced — only new Pods (created to replace crashed Pods) will use the new template. To update running Pods, you must manually delete them, which is error-prone and has no coordination logic.

A Deployment wraps a ReplicaSet and adds three critical capabilities:

  1. Rolling updates — when the Pod template changes, the Deployment creates a new ReplicaSet and carefully swaps Pods respecting maxSurge and maxUnavailable constraints. No manual deletion needed. Zero-downtime updates are possible.
  2. Rollback — old ReplicaSets are retained at 0 replicas. kubectl rollout undo instantly scales the old ReplicaSet back up.
  3. Update history — change causes and revision annotations are tracked, enabling auditable deployment history.

In practice, you interact with Deployments, and Kubernetes manages the ReplicaSets on your behalf. Understanding ReplicaSets is important for debugging (they explain why kubectl get rs shows multiple entries with 0 replicas), but creating them directly is an antipattern.

Q3: What is a DaemonSet and when would you use one instead of a Deployment?

A DaemonSet ensures exactly one Pod runs on every node (or a subset of nodes). Unlike a Deployment — where you specify a replica count and Kubernetes decides which nodes to use — a DaemonSet creates one Pod per matching node, automatically expanding as nodes are added and contracting as nodes are removed.

Use a DaemonSet when the workload must interact with every node's local resources:

  • Log collection — Fluentd must read /var/log/containers/ on every node. You cannot route all log files to a centralized Deployment — the agent must be local to where the files exist.
  • Node metrics — Prometheus Node Exporter must access host-level hardware metrics. These are available only locally per node.
  • Network plugins — CNI agents like Calico and Cilium program the Linux kernel network data plane. They must run on every node to handle routing for all Pods on that node.
  • Security agents — Falco reads eBPF/kernel events. Each node's kernel events are local to that node.

The distinguishing question is: "Does this workload need access to the local node's resources (filesystem, kernel, hardware) or does it serve network requests?" If local resources — DaemonSet. If network-served — Deployment.

Q4: How do you perform a zero-downtime rolling update in Kubernetes?

Zero-downtime rolling updates require three things to work together correctly:

  1. Correct rolling update strategy — set maxUnavailable: 0 in the Deployment spec. This ensures the cluster never has fewer running replicas than the desired count. New Pods are added before old ones are removed.
  2. Readiness probe — the new Pod must have a readiness probe configured. Kubernetes only removes an old Pod and marks the new one as "ready to receive traffic" after the readiness probe passes. Without a readiness probe, traffic is routed to a Pod that may still be starting up, causing request failures.
  3. Graceful shutdown — when Kubernetes terminates an old Pod, it sends SIGTERM to the container. The application must handle SIGTERM gracefully: stop accepting new requests, finish in-flight requests, then exit. Spring Boot's actuator supports graceful shutdown via server.shutdown=graceful. Node.js apps need explicit SIGTERM handlers. Without graceful shutdown, in-flight requests are dropped when the Pod is terminated.

Optional but recommended: set terminationGracePeriodSeconds to a value longer than your P99 request latency (e.g., 30 seconds for most web services). This gives the Pod time to drain before Kubernetes forcibly terminates it with SIGKILL.

With these three in place, a rolling update looks like: new Pod starts → readiness probe passes → traffic shifts to new Pod → old Pod receives SIGTERM → old Pod finishes in-flight requests → old Pod exits → Kubernetes repeats for the next Pod.

Q5: What is a Kubernetes Job and how does it differ from a Deployment? When would you run a database migration as a Job?

A Job creates Pods that run to completion. Once the Pod exits successfully, the Job is done. Kubernetes does not restart the Pods after success. This makes Jobs suitable for finite, one-time tasks — the exact opposite of Deployments, which continuously keep Pods running.

Key differences:

  • Deployment Pods use restartPolicy: Always — they are restarted whenever they exit. Job Pods use OnFailure or Never — they are only restarted on failure, and never after success.
  • Deployments run indefinitely. Jobs have a definite end state: Complete or Failed.
  • Jobs track successful completions; Deployments track replica availability.

Why run DB migrations as a Job (not in app startup code):

  1. Race conditions — if 3 new Deployment Pods start simultaneously and all try to run Flyway migrations, all three contend on the same database schema lock. This can cause deadlocks or double-migration errors. A Job runs once and completes before the Deployment Pods start.
  2. Failure isolation — a failed migration stops the Job and blocks the deployment pipeline. This is exactly what you want. If migrations run in app startup, a failed migration causes the Pod to crash-loop indefinitely while the old version continues serving traffic — making it harder to detect.
  3. Auditability — Job completion is a distinct Kubernetes event. You can gate the Deployment rollout on Job completion with CI/CD logic: kubectl wait --for=condition=complete job/migration.
Q6: A StatefulSet Pod is stuck in Pending state. How do you diagnose and resolve it?

A StatefulSet Pod stuck in Pending typically has one of three root causes: scheduling constraints, PVC provisioning failure, or an image pull error. Here is the systematic diagnosis:

Step 1: Describe the Pod to read events

kubectl describe pod postgres-2 -n ticketflow-prod
# Focus on the "Events" section at the bottom

Root cause 1 — Insufficient resources: Event says 0/3 nodes are available: 3 Insufficient cpu. The Pod's resource requests exceed what any single node has available. Solution: reduce resource requests, add nodes to the cluster, or check for other Pods with over-provisioned requests occupying capacity.

Root cause 2 — PVC not bound: Event says persistentvolumeclaim "data-postgres-2" not found or the PVC is in Pending state. Check PVC status:

kubectl get pvc data-postgres-2 -n ticketflow-prod
kubectl describe pvc data-postgres-2 -n ticketflow-prod

If the PVC is Pending: the StorageClass may not have a provisioner available, the storage class name is wrong, or the zone where the Pod wants to schedule does not have available EBS capacity. Check kubectl get storageclass and the provisioner logs.

Root cause 3 — Taint/toleration mismatch: Event says node(s) had taints that the pod didn't tolerate. Add the required toleration to the StatefulSet Pod spec for the tainted node pool.

Root cause 4 — Topology constraint: topologySpreadConstraints or podAntiAffinity rules may prevent scheduling when the cluster does not have enough nodes in different zones. Check the constraints and the node topology labels with kubectl get nodes --show-labels.

Note: because StatefulSet Pods start in strict order, postgres-2 stuck in Pending blocks any future ordered operations (scale-up, rolling updates). Resolving it unblocks the entire StatefulSet.
