ArgoCD & GitOps

Git as the single source of truth — automatically reconcile your Kubernetes cluster state to whatever is declared in your repository.

The Story Behind GitOps

🏦 The Bank That Deployed With Runbooks

In 2019, a well-known fintech company had 40 microservices running on Kubernetes. Deployments were done by an operations team who would SSH into the cluster, run kubectl apply -f ... commands from their laptops, and record what they changed in a Confluence page. Rollbacks meant trying to remember which YAML was applied last week and hoping someone had saved it.

On a Friday afternoon, a routine deployment to the payment-service contained a misconfigured resource limit. Within minutes, the pods were OOMKilled and the service went down. The on-call engineer started reverting changes — but which changes? In what order? Which environment variables had been tweaked directly on the cluster since the last deployment?

Four hours of downtime. A post-mortem that identified a single root cause: there was no single source of truth for what the cluster was supposed to look like. The cluster had drifted far from anything written down. Nobody could confidently describe the live state of the system.

GitOps, and specifically ArgoCD, was invented to solve exactly this problem. The rule is simple: if it is not in Git, it does not exist.

What is GitOps?

🌱 Beginner Analogy

Imagine your home thermostat. You set it to 21°C. The thermostat continuously checks the current room temperature and turns the heating on or off to match. It does not care how the temperature got where it is — it just keeps correcting it back to 21°C.

GitOps is the thermostat for your Kubernetes cluster. Your Git repository declares "the cluster should look like this." ArgoCD continuously checks whether the actual cluster matches that declaration and corrects any drift automatically. Someone deleted a deployment manually? ArgoCD recreates it. A ConfigMap was edited directly? ArgoCD reverts it. The desired state in Git always wins.

GitOps is a set of operational principles, not a specific technology. The four core principles (from the OpenGitOps specification) are:

  1. Declarative — the desired state of the system is expressed declaratively (YAML manifests, Helm values, Kustomize overlays). You describe what you want, not how to get there.
  2. Versioned and immutable — the desired state is stored in a version control system (Git). Every change is a commit. Every state in history is reproducible. Rollback is just checking out a previous commit.
  3. Pulled automatically — approved changes are applied to the system automatically by a software agent (ArgoCD). Humans do not manually run apply commands. The agent pulls from Git — Git never pushes to the cluster.
  4. Continuously reconciled — software agents continuously observe the actual state and attempt to match it to the desired state. Drift is automatically corrected without human intervention.

ArgoCD Architecture

ArgoCD is a Kubernetes-native GitOps tool. It runs inside the cluster it manages and has three core components:

1. Application Controller

The heart of ArgoCD. A Kubernetes controller that continuously watches two things: the desired state (Git repository) and the live state (Kubernetes API). It compares them and determines whether the application is Synced, OutOfSync, or in an Unknown state. When configured for auto-sync, it drives reconciliation loops to eliminate drift.

2. Repo Server

Handles all interaction with Git repositories. It clones repos, fetches updates on a configurable polling interval (default 3 minutes), and generates Kubernetes manifests from source templates. The Repo Server supports three source types: raw YAML manifests, Helm charts (generating manifests from values.yaml), and Kustomize overlays. The generated manifests are cached to avoid expensive template rendering on every sync check.

3. API Server

Exposes a gRPC/HTTP API consumed by both the ArgoCD Web UI and the argocd CLI. Handles authentication (local users, SSO via Dex — OIDC, SAML, GitHub OAuth), authorization (RBAC policies for which teams can sync which applications), and webhook receivers (GitHub/GitLab webhooks trigger immediate sync checks instead of waiting for the polling interval).

4. Dex (Identity Service)

An optional OIDC identity provider bundled with ArgoCD. Delegates authentication to external providers: GitHub OAuth, Google Workspace, Okta, Azure AD. In enterprise environments, Dex is how SSO is wired into ArgoCD so engineers log in with their corporate credentials.

5. Redis

ArgoCD uses Redis as an in-memory cache for repository state, application manifests, and cluster live state data. Reduces load on both the Kubernetes API server and the Git repository by caching results of expensive operations.

💡 Architecture Note

ArgoCD is a pull-based GitOps tool — the agent inside the cluster pulls from Git. This is architecturally superior to push-based CI/CD (where GitHub Actions kubectl apply directly) because the cluster never needs to expose its API server to the internet and no external credentials need to be distributed to CI runners.

Enterprise Scenario: TicketFlow GitOps Setup

🏢 TicketFlow Production GitOps

TicketFlow manages 6 microservices deployed to AWS EKS across three environments: dev, staging, and production. The GitOps repository (github.com/ticketflow/k8s-manifests) is the single source of truth for all cluster state.

Repository layout:

  • apps/ — ArgoCD Application manifests (one per service per environment)
  • services/user-service/, services/ticket-service/, ... — Helm chart values per environment
  • infra/ — cluster-level resources: namespaces, RBAC, NetworkPolicies, Prometheus rules
  • argocd/ — the App-of-Apps root application manifest

When a developer merges a PR to the app repo, GitHub Actions builds and pushes to nexus.ticketflow.io:5000, then opens an automated PR against k8s-manifests updating the image tag in the appropriate values.yaml. A second engineer reviews and merges. ArgoCD detects the change within 3 minutes (or immediately via webhook) and rolls out the new version. No engineer ever runs kubectl apply directly against production.

ArgoCD Application Manifest — ticket-service

# apps/ticket-service-production.yaml
# This ArgoCD Application manifest lives in the GitOps repo
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: ticket-service-production
  namespace: argocd
  # Cascade deletion: when this Application is deleted, also delete K8s resources
  finalizers:
    - resources-finalizer.argocd.argoproj.io
spec:
  project: ticketflow-production

  # Source: where to find the manifests
  source:
    repoURL: https://github.com/ticketflow/k8s-manifests.git
    targetRevision: main
    path: services/ticket-service/overlays/production
    # If using Helm:
    # helm:
    #   releaseName: ticket-service
    #   valueFiles:
    #     - values-production.yaml

  # Destination: where to deploy
  destination:
    server: https://kubernetes.default.svc   # in-cluster
    namespace: production

  # Sync policy
  syncPolicy:
    automated:
      prune: true          # delete resources removed from Git
      selfHeal: true       # revert manual cluster changes
      allowEmpty: false    # don't accidentally sync empty manifests
    syncOptions:
      - CreateNamespace=true
      - PrunePropagationPolicy=foreground
      - PruneLast=true     # prune old resources AFTER new ones are healthy
    retry:
      limit: 5
      backoff:
        duration: 5s
        factor: 2
        maxDuration: 3m

  # Health assessment customization
  ignoreDifferences:
    - group: apps
      kind: Deployment
      jsonPointers:
        - /spec/replicas  # ignore replica count — HPA manages this

CI Pipeline: Update Image Tag in GitOps Repo

# .github/workflows/ticket-service-ci.yml (excerpt)
# Runs in the APPLICATION repo (not the GitOps repo)
name: ticket-service CI/CD

on:
  push:
    branches: [main]
    paths: ['services/ticket-service/**']

jobs:
  build-and-update-gitops:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Build and push to Nexus
        run: |
          IMAGE_TAG="${{ github.sha }}"
          docker build -t nexus.ticketflow.io:5000/ticket-service:${IMAGE_TAG} \
            -f services/ticket-service/Dockerfile \
            services/ticket-service/
          docker push nexus.ticketflow.io:5000/ticket-service:${IMAGE_TAG}
          echo "IMAGE_TAG=${IMAGE_TAG}" >> $GITHUB_ENV
        env:
          NEXUS_USER: ${{ secrets.NEXUS_USER }}
          NEXUS_PASS: ${{ secrets.NEXUS_PASS }}

      - name: Update image tag in GitOps repository
        uses: actions/checkout@v4
        with:
          repository: ticketflow/k8s-manifests
          token: ${{ secrets.GITOPS_PAT }}
          path: k8s-manifests

      - name: Patch kustomization.yaml with new image tag
        run: |
          cd k8s-manifests
          # Use kustomize edit to update the image tag
          kustomize edit set image \
            nexus.ticketflow.io:5000/ticket-service=nexus.ticketflow.io:5000/ticket-service:${{ env.IMAGE_TAG }} \
            --kustomization services/ticket-service/overlays/staging/kustomization.yaml

          git config user.name "github-actions[bot]"
          git config user.email "github-actions[bot]@users.noreply.github.com"
          git commit -am "chore: update ticket-service to ${{ env.IMAGE_TAG }}"
          git push
        # ArgoCD will detect this push and sync staging automatically

Behind the Scenes

What happens when ArgoCD detects an OutOfSync application?
  1. The Repo Server polls the Git repository on its configured interval (default: 3 minutes) or receives an immediate webhook notification from GitHub. It fetches the latest commit for the tracked branch and checks its SHA against the cached SHA.
  2. If the SHA has changed, the Repo Server clones/fetches the new commit, runs the appropriate renderer (plain YAML, Helm template render, Kustomize build), and produces a flat list of Kubernetes resource manifests from the desired state.
  3. The Application Controller receives the rendered manifests and compares them against the live state fetched from the Kubernetes API server. It performs a structural diff — it does not compare raw YAML strings but normalized Kubernetes resource objects. Fields annotated with kubectl.kubernetes.io/last-applied-configuration are excluded from the diff.
  4. If any resource differs (different image tag, different replica count, missing resource, extra resource), the Application status transitions to OutOfSync. The specific diff is recorded and visible in the ArgoCD UI.
  5. If automated.selfHeal: true is set, the Application Controller immediately enqueues a sync operation. If automated is not set, the Application stays OutOfSync until a human triggers a sync via UI or CLI.
  6. During sync, the Application Controller applies resources to the Kubernetes API server using server-side apply (kubectl apply --server-side). Resources are applied in dependency order: Namespaces first, then CRDs, then RBAC, then Deployments and Services. This ordering is controlled by ArgoCD sync waves.
  7. After applying, ArgoCD polls the Kubernetes API to assess health. For Deployments, "Healthy" means availableReplicas == desiredReplicas. For Jobs, it means completion. For custom resources, ArgoCD uses Lua-based health assessment scripts (customizable per CRD kind).
  8. Once all resources are Healthy and the applied state matches the desired state, the Application transitions to Synced + Healthy. A sync history entry is written with the Git commit SHA, timestamp, and result.

Prune behaviour: If prune: true is set and a resource exists in the cluster but no longer exists in Git (e.g., a Service was deleted from the manifests), ArgoCD deletes it from the cluster. Without prune, orphaned resources remain. With PruneLast=true, ArgoCD waits until all new resources are healthy before deleting the old ones — enabling zero-downtime resource replacement.

How does self-healing work and when is it triggered?

Self-healing is triggered when the live state diverges from the desired state through any means other than ArgoCD itself: a developer ran kubectl edit deployment, a bug in a controller modified a resource, an HPA scaled replicas and the deployment was then manually edited, etc.

The Application Controller watches Kubernetes resources via informers (watch API). When any watched resource changes, it re-evaluates the diff. If selfHeal: true, it immediately queues a sync — reverting the change back to Git's desired state.

Self-heal has a self-heal timeout (default: 5 seconds) to debounce rapid changes and avoid sync storms. After detection, ArgoCD waits 5 seconds before syncing, giving time for legitimate in-progress operations (like rollouts managed by ArgoCD itself) to complete without triggering unnecessary re-syncs.

Important caveat: fields listed in ignoreDifferences are excluded from self-healing. This is how HPA-managed replica counts are preserved — you add /spec/replicas to ignoreDifferences for Deployments managed by HPA, and ArgoCD will not revert HPA's scaling decisions back to the value in Git.

How does ArgoCD Projects work for multi-team access control?

An ArgoCD Project (AppProject) is a grouping construct that enforces boundaries on what Applications within it can do:

  • Source repos — which Git repositories Applications in this project may use as source. A project for the payments team can be restricted to only the payments GitOps repo.
  • Destination clusters and namespaces — which K8s clusters and namespaces Applications in this project may deploy to. The payments project can only deploy to the payments namespace, not kube-system.
  • Allowed resource kinds — which Kubernetes resource types are permitted. You can block projects from creating ClusterRoles or PersistentVolumes.
  • RBAC roles — ArgoCD RBAC (defined in argocd-rbac-cm ConfigMap) maps SSO groups to project-level roles. The frontend team's GitHub team can sync Applications in the frontend project but cannot touch infra project applications.

At TicketFlow, there are four AppProjects: ticketflow-dev (all engineers can sync), ticketflow-staging (senior engineers + CI bots can sync), ticketflow-production (only SRE team and automated sync), and ticketflow-infra (SRE only, cluster-level resources).

Common Mistakes

Mistake 1: Using the Application Repository as the GitOps Repository

Developers put their Kubernetes manifests in a k8s/ folder inside the same repository as their application code. Every code commit triggers an ArgoCD sync, even if only a unit test changed. ArgoCD's sync history becomes noise — you cannot tell when an actual Kubernetes change was deployed versus when only app code changed.

Fix: Maintain a separate GitOps repository exclusively for Kubernetes manifests. The CI pipeline updates it via automated commits. This gives you a clean audit trail of exactly what changed in the cluster, when, and why — completely independent of application code history.

Mistake 2: Enabling Auto-Sync on Production Without prune and selfHeal Thought-Out

Teams enable automated: {} on production but forget about prune: true, leaving orphaned resources that consume cluster resources indefinitely. Or they enable selfHeal: true without adding ignoreDifferences for HPA-managed replicas, causing ArgoCD to fight HPA — ArgoCD resets replicas to 3, HPA scales to 10, ArgoCD resets to 3, ad infinitum, generating hundreds of unnecessary sync operations.

Fix: For production: enable prune: true with PruneLast=true. Add ignoreDifferences for /spec/replicas on all Deployments managed by HPA. Test auto-sync behavior in staging for at least two weeks before enabling in production.

Mistake 3: Storing All Environments in One Branch

All three environments (dev/staging/production) track the same main branch. Merging a staging-only experimental feature to main immediately triggers a production sync. Environments lose their independence — promotion becomes implicit and uncontrolled.

Fix: Use directory-based environment separation (preferred) within a single main branch: overlays/dev, overlays/staging, overlays/production. Each ArgoCD Application points to its own overlay path. Promote between environments by running kustomize edit set image in the specific overlay and committing. Promotion is explicit, reviewable, and auditable.

Mistake 4: Not Setting ignoreDifferences for Operator-Managed Fields

Some Kubernetes operators and controllers mutate resource fields after creation — cert-manager adds annotations to Ingresses, Istio injects sidecar containers into pod specs, AWS Load Balancer Controller adds annotations to Services. ArgoCD sees these mutations as drift and marks the Application as OutOfSync permanently, generating alert noise and causing spurious syncs that remove the operator's additions.

Fix: Add ignoreDifferences rules for fields managed by operators. For Istio: ignore the spec.template.spec.containers count. For cert-manager: ignore metadata.annotations matching cert-manager prefixes. Identify these during staging deployment before production rollout.

Mistake 5: No ArgoCD High Availability Setup

ArgoCD is deployed as a single replica of each component. The argocd-application-controller pod crashes and no deployments happen for 20 minutes while it restarts — including an urgent hotfix deployment during an incident. The tool meant to eliminate downtime itself causes downtime.

Fix: Run ArgoCD in HA mode: 3 replicas of argocd-server, 3 replicas of argocd-repo-server, a StatefulSet for argocd-application-controller with sharding enabled (one shard per N applications). Use the argocd-ha install manifests. Back up ArgoCD's configuration (ApplicationSet, AppProject, and Application resources) to Git — which you already have if you follow the App-of-Apps pattern.

Architect Perspective

Architect Note

GitOps is not just an operational pattern — it is an organizational pattern. When Git is the sole mechanism for cluster changes, several organizational benefits emerge that go beyond deployment automation:

  • Immutable audit trail: every cluster change is a Git commit with an author, timestamp, and message. Compliance teams can audit exactly what was deployed to production, when, and by whom — without querying Kubernetes audit logs or asking engineers.
  • Pull request as a change management process: production changes require a PR, a review, and a merge. This enforces four-eyes principles without a separate change management tool. The PR review IS the change advisory board approval for most organizations.
  • Disaster recovery becomes trivial: to rebuild a destroyed cluster, create a new cluster and apply the ArgoCD bootstrap manifest. ArgoCD reconciles the entire desired state within minutes. The GitOps repo is your runbook, your DR plan, and your configuration database all in one.
  • Progressive delivery integration: combine ArgoCD with Argo Rollouts for canary deployments and blue-green strategies. The rollout strategy is declared in Git like everything else. A canary weight of 10% → 50% → 100% is a series of commits, reviewable and revertible.

When GitOps does not fit: GitOps assumes all desired state can be expressed as declarative Kubernetes manifests. Stateful workloads with complex migration procedures (database schema migrations, data backups before upgrades) require orchestration beyond what ArgoCD provides. Use ArgoCD Sync Waves with pre-sync and post-sync hooks for these cases, or complement with a tool like Argo Workflows for the procedural parts.

Security boundary: the GitOps repository is now a privileged artifact — committing to it is equivalent to deploying to production. Protect the main branch with required reviews, signed commits (GPG/SSH commit signing), and branch protection rules. A compromised GitOps repo gives an attacker control over your entire cluster.

Production Readiness Checklist

  • GitOps repository is separate from application code repositories
  • ArgoCD deployed in HA mode (3 replicas of server and repo-server, sharded app-controller)
  • All Applications organized under AppProjects with scoped source repos and destination namespaces
  • SSO configured via Dex (GitHub OAuth or corporate OIDC) — no shared admin passwords
  • RBAC policies defined: dev team syncs dev, SRE syncs production, no one has wildcard access
  • Auto-sync enabled with prune: true and PruneLast: true on all production Applications
  • selfHeal enabled and ignoreDifferences configured for HPA-managed replicas and operator-injected fields
  • App-of-Apps or ApplicationSet pattern used — no manually created Applications that are not tracked in Git
  • GitHub/GitLab webhooks configured for instant sync trigger instead of relying on polling
  • ArgoCD notifications configured: Slack/PagerDuty alerts for sync failures and degraded health
  • Sync waves configured for ordered rollout: infra resources before application resources
  • GitOps repo branch protection: required PR reviews, signed commits, no direct push to main
  • Argo CD disaster recovery tested: cluster rebuilt from GitOps repo bootstrap procedure
  • Image update automation configured (Argo CD Image Updater or CI-driven tag updates in GitOps repo)

Hands-On Lab: Deploy TicketFlow ticket-service via ArgoCD

Lab — Requires: kubectl + a running K8s cluster (minikube, kind, or EKS) + argocd CLI

This lab walks through installing ArgoCD, creating a GitOps repository structure, and deploying the TicketFlow ticket-service using a real ArgoCD Application manifest.

Step 1: Install ArgoCD

# Start minikube if using local cluster
minikube start --cpus=4 --memory=6144

# Install ArgoCD
kubectl create namespace argocd
kubectl apply -n argocd \
  -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml

# Wait for ArgoCD to be ready
kubectl -n argocd wait --for=condition=available deployment \
  argocd-server argocd-repo-server --timeout=120s

# Get initial admin password
ARGOCD_PASSWORD=$(kubectl -n argocd get secret argocd-initial-admin-secret \
  -o jsonpath="{.data.password}" | base64 -d)
echo "Admin password: $ARGOCD_PASSWORD"

# Port-forward the UI
kubectl port-forward svc/argocd-server -n argocd 8443:443 &
# Open https://localhost:8443 — accept the self-signed cert warning

Step 2: Create a GitOps repository structure locally

# Create a local GitOps repo (in a real scenario this would be on GitHub)
mkdir -p ~/ticketflow-gitops/services/ticket-service/base
mkdir -p ~/ticketflow-gitops/services/ticket-service/overlays/dev
cd ~/ticketflow-gitops
git init

# Create the base deployment manifest
cat > services/ticket-service/base/deployment.yaml << 'EOF'
apiVersion: apps/v1
kind: Deployment
metadata:
  name: ticket-service
spec:
  replicas: 1
  selector:
    matchLabels:
      app: ticket-service
  template:
    metadata:
      labels:
        app: ticket-service
    spec:
      containers:
        - name: ticket-service
          image: nginx:1.25-alpine   # using nginx as stand-in for demo
          ports:
            - containerPort: 80
---
apiVersion: v1
kind: Service
metadata:
  name: ticket-service
spec:
  selector:
    app: ticket-service
  ports:
    - port: 80
      targetPort: 80
EOF

# Create the dev overlay kustomization
cat > services/ticket-service/overlays/dev/kustomization.yaml << 'EOF'
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
resources:
  - ../../base
namespace: dev
commonLabels:
  environment: dev
EOF

# Commit everything
git add .
git commit -m "initial: ticket-service manifests"

Step 3: Register the repository and create the Application

# Login to ArgoCD CLI
argocd login localhost:8443 \
  --username admin \
  --password "$ARGOCD_PASSWORD" \
  --insecure

# Create the dev namespace
kubectl create namespace dev

# Create the ArgoCD Application pointing at the local repo
# (in a real scenario use the GitHub HTTPS URL)
argocd app create ticket-service-dev \
  --repo file:///$(pwd) \
  --path services/ticket-service/overlays/dev \
  --dest-server https://kubernetes.default.svc \
  --dest-namespace dev \
  --sync-policy automated \
  --self-heal \
  --auto-prune

# Check the application status
argocd app get ticket-service-dev

# Watch the sync happen
argocd app wait ticket-service-dev --health

# Verify pods are running
kubectl get pods -n dev

Step 4: Simulate a change and observe reconciliation

# Manually edit the deployment in the cluster (simulate an operator)
kubectl scale deployment ticket-service -n dev --replicas=3

# ArgoCD should detect this drift within 5 seconds (selfHeal)
# Watch it revert back to 1 replica
watch argocd app get ticket-service-dev

# Now make a legitimate change via Git (the right way)
# Update replica count in the overlay
cat > services/ticket-service/overlays/dev/kustomization.yaml << 'EOF'
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
resources:
  - ../../base
namespace: dev
commonLabels:
  environment: dev
patches:
  - patch: |-
      - op: replace
        path: /spec/replicas
        value: 2
    target:
      kind: Deployment
      name: ticket-service
EOF

git add . && git commit -m "feat: scale ticket-service to 2 replicas in dev"

# Trigger immediate sync (instead of waiting for poll)
argocd app sync ticket-service-dev

# Verify 2 pods now running
kubectl get pods -n dev

Step 5: Rollback via Git revert

# Check sync history — shows all previous Git commits ArgoCD synced
argocd app history ticket-service-dev

# Revert the last commit (go back to 1 replica)
git revert HEAD --no-edit
git push   # or just commit locally if using file:// repo

# Sync ArgoCD to pick up the revert immediately
argocd app sync ticket-service-dev --wait

# Confirm back to 1 replica
kubectl get deployment ticket-service -n dev

echo "Lab complete! You deployed via GitOps and practiced rollback."

Key observations from this lab:

  1. The cluster state was corrected within seconds of the manual scale — this is self-healing in action.
  2. The legitimate change via Git commit was picked up and applied automatically — zero kubectl apply commands from you.
  3. Rollback was a git revert + git push — no special ArgoCD rollback knowledge required.
  4. The full history of what was deployed is visible as Git commits — the GitOps repo is your deployment log.

Interview Questions

These are the most commonly asked ArgoCD and GitOps questions in senior DevOps, platform engineering, and architect-level interviews.

Q1: What is GitOps and how does it differ from traditional CI/CD?

GitOps is an operational framework that uses Git as the single source of truth for declarative infrastructure and application state. The desired state of the system is always expressed in Git, and a software agent continuously reconciles the live system to match that state.

The key difference from traditional CI/CD is the direction of deployment:

  • Traditional CI/CD (push-based): A CI runner (e.g., GitHub Actions) has cluster credentials and actively pushes changes — runs kubectl apply, helm upgrade, etc. The cluster is a passive target.
  • GitOps (pull-based): An agent inside the cluster (ArgoCD) pulls from Git. The cluster is an active participant. No external system holds cluster credentials. The cluster owns its own state reconciliation.

Other differences: GitOps provides continuous reconciliation (auto-corrects drift, not just deploys once), complete audit trail (every change is a Git commit), and built-in rollback (revert a commit). Traditional CI/CD deploys once and does not monitor for subsequent drift.

In practice, TicketFlow uses both: GitHub Actions for CI (test, build, push image) and ArgoCD for CD (deploy, reconcile). CI updates the GitOps repo; ArgoCD deploys from it.

Q2: What are the four core components of ArgoCD and what does each do?

1. Application Controller — a Kubernetes controller that continuously compares the desired state (from Git, rendered by Repo Server) against the live state (from Kubernetes API). It determines sync status (Synced/OutOfSync), triggers automatic synchronization when configured, and assesses resource health. It is the core reconciliation engine.

2. Repo Server — responsible for all Git interactions. Clones repositories, checks for new commits, and generates (renders) Kubernetes manifests from source format: raw YAML, Helm chart values, or Kustomize overlays. Caches rendered manifests to avoid redundant rendering. Completely stateless — can be scaled horizontally.

3. API Server — exposes the gRPC/HTTP API consumed by the Web UI and argocd CLI. Handles authentication (SSO via Dex), authorization (RBAC), and real-time streaming of application state updates via Server-Sent Events. Also receives Git webhooks to trigger immediate sync checks.

4. Dex — an optional OIDC identity provider that federates authentication to external systems: GitHub, Google, Okta, Azure AD. Allows engineers to log into ArgoCD with their corporate SSO credentials instead of a local username/password.

Supporting services: Redis (caching of cluster state and rendered manifests), ApplicationSet Controller (generates Applications from templates using generators like Git directory scanning and cluster enumeration).

Q3: What is the App-of-Apps pattern and why is it used?

The App-of-Apps pattern solves the bootstrapping problem: how do you manage many ArgoCD Applications in a GitOps way when ArgoCD itself manages resources declaratively?

The pattern works like this: you create one root Application that points to a Git directory containing YAML files, where each YAML file defines an ArgoCD Application resource. The root Application is the only thing you apply manually (kubectl apply -f root-app.yaml). From that point, every other Application is managed by ArgoCD itself — adding, updating, or deleting Application manifests in Git is all that is needed to manage the entire set of deployed services.

Benefits:

  • All ArgoCD Application definitions are in Git — not created manually via CLI or UI.
  • Adding a new microservice = add one YAML file to the apps/ directory and commit.
  • Deleting a service = delete its Application YAML file from Git — ArgoCD removes the Application and (with prune enabled) its Kubernetes resources.
  • The entire cluster configuration is bootstrappable from Git in minutes after a disaster.

App-of-Apps vs ApplicationSet: ApplicationSet is the newer, more powerful evolution. Where App-of-Apps requires manually maintaining individual Application YAML files, ApplicationSet generates them dynamically from templates and generators (Git directory scanning, cluster enumeration, list generators). For new projects, ApplicationSet is preferred. App-of-Apps is still valuable for heterogeneous Applications that cannot be expressed with a uniform template.

Q4: How do you handle secrets in a GitOps workflow?

Secrets are the most challenging aspect of GitOps because Git repositories should never contain plaintext secret values — they may be public, or at minimum accessible to many engineers.

There are three main approaches used in production GitOps environments:

1. Sealed Secrets (Bitnami) — an in-cluster controller that holds a private key. You encrypt secrets using the public key (kubeseal CLI) and commit the encrypted SealedSecret YAML to Git. Only the Sealed Secrets controller in that specific cluster can decrypt it. Safe to commit. Limitation: rotation requires re-encrypting and committing; tight coupling to the cluster's key pair.

2. External Secrets Operator (ESO) — the preferred enterprise approach. ExternalSecret resources in Git reference secrets in external vaults (AWS Secrets Manager, HashiCorp Vault, Azure Key Vault, GCP Secret Manager). The operator fetches the actual secret values from the vault and creates Kubernetes Secrets. Git contains only the reference path (secretsmanager/ticketflow/production/db-password), never the value. At TicketFlow: all production secrets are in AWS Secrets Manager; ExternalSecret manifests are committed to Git.

3. ArgoCD Vault Plugin (AVP) — a plugin that integrates with Helm/Kustomize rendering. Manifest templates contain placeholders (<path:secret/data/db#password>). The plugin fetches values from Vault at render time, substituting placeholders before ArgoCD applies manifests. Useful when all config is already in HashiCorp Vault.

What not to do: commit plaintext Kubernetes Secrets (even base64-encoded), use kubectl create secret manually (bypasses GitOps), or store secrets in Helm values files committed to Git.

Q5: How do you perform a rollback with ArgoCD and what are the different options?

ArgoCD offers three rollback mechanisms, with different trade-offs:

1. Git Revert (recommended) — the purest GitOps approach. Run git revert <commit-sha> on the GitOps repository and push. This creates a new commit that undoes the problematic change. ArgoCD detects the new commit and reconciles the cluster to the reverted state. This is the recommended approach because: the rollback itself is auditable (a commit with a message and author), it preserves forward history, and it follows the GitOps principle that all changes go through Git.

2. ArgoCD CLI Rollback — argocd app rollback <app-name> <history-id>. ArgoCD syncs the application to a previous Git revision from its sync history. Fast and convenient during an incident. Limitation: this puts the application in a state where Git (main branch) says one thing but the cluster is running something different. ArgoCD marks the application as OutOfSync. You must follow up with a proper Git revert to restore alignment.

3. ArgoCD UI History — in the ArgoCD web interface, navigate to the application, click History, select a previous sync revision, and click Rollback. Equivalent to the CLI approach.

At TicketFlow: during incidents, use the CLI rollback for immediate recovery (fast). Then, within 30 minutes of recovery, open a PR with a proper git revert to restore Git as the source of truth. This two-step process balances recovery speed with GitOps integrity.

Note: if automated.selfHeal: true is configured, ArgoCD will immediately re-sync the application to the current Git HEAD after a CLI rollback, undoing the rollback. Disable automated sync before performing a CLI rollback in a selfHeal-enabled environment.

Q6: What is an ApplicationSet and when would you use it over App-of-Apps?

An ApplicationSet is a custom resource that generates multiple ArgoCD Application objects from a single template, using generators to enumerate the variations. Instead of maintaining 18 separate Application YAML files (6 services × 3 environments), you write one ApplicationSet with a generator that produces all 18 automatically.

Generator types:

  • Git generator — scans a Git repository for directories or files matching a pattern. A new directory services/notification-service/overlays/dev automatically creates a new Application without touching the ApplicationSet manifest.
  • Cluster generator — enumerates registered ArgoCD clusters matching label selectors. Used for deploying the same application to multiple clusters (multi-region, multi-tenant).
  • List generator — a static list of key-value pairs. Simple and explicit, useful when the variations are known and enumerable.
  • Matrix generator — computes the Cartesian product of two generators (e.g., all services × all clusters).
  • SCM provider generator — scans GitHub/GitLab organizations for repositories matching a filter, generating one Application per repo.

Use ApplicationSet when: you have many similar Applications (same structure, different parameters like service name/environment/cluster), when onboarding new services should be zero-config (just create the directory), or when deploying to multiple clusters.

Use App-of-Apps when: your Applications are heterogeneous (different source formats, different sync policies, different project assignments) and cannot be expressed with a uniform template. Also use App-of-Apps as the root that deploys ApplicationSets themselves — a common pattern at scale.

Q7: How would you design a promotion workflow across dev, staging, and production with ArgoCD?

A robust multi-environment promotion workflow with ArgoCD typically follows this structure:

Repository layout: a single GitOps repository with Kustomize overlays per environment — overlays/dev, overlays/staging, overlays/production. Each overlay contains a kustomization.yaml that patches the base manifests with environment-specific values (image tags, replica counts, resource limits, environment variables).

Promotion flow:

  1. CI builds image: GitHub Actions builds nexus.ticketflow.io:5000/ticket-service:sha-abc123 and pushes to Nexus.
  2. Auto-deploy to dev: CI opens a PR against the GitOps repo updating the image tag in overlays/dev/kustomization.yaml. An auto-merge bot (or --auto-merge GitHub label) merges it immediately. ArgoCD picks it up and deploys to dev within minutes.
  3. Promote to staging: after automated integration tests pass in dev, CI (or a developer) opens a second PR updating the image tag in overlays/staging/kustomization.yaml. Requires one peer review. ArgoCD deploys to staging on merge.
  4. Promote to production: after staging sign-off, a PR updates overlays/production/kustomization.yaml. Requires two senior reviews + SRE approval. Merged during a deployment window. ArgoCD auto-syncs production.

Progressive delivery: combine with Argo Rollouts for canary deployments in production. The production Application points to an argo-rollout resource instead of a plain Deployment. Rollout strategy (canary steps: 10% → analysis → 50% → analysis → 100%) is declared in Git. ArgoCD deploys the Rollout; Argo Rollouts controller manages the traffic split.

Key safeguards: branch protection on main in the GitOps repo, required status checks (automated tests passing in source repo before GitOps PR can be merged), CODEOWNERS file requiring SRE approval for production overlay changes, and notifications to Slack on every production sync.
