Troubleshooting Playbook

Systematic methodologies and kubectl command references to diagnose and resolve Kubernetes production incidents fast.

The Story Behind the Playbook

🚨 2:47 AM — Production Is Down

The on-call alert fires. The TicketFlow payment-service is returning 503s. Revenue is bleeding. Your phone is ringing. You have no idea where to start.

The engineer who panics and clicks randomly through dashboards will take 90 minutes to resolve what should take 10. They'll restart the wrong pod, roll back the wrong deployment, and escalate three times before finding the actual cause: a misconfigured ConfigMap that prevents the pod from starting.

The engineer with a playbook opens a terminal and works through a systematic decision tree. Observe the symptoms. Isolate the scope. Form a hypothesis. Test it. Fix it. Within 8 minutes, the service is restored and a post-mortem is being drafted. The difference is not raw skill — it is methodology. A troubleshooting playbook converts chaos into procedure. This page is that playbook.

The Debug Methodology

🥪 Beginner Analogy

Think of debugging like a doctor diagnosing a patient. A good doctor doesn't immediately prescribe surgery. They observe symptoms (fever, pain location), narrow down the problem (blood test, X-ray), form a hypothesis (likely appendicitis), test it (confirm with imaging), then apply the precise fix (surgery — not chemotherapy). Randomly treating every possible disease wastes time and can make things worse. Kubernetes troubleshooting works exactly the same way.

Every Kubernetes incident should follow the same five-step scientific method. Skipping steps leads to misdiagnosis, wasted effort, and sometimes making the problem worse.

Step 1: Observe — What exactly is broken?

Start by gathering facts, not guessing. What does the monitoring say? Which service is affected? What is the exact error? Since when? How many pods are affected? Is it all pods or just some? Is traffic still partially routing? Never form a hypothesis before you have complete observations.

Step 2: Isolate — What is the blast radius?

Is the problem in one pod, one node, one namespace, or the whole cluster? Is it one deployment or multiple? Does it affect only one AZ? Isolation tells you whether you have an application bug, a node failure, a network partition, or a cluster-level issue. Each has a completely different fix path.

Step 3: Hypothesize — What is the most likely cause?

Based on your observations and blast radius, rank the most probable causes. A pod in CrashLoopBackOff most likely has an application error or misconfigured environment. A Pending pod is almost certainly a scheduling failure — resource starvation or affinity conflict. State a concrete hypothesis before you act.

Step 4: Test — Confirm or refute the hypothesis

Use read-only commands (kubectl describe, kubectl logs, kubectl get events) to confirm your hypothesis without making changes. Only escalate to making changes when you are confident in the diagnosis. Applying a fix before confirming the cause is how you create new incidents during an existing incident.

Step 5: Fix — Apply the minimal targeted change

Apply the smallest change that resolves the confirmed root cause. Document exactly what you changed and why. After fixing, verify the service has recovered and monitor for 5 minutes before standing down. Then write a post-mortem.

CrashLoopBackOff — Causes and Fix

CrashLoopBackOff is the most common Kubernetes failure mode. It means: the container starts, immediately exits with a non-zero code, Kubernetes restarts it, and the cycle repeats with exponential backoff (10s, 20s, 40s, up to 5 minutes between attempts).

Common Root Causes

  • Application crash at startup — the process throws an unhandled exception before it can bind to its port. Check kubectl logs --previous to see the stack trace from the last crash.
  • Missing or wrong environment variable — the app tries to read DB_HOST or JWT_SECRET from the environment, finds it missing or wrong, and exits with code 1. The fix is in the ConfigMap or Secret, not the application.
  • Wrong CMD / entrypoint — the Dockerfile CMD runs a script that does not exist in the image, or the entrypoint expects arguments that are not passed. The container exits immediately with "executable not found."
  • Port conflict — the app tries to bind to a port already in use by another process in the same pod. Rare but possible with multi-container pods.
  • Failing health check causing restart — a liveness probe fails immediately after startup because the probe is misconfigured (wrong path, wrong port, or initialDelaySeconds too short). Kubernetes restarts the pod thinking it is unhealthy.
  • Permissions error — the container runs as a non-root user but tries to write to a directory owned by root. Fix with a securityContext fsGroup or initContainer that chowns the directory.

OOMKilled — Causes and Fix

OOMKilled (Out of Memory Killed, exit code 137) means the container's process exceeded its configured memory limit and the Linux kernel's OOM killer terminated it. Kubernetes then reports the status as OOMKilled in the pod description.

Common Root Causes

  • Memory limit set too low — the application legitimately needs more memory than allocated. The JVM's default heap may exceed the container limit. Node.js V8 heap defaults can exceed container limits on high-traffic services.
  • Memory leak — the application slowly leaks memory over time. Each request allocates objects that are never garbage collected. Memory grows steadily until the OOM killer fires. The fix is in the application, not the limit.
  • JVM not container-aware — Java versions before JDK 8u191 / JDK 10 did not respect cgroup memory limits. The JVM saw the full host memory (e.g., 64 GB) and allocated a heap proportional to that, far exceeding the container's 512 MB limit.
  • Sudden traffic spike — the service handles more concurrent requests than anticipated, each allocating in-memory caches or buffers, collectively exceeding the limit.

Diagnostic Commands

# Confirm OOMKilled and see which container was killed
kubectl describe pod ticket-service-6c8b4d9f-p7kqm -n ticketflow
# Look for: Last State: Terminated  Reason: OOMKilled  Exit Code: 137

# Check current memory usage with metrics-server
kubectl top pod -n ticketflow
kubectl top pod ticket-service-6c8b4d9f-p7kqm -n ticketflow --containers

# Check memory limits currently configured
kubectl get pod ticket-service-6c8b4d9f-p7kqm -n ticketflow \
  -o jsonpath='{.spec.containers[*].resources}'

# View historical memory from Prometheus (if installed)
# Example PromQL: container_memory_working_set_bytes{pod=~"ticket-service.*"}

# Check if JVM is respecting container limits (Spring Boot actuator)
kubectl exec -it ticket-service-6c8b4d9f-p7kqm -n ticketflow -- \
  curl -s localhost:8080/actuator/metrics/jvm.memory.max

Fix: Tune JVM for Containers (ticket-service is Spring Boot)

# deployment.yaml for ticket-service
containers:
- name: ticket-service
  image: nexus.ticketflow.io:5000/ticket-service:v3.1.0
  env:
  # Tell JVM to use container memory limits (JDK 11+)
  - name: JAVA_OPTS
    value: "-XX:+UseContainerSupport -XX:MaxRAMPercentage=75.0 -XX:InitialRAMPercentage=50.0"
  resources:
    requests:
      memory: "512Mi"
      cpu: "250m"
    limits:
      memory: "1Gi"   # Increase limit if app legitimately needs more
      cpu: "1000m"

ImagePullBackOff — Causes and Fix

ImagePullBackOff means the kubelet on the node tried to pull the container image and failed. After the first failure (ErrImagePull), Kubernetes applies exponential backoff and the status changes to ImagePullBackOff.

Common Root Causes

  • Image does not exist — the image name or tag is wrong. A typo in the deployment manifest, or the image was never pushed to the registry, or the tag was deleted.
  • Registry authentication failure — the node cannot authenticate to a private registry (Nexus, ECR). The imagePullSecret is missing, expired, or not attached to the ServiceAccount.
  • Registry unreachable — the registry is down, the network policy blocks egress to the registry, or DNS for nexus.ticketflow.io is not resolving from the node.
  • Image too large / node disk full — the kubelet cannot store the pulled image because the node's disk is full. Less common but devastating when it happens on all nodes simultaneously.

Diagnostic Commands

# Describe the pod — Events will show the exact error
kubectl describe pod notification-service-5f9c7b8d-hx4nt -n ticketflow
# Look for: Failed to pull image "nexus.ticketflow.io:5000/notification-service:v1.8.0"
#           Error: pull access denied / manifest unknown / no such host

# Verify the image exists in the registry
# (from a machine with registry access)
docker manifest inspect nexus.ticketflow.io:5000/notification-service:v1.8.0

# Check if the imagePullSecret exists in the namespace
kubectl get secret nexus-registry-secret -n ticketflow

# Verify the ServiceAccount has the imagePullSecret attached
kubectl get serviceaccount default -n ticketflow -o yaml
kubectl get serviceaccount ticketflow-sa -n ticketflow -o yaml

# Test registry connectivity from inside the cluster
kubectl run registry-test --image=curlimages/curl --restart=Never -n ticketflow \
  --command -- curl -v https://nexus.ticketflow.io:5000/v2/

Fix: Create and Attach imagePullSecret

# Create the registry credential secret
kubectl create secret docker-registry nexus-registry-secret \
  --docker-server=nexus.ticketflow.io:5000 \
  --docker-username=ci-bot \
  --docker-password=$NEXUS_PASSWORD \
  --docker-email=ci@ticketflow.io \
  -n ticketflow

# Option A: Attach to a specific deployment
# In the deployment manifest:
# spec.template.spec.imagePullSecrets:
# - name: nexus-registry-secret

# Option B: Attach to ServiceAccount (all pods in namespace use it automatically)
kubectl patch serviceaccount default -n ticketflow \
  -p '{"imagePullSecrets": [{"name": "nexus-registry-secret"}]}'

Pending Pods — Resource Starvation and Affinity

A Pending pod means the Kubernetes scheduler cannot find a suitable node to run it. The pod exists in etcd but no kubelet has claimed it. It will stay Pending forever unless the underlying constraint is resolved.

Root Cause 1: Insufficient Resources

Every node has allocatable CPU and memory. When all nodes are at or near capacity, new pods cannot be scheduled. This is the most common cause of Pending in production — a traffic spike triggers HPA to scale up, but the cluster has no spare capacity.

# See node resource capacity and what is already allocated
kubectl describe nodes | grep -A5 "Allocated resources"

# Or use a cleaner format:
kubectl get nodes -o custom-columns=\
"NAME:.metadata.name,CPU:.status.capacity.cpu,MEMORY:.status.capacity.memory,\
CPU_ALLOC:.status.allocatable.cpu,MEM_ALLOC:.status.allocatable.memory"

# See what is consuming resources on each node
kubectl top nodes

# See the scheduler's specific rejection reason
kubectl describe pod order-service-84b6f9c5-tw3xp -n ticketflow
# Look for Events: 0/3 nodes are available: 3 Insufficient memory

# Check if Cluster Autoscaler is configured and scaling
kubectl get events -n kube-system | grep -i "scaled up\|scale"
kubectl logs -n kube-system -l app=cluster-autoscaler --tail=50

Root Cause 2: Node Affinity / Taints and Tolerations

A pod may require specific nodes (via nodeSelector or node affinity) that have no capacity, or the node has a taint the pod does not tolerate.

# Check the pod's scheduling constraints
kubectl get pod user-service-9b4c7f2a-mn8vk -n ticketflow -o yaml | grep -A20 "affinity\|nodeSelector\|tolerations"

# Check node labels — affinity rules reference these
kubectl get nodes --show-labels

# Check node taints — pods must tolerate them or they won't schedule
kubectl describe nodes | grep -i taint

# Example: EKS GPU nodes often have:
# node.kubernetes.io/disk-pressure: NoSchedule
# If your pod has no toleration for this, it won't schedule on that node

# Verify which nodes match an affinity rule
kubectl get nodes -l topology.kubernetes.io/zone=us-east-1a

Service Not Routing — Endpoint Check

A Service routes to pods by selecting pods whose labels match the Service's selector. If the selector does not match any pod labels, the Service has zero endpoints and all traffic returns 503.

# Step 1: Check if the Service has any endpoints
kubectl get endpoints api-gateway-svc -n ticketflow
# ENDPOINTS column shows  = no matching pods = 503

# Step 2: Compare Service selector to Pod labels
kubectl get svc api-gateway-svc -n ticketflow -o yaml | grep -A5 selector
# selector:
#   app: api-gateway
#   version: v2

kubectl get pods -n ticketflow --show-labels | grep api-gateway
# If pods are labelled app=api-gateway but missing version=v2
# the selector does NOT match and endpoints will be empty

# Step 3: Check if pods pass readiness probes
# Kubernetes only adds a pod to endpoints if it is READY
kubectl get pods -n ticketflow | grep api-gateway
# READY column must show 1/1 (or N/N for multi-container pods)

# Step 4: Describe the Service for full detail
kubectl describe svc api-gateway-svc -n ticketflow

# Step 5: Test connectivity directly to a pod IP (bypass the Service)
POD_IP=$(kubectl get pod api-gateway-7d8b6c9f-wr4tz -n ticketflow \
  -o jsonpath='{.status.podIP}')
kubectl run curl-test --image=curlimages/curl --restart=Never -n ticketflow \
  --command -- curl -v http://$POD_IP:80/health

# Step 6: Check NetworkPolicy — may be blocking the Service's source
kubectl get networkpolicy -n ticketflow
kubectl describe networkpolicy allow-api-gateway -n ticketflow

Node NotReady

A NotReady node means the kubelet on that node has stopped posting heartbeats to the control plane. Kubernetes waits node-monitor-grace-period (default 40 seconds) before marking the node NotReady, then waits pod-eviction-timeout (default 5 minutes) before evicting pods to other nodes.

# See all nodes and their status
kubectl get nodes
# STATUS column shows NotReady for the unhealthy node

# Full details of what is wrong
kubectl describe node ip-10-0-1-234.ec2.internal
# Look for:
# - Conditions section: MemoryPressure, DiskPressure, PIDPressure, Ready
# - Events: kubelet stopped posting heartbeat / node not reachable

# Check kubelet logs on the node (requires SSH or SSM access)
# On EKS managed nodes via SSM:
aws ssm start-session --target i-0abc123def456789a
sudo journalctl -u kubelet -n 200 --no-pager

# Check what pods were on the problematic node
kubectl get pods --all-namespaces \
  --field-selector spec.nodeName=ip-10-0-1-234.ec2.internal

# Force delete stuck Terminating pods from NotReady node (nuclear option)
# Only use when node is truly gone and will never return
kubectl delete pod <pod-name> -n ticketflow --grace-period=0 --force

# Cordon the node (mark unschedulable but don't evict)
kubectl cordon ip-10-0-1-234.ec2.internal

# Drain the node (evict all pods gracefully, then cordon)
kubectl drain ip-10-0-1-234.ec2.internal \
  --ignore-daemonsets --delete-emptydir-data --force

Enterprise Scenario: TicketFlow — 5 Real Incidents

🏢 Incident 1 — payment-service CrashLoopBackOff (P1)

Alert: payment-service pods restarting every 30 seconds. Checkout is broken. Revenue blocked.

Observe: kubectl get pods -n ticketflow shows RESTARTS: 12 in 6 minutes. Status: CrashLoopBackOff.

Isolate: Only payment-service pods affected. ticket-service and user-service are healthy. Problem is application-scoped, not node-scoped.

Hypothesize: Recent deployment 8 minutes ago. Most likely a config change introduced a bad value.

Test: kubectl logs payment-service-7f8b9c-xr2pw -n ticketflow --previous reveals: ValueError: STRIPE_API_KEY must start with 'sk_' — got 'pk_live_...'. A deploy used the publishable key instead of the secret key.

Fix: kubectl edit secret payment-secrets -n ticketflow, correct the STRIPE_API_KEY value. kubectl rollout restart deployment/payment-service -n ticketflow. Pods come up healthy in 45 seconds.

Post-mortem action: Add a startup validation check in payment-service that validates the API key format before binding to the port.

🏢 Incident 2 — ticket-service OOMKilled (P2)

Alert: ticket-service has restarted 4 times in 2 hours. Grafana shows memory climbing to 512Mi then dropping sharply (OOM kill).

Observe: kubectl describe pod ticket-service-6c8b4d9f-p7kqm -n ticketflow shows Exit Code 137, Reason: OOMKilled.

Isolate: Memory climbs steadily, not spikes. Pattern suggests memory leak, not traffic spike.

Hypothesize: Spring Boot heap growing without bound. Hibernate L2 cache may be misconfigured (unbounded cache size).

Test: Via actuator endpoint: curl localhost:8080/actuator/metrics/jvm.memory.used. Heap grows 50 MB per hour. Cache inspection shows Hibernate second-level cache has no eviction policy. 800k ticket objects cached indefinitely.

Fix (immediate): Increase memory limit from 512Mi to 1Gi to stop OOM kills. kubectl set resources deployment/ticket-service -n ticketflow --limits=memory=1Gi.

Fix (permanent): Add Caffeine cache size limits to Spring Boot config. Redeploy with spring.cache.caffeine.spec=maximumSize=10000,expireAfterWrite=10m.

Behind the Scenes

How does CrashLoopBackOff exponential backoff actually work?

When a container crashes, the kubelet tracks consecutive failures in the pod's containerStatuses. After the first crash, it waits 10 seconds before restarting. If the container crashes again, it waits 20 seconds. Then 40s, 80s, 160s, 320s, and then it caps at 300 seconds (5 minutes). This is the full backoff sequence.

The purpose of this backoff is to protect the node from thrashing — a container that crashes immediately consuming CPU in a tight restart loop would prevent the node from running other workloads. Exponential backoff gives operators time to react and makes the failure visible (you can see the RESTARTS counter increasing) without hammering the system.

The backoff resets to 10 seconds if the container stays up for more than 10 minutes. This means if you have a service that crashes after 15 minutes due to a memory leak, the backoff restarts from 10 seconds every time, and the RESTARTS counter may stay low even though there is a recurring failure.

Key insight: A pod in CrashLoopBackOff at 300 seconds interval may show only 3–4 restarts in an hour. Do not use the restart count alone to assess severity — check the events and the time of the last restart.

Why does a Service with endpoints still return 503?

There are several scenarios where kubectl get endpoints shows pod IPs but traffic still fails:

  • Pod is Ready but not actually healthy — the readiness probe passes (maybe it only checks GET /) but the pod is throwing 500s on the actual business endpoint. The Service correctly routes to it, but the app returns an error.
  • NetworkPolicy blocking traffic — a NetworkPolicy exists that allows the Service selector to work but blocks traffic from the actual source (e.g., the ingress controller's namespace). kubectl get networkpolicy -n ticketflow and inspect each policy's ingress rules.
  • Service port mismatch — the Service's targetPort does not match the container's actual listening port. kubectl describe svc shows TargetPort: 8080/TCP but the container is listening on 3000.
  • kube-proxy not syncing — in rare cases, kube-proxy on a node may have stale iptables rules and fail to route to new endpoints. Restarting the kube-proxy DaemonSet pod on the affected node resolves this.
  • Headless Service and DNS — if using a headless Service (ClusterIP: None), DNS returns the pod IP directly. If the pod's IP changed (pod restarted) but a client has the old IP cached, it will connect to nothing. Check nslookup service-name.namespace.svc.cluster.local from inside a pod.
How does the Kubernetes scheduler decide which node to place a pod on?

The scheduler works in two phases: filtering and scoring.

Filtering eliminates nodes that cannot run the pod. The scheduler checks: Does the node have enough allocatable CPU and memory? Does the node have all required taints tolerated by the pod? Does the node match the pod's nodeSelector and nodeAffinity rules? Is the node healthy (Ready condition)? Are the required ports available? If a node fails any filter, it is eliminated.

Scoring ranks the remaining nodes. The scheduler runs a set of scoring plugins on each candidate node. Key scoring factors include: LeastAllocated (prefer nodes with more free resources to spread load), NodeAffinity (prefer nodes that match preferred affinity rules), SpreadsConstraints (honor topologySpreadConstraints to spread pods across zones), and BalancedAllocation (avoid nodes where CPU is free but memory is near-full, or vice versa).

The node with the highest aggregate score wins and the pod is bound to it. If multiple nodes tie, one is chosen arbitrarily.

Why this matters for troubleshooting: If a pod is Pending and the events say "0/5 nodes are available: 5 node(s) didn't match node affinity," it means ALL nodes failed the filtering phase — not just one. Check whether your affinity rules are overly restrictive.

Common Mistakes

Mistake 1: Restarting Pods Before Reading Logs

The instinct is to "just restart it." But kubectl rollout restart destroys the crashing container and its logs. You lose the crash evidence. The pod may restart successfully (masking the issue) and crash again in 2 hours when you have less context.

Fix: Always run kubectl logs --previous and kubectl describe pod BEFORE making any changes. Collect evidence first. Restarts come later, after diagnosis.

Mistake 2: Ignoring the Events Section in kubectl describe

Engineers read the top half of kubectl describe pod (the metadata, labels, container config) and miss the Events section at the very bottom. The Events section is the most valuable part — it shows exactly what Kubernetes tried to do, in chronological order, with specific error messages.

Fix: Scroll to the bottom of every kubectl describe output. The Events section is where the root cause usually announces itself. Alternatively, use kubectl get events --sort-by=.lastTimestamp -n ticketflow for a global event feed.

Mistake 3: Confusing Container Running State with Application Health

A pod showing Running and 1/1 in READY does NOT mean the application is working correctly. It means the container process is running and readiness probes are passing. The app could be serving 500s on every request, logging thousands of errors, or stuck in a degraded state that readiness probes don't catch.

Fix: Always cross-reference pod status with application metrics (Prometheus error rate, Grafana dashboards) and application logs. Running is a necessary but not sufficient condition for health.

Mistake 4: Setting Memory Requests = Memory Limits (Burstable vs Guaranteed QoS)

Setting requests equal to limits gives the pod the Guaranteed QoS class, meaning it will never be OOM-evicted (unless it exceeds its own limit). But if the limit is set too low, the pod gets OOMKilled by its own limit instead of being gracefully evicted. Many teams set requests=limits without understanding this trade-off.

Fix: For services with variable memory patterns, set requests lower than limits (Burstable QoS). This allows the container to burst when needed while ensuring baseline resources are reserved. For critical services where predictability matters more than efficiency, Guaranteed QoS is appropriate — but ensure the limit is actually sufficient.

Mistake 5: Rolling Back a Deployment Without Checking If the Previous Version Is Also Broken

During an incident, the reflex is "rollback!" But if the previous version also had a latent bug that was triggered by a database migration or config change that happened independently, rolling back won't help — you'll just have a different broken version.

Fix: Before rolling back, verify that the previous version was actually healthy. Check the rollout history (kubectl rollout history) and correlate with monitoring — was the error rate zero before the deploy? If yes, rollback is safe. If the error rate was already elevated, rollback is not the fix.

Architect Perspective

Architect Note

A mature troubleshooting capability is a system design concern, not just an operational concern. The best on-call engineers are fast not because they are smarter — they are fast because they designed observability into the system before it went to production.

  • Structured logging wins incidents. A log line that says {"level":"error","msg":"DB connection failed","host":"postgres-primary.ticketflow.svc","port":5432,"retry":3,"duration_ms":5002} tells you the root cause instantly. A log line that says ERROR: connection failed tells you nothing. Design logs as machine-parseable structured events with context fields, not human-readable sentences.
  • Readiness probes are incident prevention. A well-tuned readiness probe (GET /health/ready that checks DB connectivity, cache reachability, and external dependencies) prevents traffic from routing to a pod that is technically running but cannot serve requests. This converts cascading failures into isolated pod failures — a 10x reduction in blast radius.
  • Resource requests are SLO commitments. Setting correct resource requests is not about frugality — it is about guaranteeing that the scheduler places pods on nodes with the resources they actually need. Underspecified requests lead to node overcommit, OOMKills, and unpredictable latency spikes. Profile your services under peak load and set requests at P95 steady-state usage.
  • Runbooks are leverage. A 2-page runbook per critical service that documents the 5 most likely failure modes, the exact commands to diagnose each, and the fix steps multiplies the incident resolution capability of your entire team. The best runbooks are tested quarterly by having engineers follow them cold on a staging incident.
  • Post-mortems are the compounding return. The incident itself has a negative ROI. The post-mortem has a positive compounding ROI if acted on. Every post-mortem should produce at minimum: one alert tuned to detect this failure class earlier, one runbook update, and one architectural change that reduces the probability or blast radius of the next incident.

Troubleshooting Readiness Checklist

Before a production incident happens, ensure these are in place so you can debug fast when it does.

  • Readiness probes configured on all Deployments — checks DB, cache, and upstream dependencies, not just HTTP 200
  • Liveness probes configured with appropriate initialDelaySeconds (avoid premature kills during startup)
  • Memory and CPU limits set on all containers — prevents noisy-neighbor OOM kills and resource starvation
  • Structured (JSON) logging in all services — parseable by Loki, Elasticsearch, and CloudWatch Logs Insights
  • Application logs include trace IDs — correlate logs across services during distributed incidents
  • Prometheus metrics exported from all services — at minimum: error rate, latency p95, active connections
  • PodDisruptionBudgets defined for all critical Deployments — prevents full service outage during node drains
  • Cluster Autoscaler configured with appropriate min/max node counts aligned with HPA maxReplicas
  • imagePullSecrets created and attached to ServiceAccounts in every namespace that uses private images
  • Resource quotas set on all namespaces — prevents a single team's runaway Deployment from starving the whole cluster
  • kubectl access configured for all on-call engineers with appropriate RBAC (read-only for most, exec for senior SREs)
  • Runbooks exist and are linked from PagerDuty/Opsgenie alerts — not buried in Confluence
  • kubectl debug / ephemeral containers enabled on the cluster for production pod debugging without redeployment
  • Termination grace period set appropriately per service (Spring Boot needs 30s+; Go services may only need 5s)
  • Events retention configured — default 1h is too short; use Grafana Loki or a custom event logger for historical events

Interview Questions

These questions are asked in senior SRE, DevOps engineer, and platform engineer interviews. Each answer demonstrates both conceptual understanding and hands-on experience.

Q1: A pod is in CrashLoopBackOff. Walk me through your exact diagnostic process.

My process is methodical before making any changes. First, I run kubectl get pods -n <namespace> to confirm the status and check the restart count. Then I immediately run kubectl logs <pod> --previous to get the logs from the crashed container — not the current one. This is the single most important command: it shows what the process printed to stdout/stderr before it died.

Next I run kubectl describe pod <pod> and scroll to the Events section. This tells me the exit code (137 = OOMKilled, 1 = application error, 127 = command not found), any image pull failures, and liveness probe failures.

Based on what I find: if it is an exit code 1 with an application error in logs, I check ConfigMaps and Secrets for misconfigured values. If it is exit code 137, I check memory limits and investigate for memory leaks. If it is exit code 127, I check the Dockerfile CMD for a missing binary. Only after identifying the root cause do I apply a fix and verify with kubectl rollout status.

Q2: A pod shows Running but the Service returns 503. What do you check?

Running does not mean healthy. I work through a systematic checklist. First: kubectl get endpoints <service-name> -n <namespace>. If the Endpoints column shows <none>, the Service has no backend pods. I then compare the Service's selector to pod labels with kubectl describe svc and kubectl get pods --show-labels — a label mismatch is the most common cause.

If endpoints exist, I check whether the pods are passing readiness probes: kubectl get pods must show READY as 1/1 (or N/N). A pod in Running state with readiness failing will not be added to endpoints. I check the readiness probe configuration with kubectl describe pod.

If endpoints exist and pods are ready, I check the Service's targetPort against the actual container port the app is listening on — a port mismatch causes 503. Then I check NetworkPolicy for rules that might block traffic from the ingress source to the service. Finally, I port-forward directly to a pod IP to test whether the issue is in the Service layer or the application layer itself.

Q3: What is the difference between OOMKilled and Evicted, and when does each happen?

OOMKilled happens when a container exceeds its configured memory limit. The Linux kernel's OOM killer sends SIGKILL (exit code 137) to the process. This is enforced at the container level by cgroups. The pod status shows OOMKilled in the container's last state. The fix is either to increase the memory limit or fix the memory leak in the application.

Evicted happens at the node level when the node itself is under memory pressure — not when a single container exceeds its limit. When node memory falls below the eviction threshold (by default, when less than 100Mi is available on the node), the kubelet evicts pods according to their QoS class: BestEffort pods first, then Burstable pods, and finally Guaranteed pods are never evicted due to memory pressure (they may only be OOMKilled if they exceed their own limit).

In short: OOMKilled is about the container exceeding its own limit. Eviction is about the node protecting itself from memory exhaustion by removing lower-priority pods. You can distinguish them by the pod status: OOMKilled shows in kubectl describe pod under container last state; evicted pods show STATUS: Evicted in kubectl get pods with a reason in the pod status.

Q4: A pod is Pending. How do you diagnose whether it is a resource issue or an affinity issue?

The answer is always in kubectl describe pod <pod-name>, specifically the Events section. Kubernetes gives precise messages. For resource starvation: 0/5 nodes are available: 5 Insufficient memory or 3 Insufficient cpu, 2 node(s) had untolerated taint. For affinity: 0/5 nodes are available: 5 node(s) didn't match Pod's node affinity/selector.

For resource issues, I then run kubectl describe nodes | grep -A5 "Allocated resources" to see how full each node is. If all nodes are near capacity, the fix is to add more nodes or reduce the pod's resource requests. For a production cluster, I also check Cluster Autoscaler — it should add nodes automatically unless the ASG max is reached.

For affinity issues, I get the pod's affinity rules with kubectl get pod -o yaml | grep -A30 affinity and compare to actual node labels with kubectl get nodes --show-labels. The affinity rule may reference a label that no node has, or may require a zone that is at capacity. The fix is either to update the affinity rule or to add the required labels to existing nodes.

Q5: How do you debug a pod that you cannot exec into because it uses a distroless image with no shell?

Distroless images (which contain only the application binary and its dependencies) have no shell, no package manager, and often no common debugging tools. There are three approaches.

Option 1: kubectl debug with ephemeral containers (Kubernetes 1.23+). Run kubectl debug -it <pod-name> --image=busybox --target=<container-name>. This injects a temporary debug container into the existing pod that shares the same PID namespace. You get a full shell and can inspect the process, network, and filesystem of the distroless container from the debug sidecar. This is the cleanest approach and does not require redeploying the pod.

Option 2: Copy the pod spec with a debug image. Run kubectl debug -it <pod-name> --image=busybox --copy-to=debug-pod. This creates a new pod with the same spec but replaces the container image with busybox. Useful when the pod is not running at all (so ephemeral containers cannot attach).

Option 3: Use logs and metrics. If you cannot exec in, rely on structured logs (kubectl logs), Prometheus metrics, and distributed traces. A well-instrumented distroless container rarely needs exec debugging because the application emits all the diagnostic information you need externally. This is the argument for making observability a design requirement, not an afterthought.

Q6: How would you investigate a gradual memory leak in the ticket-service Spring Boot application running on Kubernetes?

A gradual memory leak manifests as steadily increasing memory usage over hours, eventually triggering OOMKilled. Investigating it requires both Kubernetes-level and application-level tools.

Step 1: Confirm the pattern. Pull the Prometheus metric container_memory_working_set_bytes{pod=~"ticket-service.*"} in Grafana and look at the 24-hour trend. A sawtooth pattern (steady growth, sharp drop at OOMKill, growth resumes) is the signature of a leak.

Step 2: Identify which memory region is leaking. Spring Boot Actuator exposes JVM metrics. Use kubectl port-forward to access /actuator/metrics/jvm.memory.used and compare heap vs off-heap. Check jvm.gc.memory.promoted — if old-gen is growing across GC cycles, that is a heap leak. If off-heap is growing (Metaspace or direct buffers), that is a different problem class.

Step 3: Trigger a heap dump. During a live pod (before it OOMKills): kubectl exec ticket-service-6c8b4d9f-p7kqm -- java -jar jcmd.jar <PID> GC.heap_dump /tmp/heapdump.hprof. Then copy it out: kubectl cp ticketflow/ticket-service-6c8b4d9f-p7kqm:/tmp/heapdump.hprof ./heapdump.hprof.

Step 4: Analyze the heap dump. Open in Eclipse Memory Analyzer (MAT). Use the "Leak Suspects" report. In my experience with Spring Boot, the most common causes are: an unbounded Hibernate L2 cache, an in-memory event queue that is not drained, or an EhCache/Caffeine cache without expiration policies.

Step 5: Immediate mitigation while the fix is developed. Set a slightly higher memory limit to reduce OOMKill frequency. Add a Kubernetes CronJob to restart the pod every 24 hours as a temporary circuit breaker. This keeps the service running while the proper fix is deployed.

Q7: A Node is NotReady. What is your runbook — step by step?

A NotReady node is a potential data loss and availability event. I move through this runbook in order:

Step 1: Assess blast radius. kubectl get nodes — is it one node or multiple? One node is likely an instance issue. Multiple nodes simultaneously suggests a network partition, AZ outage, or control plane problem. kubectl get pods --all-namespaces --field-selector spec.nodeName=<node-name> — see which pods are affected and how critical they are.

Step 2: Check node conditions. kubectl describe node <node-name> — look at the Conditions section. MemoryPressure, DiskPressure, or PIDPressure true indicates resource exhaustion. NotReady due to kubelet heartbeat timeout means the control plane lost contact with the node.

Step 3: Attempt recovery. If on EKS, check if the EC2 instance is reachable via SSM. If the instance is running but kubelet is stopped: sudo systemctl restart kubelet. Check kubelet logs: journalctl -u kubelet -n 100 for the failure reason.

Step 4: If node is unrecoverable. Cordon the node (kubectl cordon <node>) to prevent new scheduling. Drain it (kubectl drain <node> --ignore-daemonsets --delete-emptydir-data) to evict existing pods to healthy nodes. Verify all pods reschedule successfully. On EKS, terminate the instance — the Auto Scaling Group will replace it with a fresh node automatically.

Step 5: Post-mortem. Check CloudWatch metrics for the instance: CPU, memory, disk, and network. Was there a kernel panic? A disk I/O saturation event? A runaway process? Create a CloudWatch alarm for node conditions to catch future cases earlier.
