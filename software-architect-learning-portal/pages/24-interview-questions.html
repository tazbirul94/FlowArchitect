Answering Architecture Questions: The STAR Method

Interview Technique

Architecture and system design questions rarely have one correct answer. Interviewers are evaluating your reasoning process, not just your conclusion. Use the STAR structure to give answers that demonstrate senior-level thinking:

  • Situation — Set the context. What was the scale, the constraints, the team size? "TicketFlow's payment-service was processing 50K transactions/day, required PCI compliance, and had a team of 3 engineers."
  • Task — Define the specific engineering challenge. "We needed zero-downtime deployments with instant rollback capability if a payment bug appeared in production."
  • Action — Describe the technical decisions and trade-offs considered. "We evaluated blue-green vs canary. Blue-green was simpler but doubled infrastructure cost. We chose canary with 5% traffic splitting using Argo Rollouts, shifting to 100% only after 15 minutes of clean error rates."
  • Result — Quantify the outcome. "Deployment lead time dropped from 2 hours to 12 minutes. We had three rollbacks in six months — each completed in under 90 seconds with zero customer-visible errors."

Common mistake: Jumping directly to the Action without establishing Situation and Task. This makes your answer sound like a textbook definition rather than earned experience. Always anchor your answer in a concrete scenario — real or credibly hypothetical.

For trade-off questions: Never say "it depends" and stop there. That answer is worth zero points. Say "it depends on X, Y, and Z — and here is how I would evaluate each." Then evaluate them. That is a senior-level answer.

Docker — 10 Questions

JuniorQ1: What is the difference between a Docker image and a Docker container?

An image is a read-only, immutable blueprint built from a Dockerfile. It contains your application binary, its runtime, libraries, and configuration defaults stacked as layers. Images live in registries (Docker Hub, Nexus, ECR) and are pulled to machines that need them.

A container is a running instance of an image. Docker adds a thin writable layer on top when the container starts. This writable layer is discarded when the container is removed — the image underneath is unchanged.

Analogy: An image is a class definition. A container is an object instantiated from that class. One image can spawn thousands of containers simultaneously; each is isolated from the others even though they share the same image layers on disk.

TicketFlow context: The nexus.ticketflow.io:5000/payment-service:2.3.1 image is built once in CI and pushed to Nexus. Kubernetes pulls it and runs 3 replicas — 3 containers from one image, each isolated, each handling a slice of payment traffic.

JuniorQ2: What are Docker layers and why do they matter for build performance?

Each instruction in a Dockerfile (FROM, RUN, COPY, etc.) creates an immutable filesystem layer stacked on top of the previous one. Layers are content-addressed — identified by a SHA256 hash of their contents.

Why they matter: Docker caches each layer. On rebuild, Docker only re-executes instructions where the layer's inputs have changed — all subsequent layers are also invalidated. This means instruction order is a performance decision:

# SLOW — COPY source code before installing deps
# Any code change invalidates the npm install layer
FROM node:18-alpine
COPY . .
RUN npm ci
CMD ["node", "src/index.js"]

# FAST — install deps first, copy source second
# npm install is cached until package.json changes
FROM node:18-alpine
COPY package*.json ./
RUN npm ci --only=production
COPY src/ ./src/
CMD ["node", "src/index.js"]

In the fast pattern, a code change in src/ only invalidates the last COPY layer — npm ci is served from cache. On a monorepo with dozens of services like TicketFlow, this reduces CI build times from 8 minutes to under 60 seconds per service.

JuniorQ3: How does Docker achieve container isolation on Linux?

Docker uses two Linux kernel primitives — namespaces and cgroups — to create the illusion of isolation. Containers do not contain their own kernel; they share the host kernel.

Namespaces control what a container can see:

  • pid — isolated process tree; PID 1 inside the container is the app process, not systemd
  • net — own network stack, own IP address, own port space
  • mnt — own filesystem view via OverlayFS
  • uts — own hostname
  • ipc / user — isolated IPC and UID/GID mapping

Control groups (cgroups) control what a container can use: maximum CPU time, memory, and disk I/O. If a container exceeds its memory limit, the kernel OOM-kills its process — preventing a single container from taking down the host.

Important caveat: This is process-level isolation, not hardware-level isolation. A sufficiently privileged container can interact with the host kernel. For untrusted workloads, use Kata Containers or gVisor which add a kernel boundary.

JuniorQ4: What is a multi-stage Docker build and when would you use it?

A multi-stage build uses multiple FROM instructions in a single Dockerfile. Each stage is a separate environment. You copy only the artifacts you need from one stage into the next, leaving build tools behind.

# Stage 1: Build (has JDK, Maven, test dependencies)
FROM maven:3.9-eclipse-temurin-17 AS builder
WORKDIR /app
COPY pom.xml .
RUN mvn dependency:go-offline -B
COPY src/ ./src/
RUN mvn package -DskipTests

# Stage 2: Runtime (has only JRE — ~200MB vs ~600MB)
FROM eclipse-temurin:17-jre-alpine AS runtime
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
WORKDIR /app
COPY --from=builder /app/target/ticket-service.jar ./app.jar
USER appuser
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]

Benefits:

  • Smaller images: TicketFlow's ticket-service went from 620 MB (with Maven) to 185 MB (JRE only) — 70% reduction.
  • Smaller attack surface: No compiler, no build tools, no test dependencies in production.
  • Single source of truth: Build and runtime defined in one file, no separate CI build scripts.
MidQ5: What is the difference between CMD and ENTRYPOINT in a Dockerfile?

Both define what runs when a container starts — the difference is in how they interact with arguments passed via docker run.

CMD provides default arguments that are completely replaced if the user passes a command to docker run:

CMD ["node", "src/index.js"]
# docker run user-service            → runs node src/index.js
# docker run user-service bash       → runs bash (CMD replaced)

ENTRYPOINT is the fixed executable. Arguments from CMD or from docker run are appended to it:

ENTRYPOINT ["node"]
CMD ["src/index.js"]
# docker run user-service            → node src/index.js
# docker run user-service src/debug  → node src/debug (CMD replaced, ENTRYPOINT fixed)

Best practice: Use ENTRYPOINT for the main executable and CMD for default arguments. This lets operators override arguments without having to repeat the binary name. For application containers, always use the exec form (JSON array) rather than shell form — shell form runs the process as a child of /bin/sh, which means SIGTERM from docker stop is not forwarded to your app, causing slow shutdowns and potential data loss.

MidQ6: How do Docker volumes differ from bind mounts? When would you use each?

Named volumes are managed entirely by Docker. They are stored in /var/lib/docker/volumes/, have no dependency on host filesystem structure, are portable across machines, and survive container deletion. They are the correct choice for production stateful data.

Bind mounts map a specific host path into the container. They depend on the host directory existing at exactly the right path, making them non-portable. They are the correct choice for development workflows where you want live code reloading.

# docker-compose.yml — TicketFlow development
services:
  user-service:
    image: node:18-alpine
    volumes:
      # Bind mount: live reload during development
      - ./user-service/src:/app/src
    command: node --watch src/index.js

  postgres:
    image: postgres:15-alpine
    volumes:
      # Named volume: data persists across container rebuilds
      - postgres_data:/var/lib/postgresql/data
    environment:
      POSTGRES_DB: ticketflow

volumes:
  postgres_data:

Rule: If the data must survive and you care about portability, use a named volume. If you are mounting source code for hot-reload during development, use a bind mount — but never use a bind mount for database data in production; it creates implicit coupling between the container and a specific host path.

MidQ7: How would you reduce a Docker image from 900 MB to under 100 MB?

Systematic image size reduction follows a checklist:

  1. Use Alpine or distroless base images. node:18 (Debian) is 950 MB. node:18-alpine is 175 MB. gcr.io/distroless/nodejs18 is ~100 MB. Same Node runtime, radically different base OS.
  2. Multi-stage builds. Build in a full image, copy only compiled artifacts to a minimal runtime image. The build toolchain never reaches production.
  3. Combine RUN commands. Each RUN creates a layer. Chaining with && and cleaning up in the same layer keeps the size down: RUN apt-get update && apt-get install -y curl && rm -rf /var/lib/apt/lists/*
  4. Use .dockerignore. Prevents node_modules, .git, test fixtures, and dev configs from entering the build context and the image.
  5. Only copy production dependencies. For Node.js: RUN npm ci --only=production excludes devDependencies. For Python: use requirements-prod.txt without test libraries.
  6. Remove package manager caches in the same RUN layer as the install.

TicketFlow result: user-service reduced from 920 MB to 87 MB using Node 18 Alpine + multi-stage + production-only deps. This cut ECR storage costs by 65% and reduced pod startup time from 45s to 8s on cold nodes.

MidQ8: What Docker security practices are mandatory before deploying to production?

Non-negotiable security baseline for any production container:

  • Run as non-root. Add USER 1001 (or a named non-root user) in the Dockerfile before CMD/ENTRYPOINT. Root inside a container is root on the host if the container escapes.
  • Read-only root filesystem. Add --read-only or set readOnlyRootFilesystem: true in the Kubernetes security context. Prevents malware from writing to the container filesystem.
  • No secrets in images. Never ENV or COPY credentials into an image. Secrets must be injected at runtime via Kubernetes Secrets, AWS Secrets Manager, or Vault.
  • Image scanning. Integrate Trivy into CI before the image is pushed. Fail the build on CRITICAL or HIGH CVEs. TicketFlow's pipeline: trivy image --exit-code 1 --severity CRITICAL nexus.ticketflow.io:5000/payment-service:$TAG
  • Pin base image digests. FROM node:18-alpine@sha256:abc123 — not just a tag, which is mutable.
  • Drop all Linux capabilities. Containers inherit a default set of capabilities. In Kubernetes, set capabilities.drop: [ALL] and add back only what is explicitly needed.
SeniorQ9: A container restarts every 90 seconds. Walk through your debugging process.

This is a CrashLoopBackOff scenario — the most common production container failure pattern. Systematic approach:

Step 1: Establish the failure mode.

kubectl describe pod payment-service-abc123
# Look at: Last State exit code, Reason (OOMKilled vs Error)
# Exit code 137 = OOMKilled (SIGKILL from kernel)
# Exit code 1   = Application crash
# Exit code 143 = SIGTERM (graceful shutdown — but why is it restarting?)

Step 2: Read the previous container's logs (current container may have no logs if it crashes on startup):

kubectl logs payment-service-abc123 --previous
# Shows the last run's stdout/stderr before crash

Step 3: Check resource limits. OOMKilled means the container hit its memory limit. Solution: either fix the memory leak or raise the limit — but understand which it is first.

Step 4: Check liveness probe configuration. An overly aggressive liveness probe (short timeout, high sensitivity) will kill a healthy container that was momentarily slow. A startup-heavy app (JVM, large model load) needs startupProbe or a long initialDelaySeconds.

Step 5: Check readiness — is the container reaching a healthy state at all? If readiness never becomes true, traffic is never routed to it but liveness will eventually kill it.

Common root causes at TicketFlow: payment-service had a 256Mi memory limit that was fine for normal load but insufficient during reconciliation jobs at midnight. Raised to 512Mi, added a memory usage alert at 80%, added heap dump on OOM for post-mortem analysis.

SeniorQ10: Explain Docker's OverlayFS and how image layers are composed at runtime.

OverlayFS is a union filesystem — it overlays multiple directories so they appear as one. Docker uses it to implement the layered image model efficiently.

When Docker runs a container from an image with N layers, OverlayFS creates a mount with:

  • lower dirs — the N read-only image layers stacked from bottom (base image) to top (most recent Dockerfile instruction)
  • upper dir — a single writable layer for the running container
  • work dir — internal OverlayFS bookkeeping directory
  • merged dir — the unified view presented to the container

Copy-on-write (CoW): When a container modifies a file from a read-only layer, OverlayFS copies the file up to the writable upper layer first, then modifies it there. The original read-only layer is untouched. This is why multiple containers can share the same image layers on disk without interfering with each other — each only incurs storage cost for the files it actually modifies.

Performance implication: The first write to a large file from a lower layer triggers a full CoW copy to the upper layer — even if you only modified one byte. For write-heavy workloads (databases, log files, compiled outputs), always use volumes rather than the container filesystem. The CoW overhead on a 1 GB file being written to constantly is significant.

Why this matters for engineers: Understanding OverlayFS explains why docker build is fast (unchanged layers are reused), why images on disk are smaller than their apparent total size (shared base layers counted once), and why you should never write high-throughput data to the container filesystem.

Kubernetes — 15 Questions

JuniorQ11: What problem does Kubernetes solve that Docker alone does not?

Docker solves the "runs the same everywhere" problem at the level of a single container on a single machine. Kubernetes solves the cluster orchestration problem: how do you run hundreds of containers across dozens of machines reliably?

What Kubernetes adds on top of Docker:

  • Scheduling: decides which node has capacity to run a new pod, considering CPU/memory requests, node affinity, taints/tolerations.
  • Self-healing: restarts crashed containers, replaces pods on failed nodes, kills containers that fail health checks.
  • Scaling: HPA scales replica count based on CPU/memory/custom metrics. Cluster Autoscaler adds/removes nodes.
  • Service discovery & load balancing: every Service gets a stable DNS name and virtual IP. No manual registration needed.
  • Rolling deployments & rollbacks: update all replicas gradually without downtime; roll back with a single command if metrics degrade.
  • Config and secret management: ConfigMaps and Secrets decouple configuration from image, injected at runtime.

TicketFlow before Kubernetes: 6 services, each running on dedicated VMs provisioned manually. Failover was manual. Scaling required a request to the infra team. A node failure meant contacting on-call at 2 AM. After EKS migration: all of the above is automated.

JuniorQ12: What is the difference between a Pod, a Deployment, and a ReplicaSet?

Pod: the smallest deployable unit in Kubernetes. One or more tightly coupled containers sharing a network namespace and storage. Pods are ephemeral — they are not rescheduled if they die; a new Pod with a new IP is created.

ReplicaSet: ensures a specified number of identical Pod replicas are running at all times. If a Pod dies, the ReplicaSet controller creates a replacement. You rarely create ReplicaSets directly.

Deployment: manages ReplicaSets and adds rolling update and rollback capabilities. When you update a Deployment's image, it creates a new ReplicaSet with the new version and scales it up while scaling down the old one. Previous ReplicaSets are retained for rollback.

Hierarchy: Deployment → ReplicaSet → Pods

apiVersion: apps/v1
kind: Deployment
metadata:
  name: user-service
spec:
  replicas: 3                          # ReplicaSet will maintain 3 Pods
  selector:
    matchLabels:
      app: user-service
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 1
      maxSurge: 1
  template:                            # Pod template
    metadata:
      labels:
        app: user-service
    spec:
      containers:
      - name: user-service
        image: nexus.ticketflow.io:5000/user-service:3.1.2

Rule: always use Deployments, not bare Pods or ReplicaSets directly. Deployments are the standard mechanism for stateless application workloads.

JuniorQ13: What is the difference between a ClusterIP, NodePort, and LoadBalancer Service?

Kubernetes Services expose a set of Pods under a stable DNS name and virtual IP. Three exposure levels:

ClusterIP (default): virtual IP reachable only inside the cluster. Pods call each other by Service DNS name: http://ticket-service:8080. The correct type for internal microservice communication — not exposed externally.

NodePort: opens a port (30000–32767 range) on every node in the cluster. External traffic reaching any node IP on that port is forwarded to the Service. Used for development/testing. Rarely used in production — requires knowledge of node IPs, bypasses proper load balancing.

LoadBalancer: provisions a cloud load balancer (AWS ALB/NLB, GCP LB) that forwards traffic to the Service. The correct production mechanism for externally accessible services. In TicketFlow's EKS cluster, the api-gateway Service is type LoadBalancer — AWS provisions an NLB with a public DNS name. All other services are ClusterIP.

Ingress (not a Service type, but related): a single LoadBalancer in front of an Ingress controller (NGINX, AWS ALB) that routes traffic to multiple ClusterIP services based on host/path rules. More cost-efficient than one LoadBalancer per service.

JuniorQ14: What are Kubernetes resource requests and limits, and why are both required?

Requests: the minimum resources the container is guaranteed. The scheduler uses requests to decide which node can fit the pod. If a node has 2 CPU free and a pod requests 3 CPU, it will not be scheduled on that node.

Limits: the maximum resources the container can consume. If a container exceeds its memory limit, the kernel OOM-kills it. If it exceeds its CPU limit, it is throttled (not killed).

resources:
  requests:
    cpu: "250m"      # 0.25 cores guaranteed for scheduling
    memory: "256Mi"  # 256 MB guaranteed
  limits:
    cpu: "500m"      # throttled if exceeds 0.5 cores
    memory: "512Mi"  # OOMKilled if exceeds 512 MB

Why both are required:

  • Without requests: the scheduler cannot make informed placement decisions. All pods look the same and nodes get overcommitted.
  • Without limits: one misbehaving pod can consume all node resources, starving neighboring pods (the "noisy neighbour" problem). payment-service at TicketFlow had a memory leak that consumed all 16 GB of RAM on a node, taking down 8 other services before limits were set.
  • Without both: Kubernetes assigns QoS class "BestEffort" — your pod is the first evicted when the node is under memory pressure.
JuniorQ15: What is a ConfigMap and when would you use it versus a Secret?

ConfigMap: stores non-sensitive configuration as key-value pairs or file content. Environment-specific settings like feature flags, log levels, service URLs, connection pool sizes. Not encrypted at rest by default.

Secret: stores sensitive data (passwords, tokens, TLS certificates). Base64-encoded in etcd — not encrypted by default either, but can be encrypted at rest with KMS envelope encryption. Access controlled by RBAC. Never logged by kubectl get pods.

# ConfigMap — safe to store in git (non-sensitive)
apiVersion: v1
kind: ConfigMap
metadata:
  name: ticket-service-config
data:
  LOG_LEVEL: "info"
  MAX_POOL_SIZE: "20"
  NEXUS_REGISTRY: "nexus.ticketflow.io:5000"

---
# Secret — never store plaintext in git
apiVersion: v1
kind: Secret
metadata:
  name: ticket-service-secrets
type: Opaque
stringData:
  DB_PASSWORD: "from-sealed-secrets-or-external-secrets"
  JWT_SECRET: "from-vault"

Best practice: Use Sealed Secrets or External Secrets Operator to manage Kubernetes Secrets. Raw Secret manifests should never be committed to git — even base64-encoded values are trivially decoded. TicketFlow uses AWS Secrets Manager with External Secrets Operator: secrets live in AWS, ESO syncs them into Kubernetes Secrets, Argo CD never sees the plaintext values.

MidQ16: Explain the difference between liveness, readiness, and startup probes.

All three probe types use the same mechanism (HTTP GET, TCP socket, or exec command) but trigger different Kubernetes actions on failure:

Liveness probe: "Is this container alive or should it be restarted?" Failure restarts the container. Use for detecting deadlocks, infinite loops, or zombie states the app cannot recover from on its own. Never make a liveness probe check downstream dependencies — if your database is down, you do not want all your app pods restarting simultaneously.

Readiness probe: "Is this container ready to receive traffic?" Failure removes the pod from the Service's endpoint list — traffic is not sent to it. The pod stays running. Use for checking if the app has finished startup, if a circuit breaker has opened, or if the app is temporarily overloaded. This is the mechanism for zero-downtime deployments.

Startup probe: "Has this container started up successfully?" Disables liveness and readiness probes until it succeeds. Critical for slow-starting apps (JVMs loading 500 MB of classes, ML models loading, database migration runs). Without a startup probe, a JVM app taking 90 seconds to start will be killed by liveness probes before it ever becomes healthy.

# ticket-service (Spring Boot — slow JVM startup)
startupProbe:
  httpGet:
    path: /actuator/health/liveness
    port: 8080
  failureThreshold: 30    # 30 * 10s = 5 minutes max startup time
  periodSeconds: 10
livenessProbe:
  httpGet:
    path: /actuator/health/liveness
    port: 8080
  periodSeconds: 30
readinessProbe:
  httpGet:
    path: /actuator/health/readiness
    port: 8080
  periodSeconds: 10
MidQ17: What is a StatefulSet and when is it required over a Deployment?

StatefulSets are for workloads where each replica needs a stable, unique identity across restarts — typically databases and distributed systems that use peer discovery by hostname.

What StatefulSets provide that Deployments do not:

  • Stable network identity: pods are named pod-0, pod-1, pod-2 — predictable, not random hashes. DNS: redis-0.redis-headless.default.svc.cluster.local
  • Ordered startup and shutdown: pod-0 starts first and must be ready before pod-1 starts. Shutdown is reverse order. Critical for primary-replica databases.
  • Persistent Volume per replica: each pod gets its own PVC. pod-0's data does not move to pod-1 on restart. The PVC is bound to pod-0 permanently.

Use StatefulSets for: PostgreSQL, MySQL, Cassandra, Elasticsearch, Redis Sentinel/Cluster, Kafka, ZooKeeper.

Do not use StatefulSets for: stateless web servers, APIs, or batch workers — even if they occasionally write to disk. Use Deployments with volumes for those.

TicketFlow: PostgreSQL for ticket-service runs as a 3-replica StatefulSet (1 primary + 2 read replicas). The primary is always postgres-0. Application connection strings point to postgres-0 for writes and the headless service for reads.

MidQ18: How does Horizontal Pod Autoscaling work and what are its limitations?

HPA watches a metric (CPU utilization, memory, or a custom metric) and adjusts the replica count of a Deployment or StatefulSet to maintain a target value.

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: order-service-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: order-service
  minReplicas: 2
  maxReplicas: 20
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300  # Wait 5 min before scaling down

How it works: The HPA controller polls metrics every 15 seconds. It calculates desired replicas as: ceil(current_replicas * current_metric / target_metric). Scale-up is immediate; scale-down has a stabilization window to prevent flapping.

Limitations:

  • Latency: 60–90 seconds from metric spike to new pods being ready (HPA poll → schedule → pull image → start → pass readiness probe). For bursty traffic (flash sales, events), this is too slow. Mitigation: pre-scale before known events, use KEDA for event-driven scaling.
  • Node capacity: HPA adds pods but if no node has capacity, pods stay Pending. Requires Cluster Autoscaler to add nodes — another 2–5 minutes. Design for this lag.
  • CPU is a lagging indicator: CPU spikes after the problem starts. Custom metrics (request queue depth, active connections) are leading indicators. TicketFlow uses a custom metric from Prometheus: request queue depth on order-service triggers scale-out before CPU saturates.
MidQ19: What is a Kubernetes Namespace and how does it provide multi-tenancy?

A Namespace is a virtual cluster within a physical Kubernetes cluster. Resources (Pods, Services, Deployments, ConfigMaps) in one Namespace are isolated from resources in another Namespace by name. Cross-namespace communication is possible but requires full DNS names.

What Namespaces provide:

  • Name scoping: two teams can each have a Service named api in different Namespaces without collision.
  • RBAC scope: a team can have full admin rights to their Namespace without touching other Namespaces.
  • ResourceQuota: limit total CPU, memory, and object count per Namespace.
  • NetworkPolicy scope: default-deny all ingress except from pods in the same Namespace.

What Namespaces do NOT provide: node isolation, kernel isolation, or security boundaries sufficient for untrusted tenants. Two pods in different Namespaces on the same node share the same kernel. For true multi-tenancy with untrusted tenants, use separate clusters or virtual clusters (vCluster).

TicketFlow Namespace strategy: production, staging, development, monitoring, argocd. Each team deploys to their environment Namespace. The monitoring Namespace runs Prometheus/Grafana/Loki with scrape permissions cluster-wide. ResourceQuotas prevent staging workloads from consuming production-level resources.

MidQ20: How does Kubernetes DNS work for service discovery?

CoreDNS runs as a Deployment in the kube-system namespace and serves as the cluster-wide DNS resolver. Every Pod's /etc/resolv.conf is configured to use the CoreDNS service IP.

DNS name format for Services:

# Full DNS name
<service-name>.<namespace>.svc.cluster.local

# Examples — from any pod in the cluster:
http://ticket-service.production.svc.cluster.local:8080
http://payment-service.production.svc.cluster.local:8080

# Within the same namespace — short name works
http://ticket-service:8080      # resolves to ticket-service.production.svc.cluster.local

# Cross-namespace — must use full name or at least namespace
http://ticket-service.staging:8080

Headless Services (ClusterIP: None) are used for StatefulSets. Instead of a single virtual IP, DNS returns the A records for every individual pod IP. This allows direct pod addressing: postgres-0.postgres-headless.production.svc.cluster.local.

Debug DNS failures:

kubectl run debug --image=busybox --rm -it -- nslookup ticket-service.production
kubectl run debug --image=busybox --rm -it -- wget -qO- http://ticket-service.production:8080/health
MidQ21: What is a PersistentVolume and PersistentVolumeClaim? How does dynamic provisioning work?

PersistentVolume (PV): a piece of storage in the cluster provisioned by an admin or dynamically by a StorageClass. It has a lifecycle independent of any Pod — it exists until explicitly deleted.

PersistentVolumeClaim (PVC): a request for storage by a Pod. Specifies size, access mode (ReadWriteOnce, ReadWriteMany), and optionally a StorageClass. Kubernetes binds a PVC to a matching PV.

Dynamic provisioning: eliminates pre-provisioning PVs manually. A StorageClass defines a provisioner (AWS EBS, GCP PD, Azure Disk). When a PVC is created with that StorageClass, the provisioner automatically creates the underlying storage and a PV, then binds them.

# StorageClass — defined once by the platform team
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: gp3-encrypted
provisioner: ebs.csi.aws.com
parameters:
  type: gp3
  encrypted: "true"
reclaimPolicy: Retain       # Keep the EBS volume if PVC is deleted

---
# PVC — created per StatefulSet replica automatically
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: postgres-data-postgres-0
spec:
  accessModes: [ReadWriteOnce]
  storageClassName: gp3-encrypted
  resources:
    requests:
      storage: 100Gi

SeniorQ23: Explain PodDisruptionBudget and why it is critical for production clusters.

A PodDisruptionBudget (PDB) sets constraints on how many pods of an application can be voluntarily disrupted simultaneously. It protects applications during node drains (cluster upgrades, spot instance terminations, manual maintenance).

Why it matters: without a PDB, a Kubernetes node drain can evict all replicas of a Deployment simultaneously — causing a complete service outage. This happens during every EKS cluster version upgrade and every spot instance interruption.

apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: payment-service-pdb
spec:
  minAvailable: 2       # At least 2 replicas must be available at all times
  # OR: maxUnavailable: 1  # At most 1 replica can be disrupted
  selector:
    matchLabels:
      app: payment-service

Rules for setting PDB values:

  • For a 3-replica Deployment: minAvailable: 2 (allows 1 disruption at a time — prevents total outage)
  • For a 2-replica Deployment: minAvailable: 1 (allows draining one node while keeping one replica running)
  • Never set minAvailable equal to replicas — the drain will block forever, preventing cluster upgrades

TicketFlow incident: AWS EKS upgrade to 1.28 drained all nodes in the payment-service node group in parallel. No PDB was configured. All 3 payment-service replicas were evicted simultaneously — 4 minutes of payment downtime. After the incident: PDB with minAvailable: 2 added to all production Deployments as a mandatory Helm chart requirement enforced by OPA policy.

SeniorQ24: What is the Kubernetes control plane and what happens when it goes down?

The control plane is the brain of the cluster — it manages desired state, scheduling, and the API surface. Components:

  • kube-apiserver: the single entry point for all API operations. kubectl, controllers, and nodes all communicate with it. Stateless — can run multiple replicas.
  • etcd: distributed key-value store holding all cluster state (pod specs, node info, secrets, configmaps). The source of truth. Requires quorum (2N+1 nodes).
  • kube-scheduler: watches for unscheduled pods and assigns them to nodes based on resource availability, affinity rules, and taints.
  • kube-controller-manager: runs the control loops — ReplicaSet controller, Deployment controller, endpoint controller, etc.
  • cloud-controller-manager: integrates with cloud APIs (AWS, GCP, Azure) for load balancer provisioning, node lifecycle, storage provisioning.

What happens when the control plane is down:

  • Running workloads continue running. Pods already scheduled on nodes keep running. The kubelet on each node manages its local containers independently of the API server.
  • No new scheduling. New pods cannot be placed. Failed pods are not replaced because the controllers are down.
  • No configuration changes. kubectl commands fail. Argo CD cannot sync. No deployments, no ConfigMap updates, no Secret rotations.
  • Load balancers and Services still work — kube-proxy's iptables rules remain intact on nodes.

Why EKS/GKE/AKS: managed Kubernetes services run the control plane for you with multi-AZ HA. etcd is replicated across 3 AZs. The API server has multiple replicas behind a load balancer. Control plane HA is solved by the cloud provider. At TicketFlow on EKS, zero control plane incidents in 18 months.

SeniorQ25: How would you design pod scheduling for a multi-tenant cluster with strict workload isolation?

Multi-tenant isolation in Kubernetes requires layering multiple mechanisms — no single feature provides full isolation.

1. Namespace-based separation — one Namespace per tenant. RBAC prevents cross-namespace access. ResourceQuotas enforce per-tenant resource budgets.

2. Node groups with taints and tolerations — dedicate node groups to specific tenants or workload classes. Taint the nodes; only pods with matching tolerations are scheduled there:

# Node taint (set on the node group)
kubectl taint nodes node-group-payment dedicated=payment:NoSchedule

# Pod toleration (in the payment-service Deployment)
tolerations:
- key: "dedicated"
  operator: "Equal"
  value: "payment"
  effect: "NoSchedule"
nodeSelector:
  node-group: payment

3. Pod anti-affinity — spread replicas across nodes/AZs so a single node failure does not take down the service:

affinity:
  podAntiAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
    - labelSelector:
        matchLabels:
          app: payment-service
      topologyKey: kubernetes.io/hostname

4. NetworkPolicies — default-deny all ingress; explicitly allow only required service-to-service communication.

5. OPA/Gatekeeper policies — enforce that all pods have resource limits, non-root user, no privileged containers, and required labels.

CI/CD + GitOps — 8 Questions

JuniorQ26: What is the difference between Continuous Integration, Continuous Delivery, and Continuous Deployment?

Continuous Integration (CI): every code commit triggers an automated pipeline that builds the code, runs unit tests, runs integration tests, and produces a deployable artifact. The goal: detect integration bugs within minutes of a merge, not weeks later. CI lives in the developer loop — fast feedback is everything. TicketFlow CI: GitHub Actions runs on every PR, builds the Docker image, runs tests, scans with Trivy, pushes to Nexus.

Continuous Delivery (CD): the deployable artifact is automatically promoted through environments (dev → staging) and is always in a state ready to deploy to production. The production deployment itself requires a human approval gate. The pipeline handles the mechanics; a human makes the business decision. Used when production deployments have regulatory, business, or communication requirements.

Continuous Deployment: production deployment is also automated — every green commit reaches production without human approval. Requires very high automated test coverage and confidence in rollback mechanisms. Suitable for companies with mature engineering cultures and feature-flag-driven releases. Netflix, Facebook, and Amazon deploy hundreds of times per day this way.

TicketFlow model: CI + Continuous Delivery. Every merge to main auto-deploys to staging via Argo CD. Production deployment requires team lead approval in GitHub Actions — the actual deploy is then fully automated via Argo CD sync.

JuniorQ27: What is GitOps and how does it differ from traditional CI/CD?

GitOps is an operational model where the Git repository is the single source of truth for both application code and infrastructure/deployment configuration. A GitOps operator (Argo CD, Flux) continuously reconciles the cluster state with the desired state declared in Git.

Traditional CI/CD: the pipeline has push-access to the cluster. A Jenkins job or GitHub Actions workflow runs kubectl apply or helm upgrade directly against the cluster. The pipeline is the agent of change.

GitOps: the pipeline only pushes to Git (updating the image tag in a Helm values file). The cluster's GitOps operator detects the Git change and pulls the new state into the cluster. The cluster is the agent of change, pulling from Git — not being pushed to.

Key benefits of GitOps:

  • Auditability: every production change is a git commit with author, timestamp, and PR review. No undocumented kubectl commands.
  • Drift detection: if someone manually patches a Deployment via kubectl, Argo CD detects the divergence from Git state and can auto-revert or alert.
  • Rollback: git revert is a production rollback. No special runbook needed.
  • Credential security: the cluster does not need push credentials to the pipeline. Argo CD only needs read access to the Git repo.

MidQ29: What is Argo CD's sync strategy and what is the difference between automated and manual sync?

Argo CD continuously compares the live cluster state with the desired state in Git. When they diverge, the application is "OutOfSync." A sync operation reconciles the cluster to match Git.

Manual sync (default): Argo CD detects drift and shows the application as OutOfSync, but waits for a human to click "Sync" in the UI or run argocd app sync. Appropriate for production environments where human review is required before applying changes.

Automated sync: Argo CD automatically applies Git changes to the cluster without human intervention. Combined with auto-prune (deletes resources removed from Git) and self-heal (reverts manual kubectl changes). Appropriate for staging/development environments or very mature production pipelines with excellent test coverage.

apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: payment-service-production
spec:
  syncPolicy:
    automated:
      prune: true       # Delete resources removed from Git
      selfHeal: true    # Revert manual kubectl changes
    syncOptions:
    - CreateNamespace=true
    retry:
      limit: 3
      backoff:
        duration: 5s
        maxDuration: 3m

TicketFlow strategy: staging uses automated sync with selfHeal. Production uses manual sync for the final approval step, but the sync itself is triggered by the CI pipeline via the Argo CD API after a team-lead approval in GitHub Actions — fully automated mechanics, human-controlled trigger.

MidQ30: How do you handle secrets in a GitOps workflow where everything is in Git?

Raw Kubernetes Secrets cannot be committed to Git — even base64-encoded values are trivially decoded. Two production-grade patterns:

Option 1: Sealed Secrets (Bitnami)

A controller running in the cluster holds a private key. You encrypt secrets client-side using the corresponding public key with kubeseal. The resulting SealedSecret CRD is safe to commit to Git — only the in-cluster controller can decrypt it:

kubectl create secret generic db-password \
  --from-literal=password=mysecret \
  --dry-run=client -o yaml | \
  kubeseal --controller-name=sealed-secrets --format yaml \
  > payment-service-db-sealed.yaml
# This file is safe to commit to git

Option 2: External Secrets Operator (ESO) — preferred for cloud environments:

Secrets live in AWS Secrets Manager / HashiCorp Vault. ESO syncs them into Kubernetes Secrets on a schedule. Git only contains an ExternalSecret CRD specifying which secret to fetch and where to put it — no secret values ever touch Git:

apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: payment-db-secret
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: aws-secrets-manager
    kind: ClusterSecretStore
  target:
    name: payment-db-secret   # creates this Kubernetes Secret
  data:
  - secretKey: DB_PASSWORD
    remoteRef:
      key: ticketflow/production/payment-service
      property: db_password

TicketFlow uses ESO with AWS Secrets Manager. Secret rotation in AWS Secrets Manager automatically propagates to Kubernetes within 1 hour. Zero secrets in Git. Argo CD manages the ExternalSecret manifests (safe to commit) but never sees the secret values.

MidQ31: What is image promotion and why is rebuilding for each environment wrong?

Image promotion is the practice of using the exact same Docker image (identified by immutable SHA digest) across all environments — dev, staging, production. The image is built once in CI and promoted through environments by updating the deployment configuration.

Why rebuilding is wrong: rebuilding the same source code for staging and again for production produces two different artifacts. Even with identical source code, a rebuild can produce a different image due to:

  • Updated base image (new security patches changed behavior)
  • Updated transitive dependencies (npm/pip resolved newer versions)
  • Different compiler optimization flags or build environment
  • Time-sensitive operations during build (fetching a config from a remote)

When staging passes tests and production fails, you cannot determine if the failure is due to environment differences or artifact differences. With promotion, if it passed in staging, you have high confidence it will pass in production — it is the identical binary.

TicketFlow promotion flow:

# CI builds once — SHA becomes the immutable identifier
IMAGE=nexus.ticketflow.io:5000/order-service:abc1234def5

# Promotion: just update the values file for each environment
# staging/values.yaml  → image.tag: abc1234def5
# production/values.yaml → image.tag: abc1234def5 (after staging validation)

# Verify same digest before production promotion
docker inspect --format='{{index .RepoDigests 0}}' $IMAGE

Architecture + System Design — 10 Questions

JuniorQ34: What is the difference between monolithic and microservices architecture?

Monolith: all application functionality is deployed as a single unit. One codebase, one deployable artifact, one process. The user-service, ticket-service, and payment-service are all one application. Simple to develop initially; every change requires deploying the whole application.

Microservices: functionality is split into independent services, each deployed separately, each owning its own data store, communicating over APIs (HTTP/REST or message queues). Each service can be deployed, scaled, and developed independently.

Monolith advantages: simpler local development, no network latency between components, simpler transactions (in-process), easier to debug (one log stream, one trace). Correct choice for early-stage products and small teams.

Microservices advantages: independent deployability (deploy payment-service without affecting user-service), independent scaling (scale order-service during flash sales without scaling notification-service), polyglot technology choices, fault isolation (payment-service outage does not crash ticket-service).

Microservices costs: distributed systems complexity (network failures, eventual consistency, distributed transactions), operational overhead (6 services × 3 replicas = 18 pods to manage), observability complexity (tracing a request across 6 services requires distributed tracing).

Rule: start with a well-structured monolith. Extract services when you have identified stable boundaries with different scaling or deployment needs — not because microservices are fashionable.

JuniorQ35: What is an API gateway and what problems does it solve in a microservices architecture?

An API gateway is a single entry point for all client requests to a microservices backend. Clients talk to one address; the gateway routes to the correct service and handles cross-cutting concerns.

Problems it solves:

  • Client complexity: without a gateway, a mobile app must know the addresses of all 6 services. With a gateway, it knows one URL. Services can be renamed, moved, or split without changing clients.
  • Cross-cutting concerns in one place: authentication (JWT validation), rate limiting, SSL termination, request logging, CORS headers — implemented once in the gateway, not duplicated in every service.
  • Protocol translation: gateway accepts REST from mobile clients and translates to gRPC internally for service-to-service calls.
  • Request aggregation: a mobile "home screen" request might need data from user-service, ticket-service, and notification-service. The gateway can fan out and aggregate in one network hop from the client's perspective.

TicketFlow: NGINX api-gateway handles: JWT validation (rejects unauthenticated requests before they reach services), rate limiting (100 req/s per user for ticket purchases), SSL termination, and routing — /api/tickets/* → ticket-service, /api/payments/* → payment-service, /api/users/* → user-service.

JuniorQ36: Explain CAP theorem in plain terms and give an example of a real trade-off.

CAP theorem states that a distributed system can guarantee at most two of these three properties simultaneously:

  • Consistency (C): every read returns the most recent write or an error. All nodes see the same data at the same time.
  • Availability (A): every request receives a response (not necessarily the latest data). The system never returns an error due to unavailability.
  • Partition tolerance (P): the system continues operating even if network messages between nodes are lost or delayed.

In practice, network partitions happen — you cannot design around them. So the real choice is C vs A during a partition.

CP systems (choose consistency over availability): during a partition, refuse requests rather than return stale data. Examples: ZooKeeper, etcd, Consul, MongoDB (with write concern majority). Use when stale reads are unacceptable — financial balances, inventory counts, distributed locks.

AP systems (choose availability over consistency): during a partition, continue serving requests but may return stale data. Examples: Cassandra, DynamoDB (eventually consistent reads), CouchDB. Use when stale reads are acceptable — user profile data, shopping cart (eventual consistency is fine, the cart reconciles), DNS.

TicketFlow trade-off: ticket inventory uses a CP datastore (PostgreSQL with strong consistency). We would rather return an error during a partition than oversell tickets. User profile data (last login, preferences) uses an AP cache (DynamoDB) — showing a 30-second-old profile is acceptable; an error is not.

MidQ37: What are the patterns for handling distributed transactions in microservices?

Microservices own their own databases — there is no shared database, so traditional ACID transactions across services are impossible. Two main patterns:

Saga Pattern: a long-lived transaction decomposed into a sequence of local transactions, each publishing an event or message. Two implementations:

  • Choreography saga: each service listens for events and reacts. No central coordinator. Simple for small flows; hard to debug and trace for complex ones.
  • Orchestration saga: a central saga orchestrator tells each service what to do and handles failures. Explicit compensation transactions undo completed steps when a later step fails.
TicketFlow — Create Order Orchestration Saga:

1. order-service creates order (status: PENDING)
2. Orchestrator → payment-service: charge card
   - Success → continue
   - Failure → compensate: cancel order in order-service
3. Orchestrator → ticket-service: reserve ticket
   - Success → continue
   - Failure → compensate: refund payment, cancel order
4. Orchestrator → notification-service: send confirmation
5. order-service updates order (status: CONFIRMED)

Transactional Outbox Pattern: solves the dual-write problem (write to DB + publish to message queue atomically). Write the event to an outbox table in the same DB transaction as the business data. A separate process tails the outbox table and publishes events to Kafka. Ensures exactly-once event publication even if the service crashes between the write and the publish.

Rule: avoid distributed transactions where possible by designing service boundaries around aggregate roots. If two things must always change atomically, they may belong in the same service.

MidQ38: What is the difference between horizontal and vertical scaling? When would you use each?

Vertical scaling (scale up): add more resources to the existing instance — more CPU cores, more RAM, faster storage. Simple — no application changes required. Limited by the largest available hardware. Creates a single point of failure.

Horizontal scaling (scale out): add more instances of the application. Distribute load across multiple machines. Requires the application to be stateless (no local state between requests) or to externalize state (sessions in Redis, uploads in S3). No hardware upper bound — add as many instances as needed.

When to use vertical: stateful systems that are hard to shard (legacy databases, single-threaded apps), early-stage products where horizontal complexity is premature, or as a short-term fix while engineering horizontal scalability.

When to use horizontal: stateless web services, microservices, API servers, queue consumers — anything where you can run multiple identical copies. This is the default for Kubernetes workloads.

TicketFlow approach:

  • user-service, ticket-service, payment-service, order-service: horizontal — HPA scales 2 to 20 replicas based on CPU.
  • PostgreSQL: vertical for the primary (largest RDS instance class), horizontal for read replicas.
  • Redis: vertical — single-threaded architecture means horizontal sharding only helps for keyspace partitioning, not throughput per key.
MidQ39: How would you design a system to handle 100x traffic spikes (e.g., ticket sales for a major event)?

This is a well-known TicketFlow challenge — a concert announcement drops at 10 AM and 500K users simultaneously try to buy tickets. The system must handle this without overselling, without going down, and ideally without 10x normal infrastructure cost during quiet periods.

Design layers:

  1. CDN + Edge caching: static assets (event pages, images) served from CloudFront. Eliminates 70% of traffic before it hits the origin.
  2. Virtual waiting room: at sale start, new requests enter a queue. Users see their position. Queue issues tokens to drain the backend at a controlled rate. Prevents the backend from ever seeing 500K concurrent requests — it sees a steady 10K/min for 50 minutes.
  3. Pre-scale before the event: 30 minutes before sale start, scale order-service and ticket-service to maximum replicas. Cluster Autoscaler has 5-minute lag — pre-warm eliminates that risk.
  4. Inventory as a distributed counter: ticket inventory held in Redis with atomic DECR. One DB write per confirmed purchase — Redis absorbs the concurrency.
  5. Async payment: accept order synchronously, process payment asynchronously via a Kafka queue. User gets instant "order reserved" confirmation; payment processes within seconds. Decouples the slow payment path from the critical path.
  6. Circuit breaker on payment-service: if payment processor is slow, open the circuit and queue requests rather than letting latency pile up and cascade.

Result at TicketFlow: 50K tickets for a stadium event sold in 3 minutes. Peak load: 80K req/s to the waiting room, 3K req/s to order-service. Zero oversells. Zero downtime. Infrastructure cost: 4× normal for 20 minutes.

MidQ40: What is idempotency and why is it critical in distributed systems?

An operation is idempotent if performing it multiple times produces the same result as performing it once. GET /tickets/123 is naturally idempotent — calling it 100 times returns the same ticket. POST /orders is not — calling it 100 times creates 100 orders.

Why distributed systems require idempotency: in any distributed system, requests may be retried due to network failures, timeouts, or consumer restarts. Without idempotency, a retried payment creates a duplicate charge. A retried order creates a duplicate booking.

Implementation: idempotency keys

# Client generates a unique key per logical operation
# Sends it with every attempt of that operation
POST /payments
Idempotency-Key: order-abc123-payment-attempt-1
Content-Type: application/json
{"amount": 150.00, "card": "tok_visa"}

# payment-service stores the result keyed by idempotency key
# On retry: returns the stored result immediately without re-processing
# Prevents duplicate charges even if the client retries 5 times

TicketFlow implementation: payment-service uses a Redis cache keyed by idempotency_key with 24-hour TTL. On first call: process payment, store result in Redis. On subsequent calls with same key: return cached result, skip payment processor. This pattern prevented 3 duplicate charges in the first month after a Kafka consumer restart bug caused message redelivery.

SeniorQ41: Design the data architecture for a microservices system — how do you handle the database-per-service pattern in practice?

Database-per-service means each service owns its data and no other service can query it directly. This enables independent deployment and technology choices but creates challenges for queries that span service boundaries.

TicketFlow data ownership:

  • user-service → PostgreSQL (users, profiles, authentication)
  • ticket-service → PostgreSQL + Redis (ticket inventory, reservations, Redis for hot inventory counts)
  • payment-service → PostgreSQL (transactions, audit log — isolated for PCI compliance)
  • order-service → PostgreSQL (orders, order items, status history)
  • notification-service → DynamoDB (notification templates, delivery logs — high write volume, no relational queries)

Cross-service data challenges and patterns:

Challenge 1: Reporting queries that join across services. Solution: CQRS + an event-sourced read model. Each service publishes domain events to Kafka. A reporting service consumes all events and maintains a denormalized read model (e.g., in Elasticsearch) optimized for the report queries. The reporting service has eventual consistency — fine for analytics, not for transactional queries.

Challenge 2: Referential integrity across services. order-service stores user_id and ticket_id as foreign-key-style references, but there is no database-level foreign key constraint across services. Solution: validate references at write time via synchronous API call, and handle eventual deletion gracefully (tombstone records, soft deletes, explicit "account deleted" event handling).

Challenge 3: Shared lookup data (countries, currencies, event categories). Solution: either replicate via events (each service maintains its own copy), or extract a dedicated read-only reference data service with aggressive caching.

Rule: resist the temptation to share a database between services for "convenience." The short-term convenience creates long-term coupling — you cannot upgrade service A's schema without coordinating with service B, defeating the purpose of microservices.

SeniorQ42: How do you define and meet SLOs? Walk through setting one for a ticketing API.

SLO (Service Level Objective) is a target value for a service reliability metric. It is the internal commitment that drives engineering decisions. SLA (Service Level Agreement) is the external, contractual commitment with customers — typically more lenient than the SLO.

SLO definition process for TicketFlow's ticket purchase API:

Step 1: Define the SLI (Service Level Indicator) — the measurable signal:

  • Availability: successful_requests / total_requests (5xx = failure)
  • Latency: P99 response time for POST /orders

Step 2: Set the SLO target based on user impact analysis:

  • Availability SLO: 99.9% over 30 days (allows 43.8 min downtime/month)
  • Latency SLO: P99 < 500ms over 30 days

Step 3: Calculate error budget: 0.1% of requests can fail. At 100K req/day, that is 100 failing requests/day. When the budget is consumed, reliability work takes priority over feature development.

# Prometheus alert — SLO burn rate alert (Alertmanager)
alert: HighErrorRateBurndown
expr: |
  (
    sum(rate(http_requests_total{job="order-service",status=~"5.."}[1h]))
    /
    sum(rate(http_requests_total{job="order-service"}[1h]))
  ) > 0.001 * 14.4  # 14.4x burn rate = budget exhausted in 2 days
for: 5m
labels:
  severity: critical
annotations:
  summary: "order-service burning error budget at 14x rate"

Step 4: Make engineering decisions from the SLO: if P99 is at 490ms and trending toward the 500ms SLO, that drives a performance improvement sprint — not because someone is complaining, but because the data shows future SLO breach. SLOs transform reliability from subjective ("feels slow") to objective ("burning budget at 3x rate").

SeniorQ43: How would you approach migrating a monolith to microservices without downtime?

The strangler fig pattern is the standard approach — incrementally replace the monolith by routing traffic for specific capabilities to new services, until the monolith handles nothing and can be decommissioned.

Migration phases for TicketFlow:

Phase 1: Identify seams. Find capabilities with clean data and API boundaries. Start with the simplest, most isolated one. notification-service was first — it had no reads from the monolith's DB, only received events.

Phase 2: Build the new service in parallel. New notification-service running in Kubernetes, consuming events from the monolith via Kafka. Zero traffic initially.

Phase 3: Shadow mode. Both the monolith and new service receive all events. Compare outputs. Fix divergences in the new service until they match.

Phase 4: Traffic migration via API gateway. Route 5% of notification traffic to new service. Monitor. Increase to 50%, then 100%. The monolith's notification code still runs but its traffic goes to zero. Rollback = change a single NGINX config line.

Phase 5: Decommission. Delete the notification code from the monolith. Migrate the data. Remove the NGINX routing fallback.

Rules for safe migration:

  • Never cut over a DB until after the code migration is stable
  • Dual-write patterns for data migration (write to both old and new DB simultaneously, validate, then switch reads)
  • Feature flags to roll back in seconds without a deployment
  • Never try to split multiple services simultaneously — one at a time

JuniorQ44: What are the three pillars of observability? How do they differ?

The three pillars of observability are metrics, logs, and traces. They answer different questions about your system:

Metrics: numerical measurements aggregated over time. Answer: "What is happening?" CPU at 85%, error rate at 0.3%, P99 latency 340ms, queue depth 1200. Stored efficiently as time series (Prometheus). Low cardinality, queryable. Drive dashboards and alerts. Can tell you something is wrong but not why.

Logs: discrete events with context. Answer: "What happened?" "2024-01-15 10:23:41 ERROR payment-service — Stripe timeout after 3s for order abc123 user 456." High cardinality, full detail. Expensive to store at scale. Drive investigation after an alert fires. TicketFlow: Loki with Grafana, structured JSON logs from all services.

Traces: the path of a single request across multiple services. Answer: "Where did the time go?" A 2-second order creation: api-gateway 5ms → order-service 1800ms (DB query 1750ms!) → notification-service 200ms. Distributed tracing with correlation IDs. TicketFlow uses OpenTelemetry + Jaeger/Tempo. Essential for diagnosing latency in microservices — without traces, a slow endpoint could be any of 6 services.

Effective observability uses all three together: a metric alert fires → you query logs for the time window → a log shows a specific request ID → you look up the trace for that request ID to pinpoint the slow span.

MidQ45: How would you debug a latency spike in production using the observability stack?

Scenario: P99 latency for POST /orders at TicketFlow spiked from 200ms to 3200ms at 14:30. An alert fired. Structured debugging:

Step 1: Metrics — establish scope. Open Grafana. Check which service's latency spiked. Was it order-service, or did the latency originate in a downstream dependency? Check CPU, memory, and active connections for order-service during the window.

Step 2: Correlate with infrastructure events. Did a deployment happen at 14:30? Did HPA scale down replicas? Did a cronjob run at that time? Check the Kubernetes events panel in Grafana.

Step 3: Logs — find failing requests. In Loki:

{app="order-service"} |= "duration_ms" | json | duration_ms > 1000
# Find slow requests — each log line should have a trace_id

Step 4: Traces — find the slow span. Take the trace_id from a slow log entry, open Jaeger/Tempo. The flame graph shows: order-service spent 2900ms in a single database query. The query was a SELECT * FROM orders WHERE user_id = ? — missing an index.

Step 5: Fix and verify. CREATE INDEX CONCURRENTLY idx_orders_user_id ON orders(user_id); — CONCURRENTLY means no table lock, zero downtime. P99 returns to 180ms within 2 minutes of index creation. Post-mortem: add slow query logging threshold of 100ms; add the query to the regression test suite.

MidQ46: What is Kubernetes RBAC and how would you design it for a team of 20 engineers?

RBAC (Role-Based Access Control) controls who can perform which API operations on which Kubernetes resources. Components: Role or ClusterRole (defines permissions), ServiceAccount/User/Group (the subject), and RoleBinding or ClusterRoleBinding (binds role to subject).

RBAC design for TicketFlow — 20 engineers across 3 teams:

# Namespace-scoped developer role
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: developer
  namespace: production
rules:
- apiGroups: ["apps"]
  resources: ["deployments"]
  verbs: ["get", "list", "watch"]         # Read: allowed
- apiGroups: ["apps"]
  resources: ["deployments/scale"]
  verbs: ["update"]                        # Manual scale: allowed
- apiGroups: [""]
  resources: ["pods", "pods/log", "pods/exec"]
  verbs: ["get", "list", "watch", "create"] # Debug: allowed
- apiGroups: [""]
  resources: ["secrets"]
  verbs: []                                # Secrets: forbidden

Role hierarchy:

  • developer: read + logs + exec in team namespace. Cannot modify deployments or read secrets.
  • senior-developer: adds ability to patch deployments, restart pods.
  • platform-engineer: full access to development/staging namespaces; read-only to production.
  • sre-oncall: full access to production during incident (time-limited via JIT access with Teleport or AWS SSO).
  • ci-service-account: minimal — only allowed to update image tags on specific Deployments. No delete, no secret access.

Key principle: no human has permanent write access to production. Access is granted just-in-time during incidents and revoked automatically. All access is logged via Kubernetes audit logs shipped to CloudWatch.

SeniorQ47: How would you implement zero-trust networking in a Kubernetes cluster?

Zero trust means: no implicit trust based on network location. Every service-to-service communication is authenticated, authorized, and encrypted — even within the cluster.

Layer 1: NetworkPolicies — default deny all, explicit allow.

# Default deny all ingress in production namespace
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
  namespace: production
spec:
  podSelector: {}    # applies to all pods
  policyTypes: [Ingress, Egress]

---
# Explicitly allow: only api-gateway can reach ticket-service
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: ticket-service-ingress
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: ticket-service
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: api-gateway
    ports:
    - port: 8080

Layer 2: mTLS via Service Mesh (Istio / Linkerd). Even with NetworkPolicies, traffic between pods is unencrypted by default. A service mesh injects a sidecar proxy (Envoy) into every pod. Sidecars handle mTLS automatically — every connection is mutually authenticated and encrypted. A compromised pod cannot impersonate another service because it lacks the correct TLS certificate.

Layer 3: Authorization policies in Istio. Beyond encryption, Istio AuthorizationPolicy enforces that order-service may only call POST /payments on payment-service — not admin endpoints. Layer 7 policy enforcement.

Layer 4: Workload Identity. Pods use Kubernetes ServiceAccounts mapped to IAM roles (IRSA on EKS). payment-service's ServiceAccount has exactly the AWS permissions it needs — read from Secrets Manager, write to S3 for receipts. Nothing more. If the pod is compromised, the blast radius is contained to those specific AWS actions.

SeniorQ48: How do you detect and respond to a compromised container in production?

Container compromise detection requires runtime security tooling — static image scanning is insufficient because the compromise happens at runtime.

Detection layer 1: Falco (runtime threat detection). Falco monitors system calls in real time using eBPF. Pre-built rules detect anomalous behavior:

# Falco rules — examples of what triggers alerts
- rule: Shell Spawned in Container
  desc: A shell was spawned in a container (potential compromise)
  condition: spawned_process and container and shell_procs
  output: "Shell in container (user=%user.name command=%proc.cmdline)"
  priority: WARNING

- rule: Unexpected Outbound Connection
  desc: Container connecting to unexpected external IP
  condition: outbound and not expected_outbound_destinations
  output: "Unexpected outbound from %container.name to %fd.rip"
  priority: CRITICAL

Detection layer 2: anomaly detection on metrics. Sudden CPU spike on a payment-service pod that normally uses 200m CPU is suspicious. An alert on "pod CPU 10× higher than 7-day average" can detect crypto-mining malware.

Detection layer 3: audit logs. Kubernetes audit logs record every API call. A compromised container that attempts to read secrets or escalate privileges via the Kubernetes API generates audit log entries.

Response runbook for TicketFlow:

  1. Isolate immediately: apply a NetworkPolicy that blocks all egress from the compromised pod. Remove it from the Service endpoints (remove readiness label). Do not delete it — preserve forensic evidence.
  2. Capture forensics: kubectl exec into the pod to capture running processes, network connections, modified files. Or use kubectl debug with an ephemeral container if exec is too risky.
  3. Rotate all secrets the pod had access to. Assume all secrets accessible via the pod's ServiceAccount are compromised. Rotate immediately.
  4. Investigate the root cause: which CVE was exploited? Which image layer? Was it a supply chain attack on a base image or dependency?
  5. Deploy a clean replacement from a verified image digest.
  6. Post-mortem and preventive controls.
SeniorQ49: Design an alerting strategy that minimizes false positives while catching real incidents before users notice.

Alert fatigue is one of the most dangerous conditions in an ops team — when every alert is a false positive, engineers stop responding and the real incident goes unnoticed. The solution is a tiered, symptom-based alerting strategy.

Principle 1: Alert on symptoms, not causes. Users experience slow responses — alert on latency and error rate, not on CPU or memory. CPU at 90% might be normal during batch processing. 5% error rate is never normal.

Principle 2: Multi-window burn rate alerts (Google SRE model). A single threshold alert fires for brief spikes that resolve on their own. Burn rate compares current error rate against the rate that would exhaust the error budget:

# Page immediately (critical) — burning budget in 1 hour
alert: ErrorBudgetBurnCritical
expr: |
  error_rate_1h > (14.4 * monthly_error_budget_rate) AND
  error_rate_5m > (14.4 * monthly_error_budget_rate)

# Ticket (warning) — burning budget in 6 hours
alert: ErrorBudgetBurnWarning
expr: |
  error_rate_6h > (6 * monthly_error_budget_rate) AND
  error_rate_30m > (6 * monthly_error_budget_rate)

Principle 3: Alert ownership. Every alert must have a clear owner (team), a documented runbook link in the annotation, and a resolution time expectation. If an alert has no runbook, it is not ready for production.

Principle 4: Dead man's switch. Alert if your alerting pipeline itself is silent — if Prometheus has not scraped payment-service in 5 minutes, alert immediately. You cannot trust absence of alerts if the monitoring itself could be broken.

TicketFlow alert inventory: 12 paging alerts (all symptom-based, SLO-driven), 28 warning tickets (leading indicators: disk usage >70%, certificate expiry <30 days, HPA at max replicas). Zero noisy alerts — if an alert fires more than twice without incident action, it is reviewed and either fixed or deleted within 48 hours.

SeniorQ50: Walk through a complete post-mortem for a production incident. What makes a post-mortem effective?

A post-mortem is a structured retrospective after a production incident aimed at understanding root causes and preventing recurrence. The most important rule: blameless. Engineers make decisions based on the information available at the time. The system had the conditions that allowed the failure — fix the system, not the person.

TicketFlow incident — payment-service outage, 23 minutes:

Timeline:

  • 14:30 — payment-service error rate rises to 45%. Alertmanager fires P1 alert.
  • 14:33 — on-call engineer acknowledges. Initial investigation: all pods CrashLoopBackOff.
  • 14:35 — previous container logs show: ERROR: Cannot connect to vault.internal:8200 — connection refused
  • 14:37 — root cause identified: HashiCorp Vault was upgraded 10 minutes earlier. The new Vault version changed the AppRole mount path. payment-service's startup sequence cannot fetch its DB credentials and crashes.
  • 14:41 — hotfix: updated Vault mount path in payment-service ConfigMap. Rolling restart begins.
  • 14:53 — all pods healthy. Error rate returns to 0.02%. Incident resolved.

Root cause: Vault upgrade was not included in payment-service's change dependency matrix. No integration test verified the startup credential fetch after infrastructure changes.

Action items (time-boxed, assigned owners):

  1. [SRE — 1 week] Add Vault AppRole integration test to payment-service CI pipeline. Fails CI if mount path changes.
  2. [Platform — 2 weeks] Implement graceful startup retry — payment-service should retry credential fetch with exponential backoff for 5 minutes before crashing, absorbing transient Vault restarts.
  3. [SRE — 3 days] Add a synthetic canary that tests payment-service startup from scratch every 30 minutes in staging.
  4. [Platform — 1 week] Create infrastructure change communication process — Vault/Kubernetes upgrades require notification to service owners 48 hours in advance.

What makes a post-mortem effective: specific action items with owners and deadlines (not vague "improve testing"), root cause analysis that asks "why" five times (5 Whys), public sharing with the whole engineering org (others learn from it), and a follow-up review in 30 days to verify all action items were completed.
