Container Registries & Nexus

Where images live, how they are versioned, secured, and promoted from development all the way to production.

The Story Behind Registries

📦 The Art Gallery Problem

Imagine you are a sculptor. You create a beautiful bronze statue — your masterpiece. Now imagine if, every time someone wanted to see it, you had to rebuild the entire sculpture from scratch in front of them. No storage. No gallery. Just raw materials and your hands every time.

That would be absurd. You cast the statue once, put it in a gallery, and anyone who wants to see it visits the gallery and finds it exactly as you made it.

This is precisely what container registries solve. You build a Docker image once — your precisely crafted, tested, reproducible artifact — and store it in a registry. Your CI pipeline, your staging environment, your production Kubernetes cluster, your colleague's dev machine: they all pull the same image from the same gallery. Nothing is rebuilt. Nothing drifts. The statue is always exactly what you carved.

But just like galleries have different rules — some are public, some require membership, some are run by enterprises with strict curation policies — registries vary enormously in their features, access control, security scanning, and where they live. Choosing the right registry is an architectural decision with serious production implications.

What Is a Container Registry?

🥪 Beginner Analogy

A container registry is like GitHub, but for Docker images instead of source code. Just as you push code to GitHub and pull it on another machine, you push Docker images to a registry and pull them wherever you need to run them. The registry stores your versioned images, lets you tag specific releases, and controls who can read or write them.

The difference is that images are binary artifacts — not text you can read in a browser. They are layered filesystems assembled from layer blobs, described by a manifest file. The registry stores the blobs and manifests and serves them on demand over HTTPS.

Every Docker image reference has three parts:

  • Registry host — where to download from. Examples: docker.io (Docker Hub, the default), nexus.ticketflow.io:5000 (your private Nexus), 123456789.dkr.ecr.eu-west-1.amazonaws.com (Amazon ECR).
  • Repository (image name) — the name of the image, optionally namespaced. Example: ticketflow/user-service.
  • Tag or digest — the specific version. Example: 1.4.2 (tag, mutable pointer), or @sha256:a8f3... (digest, immutable content address).

Combined: nexus.ticketflow.io:5000/ticketflow/user-service:1.4.2

💡 Why Most Enterprises Choose Nexus

Large engineering organizations typically manage many artifact types: Docker images, Java JARs, npm packages, Python wheels, Helm charts, apt packages for OS bootstrapping. Nexus handles all of them in a single, self-hosted installation behind the corporate firewall. This means one authentication system, one audit log, one backup, one network policy. Cloud-native registries (ECR, ACR) only manage container images, forcing separate infrastructure for other artifact types. For TicketFlow — running on AWS EKS with Java, Node, Python, and Go services, plus Helm charts — Nexus is the practical choice.

CI Pushes to Nexus — Kubernetes Pulls on Deploy

This is the canonical image lifecycle in a production system. The image is built exactly once in CI, pushed to the registry, and then pulled by every environment that needs it — never rebuilt.

The key principle: the same image tag (1.4.2-abc1234) flows through all three environments. No rebuilds. No environment-specific images. What you validated in staging is byte-for-byte identical to what runs in production.

Image Tagging Strategy

Tags are your primary mechanism for versioning and promoting images. A disciplined tagging strategy is what makes image promotion pipelines reliable. Here is the production-grade approach TicketFlow uses:

Semantic Versioning (semver)

Semantic versioning uses three numbers: MAJOR.MINOR.PATCH. 1.0.0 → 1.0.1 (patch: bug fix) → 1.1.0 (minor: new feature, backward compatible) → 2.0.0 (major: breaking change). For container images, semver tags are applied at release time, not at every commit.

# Build and tag with full semver for a release
docker build -t nexus.ticketflow.io:5000/ticketflow/user-service:1.4.2 .

# Also apply the minor version floating tag (for internal tooling that wants latest patch)
docker tag nexus.ticketflow.io:5000/ticketflow/user-service:1.4.2 \
           nexus.ticketflow.io:5000/ticketflow/user-service:1.4

# Push both tags
docker push nexus.ticketflow.io:5000/ticketflow/user-service:1.4.2
docker push nexus.ticketflow.io:5000/ticketflow/user-service:1.4

Git SHA Tagging

Every CI build should also be tagged with the short Git commit SHA. This creates an immutable link between the running container and the exact source code that produced it. It is invaluable for debugging production incidents: you see the image tag, look up the SHA, find the commit, and know exactly what code is running.

# In CI (GitHub Actions), GIT_SHA is the 7-char short commit hash
GIT_SHA=$(git rev-parse --short HEAD)
IMAGE_NAME="nexus.ticketflow.io:5000/ticketflow/user-service"

# Tag with both semver and SHA — the SHA tag is the CI build artifact
docker build \
  -t "${IMAGE_NAME}:1.4.2" \
  -t "${IMAGE_NAME}:1.4.2-${GIT_SHA}" \
  .

# The SHA-tagged image is what CI promotes through environments
# Semver tag is applied at release time by the release pipeline
docker push "${IMAGE_NAME}:1.4.2"
docker push "${IMAGE_NAME}:1.4.2-${GIT_SHA}"

The :latest Trap

❌ The :latest Trap — Do Not Use in Production

The :latest tag is not a version — it is a mutable alias that points to the most recently pushed image. It has no semantic meaning beyond "last push". Using it in Kubernetes deployments is particularly dangerous because imagePullPolicy: IfNotPresent (the default) means Kubernetes will not pull a new image if one already exists locally under that tag. Your nodes could silently be running different versions of the same service. Even with imagePullPolicy: Always, you lose the ability to roll back to a specific previous version — all you can say is "the last one".

Fix: Use :latest only for local development convenience (docker compose up on a dev machine). Never reference :latest in CI pipelines, Kubernetes manifests, or Helm values files. Always use a specific, immutable tag. In critical production workloads, use digest pinning: image: nexus.ticketflow.io:5000/ticketflow/user-service@sha256:a8f3b2c1...

Enterprise Scenario: TicketFlow's Image Promotion Pipeline

🏢 TicketFlow — Image Promotion in Production

TicketFlow runs six microservices on AWS EKS. Images are stored in Nexus at nexus.ticketflow.io:5000. Here is what happens when a developer merges a feature branch to main:

  1. CI build triggers — GitHub Actions runs on the merge commit. It builds each changed service's image, tags it <semver>-<sha>, runs Trivy for CVE scanning, and pushes to Nexus only if scanning passes.
  2. Dev auto-deploy — Argo CD detects the new image tag in the Nexus registry (polling every 3 minutes), updates the Helm values file for the dev namespace, and deploys automatically. Developers can verify their change within 5 minutes of merging.
  3. QA promotion — After QA sign-off, a release engineer runs the promotion script: it re-tags the exact dev image as staging and updates the Helm values file. No rebuild. No code change. The promotion is a tag operation.
  4. Production promotion — After staging validation (automated load tests + manual smoke tests), the same image is promoted to stable and deployed to production. The full semver tag (1.4.2) is also applied at this point to mark the official release.
  5. Rollback — If production has an incident, the rollback is a Helm rollback to the previous release. Helm remembers which image tag was deployed. The previous image is still in Nexus. Rollback completes in under 60 seconds.

Registry Authentication

Private registries require authentication before you can push or pull images. Docker's authentication model is straightforward but has nuances that catch developers off guard in CI environments.

docker login — How It Works

# Log in to Nexus (stores credentials in ~/.docker/config.json)
docker login nexus.ticketflow.io:5000 \
  --username ci-deploy \
  --password-stdin <<< "$NEXUS_PASSWORD"

# Log in to Docker Hub (default registry)
docker login --username myuser --password-stdin <<< "$DOCKERHUB_TOKEN"

# Log in to Amazon ECR (uses AWS CLI to generate a temporary token)
aws ecr get-login-password --region eu-west-1 | \
  docker login --username AWS --password-stdin \
  123456789.dkr.ecr.eu-west-1.amazonaws.com

# Log in to Azure ACR
az acr login --name ticketflowacr

# Verify credentials are stored
cat ~/.docker/config.json
💡 CI/CD Credential Management

In GitHub Actions, never hardcode registry credentials. Store them as encrypted repository secrets (NEXUS_USERNAME, NEXUS_PASSWORD) and reference them as ${{ secrets.NEXUS_PASSWORD }}. Use a dedicated CI service account with minimal permissions — read/write to specific repositories only, never admin. Rotate service account passwords regularly (every 90 days). For ECR, use OIDC federation so CI never holds a long-lived AWS credential at all.

Nexus Setup Walkthrough

Setting up Nexus as a Docker registry involves three steps: install Nexus, create a Docker hosted repository, and configure a proxy repository to cache Docker Hub images (avoiding rate limits).

# docker-compose.yml — Run Nexus locally or on a server
version: "3.9"
services:
  nexus:
    image: sonatype/nexus3:3.66.0
    container_name: nexus
    ports:
      - "8081:8081"    # Nexus UI and API
      - "5000:5000"    # Docker hosted repository (push/pull)
      - "5001:5001"    # Docker proxy repository (Docker Hub cache)
    volumes:
      - nexus-data:/nexus-data
    environment:
      INSTALL4J_ADD_VM_PARAMS: "-Xms2703m -Xmx2703m -XX:MaxDirectMemorySize=2703m"

volumes:
  nexus-data:
# Step 1: Start Nexus and get the initial admin password
docker compose up -d nexus
# Wait ~2 minutes for Nexus to start
docker exec nexus cat /nexus-data/admin.password
# Open http://localhost:8081 and complete setup wizard

# Step 2: Create repositories via Nexus REST API (after initial setup)
# (Or use the UI: Administration → Repository → Repositories → Create repository)

# Create a Docker hosted repository (where you push your own images)
curl -u admin:$NEXUS_ADMIN_PASS \
  -X POST http://localhost:8081/service/rest/v1/repositories/docker/hosted \
  -H "Content-Type: application/json" \
  -d '{
    "name": "docker-internal",
    "online": true,
    "storage": {"blobStoreName": "default", "strictContentTypeValidation": true, "writePolicy": "allow_once"},
    "docker": {"v1Enabled": false, "forceBasicAuth": true, "httpPort": 5000}
  }'

# Create a Docker proxy repository (caches Docker Hub — avoids rate limits)
curl -u admin:$NEXUS_ADMIN_PASS \
  -X POST http://localhost:8081/service/rest/v1/repositories/docker/proxy \
  -H "Content-Type: application/json" \
  -d '{
    "name": "docker-hub-proxy",
    "online": true,
    "storage": {"blobStoreName": "default", "strictContentTypeValidation": true},
    "proxy": {"remoteUrl": "https://registry-1.docker.io", "contentMaxAge": 1440, "metadataMaxAge": 1440},
    "docker": {"v1Enabled": false, "forceBasicAuth": true, "httpPort": 5001},
    "dockerProxy": {"indexType": "HUB"}
  }'

Configure Docker Daemon to Trust the Registry

// /etc/docker/daemon.json — on every machine that needs to push/pull from Nexus
// Required when running Nexus over HTTP (not HTTPS) — common in dev/internal setups
{
  "insecure-registries": ["nexus.ticketflow.io:5000"],
  "registry-mirrors": ["http://nexus.ticketflow.io:5001"]
}
# Restart the Docker daemon after editing daemon.json
sudo systemctl restart docker

# Verify the configuration was picked up
docker info | grep -A5 "Insecure Registries"
docker info | grep -A5 "Registry Mirrors"

Push and Pull Commands with TicketFlow

Building and Pushing All TicketFlow Services

#!/bin/bash
# build-and-push.sh — Build all TicketFlow services and push to Nexus
set -euo pipefail

REGISTRY="nexus.ticketflow.io:5000"
NAMESPACE="ticketflow"
VERSION="${1:-1.4.2}"
GIT_SHA=$(git rev-parse --short HEAD)

# Authenticate
echo "$NEXUS_PASSWORD" | docker login "$REGISTRY" -u ci-deploy --password-stdin

# Build and push each service
SERVICES=(
  "user-service:services/user-service"
  "ticket-service:services/ticket-service"
  "payment-service:services/payment-service"
  "order-service:services/order-service"
  "notification-service:services/notification-service"
  "api-gateway:services/api-gateway"
)

for entry in "${SERVICES[@]}"; do
  SERVICE="${entry%%:*}"
  CONTEXT="${entry##*:}"
  IMAGE="${REGISTRY}/${NAMESPACE}/${SERVICE}"

  echo "==> Building ${SERVICE}..."
  docker build \
    --tag "${IMAGE}:${VERSION}" \
    --tag "${IMAGE}:${VERSION}-${GIT_SHA}" \
    --label "git.sha=${GIT_SHA}" \
    --label "build.version=${VERSION}" \
    "${CONTEXT}"

  echo "==> Pushing ${SERVICE}..."
  docker push "${IMAGE}:${VERSION}"
  docker push "${IMAGE}:${VERSION}-${GIT_SHA}"

  echo "==> ${SERVICE} pushed successfully"
done

echo "All services pushed to ${REGISTRY}"

Behind the Scenes

What happens when you run docker push nexus.ticketflow.io:5000/ticketflow/user-service:1.4.2?
  1. The Docker CLI reads your stored credentials from ~/.docker/config.json for the host nexus.ticketflow.io:5000.
  2. Docker sends a POST /v2/<name>/blobs/uploads/ request to the Nexus registry API to initiate a push session and receive an upload URL.
  3. For each image layer that Nexus does not already have (checked via a HEAD request against its content-addressable storage), Docker uploads the compressed layer blob using a chunked HTTP PUT.
  4. Once all layers are uploaded, Docker pushes the image manifest — a JSON document describing the image layers, their digests, the image config, and metadata. The manifest is stored under both the tag name and its SHA256 digest.
  5. Nexus stores blobs in its blobstore (by default at /nexus-data/blobs/default/) keyed by their SHA256 digest. The same layer shared across multiple images is stored exactly once — content-addressed deduplication.
  6. The tag mapping (e.g., user-service:1.4.2 → manifest SHA256) is updated in Nexus's metadata database.
  7. On docker pull, the process reverses: Docker fetches the manifest by tag, then downloads only the layers it does not already have locally (delta pull based on layer digest matching).

This is why the Docker Registry V2 API (which all modern registries implement — Nexus, ECR, ACR, Harbor, GCR) is content-addressable: images are addressed by what they contain (SHA256 digest), not where they came from. This property is what makes image promotion safe: you can re-tag an image in-registry without touching the bits at all.

How does Nexus proxy Docker Hub? What happens on the first pull?

When you configure Nexus as a Docker Hub proxy and a developer pulls node:18-alpine:

  1. Docker sends the pull request to your Nexus proxy port (5001).
  2. Nexus checks its local blobstore for the requested tag's manifest. On the first pull, it is not there.
  3. Nexus forwards the request upstream to https://registry-1.docker.io, authenticating with the Docker Hub credentials you configured in the proxy repository settings.
  4. Docker Hub returns the manifest and layer blobs. Nexus streams them back to the developer's Docker daemon and simultaneously stores them locally.
  5. The next developer who pulls node:18-alpine gets it directly from Nexus's local cache — no Docker Hub round-trip, no rate limit consumption, much faster download.

The cache TTL is configurable (default 1440 minutes = 24 hours for metadata). This means the proxy checks for upstream updates at most once per day per image tag — protecting you from Docker Hub's 100 pull/6-hour anonymous rate limit, which is trivially exceeded by CI systems rebuilding dozens of images per hour.

Common Mistakes

❌ Mistake 1: Overwriting Released Tags

A CI pipeline pushes user-service:1.4.2. A hotfix is applied, the Dockerfile is updated, and another push overwrites user-service:1.4.2 with different bits. Now every system pulling 1.4.2 gets different content depending on when they last pulled. Rollbacks become unreliable — "rolling back to 1.4.2" is meaningless if 1.4.2 has changed.

Fix: Configure Nexus with writePolicy: "allow_once" on the hosted repository. This prevents overwriting existing tags. Released tags are immutable. Use a new version number for each distinct artifact. Only development/snapshot repositories should allow overwrites.

❌ Mistake 2: Storing Registry Credentials in Docker Compose Files

Developers add registry credentials to docker-compose.yml or hardcode them in CI pipeline YAML files. These files get committed to Git. Credentials are now exposed to everyone with repo access and to GitHub's audit history permanently.

Fix: Use environment variables. In docker-compose, reference $NEXUS_PASSWORD from the environment. In CI, use the secrets management feature of your CI platform (GitHub Actions secrets, GitLab CI variables, Jenkins credentials). Never write credentials in files that could be committed.

❌ Mistake 3: Not Setting Up a Docker Hub Proxy

Teams pull base images directly from Docker Hub during CI builds. Docker Hub rate-limits anonymous pulls to 100/6h and authenticated pulls to 200/6h (free plan). A team of 10 developers each running multiple CI builds per day easily exceeds these limits. Builds start failing with 429 Too Many Requests at the worst possible time.

Fix: Set up a Nexus Docker proxy repository for Docker Hub from day one. Configure the Docker daemon's registry-mirrors to route Docker Hub pulls through your proxy. All base image pulls are cached locally — no rate limits, faster downloads, and you have a copy of your base images even if Docker Hub has an outage.

❌ Mistake 4: Skipping Image Scanning

Teams push images to production without scanning for known vulnerabilities (CVEs). A base image like node:18 can contain dozens of critical CVEs inherited from the Debian or Alpine packages it bundles. Without scanning, you ship known vulnerabilities to production and may violate security compliance requirements (SOC 2, PCI DSS, ISO 27001).

Fix: Integrate Trivy (or Snyk) into the CI pipeline as a gate before the push step. Fail the pipeline on critical CVEs. Run scheduled re-scans of images already in the registry (Nexus IQ Server, or scheduled Trivy runs via cron). Update base images monthly as a maintenance task.

❌ Mistake 5: One Nexus Repository for Everything

Engineers create a single Docker hosted repository in Nexus and push all images there — dev builds, release candidates, stable releases, base images from Docker Hub, everything mixed together. There is no way to enforce promotion policies, apply different retention rules per environment, or grant different teams access to different image sets.

Fix: Create separate repositories per lifecycle stage: docker-dev (short retention, allow overwrites, developer access), docker-release (immutable tags, write-once, CI access only), docker-hub-proxy (read-only proxy, no direct pushes). Use a Nexus Group repository that aggregates all three — developers point at the group for pulls and get everything in one endpoint.

Architect Perspective

🏛️ Architect Note

The registry is not a storage detail — it is a deployment control plane. How you structure your registry directly determines how much control you have over what reaches production. Three architectural principles to drive every registry decision:

  • The registry is the source of truth for deployable artifacts. Not Git, not CI artifacts, not S3 buckets — the registry. Everything in Kubernetes, Helm, and Argo CD references image tags in the registry. If an image is not in the registry, it cannot be deployed. This means the registry must be highly available, backed up, and access-controlled with the same rigour as your database.
  • Promotion is a tag operation, not a build operation. Never rebuild an image to promote it. A "rebuild for production" is an invisible code path that was never tested. Instead: build once, tag with an immutable identifier (SHA + semver), promote by applying environment tags to the same image. The bits are identical. The metadata (tag) changes. This is the only model that guarantees "what you tested is what you deployed".
  • Registry strategy must be decided before CI/CD strategy. Your registry choice constrains your CI/CD options. If you choose ECR, you are tied to AWS IAM for authentication — fine if you are all-in on AWS, problematic if you run hybrid deployments. If you choose Nexus, you need to operate it. If you choose Harbor, you need Kubernetes to run it. Make the registry decision first; it shapes everything downstream.

On Nexus specifically: Nexus is operationally heavier than a managed cloud registry, but for organizations with heterogeneous artifact needs (Docker + Maven + npm + Python + Helm + OS packages), it provides a level of consolidation that is genuinely valuable. One auth system, one backup job, one RBAC model, one monitoring target. The operational overhead is real but bounded; the consolidation value compounds over years. For pure cloud-native AWS shops with no polyglot artifact requirements, ECR is simpler and zero-ops. For everyone else, Nexus usually wins.

Production Readiness Checklist

  • Private registry is deployed and accessible only within the VPC / corporate network (never public-facing for production images)
  • TLS/HTTPS is configured on the registry — no production system uses HTTP for image transport
  • Separate repositories exist for dev snapshots, release candidates, and stable releases with appropriate write policies
  • Docker Hub proxy repository configured — no direct Docker Hub pulls from CI or production nodes
  • All image tags use semver + git SHA format — :latest is never used in Kubernetes manifests or Helm values
  • Release repository configured with writePolicy: allow_once — published tags cannot be overwritten
  • CI service account has push-only access to dev/release repositories; production nodes have pull-only access
  • Trivy (or equivalent) CVE scanning runs in CI as a pipeline gate — critical CVEs block the push step
  • Registry storage (Nexus blobstore) is on a durable volume (EBS, NFS) — not the container's ephemeral filesystem
  • Registry backup is automated and tested — a registry failure should not prevent deploying a hotfix from backup
  • Image retention policy configured — dev images expire after 30 days; release images retained for 1 year minimum
  • Kubernetes imagePullSecrets configured for all namespaces that pull from the private registry
  • Registry health is monitored — disk usage, pull latency, and availability alerts configured in Prometheus/Grafana

Interview Questions

These are the most frequently asked container registry questions in DevOps and software architect interviews. Each answer is production-grade — not a textbook definition.

Q1: What is a container registry and how does it differ from Docker Hub?

A container registry is a storage and distribution system for container images. It stores image layers (compressed tarballs) and manifests (JSON documents describing the layers) in a content-addressable store. The Docker Registry V2 API (now the OCI Distribution Spec) is the standard protocol — any compliant tool (Docker, Podman, Kubernetes, Helm) can push and pull from any compliant registry.

Docker Hub (hub.docker.com) is one specific registry — the public default hosted by Docker Inc. It is the registry Docker uses when you run docker pull nginx with no hostname prefix. Docker Hub hosts millions of public images and is rate-limited (100 pulls/6h for anonymous, 200/6h for free authenticated accounts).

The distinction matters architecturally: Docker Hub is a public, third-party SaaS. In production, organizations run their own private registry (Nexus, Harbor, ECR, ACR) for security, reliability, and control. Pulling base images via a private proxy of Docker Hub eliminates rate-limit issues and ensures builds succeed even if Docker Hub has an outage.

Q2: What is the difference between a tag and a digest when referencing a Docker image?

A tag is a mutable human-readable pointer to a manifest. user-service:1.4.2 is a tag — it currently points to a specific manifest, but a new push can overwrite it to point to a completely different manifest. If you push new bits under the same tag, anyone pulling that tag tomorrow gets different content than someone who pulled today.

A digest is an immutable content address — the SHA256 hash of the manifest file itself. user-service@sha256:a8f3b2c1... will always, forever, refer to the exact same bits. You cannot overwrite a digest because it is computed from the content. If the content changes, the digest changes.

When to use each:

  • Development and CI tagging — use tags like 1.4.2-abc1234. Human-readable, version-controlled.
  • Critical production workloads — use digests. Kubernetes manifests and Helm values can reference image: registry/service@sha256:.... This is the only truly immutable reference. Even if an attacker compromises the registry and overwrites a tag, a digest-pinned deployment is unaffected.

Best practice: record the digest at build time (docker inspect --format='{{index .RepoDigests 0}}'), store it in your Helm chart or deployment manifest for production, and use the human-readable tag only for CI tooling and human communication.

Q3: How would you implement an image promotion pipeline from dev to production?

An image promotion pipeline has one core principle: build once, promote the same artifact through environments. Never rebuild for a higher environment.

The pipeline:

  1. CI Build — On merge to main, build the image and tag it <semver>-<git-sha>. Run CVE scanning. If scanning passes, push to the docker-dev repository in Nexus. Apply a dev promotion tag.
  2. Dev deployment — Argo CD detects the updated dev tag and deploys to the dev namespace on EKS. Developers validate.
  3. Promote to staging — A promotion script (triggered manually or by a gate) calls the Nexus API to copy the manifest from docker-dev to docker-release with the tag staging. No rebuild. No new push from CI.
  4. Staging deployment and testing — Argo CD deploys. Integration tests and load tests run against staging. QA signs off.
  5. Promote to production — Same promotion script moves the image to the stable tag. Applies the final semver tag (e.g., 1.4.2) as the official release marker. Argo CD deploys to production namespace.
  6. Rollback — If production breaks, re-apply the previous stable tag to the previous image manifest. Kubernetes pulls the previous image. Helm rollback handles the manifest changes.

The key: promotions are tag operations in the registry, not re-executions of the build pipeline. This guarantees that what you tested in staging is byte-for-byte identical to what runs in production.

Q4: Why would a large enterprise choose Nexus over Amazon ECR, even when running on AWS?

ECR is excellent if your only artifact type is Docker images and your entire stack runs on AWS. But large enterprises typically have more complex requirements:

  • Polyglot artifacts — Java teams use Maven/Gradle and need a Maven repository. Node teams use npm. Python teams use PyPI (pip). Go teams use module proxies. Helm charts need a chart repository. All of these can live in Nexus, but not in ECR. Running a separate tool for each artifact type multiplies operational complexity.
  • Data residency and air-gapped environments — regulated industries (finance, healthcare, government) may be unable to use cloud-managed services for artifact storage due to data sovereignty requirements or security classifications. Self-hosted Nexus runs in the corporate data centre with no external dependencies.
  • Proxy caching — Nexus can proxy Docker Hub, Maven Central, npm registry, PyPI, and others — caching every dependency used by every team behind the corporate firewall. This eliminates external dependency on public repositories and prevents supply-chain attacks via dependency confusion. ECR cannot proxy non-ECR registries.
  • Multi-cloud or on-premise workloads — if the organization runs workloads on-premise, on Azure, or on GCP alongside AWS, a single Nexus instance serves all of them. ECR is AWS-specific.

The trade-off is real: Nexus requires operational investment (hardware, upgrades, backups, HA configuration). ECR is zero-ops for teams already on AWS. The decision ultimately comes down to the breadth of the artifact portfolio and the degree of cloud vendor commitment.

Q5: A Kubernetes pod is stuck in ImagePullBackOff. How do you diagnose and fix it?

ImagePullBackOff means the Kubernetes node failed to pull the container image. It is one of the most common issues in Kubernetes deployments and has a handful of root causes:

  1. Diagnose the exact error: 
    kubectl describe pod <pod-name> -n <namespace>
# Look at "Events" section — the error message is specific
  2. Wrong image tag — the most common cause. The tag specified in the deployment manifest does not exist in the registry. Fix: verify the tag exists in Nexus (curl -u ci:pass nexus.ticketflow.io:5000/v2/ticketflow/user-service/tags/list) and correct the deployment manifest.
  3. Missing imagePullSecret — the namespace does not have credentials to pull from the private registry. Fix: create a Kubernetes Secret of type kubernetes.io/dockerconfigjson and reference it in the deployment's imagePullSecrets: 
    kubectl create secret docker-registry nexus-credentials \
  --docker-server=nexus.ticketflow.io:5000 \
  --docker-username=ci-deploy \
  --docker-password=$NEXUS_PASSWORD \
  -n <namespace>
  4. Registry unreachable from nodes — network policy or security group blocks EKS nodes from reaching nexus.ticketflow.io:5000. Fix: verify connectivity from a node (kubectl run -it --rm debug --image=alpine -- wget -qO- nexus.ticketflow.io:5000/v2/) and adjust security groups / network policies.
  5. TLS/certificate issue — Nexus uses a self-signed certificate or a cert from an internal CA that the node does not trust. Fix: configure the container runtime (containerd or docker) on the node with the CA certificate, or switch Nexus to a Let's Encrypt certificate.
Q6: What is a Docker Hub proxy repository in Nexus and why is it important in production?

A Docker Hub proxy repository in Nexus is a pass-through cache. When a developer or CI system pulls a public image (e.g., node:18-alpine), the request goes to Nexus instead of directly to Docker Hub. Nexus fetches the image from Docker Hub on the first request, caches all layers locally, and serves every subsequent request for the same image from the local cache — without any Docker Hub round-trip.

Why this matters in production:

  • Rate limit elimination — Docker Hub limits anonymous pulls to 100/6h and free authenticated pulls to 200/6h. A CI system rebuilding 20 services an hour easily exhausts these limits, causing builds to fail with 429 Too Many Requests. With a proxy, the rate-limit budget is consumed only on cache misses — typically <1% of pulls.
  • Build speed — Local cache pulls at LAN speed (1 Gbps) vs Docker Hub (variable internet). A 200 MB base image pull from Docker Hub takes 10–30 seconds; from a local Nexus on the same network, under 1 second.
  • Resilience — Docker Hub has periodic outages. Without a proxy, an outage stops all builds. With Nexus caching all previously used base images, builds continue unaffected. Your builds are not dependent on an external SaaS during deployments.
  • Audit and policy enforcement — all base image pulls are logged in Nexus. Security teams can audit which base images are in use across the organisation, enforce policies (e.g., only Alpine-based images allowed), and detect if a team is pulling a base image with known critical CVEs.

At TicketFlow, the Docker Hub proxy eliminated random CI build failures that were blamed on "flaky tests" for months — they were actually Docker Hub rate-limit failures. After configuring the proxy, those failures disappeared entirely.
