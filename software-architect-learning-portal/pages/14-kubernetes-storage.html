Kubernetes Storage

Master PersistentVolumes, PersistentVolumeClaims, StorageClasses, and stateful workloads so your databases survive pod restarts, node failures, and cluster upgrades.

The Story Behind Kubernetes Storage

🗄️ The Stateless Dream vs Reality

Kubernetes was designed with stateless applications in mind. The original philosophy was elegant: pods are ephemeral, disposable, interchangeable. If a pod dies, the scheduler creates a new one. If a node fails, pods migrate. Nothing should depend on local disk.

Then reality arrived. Every serious application has a database. Databases are not stateless — they are the opposite. PostgreSQL stores your tickets. Redis caches your sessions. Elasticsearch indexes your events. These services cannot simply be restarted and pointed at an empty disk. They need persistent storage that outlives any individual pod, survives node failures, and can be backed up, expanded, and migrated.

Kubernetes solved this with a three-layer abstraction: StorageClass (defines how storage is provisioned), PersistentVolume (the actual storage resource), and PersistentVolumeClaim (the application's request for storage). This decoupling lets developers ask for "10 GB of fast SSD storage" without knowing or caring whether the underlying platform is AWS EBS, NFS, Ceph, or a local SSD. The platform team provisions it; the application team consumes it.

Storage in Plain English

🏠 Beginner Analogy: Renting Storage Units

Think of Kubernetes storage like a storage facility business:

  • StorageClass is like a type of unit the facility offers — climate-controlled (fast SSD), standard (magnetic disk), shared access (NFS). It defines the tier and how units are built.
  • PersistentVolume (PV) is an actual storage unit — a 10m x 10m room, physically built, with a lock on the door. It exists independently of any tenant.
  • PersistentVolumeClaim (PVC) is your rental contract — "I need a climate-controlled unit that's at least 50 sq ft." The facility matches you to an available unit (or builds a new one if dynamic provisioning is enabled) and hands you the key.
  • Your pod is the tenant. It uses the unit (mounts the PVC as a directory) without knowing or caring who built it or what type of lock is on the door.

When you leave (pod is deleted), your stuff stays in the unit (data is preserved). The rental contract (PVC) can be re-attached to a new pod. This is how your database survives a pod restart.

Technical Deep Dive

Volumes vs PersistentVolumes: What Is the Difference?

Kubernetes has two distinct storage concepts that are frequently confused:

Aspect Volume PersistentVolume (PV)
Lifecycle Tied to the Pod — dies when the pod dies Independent of any Pod — survives pod deletion
Defined in Pod spec directly (inline) Separate cluster-level resource
Claimed by Pod directly PersistentVolumeClaim (PVC), which pod references
Portability Tied to the pod definition Decoupled from consumer — reusable
Use case Shared temp space between containers in a pod (emptyDir), config injection (configMap, secret) Databases, file stores, anything needing durable persistence
Examples emptyDir, configMap, secret, hostPath AWS EBS, EFS, GCE PD, NFS, Ceph RBD, local PV

PVC Lifecycle: The Four Phases

A PVC moves through distinct phases from creation to release:

  • Pending — The PVC has been created but no matching PV has been bound yet. The cluster is searching for an available PV that satisfies the claim (correct size, access mode, StorageClass). With dynamic provisioning, this phase is brief — the StorageClass creates a PV on demand.
  • Bound — A matching PV has been found and bound to the PVC. The relationship is exclusive (for RWO volumes): one PV binds to exactly one PVC and vice versa. The pod can now mount the PVC.
  • Released — The PVC has been deleted, but the PV still exists. The PV's data is intact. The PV is not yet available for a new claim because it may contain sensitive data from the previous tenant.
  • Failed — Automatic reclaim of the PV has failed. Requires manual intervention.

Access Modes

Access modes define how many nodes can mount the volume simultaneously and with what permissions:

Mode Abbreviation Description Typical Backend
ReadWriteOnce RWO One node can mount read-write. Multiple pods on the same node can access it. AWS EBS, GCE PD, Azure Disk
ReadOnlyMany ROX Many nodes can mount read-only simultaneously. NFS, CephFS, EFS
ReadWriteMany RWX Many nodes can mount read-write simultaneously. AWS EFS, NFS, CephFS, Azure Files
ReadWriteOncePod RWOP Only one specific pod can mount read-write (Kubernetes 1.22+). EBS CSI driver

StorageClass and Dynamic Provisioning

Before StorageClasses, administrators manually pre-provisioned PersistentVolumes — a painful, slow, error-prone process. Dynamic provisioning changed everything: when a PVC is created requesting a StorageClass, Kubernetes automatically calls the StorageClass's provisioner to create a real storage volume (e.g., an AWS EBS disk) and a corresponding PV — all without human intervention.

Key StorageClass parameters:

  • provisioner — the plugin that creates the storage (e.g., ebs.csi.aws.com, efs.csi.aws.com, disk.csi.azure.com)
  • reclaimPolicy — what happens to the PV when the PVC is deleted: Delete (destroys the underlying volume — risky for production data) or Retain (keeps the volume for manual cleanup)
  • volumeBindingMode — Immediate (PV created as soon as PVC is created) or WaitForFirstConsumer (PV created when a pod first mounts the PVC — preferred for zone-aware provisioning)
  • allowVolumeExpansion — whether the volume can be resized after creation
  • parameters — driver-specific settings (e.g., EBS volume type: gp3, IOPS, encryption)

AWS EBS vs EFS: When to Use Which

Aspect AWS EBS (gp3) AWS EFS
Access mode RWO — one node at a time RWX — many nodes simultaneously

Cost Pay per GB provisioned Pay per GB stored (scales automatically)

Encryption KMS encryption available KMS encryption available

StatefulSet and PVC Templates

StatefulSets are the Kubernetes workload resource designed for stateful applications. Unlike Deployments, StatefulSets provide:

  • Stable, unique network identities — pods are named name-0, name-1, etc. The identity is preserved across restarts.
  • Stable persistent storage via volumeClaimTemplates — each pod in a StatefulSet gets its own dedicated PVC automatically created from the template. postgres-0 gets data-postgres-0, postgres-1 gets data-postgres-1. Each pod always remounts its own volume.
  • Ordered deployment and scaling — pods are created and deleted in order (0, 1, 2 ... for scale up; reverse for scale down), critical for leader-election replication setups.

Volume Expansion

When a StorageClass has allowVolumeExpansion: true, you can resize a PVC by editing the spec.resources.requests.storage field. The CSI driver expands the underlying volume and the filesystem. Shrinking volumes is not supported (it risks data loss). Note: the pod may need to be restarted for the filesystem to reflect the new size in some configurations.

Enterprise Scenario: TicketFlow PostgreSQL on EKS

🏢 TicketFlow: Making ticket-service's Database Production-Ready

TicketFlow's ticket-service (Spring Boot) stores all ticket data in PostgreSQL. Early on, the team ran PostgreSQL as a Deployment with emptyDir storage — every pod restart wiped the database. After a scheduled node drain during an AWS maintenance window destroyed three hours of ticket data, the team was tasked with implementing proper persistent storage.

Requirements identified by the architect team:

  • PostgreSQL data must survive pod restarts, deployments, and node failures
  • Storage must be fast — the ticketing platform processes 50,000 events/hour during peak concert sale events
  • Volumes must be encrypted at rest (SOC 2 compliance)
  • The team must be able to expand storage without downtime as the ticket catalog grows
  • Backups must be possible without stopping the database

Solution chosen: AWS EBS gp3 via the EBS CSI driver with a custom StorageClass, deployed as a StatefulSet with a volumeClaimTemplate. EBS was chosen over EFS because PostgreSQL is a single-writer database (RWO is sufficient) and EBS gp3 delivers the IOPS needed for high-concurrency transaction workloads.

Code Examples

StorageClass: gp3 EBS with Encryption

# storageclass-gp3-encrypted.yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: gp3-encrypted
  annotations:
    # Make this the default StorageClass for the cluster
    storageclass.kubernetes.io/is-default-class: "true"
provisioner: ebs.csi.aws.com
volumeBindingMode: WaitForFirstConsumer   # Provision in the same AZ as the pod
reclaimPolicy: Retain                     # Keep the EBS volume when PVC is deleted
allowVolumeExpansion: true                # Allow kubectl edit pvc to increase size
parameters:
  type: gp3
  iops: "3000"                            # Baseline IOPS for gp3
  throughput: "125"                       # MB/s baseline throughput
  encrypted: "true"                       # Encrypt with AWS KMS
  # kmsKeyId: arn:aws:kms:us-east-1:...  # Optionally specify a CMK

Behind the Scenes

What happens when a StatefulSet pod is scheduled and mounts an EBS volume?
  1. The StatefulSet controller detects that postgres-0 does not yet have a PVC named postgres-data-postgres-0. It creates the PVC from the volumeClaimTemplate spec.
  2. The PVC is created with status.phase: Pending and references the gp3-encrypted StorageClass.
  3. Because the StorageClass uses volumeBindingMode: WaitForFirstConsumer, the Provisioner does NOT act yet. It waits until a pod is actually scheduled to a node.
  4. The scheduler assigns postgres-0 to a node in us-east-1a. The WaitForFirstConsumer binding mode kicks in: the provisioner now knows the AZ and creates an EBS volume in us-east-1a (not 1b or 1c). This prevents a cross-AZ mount that would fail.
  5. The EBS CSI driver calls the AWS API to create a gp3 EBS volume. AWS returns a volume ID (vol-0abc123def456). The driver creates a PersistentVolume object in Kubernetes referencing this volume ID.
  6. The PV is bound to the PVC. The PVC status.phase transitions to Bound.
  7. The kubelet on the scheduled node detects the pod has a PVC-backed volume. It calls the EBS CSI node plugin to attach the EBS volume to the EC2 instance via the AWS API.
  8. Once attached, the kubelet calls the CSI node plugin again to mount the device (format the filesystem if it's a new volume — ext4 by default, or xfs if specified) and mount it at /var/lib/postgresql/data inside the pod's filesystem namespace.
  9. The pod starts. PostgreSQL initializes at /var/lib/postgresql/data/pgdata (the subdirectory) and begins accepting connections.

When the pod is deleted and rescheduled: if it lands on the same node, the volume is already attached — only the container restart happens. If it lands on a different node, the EBS volume detaches from the old EC2 instance and re-attaches to the new one before the pod starts. This is why EBS (RWO) works fine for single-replica databases — it follows the pod across nodes.

How does volume expansion work under the hood?
  1. You patch the PVC's spec.resources.requests.storage from 50Gi to 100Gi.
  2. The PV controller detects the size mismatch between PVC request and PV capacity. It calls the EBS CSI driver's ControllerExpandVolume endpoint.
  3. The CSI driver calls the AWS ModifyVolume API to resize the EBS volume from 50 GB to 100 GB. AWS performs this online — no detach required for gp3.
  4. The PV's spec.capacity.storage is updated to 100Gi.
  5. For the filesystem inside the pod to see the new space, the CSI driver must run NodeExpandVolume — this calls resize2fs (ext4) or xfs_growfs (xfs) inside the pod's node. This may happen automatically or require the pod to restart, depending on the CSI driver version and Kubernetes version (1.24+ supports online filesystem expansion for most drivers).
  6. The PVC status.capacity.storage updates to 100Gi once the filesystem resize is complete.
What is the difference between Retain, Delete, and Recycle reclaim policies?

When a PVC is deleted, the PV's reclaimPolicy determines what happens to the underlying storage:

  • Delete — The underlying storage (EBS volume, GCE PD, etc.) is automatically deleted when the PVC is deleted. This is the default for dynamically provisioned volumes. Convenient, but dangerous: a helm uninstall or accidental namespace deletion can destroy production data. For TicketFlow's PostgreSQL, the architect explicitly set Retain to prevent this.
  • Retain — The PV and underlying storage are preserved after PVC deletion. The PV moves to Released state. An administrator must manually clean up. Safer for production databases — you can recover data even after an accidental PVC deletion.
  • Recycle — Deprecated. Was only supported by NFS and hostPath. Performed a basic rm -rf of the volume contents and made it available for a new claim. Do not use.

Production recommendation: Use Retain for all stateful workloads with real data. Use Delete only for truly ephemeral stateful workloads (e.g., test databases in CI).

Common Mistakes

Mistake 1: Using a Deployment Instead of a StatefulSet for Databases

A Deployment with a PVC reference works for single-replica databases — until you scale to 2 replicas. Both pods attempt to mount the same RWO EBS volume. One pod comes up; the other is stuck in ContainerCreating forever with a Multi-Attach error: volume is already exclusively attached to one node error. Databases on Deployments also lose their stable identity: after a reschedule, PostgreSQL data initialized by postgres-deployment-abc is mounted by postgres-deployment-xyz — technically fine with PVCs, but fragile without stable network names for replication setups.

Fix: Always use a StatefulSet for databases. Use volumeClaimTemplates to give each replica its own dedicated PVC. The StatefulSet controller manages the binding between pod identity and PVC identity.

Mistake 2: Using reclaimPolicy Delete on Production Databases

A developer runs helm uninstall ticketflow-postgres during a debugging session. The Helm chart deletes the StatefulSet, which deletes the pods. Kubernetes then deletes the PVCs because the StatefulSet is gone. The reclaimPolicy is Delete (the default for EBS CSI). The EBS volumes are destroyed. The PostgreSQL data is gone. This has happened at real companies — it is not a hypothetical.

Fix: Set reclaimPolicy: Retain on all StorageClasses used for databases. Add kubectl annotate pvc postgres-data-postgres-0 helm.sh/resource-policy=keep to protect Helm-managed PVCs from deletion during helm uninstall.

Mistake 3: Mounting EBS Across Availability Zones

A StorageClass with volumeBindingMode: Immediate creates the EBS volume immediately when the PVC is created — before a pod is scheduled. If the volume is created in us-east-1a but the pod is scheduled to a node in us-east-1b, the pod will be stuck with Volume is not available in the requested zone. EBS volumes are AZ-bound.

Fix: Always use volumeBindingMode: WaitForFirstConsumer on EBS StorageClasses. This delays PV creation until the pod is scheduled, so the volume is guaranteed to be created in the correct AZ.

Mistake 4: Writing PostgreSQL Data Directly to the Mount Root

PostgreSQL is started with PGDATA=/var/lib/postgresql/data — which is the mount root of the EBS volume. On a brand new EBS volume, PostgreSQL initializes the directory and writes a PG_VERSION file. Everything seems fine. But the EBS CSI driver also writes a lost+found directory at the filesystem root during mount. PostgreSQL refuses to initialize if PGDATA is non-empty. Result: initdb: error: directory "/var/lib/postgresql/data" exists but is not empty on pod restart with a recycled EBS volume.

Fix: Always set PGDATA to a subdirectory within the mount: PGDATA=/var/lib/postgresql/data/pgdata. The lost+found at the root does not interfere, and PostgreSQL initializes cleanly into the subdirectory.

Mistake 5: Not Setting fsGroup in Pod securityContext

A pod runs as non-root user 999 (postgres). The EBS volume is mounted with root ownership. The postgres process cannot write to /var/lib/postgresql/data. The container crashes with a permission denied error on startup.

Fix: Set spec.securityContext.fsGroup: 999 in the pod spec. The fsGroup setting instructs the kubelet to chown the mounted volume to the specified GID before the container starts, making the volume writable by the application user.

Architect Perspective

Architect Note

Kubernetes storage is where the "cattle, not pets" philosophy meets its limits. Databases are pets — they have names, stable identities, data that cannot be discarded. The storage abstractions (PV/PVC/StorageClass) are Kubernetes' answer to this tension, and they are well-designed, but the architect must understand the trade-offs at each layer.

RWO vs RWX is a workload classification decision, not just a storage decision: If you're using RWX (EFS) for a stateful service because you want to scale horizontally, you are likely dealing with an application that should be redesigned to use a proper database with connection pooling (PgBouncer) and read replicas — not shared filesystem hacks. RWX is appropriate for genuinely shared read-heavy content (ML training datasets, CMS media libraries, shared configuration directories), not as a workaround for application architecture problems.

StatefulSets are not enough for production databases: StatefulSets give you stable identities and persistent storage, but they do not give you: automated failover, replication management, backup scheduling, or point-in-time recovery. For TicketFlow's PostgreSQL, consider the Zalando Postgres Operator or CloudNativePG — these are Kubernetes operators that manage PostgreSQL clusters as first-class objects, including streaming replication, automated failover, and scheduled backups to S3. The StatefulSet you write manually is good for learning and single-replica deployments; operators are what production clusters use.

Storage cost matters at scale: EBS gp3 costs approximately $0.08/GB-month. 100 GB = $8/month. Fine. But if you have 50 microservices each with a 100 GB EBS volume, that is $4,800/month in storage alone — before IOPS costs. Audit your PVC sizes regularly. Use VolumeSnapshot and restore to right-size over-provisioned volumes. Consider AWS RDS for relational databases in production — managed RDS often costs less total than self-managed PostgreSQL on EKS when you factor in the operational burden (patches, backups, failover, monitoring).

Backup strategy is orthogonal to storage strategy: A Retain reclaimPolicy is not a backup. A Retain policy means you will not lose data due to accidental deletion — it does not protect against data corruption, ransomware, or application bugs that write bad data. For TicketFlow's PostgreSQL: use pgBackRest or Barman for WAL archiving to S3. Use VolumeSnapshot for crash-consistent EBS snapshots (point-in-time). Test restores quarterly.

Production Readiness Checklist

  • StorageClass uses reclaimPolicy: Retain for all production databases
  • StorageClass uses volumeBindingMode: WaitForFirstConsumer for zone-aware EBS provisioning
  • allowVolumeExpansion: true set on StorageClass so volumes can grow without recreating
  • All databases deployed as StatefulSets, not Deployments
  • PostgreSQL PGDATA set to a subdirectory within the mount (e.g., /data/pgdata), not the root
  • spec.securityContext.fsGroup set correctly so non-root processes can write to mounted volumes
  • EBS volumes encrypted at rest using KMS (set encrypted: "true" in StorageClass parameters)
  • PVC sizes right-sized — over-provisioning wastes money; under-provisioning causes outages
  • Helm-managed PVCs annotated with helm.sh/resource-policy: keep to survive helm uninstall
  • Backup strategy implemented: WAL archiving to S3 (pgBackRest) AND EBS snapshots (VolumeSnapshot)
  • Restore procedure documented and tested — know your RTO before an incident, not during
  • PVC storage usage monitored in Grafana — alert at 80% full before hitting the limit
  • StatefulSet podManagementPolicy: OrderedReady for databases (not Parallel, which can corrupt replication)
  • Node affinity or topology spread constraints configured if pods must stay in specific AZs

Interview Questions

These questions are asked in senior engineer and platform engineer interviews. Click each to reveal a full answer.

Q1: What is the difference between a Volume, a PersistentVolume, and a PersistentVolumeClaim in Kubernetes?

These three concepts form a layered abstraction for storage in Kubernetes, and they are frequently confused:

Volume is storage defined inline inside a Pod spec. It shares the Pod's lifecycle — it is created when the pod starts and destroyed when the pod dies. Volumes are used for ephemeral shared storage between containers in the same pod (emptyDir), injecting configuration (configMap, secret), or accessing host paths (hostPath). Volumes do not survive pod deletion.

PersistentVolume (PV) is a cluster-level storage resource that exists independently of any pod. It represents actual storage — an EBS disk, an NFS share, a Ceph volume. It is provisioned either manually by an administrator or automatically by a StorageClass's provisioner. A PV has a lifecycle independent of any workload.

PersistentVolumeClaim (PVC) is a namespace-scoped resource that represents an application's request for storage. The developer specifies how much storage they need and what access mode is required. The Kubernetes control plane binds the PVC to a matching PV (or creates one dynamically). The pod references the PVC by name in its spec. The PVC decouples the application from the specific storage implementation — the developer doesn't need to know if the backend is EBS, EFS, or Ceph.

The binding: One PVC binds to exactly one PV (for RWO). The binding is exclusive for RWO volumes. For RWX volumes like EFS, multiple PVCs can reference the same underlying EFS file system, but each claim is still its own PVC-PV pair.

Q2: What are the Kubernetes storage access modes and when would you use each?

Access modes define how a volume can be mounted across nodes:

  • ReadWriteOnce (RWO) — The volume can be mounted read-write by a single node at a time. Multiple pods on the same node can access it. This is the most common mode for block storage like AWS EBS, Azure Disk, and GCE PD. Use for: databases (PostgreSQL, MySQL), single-replica stateful services. Attempting to mount an RWO EBS volume on two nodes simultaneously results in a Multi-Attach error.
  • ReadOnlyMany (ROX) — The volume can be mounted read-only by many nodes simultaneously. Use for: pre-built static content that multiple pods need to read (ML model files, static assets, reference datasets). Typically backed by NFS, EFS, or CephFS.
  • ReadWriteMany (RWX) — The volume can be mounted read-write by many nodes simultaneously. Use for: shared application data that multiple pods write to (CMS uploads, shared logs, ML training data staging). Backed by NFS, AWS EFS, Azure Files, CephFS. Avoid for databases — concurrent writes from multiple processes require application-level locking.
  • ReadWriteOncePod (RWOP) — Added in Kubernetes 1.22. Stricter than RWO: only a single specific pod can mount the volume read-write. Even multiple pods on the same node cannot share it. Use for: workloads where you need a guarantee that exactly one pod has access at any time, even within a node (critical financial transaction processors, etc.).

Key rule: The access mode must be supported by the underlying storage driver. EBS only supports RWO (and RWOP). EFS supports RWX. Requesting RWX from an EBS StorageClass will result in a failed PVC binding.

Q3: Why use a StatefulSet instead of a Deployment for a database?

A Deployment treats all pods as identical, interchangeable units — exactly what you want for stateless services. A StatefulSet provides additional guarantees that databases require:

  • Stable, unique pod identity — StatefulSet pods are named name-0, name-1, etc. This identity is preserved across restarts and rescheduling. Deployment pods get random hash suffixes that change on every restart. For database clusters with named replicas (PostgreSQL primary, replica-1, replica-2), stable names are essential for replication configuration.
  • Stable network identity — with a headless Service, each StatefulSet pod gets a stable DNS name: postgres-0.postgres-headless.ticketflow.svc.cluster.local. Deployment pods have no stable DNS per pod. Databases often need to address specific replicas (primary vs replica) by stable hostname.
  • Per-pod PVC binding via volumeClaimTemplates — each StatefulSet replica gets its own dedicated PVC automatically. postgres-0 always mounts data-postgres-0, never data-postgres-1. With a Deployment, all pods fight over the same PVC. On RWO storage, only one pod wins. On RWX storage, all pods share one filesystem — incorrect for databases that need isolated data directories.
  • Ordered lifecycle management — StatefulSets start pods in order (0 before 1 before 2) and stop them in reverse order. This is critical for databases where the primary must be healthy before replicas start, and replicas should drain before the primary stops.

Can you use a Deployment for a single-replica database? Technically yes — if you manually create a PVC and reference it in the deployment's volumes. It works for single replica, but you lose stable naming, per-pod PVCs, and ordered lifecycle. When you later need to scale or add replicas, you will need to migrate to a StatefulSet anyway. Start with StatefulSet.

Q4: What is dynamic provisioning and how does a StorageClass enable it?

Before dynamic provisioning, storage was manually managed: a cluster administrator had to pre-create PersistentVolumes (by manually creating EBS volumes, NFS shares, etc.) and then developers claimed them with PVCs. This was slow, required admin involvement for every deployment, and frequently resulted in PVCs stuck in Pending because no matching PV was available.

Dynamic provisioning automates this: when a PVC is created referencing a StorageClass, the StorageClass's provisioner automatically creates the underlying storage and a corresponding PV — with no manual admin work required.

How it works technically:

  1. Developer creates a PVC referencing storageClassName: gp3-encrypted.
  2. The PV controller detects the PVC is unbound and has a StorageClass.
  3. The controller calls the StorageClass's provisioner (e.g., ebs.csi.aws.com) via the CSI (Container Storage Interface) API's CreateVolume RPC.
  4. The provisioner (the EBS CSI driver running as a pod in the cluster) calls the AWS API to create an EBS volume with the parameters defined in the StorageClass (gp3, encrypted: true, etc.).
  5. The provisioner creates a PV object in Kubernetes with a reference to the new EBS volume ID.
  6. The PV controller binds the PV to the PVC. The PVC status becomes Bound.
  7. The pod can now mount the PVC.

CSI (Container Storage Interface) is the standard API that decouples Kubernetes from storage drivers. Any storage vendor can write a CSI driver that implements the CSI spec and instantly work with any CSI-compatible Kubernetes cluster. This replaced the old in-tree volume plugins that required Kubernetes core changes to add new storage backends.

Q5: What happens to a PersistentVolume when its PVC is deleted — and how do you protect against accidental data loss?

When a PVC is deleted, what happens to the underlying PV and storage depends entirely on the StorageClass's reclaimPolicy:

  • Delete (default for dynamic provisioning) — The PV object is deleted AND the underlying storage (EBS volume, GCE PD, etc.) is deleted. This is the most dangerous setting. Any data stored on the volume is permanently destroyed. This is the default because it prevents storage leaks in dynamic environments, but it is catastrophic for production databases if triggered accidentally.
  • Retain — The PV object persists with status Released. The underlying storage (EBS volume) is preserved. No data is lost. An administrator must manually recover: they can create a new PVC referencing the existing PV's volume handle, or manually delete the PV object after verifying the data is no longer needed.

Multi-layer protection strategy for production:

  1. StorageClass reclaimPolicy: Retain — First line of defense. EBS volume survives PVC deletion.
  2. Helm resource policy annotation — Add helm.sh/resource-policy: keep to PVC metadata. This prevents helm uninstall from deleting the PVC even if it is managed by a Helm chart.
  3. Kubernetes resource finalizers — Some operators add finalizers to PVCs that prevent deletion until the finalizer is explicitly removed. The CloudNativePG operator does this.
  4. RBAC restrictions — Restrict who can delete PVCs in production namespaces. Developers should not have delete permission on PVCs in production.
  5. Velero or EBS snapshots — Even with Retain policy, protect against data corruption and ransomware with regular automated snapshots to a separate account.
Q6: When would you choose AWS EFS over EBS for Kubernetes storage — and what are the trade-offs?

The choice between EFS and EBS maps directly to the access mode your workload requires and the performance characteristics you need.

Choose EBS (gp3) when:

  • Your workload is a traditional relational database (PostgreSQL, MySQL) or any single-writer stateful service. Databases are single-writer by design — they do not need RWX. EBS gp3 delivers 3,000 IOPS baseline (up to 16,000 provisioned) and sub-millisecond latency, which is critical for database transaction performance.
  • The workload runs as a StatefulSet with one pod per replica having its own isolated storage.
  • You need guaranteed IOPS. EBS IOPS are provisioned; EFS IOPS scale with throughput mode and are less predictable under burst scenarios.
  • Cost predictability matters — EBS costs are predictable ($/GB provisioned). EFS scales automatically and can be cost-effective for variable-size datasets but more expensive than EBS for large, consistently accessed datasets.

Choose EFS when:

  • Multiple pods on multiple nodes need to read and write the same files simultaneously (RWX). This is EFS's defining capability — EBS simply cannot do this.
  • You have a Deployment (not StatefulSet) that needs to share state across replicas — e.g., a CMS that writes uploaded media files and multiple pods serve them.
  • You need storage that spans multiple Availability Zones. EBS is single-AZ; EFS is regional. If a node in us-east-1a fails and your pod rescheduled to us-east-1b, EFS continues working without re-provisioning.
  • ML training workloads where multiple training pods in a distributed job need to read a shared dataset.

Key trade-off to state in an interview: EFS provides higher availability (multi-AZ) and flexibility (RWX) at the cost of higher latency (~1–10ms vs <1ms for EBS) and unpredictable performance under heavy random I/O. For TicketFlow's PostgreSQL, EBS is the correct choice. For TicketFlow's notification-service shared attachment storage, EFS is the correct choice.
