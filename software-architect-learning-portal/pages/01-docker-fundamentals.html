Docker Fundamentals

Why Docker exists, how containers work, and why every developer needs to understand them.

The Story Behind Docker

📦 The Shipping Container Revolution

Before shipping containers, every cargo ship was a chaos of differently-shaped crates, barrels, and boxes. Dockhands had to figure out how to load each one manually. It was slow, error-prone, and nothing was standardized.

In 1956, Malcolm McLean invented the standardized shipping container. Suddenly it didn't matter what was inside — every container was the same size, stacked the same way, loaded by the same cranes. Ships got 10x faster to load. Ports became predictable. Global trade exploded.

Docker did the same thing for software. Before Docker, deploying an app meant fighting different OS versions, missing libraries, conflicting dependencies, and the classic "it works on my machine" disaster. Docker gave software a standard container: pack your app and everything it needs, ship it anywhere, run it the same way every time.

The Problem Docker Solves

🔥 The "It Works On My Machine" Problem

A developer builds an app on macOS with Node 18, OpenSSL 3.0, and specific library versions pinned in package-lock.json. The staging server runs Ubuntu with Node 16 and OpenSSL 1.1. Production runs RHEL 8 with yet another set of system libraries.

Each environment behaves differently. Cryptographic functions return different results across OpenSSL versions. Native addons compiled against Node 18 crash on Node 16. Bugs only appear in production — because that's the only environment that matters, and it's the hardest one to debug. Hours and sometimes days are lost chasing environment differences instead of shipping features.

What Docker solves: Docker packages the application and its entire runtime environment into one portable, self-contained unit. The same container image runs on a developer's MacBook, in the CI pipeline, on a staging server, and in production. Same behaviour, every time, everywhere.

What is Docker?

🥪 Beginner Analogy

Think of Docker like a lunchbox. You pack your lunch (your app), your fork (the runtime), your napkin (the libraries) — everything you need. Wherever you go — home, office, school — you eat the same lunch. The cafeteria (the server) doesn't need to have your specific food. It just needs to be able to hold a lunchbox.

Docker is an open-source platform that automates the deployment of applications inside lightweight, portable units called containers. Here are the four concepts you need to know from day one:

  • Image — The recipe or blueprint. Read-only. Like a frozen snapshot of a filesystem containing your app and all its dependencies. You build it once and distribute it everywhere.
  • Container — A running instance of an image. Like a lunchbox made from the recipe. You can spin up hundreds of containers from the same image simultaneously, and each is isolated from the others.
  • Registry — Where images are stored and shared. Docker Hub is the public default. Companies use private registries like Nexus, Amazon ECR, or Harbor. Think of it as a grocery store stocked with pre-made lunchboxes.
  • Docker Engine — The software installed on a machine that creates and runs containers. Like the factory that manufactures lunchboxes from a recipe on demand.

How Docker Actually Works

Docker is not magic — it is a user-friendly interface on top of three powerful Linux kernel features that have existed for years:

1. Namespaces — Isolation

Linux namespaces control what a container can see. Each container gets its own isolated view of the system. There are six types of namespaces Docker uses:

  • pid — The container sees only its own processes. Process ID 1 inside the container is the container's init process, not the host's init.
  • net — The container gets its own network stack: its own IP address, routing table, and port space. Port 80 inside one container does not conflict with port 80 inside another.
  • mnt — The container has its own filesystem mount points. It cannot see the host filesystem unless you explicitly mount a volume.
  • uts — The container can have its own hostname.
  • ipc — Inter-process communication is isolated.
  • user — User and group IDs can be mapped differently inside the container than on the host.

2. Control Groups (cgroups) — Resource Limits

Cgroups limit what a container can use. Without cgroups, one runaway container could consume all CPU, all memory, or all disk I/O on the host, starving other containers. Cgroups enforce hard limits: this container may use at most 2 CPU cores and 512 MB of RAM. The kernel enforces these limits at the hardware level — if the container exceeds its memory limit, the OOM (Out of Memory) killer terminates its process.

3. Union File System (OverlayFS) — Layered Images

Docker images are composed of layers. Each instruction in a Dockerfile creates a new read-only layer stacked on top of the previous one. When a container runs, Docker adds a thin writable layer on top. Multiple containers can share the same base layers without duplicating them on disk. A base node:18-alpine layer downloaded once is reused by every container that uses it as a base — saving gigabytes of disk space and dramatically speeding up image pulls.

Docker Engine Architecture

Docker Engine has two main parts that you interact with directly:

  • Docker daemon (dockerd) — Runs as a background system service. Manages containers, images, networks, and volumes. Exposes a REST API over a Unix socket (/var/run/docker.sock).
  • Docker CLI (docker) — The command-line tool you type commands into. Sends requests to the daemon via the REST API. The CLI itself does no heavy lifting — it is purely a client.

Under the hood, dockerd delegates to containerd (a lower-level container runtime) which in turn calls runc, the actual component that talks to Linux kernel namespaces and cgroups to create and start containers.

💡 Windows / macOS Note

Containers require Linux kernel features. Docker Desktop on Windows and macOS runs a lightweight, optimized Linux VM (using WSL 2 on Windows, HyperKit/QEMU on macOS). When you run a container on your Mac, it runs inside this VM — completely transparent to you, but important to understand for performance and networking edge cases.

VMs vs Containers

The most common question from engineers new to containers: "How is this different from a VM?" The short answer: VMs virtualize hardware; containers virtualize the operating system. Here is the full breakdown:

Key Components Deep Dive

Docker Image

A Docker image is a read-only template used to create containers. Images are built from a series of instructions in a Dockerfile, where each instruction produces a new immutable layer stacked on top of the previous one.

  • Layers are cached — if a layer hasn't changed, Docker reuses the cached version. This makes rebuilds extremely fast during development.
  • Layers are shared — if two images both use node:18-alpine as a base, that base layer is stored only once on disk and in the registry.
  • Tagged with name:version — e.g., nginx:1.25, myapp:v2.1.0, python:3.11-slim. The tag identifies a specific version of an image.
  • Stored in registries — Docker Hub, Nexus, ECR, ACR, Harbor. You push images to a registry and pull them from anywhere.
  • Inspectable — run docker history nginx:1.25 to see every layer, its size, and the instruction that created it.
Docker Container

A container is a running (or stopped) instance of an image. You can think of it as an image brought to life.

  • Writable layer — containers get a thin, writable layer on top of the image's read-only layers. All writes during container execution go here. This layer is discarded when the container is removed.
  • Own network interface — each container gets an IP address on Docker's internal network. Containers communicate via service names (in Docker Compose or Kubernetes).
  • Own process space — the container's main process is PID 1. When PID 1 exits, the container stops.
  • Ephemeral by default — if you stop and remove a container, any data written to its filesystem is gone. For persistent data, use volumes.
  • One image, many containers — production typically runs multiple identical containers of the same image for redundancy and load distribution.
Docker Registry

A registry is a storage and distribution system for Docker images. Think of it as an artifact repository specifically for container images.

  • Docker Hub (hub.docker.com) — the public default registry. Free for public images. Widely used for official images (nginx, node, postgres, etc.). Rate-limited for anonymous pulls.
  • Nexus Repository — self-hosted, enterprise-grade artifact manager. Most large organizations use Nexus as a private registry and proxy cache. Supports Docker, Maven, npm, PyPI, Helm charts, and more in one tool.
  • Amazon ECR — AWS-native registry. Tight IAM integration. Private by default. Natural choice when running on EKS or ECS.
  • Azure ACR — Azure-native registry. Integrates with AKS and Azure DevOps pipelines.
  • Harbor — open-source, self-hosted registry with built-in image scanning (via Trivy or Clair), role-based access control, and replication. Security-focused teams often choose Harbor over Nexus for container images specifically.

In practice, most enterprises proxy Docker Hub through their internal registry (Nexus or Harbor) to avoid rate limits, cache commonly used base images, and enforce image scanning policies before images reach production.

Docker Engine Internals

Docker Engine is not a single binary — it is a stack of components, each responsible for a different layer of abstraction:

  • dockerd (Docker daemon) — the top-level daemon. Handles Docker API requests, manages images (pull/push/tag), manages container lifecycle, networks, and volumes. The entry point for everything you do with the docker CLI.
  • containerd — an industry-standard container runtime (CNCF project). dockerd delegates container lifecycle management to containerd. Kubernetes also uses containerd directly (without Docker). This is why Docker is not a required dependency for Kubernetes.
  • runc — the lowest-level component. A CLI tool that implements the OCI (Open Container Initiative) runtime specification. runc is what actually makes the system calls to create namespaces, apply cgroups, and start the container process. If you removed Docker and containerd, you could run a container with just runc and a pre-prepared filesystem bundle.
  • Docker CLI (docker) — the command you type. Sends REST API requests to dockerd over the Unix socket at /var/run/docker.sock. The CLI itself is thin — no container logic lives here.

Enterprise Scenario: TicketFlow

🏢 Real-World Example

At TicketFlow (our enterprise ticketing platform), the team had 6 microservices running in production:

  • user-service — Node.js app requiring Node 18 and specific bcrypt native addon
  • ticket-service — Java app requiring JDK 17 and specific JDBC driver versions
  • payment-service — Python app requiring Python 3.11 and specific cryptography library with C extensions
  • order-service — Go binary (statically compiled, but still has OS-level dependencies)
  • notification-service — Node.js app, but on a different version than user-service due to legacy constraints
  • api-gateway — NGINX with custom Lua modules compiled against a specific OpenResty version

Before Docker: Setting up a dev environment took 2–3 days. Every service had different runtime requirements that conflicted. The payment-service's Python crypto libraries needed a specific version of OpenSSL that broke the api-gateway's Lua compilation. New developers gave up and submitted tickets to senior engineers to fix their environment. Production deployments failed because servers had wrong library versions installed by previous manual setup. Configuration drift meant no two production servers were truly identical.

After Docker: Each service has its own Dockerfile that encapsulates its exact runtime. A docker compose up command brings all 6 services up in under 60 seconds. New developers are productive on day one — no installation documentation to follow, no missing dependencies to hunt. CI builds the exact same images that run in production. Deployments are predictable because the artifact is immutable.

Container Lifecycle

Understanding the container lifecycle is critical for debugging. A container can be in several states, and knowing the transitions helps you understand why docker ps shows nothing but docker ps -a shows your container as "Exited".

stateDiagram-v2 [*] --> Created: docker create Created --> Running: docker start Running --> Paused: docker pause Paused --> Running: docker unpause Running --> Stopped: docker stop Stopped --> Running: docker start Running --> Dead: OOM / crash Stopped --> Removed: docker rm Dead --> Removed: docker rm Removed --> [*]
  • Created — container exists as a configuration but no process has started. Allocated but not consuming CPU or memory.
  • Running — PID 1 is executing. Network interfaces are up. The container is consuming resources.
  • Paused — all processes are frozen using SIGSTOP. The container still exists in memory but consumes no CPU. Useful for taking snapshots.
  • Stopped (Exited) — PID 1 has terminated (either gracefully via SIGTERM or by completing its work). The container still exists on disk — its writable layer is preserved. You can restart it or inspect it.
  • Dead — Docker tried to stop the container but failed (e.g., OOM kill by the kernel). The container is in an error state.
  • Removed — docker rm deletes the container and its writable layer. The image it was based on is not deleted.

Behind the Scenes

What happens when you run docker run nginx?
  1. Docker CLI sends an HTTP POST request to the Docker daemon via the Unix socket at /var/run/docker.sock (on Linux) or the named pipe on Windows.
  2. The daemon checks its local image cache for nginx:latest. Images are stored in /var/lib/docker/overlay2/ on Linux.
  3. If the image is not found locally: the daemon contacts the configured registry (Docker Hub by default). It downloads the image manifest (a JSON file describing layers), then pulls each layer that isn't already cached — downloading only the delta.
  4. The daemon instructs containerd to create a container. containerd prepares an OCI runtime bundle: an extracted filesystem (assembled from OverlayFS layers) and a configuration JSON describing namespaces, cgroups, and environment.
  5. containerd calls runc with the bundle path. runc makes Linux system calls to: create a new network namespace, create a new PID namespace, create a new mount namespace and mount the OverlayFS root, apply cgroup limits for CPU and memory, and drop capabilities to reduce privileges.
  6. runc executes the container's entrypoint — in this case, the nginx master process — as PID 1 inside the new namespace.
  7. Docker daemon assigns the container an IP address on the docker0 bridge network (typically in the 172.17.0.0/16 range). A veth pair is created connecting the container's network namespace to the host's bridge.
  8. The daemon returns the container ID to the CLI, which prints it to your terminal.

This entire sequence — from command to running nginx — takes under 1 second for a locally cached image. It is orders of magnitude faster than booting a VM because no kernel is being initialized: only a process is being isolated.

Common Mistakes

❌ Mistake 1: Using :latest in Production

The latest tag is mutable — it points to whatever was most recently pushed to the registry. Today nginx:latest is 1.25. Tomorrow your CI pipeline could silently pull 1.27 with breaking configuration changes. Your staging tested 1.25; production runs 1.27. You have no idea until something breaks at 2am.

Fix: Always use specific version tags (nginx:1.25.3-alpine). In production, pin to immutable digests: FROM nginx@sha256:abc123...

❌ Mistake 2: Running Containers as Root

By default, processes inside a container run as root (UID 0). If a vulnerability in your application allows an attacker to escape the container namespace (container escapes are rare but possible), they land on the host with root privileges. This is catastrophic.

Fix: Add USER 1001 in your Dockerfile before the CMD instruction. Create a dedicated non-root user for your app. Most official base images (like node:18-alpine) already include a node user you can switch to.

❌ Mistake 3: Storing Secrets in Images

Developers write ENV DB_PASSWORD=mypassword in a Dockerfile or COPY .env /app/.env. Anyone who can pull the image — or run docker history myimage — can read the password in plain text. Even if you delete the file in a later layer, the secret is still visible in the image history.

Fix: Never put secrets in Dockerfiles or image layers. Pass secrets as environment variables at container runtime (docker run -e DB_PASSWORD=$DB_PASSWORD), use Docker secrets in Swarm, Kubernetes Secrets, or a secrets manager like AWS Secrets Manager or HashiCorp Vault.

❌ Mistake 4: Not Using .dockerignore

A COPY . . instruction without a .dockerignore file copies your entire project directory into the image: node_modules (gigabytes), .git (exposing your entire commit history), .env (secrets), IDE config files, test fixtures. This makes images enormous and leaks sensitive information.

Fix: Always create a .dockerignore file at the project root. At minimum, ignore: node_modules, .git, .env, *.md, dist, coverage, .DS_Store.

❌ Mistake 5: Thinking Containers Are Virtual Machines

Containers share the host kernel. They provide process-level isolation, not hardware-level isolation. A sufficiently privileged container process can interact with the host kernel through system calls. Running untrusted code (customer workloads, user-uploaded scripts) in a standard Docker container is not safe enough.

Fix: For untrusted or multi-tenant workloads, use gVisor (Google's kernel sandbox), Kata Containers (lightweight VMs that look like containers), or Firecracker microVMs. For trusted internal workloads, standard Docker security with non-root users, dropped capabilities, and seccomp profiles is sufficient.

Architect Perspective

🏛️ Architect Note

Containers are not just a developer convenience tool — they are an architectural primitive. When your organization adopts Docker, you are implicitly adopting a set of architectural principles:

  • Immutable infrastructure — containers are never modified in place. A new version means a new container built from a new image. This eliminates configuration drift, the silent killer of production stability. You can always rebuild the exact environment from the Dockerfile.
  • Separation of concerns — the application team owns the Dockerfile and decides what goes into the image. The platform team owns the runtime environment (Kubernetes, resource limits, network policies). Clear, contractual boundaries between teams.
  • Build once, deploy anywhere — the artifact (the image) is built once in CI and promoted through environments unchanged. What you test in staging is precisely what you deploy to production. No "re-building for prod" that introduces subtle differences.
  • The container is the unit of deployment — not a JAR file, not a WAR, not a tarball, not a Chef recipe. The container includes the application binary, its dependencies, its configuration defaults, and its runtime. Everything needed to run is self-contained.

Trade-off to understand: Container images are larger than raw binaries. A "Hello World" Go binary is 2 MB. The same binary in a minimal scratch-based Docker image is around 5 MB. With a distroless base, still around 5 MB. For serverless functions or WebAssembly edge deployments, the overhead matters — consider WASM runtimes (Fermyon, WasmEdge) instead of containers for those use cases. For everything else, the portability gain easily justifies the size overhead.

Production Readiness Checklist

Before deploying any containerized application to production, verify all of the following:

  • Images use specific version tags — not :latest
  • Containers run as a non-root user (add USER 1001 in Dockerfile)
  • .dockerignore excludes node_modules, .git, .env, and credential files
  • Images stored in a private registry (Nexus, ECR, Harbor) — not just Docker Hub
  • Images scanned for CVEs before deployment (Trivy, Snyk, or registry-integrated scanner)
  • Container CPU and memory limits set (prevents noisy neighbour problems)
  • Health checks defined in Dockerfile (HEALTHCHECK) or compose file
  • Sensitive data passed as environment variables at runtime — not baked into image layers
  • Base images pinned to digest (FROM node:18-alpine@sha256:...) for full reproducibility in production
  • Image size monitored and kept minimal — use multi-stage builds to exclude build tools from final image

Hands-On Lab: Your First Container

🧪 Lab — Requires Docker Desktop installed

This lab walks you through the essential Docker commands you will use daily. Run each block in your terminal and observe the output carefully.

Step 1: Verify Docker is working

# Pull and run the official hello-world image
docker run hello-world

# You should see a message confirming Docker is installed correctly.
# Notice: Docker pulled the image from Docker Hub automatically.

Step 2: Run a real web server

# Run nginx web server in the background (-d = detached)
# Map host port 8080 to container port 80 (-p host:container)
# Give it a name so we can reference it easily
docker run -d -p 8080:80 --name my-nginx nginx:1.25

# Verify it is running
docker ps

# Visit http://localhost:8080 in your browser — you should see the nginx welcome page

Step 3: Inspect and interact with the running container

# See the nginx access logs
docker logs my-nginx

# Follow logs in real time (Ctrl+C to stop)
docker logs -f my-nginx

# Enter the running container with an interactive shell
docker exec -it my-nginx bash

# Inside the container — explore the nginx configuration
ls /etc/nginx/conf.d/
cat /etc/nginx/conf.d/default.conf

# Exit the container shell (you are back on your host)
exit

Step 4: Inspect container metadata and resource usage

# Inspect full container details: IP address, mounts, env vars, restart policy
docker inspect my-nginx

# See real-time CPU and memory usage (--no-stream = snapshot, not live)
docker stats my-nginx --no-stream

Step 5: Work with images

# Stop and remove the container (stop = SIGTERM, rm = delete the container)
docker stop my-nginx
docker rm my-nginx

# Always use specific version tags — never :latest in production
docker pull nginx:1.25-alpine

# List all locally stored images
docker images

# Inspect the image layers — notice how thin alpine is vs full debian
docker history nginx:1.25-alpine

# See total disk space used by Docker
docker system df

What to observe during this lab:

  1. How fast the nginx container starts — compare this to how long a VM would take to boot.
  2. The size difference between nginx:1.25 (Debian-based, ~140 MB) and nginx:1.25-alpine (Alpine-based, ~40 MB).
  3. What docker inspect reveals about networking: look for the IPAddress and Networks fields.
  4. How docker exec -it gives you a fully working shell inside a running process — you're inside the container's isolated filesystem and process namespace.
  5. That docker stop and docker rm are two separate steps: stop preserves the container for restart; rm deletes it permanently.

Interview Questions

These are the most frequently asked Docker questions in software engineering and DevOps interviews. Click each question to reveal a complete answer.

Q1: What is the difference between a Docker image and a Docker container?

An image is a read-only template — a layered snapshot of a filesystem containing an application, its dependencies, and configuration. Images are immutable: once built, they do not change. They are stored in registries and pulled to machines that need to run them.

A container is a running (or stopped) instance of an image. When you start a container, Docker adds a thin writable layer on top of the image's read-only layers. Any changes made inside the container (files written, packages installed) go into this writable layer and are discarded when the container is removed.

One image can spawn hundreds of containers simultaneously. Each container is isolated from the others, even though they share the same underlying image layers.

Analogy: The image is a class definition in object-oriented programming. The container is an object instance created from that class. You instantiate many objects from one class; you run many containers from one image.

Q2: How does Docker differ from a virtual machine?

Virtual machines virtualize hardware. Each VM runs a complete operating system — a full kernel, device drivers, init system, and userland — on top of a hypervisor (VMware, Hyper-V, KVM). VMs start in 1–2 minutes and consume several GB each because they carry an entire OS.

Docker containers virtualize the operating system. Containers share the host machine's Linux kernel and use kernel features (namespaces and cgroups) for isolation. They start in under a second and consume only the memory needed by the application itself — megabytes, not gigabytes.

The trade-off: VMs provide stronger, hardware-level isolation. Containers provide far better density and speed. In practice, organizations use both: VMs for the cloud infrastructure nodes, containers for the application workloads running on those nodes.

Rule of thumb: Use VMs when you need to run different OS kernels, need full security isolation for untrusted code, or have legacy apps that require their own OS. Use containers for modern application packaging, microservices, and CI/CD pipelines.

Q3: What are Docker namespaces and cgroups?

Namespaces provide isolation — they control what a container can see. Each container gets its own view of the system through six namespace types:

  • pid — isolated process tree (container's PID 1 is not the host's PID 1)
  • net — isolated network stack (own IP, own ports)
  • mnt — isolated filesystem mounts
  • uts — isolated hostname
  • ipc — isolated inter-process communication
  • user — isolated user and group IDs

Control groups (cgroups) provide resource limits — they control what a container can use. Cgroups enforce maximum CPU time, maximum memory, and maximum disk I/O a container can consume. The kernel enforces these limits. If a container exceeds its memory limit, the kernel's OOM killer terminates its process.

Together: namespaces give the container the illusion of being the only process on the machine; cgroups ensure it cannot starve other containers of resources. Both features exist in the Linux kernel and predate Docker — Docker simply provides a convenient interface to configure and use them.

Q4: Why should you not use :latest in production?

The latest tag is mutable — it is just a pointer that is reassigned with every push. There is nothing inherently "latest" about it; it is simply the default tag when you don't specify one.

Today, nginx:latest might resolve to nginx 1.25.3. Tomorrow, after an upstream push, it could resolve to nginx 1.27.0 with changed default configuration, deprecated directives, or removed modules your app depends on.

If your CI pipeline tests with 1.25.3 but your production deployment pulls 1.27.0 (because a week passed between CI and deploy), you are running untested code in production. This is exactly the environment consistency problem Docker was built to solve — and using :latest reintroduces it.

Best practice by environment:

  • Development: nginx:1.25 (minor version pinned, allows patch updates)
  • Staging/Production: nginx:1.25.3-alpine (fully pinned)
  • Critical production: nginx@sha256:abc123... (immutable digest — cannot change even if the tag is overwritten)
Q5: What happens when a container is stopped — is data lost?

When a container is stopped (via docker stop), its process is terminated but the container still exists. Its writable layer — including any files written during its lifetime — is preserved on disk. You can restart the container with docker start and the data will still be there.

When a container is removed (via docker rm), the container and its writable layer are permanently deleted. Any data that was written to the container's filesystem during its lifetime is gone forever.

For data that must survive container removal, Docker provides two mechanisms:

  • Named volumes — managed by the Docker daemon, stored in /var/lib/docker/volumes/ on the host. Survive container deletion. Referenced by name: -v mydata:/app/data. The recommended approach for databases and stateful services.
  • Bind mounts — map a specific host directory into the container: -v /host/path:/container/path. Useful during development (mount source code into a container so changes are reflected immediately) but less portable than named volumes.

Rule: always use named volumes for any data you care about. For databases (PostgreSQL, MySQL, MongoDB), volumes are not optional — they are mandatory.
