Observability

Metrics, logs, and traces — the three pillars that let you understand what your distributed system is doing in production.

The Story Behind Observability

🔭 Flying Blind at 35,000 Feet

In the early days of aviation, pilots flew by looking out the window and trusting their gut. When clouds obscured visibility, they crashed. The solution was not to tell pilots to "try harder" — it was to instrument the cockpit. Altimeter, airspeed indicator, artificial horizon, fuel gauge. Now pilots knew the state of their aircraft even when they could not see it directly.

Distributed systems are the same. A monolith running on one server was like a small propeller plane — you could SSH in, run top, check a log file, and understand what was wrong. But a microservices system running 30 services across 100 pods on Kubernetes is a commercial jetliner. You cannot look out the window. A user reports "checkout is slow." Which service? Which pod? Which database query? Which network hop? You need instruments.

Observability is the practice of building those instruments into your system from the beginning. Not as an afterthought, but as a first-class engineering concern. When production breaks at 3am, the engineers who sleep through the incident are the ones who invested in observability. The ones paging through raw logs for 4 hours are the ones who did not.

What is Observability?

🩺 Beginner Analogy

Think of observability like a doctor diagnosing a patient. The doctor cannot look inside the body directly — so they use instruments. A stethoscope measures heart rate (metrics). The patient's symptom history is their medical record (logs). A blood test traces specific substances through the body over time (traces). Each instrument tells you something different. Together, they give the doctor enough information to diagnose and treat the problem without cutting the patient open.

Observability for software is the same idea: you give your running system instruments so you can understand its health and behaviour from the outside.

Observability is built on three complementary pillars. Each answers a different question:

  • Metrics — "How much?" Numerical measurements over time. CPU at 87%. Requests per second: 1,200. Error rate: 0.3%. P95 latency: 340ms. Metrics are cheap to store and fast to query, but tell you only aggregated numbers — not why a specific user's request failed.
  • Logs — "What happened?" Timestamped, structured records of discrete events. "User 4821 attempted checkout at 14:32:11 — payment declined: card expired." Logs tell you exactly what occurred for a specific request, but they are expensive to store at scale and hard to query across millions of records.
  • Traces — "Where did the time go?" A trace follows a single request as it flows through every service. It shows that user 4821's checkout took 3.2 seconds total: 40ms in api-gateway, 80ms in order-service, 2.9 seconds in payment-service (database query took 2.7 seconds). Traces connect cause and effect across service boundaries.

In practice, the real power comes from connecting all three: a Grafana dashboard shows P95 latency spiking (metrics), you click the spike and jump to correlated logs (logs), you pick a slow trace and see the exact database query causing the bottleneck (traces).

Technical Deep Dive

Prometheus — Metrics Collection

Prometheus is an open-source time-series database and monitoring system. It works on a pull model: Prometheus periodically scrapes HTTP endpoints (called targets) that expose metrics in a plain-text format. Every service you want to monitor must expose a /metrics endpoint.

Scrape configuration — Prometheus discovers targets either via static configuration or via service discovery (Kubernetes, AWS, Consul). For Kubernetes, Prometheus automatically discovers pods and services annotated with prometheus.io/scrape: "true".

Metric types:

  • Counter — monotonically increasing value that only goes up. Total HTTP requests, total errors. Never decremented. Use rate() to turn it into a per-second rate.
  • Gauge — a value that goes up and down. Current memory usage, number of active connections, queue depth.
  • Histogram — samples observations and counts them in configurable buckets. Used for latency and request sizes. Enables calculating percentiles (P50, P95, P99). Most important for SLO tracking.
  • Summary — similar to histogram but calculates quantiles on the client side. Less flexible than histograms for aggregation; prefer histograms.

PromQL — Prometheus Query Language. Used to query and aggregate time-series data. Core functions:

  • rate(metric[5m]) — per-second rate of a counter over last 5 minutes
  • histogram_quantile(0.95, rate(http_request_duration_seconds_bucket[5m])) — P95 latency
  • sum by (service) (rate(http_requests_total[5m])) — requests/sec grouped by service label
  • increase(metric[1h]) — total increase over 1 hour

Recording rules — pre-computed expressions saved back as new time series. Expensive queries (P95 across all pods) run once per scrape interval and stored, so dashboards load instantly instead of recomputing on every request.

Alerting rules — conditions that trigger alerts when a PromQL expression is true for a specified duration. Alerts fire to Alertmanager, which handles routing, deduplication, grouping, and notification.

Grafana — Visualization

Grafana is the visualization layer. It connects to data sources (Prometheus, Loki, Tempo, Elasticsearch, CloudWatch, and 60+ others) and renders interactive dashboards. Key concepts:

  • Data sources — configured connections to backends. One Grafana instance can query Prometheus for metrics, Loki for logs, and Tempo for traces simultaneously.
  • Dashboards — collections of panels arranged on a grid. Each panel runs a query against a data source and visualizes the result (time-series graph, stat, gauge, heatmap, table, logs, trace viewer).
  • Alerting — Grafana has its own alerting engine (introduced in Grafana 8). Alerts can query any data source, not just Prometheus. Notifications go to PagerDuty, Slack, OpsGenie, email, and webhooks.
  • Annotations — vertical lines on dashboards marking events (deployments, incidents, config changes). Automatically correlate metric spikes with deployments.
  • Variables — template variables that make dashboards dynamic. A $service variable lets a single dashboard show metrics for any service by selecting from a dropdown.

Loki — Log Aggregation

Loki is Grafana Labs' log aggregation system, designed to be cost-efficient and Prometheus-like in its approach. Unlike Elasticsearch, Loki does not index the full content of logs — it only indexes metadata labels (pod name, namespace, service, level). The log content is compressed and stored cheaply in object storage (S3).

  • Promtail — the log shipping agent. Runs as a DaemonSet on every Kubernetes node, tails container logs from /var/log/pods/, adds labels (pod, namespace, container), and pushes to Loki.
  • LogQL — Loki's query language. Two types: log queries (return log lines) and metric queries (derive metrics from logs). Example: {app="payment-service"} |= "ERROR" returns all error log lines from payment-service. rate({app="payment-service"} |= "ERROR" [5m]) returns error rate as a metric.
  • Label strategy — the most important Loki design decision. High-cardinality labels (user ID, trace ID, request ID) must not be labels — they go in the log body instead. Labels should be low-cardinality: service, environment, pod, namespace, level.

Jaeger / Tempo — Distributed Tracing

Distributed tracing follows a request through every service it touches. The two most common tools in the Kubernetes ecosystem are Jaeger (CNCF project) and Grafana Tempo (Prometheus-like, label-based, cheaply stores traces in object storage).

  • Span — the basic unit. Represents a single operation (HTTP call, database query, cache lookup). Has a start time, duration, operation name, service name, and key-value tags.
  • Trace — a collection of spans with parent-child relationships forming a tree. The root span is the entry point (incoming HTTP request to api-gateway). Each downstream call creates a child span.
  • Context propagation — trace context (trace ID, span ID) is passed between services via HTTP headers (W3C Trace Context standard: traceparent header, or legacy B3 headers). Services read the incoming context and create child spans under the same trace.
  • Sampling — tracing every request is expensive. Head-based sampling decides at the first service (1% of requests). Tail-based sampling decides after the full trace is collected (sample all traces with errors or high latency). Tail-based is more powerful but complex.
  • OpenTelemetry (OTel) — the open-source standard for instrumentation. Replaces vendor-specific SDKs (Jaeger client, Zipkin client) with a unified API. Services are instrumented with OTel; the OTel Collector receives data and forwards to Jaeger, Tempo, or any backend. Adopt OTel now — it is the industry standard.

Alertmanager — Alert Routing

Prometheus Alertmanager receives alerts fired by Prometheus (and Grafana) and handles routing, grouping, deduplication, silencing, and notification. Key concepts:

  • Routes — a tree of routing rules that match alerts by label (service, severity, team) and direct them to specific receivers (Slack channel, PagerDuty service, OpsGenie team).
  • Grouping — multiple alerts about the same outage are grouped into a single notification to avoid alert storms. Group by alertname and service means 50 firing pod alerts become one message.
  • Inhibition — suppress lower-severity alerts when a high-severity alert is active. If the entire cluster is down, suppress individual pod alerts.
  • Silences — mute alerts during maintenance windows without disabling them.
  • Repeat interval — how often to re-notify for a still-firing alert. Critical alerts: every 5 minutes. Warning alerts: every 4 hours.

SLO, SLI, and SLA

These three terms define reliability contracts and drive alert design:

  • SLI (Service Level Indicator) — the metric you measure. The ratio of good requests to total requests. P95 latency. Availability (uptime percentage). A specific, measurable number derived from real user traffic.
  • SLO (Service Level Objective) — the internal target for your SLI. "99.9% of payment-service requests will complete in under 500ms over any 30-day window." Agreed internally. Failing an SLO means you owe the team an explanation and a postmortem, not a refund.
  • SLA (Service Level Agreement) — the external, contractual commitment. "We guarantee 99.9% availability. If we fail, you get a 10% credit." Involves lawyers, contracts, and financial penalties. SLOs should be tighter than SLAs to give you a safety buffer.
  • Error budget — the allowed failure budget derived from the SLO. A 99.9% availability SLO for a 30-day window gives you 43.2 minutes of allowable downtime. If you have used 40 minutes this month, you have 3.2 minutes left. Burn the budget too fast and the team freezes new deployments until it recovers.

Enterprise Scenario: TicketFlow P95 Latency SLO

🏢 TicketFlow — payment-service SLO Crisis

TicketFlow's engineering team has defined the following SLO for payment-service:

"99.5% of payment-service HTTP requests will complete with HTTP 2xx status within 500ms (P95), measured over a rolling 30-day window."

This gives them an error budget of: 30 days × 24 hours × 60 minutes × 0.5% = 216 minutes of allowed latency SLO violations per month.

On a Tuesday afternoon, the Grafana dashboard shows the P95 latency for payment-service jumping from 180ms to 2.3 seconds. The Alertmanager fires two alerts:

  • PaymentServiceP95LatencyHigh — severity: warning — Slack #alerts
  • PaymentServiceErrorBudgetBurnRate — severity: critical — PagerDuty on-call

Investigation workflow using the three pillars:

  1. Metrics first — Grafana shows P95 latency spike started at 14:32 UTC. CPU and memory for payment-service pods are normal. The spike correlates with a Grafana annotation showing a new deployment at 14:30 UTC.
  2. Traces second — The on-call engineer opens Tempo in Grafana and filters for traces from payment-service with duration > 1s. Every slow trace shows a single span: postgres.query — SELECT * FROM payment_methods WHERE user_id = ? taking 2.1 seconds. The new deployment added a missing index removal during "cleanup".
  3. Logs to confirm — LogQL query in Grafana: {app="payment-service"} |= "slow query" | logfmt | duration > 1s confirms hundreds of slow query log lines starting at exactly 14:30 UTC.

Root cause: a database migration in the new deployment dropped an index on payment_methods.user_id by mistake. The fix (re-adding the index) is deployed at 15:10 UTC — 38 minutes of SLO violation. Error budget remaining: 178 minutes. The incident is logged, a postmortem scheduled, and migration review added to the deployment checklist.

PromQL Queries — SLO Tracking

# P95 latency for payment-service over last 5 minutes
histogram_quantile(
  0.95,
  sum by (le) (
    rate(http_request_duration_seconds_bucket{
      job="payment-service",
      status_code=~"2.."
    }[5m])
  )
)

# Request success rate (for availability SLI)
sum(rate(http_requests_total{job="payment-service", status_code=~"2.."}[5m]))
/
sum(rate(http_requests_total{job="payment-service"}[5m]))

# Error budget burn rate (30d window, 0.5% error budget)
1 - (
  sum(rate(http_requests_total{job="payment-service", status_code=~"2.."}[30d]))
  /
  sum(rate(http_requests_total{job="payment-service"}[30d]))
) / 0.005

# P95 latency broken down by HTTP method
histogram_quantile(
  0.95,
  sum by (le, method) (
    rate(http_request_duration_seconds_bucket{job="payment-service"}[5m])
  )
)

  rate(http_requests_total{namespace="ticketflow"}[5m])
)

Prometheus Recording and Alerting Rules

# /etc/prometheus/rules/ticketflow-slo.yml
groups:
  - name: ticketflow_recording_rules
    interval: 30s
    rules:
      # Pre-compute P95 latency per job — used by dashboards and alert rules
      - record: job:http_request_duration_p95:rate5m
        expr: |
          histogram_quantile(0.95,
            sum by (le, job) (
              rate(http_request_duration_seconds_bucket[5m])
            )
          )

      # Pre-compute error rate per job
      - record: job:http_error_rate:rate5m
        expr: |
          sum by (job) (rate(http_requests_total{status_code=~"5.."}[5m]))
          /
          sum by (job) (rate(http_requests_total[5m]))

  - name: ticketflow_slo_alerts
    rules:
      # P95 latency SLO violation — payment-service
      - alert: PaymentServiceP95LatencyHigh
        expr: |
          job:http_request_duration_p95:rate5m{job="payment-service"} > 0.5
        for: 2m
        labels:
          severity: warning
          team: payments
          service: payment-service
        annotations:
          summary: "payment-service P95 latency exceeds 500ms SLO"
          description: "Current P95: {{ $value | humanizeDuration }}. SLO threshold: 500ms."
          runbook: "https://wiki.ticketflow.io/runbooks/payment-service-latency"

      # Fast burn rate — error budget being consumed 14x faster than allowed
      - alert: PaymentServiceErrorBudgetBurnRate
        expr: |
          (
            1 - (
              sum(rate(http_requests_total{job="payment-service", status_code=~"2.."}[1h]))
              / sum(rate(http_requests_total{job="payment-service"}[1h]))
            )
          ) / 0.005 > 14
        for: 5m
        labels:
          severity: critical
          team: payments
          service: payment-service
        annotations:
          summary: "payment-service error budget burn rate critical"
          description: "Burning error budget {{ $value | humanize }}x faster than allowed."

      # Ticket service down
      - alert: TicketServiceDown
        expr: up{job="ticket-service"} == 0
        for: 1m
        labels:
          severity: critical
          team: platform
        annotations:
          summary: "ticket-service is unreachable"
          description: "Prometheus cannot scrape ticket-service at {{ $labels.instance }}"

app = fastapi.FastAPI()

# Auto-instrument FastAPI — creates spans for every HTTP request
FastAPIInstrumentor.instrument_app(app)

# Auto-instrument SQLAlchemy — creates spans for every DB query
SQLAlchemyInstrumentor().instrument()

@app.post("/v1/payments")
async def process_payment(payment: PaymentRequest):
    # Custom span for business-level tracing
    with tracer.start_as_current_span("payment.process") as span:
        span.set_attribute("payment.amount", payment.amount)
        span.set_attribute("payment.currency", payment.currency)
        span.set_attribute("payment.user_id", payment.user_id)

        try:
            result = await charge_card(payment)
            span.set_attribute("payment.status", "success")
            span.set_attribute("payment.transaction_id", result.transaction_id)
            return result
        except CardDeclinedError as e:
            span.set_attribute("payment.status", "declined")
            span.set_attribute("payment.decline_reason", str(e))
            span.record_exception(e)
            raise

Prometheus Metrics Instrumentation — order-service (Go)

// order_service/metrics/metrics.go
package metrics

import (
	"github.com/prometheus/client_golang/prometheus"
	"github.com/prometheus/client_golang/prometheus/promauto"
)

var (
	// Counter — total HTTP requests by method, path, and status code
	HTTPRequestsTotal = promauto.NewCounterVec(
		prometheus.CounterOpts{
			Name: "http_requests_total",
			Help: "Total number of HTTP requests handled by order-service",
		},
		[]string{"method", "path", "status_code"},
	)

	// Histogram — request latency in seconds with SLO-aligned buckets
	HTTPRequestDuration = promauto.NewHistogramVec(
		prometheus.HistogramOpts{
			Name:    "http_request_duration_seconds",
			Help:    "HTTP request duration in seconds",
			// Buckets designed around SLO threshold (500ms)
			Buckets: []float64{0.01, 0.05, 0.1, 0.25, 0.5, 0.75, 1.0, 2.5, 5.0},
		},
		[]string{"method", "path"},
	)

	// Gauge — current number of orders being processed
	ActiveOrders = promauto.NewGauge(prometheus.GaugeOpts{
		Name: "order_service_active_orders",
		Help: "Number of orders currently being processed",
	})

	// Counter — order outcomes
	OrdersProcessedTotal = promauto.NewCounterVec(
		prometheus.CounterOpts{
			Name: "orders_processed_total",
			Help: "Total orders processed by outcome",
		},
		[]string{"outcome"}, // "success", "failed", "cancelled"
	)
)

// Middleware records HTTP metrics for every request
func InstrumentHandler(next http.Handler) http.Handler {
	return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		timer := prometheus.NewTimer(
			HTTPRequestDuration.WithLabelValues(r.Method, r.URL.Path),
		)
		defer timer.ObserveDuration()

		rw := &responseWriter{ResponseWriter: w, status: 200}
		next.ServeHTTP(rw, r)

		HTTPRequestsTotal.WithLabelValues(
			r.Method,
			r.URL.Path,
			strconv.Itoa(rw.status),
		).Inc()
	})
}

Behind the Scenes

How does Prometheus's pull model work internally?

Every Prometheus scrape cycle, the Prometheus server iterates over all configured targets and sends an HTTP GET to each target's /metrics endpoint. The response is plain text in the Prometheus exposition format:

# HELP http_requests_total Total HTTP requests
# TYPE http_requests_total counter
http_requests_total{method="GET",path="/health",status_code="200"} 12847
http_requests_total{method="POST",path="/v1/payments",status_code="200"} 4821
http_requests_total{method="POST",path="/v1/payments",status_code="500"} 12

Prometheus parses this text, timestamps each sample with the current Unix timestamp (millisecond precision), and writes them to its TSDB (time-series database). The TSDB uses a chunk-based WAL (Write-Ahead Log) + block files on disk. Old data is compacted into increasingly larger blocks. Default retention is 15 days.

For Kubernetes service discovery, Prometheus queries the Kubernetes API server to discover pods, services, and endpoints. It re-discovers targets every 5 minutes by default and applies relabeling rules to transform Kubernetes metadata (annotations, labels) into Prometheus labels.

The pull model has important implications for security: Prometheus needs network access to all scrape targets, but targets never need credentials to push data out. Firewall rules control who can scrape whom. Compare this to push models (StatsD, InfluxDB line protocol) where agents push outward — useful for ephemeral jobs but harder to control at scale.

How does distributed trace context propagation actually work?

When a request arrives at api-gateway (NGINX), the OTel instrumentation creates a root span with a globally unique 128-bit trace ID and a 64-bit span ID. These are encoded in HTTP headers using the W3C Trace Context specification:

traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01
#            version-traceId(32 hex chars)-parentSpanId(16 hex chars)-flags

When api-gateway calls order-service, it includes this header in the downstream HTTP request. order-service's OTel SDK reads the traceparent header, extracts the trace ID and parent span ID, and creates a new child span with its own unique span ID but the same trace ID. It then passes the updated traceparent header when it calls payment-service.

Each service's OTel SDK batches spans and sends them to the OTel Collector via the OTLP protocol (gRPC or HTTP). The Collector processes, samples, and forwards spans to Tempo. Tempo stores all spans together and indexes them by trace ID, enabling the full trace tree to be reconstructed at query time.

The trace ID is also logged in application logs and included in HTTP response headers (X-Trace-ID), enabling correlation: when a user reports a slow request and provides a request ID, the engineer can look up logs by trace ID and see the exact trace in Grafana Tempo.

How does Loki's label-based indexing compare to Elasticsearch?

Elasticsearch indexes every word in every log line. This makes full-text search extremely fast but requires significant memory and CPU for the inverted index. A 1TB log corpus might require 300GB of index data in memory for fast queries. The operational overhead is substantial.

Loki takes the opposite approach: it indexes only the stream labels (pod name, service, namespace, level) — not the log content. Log content is compressed with Snappy or gzip and stored in cheap object storage (S3, GCS). This makes Loki 10-100x cheaper to operate than Elasticsearch for the same log volume.

The trade-off: queries that filter on log content (LogQL's |= operator) must scan through compressed chunks rather than hitting an inverted index. This is slower for arbitrary full-text search but fast enough for the structured queries engineers actually need in production (filter by service + error level + time range).

The key design principle: put high-cardinality data in log lines, not labels. If you add user_id="abc123" as a label, Loki creates a new index entry for every unique user ID — millions of entries. Instead, structure your logs as {"level":"error","user_id":"abc123","action":"checkout"} and use LogQL's | json | user_id="abc123" to filter after label selection. This keeps the index small and queries efficient.

Common Mistakes

❌ Mistake 1: High-Cardinality Labels in Prometheus

Adding user_id, order_id, or request_id as Prometheus metric labels creates a new time series for every unique value. With 1 million users, a single metric explodes into 1 million time series. Prometheus's memory usage is proportional to the number of active time series — this will OOM-kill your Prometheus instance and corrupt your metrics database.

Fix: Prometheus labels should be low-cardinality (fewer than ~1,000 unique values). Service name, HTTP method, status code, endpoint path are fine. User IDs, order IDs, and trace IDs are not. For high-cardinality data, use logs or traces — that is exactly what they are designed for.

❌ Mistake 2: Alerting on Symptoms Rather than Causes

Teams write alerts for every internal metric they can think of: CPU > 80%, JVM heap > 70%, database connections > 50. These are causes, not symptoms. A database with 50 connections might be perfectly healthy. A pod at 85% CPU might be processing a batch job normally. These alerts fire constantly, get ignored, and train engineers to dismiss pages as noise — until the real incident is buried in alert fatigue.

Fix: Alert on user-visible symptoms first. P95 latency, error rate, and availability SLO burn rate are user-visible. CPU and heap are context for debugging — put them on dashboards, not alerts. The Google SRE team's rule: alert only when a human needs to take action.

❌ Mistake 3: Setting SLOs Without Error Budgets

Teams set a 99.9% availability SLO but never track error budget consumption. The SLO becomes a vanity metric — looked at quarterly in a management review rather than daily as an engineering decision-making tool. Without error budget tracking, the SLO has no operational teeth.

Fix: Calculate your 30-day error budget (for 99.9%: 43.2 minutes). Track its burn rate in real time via PromQL. When burn rate exceeds 2x the sustainable rate for 1 hour (fast burn) or 6x for 6 hours (slow burn), alert the team. When the budget is nearly exhausted, halt non-critical deployments until it recovers.

❌ Mistake 4: Tracing Only the Happy Path

Engineers instrument their main business logic but skip error paths, retries, and background jobs. When an incident occurs in a retry loop or a background payment reconciliation job, there are no traces to show what happened. The benefit of distributed tracing is greatest precisely when something goes wrong.

Fix: Instrument all code paths: error handlers, retry logic, background workers, queue consumers, and cron jobs. Use OpenTelemetry's span.record_exception(e) and span.set_status(StatusCode.ERROR) in every exception handler. Configure tail-based sampling to keep 100% of error traces regardless of overall sample rate.

❌ Mistake 5: Storing Logs Unstructured

Teams write logs as free-form strings: print(f"Payment failed for user {user_id} with error {error}"). These are human-readable but machine-hostile. Loki's |= operator can filter them, but parsing and aggregating requires fragile regex. Correlating with traces is impossible because no trace ID field exists.

Fix: Use structured logging (JSON) from day one. Every log line should be a JSON object with fields like level, timestamp, service, trace_id, span_id, user_id, action, duration_ms, error. In Python, use structlog. In Node.js, use pino. In Java, use Logback with logstash-logback-encoder. In Go, use zap or slog.

Architect Perspective

🏛️ Architect Note

Observability is not a tool you install — it is a discipline you build into your system from the beginning. The difference between a team that resolves incidents in 15 minutes and one that takes 4 hours is almost always observability investment made months earlier.

Design observability into your system architecture:

  • Correlation IDs everywhere — every incoming request gets a UUID at the API gateway. This ID propagates as a trace ID in distributed tracing headers, as a field in every log line, and as a label in metrics for request-specific debugging. When a customer reports a problem with "reference number TF-4821," you can pull all logs, metrics, and traces for that exact request in seconds.
  • The USE method for infrastructure — for every resource (CPU, memory, network, disk), measure Utilization (what % is being used), Saturation (is it queued or waiting?), and Errors. This gives a consistent framework for diagnosing infrastructure bottlenecks.
  • The RED method for services — for every service, measure Rate (requests per second), Errors (errors per second), and Duration (P50/P95/P99 latency). These three metrics tell you whether a service is healthy from a user's perspective. They are the foundation of SLOs.
  • OpenTelemetry is the answer to vendor lock-in — instrument once with OTel, export to any backend. Starting with Jaeger today does not lock you into Jaeger forever. OTel separates instrumentation (in your code) from the observability backend (Jaeger, Tempo, Honeycomb, Datadog). This is the right long-term architectural decision.
  • Observability costs money — plan for it — Prometheus with 30-day retention for 100 services needs 50–200GB of disk. Loki ingesting 100GB/day of logs needs S3 and query capacity. Distributed tracing with 1% sample rate on 10,000 RPS still generates significant storage. Build observability costs into capacity planning from the start, not as a surprise afterthought when the storage bill arrives.

The maturity ladder: Start with metrics (cheapest, highest value). Add structured logs with trace IDs next. Add distributed tracing last — it requires code changes and OTel SDK integration in every service. For a new system: instrument metrics in week 1, structured logs in week 2, tracing as you add services. For a legacy system: add Prometheus exporters externally (NGINX exporter, JVM exporter, DB exporter) before touching application code.

Production Readiness Checklist

Before declaring a service production-ready from an observability standpoint, verify all of the following:

  • Every service exposes a /metrics endpoint and is scraped by Prometheus
  • RED metrics (Rate, Errors, Duration) are instrumented for every service endpoint
  • Histograms are used for latency (not summaries) with SLO-aligned bucket boundaries
  • Recording rules pre-compute expensive P95/P99 queries to keep dashboards fast
  • Alerting rules cover P95 latency SLO breach, error rate SLO breach, and service down
  • Error budget burn rate alerts exist for both fast burn (1h window) and slow burn (6h window)
  • Alertmanager routes critical alerts to PagerDuty and warning alerts to Slack
  • Grafana dashboards include the service's RED metrics, error budget, and recent deployment annotations
  • All log output is structured JSON with trace_id, level, service, and timestamp fields
  • Promtail (or Fluent Bit) is configured to collect and ship logs to Loki
  • Services are instrumented with OpenTelemetry SDK and traces exported to the OTel Collector
  • Trace IDs are included in HTTP response headers (X-Trace-ID) for user-facing error correlation
  • SLO document exists with defined SLI, SLO target, and error budget calculation
  • Runbook exists for every critical alert with diagnostic steps and escalation path
  • Grafana data sources link Prometheus metrics, Loki logs, and Tempo traces for correlated investigation

Interview Questions

These questions cover the observability concepts most frequently tested in senior engineer and architect-level interviews. Click each to reveal a complete answer.

Q1: What are the three pillars of observability and what does each tell you?

The three pillars are metrics, logs, and traces, each answering a different diagnostic question:

  • Metrics — "How much / how many?" Numerical aggregates over time: requests/second, P95 latency, error rate, CPU usage. Cheap to store, fast to query, but cannot explain why a specific request failed. Best for dashboards, SLO tracking, and alerting.
  • Logs — "What happened?" Timestamped records of discrete events for specific requests. Rich context (user ID, stack trace, SQL query text) but expensive at scale. Best for post-incident investigation and audit trails.
  • Traces — "Where did the time go across service boundaries?" A trace follows a single request through every service it touches, showing parent-child call relationships and per-service latency breakdown. Best for diagnosing latency issues and understanding inter-service dependencies in microservices architectures.

The real power is in connecting all three. A Grafana dashboard shows P95 latency spiking (metrics). You click the spike and jump to correlated error logs (logs). You find a specific failing request and follow its trace to identify which service and which database query caused the slowdown (traces).

Q2: Explain SLI, SLO, and SLA with concrete examples. What is an error budget?

SLI (Service Level Indicator) is the specific metric you measure as a proxy for user happiness. For TicketFlow's payment-service: "the ratio of HTTP requests completing within 500ms with a 2xx status code, measured over the last 5 minutes." It is a number you can compute from real traffic data using PromQL.

SLO (Service Level Objective) is the internal target for that SLI. "99.5% of payment-service requests will meet the SLI over any rolling 30-day window." Failing an SLO triggers a postmortem and potentially a deployment freeze — not a refund. SLOs are agreed to internally between engineering and product.

SLA (Service Level Agreement) is the external contractual commitment, usually to enterprise customers. "We guarantee 99.9% API availability. Failure incurs a 25% service credit." SLAs involve legal contracts and financial penalties. SLOs should be tighter than SLAs (99.5% SLO vs 99% SLA) to create a safety buffer.

Error budget is derived from the SLO: the allowed failure quota. A 99.5% SLO over 30 days gives: 30 days × 24 hours × 60 min × 0.5% = 216 minutes of allowed SLO violation. The error budget is a shared resource between engineering (stability = preserve budget) and product (new features = risk burning budget). When the budget is exhausted, no new deployments until it recovers. This creates a data-driven, emotionally neutral conversation about risk.

Q3: How does Prometheus differ from push-based monitoring systems? What are the trade-offs?

Prometheus uses a pull model: Prometheus actively scrapes HTTP /metrics endpoints on each target at a configured interval (default 15s). Targets are passive — they expose metrics and wait to be scraped.

Push-based systems (StatsD, InfluxDB line protocol, Graphite) require each service to actively send metrics to a central collector. Services need the collector's address and credentials.

Pull model advantages:

  • Prometheus always knows if a target is reachable — if scraping fails, the target is marked as down and an alert fires. With push, a silent service looks healthy.
  • Access control is simpler: Prometheus needs read access to scrape targets; targets don't need outbound permissions to the monitoring system.
  • Easier debugging: you can manually curl http://service/metrics to see exactly what Prometheus sees.

Pull model disadvantages:

  • Short-lived jobs (batch jobs, CI pipelines) finish before Prometheus can scrape them. Solution: Prometheus Pushgateway — jobs push to the gateway; Prometheus scrapes the gateway.
  • Services behind strict firewalls (Prometheus cannot reach them) require a Prometheus federation or remote write proxy.
  • At massive scale (millions of services), a single Prometheus instance cannot scrape all targets. Solution: Prometheus federation, Thanos, or Cortex for horizontal scaling.
Q4: Why is high cardinality dangerous in Prometheus? How do you handle high-cardinality data?

Prometheus stores each unique combination of metric name + label values as a separate time series. Cardinality is the total number of unique time series. Memory usage scales linearly with cardinality — typically 1–2 KB per active time series.

A label with high cardinality (many unique values) creates an explosion. A user_id label with 1 million unique users creates 1 million time series for a single metric. At 1KB each, that is 1GB of RAM just for one metric. Production Prometheus instances have crashed from cardinality explosions, taking down monitoring exactly when it is most needed.

Safe Prometheus label cardinality:

  • HTTP method: 5 values (GET, POST, PUT, DELETE, PATCH) — safe
  • Status code class: 5 values (2xx, 3xx, 4xx, 5xx, errors) — safe
  • Service name: 20 services — safe
  • User ID: 1,000,000 users — dangerous
  • Request ID: unique per request — catastrophic

How to handle high-cardinality data:

  • Logs — user IDs, order IDs, request IDs belong in log lines, not metric labels. Use structured logging and query with LogQL.
  • Traces — individual request timing and path data belong in trace spans, not metrics.
  • ClickHouse / columnar stores — for analytics use cases requiring high-cardinality metric exploration (Honeycomb and Grafana Mimir support this).
Q5: What is distributed tracing and how does context propagation work across services?

Distributed tracing captures the full lifecycle of a request as it flows through multiple services in a distributed system. Without tracing, you might know a request took 3 seconds total but have no idea which of the 8 services it touched caused the slowdown.

Core concepts:

  • Span — a single operation. HTTP request to a service, database query, cache lookup. Has start time, duration, service name, operation name, and key-value attributes.
  • Trace — a tree of spans with parent-child relationships representing the complete execution path of one request. All spans in a trace share the same trace ID.
  • Context propagation — the mechanism for passing trace identity between services.

How context propagation works step by step:

  1. api-gateway receives an HTTP request. OTel SDK creates a root span with a randomly-generated 128-bit trace ID and a 64-bit span ID.
  2. When api-gateway calls order-service (outbound HTTP), the OTel SDK automatically injects trace context into the HTTP request headers using the W3C Trace Context standard: traceparent: 00-{traceId}-{spanId}-{flags}.
  3. order-service's OTel SDK reads the traceparent header, extracts the trace ID and parent span ID, and creates a new child span with the same trace ID but a new span ID.
  4. This repeats for every downstream call. All spans are exported asynchronously to the OTel Collector.
  5. The tracing backend (Tempo/Jaeger) stores all spans and reconstructs the trace tree using the trace ID and parent-span-ID relationships.

In the OpenTelemetry model, this is all handled automatically by HTTP client/server instrumentation libraries — no manual header management required in application code.

Q6: How would you design alerting to avoid alert fatigue on a team of 10 engineers?

Alert fatigue occurs when engineers receive so many non-actionable pages that they start ignoring alerts. Solving it requires discipline in alert design, not just tooling.

Principles for alert design that avoids fatigue:

  • Alert on symptoms, not causes — P95 latency > SLO threshold and error rate > SLO budget burn rate are symptoms users feel. CPU > 80% is a cause that may or may not affect users. Causes belong on dashboards; symptoms trigger pages.
  • Every alert must be actionable — if an alert fires and the on-call engineer's only response is "I'll wait and see if it recovers," the alert should not fire. Either it needs a runbook action or it should be demoted to a dashboard-only metric.
  • Multi-window burn rate alerting (Google's approach) — instead of alerting when P95 > 500ms for 2 minutes, alert when the error budget burn rate is unsustainable. A fast-burn alert (budget burning 14x faster than sustainable over 1 hour) pages immediately. A slow-burn alert (budget burning 5x over 6 hours) creates a ticket for investigation during business hours. This catches incidents at different severity levels with appropriate urgency.
  • Severity tiers with matching response expectations — Critical (P1): wake someone up, resolve in <30 minutes. Warning (P2): Slack notification, review in <4 hours. Info: dashboard only, no page. Do not mix these — no one should be woken up for a P2.
  • Regularly review and prune alerts — run a monthly audit. Every alert that fired and required no action in the last 30 days should be deleted or demoted. Alerts should earn their place on the on-call runbook.

For a team of 10: aim for fewer than 5 P1 alert types per service, a shared on-call rotation with documented runbooks for every alert, and a weekly 15-minute alert review in the team standup to discuss false positives.

Q7: How do you correlate metrics, logs, and traces in a production incident investigation?

Effective incident investigation follows a structured flow from broad to specific, using each observability pillar in the right order:

1. Start with metrics (broad view, fast): Open the service RED dashboard. Which metric spiked? When did it start? What changed? Correlate with deployment annotations. Is it one service or multiple? One region or global? Metrics give you the "what and when" in seconds.

2. Use traces to find the bottleneck (service-level detail): In Grafana Explore, open Tempo. Filter for traces with high latency or errors in the affected time window. Pick representative slow/erroring traces. The trace waterfall view immediately shows which service and which specific operation (DB query, external API call, internal function) is taking the most time. Traces give you the "where" — the exact span causing the problem.

3. Use logs to understand the cause (request-level detail): Take the trace ID from the slow trace. Open Loki in Grafana Explore. Query: {namespace="ticketflow"} |= "trace_id=YOUR_TRACE_ID". This pulls every log line across all services for that specific request. You can now see the exact error message, the SQL query that was slow, the external API response that failed.

Technical enablers for this workflow:

  • Grafana's "Explore" view with cross-data-source navigation (click a metric spike → jump to logs for that time range)
  • Exemplars — Prometheus can store trace IDs alongside histogram samples, linking specific high-latency data points directly to their trace in Tempo
  • Structured logs with trace_id as a field (not just embedded in message text) so LogQL can filter by it
  • Deployment annotations on Grafana dashboards correlating changes with metric behavior

With this setup, the time from "alert fires" to "root cause identified" drops from hours to minutes — the difference between a P1 incident lasting 2 hours versus 15 minutes.
