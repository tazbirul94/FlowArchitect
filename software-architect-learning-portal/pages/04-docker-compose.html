Docker Compose

Define and run multi-container applications with a single YAML file. Go from zero to a full local development environment in minutes.

Story

Running a modern application isn't just one container. It's an app server, a database, a cache, a message queue, maybe a search engine. Starting them all manually — docker run this, docker run that, remember the right network, the right environment variable, the right port mapping — is like building flat-pack furniture without the instruction sheet. Every time.

Docker Compose is the instruction sheet. One YAML file describes your entire application stack: all services, how they connect, what storage they use, what environment they need. One command — docker compose up — starts everything in the right order. docker compose down tears it all down cleanly. Share the file with a teammate and they have your entire environment in minutes.

What is Docker Compose?

Beginner

Think of Docker Compose like a recipe for your entire kitchen setup. The recipe (docker-compose.yml) says: "heat the oven to 180° (start the database), prepare the dough (start the app), preheat the pan (start Redis)". You don't do each step manually every time — you just say "make dinner" and it follows the recipe.

Docker Compose reads a single YAML file and manages the full lifecycle of a multi-container application. Services are defined declaratively — you describe the desired state and Compose figures out how to achieve it. It handles creating networks so services can talk to each other by name, creating volumes so data persists across restarts, and starting services in the right order when dependencies are declared.

Compose ships as part of the Docker CLI since Docker Desktop 3.4 and Docker Engine 20.10. You invoke it as docker compose (with a space, the modern V2 plugin) rather than the older docker-compose (hyphenated, the legacy standalone tool). The two are functionally equivalent for most use cases, but V2 is the maintained version.

Full docker-compose.yml Anatomy

The following is a complete, production-informed docker-compose.yml for the TicketFlow platform — a realistic multi-service application. Every key is annotated so you can understand not just what it does, but why it's there.

# docker-compose.yml for TicketFlow Platform
# Version is optional in modern Compose (v2+), but shown for clarity
version: '3.9'

services:

  # ─────────────────────────────────────────────
  # API Gateway (NGINX reverse proxy)
  # ─────────────────────────────────────────────
  api-gateway:
    image: nginx:1.25-alpine
    container_name: ticketflow-gateway
    ports:
      - "80:80"          # host:container port mapping
      - "443:443"
    volumes:
      - ./nginx/nginx.conf:/etc/nginx/nginx.conf:ro   # bind mount, read-only
    networks:
      - frontend
    depends_on:
      user-service:
        condition: service_healthy   # wait until healthy, not just started
      ticket-service:
        condition: service_healthy
    restart: unless-stopped

  # ─────────────────────────────────────────────
  # User Service (Node.js)
  # ─────────────────────────────────────────────
  user-service:
    build:
      context: ./user-service           # Dockerfile location
      dockerfile: Dockerfile
      args:
        NODE_ENV: production
    image: ticketflow/user-service:dev   # tag the built image
    container_name: ticketflow-users
    ports:
      - "3001:3001"       # expose for local debugging
    environment:
      - NODE_ENV=development
      - PORT=3001
      - DB_HOST=postgres
      - DB_PORT=5432
      - DB_NAME=ticketflow
      - DB_USER=ticketflow
      - DB_PASSWORD=devpassword123    # dev only — use secrets in prod
      - REDIS_URL=redis://redis:6379
      - JWT_SECRET=dev-jwt-secret-change-in-prod
    env_file:
      - .env.local          # override with local .env file
    networks:
      - frontend
      - backend
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy
    healthcheck:
      test: ["CMD", "wget", "--no-verbose", "--tries=1", "--spider", "http://localhost:3001/health"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 15s
    restart: unless-stopped
    # Resource limits (Compose v2)
    deploy:
      resources:
        limits:
          memory: 512M
          cpus: '0.5'
        reservations:
          memory: 256M

  # ─────────────────────────────────────────────
  # Ticket Service (Java Spring Boot)
  # ─────────────────────────────────────────────
  ticket-service:
    build:
      context: ./ticket-service
      dockerfile: Dockerfile
    container_name: ticketflow-tickets
    ports:
      - "8080:8080"
    environment:
      SPRING_PROFILES_ACTIVE: development
      SPRING_DATASOURCE_URL: jdbc:postgresql://postgres:5432/ticketflow
      SPRING_DATASOURCE_USERNAME: ticketflow
      SPRING_DATASOURCE_PASSWORD: devpassword123
      SPRING_REDIS_HOST: redis
      SPRING_REDIS_PORT: 6379
      KAFKA_BOOTSTRAP_SERVERS: kafka:9092
    networks:
      - backend
    depends_on:
      postgres:
        condition: service_healthy
      kafka:
        condition: service_started
    healthcheck:
      test: ["CMD", "wget", "--no-verbose", "--tries=1", "--spider", "http://localhost:8080/actuator/health"]
      interval: 30s
      timeout: 15s
      retries: 3
      start_period: 60s     # Java apps take time to start
    restart: unless-stopped

  # ─────────────────────────────────────────────
  # PostgreSQL Database
  # ─────────────────────────────────────────────
  postgres:
    image: postgres:16-alpine
    container_name: ticketflow-db
    ports:
      - "5432:5432"       # expose for DB GUI tools (TablePlus, DBeaver)
    environment:
      POSTGRES_DB: ticketflow
      POSTGRES_USER: ticketflow
      POSTGRES_PASSWORD: devpassword123
      PGDATA: /var/lib/postgresql/data/pgdata
    volumes:
      - postgres_data:/var/lib/postgresql/data   # named volume — data persists
      - ./db/init:/docker-entrypoint-initdb.d:ro  # SQL init scripts
    networks:
      - backend
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ticketflow -d ticketflow"]
      interval: 10s
      timeout: 5s
      retries: 5
      start_period: 10s
    restart: unless-stopped

  # ─────────────────────────────────────────────
  # Redis Cache
  # ─────────────────────────────────────────────
  redis:
    image: redis:7-alpine
    container_name: ticketflow-redis
    ports:
      - "6379:6379"
    command: redis-server --appendonly yes --requirepass devpassword123
    volumes:
      - redis_data:/data
    networks:
      - backend
    healthcheck:
      test: ["CMD", "redis-cli", "--pass", "devpassword123", "ping"]
      interval: 10s
      timeout: 5s
      retries: 5
    restart: unless-stopped

  # ─────────────────────────────────────────────
  # Kafka (Message Broker)
  # ─────────────────────────────────────────────
  kafka:
    image: confluentinc/cp-kafka:7.5.0
    container_name: ticketflow-kafka
    ports:
      - "9092:9092"
    environment:
      KAFKA_NODE_ID: 1
      KAFKA_LISTENER_SECURITY_PROTOCOL_MAP: CONTROLLER:PLAINTEXT,PLAINTEXT:PLAINTEXT
      KAFKA_ADVERTISED_LISTENERS: PLAINTEXT://kafka:9092
      KAFKA_LISTENERS: PLAINTEXT://0.0.0.0:9092,CONTROLLER://0.0.0.0:9093
      KAFKA_CONTROLLER_LISTENER_NAMES: CONTROLLER
      KAFKA_CONTROLLER_QUORUM_VOTERS: 1@kafka:9093
      KAFKA_PROCESS_ROLES: broker,controller
      KAFKA_AUTO_CREATE_TOPICS_ENABLE: 'true'
      CLUSTER_ID: 'MkU3OEVBNTcwNTJENDM2Qk'
    volumes:
      - kafka_data:/var/lib/kafka/data
    networks:
      - backend

  # ─────────────────────────────────────────────
  # Notification Service
  # ─────────────────────────────────────────────
  notification-service:
    build: ./notification-service
    container_name: ticketflow-notifications
    environment:
      KAFKA_BOOTSTRAP_SERVERS: kafka:9092
      SMTP_HOST: mailhog
      SMTP_PORT: 1025
    networks:
      - backend
    depends_on:
      - kafka
    restart: unless-stopped
    profiles:
      - full     # only start with: docker compose --profile full up

  # ─────────────────────────────────────────────
  # MailHog (dev email catcher)
  # ─────────────────────────────────────────────
  mailhog:
    image: mailhog/mailhog:latest
    container_name: ticketflow-mail
    ports:
      - "1025:1025"   # SMTP
      - "8025:8025"   # Web UI at http://localhost:8025
    networks:
      - backend
    profiles:
      - full

# ─────────────────────────────────────────────
# Named Volumes
# ─────────────────────────────────────────────
volumes:
  postgres_data:
    driver: local
  redis_data:
    driver: local
  kafka_data:
    driver: local

# ─────────────────────────────────────────────
# Networks
# ─────────────────────────────────────────────
networks:
  frontend:
    driver: bridge
  backend:
    driver: bridge

Essential Compose Commands

All commands below assume you are in the directory that contains your docker-compose.yml file. Compose automatically discovers the file by walking up the directory tree.

# Start all services (builds images if needed)
docker compose up

# Start in background (detached mode)
docker compose up -d

# Build images and start
docker compose up -d --build

# Start specific services only
docker compose up -d postgres redis

# Start with a specific profile
docker compose --profile full up -d

# View running services and their status
docker compose ps

# Follow logs from all services
docker compose logs -f

# Follow logs from specific service
docker compose logs -f user-service

# Run a command in a service container
docker compose exec user-service bash

docker compose up -d --scale user-service=3

docker compose stop

Override Files

Docker Compose supports a file layering pattern that keeps base configuration clean while allowing per-environment customizations. Compose merges multiple files together at startup — values in later files override values in earlier ones, and new keys are added.

docker-compose.yml          ← base config (committed to git)
docker-compose.override.yml ← local dev overrides (gitignored)
docker-compose.staging.yml  ← staging config
docker-compose.prod.yml     ← prod config

When you run docker compose up with no -f flags, Compose automatically reads docker-compose.yml and then merges docker-compose.override.yml on top of it. This automatic merge only applies to these two specific filenames — all other environment files must be specified explicitly with -f.

A typical override file for local development adds hot-reload volume mounts, debug logging, and extra tooling without touching the base file that every team member shares:

# docker-compose.override.yml
# Automatically merged with docker-compose.yml
# Add to .gitignore

services:
  user-service:
    volumes:
      - ./user-service/src:/app/src   # hot reload: mount source code
    environment:
      - LOG_LEVEL=debug
      - NODE_ENV=development
    command: npm run dev              # override to use nodemon

  ticket-service:
    volumes:
      - ./ticket-service/src:/app/src

  postgres:
    ports:
      - "5432:5432"    # expose for local DB tools

# Use: docker compose up -d (auto-merges override)
# Use specific file: docker compose -f docker-compose.yml -f docker-compose.staging.yml up
Production Tip

Commit a docker-compose.override.yml.example to your repository showing the shape of the override file with placeholder values. Developers copy it to docker-compose.override.yml and fill in their local paths. The actual override file goes in .gitignore — this way everyone has the file, no one's local paths or tool choices end up in git, and the team always knows what overrides are expected.

Health Checks Deep Dive

Health checks are the mechanism that tells Docker whether a container is actually functioning — not just running. A container can be started (the process launched) while the service inside is still initialising, loading data, warming up caches, or establishing database connections. Health checks distinguish between these states.

healthcheck:
  test: ["CMD", "curl", "-f", "http://localhost:8080/health"]
  # or: test: ["CMD-SHELL", "pg_isready -U myuser"]
  interval: 30s      # how often to check
  timeout: 10s       # max time to wait for response
  retries: 3         # consecutive failures = unhealthy
  start_period: 60s  # grace period (don't count failures)

A container moves through three health states during its lifetime:

State Meaning When It Applies
starting Health check has not yet returned a result Within start_period after container start, or before first check interval elapses
healthy The most recent check exited 0 (success) After at least one successful check passes
unhealthy retries consecutive checks have failed After the retry threshold is exceeded outside the start period

The depends_on directive can use these states to control startup ordering. With condition: service_healthy, Compose polls the dependent service's health status and only starts the dependent service once the dependency reaches healthy. Without a condition, Compose only waits for the container process to start — which is almost never what you want for databases or caches.

# depends_on with conditions
depends_on:
  postgres:
    condition: service_healthy    # waits for healthy status
  kafka:
    condition: service_started    # only waits for process to start (weaker)
  migrations:
    condition: service_completed_successfully  # waits for exit code 0
Common Mistake

depends_on without condition: service_healthy only waits for the container to START — not for the service inside to be READY. A database container "starts" in 2 seconds but PostgreSQL takes 10 seconds to accept connections. Your app tries to connect and fails. Always use condition: service_healthy with a proper healthcheck on the database service. The extra three lines of YAML save hours of debugging intermittent startup failures.

Profiles

Profiles let you mark certain services as optional — they only start when explicitly requested. Services without a profiles: key always start. Services with a profile key only start when that profile is activated with --profile.

services:
  # Core services — always started (no profiles key)
  user-service:
    image: ticketflow/user-service:dev

  postgres:
    image: postgres:16-alpine

  # Optional services — only started with --profile
  notification-service:
    build: ./notification-service
    profiles:
      - full        # only start with --profile full

  mailhog:
    image: mailhog/mailhog:latest
    profiles:
      - full        # same profile as notification-service

  prometheus:
    image: prom/prometheus:latest
    profiles:
      - monitoring  # only start with --profile monitoring

  grafana:
    image: grafana/grafana:latest
    profiles:
      - monitoring
# Start only core services (no profiles)
docker compose up -d

# Start everything including monitoring
docker compose --profile monitoring up -d

# Start everything
docker compose --profile full --profile monitoring up -d

# Stop a specific profile
docker compose --profile monitoring down

Profiles shine in several scenarios: keeping a local environment lightweight by default (core database + app, no email catcher, no observability stack), enabling integration-test services only during CI, or activating a search engine only for developers working on search features. Each developer activates only the profiles they need for their current task.

Behind the Scenes

How does depends_on ordering work?

When Compose parses your file, it builds a dependency graph from all depends_on declarations. It then performs a topological sort to determine the startup order. If service A depends on B and C, then B and C must start before A. If B also depends on D, then D starts first.

With condition: service_healthy, Compose polls the dependency's health check at the interval cadence specified in the healthcheck configuration. It blocks starting the dependent service until the health check exits 0. This polling happens in a tight loop with a short sleep between checks — Compose does not simply wait for one interval to pass.

Important limitation: depends_on only controls startup order, not runtime order. If PostgreSQL crashes and restarts while your app is running, Compose will restart Postgres but will not automatically restart your app even if it loses its database connection. You need retry logic in your application code for runtime resilience. Libraries like pg-promise (Node) and Spring Retry (Java) handle connection retry automatically.

How do named volumes work internally?

When Compose creates a named volume, it calls the Docker Engine API to create a volume using the specified driver (default: local). Docker stores the volume data in a directory on the host at /var/lib/docker/volumes/<project>_<volume-name>/_data on Linux, or within the Docker Desktop VM on macOS and Windows.

Compose prefixes the volume name with the project name. If your project directory is ticketflow-local, the volume declared as postgres_data becomes ticketflow-local_postgres_data in Docker. You can see this with docker volume ls. You can override the project name with docker compose -p myproject up or the COMPOSE_PROJECT_NAME environment variable.

Named volumes persist until explicitly deleted. docker compose down removes containers and networks but leaves volumes intact. docker compose down -v also removes volumes — this permanently deletes all data in those volumes. docker volume inspect ticketflow-local_postgres_data shows the exact host path where the data lives, which you can use to back it up.

What is the Compose project and why does it matter?

Every resource Compose creates — containers, networks, volumes — is tagged with a project name. The project name defaults to the name of the directory containing docker-compose.yml. All resource names are prefixed: a container named postgres in a project called ticketflow becomes the container ticketflow-postgres-1.

This prefix is what allows Compose to know which resources belong to a given project, so docker compose down only removes the resources for the current project — not containers from other projects. It also enables multiple isolated instances of the same compose file to coexist on one machine. If two developers share the same host, they can each run the same compose file under different project names: docker compose -p alice-ticketflow up -d and docker compose -p bob-ticketflow up -d.

You can also set the project name via the COMPOSE_PROJECT_NAME environment variable or with a name: top-level key in the compose file itself (Compose Spec, supported since Compose v2.3).

Common Mistakes

Mistake 1 — Startup order race condition

Using depends_on without condition: service_healthy. The database container starts but the PostgreSQL process inside is still initialising. Your app service starts immediately after, tries to connect to the database, gets "connection refused", and crashes. Fix: add a healthcheck to the database service using pg_isready, and add condition: service_healthy to the dependent service's depends_on. Also implement application-level connection retry as a second line of defence.

Mistake 2 — Port conflicts with local services

If you already have PostgreSQL installed locally and running on port 5432, starting the postgres container with ports: "5432:5432" will fail with "port is already in use". Fix: map to a different host port — "5433:5432" exposes PostgreSQL on host port 5433 while the container still listens on 5432 internally. Services inside the Docker network still connect to postgres:5432 by service name regardless of the host port mapping.

Mistake 3 — Committing override files with sensitive data

docker-compose.override.yml typically contains developer-specific passwords, local paths, debug flags, and tool preferences. Committing it exposes secrets and creates merge conflicts. It should always be in .gitignore. Provide a docker-compose.override.yml.example in the repo instead, showing the expected structure with placeholder values. Each developer copies it and fills in their own values.

Mistake 4 — Losing database data with down -v

docker compose down removes containers and networks safely — named volumes survive. docker compose down -v also deletes all named volumes, permanently destroying any data in them. Developers sometimes run down -v to "fully reset" their environment and accidentally delete a database full of test data they needed. Understand the difference clearly: down is safe, down -v is destructive. When in doubt, run docker volume ls and back up volumes you care about before using -v.

Mistake 5 — Hard-coded secrets in docker-compose.yml

Writing DB_PASSWORD=mysecret directly in docker-compose.yml and committing the file to git exposes the secret in your version history forever — even if you change the value later, git history retains the original. Use an env_file: directive pointing to a .env or .env.local file. Add that file to .gitignore. Commit a .env.example with keys but no values so teammates know what variables are needed.

Architect Notes

Architect Note

Docker Compose is for local development and simple single-host deployments — not for production scale-out. Key limitations to understand before reaching for Compose in production:

  • No cross-host networking. All containers must run on a single machine. You cannot span a Compose application across multiple servers.
  • No automatic failover. restart: unless-stopped handles process crashes on a single host, but there is no mechanism to restart containers on a different host if the machine itself fails.
  • No built-in load balancing across scaled containers. --scale user-service=3 creates three containers but Compose does not load balance between them. You need an external proxy (NGINX, Traefik) configured to route across all instances.
  • No rolling deployments. Updating a service image brings it down and back up — there is no zero-downtime rolling update mechanism.

Compose is still the right tool for: local dev environments (every team should have one, and it should be the first thing a new hire runs), integration testing in CI (spin up a real database and Redis for your test suite instead of mocking), simple hobby projects and single-server deployments, and staging environments that mirror dev without Kubernetes complexity.

The mental model you build with Compose carries directly forward to Kubernetes. Services become Deployments. Networks become Services and NetworkPolicies. Named volumes become PersistentVolumeClaims. Healthchecks become readinessProbes and livenessProbes. Profiles become namespaces and Helm values. Compose teaches the right concepts — Kubernetes adds the scale, reliability, and operational sophistication that production demands.

Production Checklist

Before sharing a compose file with the team or deploying to any environment, verify each of the following:

  • docker-compose.override.yml is in .gitignore
  • .env.example is committed to git showing required variable names without values
  • All services have a healthcheck defined with an appropriate start_period
  • depends_on uses condition: service_healthy for all database-dependent services
  • Named volumes are used for all stateful services (postgres, redis, kafka)
  • Backend network is marked internal: true — no direct internet access from database tier
  • Resource limits are set via deploy.resources.limits to prevent runaway containers
  • restart: unless-stopped is set on all long-running services
  • Secrets are NOT hardcoded in docker-compose.yml — use env_file: pointing to a gitignored .env file
  • docker compose config runs without errors (validates the merged compose file before every commit)

Hands-On Lab

Lab — TicketFlow Local Dev Environment

Build a minimal but realistic local environment from scratch. You will create a Node.js service, a PostgreSQL database, and a Compose file that wires them together with proper health checks and dependency ordering. Every step is designed to make the underlying Compose mechanics visible.

# Step 1: Create project structure
mkdir ticketflow-local && cd ticketflow-local
mkdir -p user-service ticket-service db/init nginx

# Step 2: Create minimal user-service
cat > user-service/server.js << 'EOF'
const http = require('http');
const server = http.createServer((req, res) => {
  if (req.url === '/health') {
    res.writeHead(200, {'Content-Type': 'application/json'});
    res.end(JSON.stringify({ status: 'healthy', service: 'user-service' }));
  } else {
    res.writeHead(200);
    res.end('User Service OK');
  }
});
server.listen(3001, () => console.log('User service on :3001'));
EOF

cat > user-service/package.json << 'EOF'
{"name":"user-service","version":"1.0.0","main":"server.js"}
EOF

cat > user-service/Dockerfile << 'EOF'
FROM node:18-alpine
WORKDIR /app
COPY package.json .
COPY server.js .
USER node
EXPOSE 3001
HEALTHCHECK --interval=10s CMD wget -q --spider http://localhost:3001/health || exit 1
CMD ["node", "server.js"]
EOF

# Step 3: Create docker-compose.yml (minimal version)
cat > docker-compose.yml << 'EOF'
version: '3.9'
services:
  user-service:
    build: ./user-service
    ports: ["3001:3001"]
    networks: [backend]
    depends_on:
      postgres:
        condition: service_healthy
    healthcheck:
      test: ["CMD","wget","--no-verbose","--tries=1","--spider","http://localhost:3001/health"]
      interval: 10s
      timeout: 5s
      retries: 3
      start_period: 10s

  postgres:
    image: postgres:16-alpine
    environment:
      POSTGRES_DB: ticketflow
      POSTGRES_USER: ticketflow
      POSTGRES_PASSWORD: devpassword
    volumes: [postgres_data:/var/lib/postgresql/data]
    networks: [backend]
    healthcheck:
      test: ["CMD-SHELL","pg_isready -U ticketflow"]
      interval: 10s
      timeout: 5s
      retries: 5

volumes:
  postgres_data:
networks:
  backend:
EOF

# Step 4: Start and observe
docker compose up -d --build

# Step 5: Watch startup order — observe postgres becoming healthy before user-service starts
docker compose logs -f

# Step 6: Verify health — look for (healthy) in the STATUS column
docker compose ps

# Step 7: Test the service
curl http://localhost:3001/health
# Expected: {"status":"healthy","service":"user-service"}

# Step 8: Enter postgres and list databases
docker compose exec postgres psql -U ticketflow -c '\l'

# Step 9: Scale user-service to 3 instances
docker compose up -d --scale user-service=3
docker compose ps

# Step 10: Inspect volumes — note the project prefix in the volume name
docker volume ls | grep ticketflow

# Step 11: Clean up containers and networks (data survives in the named volume)
docker compose down

Interview Questions

Q1: What is the difference between docker compose stop and docker compose down?

docker compose stop sends a stop signal to the running containers and waits for them to halt gracefully. The containers continue to exist in a stopped state — their writable filesystem layer, assigned IP addresses, and configuration all persist. You can restart them with docker compose start and they pick up exactly where they left off. No data is lost. No networks are removed.

docker compose down goes further: it stops the containers and then removes them, along with the networks Compose created. Your named volumes survive — data in postgres and redis is untouched. Use down when you want a full cleanup of containers and want a fresh start on the next up.

docker compose down -v additionally removes named volumes — this permanently destroys all data stored in those volumes. Use it only when you explicitly want to wipe the database clean (e.g., to reset a dev environment to a known blank state). Never run down -v on a shared environment without communicating to the team first.

Rule of thumb: use stop when pausing work to resume later. Use down when cleaning up at end of day or switching branches. Use down -v only intentionally, never as a default "make it work again" habit.

Q2: How does depends_on work and what are its limitations?

depends_on controls the startup order of services. Without a condition: key, it only waits for the dependent container's process to start — the container exits the "created" state and enters "running". This is almost never sufficient for databases, message brokers, or caches because the service inside the container needs additional time to initialise before accepting connections.

With condition: service_healthy, Compose polls the dependency's health check until it reports the healthy status. With condition: service_completed_successfully, it waits for a one-shot container (like a database migration job) to exit with code 0 before proceeding.

Critical limitation: depends_on only governs the initial startup sequence. Once all services are running, Compose does not monitor dependencies. If PostgreSQL crashes and Compose restarts it, your application service is not automatically restarted — even though it may have lost its database connection. Implement connection retry logic in your application (most good database libraries do this automatically) so the app reconnects when the database comes back, without human intervention.

Q3: What is the purpose of Docker Compose networks and why use multiple?

Networks in Compose provide isolation and service discovery. Services on the same network can reach each other by their service name (Compose registers each service as a DNS hostname on its networks). Services on different networks cannot communicate unless they share at least one network in common.

Using multiple networks is a security pattern. In the TicketFlow example: the frontend network connects the API gateway to the user service and ticket service. The backend network connects the application services to the database, Redis, and Kafka. The API gateway is on both networks — it is the only service that bridges them. PostgreSQL and Redis are only on backend — they cannot be reached from the frontend tier directly.

Marking the backend network as internal: true is an additional security layer: no container on that network can make outbound internet connections. A compromised database container cannot reach an external attacker's server or exfiltrate data. Frontend containers can still talk to the internet (for fetching external APIs, sending webhooks) because they are only on the non-internal frontend network.

Q4: How do you handle secrets in Docker Compose for development vs. production?

For development: use an env_file: directive pointing to a .env or .env.local file. This keeps secrets out of docker-compose.yml itself. Git-ignore the .env file. Commit a .env.example to the repo with all keys present but values blank or set to obvious placeholder strings, so teammates know exactly what variables to provide.

For production: the short answer is that Docker Compose is generally not the right tool for production workloads that require real secrets management. For true production, use Kubernetes Secrets (base64 encoded, optionally encrypted at rest via etcd encryption), HashiCorp Vault with dynamic secrets and short-lived leases, or cloud-native solutions like AWS Secrets Manager, Azure Key Vault, or GCP Secret Manager with automatic rotation.

If you must use Compose in a simple single-server production deployment, Docker Swarm mode supports a secrets: top-level key — secrets are stored encrypted in the Raft log and exposed to containers as files under /run/secrets/. This is significantly more secure than environment variables because secrets never appear in docker inspect output. However, for anything beyond a hobbyist project, the right answer is a proper secrets manager, not Compose.

Q5: What is the Compose override file pattern?

docker-compose.yml contains the base, environment-agnostic configuration. It defines all services, their images or build contexts, their network membership, and their default configuration. This file is committed to git and is the source of truth that every team member and every environment starts from.

docker-compose.override.yml is automatically discovered and merged on top of the base file whenever docker compose up is run with no explicit -f flags. Merge rules: scalar values (strings, numbers) in the override replace base values; lists (ports, volumes, environment) are appended, not replaced; maps (environment as key-value) are merged with override values taking precedence.

The override file is typically gitignored because it contains developer-specific choices: hot-reload volume mounts (which depend on local paths), elevated log levels, dev-specific environment variables, and extra containers for local tooling. Each developer maintains their own copy. For staging and production, explicit -f flags are used: docker compose -f docker-compose.yml -f docker-compose.staging.yml up -d. This makes the environment explicit and auditable — no surprise automatic merges.

Q6: How do you wait for a database to be ready in Compose?

The robust solution has three layers, each addressing a different failure mode.

Layer 1 — Compose healthcheck: Add a healthcheck to the database service using a native readiness check. For PostgreSQL, use pg_isready -U myuser which tests whether the database is accepting connections on the specified user. For Redis, use redis-cli ping. Set a reasonable start_period so Compose does not mark the container unhealthy during the normal initialisation window (10–15 seconds for Postgres, 60+ seconds for a cold Kafka broker).

Layer 2 — Compose depends_on condition: Use depends_on: <db>: condition: service_healthy on every service that connects to the database. This ensures Compose does not start the application service until the database reports healthy. This handles the happy-path startup case.

Layer 3 — Application-level retry: Implement connection retry logic in your application. depends_on: condition: service_healthy only guarantees that the database was healthy at the moment your service started. After startup, the database can restart, network can hiccup, or connections can be lost. Good database client libraries (pg-promise for Node.js, HikariCP for Java, SQLAlchemy with connection pooling for Python) handle automatic reconnection. Configure retry policies with exponential backoff and a maximum retry count so the app recovers automatically without operator intervention.
