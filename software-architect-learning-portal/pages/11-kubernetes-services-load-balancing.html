Kubernetes Services & Load Balancing

How Kubernetes routes network traffic to Pods — ClusterIP, NodePort, LoadBalancer, ExternalName, headless services, DNS-based discovery, and the kube-proxy machinery underneath.

The Story Behind Kubernetes Services

📞 The Hotel Switchboard Problem

Imagine a large hotel in the 1950s. Guests need to reach staff, but staff move around constantly — a bellhop could be on any floor, a housekeeper in any room. If every guest had to know the exact room a specific staff member was in at this exact moment, the system would collapse. No one would ever reach anyone.

The solution was the hotel switchboard. A guest dials a single, stable extension — "dial 0 for housekeeping." The switchboard operator knows which staff members are available right now and connects the call to one of them. Guests never need to know individual staff locations. Staff can move freely. The switchboard handles it all.

Kubernetes Services are the hotel switchboard for your microservices. Pods are ephemeral — they get created, scaled, rescheduled, and terminated constantly. Each Pod gets a unique IP address that changes every time it is replaced. If payment-service had to know the exact Pod IP of ticket-service in order to call it, every restart of ticket-service would break payment-service. A Service gives each workload a stable, virtual IP and DNS name. Traffic sent to that stable address is automatically distributed across healthy Pods — no matter how many there are or where they live in the cluster.

Services in Plain English

🍕 Beginner Analogy

Think of a Kubernetes Service like a pizza shop's phone number. The shop might have 5 phones and 10 staff members taking orders — they change shifts, some go on break, new ones start. But the phone number never changes. You call the same number every time. The shop's internal routing system decides which available staff member picks up your call.

A Kubernetes Service is that phone number for your app. You always call ticket-service. Behind the scenes, Kubernetes decides which of the 3 (or 10, or 1) running Pods of ticket-service actually handles the request. Your calling code never has to change, no matter how many Pods exist or when they restart.

Every Kubernetes Service has three key properties:

  • Selector — a label query that identifies which Pods the Service routes traffic to. Any Pod with matching labels is automatically included as a backend endpoint.
  • ClusterIP — a virtual IP address assigned to the Service (from a dedicated IP range). This IP is stable for the lifetime of the Service and is only reachable inside the cluster.
  • Ports — the port the Service listens on, and the target port it forwards to on the Pod (these can differ).

The Four Service Types

Kubernetes provides four distinct Service types, each designed for a different networking requirement. Choosing the wrong type is a common mistake — understand each one before reaching for the keyboard.

1. ClusterIP (Default)

ClusterIP is the default Service type. It creates a virtual IP address that is only reachable from inside the cluster. Nothing outside the cluster can talk to a ClusterIP service directly. This is what you use for internal service-to-service communication — the vast majority of your Services will be ClusterIP.

  • Use for: internal microservice communication (e.g., payment-service calling ticket-service)
  • Not reachable from: outside the Kubernetes cluster
  • DNS name: ticket-service.default.svc.cluster.local
  • Stability guarantee: the ClusterIP address never changes for the lifetime of the Service

2. NodePort

NodePort extends ClusterIP by also opening a static port (between 30000–32767 by default) on every node in the cluster. External traffic sent to <any-node-IP>:<NodePort> is routed into the cluster and forwarded to the Service's Pods. NodePort is rarely used in production — it requires clients to know node IPs, exposes a non-standard port number, and provides no load balancing across nodes. It is primarily used for local development, quick demos, or when you have no cloud load balancer available.

  • Use for: local development exposure, on-premises clusters without a cloud LB, quick testing
  • Access via: http://<node-ip>:30080
  • Port range: 30000–32767 (configurable in kube-apiserver flags)
  • Risk: exposes all nodes; requires firewall rules on each node; no TLS termination

3. LoadBalancer

LoadBalancer extends NodePort by additionally provisioning an external cloud load balancer. When you create a LoadBalancer Service on AWS EKS, Kubernetes calls the AWS API to create an ELB/NLB with a public IP. That load balancer forwards traffic to the NodePort on your cluster nodes. This is the standard way to expose a single externally-facing service. TicketFlow uses this type exclusively for the api-gateway service — it is the single entry point from the internet.

  • Use for: exposing a single public-facing service (API gateway, public API)
  • Requires: a cloud provider with a load balancer controller (AWS, GCP, Azure) or MetalLB on bare metal
  • Cost: each LoadBalancer Service provisions a separate cloud LB — can become expensive with many services
  • Better pattern: one LoadBalancer for the Ingress controller, use Ingress rules for individual services (see page 12)

4. ExternalName

ExternalName is a special type that does not create a virtual IP or proxy traffic. Instead it creates a DNS CNAME alias inside the cluster. When any Pod resolves the Service name, it gets back the CNAME's target DNS name. This is useful for pointing cluster workloads at external services (a managed RDS database, a third-party API) using a stable in-cluster DNS name that can be changed without modifying application configuration.

  • Use for: aliasing external DNS names into the cluster (e.g., db.ticketflow.io → ticketflow-db.us-east-1.rds.amazonaws.com)
  • Does NOT: proxy traffic, perform health checks, or load balance
  • Migration pattern: create an ExternalName pointing at the old service; later change it to ClusterIP pointing at the new in-cluster service — zero application code changes required

Service Discovery via DNS

Kubernetes includes CoreDNS — a cluster-internal DNS server that automatically creates DNS records for every Service. This is how microservices find each other without hardcoding IP addresses.

DNS Name Format

Every Service gets a fully qualified domain name (FQDN) following this pattern:

<service-name>.<namespace>.svc.cluster.local

# Examples for TicketFlow:
ticket-service.default.svc.cluster.local
payment-service.default.svc.cluster.local
user-service.default.svc.cluster.local
api-gateway.default.svc.cluster.local
notification-service.default.svc.cluster.local

Because Kubernetes automatically adds the namespace and cluster suffix to DNS searches, in-cluster code can use just the short name:

# payment-service (Python) calling ticket-service
# Short name works because both services are in 'default' namespace
import httpx

response = httpx.get("http://ticket-service:8080/api/tickets/42")

# Full FQDN — required when calling across namespaces
# e.g., calling a service in the 'monitoring' namespace
response = httpx.get("http://prometheus.monitoring.svc.cluster.local:9090/api/v1/query")

How CoreDNS Works

When a Pod makes a DNS query for ticket-service, here is what happens:

  1. The Pod's /etc/resolv.conf (injected by kubelet) points to the CoreDNS ClusterIP address (typically 10.96.0.10).
  2. The query is sent to CoreDNS with the cluster search domains appended: ticket-service.default.svc.cluster.local.
  3. CoreDNS has a watch on the Kubernetes API and maintains a real-time map of all Service ClusterIPs. It returns the ClusterIP for ticket-service.
  4. The Pod connects to that ClusterIP. kube-proxy on the node intercepts the connection and load-balances it to one of the healthy Pod IPs registered as endpoints for the Service.

kube-proxy: The Traffic Plumber

kube-proxy runs as a DaemonSet on every node in the cluster. Its job is to implement the Service abstraction by maintaining network rules that intercept traffic destined for Service ClusterIPs and rewrite it to target actual Pod IPs. It watches the Kubernetes API for Service and Endpoint changes and keeps the rules up to date automatically.

iptables Mode (Default)

In iptables mode, kube-proxy programs Linux netfilter rules that intercept packets at the kernel level. When a packet arrives destined for a Service ClusterIP, the kernel's PREROUTING chain hits a rule that randomly selects one of the available Pod IPs and rewrites the destination (DNAT). The selection uses probability-weighted rules to achieve load balancing. This is purely kernel-space — no process needs to handle each packet individually.

# View the iptables rules kube-proxy creates for a service
# (Run on a cluster node)
sudo iptables -t nat -L KUBE-SERVICES -n | grep ticket-service

# View the actual endpoint rules (DNAT to Pod IPs)
sudo iptables -t nat -L KUBE-SVC-XXXXXXXXXXXX -n

# Example output shows probability-weighted rules:
# -j KUBE-SEP-AAAAAA  /* first pod */     statistic --probability 0.33
# -j KUBE-SEP-BBBBBB  /* second pod */    statistic --probability 0.50
# -j KUBE-SEP-CCCCCC  /* third pod */     (catches the rest)

IPVS Mode (Recommended for Large Clusters)

IPVS (IP Virtual Server) is a Linux kernel module originally built for high-performance load balancing. In IPVS mode, kube-proxy creates a virtual server per Service ClusterIP using the kernel's IPVS subsystem rather than iptables chains. IPVS uses a hash table internally, so lookup time is O(1) regardless of the number of Services — iptables is O(n) and degrades significantly at thousands of services. IPVS also supports richer load balancing algorithms: round-robin, least connections, source hashing, etc.

# Check if IPVS mode is active
kubectl get configmap kube-proxy -n kube-system -o yaml | grep mode

# View virtual servers created by kube-proxy in IPVS mode
sudo ipvsadm -L -n

# Example output:
# TCP  10.96.10.5:8080 rr          <-- ticket-service ClusterIP, round-robin
#   -> 10.244.1.10:8080  Masq  1  0  0   <-- Pod 1
#   -> 10.244.2.10:8080  Masq  1  0  0   <-- Pod 2

Headless Services (StatefulSets)

A headless Service is created by setting clusterIP: None. Kubernetes does not assign a virtual IP to the Service. Instead, DNS queries return the IP addresses of all backing Pods directly — the client receives a list of A records, one per Pod. This bypasses kube-proxy entirely and gives the client full visibility and control over which Pod to talk to.

Why StatefulSets Need Headless Services

Regular Services are designed for stateless workloads where any Pod is interchangeable. StatefulSets run stateful applications (databases, message queues, distributed caches) where each Pod has a stable identity and may have a different role — in a PostgreSQL cluster, the primary Pod accepts writes while replica Pods handle reads. The application must be able to address individual Pods by name.

A headless Service paired with a StatefulSet creates stable, predictable DNS names for each Pod:

# With headless Service named 'postgres' and StatefulSet named 'postgres':
# Each Pod gets a stable DNS name:
postgres-0.postgres.default.svc.cluster.local  <-- primary
postgres-1.postgres.default.svc.cluster.local  <-- replica 1
postgres-2.postgres.default.svc.cluster.local  <-- replica 2

# These names survive Pod restarts — postgres-0 always resolves to the
# same StatefulSet Pod, even if it gets rescheduled to a different node

Enterprise Scenario: TicketFlow Service Mesh

🏢 TicketFlow — How Services Wire the Platform

TicketFlow runs 6 microservices on AWS EKS. Here is the complete networking topology and why each Service type was chosen:

  • api-gateway (NGINX) — type: LoadBalancer. The only public entry point. AWS provisions an NLB with a stable public IP. All HTTP/HTTPS traffic from the internet enters here. TLS is terminated at the NLB.
  • user-service (Node.js) — type: ClusterIP. Only called by api-gateway internally. Stable internal DNS: user-service.default.svc.cluster.local.
  • ticket-service (Spring Boot) — type: ClusterIP. Called by api-gateway, payment-service, and order-service. Port 8080 on the Service maps to port 8080 on Pods.
  • payment-service (Python) — type: ClusterIP. Called by order-service. Calls ticket-service to validate ticket availability before charging. Calls an ExternalName Service to reach the Stripe API via an internal alias.
  • order-service (Go) — type: ClusterIP. Orchestrates ticket-service + payment-service. High throughput — scaled to 5 replicas during peak. The Service transparently distributes load across all 5 Pods.
  • notification-service (Node.js) — type: ClusterIP. Receives events from other services. Connects to SES via an ExternalName Service: email-provider → email.us-east-1.amazonaws.com.

This design means only one cloud load balancer is provisioned (saving ~$20/month per avoided LB). All internal traffic uses ClusterIP — no external exposure, no unnecessary cost, no attack surface.

payment-service calling ticket-service (Python)

# payment-service/src/services/ticket_validator.py
import httpx
import logging
from typing import Optional

logger = logging.getLogger(__name__)

# DNS name resolves to ticket-service ClusterIP via CoreDNS
# No hardcoded IPs — this works regardless of Pod restarts or scaling
TICKET_SERVICE_URL = "http://ticket-service:8080"

async def validate_ticket_availability(
    ticket_id: str,
    quantity: int,
    correlation_id: str
) -> dict:
    """
    Check ticket availability before processing payment.
    Called as part of the order-service orchestration flow.
    """
    async with httpx.AsyncClient(timeout=5.0) as client:
        try:
            response = await client.get(
                f"{TICKET_SERVICE_URL}/api/v2/tickets/{ticket_id}/availability",
                params={"quantity": quantity},
                headers={
                    "X-Correlation-ID": correlation_id,
                    "X-Source-Service": "payment-service",
                }
            )
            response.raise_for_status()
            data = response.json()

            logger.info(
                "ticket_availability_check",
                ticket_id=ticket_id,
                available=data["available"],
                quantity_requested=quantity,
            )
            return data

        except httpx.TimeoutException:
            logger.error("ticket_service_timeout", ticket_id=ticket_id)
            raise ServiceUnavailableError("ticket-service timeout")
        except httpx.HTTPStatusError as e:
            logger.error("ticket_service_error", status=e.response.status_code)
            raise

Behind the Scenes

What happens when payment-service calls ticket-service for the first time?
  1. DNS resolution: The Python runtime in payment-service Pod calls httpx.get("http://ticket-service:8080/..."). The OS resolver consults /etc/resolv.conf, which lists CoreDNS at 10.96.0.10 as the nameserver, with search domain default.svc.cluster.local.
  2. CoreDNS lookup: CoreDNS receives a query for ticket-service.default.svc.cluster.local. CoreDNS maintains an in-memory map built by watching the Kubernetes API. It returns the ClusterIP: 10.96.10.5. DNS TTL is 30 seconds — the result is cached in the Pod's resolver for that duration.
  3. TCP connection to ClusterIP: The HTTP client opens a TCP connection to 10.96.10.5:8080. The kernel on the node processes the outgoing packet.
  4. iptables DNAT intercept: kube-proxy has pre-programmed a PREROUTING/OUTPUT rule in netfilter that matches destination 10.96.10.5:8080. The kernel randomly selects one of the three Endpoints (Pod IPs: 10.244.1.10, 10.244.2.10, 10.244.3.10) using probability-weighted rules and rewrites the destination IP (DNAT). The source IP is also NAT'd (MASQUERADE) so the response routes back correctly.
  5. Pod receives connection: The selected ticket-service Pod at 10.244.2.10:8080 receives the connection. From the Pod's perspective it is a direct connection — it has no visibility into the ClusterIP or the NAT that occurred.
  6. Response returns: The response travels back through the conntrack table, which maintains the DNAT mapping. The kernel rewrites the source IP back to 10.96.10.5 (SNAT) so the calling service sees the ClusterIP in the response, not a Pod IP. The HTTP response arrives in payment-service.
  7. Subsequent calls: If a Pod fails or is scaled, kube-proxy detects the Endpoint change (via API watch) and updates the iptables rules within milliseconds. The DNS TTL of 30 seconds means some calls may briefly use a stale ClusterIP — but the ClusterIP never changes, so this is not an issue. New Pods are added to the iptables rules automatically.
What happens when a Pod backing a Service is terminated?
  1. Kubernetes marks the Pod as Terminating. The endpoint controller removes this Pod's IP from the Service's Endpoints object immediately.
  2. kube-proxy detects the Endpoints change via its API watch and removes the terminated Pod's IP from the iptables rules. This happens within ~100ms on a well-tuned cluster.
  3. The Pod's container receives SIGTERM. Kubernetes waits for terminationGracePeriodSeconds (default 30s) before sending SIGKILL.
  4. In-flight requests that were already routed to this Pod will complete if the app handles SIGTERM gracefully. New requests will never reach this Pod — they go to the remaining healthy Pods.
  5. The key risk: there is a small window between Pod Termination and kube-proxy updating iptables where a new connection might still be routed to the terminating Pod. This is why well-behaved apps handle SIGTERM with a brief delay before shutting down the listener — to let kube-proxy catch up.

Common Mistakes

❌ Mistake 1: Service Selector Does Not Match Pod Labels

The Service has selector: app: ticket-service but the Deployment pods are labeled app: ticketservice (no hyphen). The Service has zero Endpoints — all traffic silently times out. This is the most common mistake when setting up Services for the first time.

Diagnose: kubectl get endpoints ticket-service. If the Endpoints list shows <none> where you expect Pod IPs, your selector does not match any Pods. Run kubectl get pods --show-labels to compare actual labels against the Service selector.

Fix: Ensure the Service spec.selector matches the Pod template metadata.labels exactly — case-sensitive, hyphen-sensitive.

❌ Mistake 2: Creating a LoadBalancer for Every Microservice

A team creates 10 microservices and gives each one a type: LoadBalancer because "it needs to be accessible." On AWS EKS, each LoadBalancer Service provisions a separate NLB or Classic ELB. That is 10 load balancers × $18/month = $180/month in unnecessary LB costs, plus 10 public IPs in your security surface.

Fix: Use a single LoadBalancer Service for an Ingress controller (nginx-ingress or AWS Load Balancer Controller). Route all public traffic through the Ingress with host/path rules. All internal microservices use ClusterIP. See page 12 for Ingress deep-dive.

❌ Mistake 3: Hardcoding Pod IP Addresses

A developer finds a Pod IP by running kubectl get pods -o wide and hardcodes 10.244.3.7:8080 into the application config. Pods are ephemeral — they can be rescheduled, scaled, or replaced at any time. That IP will stop working the next time the Pod is terminated. Pod IPs are not stable and must never be used directly.

Fix: Always use Service DNS names (http://ticket-service:8080). Service ClusterIPs and DNS names are stable for the lifetime of the Service. That is exactly what Services exist to provide.

❌ Mistake 4: Wrong targetPort Causes Connection Refused

The Service defines port: 8080 and targetPort: 8080 but the application inside the container actually listens on port 3000 (a Node.js app). Connections to the Service reach the Pod but get an immediate ECONNREFUSED because nothing is listening on 3000 at the Pod's 8080. Error messages in the calling service say "connection refused" rather than "timeout," which is the key diagnostic clue.

Fix: The targetPort must match the actual port the container process is listening on. Best practice: use named ports. In the Pod template, name the container port (name: http), then reference that name in the Service's targetPort: http — this makes the mapping explicit and self-documenting.

❌ Mistake 5: Using NodePort in Production for External Traffic

A team exposes their API using NodePort :30080 for external access, reasoning "it works just like LoadBalancer but cheaper." In production: if a node goes down, clients hitting that node get connection refused — there is no health checking of nodes. NodePort provides no TLS termination. The non-standard port (30080 vs 443) gets blocked by enterprise firewalls. Auto-scaling adds new nodes that clients don't know about. NodePort was not designed for production external traffic.

Fix: Use LoadBalancer type (with a single Ingress controller) for any externally-facing production service. The cloud load balancer handles node health checks, SSL termination, and automatic node registration.

Architect Perspective

🏛️ Architect Note

Kubernetes Services are a stable contract boundary. When you define a Service, you are declaring: "this is the stable interface for this workload." The Service name, port, and protocol form a contract that all consumers depend on. The underlying Pods — their count, IPs, node placement — are implementation details that you can change freely.

This separation is architecturally significant for several reasons:

  • Deployment independence: You can replace all Pods behind a Service (rolling update, blue-green) without the callers noticing. As long as the Service contract is preserved, callers are unaffected.
  • The single LoadBalancer pattern: Production clusters should have exactly one external LoadBalancer — the one fronting the Ingress controller. Everything behind it uses ClusterIP. This principle — one gateway, many internal services — is the architecture that scales to hundreds of services without runaway cloud costs.
  • Services are not enough for resilience: Services provide load balancing and Pod discovery, but they do not handle retries, circuit breaking, timeouts, or mutual TLS. For production microservice resilience, Services are the foundation that a service mesh (Istio, Linkerd) or application-level client library (Resilience4j, Polly) builds on top of. Services handle the "where" — service mesh handles the "how well."
  • ExternalName for migration: When migrating from a legacy system to an in-cluster service, ExternalName provides a zero-code-change migration path. Start with ExternalName pointing at the legacy system. Once the in-cluster replacement is ready, change the Service type to ClusterIP — all callers automatically route to the new service. This technique is valuable for incremental strangler fig migrations.
  • Endpoint slices at scale: In clusters with hundreds of Services and thousands of Pods, the original Endpoints API becomes a bottleneck — every change to any endpoint rebuilds the entire object and floods the API server. Kubernetes 1.21+ introduced EndpointSlices (enabled by default), which shard endpoints into groups of 100. kube-proxy on each node only needs to sync the slices that changed. At 5,000+ Pods, this is not optional — it is the difference between a responsive cluster and a broken one.

Production Readiness Checklist

  • All internal microservices use type: ClusterIP — not NodePort or LoadBalancer
  • Only one type: LoadBalancer Service exists per cluster (or per namespace), fronting the Ingress controller
  • Service selector labels exactly match Pod template labels — verified with kubectl get endpoints
  • Named container ports are used in Pod templates and referenced by name in Service targetPort
  • terminationGracePeriodSeconds set appropriately and apps handle SIGTERM gracefully to avoid dropped connections during rolling updates
  • LoadBalancer Services on AWS EKS use NLB annotations, not Classic ELB (better performance, lower cost)
  • Service port names follow protocol naming convention (http, grpc, tcp) for Istio/service mesh compatibility
  • ExternalName Services used instead of hardcoded external FQDNs in application configs
  • Kubernetes 1.21+ clusters have EndpointSlice feature enabled (default) — do not disable it
  • Network policies applied to restrict which Services can communicate with each other (principle of least privilege)
  • kube-proxy mode set to ipvs for clusters with >500 Services to avoid iptables O(n) performance degradation
  • Service discovery tested with kubectl run dns-test ... nslookup <service> before go-live

Hands-On Lab: Expose user-service

🧪 Lab — Requires a running Kubernetes cluster (minikube, kind, or EKS)

This lab deploys user-service and exposes it via ClusterIP, then demonstrates DNS discovery, port-forwarding, and endpoint inspection. Estimated time: 20 minutes.

Step 1: Deploy user-service

# Create the Deployment and Service from inline YAML
kubectl apply -f - <<'EOF'
apiVersion: apps/v1
kind: Deployment
metadata:
  name: user-service
  namespace: default
spec:
  replicas: 2
  selector:
    matchLabels:
      app: user-service
  template:
    metadata:
      labels:
        app: user-service
        version: "1.0"
    spec:
      containers:
        - name: user-service
          image: nginx:1.25-alpine    # Using nginx as a stand-in for the lab
          ports:
            - name: http
              containerPort: 80
          readinessProbe:
            httpGet:
              path: /
              port: 80
            initialDelaySeconds: 5
            periodSeconds: 5
---
apiVersion: v1
kind: Service
metadata:
  name: user-service
  namespace: default
spec:
  type: ClusterIP
  selector:
    app: user-service
  ports:
    - name: http
      port: 3000
      targetPort: http    # References the named containerPort
EOF

Step 2: Verify Service and Endpoints

# Check Service was created — note the ClusterIP assigned
kubectl get svc user-service

# Verify Endpoints — should show 2 Pod IPs (one per replica)
# If this shows <none>, your selector does not match your Pod labels
kubectl get endpoints user-service

# Describe the Service for full details (ports, selector, events)
kubectl describe svc user-service

# Check that Pods are Ready (READY should show 2/2)
kubectl get pods -l app=user-service -o wide

Step 3: Test DNS discovery inside the cluster

# Launch a temporary debug Pod to test DNS resolution
kubectl run dns-tester \
  --image=busybox:1.28 \
  --rm \
  -it \
  --restart=Never \
  -- sh

# Inside the debug Pod:
# 1. Check /etc/resolv.conf — verify CoreDNS address and search domains
cat /etc/resolv.conf

# 2. Resolve user-service using short name (within same namespace)

# 3. Resolve using full FQDN

# 4. Make an HTTP request using the Service DNS name
wget -qO- http://user-service:3000/

# 5. Run multiple requests to observe load balancing
for i in $(seq 1 5); do wget -qO- http://user-service:3000/ 2>&1 | head -1; done

exit  # Back to host — the debug Pod is automatically removed

Step 4: Port-forward for local access (no NodePort needed)

# Forward local port 9090 to user-service port 3000
# This creates a tunnel — no NodePort or LoadBalancer needed

# The port-forward is session-scoped — Ctrl+C to stop

Key observations from this lab:

  1. The Endpoints object updates within seconds of Pod readiness changes — this is kube-proxy staying in sync.
  2. DNS short names work because /etc/resolv.conf adds default.svc.cluster.local as a search domain automatically.
  3. kubectl port-forward is the correct way to test internal ClusterIP Services locally — not NodePort, not SSH tunneling to nodes.
  4. A Service with zero Endpoints means either: (a) the selector does not match any Pods, or (b) Pods exist but are not yet Ready (failing readiness probes).

Interview Questions

These questions appear frequently in Kubernetes and platform engineering interviews. Each answer is written to demonstrate depth beyond surface definitions.

Q1: What is a Kubernetes Service and why is it necessary? Why can't microservices just talk directly to Pod IPs?

A Kubernetes Service is a stable, virtual network endpoint that provides a consistent address for a dynamic set of Pods. It is necessary because Pods are ephemeral — they are created and destroyed constantly due to scaling events, rolling updates, node failures, and scheduling decisions. Each new Pod receives a new, randomly assigned IP address.

If payment-service hardcoded the IP of a ticket-service Pod, that IP would become invalid the next time the Pod was rescheduled. The calling service would need to re-discover the new IP somehow. At scale with dozens of services and hundreds of Pods, this becomes unmanageable.

A Service solves this by providing: a stable ClusterIP that never changes, automatic registration/deregistration of healthy Pods as Endpoints, load balancing across all healthy Pods, and a stable DNS name via CoreDNS. Services decouple the consumer (who needs a stable address) from the producer (whose Pods change constantly).

Q2: Explain the difference between ClusterIP, NodePort, and LoadBalancer. When would you choose each?

ClusterIP is the default. It creates a virtual IP accessible only from within the cluster. Use it for all internal service-to-service communication. In a typical microservices platform, 90%+ of your Services should be ClusterIP. It has zero cloud cost and is the most secure choice since the service is unreachable from outside.

NodePort extends ClusterIP by additionally opening a static port (30000–32767) on every cluster node. External traffic can reach the service via <node-IP>:<nodePort>. Use it for local development clusters, on-premises setups without a cloud LB, or quick testing. Avoid it in production for external traffic — no health-checked load balancing across nodes, non-standard port numbers, and all nodes are exposed.

LoadBalancer extends NodePort by provisioning an external cloud load balancer (AWS NLB, GCP Network LB, Azure Load Balancer). This is the standard way to expose a service to the internet. However, each LoadBalancer Service provisions a separate, billed cloud resource. The recommended pattern is: one LoadBalancer for the Ingress controller, and all other services use ClusterIP with Ingress rules for external routing.

Q3: How does Kubernetes DNS-based service discovery work? What is CoreDNS's role?

When a Pod is created, kubelet injects a /etc/resolv.conf file that points the DNS nameserver to CoreDNS's ClusterIP (typically 10.96.0.10) and sets search domains including default.svc.cluster.local.

CoreDNS runs as a Deployment in the kube-system namespace and watches the Kubernetes API for Service creation/deletion events. For every Service, it maintains a DNS record: <service-name>.<namespace>.svc.cluster.local pointing to the Service's ClusterIP.

When application code calls http://ticket-service:8080, the OS resolver queries CoreDNS with ticket-service.default.svc.cluster.local (after appending search domains). CoreDNS returns the ClusterIP. The application connects to that ClusterIP, and kube-proxy intercepts the connection to load-balance it to a real Pod IP.

Short names work within the same namespace because the search domain is appended automatically. Cross-namespace calls require the full FQDN: ticket-service.payments.svc.cluster.local.

Q4: What is kube-proxy and how does it implement load balancing in iptables mode vs IPVS mode?

kube-proxy is a DaemonSet that runs on every cluster node. Its job is to implement the Service abstraction by maintaining network rules that route traffic from ClusterIPs to actual Pod IPs. It watches the Kubernetes API for Service and Endpoints changes and updates routing rules accordingly.

iptables mode (default): kube-proxy programs netfilter chains in the Linux kernel's NAT table. When a packet destined for a ClusterIP hits the PREROUTING or OUTPUT chain, a DNAT rule randomly selects one of the Endpoints (Pod IPs) using probability-weighted rules and rewrites the destination. This is entirely kernel-space — no userspace process handles individual packets. However, iptables rule evaluation is O(n) — with 10,000 Services, every packet must traverse all rules linearly. This degrades performance at scale.

IPVS mode (recommended at scale): kube-proxy uses the Linux kernel's IPVS (IP Virtual Server) subsystem, which uses hash tables for O(1) lookup regardless of the number of Services. IPVS also supports richer algorithms: round-robin, least connections, source IP hash, etc. For clusters with more than ~500 Services, IPVS mode is strongly recommended to avoid the iptables performance cliff. Enable it by setting mode: ipvs in the kube-proxy ConfigMap.

Q5: What is a headless Service and when would you use one?

A headless Service is created by setting clusterIP: None in the Service spec. Unlike regular Services, Kubernetes does not assign a virtual IP. Instead, DNS queries for the Service name return multiple A records — one for each backing Pod's IP. kube-proxy does not create any iptables/IPVS rules for headless Services because there is no VIP to intercept.

The primary use case is StatefulSets. StatefulSets run stateful workloads like databases and message queues where individual Pods have distinct roles. A headless Service paired with a StatefulSet creates stable, predictable DNS names: postgres-0.postgres.default.svc.cluster.local, postgres-1.postgres.default.svc.cluster.local, etc. These names survive Pod restarts and rescheduling — a critical requirement for replication and primary/replica configurations.

Headless Services are also used when you want the client to perform its own load balancing or service discovery — for example, with gRPC's client-side load balancing, where the client needs to know all Pod IPs to establish connections to each one directly.

Q6: A Service has been created but traffic is not reaching the Pods — how do you diagnose this?

This is one of the most common Kubernetes networking issues. Here is a systematic diagnostic approach:

  1. Check Endpoints: kubectl get endpoints <service-name>. If the output shows <none>, the selector does not match any Pods — the Service is routing to nothing. Compare the Service selector (kubectl describe svc) against actual Pod labels (kubectl get pods --show-labels).
  2. Check Pod readiness: If Endpoints shows IPs but traffic fails, verify Pods are Ready: kubectl get pods -l <selector>. Only Ready Pods are included as Endpoints. Failing readiness probes prevent a Pod from receiving traffic even if it is running.
  3. Check targetPort: Verify targetPort matches the port the container actually listens on. A ECONNREFUSED error (not timeout) suggests the Pod is reachable but nothing is listening on the target port.
  4. Test DNS from inside the cluster: Run a debug Pod (kubectl run test --image=busybox -- nslookup <service>). If DNS resolution fails, the problem is in CoreDNS, not kube-proxy.
  5. Check Network Policies: If all the above are correct, a NetworkPolicy may be blocking the traffic. kubectl get networkpolicy and check if the policy allows the specific source-to-destination traffic.
  6. Check kube-proxy rules: On a node, run iptables -t nat -L KUBE-SERVICES -n to verify kube-proxy has programmed rules for the Service's ClusterIP.
Q7: What is an ExternalName Service and what problem does it solve in a migration scenario?

An ExternalName Service is a special Service type that creates a DNS CNAME alias. When any Pod in the cluster resolves the Service name, CoreDNS returns the CNAME target instead of a ClusterIP. kube-proxy is not involved — no VIP, no iptables rules, no load balancing. It is purely a DNS alias.

The primary value is decoupling application code from external service locations. Instead of hardcoding ticketflow-db.us-east-1.rds.amazonaws.com in 5 services, each service uses the in-cluster name postgres-primary which is an ExternalName pointing at the RDS endpoint. If the RDS hostname changes (e.g., after a failover or migration to a different instance), you update the ExternalName Service once — no application config changes needed.

The migration pattern is powerful: when migrating a dependency from external to in-cluster, create an ExternalName Service first, point all consumers at it. When the in-cluster replacement is ready, change the Service type from ExternalName to ClusterIP with a selector pointing at the new Pods. The switch is atomic from DNS's perspective — zero application code changes across all consuming services. This is the Strangler Fig pattern applied at the Kubernetes networking layer.
