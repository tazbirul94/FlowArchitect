Kubernetes Config & Secrets

Manage application configuration and sensitive credentials safely in Kubernetes — from ConfigMaps and Secrets to Sealed Secrets and External Secrets Operator.

The Story Behind Config & Secrets Management

🔑 The Hardcoded Password Disaster

It is 3 AM. The TicketFlow on-call engineer gets paged: the ticket-service is down in production. After frantic investigation, the culprit is found — a developer rotated the PostgreSQL database password after a security audit, but the new password only made it into staging. Production containers still have the old, now-rejected password baked directly into the Docker image as an environment variable in the Dockerfile.

Worse, a security scan the next morning reveals that three months of git history contain the production database password in plain text. The password has been rotated, but the history — and anyone who cloned the repository — now holds a record of what production credentials once looked like. The security team requires a full audit, all tokens to be rotated, and an incident report filed.

This disaster is completely avoidable. Kubernetes provides first-class primitives for separating configuration from code: ConfigMaps for non-sensitive settings and Secrets for credentials — each injected into containers at runtime without ever touching the image or the source code repository. Combined with tools like Sealed Secrets or the External Secrets Operator, you get a full secret lifecycle that satisfies security auditors and lets developers sleep at night.

The Simple Explanation

🥪 Beginner Analogy

Think of your application container as a vending machine. The vending machine (your app) needs two kinds of information to work: settings like "how long to wait before timing out" (non-sensitive, you can write these on a sticky note on the machine) and secrets like "the master unlock code" (sensitive, you definitely should not write those on a sticky note).

A ConfigMap is the sticky note — publicly visible configuration that changes between environments but is not secret. A Secret is the locked box handed to the machine privately — the database password, the API key, the TLS certificate. The machine reads both at startup, but you never need to bake either one into the machine's firmware (your container image). You can swap the sticky note or replace the locked box without rebuilding the machine at all.

Configuration management in Kubernetes answers a deceptively simple question: how do you give your app the settings it needs without hardcoding them into your container image? The answer matters for three reasons:

  • Environment promotion — the same image should run in dev, staging, and production. Only the configuration changes between environments. If config is baked into the image, you need different images per environment — defeating the "build once, deploy anywhere" promise of containers.
  • Security — credentials in container images or git history are a serious vulnerability. Anyone with pull access to a registry, or read access to the repository, gets the credentials.
  • Operational flexibility — configuration changes (feature flags, connection strings, log levels) should not require a full image rebuild and redeployment. With ConfigMaps and Secrets, you can update configuration and roll it out independently.

ConfigMaps and Secrets — Technical Deep Dive

ConfigMap

A ConfigMap is a Kubernetes API object that stores non-confidential key-value pairs. The data is stored in etcd (Kubernetes' backing store) unencrypted and in plain text. ConfigMaps are appropriate for:

  • Application-level settings (log level, feature flags, thread pool sizes)
  • External service endpoints that vary by environment (dev DB host vs prod DB host)
  • Configuration files (nginx.conf, application.properties, logback.xml)

ConfigMaps can hold up to 1 MiB of data. For larger config files, consider a dedicated config store.

Secret

A Secret is structurally identical to a ConfigMap but is intended for sensitive data. Kubernetes Secrets are base64-encoded, not encrypted by default. This is a critical distinction: base64 is an encoding, not encryption. Anyone with kubectl get secret access can decode the values in seconds.

Secrets are safer than ConfigMaps only because:

  • They are not printed in pod specs or logs by default
  • They can be excluded from etcd backups and audit logs
  • RBAC can grant read access to Secrets separately from ConfigMaps
  • They can be encrypted at rest in etcd using EncryptionConfiguration

For real encryption, you need additional tooling: Sealed Secrets (encrypt secrets in git) or External Secrets Operator (pull from AWS Secrets Manager, HashiCorp Vault, etc.).

Three Ways to Consume ConfigMaps and Secrets

Method How it works Best for Hot reload on update
env (individual key) Each key mapped explicitly to a named env var Small number of known keys; explicit mapping required No — requires pod restart
envFrom (all keys) All keys from ConfigMap/Secret become env vars automatically Many settings; no explicit mapping needed No — requires pod restart
Volume mount (files) Each key becomes a file on a mounted filesystem path Config files (nginx.conf, .properties); apps that watch for changes Yes — kubelet updates files within ~1 min

Sealed Secrets (Bitnami)

Sealed Secrets solve the "secrets in git" problem. A SealedSecret is a Kubernetes custom resource that contains a Secret encrypted with the cluster's public key. The Sealed Secrets controller (running in the cluster) decrypts it and creates the real Secret. You can safely commit SealedSecret YAML to git — only the controller in your specific cluster can decrypt it.

External Secrets Operator (ESO)

ESO pulls secrets from external stores (AWS Secrets Manager, AWS SSM Parameter Store, HashiCorp Vault, GCP Secret Manager, Azure Key Vault) and synchronizes them as Kubernetes Secrets. This is the gold standard for production: secrets live in a dedicated secrets manager with audit trails, rotation policies, and access controls — not in Kubernetes etcd at all.

Enterprise Scenario: TicketFlow DB Credentials

🏢 TicketFlow — ticket-service on AWS EKS

TicketFlow's ticket-service (Spring Boot) connects to a PostgreSQL RDS instance. It needs three categories of configuration:

  • Non-sensitive config — database host, port, pool size, log level, feature flags. These vary per environment but are not sensitive. Stored in ConfigMaps.
  • Sensitive credentials — database username, database password, JWT signing key. Stored in AWS Secrets Manager and synchronized to Kubernetes Secrets via External Secrets Operator.
  • TLS certificates — the mutual TLS certificate and private key for service-to-service communication. Mounted as files via Kubernetes Secrets (volume mount, not env vars — private keys should never be in environment variables).

The policy at TicketFlow: nothing sensitive ever touches the git repository or the container image. ConfigMap YAML files are committed to git (and reviewed in PRs). SealedSecret YAML is committed to git for low-sensitivity secrets (JWT audience, non-prod API keys). High-sensitivity secrets (DB passwords, signing keys) exist only in AWS Secrets Manager, synchronized to Kubernetes at runtime by ESO.

Pod spec — all three injection methods

# deployment-ticket-service.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: ticket-service
  namespace: ticketflow
spec:
  replicas: 3
  selector:
    matchLabels:
      app: ticket-service
  template:
    metadata:
      labels:
        app: ticket-service
    spec:
      serviceAccountName: ticket-service-sa  # RBAC: minimal permissions
      containers:
        - name: ticket-service
          image: nexus.ticketflow.io:5000/ticket-service:v2.4.1
          ports:
            - containerPort: 8080

          # ── Method 1: env — individual keys from ConfigMap ──────────
          env:
            - name: DB_HOST
              valueFrom:
                configMapKeyRef:
                  name: ticket-service-config
                  key: DB_HOST
            - name: DB_PORT
              valueFrom:
                configMapKeyRef:
                  name: ticket-service-config
                  key: DB_PORT
            # Sensitive values from Secret (individual key)
            - name: DB_USERNAME
              valueFrom:
                secretKeyRef:
                  name: ticket-db-credentials
                  key: DB_USERNAME
            - name: DB_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: ticket-db-credentials
                  key: DB_PASSWORD

          # ── Method 2: envFrom — ALL keys from ConfigMap at once ─────
          envFrom:
            - configMapRef:
                name: ticket-service-config
            # Caution: envFrom from a Secret dumps all keys as env vars.
            # Avoid for private keys or multi-line certs — use volumeMount.

          # ── Method 3: volumeMount — config file + TLS certs ─────────
          volumeMounts:
            # Mount application.properties from ConfigMap key
            - name: app-config-volume
              mountPath: /app/config/application.properties
              subPath: application.properties
              readOnly: true
            # Mount TLS cert and key from Secret as files
            - name: tls-volume
              mountPath: /app/tls
              readOnly: true

          resources:
            requests:
              cpu: "250m"
              memory: "512Mi"
            limits:
              cpu: "1000m"
              memory: "1Gi"

      volumes:
        - name: app-config-volume
          configMap:
            name: ticket-service-config
        - name: tls-volume
          secret:
            secretName: ticket-service-tls
            defaultMode: 0400  # owner read-only — important for private keys

Sealed Secrets — safe to commit to git

# Install the kubeseal CLI
brew install kubeseal  # macOS
# or
wget https://github.com/bitnami-labs/sealed-secrets/releases/download/v0.24.0/kubeseal-0.24.0-linux-amd64.tar.gz

# Fetch the cluster's public certificate
kubeseal --fetch-cert \
  --controller-namespace kube-system \
  --controller-name sealed-secrets-controller \
  > sealed-secrets-cert.pem

# Seal an existing Secret (reads from stdin, outputs SealedSecret)
kubectl create secret generic ticket-jwt-config \
  --namespace ticketflow \
  --from-literal=JWT_AUDIENCE=ticketflow-api \
  --dry-run=client -o yaml \
  | kubeseal --cert sealed-secrets-cert.pem --format yaml \
  > sealed-secret-jwt-config.yaml

# The output file is safe to commit to git — only the cluster can decrypt it
cat sealed-secret-jwt-config.yaml
# sealed-secret-jwt-config.yaml — safe to commit to git
apiVersion: bitnami.com/v1alpha1
kind: SealedSecret
metadata:
  name: ticket-jwt-config
  namespace: ticketflow
spec:
  encryptedData:
    # These ciphertext blobs are encrypted with the cluster's RSA public key
    JWT_AUDIENCE: AgBy8hW9pZ3k1k7bfXwl...long-encrypted-blob...
  template:
    metadata:
      name: ticket-jwt-config
      namespace: ticketflow

External Secrets Operator — pull from AWS Secrets Manager

# external-secret-ticket-db.yaml
# Requires ESO installed: helm install external-secrets external-secrets/external-secrets

# Step 1: SecretStore — defines where to pull secrets from
apiVersion: external-secrets.io/v1beta1
kind: SecretStore
metadata:
  name: aws-secrets-manager
  namespace: ticketflow
spec:
  provider:
    aws:
      service: SecretsManager
      region: eu-west-1
      auth:
        jwt:
          serviceAccountRef:
            name: ticket-service-sa  # Uses IRSA (IAM Roles for Service Accounts)
---
# Step 2: ExternalSecret — defines which secrets to sync and how
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: ticket-db-credentials
  namespace: ticketflow
spec:
  # Sync every 1 minute — supports automatic rotation
  refreshInterval: 1m

  secretStoreRef:
    name: aws-secrets-manager
    kind: SecretStore

  target:
    name: ticket-db-credentials  # Name of the Kubernetes Secret to create/update
    creationPolicy: Owner

  data:
    # Pull DB_USERNAME from AWS Secrets Manager path
    - secretKey: DB_USERNAME
      remoteRef:
        key: ticketflow/production/ticket-service/db
        property: username

    # Pull DB_PASSWORD from AWS Secrets Manager path
    - secretKey: DB_PASSWORD
      remoteRef:
        key: ticketflow/production/ticket-service/db
        property: password

    # Pull the entire JWT signing key secret (single-value secret)
    - secretKey: JWT_SIGNING_KEY
      remoteRef:
        key: ticketflow/production/ticket-service/jwt-signing-key

RBAC — restricting Secret access

# rbac-ticket-service.yaml
# ServiceAccount for ticket-service (minimal footprint)
apiVersion: v1
kind: ServiceAccount
metadata:
  name: ticket-service-sa
  namespace: ticketflow
  annotations:
    # IRSA: this ServiceAccount is allowed to assume the IAM role
    eks.amazonaws.com/role-arn: arn:aws:iam::123456789012:role/ticketflow-ticket-service-role
---
# Role: what ticket-service-sa can do with secrets in this namespace
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: ticket-service-role
  namespace: ticketflow
rules:
  # Can read its own ConfigMap
  - apiGroups: [""]
    resources: ["configmaps"]
    resourceNames: ["ticket-service-config"]
    verbs: ["get", "watch", "list"]
  # Can read ONLY its own secrets — not all secrets in the namespace
  - apiGroups: [""]
    resources: ["secrets"]
    resourceNames: ["ticket-db-credentials", "ticket-service-tls", "ticket-jwt-config"]
    verbs: ["get"]
  # Cannot list all secrets (prevents enumeration)
---
# RoleBinding: bind the Role to the ServiceAccount
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: ticket-service-rolebinding
  namespace: ticketflow
subjects:
  - kind: ServiceAccount
    name: ticket-service-sa
    namespace: ticketflow
roleRef:
  kind: Role
  apiGroupName: rbac.authorization.k8s.io
  name: ticket-service-role

Behind the Scenes

How does Kubernetes store and serve ConfigMaps and Secrets internally?

Storage in etcd: ConfigMaps and Secrets are persisted as objects in etcd, the distributed key-value store that backs all Kubernetes state. By default, the data is stored in plain text (base64 for Secrets, which is identical to plain text from a security standpoint). You can enable encryption at rest via an EncryptionConfiguration on the API server, which encrypts Secret data before writing it to etcd using AES-CBC or AES-GCM with a KMS provider.

Delivery to pods — environment variables: When the kubelet creates a pod, it calls the API server to get the pod spec. If the spec references ConfigMap/Secret keys via env or envFrom, the kubelet fetches those values from the API server and injects them into the container's environment before starting the container process. Environment variable values are frozen at container start — updating the ConfigMap or Secret does not update the running container's environment; you must restart the pod.

Delivery to pods — volume mounts: Volume-mounted ConfigMaps and Secrets work differently. The kubelet fetches the data and writes it to a tmpfs (in-memory filesystem) directory on the node. The container sees files at the mount path. Kubernetes keeps a watch on the ConfigMap/Secret, and when it changes, the kubelet updates the files on tmpfs within approximately one minute (controlled by the --sync-frequency kubelet flag). Applications that use inotify or periodic file reads can pick up changes without restarting.

Projected volumes: Multiple ConfigMaps, Secrets, and other sources (like service account tokens and downward API data) can be merged into a single volume mount using a projected volume type. This avoids cluttering the pod spec with many separate volume entries.

Secret encryption at rest (best practice): On AWS EKS, enable envelope encryption on the EKS cluster using a KMS key. EKS will use the KMS key to encrypt the DEK (data encryption key) that encrypts Secrets in etcd. This satisfies PCI-DSS and SOC 2 requirements for encryption at rest.

How does External Secrets Operator work under the hood?

ESO is a Kubernetes operator — a set of controllers that extend the Kubernetes API with custom resources (ExternalSecret, SecretStore, ClusterSecretStore).

  1. You create an ExternalSecret resource specifying: which SecretStore to use, which external secret paths to fetch, how to map them to Kubernetes Secret keys, and how often to re-sync (refreshInterval).
  2. The ESO controller watches for ExternalSecret objects. When it detects one, it authenticates to the external provider using the credentials configured in the SecretStore (for AWS, this is typically IRSA — IAM Roles for Service Accounts, which uses a projected service account token that AWS STS can verify).
  3. ESO fetches the secret values from the external provider and creates or updates a Kubernetes Secret with those values.
  4. ESO stores a hash of the fetched values. On the next sync interval, it fetches again and compares hashes. If the external secret has changed (e.g., after a rotation), ESO updates the Kubernetes Secret. Since the Secret is volume-mounted, the kubelet propagates the update to running pods within ~60 seconds — achieving near-zero-downtime secret rotation.
  5. The external secret values themselves never leave the cluster unencrypted in transit — ESO uses TLS to communicate with AWS Secrets Manager.

Common Mistakes

❌ Mistake 1: Committing Secrets to Git

The most common and most dangerous mistake. A developer adds a secret.yaml with base64-encoded passwords to the repository "just temporarily." It gets merged, git history is immutable, and the credentials are now in every clone of the repository forever. Even force-pushing to remove the commit does not help if others have already pulled.

Fix: Add *secret*.yaml and *credentials*.yaml to .gitignore. Use SealedSecrets for secrets that must live in git. Use ESO for production secrets. Set up git pre-commit hooks (e.g., gitleaks) to scan for accidentally committed secrets before every push.

❌ Mistake 2: Treating base64 as Encryption

Seeing the kind: Secret label and the scrambled-looking base64 values, engineers assume the data is encrypted. Running echo dGlja2V0X2FwcF91c2Vy | base64 --decode takes two seconds and reveals the plaintext. Anyone with kubectl get secret RBAC permission — or read access to etcd backups — can read every Secret in plain text.

Fix: Enable etcd encryption at rest via EncryptionConfiguration (on self-managed clusters) or KMS envelope encryption (on EKS/GKE). Use ESO so that production secrets never actually live in etcd — they only exist in AWS Secrets Manager and are fetched ephemerally.

❌ Mistake 3: Using envFrom for Secrets Containing Private Keys or Certificates

Private keys and certificates are multi-line PEM-encoded strings. When injected as environment variables, they often break due to newline handling in shells, get logged in process listings (ps aux on some systems shows env vars), and are accessible to any process in the container. envFrom dumps all secret keys as env vars — if a secret contains a private key, it will be an env var.

Fix: Always mount TLS certificates and private keys as volume-mounted files with defaultMode: 0400 (owner read-only). Use env vars only for short scalar values like passwords and API keys.

❌ Mistake 4: No RBAC Restriction on Secrets

By default in many clusters, the view ClusterRole includes the ability to list and get Secrets. Service accounts created without explicit RBAC often inherit broad permissions. A compromised notification-service pod with overly broad permissions could read the ticket-db-credentials Secret that belongs to ticket-service.

Fix: Give each service account a dedicated Role with resourceNames scoping — only the specific Secret names that service needs. Never grant list on Secrets (it allows enumerating all secret names). Audit RBAC regularly with kubectl auth can-i --list --as system:serviceaccount:ticketflow:ticket-service-sa.

❌ Mistake 5: Restarting Pods to Pick Up Config Changes Instead of Using Volume Mounts

Teams update a ConfigMap and then trigger a rolling restart of all pods to apply the change. This causes unnecessary churn — rolling restarts terminate and recreate pods, interrupting in-flight requests. For high-traffic services, even a graceful rolling restart has cost.

Fix: For configuration that changes frequently (feature flags, log levels, tuning parameters), use volume-mounted ConfigMaps. The kubelet automatically propagates updates to mounted files within ~60 seconds. Applications that support hot-reload (Spring Boot Actuator refresh, NGINX -s reload, application.properties reload) can pick up changes with zero restarts.

Architect Perspective

🏛️ Architect Note

The way an organization manages secrets is a direct indicator of its security maturity. Here is how the progression typically looks:

  • Level 0 (Dangerous) — Secrets in Dockerfiles, hardcoded in source, committed to git. Every developer and every CI runner sees every production credential. Rotation requires a code change and deployment.
  • Level 1 (Basic) — Kubernetes Secrets, but not encrypted at rest, no RBAC scoping, managed manually with kubectl apply. Better than Level 0, but credentials still live unencrypted in etcd and anyone with cluster read access sees them.
  • Level 2 (Standard) — Sealed Secrets. Secrets encrypted in git. Rotation requires generating a new SealedSecret and committing. Good for most organizations that are not handling PII or payment data at scale.
  • Level 3 (Production-grade) — External Secrets Operator pulling from AWS Secrets Manager or HashiCorp Vault. Secrets have rotation policies, audit trails, versioning, and access logging. Kubernetes etcd never holds the sensitive value long-term — it is a synchronized cache. Rotation is automated and zero-downtime.

Key architectural decision — scope of secrets: Keep secrets as narrow in scope as possible. ticket-service should have exactly one credential to its own database — not a shared "service account" with access to all databases. This limits blast radius: if ticket-service is compromised, only the tickets database is at risk, not payment or user data.

Config vs. code boundary: A rule of thumb — if you need to rebuild the image to change it, it is in the wrong place. Database host, feature flags, log levels, and thread pool sizes should all be externalized. The image contains only behavior; the cluster provides all context needed to run that behavior.

Production Readiness Checklist

  • No secrets in Dockerfiles, source code, or .env files committed to git
  • .gitignore includes *secret*.yaml, *.env, and credential files
  • gitleaks or equivalent pre-commit hook scanning for accidental secret commits
  • etcd encryption at rest enabled (KMS envelope encryption on EKS)
  • All production secrets stored in AWS Secrets Manager (or HashiCorp Vault) — not bare Kubernetes Secrets
  • External Secrets Operator deployed and managing production secret sync
  • ESO refreshInterval set to ≤ 5 minutes to support rapid rotation
  • Each service has its own ServiceAccount with RBAC scoped to only its own named Secrets
  • No list permission on Secrets for any service account
  • TLS certificates and private keys mounted as files (not env vars), with defaultMode: 0400
  • ConfigMaps version-controlled in git and reviewed in pull requests
  • Secret rotation procedures documented, tested, and automated
  • Alerts configured on ESO sync failures (ESO exposes Prometheus metrics)
  • All Secret accesses logged via Kubernetes audit logs and forwarded to SIEM

Hands-On Lab: Mount Config + Secret into ticket-service

🧪 Lab — Requires kubectl access to a Kubernetes cluster (minikube or kind works)

In this lab you will create a ConfigMap and a Secret, then deploy a pod that consumes them via all three injection methods. You will see how env vars are frozen at startup while volume-mounted files update live.

Step 1: Create a namespace and ConfigMap

kubectl create namespace ticketflow-lab

kubectl create configmap ticket-service-config \
  --namespace ticketflow-lab \
  --from-literal=DB_HOST=postgres.ticketflow-lab.svc.cluster.local \
  --from-literal=DB_PORT=5432 \
  --from-literal=LOG_LEVEL=INFO \
  --from-literal=MAX_POOL_SIZE=10

kubectl get configmap ticket-service-config -n ticketflow-lab -o yaml

Step 2: Create a Secret

kubectl create secret generic ticket-db-credentials \
  --namespace ticketflow-lab \
  --from-literal=DB_USERNAME=ticket_app_user \
  --from-literal=DB_PASSWORD='LabPassword123!'

# Observe base64 encoding — this is NOT encryption
kubectl get secret ticket-db-credentials -n ticketflow-lab -o yaml

Step 3: Deploy a test pod consuming both

# Save as lab-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: ticket-service-lab
  namespace: ticketflow-lab
spec:
  containers:
    - name: ticket-service-lab
      image: busybox:1.36
      command: ["sh", "-c", "sleep 3600"]
      env:
        # Method 1: individual keys
        - name: DB_HOST
          valueFrom:
            configMapKeyRef:
              name: ticket-service-config
              key: DB_HOST
        - name: DB_PASSWORD
          valueFrom:
            secretKeyRef:
              name: ticket-db-credentials
              key: DB_PASSWORD
      envFrom:
        # Method 2: all ConfigMap keys as env vars
        - configMapRef:
            name: ticket-service-config
      volumeMounts:
        # Method 3: ConfigMap as a file
        - name: config-volume
          mountPath: /etc/ticket/config
          readOnly: true
        # Method 3: Secret as a file
        - name: secret-volume
          mountPath: /etc/ticket/secrets
          readOnly: true
  volumes:
    - name: config-volume
      configMap:
        name: ticket-service-config
    - name: secret-volume
      secret:
        secretName: ticket-db-credentials
        defaultMode: 0400
kubectl apply -f lab-pod.yaml
kubectl wait --for=condition=Ready pod/ticket-service-lab -n ticketflow-lab --timeout=60s

Step 4: Inspect how config is injected

# See env vars injected (both Method 1 individual and Method 2 envFrom)
kubectl exec -it ticket-service-lab -n ticketflow-lab -- env | grep -E 'DB_|LOG_|MAX_'

# See the ConfigMap mounted as files (one file per key)
kubectl exec -it ticket-service-lab -n ticketflow-lab -- ls /etc/ticket/config/
kubectl exec -it ticket-service-lab -n ticketflow-lab -- cat /etc/ticket/config/LOG_LEVEL

# See the Secret mounted as files (file permissions should be 0400)
kubectl exec -it ticket-service-lab -n ticketflow-lab -- ls -la /etc/ticket/secrets/
kubectl exec -it ticket-service-lab -n ticketflow-lab -- cat /etc/ticket/secrets/DB_PASSWORD

Step 5: Update ConfigMap and observe live file update

# Update LOG_LEVEL from INFO to DEBUG
kubectl patch configmap ticket-service-config -n ticketflow-lab \
  --patch '{"data": {"LOG_LEVEL": "DEBUG"}}'

# Wait ~60 seconds, then check the mounted file — it will have updated automatically
# (no pod restart needed for volume mounts)
kubectl exec -it ticket-service-lab -n ticketflow-lab -- cat /etc/ticket/config/LOG_LEVEL
# Expected output: DEBUG

# But env var is still the old value — frozen at container start
kubectl exec -it ticket-service-lab -n ticketflow-lab -- sh -c 'echo $LOG_LEVEL'
# Expected output: INFO  (still the old value until pod is restarted)

Step 6: Clean up

kubectl delete namespace ticketflow-lab

Key observations from this lab:

  1. Volume-mounted files update live (within ~60s) when the ConfigMap changes. Env vars do not — they require a pod restart.
  2. Secret files are mounted with strict permissions (0400) when defaultMode is set — only the container process owner can read them.
  3. Base64-encoded Secret values appear as plain text in the mounted files. They are decoded by Kubernetes before mounting.
  4. Both ConfigMap and Secret keys appear as individual files in the mounted directory — one file per key.

Interview Questions

These are the most frequently asked Kubernetes secrets and configuration questions in senior engineering and DevOps interviews. Each answer covers both theory and practical implications.

Q1: What is the difference between a ConfigMap and a Secret in Kubernetes, and when would you use each?

Both ConfigMap and Secret store key-value data in Kubernetes etcd and can be injected into pods the same three ways: individual env vars (env.valueFrom), bulk env vars (envFrom), or volume-mounted files. The structural difference is minimal: Secrets are base64-encoded and have a few additional safety properties.

Use ConfigMap for: non-sensitive application settings — database host, port, log level, feature flags, thread pool sizes, configuration files (nginx.conf, application.properties). These are safe to commit to git as YAML and review in PRs.

Use Secret for: any value you would not want printed in logs or visible to everyone with cluster read access — database passwords, API keys, JWT signing keys, TLS certificates, SSH private keys, OAuth tokens.

Critical nuance: Secrets are base64-encoded, not encrypted. Anyone with kubectl get secret permission, or access to etcd backups, can decode them in seconds. For true security: enable etcd encryption at rest, restrict RBAC on Secrets tightly, and use External Secrets Operator so the actual sensitive value lives in AWS Secrets Manager rather than etcd.

Q2: What are the three ways to inject a ConfigMap or Secret into a pod, and what are the trade-offs?

There are three injection methods, each with distinct trade-offs:

1. env.valueFrom (individual key mapping): Each key is explicitly mapped to a named environment variable. Verbose but precise — you control exactly which keys become which env vars, and mismatched key names are caught at pod start. Values are frozen at container start; updating the ConfigMap requires a pod restart to take effect.

2. envFrom (bulk import): All keys in a ConfigMap or Secret become environment variables automatically. Convenient but risky — a new key added to the ConfigMap silently becomes an env var, potentially overwriting an existing env var with the same name. Same frozen-at-start limitation. Avoid envFrom with Secrets containing multi-line values like PEM certificates — they break env var injection.

3. Volume mount (files): Each key becomes a file in a mounted directory (or a single file via subPath). This is the only method that supports live updates — the kubelet updates mounted files within ~60 seconds of a ConfigMap/Secret change, with no pod restart. Required for TLS certificates and private keys (which should never be env vars). Applications must be coded to re-read the file or use inotify to react to changes.

Rule of thumb: Use volume mounts for config files and certificates. Use env.valueFrom for a small number of explicit scalar values. Avoid envFrom for Secrets in production.

Q3: Are Kubernetes Secrets secure? How would you make them more secure?

Out of the box, Kubernetes Secrets provide limited security. The data is base64-encoded (not encrypted) and stored as plain text in etcd. The word "Secret" is misleading — it is more of a convention to separate sensitive data from ConfigMaps, with a few additional protections (not printed in pod specs, separate RBAC resource type, not shown in kubectl describe pod output).

Steps to make Secrets genuinely secure:

  • Encryption at rest: Enable EncryptionConfiguration on the API server to encrypt Secret data before writing it to etcd. On EKS, use KMS envelope encryption. This protects against someone gaining direct access to the etcd datastore or its backups.
  • RBAC scoping: Use resourceNames in Role definitions so each ServiceAccount can only access the specific Secrets it needs — not all Secrets in the namespace. Remove list permission on Secrets from all non-admin service accounts.
  • External Secrets Operator: The most robust approach — Secrets live in AWS Secrets Manager with audit logging, versioning, and rotation policies. ESO synchronizes them to Kubernetes ephemerally. etcd holds a synchronized copy that is refreshed periodically and expires if ESO is removed.
  • Audit logging: Enable Kubernetes audit logs to record every get, list, and watch on Secrets. Forward audit logs to a SIEM for alerting on unusual access patterns.
  • Namespace isolation: Keep sensitive workloads in dedicated namespaces with NetworkPolicies and RBAC tightly controlled.
Q4: What is Sealed Secrets and how does it solve the "secrets in git" problem?

Sealed Secrets (by Bitnami) is a Kubernetes controller and CLI tool (kubeseal) that enables safely committing encrypted secrets to git.

How it works:

  1. The Sealed Secrets controller runs in the cluster and holds an asymmetric key pair. You fetch its public certificate with kubeseal --fetch-cert.
  2. You use kubeseal to encrypt a plain Kubernetes Secret using the cluster's public key. The output is a SealedSecret custom resource containing ciphertext that only the controller can decrypt.
  3. You commit the SealedSecret YAML to git. It is safe because the ciphertext cannot be decrypted without the private key held by the controller in your specific cluster.
  4. Argo CD or Flux applies the SealedSecret to the cluster. The controller decrypts it and creates the actual Kubernetes Secret, which is available to pods.

Limitations: The private key in the controller is a single point of failure — if lost, all Sealed Secrets are undecryptable. Back up the private key securely. Sealed Secrets are cluster-specific (encrypted for one cluster's key pair), so you cannot reuse the same SealedSecret across multiple clusters without re-encrypting. For cross-cluster or multi-region setups, ESO with a central secrets manager is more appropriate.

Q5: How does External Secrets Operator enable zero-downtime secret rotation?

Zero-downtime rotation is achieved by combining three capabilities: ESO's continuous re-sync, Kubernetes volume mount live updates, and application-level credential refresh.

The rotation flow:

  1. The DBA or security automation rotates the database password in AWS Secrets Manager. AWS creates a new secret version; the old version is deprecated but not immediately deleted.
  2. ESO polls Secrets Manager at the interval defined in refreshInterval (e.g., every 1 minute). It detects the new version and updates the corresponding Kubernetes Secret.
  3. The kubelet detects the Secret update and propagates the new values to any volume-mounted files within ~60 seconds. Pods using volume mounts get the new password as a file update without restarting.
  4. Applications reading the credential file (Spring Boot applications can use Kubernetes ConfigMap reload, or read the file per-request) begin using the new password transparently.

Why env vars break this pattern: If the DB_PASSWORD is injected as an environment variable (not a volume mount), it is frozen at pod start. The pod cannot pick up the rotated password without a restart. Rolling restarts during rotation create a window where old and new passwords coexist in the running fleet — if the old password is revoked before all pods restart, some pods will fail to authenticate.

Best practice: Always mount credentials that are subject to rotation as files. Configure your database connection pool to re-authenticate when it encounters an authentication error (most production pools support this). This allows seamless rotation with no service interruption.

Q6: How would you design the secrets management strategy for a new microservices platform on AWS EKS from scratch?

A complete secrets management architecture for AWS EKS should cover: storage, access control, injection, rotation, and auditability.

Storage — AWS Secrets Manager: All secrets live in AWS Secrets Manager, organized by path: /{env}/{service-name}/{secret-name} (e.g., /production/ticket-service/db-credentials). Secrets Manager handles versioning, rotation scheduling, and cross-region replication.

Access control — IRSA: Each Kubernetes ServiceAccount is annotated with an IAM role ARN (eks.amazonaws.com/role-arn). The IAM role has an IAM policy granting secretsmanager:GetSecretValue only for the specific secret paths that service needs — not a wildcard. This is the principle of least privilege at the IAM layer.

Injection — External Secrets Operator: ESO is deployed cluster-wide. Each service has an ExternalSecret manifest (committed to the GitOps repo) that declares which secrets to pull, maps them to Kubernetes Secret keys, and sets a refreshInterval of 1–5 minutes. ESO creates and maintains the Kubernetes Secrets automatically.

Rotation — automated: Secrets Manager rotation lambdas rotate database passwords on a schedule (e.g., every 30 days for production). ESO picks up new versions within the refresh interval. Applications use volume-mounted credentials and connection pool re-auth to handle rotation transparently.

Non-sensitive config — ConfigMaps in git: Environment-specific ConfigMaps (DB_HOST, LOG_LEVEL, feature flags) are YAML files in the GitOps repository, reviewed in PRs, and applied by Argo CD. No secrets here.

GitOps repository — SealedSecrets for low-sensitivity: For non-prod environments or non-sensitive "secrets" (JWT audience, feature toggle keys), Sealed Secrets allows managing them in git without a full ESO setup.

Auditability: Enable EKS audit logging. Enable CloudTrail for AWS Secrets Manager. Set up alerts for unusual secret access patterns (e.g., a service accessing another service's secrets). Forward all logs to a centralized SIEM.
