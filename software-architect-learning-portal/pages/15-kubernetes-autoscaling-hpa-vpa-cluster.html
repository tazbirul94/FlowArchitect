Kubernetes Autoscaling: HPA, VPA & Cluster Autoscaler

Master all three layers of Kubernetes autoscaling — pod replicas, pod resource right-sizing, and node provisioning — plus KEDA for event-driven scale-to-zero.

The Story Behind Autoscaling

Concert Night at TicketFlow

It is 9:59 AM on a Friday. Beyoncé's world tour just went on sale. In 60 seconds, 400,000 fans will simultaneously hammer TicketFlow's ticket-service endpoint. At 9:55 AM, the platform is idling on 3 pods consuming a total of 600 MB of RAM. By 10:02 AM it needs to handle 80,000 requests per second.

Before autoscaling: the SRE team would pre-provision 200 pods the night before — paying for them all night, scrambling to scale down Monday morning, and still occasionally getting the numbers wrong. One Friday they under-provisioned by 40% and the site returned 503 errors for 11 minutes while Taylor Swift tickets sold out. That 11 minutes cost $2.3 million in lost sales and a viral Twitter storm.

After Kubernetes autoscaling: the cluster watches CPU, memory, and custom queue depth metrics in real time. When the concert sale opens, HPA spins up 180 new pod replicas in under 90 seconds. Cluster Autoscaler provisions 12 new EC2 nodes to accommodate them. The VPA silently adjusts container memory requests so JVM-heavy ticket-service pods get the right amount. After the rush, everything scales back down automatically. The team slept through Friday morning.

What is Autoscaling?

Beginner Analogy

Think of a restaurant. On a quiet Tuesday, two waiters can handle every table. On New Year's Eve, you call in 20 extra staff. On a regular night, the kitchen is fine with 4 burners lit — but on a busy Saturday you also open the second kitchen entirely (a new node).

Kubernetes autoscaling works exactly like this across three layers:

  • HPA (Horizontal Pod Autoscaler) — calls in more waiters (more pod copies) when demand rises.
  • VPA (Vertical Pod Autoscaler) — makes each waiter stronger by giving them the right tools (adjusts CPU/memory limits per pod).
  • Cluster Autoscaler — opens new kitchens when all existing space is full (adds new nodes to the cluster).
  • KEDA — calls in staff based on specific triggers like "the reservation phone is ringing off the hook" (event-driven scale based on queues, HTTP traffic, custom sources).

Technical Deep Dive

Horizontal Pod Autoscaler (HPA)

HPA is a Kubernetes controller that automatically adjusts the number of pod replicas in a Deployment, ReplicaSet, or StatefulSet. It operates on a control loop — by default every 15 seconds — comparing observed metric values against desired targets.

The scaling formula Kubernetes uses internally is:

desiredReplicas = ceil[currentReplicas * (currentMetricValue / desiredMetricValue)]

Example:
  currentReplicas = 3
  currentCPU     = 80% (of requested)
  targetCPU      = 50%
  desiredReplicas = ceil[3 * (80 / 50)] = ceil[4.8] = 5

HPA supports three metric sources:

  • Resource metrics — CPU and memory utilization relative to pod requests. Sourced from the metrics-server add-on (must be installed separately on EKS).
  • Custom metrics — application-level metrics exposed through the Custom Metrics API adapter (e.g., Prometheus Adapter). Examples: requests-per-second, queue depth, active WebSocket connections.
  • External metrics — metrics from systems outside the cluster entirely. Examples: SQS queue length, Kafka lag, Google Pub/Sub messages. Requires an external metrics adapter or KEDA.

Two important behavioral parameters control how aggressively HPA scales:

  • Scale-up stabilization window (default 0s) — how long HPA waits before scaling up after a spike. Low because scale-ups are urgent.
  • Scale-down stabilization window (default 300s) — how long HPA waits before scaling down after load drops. High to prevent flapping — rapid scale up/down cycles that waste resources and destabilize the application.

Vertical Pod Autoscaler (VPA)

VPA right-sizes container resource requests and limits based on actual historical and current usage. It observes how much CPU and memory pods actually use and recommends (or enforces) better-fitting requests. This solves the "ghost capacity" problem: a pod requesting 4 GB RAM that only ever uses 800 MB wastes 3.2 GB of node capacity that other pods could use.

VPA has four operating modes:

ModeBehaviorUse When
Off VPA calculates recommendations but applies nothing. You read them manually. Initial audit phase — understand current waste without touching production.
Initial VPA sets resource requests only when a pod is first created. Never updates a running pod. New deployments where you want right-sized initial requests but don't want forced restarts.
Recreate VPA evicts and recreates pods when their resources differ significantly from the recommendation. Non-critical workloads where occasional restarts are acceptable.
Auto Same as Recreate today; will support in-place resource updates when the feature graduates. Most recommended for production once in-place updates are stable.

VPA consists of three components: the Recommender (watches metrics and calculates ideal requests), the Updater (evicts pods that are too far from recommendations), and the Admission Controller (mutates new pods on creation to set the recommended requests).

Cluster Autoscaler (CA)

Cluster Autoscaler operates at the infrastructure level — adding or removing worker nodes. It watches for pods in the Pending state that could not be scheduled because no existing node has sufficient capacity. When it finds such pods, it triggers the cloud provider's API to provision new nodes from a configured node group (on EKS: an Auto Scaling Group).

CA also watches for underutilized nodes — nodes where all pods could fit onto other nodes with available capacity. After a configurable delay (default 10 minutes), it cordons the node, evicts its pods (respecting PodDisruptionBudgets), and terminates the EC2 instance.

Key configuration parameters:

  • --scale-down-delay-after-add (default 10m) — wait this long after a scale-up before considering scale-down. Prevents premature scale-down of freshly provisioned nodes.
  • --scale-down-unneeded-time (default 10m) — a node must be unneeded for this long before it is terminated.
  • --max-node-provision-time (default 15m) — if a node takes longer than this to join the cluster, CA considers the scale-up failed.

KEDA — Kubernetes Event-Driven Autoscaling

KEDA extends HPA to support event-driven scaling sources that standard HPA cannot handle natively. KEDA acts as an external metrics server and HPA controller in one. It connects to over 60 built-in scalers (event sources) and translates their metrics into HPA-compatible external metrics.

Key KEDA capabilities beyond standard HPA:

  • Scale to zero — KEDA can scale a Deployment down to 0 replicas when its event source is empty (e.g., empty SQS queue). Standard HPA has a minimum of 1. KEDA bridges 0 to 1 by bypassing HPA's minimum constraint.
  • Rich event sources — AWS SQS, Kafka consumer lag, RabbitMQ queue depth, HTTP request rate, Redis list length, Prometheus queries, cron schedules, and more.
  • ScaledObject — the KEDA CRD that defines what to scale, which scaler to use, min/max replicas, and cooldown settings.
  • ScaledJob — for batch workloads, KEDA can scale Kubernetes Jobs (not just Deployments) based on queue depth — one Job per queue message pattern.

Enterprise Scenario: TicketFlow Peak Load

TicketFlow — Concert Sale Architecture

TicketFlow's AWS EKS cluster hosts all six microservices. The ticket-service (Spring Boot) receives the most concentrated traffic spikes — up to 15,000 req/s during the first 90 seconds of a major sale. Here is the full autoscaling strategy the platform team implemented:

  • ticket-service — HPA on CPU (target 60%) with custom metric http_requests_in_flight via Prometheus Adapter. minReplicas: 3, maxReplicas: 200. VPA in Off mode (produces recommendations; team reviews weekly to right-size JVM heap settings).
  • payment-service (Python) — KEDA ScaledObject on SQS queue depth (payment confirmation queue). Scales from 2 to 50 replicas. Scale-to-zero disabled (minReplicas: 2) because payment confirmations must always process.
  • notification-service (Node.js) — KEDA ScaledObject on SQS queue depth (email/SMS queue). Full scale-to-zero enabled — notification-service sleeps at 0 pods when no events are in-flight, saving ~$180/month on EKS node capacity.
  • order-service (Go) — HPA on CPU (target 70%). Very efficient binary — rarely needs more than 5 replicas even at peak, but HPA is configured as a safety net (max: 30).
  • user-service (Node.js) — HPA on CPU + VPA in Recreate mode. VPA identified that user-service was requesting 512 MB but averaging 180 MB; after VPA, requests dropped to 200 MB, freeing capacity for 15 additional ticket-service pods per node.
  • Cluster Autoscaler — configured across 3 node groups: on-demand m5.2xlarge (always-on baseline), Spot c5.2xlarge (cost-optimized burst), and memory-optimized r5.xlarge (for VPA-expanded JVM pods). Scale-down delay: 15 minutes post-sale to avoid thrashing.

Result: The last Beyoncé sale scaled from 3 to 186 ticket-service replicas in 78 seconds. Cluster Autoscaler provisioned 9 Spot nodes. Zero 503 errors. Post-sale scale-down completed within 25 minutes, reducing AWS bill by ~72% compared to pre-provisioned static capacity.

Configuration Examples

HPA — CPU-Based Autoscaling for ticket-service

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: ticket-service-hpa
  namespace: production
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: ticket-service
  minReplicas: 3
  maxReplicas: 200
  metrics:
    # Scale on CPU utilization (relative to pod requests)
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 60
    # Scale on memory (secondary metric — both must be satisfied)
    - type: Resource
      resource:
        name: memory
        target:
          type: Utilization
          averageUtilization: 70
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 0   # Scale up immediately on traffic spike
      policies:
        - type: Pods
          value: 20                   # Add up to 20 pods per scaling period
          periodSeconds: 30
        - type: Percent
          value: 100                  # Or double pod count — whichever is larger
          periodSeconds: 30
      selectPolicy: Max               # Use the policy that allows the most scale
    scaleDown:
      stabilizationWindowSeconds: 300 # Wait 5 min before scaling down (prevent flap)
      policies:
        - type: Pods
          value: 5                    # Remove at most 5 pods per period
          periodSeconds: 60

HPA — Custom Metric (HTTP Requests In-Flight)

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: ticket-service-hpa-custom
  namespace: production
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: ticket-service
  minReplicas: 3
  maxReplicas: 200

    # Custom metric — requires Prometheus Adapter to be installed and configured
    # Metric name must match what Prometheus Adapter exposes via Custom Metrics API
    - type: Pods
      pods:
        metric:
          name: http_requests_in_flight   # exposed by ticket-service /metrics endpoint
        target:
          type: AverageValue
          averageValue: "50"              # Target: 50 in-flight requests per pod replica
    # Also keep CPU as a fallback metric
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 60

VPA — Right-Sizing user-service

apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: user-service-vpa
  namespace: production
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: user-service
  updatePolicy:
    updateMode: "Recreate"            # Evict + recreate pods with updated requests
  resourcePolicy:
    containerPolicies:
      - containerName: user-service
        minAllowed:
          cpu: "50m"                  # Never go below 50 millicores (startup need)
          memory: "128Mi"
        maxAllowed:
          cpu: "2"                    # Never exceed 2 cores (runaway protection)
          memory: "1Gi"
        controlledResources:
          - cpu
          - memory
        # VPA will update both requests AND limits
        controlledValues: RequestsAndLimits

Behind the Scenes

How does HPA actually retrieve metrics from Prometheus?

Standard Kubernetes only ships with the metrics-server, which provides CPU and memory from the kubelet. For custom metrics (like http_requests_in_flight), you need a bridge between Prometheus and the Kubernetes Custom Metrics API.

  1. Application exposes metrics — ticket-service's Spring Boot app has a /actuator/prometheus endpoint. A Prometheus instance scrapes this endpoint every 15 seconds and stores the time-series data.
  2. Prometheus Adapter installed — the Prometheus Adapter deployment runs in the cluster. It is configured with rules that map Prometheus metric names to Kubernetes Custom Metrics API paths. Example rule: http_requests_in_flight{namespace="production",pod=~"ticket-service-.*"} → exposed as pods/http_requests_in_flight.
  3. HPA queries Custom Metrics API — the HPA controller calls GET /apis/custom.metrics.k8s.io/v1beta1/namespaces/production/pods/*/http_requests_in_flight. The Prometheus Adapter intercepts this request, queries Prometheus for the current value, and returns it.
  4. HPA calculates desired replicas — using the standard formula: ceil[currentReplicas * (currentValue / targetValue)]. If current in-flight is 350 across 5 replicas (70 per pod) and target is 50, desired = ceil[5 * (70/50)] = 7 replicas.
  5. HPA updates the Deployment scale subresource — the HPA controller patches /scale on the Deployment object, which the ReplicaSet controller then acts on by creating new pods.

Latency chain: metric scrape (15s) → Prometheus query → Adapter response → HPA evaluation → Deployment patch → Pod scheduled → Pod ready. Expect 30–90 seconds from load spike to traffic-serving pod. This is why predictive scaling (scaling in advance of known events) is important for hard spikes like concert sales.

How does Cluster Autoscaler decide which node group to expand?

When Cluster Autoscaler detects pending pods, it simulates scheduling: for each node group, it asks "if I added one node from this group, would the pending pods schedule onto it?"

  1. Expander strategy — the --expander flag controls which node group wins when multiple groups could accommodate the pods:
    • least-waste — pick the group whose node type wastes the least CPU/memory after placing the pods. Best for heterogeneous workloads (TicketFlow's default).
    • random — pick randomly. Useful when you want to spread load across groups evenly.
    • most-pods — pick the group that can schedule the most pending pods. Minimizes the number of scale-up events.
    • price — pick the cheapest option. Requires cloud-provider cost data.
  2. Node template — CA does not actually launch a node to check; it uses node group templates (based on existing nodes in the group) to simulate the node's resources.
  3. PodDisruptionBudgets respected — before evicting pods from underutilized nodes during scale-down, CA checks PDBs. If evicting a pod would violate a PDB (e.g., bringing replicas below minAvailable), CA skips that node.
  4. AWS Auto Scaling Groups — on EKS, CA calls the AWS Auto Scaling API: SetDesiredCapacity to increase the ASG. The EC2 instance launches, the kubelet registers the node, and CA watches for it to enter Ready state.
How does KEDA achieve true scale-to-zero?

Standard HPA has a minimum replica count of 1. It literally cannot set replicas to 0. KEDA works around this by taking direct ownership of the Deployment scale when the replica count needs to be 0:

  1. When the SQS queue empties and the cooldown period expires, KEDA directly patches the Deployment to 0 replicas — bypassing HPA entirely for this operation.
  2. KEDA creates an HPA internally for values between 1 and maxReplicas. For the 0-to-1 transition, KEDA handles it directly.
  3. When a new message arrives in the queue, KEDA detects it in its polling loop (every pollingInterval seconds), immediately sets replicas to 1, and then hands control back to HPA for further scaling.
  4. The cold-start problem: there is a window between a message arriving and the first pod being ready to process it. For notification-service this is acceptable (< 30s). For latency-sensitive services (payment-service), keep minReplicaCount: 2 to avoid cold starts.

KEDA also supports ScaledJob for true one-to-one message processing: one Job (batch pod) per queue message, each pod processes one message and exits. This pattern guarantees ordered, isolated processing and is used by TicketFlow's invoice-generation batch worker.

Common Mistakes

Mistake 1: Running HPA and VPA Together in Conflicting Modes

HPA scales replicas based on CPU utilization. VPA changes CPU requests. When VPA reduces a pod's CPU request, the pod's CPU utilization percentage (relative to request) instantly increases — which triggers HPA to scale up. When HPA scales up, load is distributed more, CPU per pod drops — VPA then reduces requests again. The two controllers fight in a loop.

Fix: Never use VPA in Auto or Recreate mode on the same resource metric that HPA uses. The safe combination is: HPA scales on CPU + VPA in Off mode (recommendations only). If you need both controllers active, configure HPA to scale on custom metrics (e.g., request rate) and let VPA manage CPU/memory requests without triggering HPA. Alternatively, disable CPU as an HPA metric and exclusively use VPA for CPU right-sizing.

Mistake 2: No Resource Requests Set on Pods

HPA calculates utilization as actualUsage / requestedAmount. If a pod has no CPU request set (resources.requests.cpu), the denominator is zero — HPA cannot compute utilization and will refuse to scale. The HPA status will show condition AbleToScale: False with message "missing request for cpu".

Cluster Autoscaler also depends on resource requests: it schedules pods based on requested (not actual) resources. Without requests, CA has no signal to trigger node provisioning. The pod may land on an already-overloaded node, causing OOM kills or CPU throttling — not because no capacity exists but because requests were not set.

Fix: Always define resources.requests and resources.limits for every container. Use VPA in Off mode first to calibrate realistic values, then set them explicitly. Use LimitRange objects to enforce defaults cluster-wide.

Mistake 3: HPA Flapping Due to Missing Stabilization Windows

Without a scale-down stabilization window, HPA behaves like a thermostat with no hysteresis: load spikes, 20 pods are added; load drops slightly for 30 seconds, 15 pods are removed; load spikes again, 15 pods must be added. The cluster constantly adds and removes pods, every pod restart causes brief connection drops, and the churn wastes time and money.

Fix: Set behavior.scaleDown.stabilizationWindowSeconds to at least 300 (5 minutes) for most services. For services with predictable load patterns (e.g., daily peak then drop), increase to 600 or 900. Never set the scale-up window higher than 30s for services that need to respond to traffic spikes quickly.

Mistake 4: Cluster Autoscaler Cannot Scale Down Because of PodDisruptionBudget Misconfiguration

A PDB set to minAvailable: 100% (or equivalently maxUnavailable: 0) means CA can never evict any pod on an underutilized node, because doing so would momentarily drop availability below 100%. The cluster wastes money on perpetually un-drainable nodes. This is especially common with StatefulSets where operators copy PDB examples verbatim.

Fix: Set PDBs to minAvailable: N-1 (at least 1 pod can be evicted) or maxUnavailable: 1. For a 3-replica deployment: minAvailable: 2 allows CA to drain nodes safely while maintaining quorum. Review all PDBs in production with kubectl get pdb -A and flag any with 100% availability constraints.

Mistake 5: Ignoring Cooldown Periods for KEDA

Setting cooldownPeriod: 0 or a very low value for KEDA causes the notification-service to thrash: queue empties, pod scales to 0, one message arrives, pod scales to 1 (30s cold start), processes message, pod scales back to 0, another message arrives. Each scale event adds latency and node churn. The SQS poll may even show "zero messages" between polls while messages are in-flight, causing premature scale-down.

Fix: Set cooldownPeriod to at least 60–120 seconds for most queue-based workloads. For bursty queues (e.g., concert sales), set it higher (300s) so the fleet stays warm through the entire burst window. Monitor the gap between KEDA scale-to-zero events and message arrival times using CloudWatch SQS metrics + Prometheus KEDA metrics.

Architect Perspective

Architect Note

Autoscaling is not a set-and-forget feature. It is a feedback control system, and like any control system it can oscillate, saturate, or respond too slowly if tuned incorrectly. Here is how architects think about it:

  • Three layers, different time horizons — HPA operates in seconds-to-minutes, Cluster Autoscaler in minutes-to-tens-of-minutes, VPA in hours-to-days. Design your system so each layer has the capacity buffer needed to accommodate the slower layer's response time. HPA needs node capacity available before it can place pods; if CA takes 8 minutes to provision a node, you need enough existing node headroom for 8 minutes of HPA scaling.
  • Scale-out latency is a system property — from a load spike being detected to a new pod serving traffic, there are 5+ sequential steps: metric scrape → HPA evaluation → pod creation → image pull → container start → health check pass → traffic routed. For Java services with 30s JVM startup, this chain can take 2–3 minutes. Either accept this latency, use faster runtimes (Go, Node.js native for notification-service), use GraalVM native image, or pre-warm pools with predictive scaling (KEDA cron trigger).
  • Cost-conscious scaling requires target utilization calibration — setting HPA target to 50% CPU means you run at roughly 2x the required capacity on average. That is the safety buffer. For cost-optimized services, raise the target (70–80%). For revenue-critical services, lower it (40–50%). Model the cost of downtime vs. the cost of idle capacity to find the right target per service tier.
  • Predictive scaling beats reactive scaling — for TicketFlow's concert sales, the team knows exact sale times 24 hours in advance. A KEDA cron trigger pre-scales ticket-service to 30 replicas 15 minutes before sale time, eliminating the cold-start lag during the first 90 seconds of maximum traffic. Reactive autoscaling then handles anything beyond those 30 pods.
  • VPA is underutilized and high-value — most teams skip VPA because it requires restarts. The correct posture is to run VPA in Off mode on all services from day one, observe recommendations for 2 weeks, then apply them as baseline requests in your Helm chart values. This alone can reduce cluster cost by 30–40% through elimination of over-requested ghost capacity.

Production Readiness Checklist

  • All pods have resources.requests and resources.limits set — HPA and CA depend on these
  • metrics-server is installed and returning data (kubectl top pods -n production works)
  • HPA scale-down stabilization window is at least 300 seconds to prevent flapping
  • HPA scale-up policy allows aggressive burst (Percent: 100, periodSeconds: 30)
  • HPA and VPA are not both active on the same CPU metric simultaneously
  • PodDisruptionBudgets allow at least one pod to be evicted (maxUnavailable >= 1)
  • Cluster Autoscaler is deployed and has IAM permissions to modify Auto Scaling Groups
  • CA scale-down delay is tuned to your workload pattern (minimum 10m, 15m+ for bursty)
  • KEDA cooldown periods are set to prevent scale-to-zero thrashing
  • Payment-service and other latency-critical services have minReplicas >= 2 (no scale to zero)
  • Node groups have appropriate mix of on-demand (critical) and Spot (burst) instances
  • Critical pods (ticket-service, payment-service) use nodeAffinity to avoid Spot-only placement
  • VPA Off mode recommendations reviewed monthly and applied to Helm chart values
  • Predictive scaling (KEDA cron trigger) configured for known traffic peaks
  • Autoscaler metrics (HPA events, CA decisions, KEDA scale events) are visible in Grafana

Hands-On Lab: Configure HPA with Custom Metric

Lab — Requires kubectl access to a Kubernetes cluster (minikube or EKS)

In this lab you will configure an HPA on a simulated ticket-service deployment using CPU metrics, trigger scaling with a load generator, and observe the scaling behavior in real time.

Step 1: Install metrics-server (minikube only)

# Enable metrics-server on minikube
minikube addons enable metrics-server

# Verify metrics-server is running (may take 60s to start)
kubectl get pods -n kube-system | grep metrics-server

# Test that metrics are flowing
kubectl top nodes
kubectl top pods -A

Step 2: Deploy a sample ticket-service workload

# Create a namespace for the lab
kubectl create namespace lab

# Deploy a PHP-Apache app that simulates CPU-bound request processing
kubectl apply -f - <<'EOF'
apiVersion: apps/v1
kind: Deployment
metadata:
  name: ticket-service
  namespace: lab
spec:
  replicas: 1
  selector:
    matchLabels:
      app: ticket-service
  template:
    metadata:
      labels:
        app: ticket-service
    spec:
      containers:
      - name: ticket-service
        image: registry.k8s.io/hpa-example
        ports:
        - containerPort: 80
        resources:
          requests:
            cpu: "200m"
            memory: "64Mi"
          limits:
            cpu: "500m"
            memory: "128Mi"
---
apiVersion: v1
kind: Service
metadata:
  name: ticket-service
  namespace: lab
spec:
  selector:
    app: ticket-service
  ports:
    - port: 80
EOF

Step 3: Create the HPA

# Create HPA targeting 50% CPU utilization
kubectl apply -f - <<'EOF'
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: ticket-service-hpa
  namespace: lab
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: ticket-service
  minReplicas: 1
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 50
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 60   # Shorter window for lab purposes
EOF

# Verify HPA was created and is reading metrics (may show unknown initially)
kubectl get hpa -n lab
kubectl describe hpa ticket-service-hpa -n lab

Step 4: Generate load to trigger scale-up

# Open a second terminal window and run this load generator
# It sends continuous HTTP requests to saturate CPU
kubectl run -i --tty load-generator \
  --rm \
  --image=busybox:1.28 \
  --restart=Never \
  --namespace=lab \
  -- /bin/sh -c "while sleep 0.01; do wget -q -O- http://ticket-service/; done"

# Back in the first terminal, watch HPA respond in real time
# Run this in a loop every 15 seconds
watch -n 15 kubectl get hpa,pods -n lab

Step 5: Observe scaling behavior and scale down

# After you see HPA scale to multiple replicas (allow 2-3 minutes):
# 1. Stop the load generator in the second terminal (Ctrl+C)
# 2. Watch the scale-down happen after the stabilization window (60s in this lab)
watch -n 15 "kubectl get hpa -n lab && kubectl get pods -n lab"

# Check HPA events for the full scaling history
kubectl describe hpa ticket-service-hpa -n lab | grep -A 30 Events

# Expected output will show:
# SuccessfulRescale  Reason: "DesiredReplicas" → scale up message
# SuccessfulRescale  Reason: "DesiredReplicas" → scale down message

# Clean up when done
kubectl delete namespace lab

Step 6: Explore VPA recommendations (bonus)

# Install VPA (if using minikube with a recent version)
# Clone the Kubernetes autoscaler repo and run the VPA installer:
# https://github.com/kubernetes/autoscaler/tree/master/vertical-pod-autoscaler

# Create a VPA in Off mode for ticket-service
kubectl apply -f - <<'EOF'
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: ticket-service-vpa
  namespace: lab
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: ticket-service

EOF

# After 5-10 minutes of traffic, check the recommendation:
kubectl describe vpa ticket-service-vpa -n lab
# Look for the "Recommendation" section showing Target CPU and Memory values

What to observe during this lab:

  1. How long it takes from load starting to HPA detecting the metric change — typically 30–60 seconds due to metrics scrape interval.
  2. The HPA shows TARGETS: X%/50% in kubectl get hpa — watch this value change under load.
  3. After load stops, HPA holds current replicas for the stabilization window before scaling down — preventing premature scale-down if load resumes.
  4. VPA recommendation values — compare them to the requests you set manually in Step 2. In a real system, the VPA target becomes the new baseline for your Helm chart values.

Interview Questions

These are the most frequently asked Kubernetes autoscaling questions in senior DevOps, platform engineering, and software architect interviews.

Q1: What is the difference between HPA, VPA, and Cluster Autoscaler? When do you use each?

HPA (Horizontal Pod Autoscaler) adjusts the number of pod replicas within a Deployment or StatefulSet. It scales out (more copies) when load increases and scales in (fewer copies) when load decreases. Use HPA for stateless services that benefit from parallel processing — web servers, API services, queue consumers.

VPA (Vertical Pod Autoscaler) adjusts the CPU and memory requests/limits of existing pods. It right-sizes individual pods rather than adding more of them. Use VPA for services where adding replicas does not help (databases, single-threaded jobs) or where you want to eliminate resource waste and over-provisioning.

Cluster Autoscaler adjusts the number of nodes in the cluster. It provisions new nodes when pods are Pending (cannot be scheduled) and removes underutilized nodes to reduce cost. Use CA whenever your workload scales beyond the capacity of existing nodes — it is the infrastructure layer that makes HPA possible at large scale.

The key insight: these three work at different layers and different timescales. HPA needs nodes to place pods on — CA ensures those nodes exist. VPA ensures each pod uses the right amount of the node's resources. In production you typically want all three active, but must carefully avoid the HPA + VPA conflict on the same CPU metric.

Q2: Can you run HPA and VPA together? What is the conflict and how do you resolve it?

Yes, but with a critical caveat: HPA and VPA must not both control the same CPU metric simultaneously. The conflict arises from a feedback loop:

  1. VPA reduces a pod's CPU request from 500m to 200m (right-sizing).
  2. Pod's CPU utilization percentage spikes instantly — it was using 100m (20% of 500m request), now it is using 100m (50% of 200m request).
  3. HPA sees CPU at 50% — above its 40% target — and scales up replicas.
  4. More replicas mean less load per pod, CPU utilization drops.
  5. VPA observes lower CPU and further reduces requests, triggering more HPA scale-up.

Resolution strategies:

  • Option 1 (most common): Use HPA on custom/external metrics (HTTP request rate, queue depth) and VPA on CPU/memory. The two controllers operate on orthogonal signals and cannot interfere.
  • Option 2: Keep VPA in Off mode — read its recommendations and apply them manually to your Helm chart values. HPA operates on CPU normally.
  • Option 3: Disable CPU as an HPA metric entirely. Let VPA manage CPU/memory resource requests. Use HPA only on non-resource metrics via KEDA.
Q3: What is KEDA and how does it extend standard HPA?

KEDA (Kubernetes Event-Driven Autoscaling) is an open-source CNCF project that extends HPA with event-driven scaling capabilities. Standard HPA supports CPU, memory, and custom metrics through the Custom Metrics API. KEDA adds two critical capabilities:

1. True scale to zero: Standard HPA minimum is 1 replica. KEDA can scale a Deployment to 0 replicas when its event source is idle (empty queue, no HTTP traffic, no scheduled events). When events arrive, KEDA scales from 0 to 1 directly, then hands off to HPA for further scaling. This enables significant cost savings for intermittently used services.

2. Rich event source ecosystem: KEDA ships with 60+ built-in scalers: AWS SQS, Kafka consumer lag, RabbitMQ queue depth, HTTP request rate, Redis list length, Prometheus queries, cron schedules, Azure Service Bus, and more. It translates these event sources into the External Metrics API that HPA can consume.

How it works architecturally: KEDA installs as a deployment in the cluster and registers itself as an External Metrics API server. It polls its configured event sources and exposes their values through the API. HPA queries KEDA's external metrics endpoint as it would any other metrics source. For scale-to-zero, KEDA directly manages the Deployment's replica count outside of HPA.

KEDA also supports ScaledJob — scaling Kubernetes batch Jobs (not just Deployments) based on event queue depth. This enables one-job-per-message patterns ideal for heavy batch processing tasks like invoice generation or video encoding.

Q4: Why might Cluster Autoscaler fail to scale down even when nodes appear underutilized?

Cluster Autoscaler scale-down is blocked by several conditions, and this is one of the most common production mysteries teams encounter:

  1. PodDisruptionBudget blocks eviction: If a PDB sets minAvailable: 100% or maxUnavailable: 0, CA cannot evict any pod from the node. Check with kubectl get pdb -A and look for zero-disruption PDBs.
  2. Pods with local storage: Pods using emptyDir or hostPath volumes cannot be evicted by default (the data would be lost). CA skips nodes with such pods unless --skip-nodes-with-local-storage=false is set.
  3. Non-evictable pods: Pods not backed by a controller (bare pods, not in a Deployment or StatefulSet) cannot be recreated after eviction. CA will not evict them.
  4. DaemonSet pods: These are not managed by CA — they are expected on every node. A node running only DaemonSet pods is technically "empty" from CA's perspective and can be removed, but DaemonSets themselves prevent eviction of their pods (they are ignored in utilization calculations).
  5. Scale-down delay not elapsed: The --scale-down-unneeded-time and --scale-down-delay-after-add timers may still be counting. Check CA logs: kubectl logs -n kube-system -l app=cluster-autoscaler.
  6. Node utilization above threshold: CA's default scale-down threshold is 50% of allocatable resources. If actual pod resource requests sum to > 50% of node capacity, CA considers the node utilized enough to keep.

The debug command: kubectl describe cm cluster-autoscaler-status -n kube-system — the ConfigMap contains a JSON explanation of why each node group is not being scaled down.

Q5: What is the HPA scale-down stabilization window and why is it critical?

The stabilization window is a duration during which HPA tracks the highest (for scale-down) or lowest (for scale-up) desired replica count computed from metrics. HPA only acts after the window elapses without the extreme value changing.

Why it matters for scale-down: Traffic is rarely perfectly smooth. A batch job completes, CPU drops from 80% to 15% for 20 seconds, then a new batch starts and CPU returns to 75%. Without a stabilization window, HPA would scale from 10 to 2 pods in those 20 seconds, then back to 10 pods 30 seconds later. Each scale event causes brief service degradation as pods drain connections and new pods warm up.

The default 300-second (5-minute) scale-down window means HPA must observe consistently low utilization for 5 minutes before scaling down. This prevents flapping on transient dips.

Scale-up has a default window of 0 seconds — scale up immediately on any spike. This asymmetry (fast scale-up, slow scale-down) is intentional: the cost of under-provisioning (dropped requests, user-facing errors) is much higher than the cost of over-provisioning (slightly higher cloud bill).

Tuning guidance:

  • Bursty, event-driven workloads (concert sales): 600–900s scale-down window to stay warm through the full event.
  • Steady web traffic with predictable diurnal patterns: 300s (default) is usually appropriate.
  • Development/staging environments: 60–120s to avoid wasting resources during idle periods.
Q6: How would you implement predictive autoscaling for a known traffic spike like a concert ticket sale?

Reactive autoscaling (HPA detecting and responding to CPU/metric spikes) has a fundamental latency problem: from metric spike to pod ready, you have 30–180 seconds depending on image pull time and application startup. For a concert sale where peak load arrives in the first 10 seconds, this is unacceptable.

Predictive scaling strategy for TicketFlow:

  1. KEDA CronScaler (15 min before sale): Configure a KEDA ScaledObject with a cron trigger that fires at the scheduled sale time. Set desiredReplicas: 30 during the sale window. This pre-warms 30 ticket-service pods and triggers CA to provision nodes 15 minutes before the flood. 
    triggers:
  - type: cron
    metadata:
      timezone: America/New_York
      start: 55 9 * * 5    # 9:55 AM every Friday (pre-scale before 10 AM sales)
      end: 30 11 * * 5      # 11:30 AM (scale down after sale rush)
      desiredReplicas: "30"
  2. HPA takes over during the sale: Once the sale opens, reactive HPA adds replicas on top of the 30 pre-warmed pods. Since node capacity already exists (CA provisioned during the cron pre-scale), new pods schedule instantly.
  3. Graceful scale-down after the sale: Increase stabilizationWindowSeconds to 900s on sale days to prevent premature scale-down in case of late traffic bursts (fans retrying failed purchases).
  4. Warm the JVM: For Spring Boot ticket-service, configure Kubernetes readiness probes with a initialDelaySeconds: 30 and a startup probe to ensure pods are JVM-warmed before receiving traffic. The 30-second buffer allows JIT compilation to kick in before the first real requests arrive.

This hybrid approach — predictive cron pre-scale + reactive HPA for overflow — gives you the reliability of pre-provisioning without the cost of running those resources 24/7. The cron trigger brings capacity online only during the event window.
